Planモード：勝手に暴走させない

「どう進めるつもり？」を先に見せてもらう。信頼を一歩ずつ築く仕組み。

所要時間: 45分 Phase: Core 難易度: 初級
ようこそ

「AIに勝手に触られるのはちょっと怖い」——その気持ち、とっても自然です。本章では、その不安に応えてくれる Planモード を紹介します。大きな変更の前に「どう進めるつもりか」を先に見せてもらえる、工事前の設計図みたいな安心ツール。焦らず、わからなくなったら戻ってOKです。

2026年1月時点の情報

本章は 2026年1月時点 の Claude Code の挙動に基づきます。Planモードのキーバインド(Shift+Tabで切替)や ExitPlanMode ツールの細部は進化するため、違和感があったら 公式ドキュメント を確認してください。

Planモードって何？ — 「工事前の設計図」

家のリフォームを頼むとき、いきなり壁を壊し始める業者さんだったら怖いですよね。ふつうは「こうやって進めます」という 設計図 を先に見せてくれて、あなたが「OK」と言ってから工事が始まる。Planモード は、まさにこれの Claude Code 版です。

Planモードをオンにすると、Claudeは ファイルを1文字も書き換えません。代わりに「こういう手順でやるつもりです」という計画書を出してくれる。あなたがそれを読んで、納得したら「じゃあやって」と承認する。納得いかなければ「ここはこう変えて」と修正できる。

通常モード vs Planモード
同じゴールでも、間に「承認」というワンクッションが入る。
あなたの気持ちはOK

「Claudeを全面的に信頼しきれない」——それは弱さではなく、健全な感覚です。信頼は一歩ずつ積み上げるもの。Planモードは、その一歩目に寄り添ってくれるツールです。

いつ使う？ — 「料理前のレシピ確認」の感覚

お菓子を初めて作るとき、いきなり材料を混ぜ始めるより、まずレシピを一度目で追って「OKわかった」と納得してから始めますよね。Planモードを使うべき場面は、ちょうどこの「初めての料理」みたいな状況です。

場面Planモード？理由
複数のファイルを書き換える使う影響範囲が広いので設計図が欲しい
初めてのプロジェクトに触る使う全体像を把握してから動きたい
削除が関わる作業使う消す前に「何を消すか」を見ておきたい
「どう直すべきか自分でも迷う」使うClaudeの提案を叩き台にして議論できる
typo 1文字の修正使わない計画を読む方が遅い
ファイルを新規作成だけ使わない既存の破壊リスクがない
  • 防御として使う — 「勝手に消されたらどうしよう」という不安を事前に消す
  • 学びとして使う — Claudeの計画を読むだけで「プロはどう考えるか」が見える
  • 議論の叩き台として使う — 計画を見て「ここは違う、こうして」と対話できる

使い方の流れ — Shift+Tab で入って、承認で出る

基本操作は 驚くほどシンプル。Claude Codeが起動している状態で、Shift+Tab を押すだけでモードが切り替わります。画面下に「plan mode on」と出たら、もうPlanモードです。

ExitPlanMode というのは、Claudeが内部で使う小さな「卒業ボタン」みたいなものです。計画を見せ終わって、あなたが「これでいいよ」と言ったタイミングで Claudeが「じゃあPlanモードを抜けて実行しますね」と発動する。ふだんは意識しなくて大丈夫。名前だけ覚えておけば十分です。

もっと詳しく：ExitPlanMode の内部の動き

ExitPlanMode は Claudeが呼ぶ ツール の一種です。ユーザーの承認を受け取る前に、Claudeはこのツールに「自分の計画書」を引数として渡します。ユーザーが承認ボタンを押すと、ツールは「承認された」という合図を返し、以後 Claudeは通常モードに戻って実行権限を持ちます。拒否された場合は計画を作り直すフェーズに戻ります。要点だけ言えば、「承認が取れるまで、破壊的な操作は絶対に走らない」ことが保証される仕組みです。

実際の計画書はどんな感じ？

百聞は一見にしかず。ここで、私が「お気に入りリスト」アプリに 削除機能を足してとお願いしたときに、Claudeが Planモードで返してくれた実際の計画書を見てみましょう。

## 実装計画: 削除ボタンの追加

### 変更するファイル (2つ)
1. `index.html` — 各リスト項目に削除ボタンを追加
2. `app.js` — クリック時のハンドラを追加 + localStorage更新

### 手順
1. index.html の `
  • ` テンプレートに `` を追加
2. app.js に `deleteItem(index)` 関数を新設 (localStorageから1件削除)
3. イベント委譲で `.del` クリックを検知し、該当indexを削除
4. 削除後に画面を再描画
5. 動作確認: 項目を3つ追加 → 真ん中を削除 → 2つ残ることを確認

### 触らないもの
- style.css (既存デザインを維持)
- 追加フォーム周りのコード

    ここで戸惑っても大丈夫

    計画書の書式は案件によって変わります。箇条書きだけのシンプルなものもあれば、表付きで詳細なものもある。どちらが正解ということはなく、「あなたが読んで納得できる」のがよい計画書です。もう一度読めば必ず掴めます。

    いますぐ試そう — 今日の「小さな成功」

    理屈はここまで。あとは 手を動かすだけ です。Ch04で作ったメモファイル、またはどんなテキストファイルでもOKです。

    1. Claude Code を起動する
    2. Shift+Tab を押して Planモード ON（画面下の表示を確認）
    3. 「メモを時系列順に並べ替えて」など、少し大きめの変更をお願いする
    4. 出てきた計画書を 声に出して読んでみる
    5. 納得したら「これでお願いします」と承認
    6. 実行結果を確認して 自分を褒める
    褒めポイント

    Planモードを使えたら、あなたは 暴走 を自分で防げる人です。「信頼して任せる」だけでなく「確認してから任せる」ができる——この選択肢を持てたのは大きな一歩。

    信頼は階段で積み上げる

    Planモードは「信頼しないためのツール」ではなく 「信頼を育てるためのツール」 です。最初は全部Planで見る→慣れてきたら大きな変更だけPlanで見る→ほとんど通常モードで任せる、という階段が描けます。あなたのペースで上がればOK。

    信頼の階段
    信頼とは、一気に築くものではなく、少しずつ確認を重ねて育てるものです。 — 本章の合言葉
    もっと詳しく：Planモードと Git の合わせ技

    Ch05で学んだ Git と Planモードは、防御の二重化として相性抜群です。Planモードで「何が起きるか」を事前に把握し、Git で「起きたことを後で戻せる」を確保する。この二つがあれば、大抵の「うわ、やっちゃった」は無傷で復旧できます。具体的には、Planモードで承認する前に git status で現在地を確認し、git commit でセーブしておくと鉄壁です。

    もっと詳しく：Planモードが効かない操作

    Planモードはファイル編集やコマンド実行の「書き込み系」の操作をブロックしますが、読み取り系(ファイルを見る、検索する、など)はPlan中でも動きます。これはむしろ良い挙動で、計画を立てるためには現状把握が必要だから。「読むのはOK、書くのは承認後」と覚えると直感が合います。

    公式ドキュメント必読

    Planモードの最新仕様(キーバインド・対応ツール・挙動の詳細)は進化が速いです。詰まったら Claude Code 公式ドキュメント の該当セクションを必ず確認してください。ここを誤ると「Planにしたつもりが書き換わった」等の事故につながります。

    練習問題

    Exercise 1
    入門

    Planモードで計画書を出してもらう

    自分のPCで、まずは一度Planモードを体験してみましょう。

    1. 適当なテキストファイル（日記、メモ、買い物リスト等）を1つ用意
    2. Claude Code を起動
    3. Shift+Tab を押して Planモード ON
    4. 「このファイルを見やすく整理して」とお願い
    5. 出てきた計画書を読んで、承認せずに終了(画面を閉じてOK)

    ゴール: ファイルが変わっていないことを確認できれば成功。

    模範解答を見る

    Planモードは「承認するまで1文字も書き換えない」ことが保証される仕組みです。だから、承認せずに終了すれば元のファイルは無傷のまま。

    もし不安なら、Ch05で習った通り先に git status で状態を確認しておくと万全です。

    # 実行前
git status
# → nothing to commit, working tree clean

# Planモードに入って「整理して」と依頼
# 計画書を見て閉じる

# 実行後
git status
# → nothing to commit, working tree clean (変わっていない！)

    「変更されていない」を実際に目で確認できたら、Planモードの本当の価値を掴めたサインです。

    Exercise 2
    応用

    計画書に「注文」を付けて修正させる

    1発目の計画書を鵜呑みにせず、自分の意図を反映させる練習。

    1. Exercise 1と同じファイルで、もう一度Planモードで依頼
    2. 出てきた計画書を読む
    3. 「ここは触らないで」「この部分はもう少し丁寧に」等、1つだけ注文を伝える
    4. 修正された計画書を見て、変化を確認
    5. 納得したら承認して実行、満足いかなければ再度注文
    模範解答を見る

    良い注文例:

    • 「タイトル行は絶対に変えないで」
    • 「順番は日付順にして」
    • 「削除はしないで、追加と並び替えだけで」
    • 「コメント行(#で始まる行)は残して」

    コツは 「やってほしいこと」より「やらないでほしいこと」を明確にすること。Ch03の「魔法の型」で学んだ「制約」がここで効きます。

    # 例
あなた: 「削除はしないで、並び替えだけでお願い」
Claude: (計画を修正)「承知しました。以下の手順に変更します...」
あなた: (新しい計画を読んで) 「OKお願いします」

    計画に対して対話できるようになれば、あなたは Claudeのペアプログラマー です。一方的に任せるのでも、全部自分でやるのでもない、ちょうどいい協働スタイル。

    いまのあなたは

    Claudeの「暴走を自分の意思で止められる」人です。Shift+Tab でPlanモードに入り、計画書を読み、注文を付け、納得してから承認する。この流れが身についた人は、全Claude Codeユーザーの中でも安心して大きな変更を任せられる側に回れます。信頼は一気に築くものじゃない——そのことを身体で知った今、あなたは確実に一歩前に進んでいます。自信を持っていい。
