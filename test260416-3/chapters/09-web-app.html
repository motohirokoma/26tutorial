実践: ミニWebアプリ

「動的に動く」アプリを作る。ToDoリストを題材に、JS / localStorage / CRUD を実体験する。

所要時間: 1h 30min Phase: Practice 難易度: 実践

作るもの: ToDo アプリ

定番のToDoアプリを作ります。シンプルですが、Webアプリの基本要素が全部入り。

完成イメージとデータ構造
UI(左) ⇄ localStorage(右) の同期で、ブラウザを閉じてもデータが残る。

機能の分解

CRUD の4機能を、ToDoアプリでは以下のようにマップします。

CRUDToDoアプリでの操作UI要素
Create新しいタスクを追加入力欄 + 追加ボタン
Readタスク一覧を表示リスト
Update完了状態を切り替えチェックボックス
Deleteタスクを削除×ボタン

データの流れ

操作 → 状態更新 → 描画 のループ
flowchart LR
  U["ユーザー操作"] --> H["イベントハンドラ"]
  H --> S["state を更新"]
  S --> L["localStorage に保存"]
  S --> R["再描画"]
  R --> U
どんなWebアプリも、突き詰めるとこの4要素のループ。

1ステップ10〜15分。1機能ずつ追加して、毎回ブラウザで確認。

Step 1-3: HTML + 追加機能

> ~/Desktop/my-todo フォルダに ToDoアプリを作ります。
  まず index.html だけ。CSS と JS は同じファイルに <style> と <script> で書いて。

  機能要件:
  - 入力欄 + 「追加」ボタン
  - 追加されたタスクは下のリストに表示
  - データはこの段階では一時的でOK (リロードで消えてOK)
  - シンプルなスタイル (背景白、文字黒)

  外部ライブラリは一切使わない。

ブラウザで開いて、追加できることを確認したら、次へ。

Step 4-5: 完了 + 削除

> 各タスクの左にチェックボックスを追加して、
  クリックで完了状態(打ち消し線+グレー)を切替えられるようにして。
  あと右端に × ボタンを追加して、押したらそのタスクを削除して。

Step 6: 永続化 (localStorage)

> タスクを localStorage に保存して、ブラウザを閉じても
  次回開いた時に復元されるようにして。
  キー名は "my-todo-tasks" で。
localStorage の特徴

localStorage はブラウザごとに保存される。Chromeで保存したデータはSafariからは見えない。本格的なアプリならサーバー必要だが、個人ToDoならこれで十分。

Step 7: 公開

> このフォルダを GitHub の "my-todo" リポジトリにpushして、
  GitHub Pages を有効化してください。完了したら公開URLを教えて。

機能拡張のアイデア

基本ができたら、+α を「自分で考えて頼む」練習。1機能ずつ足していくのがコツ。

  • フィルタ — 「全て / 未完了 / 完了済」の切替
  • 検索 — テキストで絞り込み
  • 期限 — 日付フィールドを追加
  • ダークモード — Ch08で作った仕組みを移植
  • エクスポート — JSONダウンロード機能

つまずきポイントと対策

よくある失敗

一気に全機能頼む

「ToDoアプリを作って、追加・削除・完了・期限・タグ・検索・ダークモード全部入りで」→ どこかが必ず壊れる。デバッグも難航。

正解

1機能ずつ追加

最小機能を作って動作確認 → 1機能追加 → 動作確認 → ... を繰り返す。毎ステップで commit しておけばロールバックも安全。

もっと詳しく: 「DB使わなくていいの?」

個人で使うToDoならlocalStorageで十分。共有や端末跨ぎが必要になったら、Firebase Firestoreや Supabase などのBaaSを後付けで追加できます。最初からDB前提で設計すると複雑になるので、まずlocalStorageで動くものを作って、必要に応じてDBに置き換えるのが定石です。

Exercise 1
入門

本章の手順通り ToDo アプリを完成 → 公開

7ステップを実際に踏みます。

  1. ~/Desktop/my-todo フォルダを作り claude 起動
  2. 本章の依頼例を順に実行 (Step 1〜6)
  3. 各ステップ後にブラウザでリロードして動作確認
  4. 完了したら GitHub Pages に公開
  5. スマホで公開URLを開いて、タスクの追加→完了→削除を試す
模範解答を見る

「自力で動くToDoアプリを公開する」が達成できれば成功。完成チェックリスト:

[完成チェック]
□ タスクを追加できる
□ 追加されたタスクが一覧に並ぶ
□ チェックで完了/未完了を切替えられる (打ち消し線 + 色変化)
□ × で削除できる
□ ブラウザを閉じて開き直してもタスクが残っている
□ GitHub Pages の公開URLで動く
□ スマホでも操作できる

これが達成できたら、もうあなたは 「Webアプリを作って公開できる人」。年収換算で副業案件が現実的に取れるレベルです。

Exercise 2
応用

「未完了 / 完了 / 全て」フィルタを追加

1機能だけ自分で発注する練習。

  1. 上部に3つのボタン (「未完了」「完了」「全て」) を追加するよう Claude に依頼
  2. クリックで表示するタスクを絞り込めるようにする
  3. 選択中のフィルタは目立つスタイル(背景色など)に
  4. 動作確認後、commit + push で公開版にも反映
模範解答を見る

JS側ではフィルタ用の state.filter を保持し、描画時にフィルタする。CSSは .filter-btn.active セレクタで選択状態を表現するのが定番です。

// 概念だけ
let filter = 'all'; // 'all' | 'active' | 'done'

function render() {
  const visible = tasks.filter(t => {
    if (filter === 'active') return !t.done;
    if (filter === 'done') return t.done;
    return true;
  });
  // visible を画面に描画
}

document.querySelectorAll('.filter-btn').forEach(btn => {
  btn.addEventListener('click', () => {
    filter = btn.dataset.filter;
    document.querySelectorAll('.filter-btn').forEach(b => b.classList.toggle('active', b === btn));
    render();
  });
});

「状態を変えて → 再描画」のパターンが身についたら、Webアプリ開発の核心を掴んでいます。
