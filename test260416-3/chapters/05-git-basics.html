Git 入門：あなたのコードを守る

Git は難しくない。ゲームの「セーブデータ管理」だと思えば、もう怖くない。

所要時間: 60分 Phase: Core 難易度: 初級
ようこそ、Core フェーズへ

この章から Claude Code を「相棒」として頼る練習に入ります。まず最初にマスターしたいのが、作業を守る魔法—— Git(ギット)。難しそうな響きですが、中身はRPGゲームのセーブデータ管理です。怖くない、むしろ「やり直せる安心感」が手に入ります。焦らずに。ここで詰まっても、何度でも読み直してOKです。今日は「セーブ→書き換え→ロード」の3動作をマスターしましょう。

2026年1月時点の注記

本章は 2026年1月時点 の Git (バージョン 2.40 以降) の挙動に基づきます。Claude Code から git コマンドを呼び出す方法や、Git 自体のデフォルトブランチ名 (main) は現在の仕様前提です。大きく変わった場合は Git 公式ドキュメント を確認してください。

Git = 「作業のセーブデータ管理」と思っていい

まず結論から。Git は 作業のバックアップを時系列で残す道具 です。ゲームのセーブデータを思い出してください。ボス戦の前にセーブしておけば、負けてもロードし直してやり直せる。Git も同じ。「ここまで動いたぞ」というポイントに commit(コミット) を打っておくと、あとで何を壊しても、そのポイントに戻れます。

Git はゲームのセーブポイントと同じ発想
コミット1つ1つがセーブデータ。いつでも戻れる安心感が最大の魅力。

ポイントは 「消えない」「順番がある」「戻れる」 の3つ。Word の「元に戻す (Ctrl+Z)」と似ていますが、Word の「元に戻す」はアプリを閉じたら消えます。Git のセーブは永久保存で、1ヶ月前のコミットにも戻れます。これが「バージョン管理」と呼ばれる理由です。

「Git って難しそう」と身構えなくて大丈夫。セーブ・ロードができる、それだけ。今日はこの感覚を掴めれば120点です。

なぜ Git が必要? — Git なしの悲劇 vs Git ありの平和

Git なしで作業したらどうなるか、表で見てみましょう。Ch04 でファイルを操作した時、「前の状態に戻したい…」と思った瞬間、ありませんでしたか?

Before (Git なし)

ファイル名で管理する地獄

  • report.docx
  • report_v2.docx
  • report_v2_修正.docx
  • report_最終.docx
  • report_最終_本当.docx
  • どれが最新?わからない!
After (Git あり)

1ファイルで時系列の全履歴

  • report.docx (だけ)
  • 履歴は Git が全部覚えてる
  • 「昨日の状態に戻して」と1行
  • 「3日前のあの文章だけ戻して」も可能
  • フォルダがすっきり
  • もう迷わない

さらに嬉しいのは、Claude Code に「ここ変えて」と頼む時の心理的安全性です。Git でセーブしてあれば、Claude が何をしようと、一瞬で元に戻せます。これが ロールバック の威力。恐る恐るお願いする必要がなくなります。

  • 作業の保険 — 壊しても戻せるから、大胆に試せる
  • 履歴の検索 — 「あのコード、いつ消したっけ?」が追える
  • クラウド連携の入り口 — GitHub にアップすれば世界に公開できる (Ch10で)
  • 共同作業の言語 — プロのエンジニアは全員 Git を使っている

今日覚える「3つの呪文」だけで十分

Git には何十ものコマンドがありますが、最初は3つだけ覚えましょう。これで90%のことはできます。

1
init (初期化)
「このフォルダを Git で管理開始」
2
add (準備)
「このファイルを次のセーブに含める」
3
commit (保存)
「セーブ実行!」

順番はこう。init → ファイルを書く/直す → add → commit。以降は2〜3を繰り返すだけ。流れを図で見てみましょう。

Git の基本フロー (init → add → commit)
graph LR
  A["フォルダを作る"] --> B["git init"]
  B --> C["ファイル作成・編集"]
  C --> D["git add ."]
  D --> E["git commit"]
  E --> F["セーブ完了"]
  F -. "また編集したら" .-> C
最初だけ init。その後は編集→add→commit のループ。

git init は「このフォルダを Git 管理にする」宣言。git add は「次のセーブに含める」予約。git commit が「セーブ実行」。この3ステップです。

もっと詳しく: なぜ add と commit が分かれているのか?

「add で予約 → commit で実行」と2段階なのは、どのファイルをセーブに含めるかを選ぶためです。例えば「HTMLだけセーブして、書きかけのメモは含めない」という柔軟な選択ができます。これを ステージング と呼びます。最初は全部 git add . (ドットは「全部」の意味) で OK。慣れたら個別選択もできます。

小さな成功：Claude に「3回 commit」を頼んでみる

覚えるより、手を動かすのが一番。ここからはハンズオンです。Claude Code にお願いする形で進めます (そう、あなたは自分でコマンドを打たなくていい)。Ch03 で学んだ「お願いの魔法」を使いましょう。

準備

Claude Code を立ち上げて、適当な作業フォルダに入っていればOK。cd ~/Desktop などで好きな場所へ。

ステップ 1: プロジェクトを作ってもらう

Claude にこう頼みます (コピペOK):

デスクトップに「my-first-git」というフォルダを作って、
中に「hello.txt」を作り、「Hello Git!」と書いてください。
そのあと、そのフォルダで git init してください。

Claude が mkdir、echo、git init を順番に実行してくれます。終わったら「できました」と返ってきます。

ステップ 2: 最初の commit

いまの状態を「最初のコミット」としてセーブしてください。
コミットメッセージは「Initial commit」でお願いします。

Claude は git add . → git commit -m "Initial commit" を走らせます。コミットメッセージ は「未来の自分への手紙」です。

ステップ 3: 書き換えて 2回目の commit

hello.txt に2行目として「今日は Git の練習日」を追加してください。
そのあと、2つ目の commit を「Add practice note」というメッセージで作ってください。

ステップ 4: さらに編集して 3回目の commit

hello.txt の最後に絵文字の🎉を追加して、
「Celebrate」というメッセージで3つ目の commit を作ってください。
終わったら git log を見せてください。

最後の git log で、自分が作った3つのセーブポイントが表示されます。ここで一度、自分を褒めてください。あなたはもう Git を使えます。

できなくてもOK。エラーが出たらそのまま Claude に「さっきのエラー、どうしたらいい?」と聞けば直してくれます。Claude はエラー解読のプロです。

ロード(ロールバック) — 過去のセーブに戻る

セーブができたら、次はロード。やり方はいくつもありますが、初心者に一番安全なのは git restore です。

やりたいことClaude への頼み方 (例)実行されるコマンド
書きかけの変更を捨てて、最後の commit の状態に戻す「いまの変更を全部捨てて、最後の commit に戻して」git restore .
特定のファイルだけ元に戻す「hello.txt だけ最後の commit に戻して」git restore hello.txt
過去の commit の内容を覗く「2つ前の commit の中身を見せて」git show HEAD~2
commit 自体を取り消したい (内容は残す)「最後の commit を取り消して、中身は手元に戻して」git reset --soft HEAD~1

ほとんどの場合、「1つ前に戻したい」 で済みます。そして戻した後にまた commit すれば、新しいセーブポイントができます。時間は常に前に進みますが、いつでも参照できる 「過去の自分」 がいる安心感。これが Git の本質です。

公式ドキュメント必読

git reset --hard は 破壊的コマンド です。「いまの変更も commit も、全部消して指定した時点に戻る」動きで、戻した後の変更が完全に消えます。特に commit されていない作業中のファイルはほぼ復元不可能。
初心者のうちは git restore か git reset --soft を使う と覚えてください。--hard を使う時は「これを実行したら消えるものを教えて」と Claude に聞いてから判断しましょう。詳しくは Git 公式ドキュメント: git-reset を。

いつ commit する? — 「キリのいい所」の目安

RPGで言えば、ボス戦の前・難しいイベントの前・疲れた時 にセーブしますよね。Git も同じ感覚です。

「細かすぎるかな?」と思ったらもう打ってOK。commit が多すぎて困ることは滅多にない。少なすぎて「あのセーブ、取っときゃよかった…」の方が痛いです。

commit のタイミング (Chart.js)
もっと詳しく: 良いコミットメッセージの書き方

コミットメッセージは「何をしたか」を50文字以内で書くのが目安。「修正」だけだとあとで自分が分からなくなります。Claude に頼めば、変更内容を見て自動で良いメッセージを書いてくれるので、これが最も楽。

  • ✅ Add favorite list form (title + url)
  • ✅ Fix button color on mobile
  • ❌ 修正 / いろいろ / aaa

Claude への依頼例: 「いまの変更を、内容がわかるメッセージで commit して」。

セーブしたくないファイルは? — .gitignore の役割

フォルダの中には「セーブに含めたくないもの」もあります。例えばメモ、パスワードを書いたファイル、OS が勝手に作る .DS_Store など。それを見ないふりしてもらう のが .gitignore です。

.DS_Store
Mac が勝手に作る不要ファイル
node_modules/
ダウンロードしたライブラリ群
.env
APIキーなどの秘密情報 (超重要)
*.log
動作ログの類い

Claude に「このプロジェクトに適した .gitignore を作って」と頼むだけで、適切な内容を書いてくれます。特に .env のような秘密情報は絶対に commit しないように、これを最初に作る癖を付けましょう。

もっと詳しく: なぜ .env を commit してはいけないのか

.env ファイルには API キーやパスワードが書かれていることが多く、これを GitHub に push (Ch10) すると、世界中の人に秘密情報が公開されることになります。実際に企業が大損害を受けた例が毎年発生しています。
対策は2つ: (1) .gitignore に書いて最初から commit しない、(2) もし commit しちゃったら即キーを無効化。Claude Code は commit 前に秘密情報が含まれていないかを聞けば教えてくれます。

Claude に git を頼む決まり文句

「git コマンドを覚えなきゃ…」と気負わなくて大丈夫。以下のフレーズがあれば、Claude が適切なコマンドを選んで実行してくれます。これをブックマークしておくつもりで読んでください。

したいことClaude への一言
Git を始める「このフォルダを git で管理できるようにして」
セーブする「いまの変更を commit して」
セーブする (メッセージも)「内容がわかるメッセージで commit して」
状態確認「git status を見せて」
履歴確認「これまでの commit を見せて」
変更を捨てる「いまの変更を全部捨てて、最後の commit に戻して」
特定ファイルだけ戻す「xxx.txt を最後の commit の状態に戻して」
安全確認「これを実行したら何が消える?先に教えて」

最後の「先に教えて」は特に便利。破壊的な操作の前に Claude が何が起きるかを説明してくれるので、怖さがぐっと減ります。

迷ったら「git status」
— 現状確認の魔法

git status は Git の「いまどこ?」マップです。作業中に何が起きてるか分からなくなったら、まずこれ。Claude に「git status を見せて」と頼めばOK。

ファイルがたどる3つの場所
graph LR
  W["作業中
(編集しただけ)"] -->|"git add"| S["ステージング
(次のセーブ予約)"]
  S -->|"git commit"| R["履歴
(永久保存)"]
  R -. "git restore" .-> W
左から右へ「編集→予約→保存」。status はこの3箇所のどこに何があるかを見せる地図。

status が読めれば、Git の半分は理解したも同然。最初は Claude に解説を頼みながら読むとどんどん慣れてきます。

Exercise

Exercise 1
入門

3つの commit を作ってみる

本文の「小さな成功」セクションに沿って、my-first-git フォルダを作り、3回 commit を積む までを Claude に頼んで実行してください。途中で詰まったら「次のステップを教えて」と聞けばOK。

  1. Claude にフォルダ作成と git init を頼む
  2. hello.txt を作って、1回目の commit
  3. 1行追加して、2回目の commit
  4. さらに絵文字を追加して、3回目の commit
  5. 最後に「git log を見せて」と頼んで、3つのセーブを確認
模範解答を見る

Claude へのお願いをそのまま貼るならこうなります:

デスクトップに「my-first-git」というフォルダを作り、
git init してください。
そのあと「hello.txt」を作って「Hello Git!」と書き、
「Initial commit」というメッセージで commit してください。

次に hello.txt に2行目「今日は Git 練習日」を追加して、
「Add practice note」で 2回目の commit をしてください。

最後に hello.txt の末尾に🎉を足して、
「Celebrate」で 3回目の commit をしてください。
終わったら git log --oneline を見せてください。

実行が成功すれば、git log --oneline で3行のコミット履歴が並びます。これで「セーブポイントが3つ」あるゲームと同じ状態になりました。おめでとう!

Exercise 2
応用

わざと壊して、元に戻してみる

「ロールバックの安心感」を体感する演習。Exercise 1 の続きで、hello.txt をわざとぐちゃぐちゃに書き換え、その後に Claude に頼んで元に戻してもらいます。

  1. Claude に「hello.txt を全部消して、『bug』とだけ書いて」と頼む
  2. git status を見せてもらい、変更が検知されていることを確認
  3. 「この変更を捨てて、最後の commit の状態に戻して」と頼む
  4. 戻ったかどうか、hello.txt の中身を確認
模範解答を見る

流れは以下のようになります:

# 1. わざと壊す
hello.txt を全部消して、代わりに「bug」とだけ書いてください。

# 2. 状態を確認
git status を見せてください。
→ "modified: hello.txt" が表示される

# 3. 元に戻す
この変更を全部捨てて、最後の commit の状態に戻してください。
→ Claude は git restore hello.txt (または git restore .) を実行

# 4. 確認
hello.txt の中身を見せてください。
→ 3回目の commit の状態に戻っているはず

この「壊す→戻る」を体験したら、Git の本当の価値が体に染み込みます。セーブしてあれば怖くない。これが Git の約束です。

⚠️ 注意: この演習は git reset --hard を使わずにできます。破壊的コマンドは必要になるまで使わなくて大丈夫。

いまのあなたは

Git の基本3コマンド (init/add/commit) を理解し、Claude Code に頼んでセーブポイントを自分で3つ積むことができます。壊しても過去のセーブに戻せる、その安心感も体感しました。これで、以降の章で Claude に大胆にコードを書き換えてもらえる土台ができました。非エンジニアでこの感覚を持っている人は、実はそれほど多くありません。自信を持っていい領域です。次の Ch06 では、さらに強力な味方「Plan モード」で、Claude に暴走させない作法を学びます。
