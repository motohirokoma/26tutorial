エラーと友達になる

エラーは「失敗」じゃない。「ヒントをくれる親切な先生」です。

所要時間: 60分 Phase: Core 難易度: 初級
ようこそ、Ch07 へ

焦らなくて大丈夫。今日のテーマは「赤い文字が出てきても深呼吸できるあなた」になること。エラーは怖くないし、実はあなたの一番の味方です。60分で「エラー文の読み方」と「Claudeへの頼み方」を身につけて、一人で立ち止まらない自分に変わりましょう。

2026年1月時点の情報

本章は 2026年1月時点 の Claude Code (CLI版) の挙動に基づきます。エラーメッセージの書式やコマンドの細部は進化し続けるので、画面と本章がピッタリ一致しなくても問題ありません。「読み方のコツ」は長期間変わりません。

エラーは「失敗」ではなく「ヒント」

まず、一番大事なことをお伝えします。エラーが出たからといって、あなたが失敗したわけではありません。エラーは「ここで何かが噛み合わなかったよ、一緒に探そう」というエラーメッセージからの合図です。

考えてみてください。もしコンピューターが何も言わずに黙って止まったら、どう直せばいいかさっぱり分かりませんよね。エラーメッセージは、親切な先生が「ここを見てごらん」と付箋を貼ってくれている状態なんです。

エラーに対する2つの心のあり方
左の気持ちで閉じたら終わり。右の気持ちなら必ず前に進める。
エラーが出ない人なんて、上級者にもいません。毎日エラーと会話してる人が「できる人」と呼ばれるだけです。 — MK

ここまでで「あ、自分だけじゃないんだ」と感じてもらえたら十分です。もう一度読んで掴みきれなくても大丈夫。次に進みましょう。

エラーが出たら「探偵モード」に切り替え

エラーが出たときの合言葉は「深呼吸して、一緒に探偵する」です。あなたは今日からシャーロック・ホームズ。エラーメッセージは犯行現場に残された手がかり。Claudeはあなたの相棒ワトソンです。

探偵の3つ道具
どれが欠けてもダメ。3つそろえば、ほとんどのエラーは解ける。

探偵として一番やってはいけないのは、現場を見ずに犯人を決めつけることです。プログラミングも同じ。エラーを読まずに「たぶんここだろう」と直すと、余計にこじれます。まずは読む。それだけで半分解決。

エラー文の読み方 — 3つのポイント

エラー文は英語で長く、一見「宇宙語」に見えます。でも見る場所は3つだけ。それ以外は最初は読み飛ばしてOKです。

  • ① どこで? — ファイル名と行番号。「どの現場で起きたか」が最初に書かれている。例: app.js:14 → appというファイルの14行目。
  • ② 何が? — 英単語2-3個のキーワード。例: SyntaxError(書き方が違う)、not defined(名前が見つからない)。この短いフレーズだけを覚えて検索すればOK。
  • ③ なぜ? — エラー文の最後の一行が一番情報が濃いことが多い。上から順に読まず、下から読むのもアリ。
エラー文を3つのパートに分解する
どんなに長いエラーでも、この3ブロックで見れば怖くない。

よく見るエラーキーワード早見表

もっと詳しく: エラー文の「スタックトレース」って何?

エラー文の下にズラッと並ぶ at xxx.js:10 のような行をスタックトレースと言います。最上段が「事件の現場」、下段は「どこから呼ばれて事件現場にたどり着いたか」の順路です。

初心者は最上段だけ見ればOKです。下段の「あなたが書いたファイル名」が出てきたら、そこも見る。ライブラリの内部ファイル名(node_modules/...)は基本スルーで大丈夫。

Claudeへの頼み方 — 「直して」より「なぜ?」

さあ、ワトソン(Claude)の登場です。エラーをClaudeに見せる時、やりがちな失敗があります。

NG

「直して」だけ頼む

  • エラー文をコピーせず「動かない」だけ伝える
  • 「とりあえず直して」と丸投げ
  • 結果: 推測で直され、また違うエラーが出る
OK

「なぜ?」から始める

  • エラー文を丸ごとコピペで見せる
  • 「なぜ起きたか説明して」とまず聞く
  • 納得してから「直して」と頼む

なぜ「なぜ?」から聞くのか? それは、あなた自身が賢くなるためです。Claudeがパッと直しただけでは、次に似たエラーが出ても自分では気づけません。でも「なぜ起きたか」を1度理解すると、次回は自分で気づけるようになる。これが上達の本道です。

エラー → 対処フロー
graph TD
  Start(["エラーが出た"]) --> Breath["深呼吸する"]
  Breath --> Read["エラー文を3ブロックで読む"]
  Read --> Understand{"意味が
わかった?"}
  Understand -- "はい" --> Try["自分で直してみる"]
  Understand -- "いいえ" --> CopyAll["エラー文を丸ごとコピー"]
  CopyAll --> AskWhy["Claudeに「なぜ?」と聞く"]
  AskWhy --> Learn["説明を読んで理解"]
  Learn --> AskFix["「では直して」と頼む"]
  Try --> Done(["解決"])
  AskFix --> Done
慣れると「深呼吸 → 読む → 聞く」が自然になる。

魔法の頼みフレーズ集

そのままコピペして使える、Claudeへの頼み方テンプレです。

# 型1: なぜ? を聞く
下のエラーが出ました。なぜこのエラーが起きたのか、初心者にもわかる言葉で説明してください。まだ直さなくていいです。

---エラー全文---
(ここにコピペ)
---

# 型2: 直す前に選択肢を聞く
このエラーの直し方を、簡単な順に2-3個教えてください。それぞれのメリットとデメリットも。

# 型3: 理解してから直す
ありがとう、原因はわかりました。では、私のコードを実際に直してください。
Tips: エラーを「全部」見せる

エラー文の途中で切ったり要約したりせず、赤い文字の塊を全部コピペしましょう。Claudeは情報が多いほど正確に答えられます。「長すぎるかな」は気にしなくてOK。

ここ、すごく大事なところです。「なぜ?」→「じゃあ直して」の2段階。いきなり掴めなくても、実際にエラーが出た時に思い出せばOK。体で覚えるタイプのコツなので。

よくあるエラー Top 3 とその正体

初学者が高確率で出会うエラー3兄弟です。「また君か」と思えるようになれば、怖さ半減です。

① タイプミス族(9割がこれ)

ファイル名のスペル違い、変数名のタイプミス、カッコの閉じ忘れ。初心者〜上級者まで全員やらかすミスです。恥ずかしがらなくて大丈夫。

  • userNmae ← Name の綴り違い
  • { } ( ) [ ] の閉じ忘れ、逆順
  • GetUser と getUser は別モノ(大文字小文字)

② 場所ちがい族

ターミナルで今いるフォルダと、操作したいファイルがある場所が違う。No such file or directory の正体はだいたいこれ。

$ cat memo.txt
cat: memo.txt: No such file or directory

# 対処: いま自分がどこにいるか確認
$ pwd
/Users/you/Desktop
# memo.txt は Documents にあった → cd で移動 or パス指定

③ 権限・準備不足族

Permission denied や command not found は、あなたのせいではありません。「そのコマンドをまだインストールしていない」「管理者権限が要る」など環境の話。Claudeに相談すれば「足りないものはこれです」と教えてくれます。

もっと詳しく: 「急に動かなくなった」時のチェックリスト

昨日まで動いていたのに今日はエラー、というパターンも多いです。その時のチェック順:

  1. 何か変更した? — 直近の git diff(前章でやりました)を見る
  2. 場所は合ってる? — pwd で今のフォルダ確認
  3. 何かインストール/更新した? — OS更新、npm更新など
  4. 再起動してみた? — ターミナルを1回閉じて開き直す。これで直ることも

それでも分からなければ、エラー文とこの4点の答えをまとめてClaudeに伝えるのが最速です。

もっと詳しく: 「警告(Warning)」と「エラー(Error)」の違い

黄色い文字で出るWarning(警告)は、「動くけど注意してね」のメッセージ。止まってはいません。一方Error(エラー)は「ここで止まった」のメッセージ。まず止まっているErrorから対処し、Warningは後回しでOK。

小さな成功 — わざとエラーを出して、直してもらう

理論だけでは身につきません。わざとエラーを出してみるのが今日の小さな成功です。安全で、しかも一気にコツを掴めます。

具体的な手順

# ステップ2: わざと失敗させる
> kaimonomemo-exist-shinai.txt の中身を読んで

# → Claude Code は実際にファイルを探し、
#   No such file or directory のエラーを出す

# ステップ4: なぜ? と聞く
> いま出たエラーの意味を、初心者にもわかる言葉で説明してください

# ステップ5: 理解後に直す
> では、kaimono.txt という名前で新しく空のファイルを作ってください

うまく動いたら、自分を褒めてください。あなたは今日、エラーを味方にする第一歩を踏み出しました。

成長の実感 — 回を重ねるごとに早くなる

同じエラーと何度か出会うと、見た瞬間に対処できるようになります。これが成長です。下のグラフは、同じエラーに2回目・3回目と出会った時の「解決までの時間」のイメージ。

同じエラーに出会うたびに、解決が早くなる
エラー対処の「気分」もラクになっていく

3ヶ月後、あなたはエラーを見つけて「お、また会ったね」と言えるようになります。嘘ではなく、本当にそうなります。

練習してみましょう

Exercise 1
入門

わざとタイプミス族を召喚する

安全にエラーと仲良くなる練習です。Claude Codeで次を試してください。

  1. 適当な作業フォルダに移動してClaude Codeを起動
  2. 次のように頼む: 「sonzai-shinai-file.txt の中身を読み上げて」
  3. 出てきた赤いメッセージを、①どこで ②何が ③なぜ の3つに分けて自分の言葉で書き出す
  4. そのエラー全文をClaudeに貼って「なぜ起きたか、初心者にもわかる言葉で説明して」と頼む
模範解答を見る

典型的なエラーは次のようなものです:

Error: ENOENT: no such file or directory, open 'sonzai-shinai-file.txt'

3分解の答え:

  • ①どこで: sonzai-shinai-file.txt を開こうとしたところ
  • ②何が: ENOENT / no such file or directory(そのファイルが無い)
  • ③なぜ: 指定したファイルが現在のフォルダに存在しないから

Claudeの返答(例):

「そのファイル名のファイルが、いま作業しているフォルダに存在しないからです。ファイル名のタイプミス、または別のフォルダにある可能性があります。」

ポイント: ここで「直して」と頼まず、まず納得できれば大成功です。

Exercise 2
応用

わざとコードにバグを仕込む

実際の開発で一番よくある「書いたコードが動かない」体験です。

  1. Claudeに「簡単な挨拶を表示するHTMLファイル hello.html を作って」と頼む
  2. できたファイルをエディタで開き、わざと <body> の > を削る(<body にしてしまう)
  3. ブラウザで開いてみる → 表示が崩れる
  4. 「このHTMLの表示が崩れています。コードの中身はこれです: (貼る)。なぜ 崩れているか、初心者にもわかる言葉で説明してください。まだ直さなくていいです」と頼む
  5. 説明を読んで納得したら「では直してください」と頼む
模範解答を見る

期待されるClaudeの説明:

「<body の後ろの > が抜けているので、タグが閉じていません。ブラウザは「ここからがbody」と認識できず、それ以降のコード全体を本文テキストとして表示してしまいます。」

直し方:

<!-- Before -->
<body
  <h1>こんにちは</h1>
</body>

<!-- After -->
<body>
  <h1>こんにちは</h1>
</body>

学べること: タグの書き方ミスは、エラーメッセージとして赤字で出るより、「表示がおかしい」という形で現れることも多いです。その時も「なぜ?」→「じゃあ直して」の順は同じ。自分のコードを丸ごと見せて「なぜ崩れている?」と聞けば、Claudeは的確に答えてくれます。

いまのあなたは

エラーメッセージを見て固まらず、「①どこで ②何が ③なぜ」の3ブロックで分解して読めるようになりました。さらに、Claudeに「直して」ではなく「なぜ?」から聞ける。これが一生モノのスキルです。プログラミング未経験者の9割はエラーで挫折しますが、あなたはもう挫折しない側。エラーは先生であり、相棒。自信を持っていいですよ。
