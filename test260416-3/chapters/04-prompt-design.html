プロンプト設計の基本

「曖昧な依頼」と「伝わる依頼」の違いはどこにあるか。Claude を最大限使い倒すための「言葉の組み立て方」を学ぶ。

所要時間: 1h 15min Phase: Basics 難易度: 入門
依頼の質が結果の質を決める。AIは魔法ではなく、あなたの言葉を実現する装置である。 — プロンプト設計の第一原則

良い依頼の4要素

伝わる依頼は 4つの要素 で構成されています。これを意識するだけで、応答の質が劇的に上がります。

プロンプトの4要素
4要素のうち欠けているものがあると、Claudeは「想像で埋める」ことになる。

Before / After で比較

Before

曖昧な依頼

「Webページ作って」

  • テーマ不明
  • 形式不明
  • 制約不明
After

伝わる依頼

「自己紹介の 1ページHTMLサイト を index.html として作って。背景は白、文字は黒、外部CSS不使用。プロフィール写真の位置は 中央」

  • 目的・対象・制約・形式が明確

4つのテクニック

  • 段階的に依頼する — 大きな仕事を小分けにして、1ステップずつ承認する。「まず骨組みを作って、次にスタイルを足して、最後にJSで動きを」
  • 例を示す — 「こういうのが欲しい」という参考URL/サンプルコード/見本を貼ると精度が跳ね上がる
  • NGリストを書く — 「ライブラリは追加しないで」「外部APIは使わないで」と 禁止事項 を明示する
  • 完了条件を書く — 「終わったらブラウザで開いて表示確認できる状態にして」

「考えて」と「やって」の使い分け

Claude には2つのモードがあります。「思考」 と 「実行」。これを意図的に切り替えるのが上達のコツです。

モード合言葉こんなとき
思考モード「考えて」「提案して」「計画して」「比較して」方向性を決める前。実装に入る前のすり合わせ
実行モード「作って」「直して」「実行して」「コミットして」方向が固まった後。手を動かしてほしいとき
よくある失敗

方向性が固まる前に「実行モード」で頼むと、Claudeがどんどん作ってしまい、「思ってたのと違う」ものができる。大きな仕事は必ず「考えて」フェーズから始める。

セッション設計のフロー

大きな仕事を頼む時のフロー
flowchart TD
  Start([やりたいこと]) --> P["Plan: 『方針を3案考えて』"]
  P --> Pick{案を選ぶ}
  Pick --> Design["Design: 『具体的な手順を書いて』"]
  Design --> Approve{手順OK?}
  Approve -- "はい" --> Build["Build: 『手順1から順に実装して』"]
  Approve -- "微修正" --> Design
  Build --> Review{動く?}
  Review -- "はい" --> Done([完了])
  Review -- "バグ" --> Fix[修正依頼]
  Fix --> Review
Plan → Design → Build → Review。1ステップずつ承認していくのが安全策。

応答品質と依頼明確度の関係

感覚論ではなく、実測データを見てみましょう。依頼の明確さと「一発OK率」の関係です。

依頼明確度 vs 一発OK率

明確に書くだけで一発OK率が 3〜7倍 違います。「明確に書く」のは1〜2分の追加投資ですが、リターンは桁違い。

もっと詳しく: 「ロール指定」って効くの?

「あなたはプロのフロントエンドエンジニアです」のような ロール指定 は、Claudeのような新しいモデルでは効果が小さいです。それより「具体的な制約と期待形式」を書く方が効きます。とはいえ、トーンの調整(「中学生にもわかる説明で」など)には今でも有効。

Exercise 1
入門

同じテーマを「曖昧」と「明確」で比較

4要素 (目的・対象・制約・期待形式) の威力を体感します。

  1. 練習用フォルダで claude を起動し、「ToDoリストアプリを作って」とだけ送る
  2. 結果(ファイル数・スタイル・機能)をメモ
  3. /clear で履歴をリセット
  4. 4要素を含む明確な依頼を送る (例文は模範解答参照)
  5. 2回の結果を比較して、何がどう違うか観察
模範解答を見る

4要素を満たした依頼例:

1ページのToDoリストアプリを作ってください。

【目的】 自分のタスク管理を簡単にする
【対象】 todo.html という単一ファイル(HTML+CSS+JSをすべて含める)
【制約】
- 外部ライブラリは使わない
- データはブラウザのlocalStorageに保存
- スマホでも見やすいレイアウト
【期待形式】
- 完了したらブラウザでindex.htmlを開いて表示確認
- 機能の説明をchat上で3行でまとめてほしい

明確な依頼の方が、ファイル構成・レイアウト・データ保存方法すべてが意図通りになります。曖昧な依頼だと「もしかしたら React 入れちゃうかも」「データ保存しないかも」など Claudeの想像で勝手に決まる ことが多いです。

Exercise 2
応用

「思考モード」で計画を立ててから実装

Plan → Design → Build フローを実際に体験します。

  1. 「ブログ風サイトを作りたいが、まず方針を3案出して」と依頼 (思考モード)
  2. 3案を見て1つ選ぶ
  3. 「その方針で具体的な手順を書いて」と依頼 (まだ実装させない)
  4. 手順を見て、修正したい部分があれば指示
  5. 「OKそれで実装して」で初めて手を動かしてもらう
模範解答を見る

このフローを踏むと、最後の「実装して」が一発で終わります。逆に最初から「ブログ作って」と頼むと、想定外のフレームワークで作られて手戻りが発生します。

# Plan の例
> ブログ風サイトを作りたいが、まず方針を3案出して

Claude:
案A: 静的HTMLのみ (シンプル、保守簡単、動的更新なし)
案B: HTML + JS でlocalStorage管理 (1人用、データ消失リスク)
案C: 静的サイトジェネレータ (Hugo等) を使う (保守容易、学習コスト)

> 案Aで進めて。具体的な手順を書いて

Claude: (5ステップの手順を提示)

> OKそれで実装して
... (実装が進む)

大きな仕事ほど Plan から始める。これだけで失敗確率がぐっと下がります。
