エラー・困ったときの対処

「動かない」「壊れた」「Claudeが暴走した」を冷静に切り分けるための、よくあるパターン集と対処フロー。

所要時間: 1h Phase: Core 難易度: 中級

困ったときの判断フロー

困ったときに 冷静に切り分ける手順 があると、何でも自分で解決できるようになります。

トラブル切り分けフロー
flowchart TD
  S([困った]) --> Q1{"claudeが起動しない?"}
  Q1 -- "はい" --> I["インストール系: API Key / Node.js"]
  Q1 -- "いいえ" --> Q2{"返事が来ない?"}
  Q2 -- "はい" --> N["ネットワーク or Anthropic 障害"]
  Q2 -- "いいえ" --> Q3{"おかしな結果?"}
  Q3 -- "はい" --> P["プロンプト不明確 or 履歴混乱"]
  Q3 -- "いいえ" --> Q4{"ファイル壊れた?"}
  Q4 -- "はい" --> G["Git で revert"]
  Q4 -- "いいえ" --> O["その他: ログ確認"]
「症状」→「原因の見当」→「対処」の3ステップで考える。

あるあるエラー集

症状原因対処
command not found: claudenpm のグローバルパスが通ってないnpm install -g をやり直す or PATH確認
API Error 401APIキー未設定 or 期限切れ環境変数を再確認、Consoleで再発行
API Error 429レート制限 (一気に頼みすぎ)30秒待つ。それでも続くならBilling確認
Insufficient creditsAPI残高ゼロAnthropic Console で課金額追加
応答が文字化けターミナルの文字コードUTF-8に設定 (Macは通常OK)
応答が極端に遅い会話履歴が長すぎ/compact または /clear

「Claudeに頼んでもうまくいかない」の対処

ツール側ではなく、応答品質の問題。3つのパターンと対策があります。

  • 意図と違うものが出る → プロンプトを4要素 (目的・対象・制約・形式) で書き直す。Ch04参照
  • 同じミスを繰り返す → /clear で履歴リセット。混乱した文脈を捨てる
  • 大規模変更で迷走 → 「まず計画だけ立てて」で Plan モード に戻す
秘伝のテク

「あなたが今困っている理由を3つ挙げてから、対策を提案して」と頼むと、Claudeが自己診断して切り口を変えてくれることがある。

暴走を止める3つの方法

緊急停止の3段階
1
Esc
処理中断
2
Ctrl+C
2回押しで終了
3
git revert
ファイルを巻き戻す
Esc → Ctrl+C → git の順で、強さを上げていく。最終手段はGit。

「動かない」を「動く」にする手順

Webサイトやスクリプトが動かない時の 標準デバッグ手順。

Step 1
エラーメッセージをまるごとClaudeに渡す
「このエラーが出た: 〜〜〜」とエラー全文を貼る。Claudeが原因と修正案を出す。
Step 2
最小再現を作る
「このエラーが出る最小のサンプルを作って」で、シンプルなテストケースを生成。
Step 3
2分割法で犯人を絞る
「直前の変更の半分を戻して試して」を繰り返し、どの変更がエラーを引き起こしたか特定。
Step 4
公式ドキュメントを参照
「公式ドキュメントの該当箇所を引用して、正しい使い方を教えて」

挫折ポイントの統計

多くの人がどこで挫折するのか、初学者の脱落ポイントを見ておきましょう。

章別の脱落率 (実測ベース)

圧倒的な山は Ch02 (インストール) と Ch06 (Git)。設定系で詰まりやすい。逆に言えば、ここを越えれば後は楽。もしCh02-06で詰まったら、この章に戻ってきて。

「助けて」の出し方

自力で解決できないときの逃げ場マップ。一番上から順に試していけば、たいてい解決します。

  • 1. エラー文を Google に貼る — 同じことで困った人がいる確率は90%超
  • 2. 公式ドキュメントを検索 — docs.claude.com で症状を検索
  • 3. GitHub Issues — anthropics/claude-code リポジトリ
  • 4. Discord / Reddit — リアルタイムに人に聞ける
もっと詳しく: 「ChatGPTに聞いてもいい?」

もちろんOK。ただし Claude Code 自体について は ChatGPT が古い情報を持っている可能性が高いので、症状や一般的な技術問題(JS/HTML/Gitなど)はChatGPTで聞いて、Claude Code固有の挙動はAnthropic公式 or Claude本体に聞くのが安全。

Exercise 1
入門

わざとエラーを起こして対処する

エラーに「初遭遇」する練習です。本番で慌てないために、安全な環境で1回経験します。

  1. Claude を起動
  2. 「broken.js を作って、わざとシンタックスエラーを含めて」と依頼
  3. 「このファイルを Node.js で実行して、出たエラーを見せて」
  4. 「このエラーの意味と修正方法を3行で説明して」
  5. 「修正版を保存して」
模範解答を見る

典型的にはこんなエラーが出ます。Claudeは原因と修正を即座に説明してくれます。

$ node broken.js
SyntaxError: Unexpected token 'const'

[原因] 関数定義中に閉じカッコが抜けている
[修正] L3 の `function foo({` の `{` を `(` に
[修正後] 動作確認OK

本物のエラー対処も同じフローです。エラー全文をClaudeに渡せば、原因と対策が瞬時に返ってきます。

Exercise 2
応用

Claude が暴走したシミュレーション

「やめさせる」体験を1回しておきましょう。

  1. Claude に「100個のテスト用HTMLファイルを test1.html から test100.html まで一気に作って」と依頼
  2. 承認画面が出たらあえて Esc で中断
  3. ls test*.html で何個できたか確認
  4. 「全部の test*.html を削除して」
  5. 削除も承認画面が出るので、内容を確認してから OK
模範解答を見る

Esc で中断したタイミングによって、できているファイル数は0〜100個までさまざま。これが「承認制があるおかげで暴走しない」体験です。

# 中断後の状態確認
$ ls test*.html | wc -l
0  # ← 承認画面で止めたので0

# あえて1つ作っておく場合
$ ls test*.html | wc -l
3  # ← 「3つだけ作って」で止めた場合

# 削除コマンド
> 全部の test*.html を削除して
[Claude が rm test*.html を提案 → 承認 → 完了]

承認の壁があるから安心して試せる。これがClaude Codeの大きな強みです。逆に「承認なし」の --dangerously-skip-permissions モードもありますが、最初は使わないこと。
