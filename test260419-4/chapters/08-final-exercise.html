最終演習：定期タスク3つ自動化

業務から3つ選んで組み込む — Schedule と /loop を2週間運用し効果測定まで

所要時間: 25分 Phase: Practice 難易度: 応用

7章の学びを「自分の業務」に落とす

ここまで Ch01-Ch07 で、Scheduled Tasks(定時のcron実行)と /loop(dynamic / interval の自走モード)、放置運用プロンプト、そして CI監視・PRサマリ・長時間タスク自走 の3つの実戦パターンを学んできました。知識をあなたの業務に3つ埋め込むのがこの最終章の目的です。

目標は「2週間のうちに3つの定期タスクを自分の業務で動かし、コスト・効果・通知負荷を数字で把握できている」状態。放置運用はコストに直結するのでザル設計は禁物、でも怖がって小さくしすぎると効果ゼロ。絶妙なバランスを実運用で掴むのが本演習です。

Tips

最初は完璧を狙わず、5分でノートに10個の定期業務を書き出し、頻度×定型度×痛み(時間/認知)の3軸で3つに絞る。先に動かしてから磨けばよいという割り切りが肝心。2週間運用してやめる1つを見つけるのがこの演習最大の収穫です。

CONCEPT × PATTERNS × OPERATE の3軸を束ねたものが、あなた専用の定期運用テンプレートになる。

Note

「3つ全部うまく行く」ことを目指さないで構いません。3つのうち1つが残れば成功、2つ残れば合格、3つ残れば上出来。放置運用は月額コストと通知疲れが2大リスクなので、データで判断して迷わず止める勇気も学びの一部です。

段階ヒント (行き詰まったら開く)

いきなり全部開くと「答え写し」になります。まず自分で1つ書いてみて、詰まったヒントだけ開いてください。5つのヒントは 候補発見 → Schedule or Loop の選択 → プロンプト → 通知設計 → コスト見積もり の順に対応しています。

もっと詳しく: ヒント1 — 候補の見つけ方 (Ch01)

定期タスク化の良い候補には5つのシグナルがあります:

  • 週1回以上発生する — 1度きりなら手で動かしたほうが速い
  • 出力の形式が決まっている — 要約・箇条書き・Slack投稿など定型
  • 判断が単純(Yes/No or 閾値) — ビルド成功?/PR増減?/記事公開済み?など
  • 頻度/時間帯が固定 — 毎朝9時・5分間隔・毎週月曜 など
  • 時間か認知が痛い — 1回30分以上か、毎日気にかけないといけない

逆に悪い候補は: ①対話的に判断が変わる / ②前回の結果を人間が見て分岐する / ③厳密な再現性が必要(生成AIは毎回ゆらぐ) / ④データ源が認証の都度手入力を要する。この4つは手作業か Slash Command のほうが総じて速いです。

Tips

候補が6個しか出ないなら1日のタイムライン(9時→10時→…)を紙に書く。「ビルド確認」「Slackトレンド確認」「競合ニュース」「自PR状況」「夜間の計算ジョブ」など、無意識でやっている定期確認業務が必ず何個か出てきます。

もっと詳しく: ヒント2 — Schedule か /loop か(モード判定) (Ch01 / Ch02 / Ch03)

判定は3択のフローチャートでほぼ一意に決まります:

条件推奨モード理由
「毎朝9時」「毎週月曜」など時刻固定Scheduled Tasks (CronCreate)cron式 `0 9 * * 1-5` で明示。時差・祝日管理もcron側
「5分ごと」「10分ごと」など短い一定間隔の監視/loop interval or Schedule(*/5 * * * *)ビルド監視・CI確認。短間隔なら /loop のほうが起動オーバーヘッド少ない
1ステップごとに状況判断して次のスリープ長を決めたい/loop dynamicScheduleWakeup を Claude 自身が呼ぶ。長時間自走(Ch07)に最適

迷ったら 「次の起動時刻は人間が事前に決められるか?」だけ自問。Yes なら Schedule、No なら /loop(とくに判断が都度なら dynamic)。

Warning

interval モードと Schedule `*/5` は似て見えますが、interval はセッション内の反復、Schedule は独立起動。長期運用・通知責務が重いなら Schedule、短期の張り付き監視なら interval が向きます(Ch01の比較表参照)。

もっと詳しく: ヒント3 — 放置運用プロンプトテンプレ (Ch04)

Ch04 の self-contained 原則に沿ったテンプレート骨格(無人実行前提):

# 目的
[何を成果としたいか1行 / 定期実行なので同じ条件で毎回動く]

# データ源
[URL / API / ファイルパス — 認証はどう通すかも明記]

# 成功条件
[このときだけ通知する、このときは静かに終わる]

# 出力形式
[Slackテキスト / JSON / ファイル書き出し どれか固定]

# 異常時の動作
- データ取得失敗: [スキップ / リトライ1回 / 即通知]
- 前回と同じ結果: [通知しない(二重通知回避)]
- 実行時間超過: [打ち切り / 部分結果通知]

# 冪等性
[同じ入力で同じ出力になる / 副作用を起こさない]

悪い例は「毎朝PR一覧を送って」だけ。良い例は「open PR のうち自分がレビュワーで48時間未反応のものだけ、Slack #pr-reminder に優先度H/M/L付きで投稿。該当0件なら静かに終了。取得エラー時は1回リトライ後に自分DMへアラート」まで書ききったもの。曖昧さを残すと放置運用で揺らぐのが最大の落とし穴です。

もっと詳しく: ヒント4 — 通知設計(二重通知回避) (Ch05 / Ch06)

放置運用で最もユーザを離脱させるのは通知疲れ。3つの原則で設計:

  1. 状態変化のみ通知 — 「ビルド失敗したまま」なら2回目以降は静かに。前回ステータスをステートファイルに保存して比較
  2. チャンネル分離 — 緊急(CI失敗=即Slack DM) / 日次サマリ(PRレビュー=#pr-daily) / 週次(メール) で音量を分ける
  3. ミュート時間の設定 — 深夜・週末は保留し翌営業日にまとめる。cron側で時間枠制御

通知経路は1タスクに1経路が鉄則。Slack・メール・pushを全部繋ぐと冗長で、どれも読まれなくなります。

Warning

Slack Webhook は秘匿情報。環境変数 `SLACK_WEBHOOK_URL` 経由で渡し、プロンプトに直書きしない。リポジトリにコミットすると即座に トークン失効が走ります(2026年1月時点の GitHub Push Protection 仕様)。

もっと詳しく: ヒント5 — コスト見積もり(放置運用の落とし穴)

定期タスクの月額コストは3変数で概算できます:

実行回数/月
cron頻度 × 稼働日
1回あたりトークン
in + out (プロンプト+出力)
単価
モデル別 $/1M tokens

例: 「5分ごとCI監視」= 月 12 × 24 × 30 = 8,640回。1回3Kトークン消費なら月25.9Mトークン。常駐系は積もるとサイフに刺さるのでドライラン時に必ず概算する。

削減テク: ①状態未変化なら早期終了(APIコール0)で8割カット ②サマリだけキャッシュし再呼び出し ③短間隔監視は業務時間内に限る(cron で `0-23/5 9-18 * * 1-5`)。運用コストの月上限を決めて、超えたら自動停止できる設計が安心。

見るべき本当の指標は「月額コスト ÷ 月の価値(時間削減分)」。3以上なら黒字、1未満ならやめる判断を躊躇しない。

# 目的
過去24時間のAI・Claude Code関連ニュースを5件に要約し、
毎朝7時に Slack #morning-news へ投稿する。

# データ源
- Hacker News 上位100件(API)
- Anthropic公式Blog(RSS)
- GitHub Trending "Agent / LLM" タグ(HTML)

# 成功条件
Claude Code / Agent SDK / 新モデル / 新機能 の
いずれかのキーワードに該当するもの Top5

# 出力形式
Slack Block Kit JSON で #morning-news に post。
各項目: タイトル / 1行要約 / URL / 重要度(★1-3)

# 異常時の動作
- 全データ源失敗: DMで自分に "morning-news: all sources failed"
- 該当0件: "本日は該当ニュースなし" と投稿

# 冪等性
同じ日を2回実行しても、投稿は1回のみ(state file で判定)

# 完了報告
- 投稿メッセージTS: {ts}
- 取得件数合計: {count}

実行手順(Claude Code CLI の例)

  --name "morning-news" \
  --prompt-file ~/claude-prompts/morning-news.md \

# 停止したくなったら

# 初回プロンプト(Ch07で学んだ形式)
あなたはリファクタ担当エージェントです。

1. ~/refactor/todo.md を読み、未チェックの先頭1項目を取得
2. その1項目だけをリファクタしテスト実行
3. 通ったら [x] にマーク、コミット作成(msg: "refactor: {item}")
4. 通らなかったら [!] にマークして理由をログに追記
5. 残項目があれば ScheduleWakeup で30秒後に再起動
6. 残項目0 or エラー3回連続 で停止し、Slack DMへ完了通知

ルール:
- 1ターンで1項目のみ(トークン爆発防止)
- テスト失敗時は3回までリトライ、超えたら次項目へ
- 現在時刻が 08:00 を過ぎたら即停止(朝のレビュー時間確保)

考え方の解説

この3つに絞った理由は3つの設計判断です。

  1. 頻度のリズム分散 — 平日毎朝(短)/ 週1(中)/ 月2-4(長)。どれかが使えなくても他が回る。通知疲れを防ぐためにも頻度と経路の分散が効く。
  2. Schedule と /loop を両方持つ — 固定時刻2件は Scheduled Tasks、自己判断しながら進む長時間タスクは /loop dynamic。2モードを実業務で使い分けるスキルが身につく。
  3. コスト構造を分散 — 小(朝$3)/ 小(週$1)/ 大だが間欠($15)。常時高コストを作らない。月間上限を決めておけば、家計簿的にも安全。
Insight

2週間で最も効いたのは朝ニュース要約でした。コスト$3/月で「毎朝の情報収集」という認知負荷が消え、代わりに重要ニュースをSlackで1回見るだけに。時間削減よりも「気にかけないといけない小さなタスク」を消したことの効用が本命だったと後から気づきました。

この3つを自分の業務で動かせる人は、もう「定期実行を知っている人」から「定期実行でワークフローを再設計できる人」に変わっています。

応用課題: チーム共有 と コスト最適化

最初の3つが定着したら、次は個人からチーム、放置から最適化へとフェーズを上げます。

応用 1
応用

自分の定期タスクをチームに展開する

2週間動いた3つのうち、最も効果が高かった1つをチームに展開してください。重要なのは、属人化を避け誰でも再現できる形にすることです。

  1. プロンプトを変数化(Slackチャンネル名・cron時刻・担当者を env化)して Git にコミット
  2. README にセットアップ手順(認証・cron登録・失敗時の連絡先)を書く
  3. 他メンバー1人に実際に起動してもらい、30分以内で動く状態まで持っていけるか検証
模範解答を見る

MKの実例(朝ニュース要約をチーム共有したケース):

  • Git化: `morning-news/` フォルダに `prompt.md` `schedule.json` `.env.example` `README.md` の4ファイル
  • 変数化: `{SLACK_CHANNEL}` `{KEYWORDS}` `{CRON}` `{TZ}` の4つ。`.env.example` に説明コメント
  • 検証: 同僚Aが READMEのみでセットアップ → 22分で動作。詰まった箇所(SlackワークスペースID取得)を README Q&A に追記
Insight

チーム展開すると「プロンプトの抜け」が可視化されます。自分1人だと気づかなかった暗黙知(例: 特定のキーワードフィルタが社内ドメイン知識に依存していた)が、他人に渡すと必ず露呈する。展開は最強のプロンプトレビュー手段と言えます。

応用 2
応用

月額コストを半分にする最適化

2週間の実測コストを見て、同等の価値を保ちつつ月額を50%削減してください。5つの削減テクから最低2つを適用すること。

  1. ①状態未変化なら早期終了 ②業務時間内のみ実行 ③安いモデルへのスイッチ ④サマリのキャッシュ ⑤実行間隔の延長 から2つ以上適用
  2. 適用前後の日次コストを Before/After で計測(最低1週間)
  3. 満足度が0.5ポイント以上下がったら元に戻し、別の組み合わせを試す
模範解答を見る

MKの最適化実例(朝ニュース要約で検証):

  • 適用した2つ: ①前日と同じTopURLなら投稿スキップ(状態未変化) + ④取得記事のタイトル/要約をキャッシュし7日間再利用
  • 結果: API呼び出し 約40%削減、コスト $3.1 → $1.7/月(45%減)
  • 満足度: 4.3 → 4.2(ほぼ維持。むしろ重複が減って読みやすくなったとの声)

他の削減パターン例:

テク典型削減率副作用

②業務時間内のみ50-65%深夜の緊急検知を失う

Warning

「安いモデル」への切替は効果絶大だが要検証。2026年1月時点で Claude は Haiku/Sonnet/Opus の階層があり、要約レベルなら Haiku で十分なケースが多いが、判断が絡む業務(CIログの原因特定など)では Sonnet 以上を推奨。満足度で切り戻す運用が安全です。
