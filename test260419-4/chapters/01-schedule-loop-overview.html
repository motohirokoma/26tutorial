Scheduled Tasks & Loop とは

両者の違いと使い分けの全体像

所要時間: 25分 Phase: Basics 難易度: 入門
この章のゴール

Claude Code を「呼んだ時だけ動く対話ツール」から、「決めた時刻に・あるいは自分のペースで勝手に動き続ける自走ツール」 に変える2つの柱——Scheduled Tasks(定時実行)と /loop(セルフペーシング)——を区別して、自分の業務のどこに何を当てるかの地図を頭に作ります。25分で全体像を掴み、Ch02以降の各モジュールに進むための土台を揃えます。

バージョン注記

本章の記述は 2026年1月時点 の Claude Code の挙動に基づきます。CronCreate ツール / /schedule スキル / /loop スキルの仕様は進化するため、コマンドが通らない場合は 公式ドキュメント を確認してください。

2つのコンセプト — 一言で

Scheduled Tasks と /loop は似て非なるものです。片方は時計が駆動し、もう片方はClaude自身が駆動します。

Scheduled Tasks vs /loop の駆動モデル
左=外部の時計が叩く / 右=Claudeが自分で次を予約する。駆動役が真逆。

正確には、Scheduled Tasks は CronCreate ツール(または /schedule スキル)で cron 式を登録することで起動します。一方 /loop は、Claude が応答の最後に ScheduleWakeup を呼ぶことで「次ターン予約」を自分に入れる仕組みです。前者は時刻で叩かれる、後者は自分で時刻を決めて生き残る、と覚えると混乱しません。

属性比較 — どこが違うのか

両者の属性を一行で並べて比較すると、用途の差がはっきりします。

  • 週1回のレポート生成
  • 月末の請求書集計

  • 「長文を章ごとに書き切る」

  • 「状況次第で次の間隔を変える」

今すぐ試す(判別クイズ・30秒)

次のタスクはどれに当てはまりますか? 心の中で答えてから下を開いてください。

  1. 毎週月曜9時に「週次PR一覧」を Slack に投げる
  2. 今走っている CI が終わるまで見張って、失敗なら原因要約を返す
  3. 200記事のブログタイトルを1つずつ生成して、全部できたら止まる
  4. いまこの場で「今日のニュース3本」を要約して欲しい
答えを見る

1 → Scheduled Tasks(定時)、2 → /loop dynamic(完了待ち・間隔可変)、3 → /loop dynamic(終了条件あり)、4 → 自動化不要(通常の対話)。

具体的な用途 6例

「自動化すると効く」典型を 6つ 並べます。自分の業務で近いものがあれば、そこが最初の導入口。

  • CI監視(/loop interval 5分 or Schedule */5 * * * *) — GitHub Actions のビルドを定期的にチェックし、失敗時のみ通知。二重通知回避 は要設計(Ch05で扱う)。
  • PRサマリ通知(Schedule 0 9 * * 1-5) — 毎朝9時に関係するPR一覧を要約して Slack へ。朝会の準備を Claude がやっておいてくれる(Ch06で実装)。
  • ニュース要約(Schedule 0 7 * * *) — 毎朝7時に特定トピックのRSS/Webを巡回して3行要約。通勤前に読む「ミニ日刊紙」になる。
  • ヘルスチェック(Schedule */10 * * * *) — 10分ごとにサイト/APIエンドポイントを叩き、ステータスが変わった時だけアラート。
  • 長文執筆(/loop dynamic) — 1万字の記事を章ごとに分割して1時間かけて書き切る。途中で進捗ファイルに保存して再開可能に(Ch07で実装)。
  • 定期リサーチ(Schedule 週次) — 毎週金曜に競合の新着プレスリリースを調査し、サマリをMarkdownで保存。月次/四半期レビューの素材になる。

もっと詳しく: Scheduled と /loop を組み合わせるパターン

どちらか片方だけでなく、組み合わせて使うパターンも実用的です。例: 「毎朝9時に Schedule で /loop を起動して、その日の調査が終わったら自動停止」。これは Schedule=キックオフ / /loop=完遂まで自走 の役割分担。朝起動のジョブが夜までに終わればOK、という緩い運用に向きます。Ch07末尾で具体パターンを扱います。

制約と4つの落とし穴

「放置運用」は便利ですが、人が見ていない所で動く以上、見えないコストやバグが積もりやすい。先に知っておくべき4点。

4つの落とし穴マップ
4つとも対策は明確。Ch03-Ch05 で各項目の具体手法を扱います。

コスト試算(放置運用の怖さを数字で)

もっとも見落とされやすい コスト を、いくつかの設定で試算してみます。「1回あたり $0.05」を仮定した場合の月額コスト(30日換算)。

間隔ごとの月額コスト試算(1回 $0.05 仮定)
5分間隔で回すと月 $400 超え。「ずっと動かし続ける」前に一度必ず試算する。
コスト警告

放置運用はコストに直結する。「とりあえず5分ごとに動かしてみましょう」を1週間放置すると、想定外の請求につながる可能性があります。最初は毎時 or 長間隔で始め、必要に応じて間隔を縮めるのが鉄則。/loop には終了条件の明示、Schedule にはCronDelete の発動条件を必ず設計に組み込みましょう(Ch04の中心トピック)。

落とし穴ごとの対策一覧

落とし穴具体的な症状対策
コスト蓄積5分間隔 × 1ヶ月 = 月 $400超の請求間隔を最低限に・終了条件を必ず設ける(Ch04)
状態保持なし「前回どこまでやったか」を毎回忘れるtodo.md などのファイルで進捗保持(Ch07)
時差・TZミス9時指定が UTC 扱いで 18時(JST)に発火cron登録時に Asia/Tokyo を明示(Ch03)
サイレント失敗エラーで止まっても誰も気づかない成功/失敗のどちらでも通知を送る(Ch04)

もっと詳しく: どの章から入るのが効率的?

(A) 定時業務の自動化をすぐ始めたい人 → Ch01→Ch03→Ch06(PRサマリ)で朝9時の通知を先に稼働。(B) 長時間タスクに興味がある人 → Ch01→Ch02→Ch07(/loop自走)で夜まで回す実戦へ。(C) 業務全体を棚卸しして選びたい人 → 素直に順番通り、Ch04で放置運用の設計原則、Ch08で自分の業務に3つ導入、で完走。どのルートでも Ch01 は地図なのでスキップしないこと。

Exercise 1
入門

3つのタスクを Schedule / /loop / 不要 に振り分ける

下記3つのタスクを、本章のフローチャートに従って分類してください。それぞれ「どれを選ぶか」と「その根拠1行」を書き出します。

  1. 毎朝8時、今日の天気を Slack に投げる
  2. 30分後にAPIが復活したか確認し、復活していたらジョブを再開する
  3. いま手元のログファイル1本をサマリしてほしい
模範解答を見る

期待される分類と根拠は次の通り。

1. Scheduled Tasks
   根拠: 時刻が決まっている・毎日繰り返す → cron 0 8 * * *

2. /loop dynamic
   根拠: 終了条件(API復活)があり、次の間隔は状況次第
        (最初は5分、3回失敗したら間隔を15分に伸ばす等)

3. 自動化不要
   根拠: 1回だけ・今すぐやりたい → 通常の対話でOK

観察ポイント: 1 と 3 の差は「繰り返しの有無」、1 と 2 の差は「時刻固定 vs 状況駆動」、2 と 3 の差は「条件成立まで動くか」。この3軸で迷わなくなれば、Ch02以降の構文学習が自分事になる。

Exercise 2
応用

自分の業務から「自動化候補」を3つ抽出してコスト試算

Section4の6ユースケース表と、Section5のコスト試算を参考に、自分の業務 から自動化すると効きそうなタスクを3つ挙げ、それぞれ次を書き出してください。抽象語禁止(「効率化」だけはNG)。

  1. タスク名と、Schedule か /loop かの選択 + cron式(または間隔)
  2. 1回あたりの実行時間目安と、月額コスト概算(例: $0.05 × 30回/月 = $1.5)
  3. 停止条件と通知設計(成功時・失敗時に誰がどこで気づくか)
模範解答を見る

例: MK自身の「チュートリアル制作ワークフロー」のケース

【1. 毎朝8時のPRサマリ通知】
  - Scheduled Tasks, cron: 0 8 * * 1-5 (平日)
  - 1回 $0.05 × 20回/月 = $1.0
  - 停止: CronDelete、通知: Slack #morning-brief

【2. 新着Issue巡回(30分ごと)】
  - Scheduled Tasks, cron: */30 * * * *
  - 1回 $0.03 × 48回/日 × 30 = $43/月
  - 停止: 新着0件が連続10回で発生頻度半減
  - 通知: 新着あった時のみSlack

【3. 週次レポート生成(/loop dynamic)】
  - /loop dynamic, 毎週金曜16時に Schedule でキック
  - 10章ぶんを1時間かけて自走生成
  - 1回 $2.0 × 4回/月 = $8/月
  - 停止: 全章完了 or 1時間経過
  - 通知: 完成したらSlackにパス通知

ポイント: 最初から「cron式 / コスト概算 / 停止条件 / 通知」の4点セットで言語化しておくと、Ch04以降の実装がそのまま移植できる形で残る。最終章 Ch08 で、ここで挙げた3つを実装まで持っていきます。
