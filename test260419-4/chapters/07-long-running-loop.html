実践：長時間タスクを /loop で自走

夜まで回すワークフロー — 進捗ファイルで状態を持たせ、1ステップずつ自走させる

所要時間: 30分 Phase: Practice 難易度: 中級
この章のゴール

10項目あるToDoリストを 1時間かけて1つずつ処理 する、現実的な「自走ワークフロー」を組み立てます。鍵は /loop dynamic(Claudeが自分で次の起動を決める)+ todo.md による状態保持 + 明確な停止条件。この3点が揃うと、朝セットして夜戻ってきたら10件のブログ記事が書き上がっている、といった運用が現実になります。

バージョン注記

本章の記述は 2026年1月時点 の Claude Code(/loop スキル、ScheduleWakeup)の挙動に基づきます。ツール名やパラメータは更新されるため、動かない場合は 公式ドキュメント を参照してください。

コスト警告

本章のワークフローは 1時間に10回以上 Claude を起動 します。1ステップあたり数千〜1万トークン消費する想定なら、1セッションで数百円〜数千円規模。必ず上限(ステップ数・時間)を todo.md と停止条件に書き込み、暴走しても有限時間で止まる設計にしてください。

シナリオ：夜まで回したいタスクたち

「自分が寝ている間に10件ぶん終わってたら助かるなあ」というタスクは、周りに転がっています。/loop dynamic は、こうした 「1ステップの実行時間が大きい／本数が多い／順次に進めたい」 タスクの得意分野です。代表的な3シナリオを見ます。

A. ブログ記事量産

10テーマを夜まで書き上げる

  • 1本 = リサーチ+構成+執筆で15〜30分
  • 10本順次で 2.5〜5時間
  • 「朝にテーマ10個用意 → 夜チェック」の運用
B. コードリファクタ

10ファイルを1件ずつ書き換え

  • 1ファイル = 読解+修正+テストで10〜20分
  • 依存関係があるため順次実行が安全
  • 各ファイル毎にコミットを分けたい
C. 調査レポート

10社を1社ずつ分析

  • 1社 = 公式+IR+SNSで15分
  • 1社ぶんずつMarkdownで書き出し
  • 途中でエラーが出ても残り9社は進む

いずれも共通するのは 「独立な単位(=1項目)が10個前後」+「全体を一気に1ターンで処理するのは無理」 という特性。ここで /loop dynamic と todo.md が効きます。

/loop dynamic の使い方 — 自分で次を予約する

/loop dynamic は ScheduleWakeup を毎ターン呼ぶことで「自分で次の起動時刻を決める」ループです。これが interval モード(固定間隔)と異なる点。重い1ステップが終わってから次を予約するので、ステップ間の休み時間を動的に調整 できます。

/loop dynamic の動き(1サイクル)
graph LR
  A["起床"] --> B["todo.md を読む"]
  B --> C{"未完の
項目ある?"}
  C -- "Yes" --> D["1項目処理"]
  D --> E["todo.md 更新"]
  E --> F["ScheduleWakeup
次の時刻を予約"]
  F --> Z["ターン終了"]
  C -- "No" --> End["完了・予約しない"]
各ターンは「読む→1個やる→記録→次を予約」の固定4ステップ。次を予約しなければ自然終了する。

interval モードとの違い

観点/loop interval/loop dynamic
次の起動固定間隔(5分ごと等)Claudeが自分で時刻決定
ステップの重さ軽い(ポーリング向き)重い(1ステップ分単位)
完了判定CronDelete で手動次を予約しない=自然終了
向いてる用途CI監視・価格監視長編執筆・リファクタ
Ch05/Ch06との関係CIポーリングで使用本章のメイン道具

最小コマンド

実運用で使うコマンドはこれだけ。/loop スキル呼び出しに続けて 間隔を省略 すると dynamic モードになります(2026年1月時点)。

# dynamic モード(Claudeが自分で次の時刻を決める)
/loop 以下のタスクを todo.md に沿って進めてください。
完了項目にチェックを付け、1つ処理したら次の起動時刻を自分で予約。
全項目完了したら予約をやめて終了してください。

# interval モード(比較用、Ch05で使用)
/loop 5m 以下のプロンプト…
ポイント

dynamic は 「間隔を書かない」ことで有効化 します。書き忘れと見分けがつきにくいので、プロンプトの冒頭に「これはdynamicループである」と明記するとデバッグが楽です。

もっと詳しく: ScheduleWakeup の仕組みと次起動の考え方

ScheduleWakeup は「N秒後またはT時刻に、このセッションを再度起床させる」というハーネス側の予約機能です。dynamicループでは毎ターン末尾にこれを呼ぶことで継続します。キャッシュTTLが5分(2026年1月時点)なので、270秒以内に次を予約すればウォーム状態で再開、それより長いと コールド再開 になります。重い1ステップの直後は数分のクールダウンを入れても構いませんが、「次の項目にすぐ取り掛かる」ならウォーム範囲を狙うのが得。

進捗ファイル todo.md で状態を持たせる

ループの各ターンは 独立した会話文脈 です。メモリで状態は引き継げないので、ディスク上のファイルで状態を表現 する必要があります。最も簡単なのが todo.md。Markdownのチェックボックスを見て「次にやるべき項目」を自己判定させます。

todo.md テンプレート(ブログ10本のケース)
# ブログ記事10本自動執筆 todo

## メタ情報
- 開始時刻: 2026-04-18 09:00
- デッドライン: 2026-04-18 21:00
- 最大ステップ: 12(10本 + リトライ2枠)
- 1ステップ間隔: 5分以内(ウォーム再開狙い)

## タスク
- [x] 1. 「Claude Code入門」記事を articles/01.md に書く
- [x] 2. 「サブエージェント活用」記事を articles/02.md に書く
- [ ] 3. 「/loop で夜まで自走」記事を articles/03.md に書く ← 次
- [ ] 4. 「CronCreate の使い方」記事を articles/04.md に書く
- [ ] 5. 「放置運用プロンプト」記事を articles/05.md に書く
- [ ] 6. 「コスト最適化」記事を articles/06.md に書く
- [ ] 7. 「エラーからの自己回復」記事を articles/07.md に書く
- [ ] 8. 「通知設計 Slack連携」記事を articles/08.md に書く
- [ ] 9. 「dynamic vs interval」記事を articles/09.md に書く
- [ ] 10. 「最終演習: 自業務3つ自動化」記事を articles/10.md に書く

## ログ(各ターンで追記)
- 09:00 start: 1件目開始
- 09:23 done: 1件目 OK(1,540字)
- 09:27 done: 2件目 OK(1,820字)
- 09:51 start: 3件目 開始予定

## 停止条件
- 全項目 [x] → 予約せず終了
- 21:00 到達 → 未完項目を残して終了
- 同じ項目で3回エラー → その項目を SKIP としてマーク、次へ
チェックボックス・メタ情報・ログ・停止条件の4ブロック。これだけで自走に必要な情報が全て揃う。

なぜ Markdown で十分なのか

  • 人間が読める — 夜中に覗いたら進捗が一目でわかる。JSONやDBは重すぎる
  • git で差分追跡 — 各ターンの終わりに commit すれば、後から「どの項目で何分かけたか」が復元可能
  • Claudeが編集しやすい — 「- [ ] を - [x] に置換」のような操作はLLMが間違えにくい
  • 壊れても復旧可能 — 最悪手でチェックを付け直せる。バイナリ状態ファイルと違い救済しやすい
今すぐ試す(空のtodo.mdを作る)

今の作業ディレクトリで、次を実行してベースを作ってください。

cat > todo.md <<'EOF'

## メタ情報

- 最大ステップ: 10
## タスク
- [ ] 1. (ここに項目を書く)
- [ ] 2.
## ログ
## 停止条件

プロンプトから参照できるパスが決まると、以降のセクションが一気に具体化します。

停止条件 — 必ず止まるように設計する

dynamic ループは 「次を予約しなければ自然終了」 ですが、逆に言うと 予約し続ければ永遠に続く。暴走を防ぐには、3つの停止条件 を必ずプロンプトに書き込みます。

3つの停止条件の関係
完了・エラー・時間切れ。この3つのどれかで必ず止まるよう仕組む。

3条件を詳しく

条件検出動作todo.md への記録
完了全項目 [x]ScheduleWakeup を呼ばず終了「完了: N件」と末尾に追記
エラー3連発同一項目で3回失敗その項目を SKIP にして次へ、最終的に報告「SKIP: 項目#K 理由=...」
時間切れ現在時刻 >= deadline未完項目を残し終了「中断: 21:00到達、残N件」
最大ステップステップ数が上限同上(保険)「中断: step limit」
コスト制御の要

時間切れと最大ステップは 二重の安全装置。片方だけだと、項目ごとの実時間が想定より伸びた時に止まらなくなります。両方書き込むのが常道です。

もっと詳しく: エラー時の再試行戦略(指数バックオフまで行くか?)

個人運用レベルでは 「3連敗で SKIP」 で十分です。指数バックオフ・リトライキューのような重装備は、100項目・24時間運用など規模が大きくなってから検討すれば十分。ポイントは「エラー項目が原因でループ全体が止まらないこと」。1件失敗しても残り9件の記事が書き上がる方が、全件止まるより圧倒的に価値が高い。

Step 2. /loop dynamic を起動

Claude Code を開いたディレクトリで、次のプロンプトを送ります。

/loop 【dynamicループ・自走モード】
あなたは todo.md に従ってブログ記事を1本ずつ書き上げるループです。
毎ターン必ず次のフローを実行してください。

1) todo.md を Read する
2) 停止条件を先にチェック(完了/時間切れ/ステップ上限)
   - 条件を満たしたら ScheduleWakeup を呼ばず終了(ログに理由を記録)
3) 未完の最初の項目を1つ選ぶ
4) その項目を1本の記事として articles/NN.md に書く(500字前後、日本語)
5) 書けたら todo.md の [ ] を [x] に更新、ログに「HH:MM done: N件目」を追記
6) 失敗したら連続失敗カウンタを+1(3になったら SKIP にしてログ記録)
7) ScheduleWakeup で 5分後に次のターンを予約
8) ターン終了

【重要】
- 会話の前提に頼らず、毎ターン todo.md だけから状態を復元すること
- ログは時刻つきで追記、上書きしないこと
- 成果物は articles/NN.md に書き、チャット上に長文を出さない

Step 3. 走らせながら進捗を確認

ブログ10本の進捗(想定)60分で完走
1時間の自走イメージ(縦=累計記事数)
約6分に1本ペースで10本。3件目で1回エラー→再試行して継続、という現実的な曲線。
1ターンあたり時間の内訳(平均)
リサーチと執筆でほぼ全部。todo.md 更新は秒単位、間隔待ちは5分の「固定コスト」。

Step 4. 回収とふりかえり

1時間後、articles/ に10本のMarkdownが並びます。todo.md のログで「何件目で何分かかったか」も見えます。うまくいったらそのまま git commit、うまくいかなかった項目は SKIP ログから手動再実行できます。

コスト見積もり(2026年1月時点の感覚値)

1本500字のブログ記事 × 10本。1ターン入出力3〜8Kトークン想定で、合計数十〜百数十円 規模。放置時間が伸びるほど積み上がるので、必ず deadline と max-step の両方を todo.md に書いておく こと。途中でコストが想定を超えたら Ctrl+C で止めて、次回以降に間隔を長くする・項目数を減らす等で調整します。

ここまで組めれば、あとは題材を入れ替えるだけで リファクタ自走・分析レポート自走 に転用できます。

Exercise 1
入門

todo.md を書いて dynamicループを3ターンだけ回す

本章の Step 1-2 を、項目数だけ 3個に減らして 実行してみましょう。成功基準は「3本の成果物ファイルが articles/ に並ぶこと」と「todo.md のログに3回の done が記録されていること」。

  1. 項目は軽いテーマ3つでOK(例: 好きな本 / 好きな映画 / 好きな食べ物を各200字で紹介)
  2. 間隔は1分に短縮(デバッグしやすさ優先)、max-step は 5 に設定
  3. 走り終わったら todo.md のログと articles/*.md の両方を確認
模範解答を見る

最小形の todo.md とプロンプトはこの通り。

# todo.md
## メタ情報
- デッドライン: 開始+15分 / 最大ステップ: 5 / 間隔: 1分
## タスク
- [ ] 1. 好きな本を200字で紹介 → articles/01.md
- [ ] 2. 好きな映画を200字で紹介 → articles/02.md
- [ ] 3. 好きな食べ物を200字で紹介 → articles/03.md
## ログ
## 停止条件
- 全項目 [x] → 終了
- 15分到達 → 終了

# プロンプト
/loop 【dynamic】todo.md を読み、未完1項目を処理し、
成果を articles/NN.md に書き、todo.md を更新し、
1分後に次を ScheduleWakeup で予約。完了/時間切れで終了。

観察ポイント:(1) 1ターン目で1件目が done になる (2) ログの時刻が約1分間隔で並ぶ (3) 3件終わったら次が予約されず自然停止。ここで「dynamic は予約をやめれば止まる」が体感できる。

Exercise 2
応用

リファクタ自走ワークフローを設計する

自分の業務リポジトリを1つ選び、「型エラーを1ファイルずつ直していくループ」を紙の上で設計してください。実装しなくてOK、設計書だけ作ります。

  1. 対象ファイル10個を todo.md の形で列挙(相対パスで)
  2. 1ステップで行う処理(Read → 修正 → tsc で検証 → commit)の内訳
  3. エラー発生時の扱い(即SKIP vs 2回リトライ、どちらを選ぶか理由つき)
  4. コスト見積もり(1ステップ=X円想定、合計Y円まで)
  5. 「これは dynamic でなく interval の方が適している」ケースがあれば説明
模範解答を見る

模範解答の例(TypeScriptリポジトリを想定)

【todo.md】
- [ ] 1. src/api/user.ts — TS2345 修正
- [ ] 2. src/api/order.ts — TS2322 修正
(...以下10件)

【1ステップ】
1) 対象ファイルを Read
2) 型エラーを直す(既存ロジック維持)
3) npx tsc --noEmit で検証(エラー残存なら再試行)
4) git add + git commit -m "fix(types): {file}"
5) todo.md 更新・ログ追記

【エラー時】 2回リトライ → ダメなら SKIP
  理由: 型エラーは原因が複雑な場合があり、1発で直らないことが多い。
        ただし3回以上はLLMがループしがちなのでSKIPで人間に回す。

【コスト】 1ステップ=約30円想定(長めのファイル含む) × 10 + 余裕2 = 360円。

【interval が向くケース】
  ・CIの結果を5分おきに見にいく監視(Ch05)
  ・1ステップが軽く、刻みが重要なポーリング系
  → 本件は1ステップ自体が重い(tsc含む)ので dynamic 向き。

ポイント: 「1ステップの重さ」で dynamic / interval を選ぶ感覚が身につくと、Ch08の最終演習で自作ワークフローを組む時に設計判断が速くなります。
