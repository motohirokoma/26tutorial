放置運用プロンプトのコツ

無人実行前提でのプロンプト設計 — 自走しても迷わないための5要素

所要時間: 30分 Phase: Basics 難易度: 中級
この章のゴール

Scheduled Tasks や /loop で走るプロンプトは 「寝ている間に1人で動く」 前提で書く必要があります。対話モードのようにユーザーが途中で補足することはできません。本章では self-contained・エラー時挙動・必須5要素・Before/After比較を通じて、コピペで動く定期タスク用プロンプトを1本書き上げるのがゴールです。本章の記述は2026年1月時点。

対話モードと放置運用の決定的な違い

対話モードでは「あれ、ちょっと違うな」と思えば即座に口頭で修正できます。しかし放置運用は 全部Claude任せ。あとからログを見る人間が起きた結果を読むだけの世界です。プロンプト設計の発想を根本から切り替える必要があります。

対話モード vs 放置運用のコンテキスト境界
放置運用では「人間による補正」が一切効かない。プロンプトだけが命綱。
  • 無人前提 — 深夜2時に実行されても誰もいない。エラーを出したときに「放置してよいか/通知するか」は事前に決めておく必要がある。
  • ステートレス起動 — 毎回まっさらなコンテキストで立ち上がる。前回の文脈も「さっき話したこと」も存在しない。
  • 後日ログ読者 — 結果を読むのは「あとから見る人間」。出力は単体で理解できる必要がある(どの時刻の・何のタスクで・結果がどうだったか)。
  • コスト直結 — 曖昧なプロンプトは試行錯誤を増やし、トークン消費が膨らむ。放置運用はコスト倍率が効くので設計の甘さが直接的な金銭ダメージになる。
コスト警告

5分ごと実行 × 24時間 = 288回/日 × 30日 = 月8640回。1回あたりのトークンを2000から5000に増やすだけで、月のコスト差は桁違いになります。プロンプトの精度と簡潔さは、そのままコスト最適化に直結します。

放置運用プロンプト必須5要素

放置運用プロンプトは以下の5要素を全部含めると安定します。1つでも抜けると、無人時に致命的な詰まり方をします。

必須5要素チェックリスト
対話モードより「異常時動作」の比重が上がる。
もっと詳しく: 「目的」が判断の北極星になる理由

例: 「毎朝9時にPRサマリを通知」というタスクで、Claude が「該当PRが0件だった」状況に出会ったとき、目的の書き方で挙動が変わります。「朝会の話題を提供する」という目的があれば「本日レビュー対象は0件です」と明示的に通知する判断ができる。目的がない場合は Claude が自分で「空通知は迷惑かな?」と忖度してスキップし、結果として「通知が来ない = 故障なのか0件なのか不明」という状態を作り出します。北極星指示を必ず書きましょう。

エラー時の挙動3択: 止まる / スキップ / リトライ

定期タスクが失敗したとき、Claude に何をさせたいか? これを事前に決めて書いておくのが放置運用の核心です。選択肢は3つ。タスクの性質で使い分けます。

エラー時動作の選択フロー
graph TD
  Err["エラー発生"] --> Q1{"冪等な
操作か?"}
  Q1 -- Yes --> Q2{"一過性の
可能性?"}
  Q1 -- No --> Stop["止まる
(人間に通知)"]
  Q2 -- Yes --> Retry["リトライ
(3回まで)"]
  Q2 -- No --> Q3{"次回で
自己回復?"}
  Q3 -- Yes --> Skip["スキップ
(ログ記録のみ)"]
  Q3 -- No --> Stop
迷ったら「止めて通知」が最も安全。サイレント失敗が一番怖い。
挙動使いどき書き方リスク
止まる (fail-fast) データ破壊の可能性がある / 判断保留したい 「エラー時は作業を中止し、Slack の #alerts にエラー全文を通知」 人の対応がボトルネックに
スキップ (skip) ニュース要約・サマリ通知など、1回抜けてOK 「エラー時は今回の実行をスキップし、次回の定時実行を待つ」 継続失敗に気付かない恐れ
リトライ (retry) API一過性エラー / ネットワーク不安定 「最大3回、指数バックオフでリトライ。全失敗で通知」 冪等でないと二重実行
冪等性を先に考える

冪等性がないタスク(例: Slackに送信 / GitHubへコメント)をリトライさせると、二重通知や重複コメントが発生します。送信先側にIDで重複チェックを持たせるか、「送信済みフラグ」をファイルに書いてから送る設計にしましょう。

もっと詳しく: スキップを選ぶときは「連続失敗検知」もセットで

スキップを安易に選ぶと、サイレント故障が起きます。対策: 「直近N回連続で失敗したら通知を出す」という簡易ガードをプロンプトに組み込みます。例: state/last-run.json に consecutive_failures を書き、3以上で人間に通知。こうすれば1回のスキップは静かに、3回続けば人を呼べます。

Before / After: 悪い例 → 良い例

放置運用で実際に失敗したプロンプトを、改善ポイントと一緒に並べます。

ケース1: CIチェック通知

Before (悪い例)

「ビルドが失敗したら教えて」

  • どのリポジトリ? どのブランチ?
  • 「失敗」の定義は? FAILURE だけ? timeout も?
  • 誰に教える? どこに?
  • 同じ失敗が続いたら毎回通知?(通知疲れ)
  • ツール取得失敗時は?
After (良い例)

5要素+重複抑止まで明記

目的: main ブランチのビルド失敗を
  5分以内に検知しチームに即共有する。

データ源:
  gh run list --repo acme/web
    --branch main --limit 5 --json status,conclusion,headSha,url

成功条件: 最新Runがsuccessなら何もせず終了。
  failure / cancelled / timed_out のいずれかなら通知。

出力形式: Slack Webhook への POST
  {"text": "[CI Fail] @: "}

異常時動作:
  - gh 呼び出し失敗: 3回リトライ、全失敗で #alerts へ
    「CI監視ジョブ自体が失敗」と通知
  - 同じ headSha で既に通知済なら重複送信しない
    (state/last-notified-sha.txt を確認)
  - 通知後は last-notified-sha.txt を更新

ケース2: 朝のPRサマリ

Before (悪い例)

「朝9時にPRまとめて送って」

  • 自分のPR? レビュー対象? 全PR?
  • 優先度の基準なし
  • 空の日の挙動不明
  • 通知先のSlack channel不明
After (良い例)

対象・並び・空ケースを定義

目的: 朝会の前(9:00)にレビューすべき
  PRをSlackで共有し、朝会のインプットにする。

データ源:
  gh pr list --search "review-requested:@me"
    --state open --json number,title,author,url,updatedAt

成功条件: 0件でも「本日レビュー対象なし」と
  明示的に通知すること(= 故障と区別可能)。

出力形式:
  Slack #dev-morning へ以下のMarkdown:
   *本日のレビュー (YYYY-MM-DD)*
   1. #123 タイトル - @author (2日前更新)
   2. ...
  件数0なら「本日レビュー対象なし ✓」

異常時動作: gh失敗時はスキップ(明日の通知を待つ)。
  ただし3日連続失敗した場合 #alerts に通知。

ケース3: 長時間リファクタ (/loop dynamic)

Before (悪い例)

「todo.md のタスクを片付けて」

  • 順番は? 上から?
  • 1ターンで何個?
  • 終了条件は?
  • エラー時に次タスクに進む? 止まる?
After (良い例)

状態管理と終了条件を明示

目的: docs/todo.md のリファクタ項目を
  1ターン1項目ずつ完了させる。

データ源: docs/todo.md の先頭から
  [ ] で始まる未完了項目を1つ取得。

成功条件:
  1. 項目の指示通りに実装・テスト通過
  2. 該当行の [ ] を [x] に更新
  3. 次ターンをScheduleWakeup(即時)で予約

出力形式: git コミット 1件/ターン。
  メッセージ: "refactor: <項目タイトル>"

異常時動作:
  - 実装失敗: 該当行を [!] に変更、
    次項目へ進む (止まらない)
  - [ ] が1つもない: ScheduleWakeup せず終了
  - 連続3項目が [!] になった: 自動停止して
    Slackへ「人間確認が必要」と通知

出力形式を「後日ログ読者」に向けて設計する

放置運用の出力は後から人間が読むログです。単体で意味が分かる密度を持たせます。必ず入れる要素は次の通り。

1行目
実行時刻 + タスク名
[2026-01-15 09:00 daily-pr]
2行目
ステータス (OK/NG/SKIP)
状態が即読める
3行目〜
要点 (3行以内)
詳細はファイルへ
# 良い出力例(Slack通知)
[2026-01-15 09:00 daily-pr] OK
本日のレビュー対象: 3件
 1. #421 認証リトライ修正 - @tanaka
 2. #419 i18nの肩書き表記統一 - @sato
 3. #417 Vercel deploy script - @ito
詳細: https://.../runs/xyz

コスト視点の出力設計

出力字数は output tokens に直結します。放置運用の出力は150字前後で型を縛るのがベスト。詳細は必ずファイルに書き出してパスだけ通知する。

出力字数とOutput Tokenコスト(日次 × 30日、モデル別傾向の概算)
字数を150字前後で型縛りすると、月次コストが大幅に縮む。読みやすさもアップ。
もっと詳しく: 詳細をファイルに書き出す利点

詳細を logs/2026-01-15/daily-pr.md のようにファイル化する利点は3つ: (1) 通知は短く、概要だけが目に入る → チャンネル汚染しない、(2) 後から詳細を見たい時だけファイルを開く → 認知負荷の最適化、(3) git にコミットすれば履歴が追える → 「いつから失敗が始まったか」の調査が楽。コスト面でも output tokens を節約できます。

再利用のコツ

このテンプレートは埋めたもの自体をリポジトリにコミットしてください。prompts/daily-pr.md のようにプロンプトファイルをレビュー対象に載せると、チームでの運用がスムーズになります。Scheduled Tasks のプロンプトから prompts/daily-pr.md を Read させるだけで呼び出せます。

放置運用アンチパターン5選

筆者が実際に踏んだ地雷を症状 → 原因 → 対策で整理します。

症状原因対策
通知が全く来ない/気づくと1週間止まってた エラー時スキップのみで連続失敗検知がない state ファイルに consecutive_failures を記録、N回で #alerts へ
同じエラー通知が深夜に100回届く リトライループ + 冪等でない通知 通知先で sha / PR番号で重複検知 or 最大回数で強制停止
コストが月5万円 → 30万円に跳ねた 曖昧プロンプトで毎回試行錯誤 + 冗長出力 必須5要素適用 + 出力150字縛り + 詳細ファイル化
「実行されているのか壊れてるのか分からない」 成功時は無音(= 通知なし)の設計 0件ヒット時も「本日0件 ✓」を通知。ハートビート代わりに
エラーログが断片的で原因が追えない 出力に時刻・タスク名・入力が無い 1行目を [時刻 task-name] status 固定に
もっと詳しく: ハートビート通知のバランス

「毎回何かしら通知する」と「通知疲れ」が起きます。バランス: 毎日タスクなら0件時も通知・5分毎の高頻度タスクは失敗時のみ通知 + 週次でハートビート。目安として通知疲れの閾値は「1日10通以上」。これを超えると read 率が激減します。重要度で通知先チャンネルを分けるのも有効。

Exercise 1
入門

必須5要素で定期タスクプロンプトを1本書く

以下のシナリオで、本章のテンプレートを使って放置運用可能なプロンプトを書き上げてください。

シナリオ: 平日毎朝8時に、前日マージされたPR数をSlack #eng-daily に通知したい。
  1. 目的・データ源・成功条件・出力形式・異常時動作 の5要素を全て埋める
  2. 0件の日の挙動を明示する
  3. エラー時は「スキップ」扱いとし、3日連続失敗で #alerts に通知する設計にする
模範解答を見る 
【目的】 前日のチームの開発進捗を朝8時に可視化し、
   朝会インプットとモメンタム維持に使う。

【データ源】
 - コマンド: gh pr list --repo acme/web
     --state merged
     --search "merged:>=YESTERDAY_DATE"
     --json number,title,author,mergedAt,url
 - YESTERDAY_DATE は実行時に 1日前(JST)を算出
 - gh 失敗時はこのジョブ全体をスキップ扱い

【成功条件】
 - 前日マージPR件数を取得し通知送信できたらOK
 - 0件の日も「前日マージPR: 0件(静かな1日でした ✓)」
   と必ず通知する(故障と区別するため)

【出力形式】
 - Slack #eng-daily へ:
   [YYYY-MM-DD 08:00 daily-merge] OK
   前日マージPR: N件
   (件があれば)
    1. #番号 タイトル - @author
    ...(最大5件、超過は「他X件」と表記)
 - 本文150字以内、詳細は logs/merged/YYYY-MM-DD.md へ

【異常時動作】
 - gh 失敗: 今回スキップ
 - state/daily-merge.json に consecutive_failures を記録
 - 3回連続失敗で #alerts に「daily-merge が3日連続失敗」
   と通知し、consecutive_failures はリセットしない
 - 成功したら consecutive_failures = 0

ポイント: 「0件も通知」「連続失敗検知」の2点が放置運用らしさ。これを書かないと、静かな日と壊れた日を区別できなくなります。

Exercise 2
応用

既存プロンプトの「異常時動作」を診断する

自分が使っている(または過去に書いた)定期タスクのプロンプトを1つ引っ張り出し、以下3つの観点で診断してください。自前のものがなければ、チームの誰かのプロンプトを見せてもらうか、GitHub で公開されている Scheduled Tasks の例で代用可。

  1. 異常時動作が「止まる / スキップ / リトライ」のどれか明示されているか?
  2. 0件ヒット / 空データ時の挙動は書かれているか?
  3. サイレント故障(連続N回失敗で通知)のガードはあるか?

欠けている要素を補う形でプロンプトを書き直し、Before → After の差分を git diff 形式で残してください。次回、実運用で何が変わったかを記録するとなお良いです。

模範解答を見る

回答例(診断対象: 「毎時ヘルスチェックURLを叩く」ジョブ):

# Before
目的: service.example.com が生きているか確認
コマンド: curl -sSf https://service.example.com/health
失敗したらSlackに通知

診断結果:

  1. 異常時動作: 「失敗したらSlack通知」のみ。リトライ有無・止まるか継続か不明 → NG
  2. 空レスポンス / 非200時の扱い: 書かれていない → NG
  3. 連続失敗検知: なし → サイレント故障リスクあり
# After
【目的】 service.example.com のup状態を毎時確認し、
      ダウン検知から最大1時間以内にチームへ通知する。

【データ源】 curl -sSf --max-time 10 https://service.example.com/health
       (exit 0 + body="OK" で正常)

【成功条件】 HTTP 200 + body="OK" なら今回OK。
       それ以外(非200/timeout/body不一致)は失敗扱い。

【出力形式】
 - 正常時: state/hc.json を更新するだけ(通知なし)
 - 異常時: #alerts へ
   [YYYY-MM-DD HH:00 health-check] NG
   応答:  / 
   連続失敗: 回目

【異常時動作】
 - curl timeout/非200: その場で最大3回リトライ(3秒/9秒/27秒)
 - 全失敗なら state/hc.json の fails を +1、通知
 - 復旧(正常応答)時は「service復旧しました ✓」を1度だけ通知
   し fails を 0 にリセット
 - fails >= 5 で PagerDuty にエスカレーション

効果の記録例: Before運用時はダウン検知から通知まで平均22分・サイレント失敗1回発生。After では最大7分以内に検知、エスカレーションで夜間も確実に気付けるようになった。
