Scheduled Tasks (CronCreate)

cron式・通知・実行スコープ — 定時実行を1本引けるようになる

所要時間: 40分 Phase: Basics 難易度: 中級
この章のゴール

Claude Codeで「毎朝9時にPRサマリをSlackに送る」「1時間後に一度だけバッチを回す」といった定時・定期実行のタスクを、自分の手で1本作れるようになります。鍵になるのは CronCreate ツール(/schedule スキル経由)と cron式の理解、それにリモート実行スコープと通知経路です。最後は実際に1時間後の1回実行トリガーを作り、出力を TaskOutput で取り出すまで通しでやります。本章の記述は2026年1月時点の情報で書かれています。

コスト警告

Scheduled Tasks は放置すると勝手に課金が走り続けるのが最大の落とし穴です。毎分起動で空回りし続けるトリガーを1つ放置するだけで、月額が数倍に膨らみます。本章の最後で CronList と CronDelete を必ず確認してください。

CronCreate と /schedule スキルの全体像

Claude Codeで定時実行をやるには2つの入口があります。どちらも最終的には同じ「トリガー(リモートエージェント設定)」を作ります。

CronCreate と /schedule の関係図
/schedule は親切UI、CronCreate は生ツール。どちらも同じ「トリガー」を作る。
  • /schedule スキル — トリガー操作の対話UI。「毎朝9時にPRサマリ送って」のような自然文で依頼できる。内部的には CronCreate/CronList/CronDelete を呼ぶ。
  • CronCreate ツール — 生のツール呼び出し。cron式・プロンプト・通知先を構造化して渡す。再現性重視のスクリプト化に向く。
  • CronList / CronDelete — 登録済みトリガーの一覧と削除。trigger_idで指定する。
  • RemoteTrigger / TaskOutput — 今すぐ手動起動する RemoteTrigger、結果のログを取り出す TaskOutput。セットで使うことが多い。
まず覚えるのはこの4つ

CronCreate(作る) / CronList(見る) / CronDelete(消す) / TaskOutput(結果を取る)。最初はこの4つだけでOK。

cron式の基本 — 5フィールドの読み方

cron式はスペース区切り5フィールドで「いつ起動するか」を表現します。覚えるのは位置と記号だけ。

cron式 5フィールドの構造
左から順に 分・時・日・月・曜。曜日は 0=日, 1=月, ... 6=土(7=日もOKな実装あり)。

よく使う記号4つ

記号意味例読み方
*任意(全部)* * * * *毎分
,列挙0 9,12,18 * * *9時/12時/18時ちょうど
-範囲0 9-17 * * *9時〜17時の毎時0分
/間隔*/5 * * * *5分ごと

頻出パターン早見表

やりたいことcron式備考
毎分起動* * * * *テスト以外で使わない。コスト爆発注意
5分ごと*/5 * * * *CIポーリングの定番
毎時0分0 * * * *毎時1回のヘルスチェック
平日の朝9時0 9 * * 1-5PRサマリ通知の定番
毎日の18時0 18 * * *退勤前の集計
毎週月曜朝10時0 10 * * 1週次レポート
毎月1日の朝0時0 0 1 * *月次集計
15分ごと営業時間内*/15 9-18 * * 1-5平日 9-18時の15分刻み
1時間後に1回だけ(1回実行はcron式とは別枠)run_at や delay 系パラメータで指定
タイムゾーン注意

cron式の時刻はトリガー側のタイムゾーンで解釈されます。2026年1月時点のリモート実行はデフォルトUTCのことが多いので、「朝9時」はtimezone="Asia/Tokyo" 指定を忘れずに。でないとJSTの18時に動いて怒られます。

もっと詳しく: crontab.guru で自分の式を検証するクセをつける

crontab.guru に式を貼ると「この式は次に何時に動く」を自然文で表示してくれます。特に */7 のような割り切れない間隔、月末指定、曜日の境界ケースは人力検算だとハマりがち。本番トリガーを作る前に一度貼って確かめるだけで、深夜3時にSlackが鳴る事故が激減します。

Claude Codeのリモート実行では標準5フィールド構文に限定されることが多いです。@hourly 等のエイリアスが通らない実装もあるので、常に5フィールドで書くクセをつけておくと安全です。

実行環境 — リモート vs ローカル

Scheduled Tasks はどこで動くかが結果の信頼性を左右します。2026年1月時点で主な選択肢は2つ。

観点リモート (Remote Agent)ローカル (自PC cron)
起動条件クラウド側で常時待機PCが起きていて Claude Code が起動中
PCスリープの影響無し(安心)スリープ中は動かない
ファイル/リポ アクセスクラウド側のworkspaceローカルの実ファイル
秘匿情報の扱いクラウドで暗号化管理ローカルにそのまま
典型用途定時通知・CIポーリングローカルプロジェクト専用の集計
コスト感起動ごと API課金が発生同じく API課金発生(加えて自PC電気代)
デバッグTaskOutputでログ取得ターミナルで直接確認可
まず選ぶべきはリモート

「通知系」「CI監視」「PRサマリ」みたいにPCが寝ていても動いてほしいタスクはリモート一択。ローカル cron を使うのは「このマシンのファイルしか触れない」ケースだけに絞ると迷いがなくなります。

通知チャネル — Slack / push / メール

無人実行の価値は通知が届いて初めて生まれます。主な3経路と使い分け。

通知チャネルの到達速度と保存性
  • Slack (Incoming Webhook) — Webhook URL を1本もらえば完了。チームで共有・検索しやすい。CIアラート、PRサマリの定番。
  • push通知(モバイル/デスクトップ) — 即時性が最強。障害通知や「5分以内に気付きたい」用途。履歴は残りにくい。
  • メール — 到達の信頼性が高く、長文も送れる。日次/週次レポートの保存先として優秀。即時性は劣る。
使い分け3秒ルール

「すぐ気付きたい → push / チームで見る → Slack / アーカイブ性優先 → メール」。同じタスクでも通知先で価値が変わります。Slack+メール の2経路送りも現場では定番。

もっと詳しく: 二重通知を避けるチェックポイント

CI失敗を5分ごとに見続けるトリガーで同じ失敗を12回通知するのは典型的な事故です。回避策は(1) 最終通知したSHAをファイルに保存し次回起動時に比較、(2) 結果が前回と同じなら通知スキップ、(3) 復旧したときだけもう一度通知——の3点セット。プロンプトに「前回通知した commit SHA を `last_notified.txt` に保存し、同じなら送信しない」と明記するだけで実装できます。

トリガー管理 — CronList / CronDelete / RemoteTrigger

作ったトリガーは必ず一覧化と停止手段を確保しておきます。放置事故を防ぐ唯一の方法です。

トリガーライフサイクル
graph LR
  A["CronCreate
作成"] --> B["CronList
一覧確認"]
  B --> C{"動作OK?"}
  C -- "はい" --> D["RemoteTrigger
手動起動で検証"]
  D --> E["TaskOutput
結果取得"]
  C -- "いいえ" --> F["CronDelete
削除して作り直し"]
  E --> G["運用継続"]
  G -.2週に1度.-> B
  F --> A
作って終わりではなく、2週間に1度 CronList で棚卸しすると放置事故が消える。

1回実行の手動テスト(超重要)

新規トリガーは必ず RemoteTrigger で手動1回実行→TaskOutput 確認の順でテストします。cron式が正しくても、プロンプトや通知先がバグっていたら無人実行では気付けません。

ダメな例

作っていきなり放置

CronCreate(
  cron="0 9 * * 1-5",
  prompt="PRを要約してSlackに",
  slack_webhook="https://..."
)
# → 翌朝9時まで動作不明。
#    通知ミスってたら月曜に気付く
良い例

作って即手動テスト

t = CronCreate(...)
# すぐ手動起動
RemoteTrigger(trigger_id=t.id)
# 30秒後に結果確認
TaskOutput(trigger_id=t.id)
# Slackに届いたか目視

出力の扱い — TaskOutput

Scheduled Tasks は無人実行なので、「何が起きたか後で見る」手段がないと運用できません。TaskOutput は最後に起動したタスクの標準出力・応答テキスト・ツール呼び出し履歴を取ってきます。

  • 何を返すか — 最終応答・ツール履歴・エラーをまとめたログオブジェクト。
  • 保持期間 — 直近N回分までキャッシュ(2026年1月時点)。長期保存したければプロンプト側でファイルやログ外部化を組み込む。
  • 落とし穴 — 「通知成功」と「タスク成功」は別物。Slackに届いてても中身が空や誤答の可能性あり、TaskOutputで中身を点検するクセを。

ハンズオン — 1時間後に1回だけ動くタスク

理論はここまで。実際にトリガーを1つ作って、1時間後に1回だけ動かし、TaskOutputで結果を取るところまでやります。最小コストで全サイクルを体感する練習です。

今すぐ試す(コスト最小構成)

「1時間後に現在時刻と一言コメントを出力するだけ」の極小タスクを1回だけ走らせます。cron式は「明日の同時刻」など未来の1点に絞るのがコツ。終わったら必ず CronDelete で片付けます。

手順(コピペで動く)

# Step 1: Claude Code で /schedule を呼ぶ
/schedule

# → 自然文で依頼
「ちょうど1時間後に1回だけ、現在のUTC時刻と
 『hello from scheduled task』という文字列を出力するトリガーを作って。
 実行後は手動削除する前提、通知はなし、ログだけ残して」

# Step 2: 戻ってきた trigger_id をメモ
# 例: trigger_id = "sch_abc123..."

# Step 3: 念のためCronListで登録確認
/schedule list

# Step 4: (任意) RemoteTriggerで今すぐテスト実行してTaskOutputで確認
# これで動作検証が早回しでできる

生のツール呼び出しで同じことをやるなら:

# CronCreate 直接呼び出し版
# (1時間後 = 現在UTC+1h の時刻でcron式を組む例)
# 例: 現在が 2026-04-18 10:23 UTC なら、11:23 UTC に1回動かしたい
# 5フィールドで1回だけ動かすのは本質的に苦手なので、
# 近似として「その時刻の特定日・特定月」で絞る

CronCreate(
  cron="23 11 18 4 *",           # 4月18日 11:23 UTC(1回きり相当)
  timezone="UTC",
  prompt="現在時刻を出力し 'hello from scheduled task' と返す。",
  notify=null,
  one_shot=true                   # サポートされていれば一回限りで自動削除
)
1回きり実行の注意

cron式は本来「繰り返し」のためのもの。1回きり実行を表現する専用パラメータ(one_shot・run_at・delay 等、実装により名称異なる)が用意されているならそちらを使う方が事故りにくいです。日付フィールドで1点を絞る方法は、必ず実行後に自分で CronDelete することを忘れずに(翌年同日に勝手に再実行されます)。

ハンズオンの全体フロー
sequenceDiagram
  participant U as あなた
  participant C as Claude Code
  participant R as Remote Agent
  U->>C: /schedule 「1時間後に1回」
  C->>R: CronCreate(cron, prompt, one_shot)
  R-->>C: trigger_id を返す
  U->>C: /schedule list で確認
  U->>C: RemoteTrigger(trigger_id) 手動検証
  R-->>C: タスク実行ログ
  U->>C: TaskOutput(trigger_id) で中身確認
  Note over U,C: 1時間後、自動実行
  R->>R: 自動起動してプロンプト実行
  U->>C: TaskOutput(trigger_id) 本番結果確認
  U->>C: CronDelete(trigger_id) 片付け
作る→手動検証→待つ→結果確認→消す。5ステップが無人運用の基本形。

うまくいったかのチェックリスト

  • trigger_id が返ってきた — 作成成功。戻り値を必ずメモ。
  • CronList に出る — 登録が反映されている。
  • RemoteTrigger 手動起動で TaskOutput に期待文字列 — プロンプトが正しく動く。
  • 1時間後の自動実行後、再度 TaskOutput で結果確認 — スケジュール部分も正常。
  • CronDelete で完全削除、CronList が空に — 放置事故ゼロで完了。

ここまでできれば、あとはプロンプトの中身を差し替えるだけで「CI監視」「PRサマリ」「週次レポート」すべて同じ型で作れます。Ch04ではこの型を放置運用プロンプトに進化させるコツを学びます。

もっと詳しく: cron式の「1回きり」を安全に運用する2パターン

実装に one_shot / run_at が無い環境では、次の2パターンで擬似的に1回実行を作ります。
パターンA(推奨): 実行用プロンプトの先頭に「このタスクは1回のみ。実行後 CronDelete(trigger_id=self.id) を必ず呼ぶ」と書く。タスク自身が自分を削除する形。
パターンB: 日付と月をピンポイント指定(例 23 11 18 4 *)し、カレンダーに「翌日削除」リマインダーを入れる。人力運用の安全網。
どちらでも、作成から24時間以内にCronListで消えているか確認する習慣を組みにしてください。

Exercise 2
応用

自分の環境で1時間後1回実行トリガーを通しでやる

本章のハンズオンを実施し、以下5点を自分の言葉でメモしてください。

  1. 作ったトリガーの trigger_id と指定したcron式
  2. RemoteTrigger 手動起動時の TaskOutput の抜粋(応答テキストの先頭50字)
  3. 1時間後の自動実行で、想定通りに動いたかどうか
  4. 想定外だった点があれば何か(タイムゾーン違い・通知先ズレ・プロンプト不足など)
  5. 最後に CronDelete で削除し、CronList が空になったか確認
模範解答を見る

典型的な観察例(筆者の環境, 2026年1月):

  • 1. trigger_id = sch_7k2p... / cron=23 11 18 4 *, timezone=UTC, one_shot相当の日付指定
  • 2. TaskOutput 先頭 = 「Current UTC: 2026-04-18 11:23 / hello from scheduled task」
  • 3. 1時間後の自動実行では、UTCとJSTのズレで最初 9時間ずれて実行された。timezone 指定を明示して作り直したら想定通りに
  • 4. 想定外: 最初の作成で notify を指定せずに作ったら、TaskOutputに残る以外の痕跡がなかった。無人運用ではSlack or メール必須と再認識
  • 5. CronDelete 実行 → CronList が空になったことを確認して完了

学びの結論例: cron式の正しさだけでなく「timezone / 通知 / 削除」の3点セットで初めて実運用になる。1時間後テストは、本番前の練習として最強のコスパ。

※ この演習を通知先だけ変えて3回(Slack / メール / 通知なし)やると、Ch05以降の実践シナリオでいちいち調べ直さなくて済むようになります。
