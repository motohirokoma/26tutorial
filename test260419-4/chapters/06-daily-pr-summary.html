実践：毎朝9時のPRサマリ通知

朝会のインプットをSlackに自動生成する定期ジョブを、ゼロから実装する

所要時間: 30分 Phase: Practice 難易度: 中級
この章のゴール

毎朝9時、自分が関係するPRをopen PRから自動収集し、優先度付きでSlackに投下するCronCreateジョブをゼロから作ります。cron式の設計、self-containedプロンプト、Slack通知、テンプレ化してチーム展開、までを一本で。本章の記述は2026年1月時点。

シナリオ: 朝会前のPRサマリ自動化

朝会前の5分、自分のタブを開いてPRを一つずつ確認する作業が毎日発生しています。レビュー未対応のPR、自分のPRで返信が来ているもの、CIが落ちているものを見落とすと、朝会で答えられない。これをClaudeに任せて、9時までにSlackのDMに要約を投下させます。

毎朝、寝ている間にClaudeが起きてSlackに投げる。ユーザーは受け取るだけ。
このジョブで解決する痛み

朝の情報収集時間を5分→0分に圧縮。見落としによる「朝会で答えられない」事故をゼロに。さらに副作用として、レビュー忘れPRの可視化とチームメンバーのブロッカー早期発見にもつながります。

cron式の設計: 0 9 * * 1-5

CronCreateに渡すcron式は 0 9 * * 1-5。5フィールド構文で「平日の9:00ちょうど」を意味します。週末も含めたい人は 0 9 * * * に。タイムゾーンはCronCreateツールのデフォルトでJST(Asia/Tokyo)が明示設定可能です。

cron式5フィールドの分解
5フィールドを右から埋めると逆順に意味がつながる。
もっと詳しく: よく使うバリエーション
  • 0 9 * * 1-5 — 平日朝9時(本章の標準)
  • 0 9,13 * * 1-5 — 平日の9時と13時の2回
  • */15 9-17 * * 1-5 — 平日営業時間内に15分おき(常時監視系)
  • 0 9 1 * * — 毎月1日の朝9時(月次レポート)
  • 0 18 * * 5 — 毎週金曜18時(週次振り返り)

曜日表記は数字が無難。crontab.guru で検証するのを習慣に。

プロンプト設計: 自律動作する完全指示

毎朝9時にリモートセッションで起動するプロンプトは、誰も見ていない環境で自走する必要があります。リモートトリガーでは対話できないので、指示は単体で完結させます。

プロンプトに必須の7ブロック
異常時動作を書き忘れると「今朝サマリが来なかった」が静かなバグになる。

実プロンプト全文(コピペで動くレベル)

# ---- CronCreate 用プロンプト本文 ----
【目的】
 朝会前に、自分が関係する open PR の状況を1枚で
 把握できるようにする。レビュー抜け・自分PRの
 返信見逃しをゼロにする。

【データ源】
 以下を順に実行して対象PRを収集:
  1. gh pr list --search "review-requested:@me is:open" \
       --json number,title,url,repository,updatedAt,labels,statusCheckRollup \
       --limit 20
  2. gh pr list --search "author:@me is:open" \
       --json number,title,url,repository,updatedAt,reviewDecision,statusCheckRollup \
       --limit 20
 両者をマージし、number+repository で重複除去。

【分類】
 - Review Requested: 自分がレビュワー
 - My PR w/ Update: 自分作成 + 直近24h で変化
 - Stale: 最終更新から 72h 以上

【優先度】
 - Critical: statusCheckRollup に FAILURE を含む / labels に hotfix
 - High: 3日以上 更新なし(Stale) / 自分がブロッカー
 - Normal: 上記以外

【出力形式 (Slack 投稿本文)】
 先頭1行: ":sunrise: PR Summary $(date +%Y-%m-%d) - 合計N件"
 以降、優先度順に各PRを1行で:
   "{絵文字} [{repo}#{number}] {title} ({分類}, {残り時間/経過})"
 絵文字: Critical=:rotating_light: High=:warning: Normal=:white_check_mark:
 PRゼロのときは ":coffee: 本日対応すべきPRなし" の1行のみ。

【送信】
 上記本文を Slack Webhook に payload で POST:
  curl -X POST -H 'Content-Type: application/json' \
    --data "$(jq -Rn --arg t "$TEXT" '{text:$t}')" \
    "$SLACK_WEBHOOK_URL"
 SLACK_WEBHOOK_URL は Scheduled Task の環境変数から取得。

【異常時動作】
 - gh 認証失敗 → ":x: PR Summary: gh 認証切れ" を送る
 - gh コマンドタイムアウト(>30s) → ":x: PR Summary: 収集失敗" を送る
 - Slack 送信失敗 → stderr にログ残し、2回までリトライ

【完了条件】
 Slack への POST が 2xx を返せば完了。
 その他のケースでも必ず1通は投げて『沈黙』を起こさない。
もっと詳しく: 「沈黙を起こさない」設計思想

朝会前に通知が来ないと、ユーザーは「ジョブ動いた? 通知バグ? PRゼロ?」と迷います。ゼロ件通知を仕込むと「来ないこと=壊れている」と1:1で判断できるため、運用コストが劇的に下がります。異常時も空振り時も、必ず1通。

実装ステップ: 4段階で組み上げる

上のプロンプトをいきなり 0 9 * * 1-5 に仕込まない。手で1回動かす → 15分後に1回だけ試す → 日次に切り替えるの順で慣らします。

いきなり平日9時に仕込むと、バグが「翌営業日まで気づけない」事故になる。

Step1: ghコマンドを手で叩く

# 認証確認
gh auth status

# review-requested を自分に絞って取得
gh pr list --search "review-requested:@me is:open" \
  --json number,title,url,repository,updatedAt,statusCheckRollup \
  --limit 20

# author:@me 側も
gh pr list --search "author:@me is:open" \
  --json number,title,url,repository,updatedAt,reviewDecision,statusCheckRollup \
  --limit 20

想定通りのPRが返ることを目視確認。件数ゼロなら、テスト用に自分で1つPRを出すかorg横断検索に切り替えます。

Step2: プロンプトをファイルに保存

# ~/.claude/prompts/pr-summary.md に上記プロンプト本文を保存
mkdir -p ~/.claude/prompts
$EDITOR ~/.claude/prompts/pr-summary.md
# (上のプロンプトをコピペ)

プロンプトはリポジトリ外の個人設定として保存しておくと、仕事で使っているリポジトリに依存せず動かせます。

Step3: CronCreate でone-shot実行

# Claude Code 内で /schedule を呼び、one-shot で15分後に実行
/schedule create \
  --name "pr-summary-test" \
  --cron "$(date -v+15M '+%M %H * * *')" \
  --timezone "Asia/Tokyo" \
  --prompt-file ~/.claude/prompts/pr-summary.md \
  --env SLACK_WEBHOOK_URL=https://hooks.slack.com/services/XXX \
  --one-shot

# 一覧で登録確認
/schedule list
Secret の扱い

SLACK_WEBHOOK_URL はプロンプト本文に直書きしない。Scheduled Task の --env や Claude Code の secrets 管理経由で渡す。漏洩するとチャンネルにスパムを送られる実害が出ます。

Step4: 本番cronへの昇格

# one-shot が Slack に届いたら、日次に切り替え
/schedule create \
  --name "pr-summary-daily" \
  --cron "0 9 * * 1-5" \
  --timezone "Asia/Tokyo" \
  --prompt-file ~/.claude/prompts/pr-summary.md \
  --env SLACK_WEBHOOK_URL=$SLACK_WEBHOOK_URL \
  --notify-on-error slack

# テスト用のone-shotは削除
/schedule delete pr-summary-test

コスト警告: 放置前提だからこそ上限を引く

日次1回ならコストは軽微ですが、cron式のミスで連打したり、PRが大量でトークンが膨らむと月額が跳ねます。2つのガードを必ず入れます。

cron頻度 × トークン量のコスト目安(2026年1月時点)
日1回なら月¥150程度。5分おきに間違えると月¥4万。桁が違う。
リスク症状ガード
cron式ミスで連打 * 9 * * 1-5 と書くと毎分実行。月額4万円超 crontab.guru で毎回確認 / one-shot から段階導入
PR件数が膨大 gh pr list --limit を書き忘れ、数百件フェッチでトークン爆発 必ず --limit 20。超過時は要約粒度を粗く
セッション長時間化 AIが迷走して1回30分超え プロンプトに「処理は5分以内」と明記 / timeout 引数

テンプレ化: チーム展開で20人に配る

自分用に動いたら、メンバー名とSlackチャンネルだけ差し替えて配れる形にテンプレ化します。チーム10-20人に展開すると、朝会全体の情報粒度が揃います。

# team-pr-summary.template.md
# =====
# 変数(展開時に置換):
#   {{USER}}         — GitHub username (例: motohirokoma)
#   {{WEBHOOK_ENV}}  — Slack Webhook を入れる環境変数名
#   {{REPO_FILTER}}  — 対象リポ絞り込み (例: "org:myorg")
#   {{CRON}}         — cron式 (例: "0 9 * * 1-5")
# =====

【目的】
 {{USER}} さんの朝会用 PR サマリを Slack 通知する。

【データ源】
 gh pr list --search "review-requested:{{USER}} is:open {{REPO_FILTER}}" ...
 gh pr list --search "author:{{USER}} is:open {{REPO_FILTER}}" ...

(中略)

【送信】
 {{WEBHOOK_ENV}} で指定された URL に POST

# 展開スクリプト例(bash)
for user in alice bob carol; do
  sed -e "s/{{USER}}/$user/g" \
      -e "s/{{WEBHOOK_ENV}}/WEBHOOK_$user/g" \
      -e "s|{{REPO_FILTER}}|org:myorg|g" \
      -e "s/{{CRON}}/0 9 * * 1-5/g" \
      team-pr-summary.template.md > prompts/pr-summary-$user.md

  /schedule create \
    --name "pr-summary-$user" \
    --cron "0 9 * * 1-5" \
    --timezone "Asia/Tokyo" \
    --prompt-file prompts/pr-summary-$user.md \
    --env-from-secret "WEBHOOK_$user"
done
もっと詳しく: チーム展開で必ずやる3つ
  1. 各人のWebhookは個別DMに — 共通チャンネルに流すとノイズ化して誰も見なくなる
  2. opt-in にする — 強制すると「通知疲れ」でミュートされる。希望者だけ
  3. 2週間で効果レビュー — 「通知を読む頻度」アンケート。読まれないなら頻度か粒度を変える

notification fatigueを起こすと、重要な通知まで無視されます。粒度設計が最重要。

ハンズオン: 自分のリポで今すぐ実装

今すぐ試す: 15分後に届くone-shotジョブを作る

以下を順に実行して、Slack DMにPRサマリが1通届くところまで作ります。所要15-20分。

  1. gh auth status と gh pr list --search "involves:@me is:open" で認証とコマンドを確認
  2. Slackアプリで自分宛DMのIncoming Webhookを作り、URLを控える
  3. ~/.claude/prompts/pr-summary.md に本章のプロンプトをコピペ
  4. /schedule create --name pr-summary-test --cron "$(date -v+15M '+%M %H * * *')" --timezone Asia/Tokyo --prompt-file ~/.claude/prompts/pr-summary.md --env SLACK_WEBHOOK_URL=https://... --one-shot を実行
  5. 15分待つ。Slack DMに通知が来たらgreen。来なければ/schedule listと/schedule logs pr-summary-testで原因調査
  6. 動いたら--cron "0 9 * * 1-5"に切り替えてpr-summary-dailyとして登録
Exercise 1
中級

cron式を自分の生活リズムに合わせてカスタマイズ

「朝9時」はあくまでデフォルト。自分の朝会時間、リモート勤務の開始時間、時差など、実際の業務に合わせて最適なcron式を設計してください。

  1. 自分の朝会は何時? そこから逆算して10分前になるようcron式を書く
  2. リモートで海外メンバーがいる場合、タイムゾーン違いでどう扱うか検討
  3. 休み明け(月曜)はより早く通知したいなら、曜日別に2つ登録する方法を考える
模範解答を見る 
# パターンA: 朝会が9:30の場合 → 9:20に通知
cron: "20 9 * * 1-5"
timezone: "Asia/Tokyo"

# パターンB: 月曜は週末PRの溜まりをより早く把握したい
# → 月曜だけ8:30、火-金は9:20
# 2つ登録する:
/schedule create --name pr-summary-mon \
  --cron "30 8 * * 1" --prompt-file ...
/schedule create --name pr-summary-tue-fri \
  --cron "20 9 * * 2-5" --prompt-file ...

# パターンC: 米西海岸メンバーと協業 (UTC-8)
# 日本朝9時 = PT 17時(前日)。
# PTメンバーには現地朝9時で飛ばすため、別cronを登録
/schedule create --name pr-summary-pt \
  --cron "0 9 * * 1-5" \
  --timezone "America/Los_Angeles" \
  --prompt-file prompts/pr-summary-pt.md

ポイント: timezoneは必ず明示。サーバー側のデフォルトに依存するとDST(夏時間)で1時間ズレる事故が発生します。

Exercise 2
応用

PR以外の情報も同じ通知にまとめる

朝会のインプットはPRだけではありません。Issue、CIの夜間失敗、Sentry のエラーなども同じSlack DMに統合すると、朝会5分前に1通だけ見れば全部分かる状態を作れます。

  1. 自分の業務で、朝一に確認している情報ソースを3つリストアップ
  2. それぞれを取得するCLIコマンドまたはAPIを調べる
  3. プロンプトの【データ源】ブロックを拡張し、統合通知として1つのcron jobにまとめる
  4. 通知が長くなりすぎないか(1画面に収まるか)を意識し、優先度でフィルタリング
模範解答を見る 
# 統合プロンプトの【データ源】ブロック拡張例

【データ源】
 # (1) PR (既存)
 gh pr list --search "involves:@me is:open" --limit 20 ...

 # (2) 自分にアサインされた open Issue
 gh issue list --search "assignee:@me is:open" --limit 10 ...

 # (3) 昨夜失敗したCI run (定期ビルドなど)
 gh run list --workflow "nightly.yml" --status failure \
   --created ">$(date -v-1d +%Y-%m-%d)" --limit 5 ...

 # (4) Sentry の未対応エラー (優先度high以上)
 curl -s -H "Authorization: Bearer $SENTRY_TOKEN" \
   "https://sentry.io/api/0/projects/myorg/main/issues/?query=is:unresolved level:error"

【優先度統合】
 Critical: Sentry level=error かつ件数>5 / CI nightly 失敗
 High    : Review Requested PR で 3日以上
 Normal  : その他

【出力形式】
 セクション分け:
  :computer: PR (N件)
    ...
  :bug: Issue (N件)
    ...
  :construction: CI/Sentry
    ...
 総件数が 15 を超える場合は Critical/High のみ表示し、
 「(他 N 件省略、詳細は  )」の行を最後に足す。

ポイント: 情報源を統合すると画面は長くなります。15件ルールで自動省略し、詳細は別リンクに逃がす。読む側の認知コストを一定に保つのが重要です。

2026年1月時点の注記

本章の /schedule サブコマンド、CronCreate の引数名、タイムゾーン指定方法は2026年1月時点のものです。Claude Code はアップデートが活発なため、--env / --env-from-secret / --one-shot のフラグ名は変更される可能性があります。最新情報は公式ドキュメントで確認してください。
