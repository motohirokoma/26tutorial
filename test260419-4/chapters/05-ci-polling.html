実践：5分ごとのCIチェック

GitHub Actions のビルド結果を監視し、失敗時だけ Slack に通知する放置運用を作る

所要時間: 30分 Phase: Practice 難易度: 中級
この章のゴール

mainブランチに push した直後の CI (GitHub Actions) を5分おきに監視し、失敗したジョブだけ Slack 通知する放置運用を、自分のリポで実装します。Ch02(/loop) と Ch03(Scheduled Tasks) の知識を実戦投入し、interval vs cron の使い分け、二重通知回避、コスト見積もりまでを現場レベルで体得するのが目標です。本章の記述は2026年1月時点。

コスト警告

5分間隔ポーリングは1日あたり最大288回の Claude 実行になります。毎回フル起動だとトークンが嵩むので、(1) プロンプトを短く、(2) 失敗時のみ詳しく分析、(3) 夜間や週末は停止、の3つを本章で徹底します。これは Ch04「放置運用プロンプト」で学んだ原則の応用です。

シナリオ: なぜ5分ごとに監視したいのか

典型的な開発フロー: mainブランチに push → GitHub Actions で lint/test/build → 失敗したらできるだけ早く気づきたい。でも毎回 GitHub のWeb画面を開くのは面倒、しかも失敗通知が GitHub の Notifications に埋もれて気づかない問題がよくあります。

従来ワークフロー vs Claude 放置運用
検知時間を「数時間」から「数分」に圧縮する単純な改善。実装は30分で終わる。
  • 5分という間隔 — GitHub Actions の典型ジョブは 1〜5分で終わる。5分粒度なら「終わった直後」を拾える。
  • 失敗時のみ通知 — 成功まで通知すると狼少年化。Claude に「失敗時のみ Slack」と指示するのが肝。
  • ログ要約を同封 — Slack を見た瞬間に「何が壊れたか」が分かる。人間の復旧判断を加速。

/loop interval vs CronCreate */5 を比べる

「5分ごと」は両方で実現できます。どちらを選ぶか? 結論から言うと短時間の集中監視は /loop interval、長期の定常監視は CronCreate。両者の性質を表でまとめます。

観点/loop interval 5mCronCreate */5 * * * *
実行場所 手元の Claude Code セッション リモート Claude (RemoteTrigger)
PC を閉じて良いか NG (セッションが切れると止まる) OK (サーバー側で動く)
開始のしやすさ `/loop 5m ...` 1行で即開始 CronCreate で登録が必要
停止のしやすさ セッション Ctrl+C / 次ターンで停止 CronDelete で明示削除
コスト粒度 手元セッションなので把握しやすい 登録しっぱなしで忘れがち(危険)
キャッシュ効率 5分以内ならウォーム状態維持しやすい 毎回コールド(原則)
向いている用途 1〜数時間の集中監視(今日のデプロイ直後など) 毎日の定常監視(営業日中ずっとなど)
使い分けフロー
graph TD
  Q["5分ごとに CI を見たい"] --> D1{"いつまで回す?"}
  D1 -->|"数時間のみ"| L["/loop 5m を使う"]
  D1 -->|"営業日ずっと"| D2{"PC を閉じる可能性は?"}
  D2 -->|"ある"| C["CronCreate */5 * * * 1-5"]
  D2 -->|"ない"| L
  L --> E["即時開始。手元で回す"]
  C --> E2["リモートで定時実行"]
「短期・集中」=/loop、「長期・営業日ずっと」=Cron が基本の切り分け。
もっと詳しく: 厳密に5分ぴったりではないことへの注意

どちらのモードも「ぴったり5分」は保証されないという共通点があります。/loop interval は Claude の応答完了から5分測るため、応答が1分かかったら実質次は6分後。CronCreate も内部スケジューラの揺らぎで数秒〜十数秒ずれることがあります。CI 監視用途では「秒単位の厳密さ」は不要なので、これは実用上問題になりません。逆に「外形監視 SLA 計測」のように厳密性が要るなら、Claude ではなく専用の監視ツール(Datadog/Mackerel 等)の方が適しています。

前提: gh CLI と Slack Webhook を準備

Claude に渡す「道具」として以下2つがあれば十分です。ローカルにあれば Claude Code はBash tool経由で自力で叩きに行けます。

1. gh CLI 最新版をインストール

# macOS
brew install gh
gh --version   # 2.62 以降推奨 (2026年1月時点)

# Linux (Debian/Ubuntu)
sudo apt install gh

# 認証(一度だけ)
gh auth login
# → GitHub.com / HTTPS / Login with a web browser を選択

# 動作確認: 自分のリポの直近 run を取る
gh run list --repo OWNER/REPO --limit 3

2. Slack Incoming Webhook を作る

  1. Slack App を作成: https://api.slack.com/apps → Create New App → From scratch
  2. Incoming Webhooks をオン → Add New Webhook to Workspace → 送信先 channel を選ぶ
  3. 発行された URL (例: https://hooks.slack.com/services/T.../B.../xxx) を環境変数に
# ~/.zshrc などに追記
export CI_ALERT_SLACK_URL="https://hooks.slack.com/services/T.../B.../xxx"

# 動作テスト
curl -X POST -H 'Content-Type: application/json' \
  --data '{"text":"Claude CI watcher test"}' \
  "$CI_ALERT_SLACK_URL"
セキュリティ注意

Webhook URL は実質的な投稿権限そのもの。git コミットしない、Slack に貼らない、ログに出さない。Claude に渡すときも URL を直接プロンプトに書くより、環境変数名で参照させる(例: $CI_ALERT_SLACK_URL)方が安全です。

プロンプト設計: self-contained に全部書く

Ch04 で学んだ self-contained 原則を適用します。放置運用では「起動した Claude が履歴ゼロで読んでも1人で完遂できる」ようプロンプトを書きます。

監視プロンプトの7ブロック構造
「放置で動く」プロンプトは、成功時・失敗時・エラー時の3パスをすべて明記する。

完成プロンプト(コピペ可)

あなたは CI 監視 watcher です。以下を1回実行してください。
状態保持には ~/.ci-watch-state.json を使います(なければ作る)。

【目的】
 OWNER/REPO の main ブランチの最新 GitHub Actions run を確認し、
 失敗していれば Slack に通知する。成功なら何もしない。

【対象】
 - リポ: OWNER/REPO (ここは自分のリポ名に置換)
 - ブランチ: main

【手順】
 1. bash: gh run list --repo OWNER/REPO --branch main \
      --limit 1 --json databaseId,status,conclusion,headSha,displayTitle,url
 2. status == "completed" でない → 何もせず終了(まだ走行中)
 3. conclusion が "success" → stdout に 1行だけ
    「OK: run_id=<id>」と書いて終了。
 4. conclusion が "failure" / "cancelled" / "timed_out" のとき:
    a. ~/.ci-watch-state.json を読み、last_notified_run_id と比較
    b. 同じ id なら既通知なので何もせず終了
    c. 新しい id なら:
       - gh run view <id> --repo OWNER/REPO --log-failed | tail -60
       - 上記 log tail を 200字以内で要約
       - 以下 JSON を $CI_ALERT_SLACK_URL に POST
         {
           "text": ":rotating_light: CI FAILED: <displayTitle>\n<url>\n\n<log要約>"
         }
       - ~/.ci-watch-state.json を { "last_notified_run_id": <id> } で更新

【やらないこと】
 - 成功時に通知しない
 - 同じ run_id を2回通知しない
 - gh / curl が失敗してもループは止めない(stderr出力のみ)

【出力】
 標準出力に 1行だけ結果を書く。例:
 「OK: run_id=12345」または
 「NOTIFIED: run_id=12346 (失敗通知送信)」または
 「SKIP: run_id=12346 (既通知)」

二重通知回避: 状態ファイルで冪等化

単純に「失敗ならSlack」と書くと、同じ失敗ビルドが5分ごとに通知されてSlack が荒れます。これを防ぐのが状態ファイル方式です。

状態ファイルでの重複抑止(シーケンス)
sequenceDiagram
  participant C as Claude(5分おき)
  participant GH as GitHub API(gh)
  participant S as ~/.ci-watch-state.json
  participant Sl as Slack
  C->>GH: 最新 run 取得
  GH-->>C: run_id=123 (failure)
  C->>S: last_notified_run_id を読む
  S-->>C: 123 (既通知)
  C->>C: 何もしない(SKIP)
  Note over C: 5 分後
  C->>GH: 最新 run 取得
  GH-->>C: run_id=124 (failure)
  C->>S: last_notified_run_id を読む
  S-->>C: 123
  C->>Sl: 通知を送る
  C->>S: 124 に更新
鍵は「通知送信直後に状態を書き換える」。同じ run_id は二度と通知されない。

状態ファイルのスキーマ

// ~/.ci-watch-state.json
{
  "last_notified_run_id": 12345,
  "last_notified_at": "2026-01-18T10:05:00Z",
  "last_checked_at": "2026-01-18T10:10:12Z",
  "consecutive_errors": 0
}
  • last_notified_run_id — 最後に通知した run の databaseId。比較の主キー。
  • last_checked_at — 単なる稼働確認用。異常検知に使える。
  • consecutive_errors — gh/curl のエラー回数。3回でメタアラート送信。
もっと詳しく: 「再実行された failure」も新 id なら通知される

GitHub Actions で同じコミットに対して「Re-run failed jobs」を押すと新しい run_id が発行されます。これは新規 id なので watcher は再通知します。人間としては「また失敗したか」と気づきたい場面なので、この挙動は妥当です。「同じコミットは1回だけ」にしたいなら headSha で重複排除する設計に変えれば良いです(状態ファイルに last_notified_sha を持たせる)。

実装A: /loop interval で回す

まずは手軽な方から。手元 Claude Code セッションで /loop を使います。

手順

  1. 上の「完成プロンプト」を ~/ci-watcher-prompt.md に保存(自分のリポ名に置換)
  2. Claude Code セッションで以下を実行
/loop 5m @~/ci-watcher-prompt.md

これで5分おきに同じプロンプトが走ります。停止は Claude のセッションを Ctrl+C するか、次ターンで「ループ止めて」と伝えるだけ。

/loop interval 実行タイムライン(60分)
5分おきに12回発火。失敗発生時刻(例: 0:23)から最大5分以内に通知が飛ぶ。
今すぐ試す: 最初の1時間を /loop で監視

自分の直近 push 後、1時間だけこの watcher を走らせてみましょう。成功ビルドだけなら Slack に何も来ない(静かな watcher が正解)。わざと失敗コミット(例: process.exit(1) を仕込む)を push すると、5分以内に Slack へ通知が来るはずです。

実装B: CronCreate で長期運用

PC を閉じる or 営業日ずっと回したいなら CronCreate に切り替えます。Ch03 で学んだ手順の実戦適用です。

# /schedule スキル経由(Claude Code セッション内で)
/schedule create \
  --cron "*/5 9-18 * * 1-5" \
  --prompt-file ~/ci-watcher-prompt.md \
  --name "ci-watch-main"

# cron 式の意味:
#  */5    -> 5分おき
#  9-18   -> 9時〜18時だけ
#  * *    -> 毎日・毎月
#  1-5    -> 月〜金のみ
# = 営業日9-18時のみ動く設定 (夜間と土日は静か)

cron 式の設計ポイント

  • 夜間停止 — 「9-18」で営業時間外はコストゼロ。誰も気づかない深夜に実行しても意味薄い。
  • 土日停止 — 「1-5」で月〜金のみ。週末の誤通知抑止。
  • 忘れ防止 — 登録したら /schedule list でカレンダーに棚卸し。不要なら即 /schedule delete。

もっと詳しく: 失敗時のみ詳細分析でコストを抑える

ポーリング毎に log tail 60行 を全部要約するのは無駄。成功時(全体の95%以上)は「status 取得 → 成功確認 → 即終了」で最小トークンに抑えられます。失敗時だけ log を取りに行き要約する設計にすれば、平均トークンは 1/5 程度まで下がります。プロンプトに「成功時は log を読まない」と明示するのがコツです。

失敗検知ロジック: conclusion と status の違い

GitHub Actions の gh run list --json は2つのフィールドを返します。混同すると誤検知の温床です。

フィールド値意味
statusqueuedまだ走ってない(待機中)
statusin_progress走行中
statuscompleted終わった(成否は conclusion で判定)
conclusionsuccess成功
conclusionfailure失敗 ← 通知対象
conclusioncancelledキャンセルされた ← 通知対象
conclusiontimed_outタイムアウト ← 通知対象
conclusionskippedスキップ(通知不要)

つまり正しい判定は「status == completed かつ conclusion in {failure, cancelled, timed_out}」。走行中を失敗と誤判定しないため、status の確認を忘れずに。

# jq で綺麗に判定する場合(bash 単体テスト用)
gh run list --repo OWNER/REPO --branch main --limit 1 \
  --json databaseId,status,conclusion \
  | jq -r '.[0] | select(.status=="completed" and (.conclusion=="failure" or .conclusion=="cancelled" or .conclusion=="timed_out")) | .databaseId'
# → 失敗ならIDを出力、それ以外なら何も出ない
Exercise 1
中級

自分のリポで watcher を起動する

以下の手順で実際に動作させてください。

  1. gh CLI と Slack Webhook の準備(本章 Section 3)
  2. 完成プロンプト(Section 4)の OWNER/REPO を自分のリポ名に置換し、~/ci-watcher-prompt.md に保存
  3. わざと失敗するコミット(例: test コードに expect(1).toBe(2))を push
  4. /loop 5m @~/ci-watcher-prompt.md で起動
  5. 5分以内に Slack 通知が来ることを確認。さらに次のループでは重複通知が来ないことを確認
模範解答・確認ポイント

期待される挙動:

  • 1回目: 失敗検知 → Slack 通知 → state ファイル更新 → stdout に NOTIFIED: run_id=XXX
  • 2回目: 同じ run_id → SKIP(通知しない) → stdout に SKIP: run_id=XXX
  • 3回目以降: 新しい push が無ければずっと SKIP
  • 成功 push 後: 次のループから OK: run_id=YYY

うまくいかない場合のチェックリスト:

# A. gh コマンドが通るか
gh run list --repo OWNER/REPO --limit 1
# → 何も出ない場合は repo 名 or 認証を疑う

# B. Slack URL が有効か
curl -X POST -H 'Content-Type: application/json' \
  --data '{"text":"test"}' "$CI_ALERT_SLACK_URL"
# → 200 "ok" 以外なら URL を再確認

# C. 状態ファイルが書けているか
cat ~/.ci-watch-state.json
# → last_notified_run_id が最新失敗の ID になっているか

ここまで動けば「5分ごとの CI 監視」は完成。あとは運用期間を決めて /loop を止めるか、営業日運用なら CronCreate へ移行。

Exercise 2
応用

CronCreate 版に移行し、コストを営業時間限定で最適化

Exercise 1 の /loop 版を CronCreate に移植し、以下の最適化を加えてください。

  1. 営業日 (月-金) の 9-18 時のみ実行する cron 式を設定
  2. 状態ファイルに consecutive_errors を追加し、3 回連続で gh が落ちたら「watcher 自身が壊れた」を Slack に通知
  3. CronList で登録内容を確認し、不要になった時の削除コマンドをメモに残す
模範解答 
# 1. 登録
/schedule create \
  --cron "*/5 9-18 * * 1-5" \
  --prompt-file ~/ci-watcher-prompt.md \
  --name "ci-watch-main-biz"

# 2. プロンプトに以下セクションを追記
【エラー時の挙動】
 gh 実行が非ゼロ終了した場合:
  - ~/.ci-watch-state.json の consecutive_errors を +1
  - 値が 3 以上になった瞬間、Slack に
    「:warning: CI watcher が3回連続で gh コマンド失敗」
    を1度だけ送信(以後 consecutive_errors が 0 に戻るまで抑止)
 gh が成功した場合: consecutive_errors を 0 にリセット

# 3. 棚卸し
/schedule list
# → 登録一覧を確認

# 削除(プロジェクト終了時)
/schedule delete ci-watch-main-biz

このレベルまで作ると、watcher が死んでも気づける(メタアラート)、コストが読める(営業時間限定)、停止しやすい(名前管理)という3拍子揃った放置運用になります。ここが Ch05 のゴール地点です。
