/loop スキル

dynamic モード vs interval モード、自走パターン

所要時間: 30分 Phase: Basics 難易度: 初級
この章のゴール

/loop スキルの2つの動作モード(dynamic / interval)の違いを、内部で使われる ScheduleWakeup の仕組みとキャッシュの温度(ウォーム/コールド)とセットで理解します。最後に3回だけ回る簡単な /loop をハンズオンで動かして、停止の仕方まで押さえる——ここまでが30分のスコープです。Ch03 (CronCreate) に進む前の土台となる章です。

バージョン注記

本章の記述は 2026年1月時点 の Claude Code における /loop スキルの挙動に基づきます。ScheduleWakeup ツールのキャッシュ閾値(270s / 1200s)やスキルの起動フローはハーネス側で継続的に調整されているため、動作が微妙に違う場合は 公式ドキュメント と ~/.claude/skills/loop/SKILL.md を参照してください。

コスト警告

/loop は 回るたびに課金される。5分間隔で1日回せば 288回、1回 $0.05 相当でも日額 $14。**必ず停止条件**(終了フラグ/最大回数/時間帯)をプロンプトに組み込むこと。「とりあえず回して放置」は絶対にやらない。

/loop とは何か — 自分で自分を起こすスキル

/loop は、Claude Code に組み込まれているスキルで、1回動いたあとで「自分を次に起こす予約」を入れ、間隔を置いて再起動する——これを繰り返すための仕組みです。内部では ScheduleWakeup というツールを子スキルから呼び出しています。

/loop の基本フロー(1ターンごとに「次」を予約)
1ターンは「実行 → 次を予約 → 終了」のワンセット。Claudeは毎回まっさらに起きて、プロンプトとリポジトリ状態から再スタートする。

ポイントは 「Claudeは回り続けているわけではない」 という点です。ターン は終わるたび本当に消え、次のターンは ScheduleWakeup の目覚ましで呼び起こされて最初から始まります。状態は会話ログとリポジトリのファイル、それだけです。

2つのモード: dynamic と interval

/loop は同じスキルでありながら、引数に間隔を渡すかどうかで挙動が丸ごと変わります。これを覚えるだけで7割です。

観点dynamic モード(引数なし)interval モード(引数あり)
起動例/loop /my-task/loop 5m /my-task
次の間隔を決めるのはClaude 自身(毎ターン判断)ユーザ(固定)
向いてる用途進捗が不定(長文執筆/探索/試行錯誤)定期監視(CIチェック/PR確認)
キャッシュ最適化Claudeがウォーム帯を狙える固定のためコールドになりやすい
停止の仕方ScheduleWakeupを呼ばずに終わる同じく。あるいは Esc / /stop
予測しやすさ低(Claude次第)高(cron的)

  • 探索的リサーチで深さを調整しつつ巡回

  • 15分ごとに Slack 受信箱を要約

dynamic モードの起動例 
/loop README の下書きを 1セクションずつ書き進めてください。
各ターンで次の1セクションだけ書き、
section-done.md に完了したセクション番号を append。
全8セクション書き終わったら loop を終了(次のウェイクアップを予約しない)。
interval モードの起動例(5分固定) 
/loop 5m gh run list --limit 3 --json conclusion,status,name |
jq を使って最新ビルドの結果を判定。
FAILED があれば Slack Webhook に投稿、なければ何もせず終わる。
重複通知を避けるため .loop-state/last-notified.json を参照。
もっと詳しく: dynamic での「次は何秒後」をClaudeはどう決めているか

dynamic モードでは、スキル内のシステムプロンプトに「次のウェイクアップまでの秒数を合理的に選べ」という指示が入っています。Claudeは現在のタスク進捗・残量・前回の実行時間・キャッシュの温度を考慮し、ScheduleWakeupに渡す delay_seconds を決めます。「まだ書くことが多いので60秒後」「待ち時間が欲しいので180秒後」のように調整します。interval モードではこの判断をスキップし、ユーザ指定の秒数をそのまま渡します。

キャッシュ TTL 5分の意味 — ウォーム / コールドの境界

/loop の「間隔」を決めるとき、Claude API の プロンプトキャッシュの TTL(有効期限) が挙動と料金に直結します。2026年1月時点の Claude Code では、実効的に以下の3ゾーンで考えるとよい設計になります。

ウェイクアップ間隔とキャッシュ温度の関係
横軸=ウェイクアップ間隔(秒)、縦軸=相対コスト/レイテンシ。270秒以下はウォーム(ほぼキャッシュヒット)、1200秒(20分)を超えるとコールド(全再読込)。
ゾーン間隔状態体感向き
ウォーム< 270秒キャッシュ生存高速・安い短間隔反復
中間270s 〜 1200s部分ヒット徐々に劣化中間隔タスク
コールド> 1200秒キャッシュ失効毎回フル読込長間隔(CronCreateを検討)

Tips

厳密な閾値はハーネス実装側の最適化で多少動きます(2026年1月時点、Anthropicはプロンプトキャッシュ TTL を 5分 / 1時間の2段構成で提供)。本章の 270s / 1200s は「短間隔ならウォーム、長間隔ならコールド」と覚える目安として使ってください。迷ったら5分以内=/loop、20分超=Scheduleで概ね外れません。

もっと詳しく: なぜ 270秒 や 1200秒 なのか

Anthropicのプロンプトキャッシュは標準 TTL が 5分(= 300秒)。実際の読み取りでは直前のターンの読み込み時刻からタイミング次第で数十秒早く失効するため、安全マージンとして 270秒(4分半)を「ウォーム帯の上限」と置くのが現場感覚です。1200秒(20分)は「延長 TTL(1時間)を使っていても3回目以降は確実にコールドになり、/loop のうまみが消える」ラインで、運用上はここで CronCreate に切り替えた方がトータルコストが安くなります。

典型パターン 3種(短/中/長間隔)

実戦で使われる /loop の型は、ほぼ3つに分類できます。自分のタスクをこの型に当てはめて、間隔・停止条件・通知の有無だけ変えるのが最短ルートです。

  • 短間隔(60-180秒): ビルド監視型 — CI/デプロイの進捗を分単位で追う。/loop 2m、interval 固定、SlackへFAILのみ通知。ウォーム帯で高速・安価。
  • 中間隔(5-15分): 自己ペース進行型 — 長文執筆や段階的リファクタ。/loop の dynamic モードで、Claudeが進捗を見て次の間隔を決める。todo.md に状態を書き出して再開。
  • 長間隔(30分-1時間): アイドル巡回型 — PRサマリ、ニュース要約、営業日初回のヘルスチェック。コールド帯なので CronCreate の方が安いが、状態を跨ぎたい時は /loop も可。
3パターンのコスト/価値バランス
短間隔はキャッシュが効いて実コストが想像より低い。長間隔は逆にウォームが効かず、回数のわりに割高。

短間隔パターンの具体プロンプト

/loop 2m
現在の gh run list --json conclusion,status,name --limit 3 を取得。
最新の run.status が "in_progress" なら「ビルド中...」を stdout に1行出して終了。
"completed" かつ "failure" なら、.loop-state/notified.json を参照し
同じrun_idがまだ通知されていなければ Slack Webhook に投稿。
run_id を notified.json に追記。
completed かつ "success" の場合も同様に1回だけ通知して、次のウェイクアップは予約しない
(= ループ終了)。

中間隔(dynamic)の具体プロンプト

/loop todo.md の未完了タスクを上から1つ取り、実行してチェックを入れる。
タスクが大きいと感じたらサブタスクに分割して todo.md に戻す。
次のウェイクアップまでの時間は、いま実行したタスクの重さから自分で決める
(軽い編集なら120秒、重いリサーチなら10分)。
todo.md が全部完了したらウェイクアップを予約せず終了。

/loop の止め方 — 3通り

/loop は「自分で自分を起こし続ける」ので、明示的に止める手段を用意していないと放置コストが爆発 します。止め方は次の3つ、優先順位の高いものから書きます。

1
プロンプト側の終了条件
「全タスク完了 / エラー連続 / 時刻超過 で ScheduleWakeup を呼ばない」を書く
2
手動介入
実行中のターンで Esc、または「/loop を停止」と指示
3
強制クリア
Claude Code を終了、/clear、別セッションで継続せず放棄
停止条件の書き分け判定
graph LR
  A(["ターン開始"]) --> B{"状態を読む"}
  B --> C{"終了条件
に合致?"}
  C -- "Yes" --> End["ScheduleWakeupを呼ばない
→ loop終了"]
  C -- "No" --> D["タスク実行"]
  D --> E["ScheduleWakeup(N秒)"]
  E --> F(["ターン終了"])
停止は「呼ばない」という選択を Claude にさせること。プロンプトに条件を明記することで Claude がその分岐を取る。

良い /loop プロンプトには、必ず以下3つの終了条件のどれかが書かれています:

完了条件
todo.md が空 / 全チェック済み / 目標ファイル存在
失敗打ち切り
同じエラーが連続3回 / 特定ファイルに "abort" と書かれた
時間打ち切り
現在時刻が指定時刻を超えた / N ターン消化した
コスト警告(再掲)

停止条件のない /loop は 寝ている間に100回以上回って青ざめる のが定番事故。プロンプト末尾に「XX時になったら終了」「YY回実行したら終了」を機械的に必ず入れる。心配なら .loop-state/turn-count.json にカウンタを置いて参照させるのが確実。

もっと詳しく: 手動停止のつもりが「次ターンが既に予約されていた」ケース

Esc を押した瞬間、すでに ScheduleWakeup が成功して予約が入っていると、次のターンは予定通り起きてきます。確実に止めたいときは、プロンプト側に「停止フラグファイル .loop-state/stop.flag があれば次を予約しない」を仕込んでおき、手動停止したい時は touch .loop-state/stop.flag するのが安全。Ch04 (放置運用プロンプト設計) でこのパターンを詳しくやります。

ハンズオン: 3回だけ回る /loop

理屈は以上です。手を動かすのが一番効きます。3回だけ回って自動停止する、最小の /loop を動かしましょう。所要5分。

ハンズオン手順
  1. 任意の作業ディレクトリで Claude Code を開く
  2. 下のプロンプトをコピペして送信
  3. 合計3回のターンでログが書かれ、自動停止するのを観察
  4. loop-log.txt に3行出ていることを確認
/loop 30s
以下を毎ターン実行してください。

1. ./loop-log.txt が存在しなければ新規作成
2. 現在時刻を読み、"turn X at HH:MM:SS" という形式で1行 append
   (Xはこれまでのターン数+1)
3. ./loop-log.txt の行数を数える
4. 行数が 3 以上なら ScheduleWakeup を呼ばず終了
5. 未満なら次のウェイクアップを30秒後に予約して終了

このプロンプトで起きるのは次の流れです:

T+0s
1回目のターン
loop-log.txt を作成、"turn 1 at ..." を append、行数=1なので30秒後を予約して終了。
T+30s
2回目のターン
"turn 2 at ..." を append、行数=2なので再度30秒後を予約。ここでもキャッシュがウォーム帯(30s < 270s)なので高速。
T+60s
3回目のターン(最終)
"turn 3 at ..." を append、行数=3 ≥ 3 なので ScheduleWakeup を呼ばずに終了。loop はここで自然死。

終了後、cat loop-log.txt で以下のような3行が見えれば成功です。

turn 1 at 14:23:05
turn 2 at 14:23:36
turn 3 at 14:24:07
次へ

この「3回で止まる」経験があると、Ch03(CronCreate)で「回らないけど定刻に起きる」タスクが対比で理解しやすくなります。/loop はセルフペース、CronCreate は時刻固定——同じ放置運用でも能動 vs 受動の対比を次章で扱います。

Exercise 1
入門

ハンズオンを「5回で停止」に改造する

本文ハンズオンの /loop プロンプトを改造し、5回ターンで停止 するよう変更してください。さらに、途中の各ターンでログ行に「(残り N 回)」を追記する改造も加えてみましょう。

  1. 終了条件を「3回」から「5回」に変える
  2. ログ行の形式を turn X at HH:MM:SS (残り N回) に拡張
  3. 実行して、loop-log.txt が5行になり最終行に「(残り 0回)」が入ることを確認
模範解答を見る

プロンプトは下記のように変えればOK。

/loop 30s
以下を毎ターン実行してください。

1. ./loop-log.txt が存在しなければ新規作成
2. これまでの行数 L を数え、現在のターン番号を X = L + 1、残りを N = 5 - X とする
3. 現在時刻を読み、"turn X at HH:MM:SS (残り N回)" を append
4. X が 5 以上なら ScheduleWakeup を呼ばずに終了
5. 未満なら 30 秒後を予約して終了

ポイント: 終了判定はプロンプト内の条件分岐で Claude に「呼ぶ/呼ばない」を選ばせる。ここを自分で書けるようになると、任意の停止条件(時刻・ファイル存在・連続失敗)が書けるようになります。

Exercise 2
応用

dynamic モードで「todo.md を1項目ずつ片付ける」/loop を書く

interval ではなく dynamic モードを使い、手元の todo.md にある未完了タスクを1ターン1項目ずつ処理する /loop を設計してください。中身は任意のタスクでかまいません(例: "READMEに章見出しを追加"、"CHANGELOGに今日の日付を書く" など軽いもの3つ)。

  1. todo.md を5行ほど用意(チェックボックス形式 - [ ])
  2. dynamic /loop のプロンプトを書く(間隔は Claude に決めさせる)
  3. 停止条件(未完了ゼロ / 連続エラー / 時刻超過のどれか)を明記
  4. 実行してログを確認。全項目にチェックが入ったら /loop が自然終了すること
模範解答を見る

次のようなプロンプトで動きます。

/loop todo.md を先頭から読み、最初の "- [ ]" の未完了タスクを1つだけ取り出して実行してください。

- タスクが完了したら、その行を "- [x]" に書き換える
- タスクが大きいと感じたらサブタスクに分解して todo.md に挿入して良い
- 次のウェイクアップまでの秒数は自分で決めてください
  (軽い編集=120秒、重いリサーチ=600秒、など)
- 以下のいずれかに合致したら ScheduleWakeup を呼ばず終了:
  (a) todo.md に "- [ ]" が1つもない(全項目完了)
  (b) 同じタスクで連続3回失敗している(状態は .loop-state/fail.json を参照)
  (c) 現在時刻が 18:00 を超えた

観察ポイント:(1) dynamic モードでは ScheduleWakeup の delay_seconds がターンごとに変わる(ログで確認できる)。(2) 状態ファイル .loop-state/fail.json を使うことで、ターンをまたいだ失敗カウントができる。(3) 時刻打ち切り(c)が「寝てる間暴走」を防ぐ最重要の安全弁。この3つが揃って初めて「放置運用可能な /loop」。
