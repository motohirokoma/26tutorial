実践：リサーチを5並列で回す

同じ問いを多面で検証、結果を集約する実例

所要時間: 30分 Phase: Practice 難易度: 中級
この章のゴール

「新技術 X をチームに導入すべきか」という曖昧で難しい問いを、5側面(技術 / コスト / 学習曲線 / 代替案 / リスク)に分解し、5つのSubagentを1メッセージで並列起動して調査する。結果をメインAgentで統合し、経営会議に出せる1枚のサマリまで仕上げる一連の流れを、あなたの業務テーマで再現できるようになる。本章の記述は2026年1月時点の情報です。

シナリオ：新技術Xをチーム導入すべきか

想定する現場はこうです。あなたはあるチームのリードで、「Bun を本番の Node.js 代替として導入すべきか」を問われています。判断に必要な情報は広く、深く、そして多面的。ひとりで全部調べたら半日潰れますし、ChatGPT に一発で聞くと「多分イケます」みたいな薄い結論しか返ってきません。

そこでSubagentを5つ並列で起動します。各Subagentには固有の観点を与え、同じテーマを別角度で深掘りさせる。あなたはメインAgentに戻ってきた5つの独立レポートを読み、最終的な意思決定材料に統合する──これが本章の全体像です。

5並列リサーチの全体像
1つの問いを5観点に分解 → 5並列で深掘り → メインAgentが統合して1枚サマリ。

ポイントは 5つのSubagentが互いの結果を参照しない こと。依存がないから並列化できる。この「独立性」の判定は Ch04 で詳しくやりました。

5観点への分解設計

観点数は「MECE」で決めるのが鉄則ですが、本番では 3〜7個 が扱いやすいゾーン。2個だと粗すぎて意思決定できない、8個以上だと統合段階でメインAgentのコンテキストが膨らみすぎます。導入判断なら「技術 / コスト / 学習 / 代替 / リスク」の5点セットがほぼテンプレとして機能します。

観点分解ロジックツリー
graph TD
  Q["Bun を導入すべきか?"] --> F["Feasibility (実現可能性)"]
  Q --> B["Business (事業性)"]
  Q --> O["Org (組織影響)"]
  F --> A1["A1: 技術"]
  F --> A4["A4: 代替案"]
  B --> A2["A2: コスト"]
  B --> A5["A5: リスク"]
  O --> A3["A3: 学習曲線"]
上位3カテゴリに束ねると5観点の抜け漏れも検証しやすい。
観点調べる中核成果物の形式所要目安
A1 技術互換性/性能/安定性/エコシステム箇条書き+主要ベンチ数値3-5min
A2 コスト移行工数/運用費/ライセンス/CI変更表(項目×金額/人日)3-5min
A3 学習曲線教育時間/資料/コミュニティ/採用影響ロードマップ風リスト2-4min
A4 代替案Deno/Node最新/現状維持 の優劣比較比較表(軸×選択肢)3-5min
A5 リスクベンダー依存/脆弱性履歴/撤退コストリスク×影響度表3-5min

各観点で 成果物の形式を指定 することが後段の統合を楽にする最大のコツ。表なら表、ロードマップならロードマップ、と self-contained に明記します。これはCh05で学んだ原則の実戦適用です。

もっと詳しく: 観点が5つを超えそうな時どうするか

「セキュリティ」「法務」「インフラ」なども気になる──と欲が出たら、以下のどれかで調整します:

  1. A5 リスクに吸収:セキュリティと法務はリスクカテゴリの下位項目として1つのSubagentに扱わせる。prompt内で「セキュリティ観点とライセンス法務観点も含めること」と明示
  2. 第2ラウンド:まず5観点で回し、統合後に「深掘りが必要」となった1〜2テーマだけ第2ラウンドで起動
  3. 7観点並列に拡大:どうしても独立に見たいなら7並列。コストは上がるが可能。ただし8以上はメインの統合負荷が重くなる

MKの実務では 「まず5観点→結果を見て必要なら2個追加」 の二段構えが最も失敗が少ない運用です。

5つの Agent call プロンプト全文

ここからが本章のメイン。実行可能な5つのプロンプトを全部掲載 します。あなたのClaude Codeに貼り付けると、そのまま5並列リサーチが動きます。Subagentタイプは全て general-purpose(または Explore)を使用。

実行方法

Claude Code に「以下の5つを1メッセージ内で並列に Agent ツールで起動してほしい」と伝え、下記5ブロックを続けて貼り付けます。1メッセージ内に5つの Agent call を書くことで、メインAgentが内部的に並列起動します。

A1: 技術観点

subagent_type: general-purpose
description: Bun 技術調査 (互換性/性能/安定性)
prompt: |
  あなたは Node.js エコシステムに詳しい技術リサーチャーです。
  「Bun を本番の Node.js 代替として採用可能か」を技術観点から評価してください。

  【調査項目】
  1. Node.js API 互換性の現状 (2026年1月時点)
  2. 代表的ベンチマーク(HTTP/JSON/ファイルI/O)での性能差
  3. メジャーバージョン安定性と過去のbreaking change履歴
  4. パッケージエコシステム(主要ライブラリの動作状況)
  5. TypeScript/Test runner組み込み機能の成熟度

  【成果物形式】
  - 各項目を3-5行の箇条書きで
  - ベンチは具体数値(例: req/s, ms)を1つ以上引用
  - 最後に「技術観点のみでの採用推奨度: 5段階」
  - 出典URLを各項目に1本以上

  【制約】
  - 推測で断定しない。不確実な箇所は「不確実」と明記
  - 報告は800字以内
  - 完了報告の末尾に「検証済み: (Yes/No)」を明記

A2: コスト観点

subagent_type: general-purpose
description: Bun コスト試算 (移行工数/運用費)
prompt: |
  あなたは SaaS のエンジニアリングマネージャーです。
  「Bun を既存 Node.js サービスへ導入する総コスト」を試算してください。

  【前提(あなたが仮置き)】
  - 対象: Node.js 20 + Express + TypeScript のAPI 1本
  - 規模: 5万行, 月額運用費 20万円のクラウド環境
  - チーム: 4人のバックエンドエンジニア

  【調査項目】
  1. 初期移行工数(人日)と内訳(CI/CD, パッケージ差替, テスト修正)
  2. ランタイム差替によるクラウド運用費の変化試算
  3. ライセンス/商用サポート費用
  4. 既存モニタリング/APMツール対応コスト
  5. 隠れコスト(本番障害時の切り戻し準備等)

  【成果物形式】
  - 表形式: | 項目 | 単位 | 初期 | 年間 | 備考 |
  - 総額レンジ(最低/中央/最高)を円建てで
  - 前提の仮置きは冒頭に明記

  【制約】
  - 根拠のない過度な安値/高値は避ける
  - 報告は900字以内
  - 完了報告の末尾に「成果物: 表を含む」を明記

A3: 学習曲線観点

subagent_type: general-purpose
description: Bun 学習曲線調査 (教育/資料/コミュニティ)
prompt: |
  あなたは社内の Developer Enablement 担当です。
  「4人のNode.js経験者チームが Bun を使いこなすまでの学習曲線」を評価してください。

  【調査項目】
  1. 初心者が独学で追いつける公式資料の質と量
  2. 日本語での解説記事/書籍/動画の現状
  3. コミュニティ規模(Discord/GitHub stars/Stack Overflow質問数)
  4. 採用市場への影響(Bun経験者の採用難易度)
  5. チュートリアル完走から業務適用までの推定時間

  【成果物形式】
  - ロードマップ風の箇条書き
    - Week 1: 何ができるようになる
    - Week 2-3: ...
    - Month 2: 業務適用可能
  - 「習得難易度: 5段階」で評価

  【制約】
  - 個人差は「平均的経験者」を仮定して1つに絞る
  - 報告は700字以内
  - 完了報告の末尾に「参照URL数: N」を明記

A4: 代替案観点

subagent_type: general-purpose
description: Bun 代替案比較 (Deno/Node最新/現状維持)
prompt: |
  あなたは技術選定のコンサルタントです。
  「Bun を採用しない場合の現実的な代替案」を3つ挙げて、Bun と横並び比較してください。

  【候補(最低この3つを含める)】
  - Deno 2.x
  - Node.js 最新LTS(22.x系)をそのまま継続
  - 現状維持(Node.js 20 そのまま)

  【比較軸】
  1. 性能
  2. 安定性
  3. 互換性(既存コード)
  4. 将来性(5年後に主流か)
  5. 移行リスク
  6. コスト

  【成果物形式】
  - 表形式: | 軸 | Bun | Deno 2 | Node 22 | 現状維持 |
  - 各セルは「◎/○/△/✕」+ 1行補足
  - 最下行に「総合スコア」を加点式で
  - 末尾に「あなたの推奨: (1つ選ぶ)」と理由2-3行

  【制約】
  - 軸ごとの評価理由を簡潔に
  - 報告は900字以内
  - 完了報告の末尾に「推奨候補: X」を明記

A5: リスク観点

subagent_type: general-purpose
description: Bun リスク分析 (ベンダー/脆弱性/撤退)
prompt: |
  あなたはプロダクトセキュリティと事業継続の責任者です。
  「Bun を本番導入する際の主要リスク」を洗い出し、影響度×発生確率で評価してください。

  【リスクカテゴリ(最低)】
  1. ベンダーロック(Oven社の持続可能性、オープン度)
  2. 過去のCVE/セキュリティ脆弱性履歴
  3. 本番障害時のサポート入手経路
  4. 代替ランタイムへの撤退コスト
  5. 法務/ライセンス(MIT以外の混入リスク)
  6. 人材流出時の属人化

  【成果物形式】
  - 表形式: | リスク | 発生確率(H/M/L) | 影響度(H/M/L) | 緩和策 |
  - リスクは最低6個
  - 末尾に「最も注視すべきTOP3」を選出

  【制約】
  - 「可能性がある」だけで終わらせず、必ず緩和策を1行添える
  - 報告は900字以内
  - 完了報告の末尾に「High×High リスク数: N」を明記

5つとも共通するのは、「成果物形式」「文字数制約」「完了報告フォーマット」 が明示されていること。これがないと統合時に形式バラバラで読めないサマリが5つ戻ってきます(Ch05の黄金律)。

今すぐ試す

上記5ブロックを research-bun.md というファイルに保存し、Claude Code にこう指示します:「research-bun.md の内容を読み、記載された5つのAgent callを1メッセージで並列に実行して」。これだけで5並列が動きます。実行前に対象を自分の検討中テーマに差し替えれば、そのまま業務に使えます。

結果の集約と意思決定

5つのSubagentが戻ってきたら、メインAgentで統合サマリを作ります。統合指示は次の形式で与えます(これもそのまま使えます):

5つのSubagentレポート(A1〜A5)を読み、以下の形式で1枚サマリを作成してください。

【出力形式】

- 結論: GO / NO-GO / 条件付きGO
- 最大の根拠1行
- 最大のリスク1行

もっと詳しく: 5レポートが矛盾した時の対処

A1(技術)が「◎」なのにA5(リスク)が「H×H」多数、のように観点間で結論が割れる ケースは頻出します。これは失敗ではなく「正しく多面で見えた証拠」。対処は3パターン:

  1. 追加Subagentで争点を深掘り:矛盾の原因1つに絞って第2ラウンド実行
  2. 人間の判断軸を追加:「このチームはリスク回避文化か攻めか」をルーブリックに明記
  3. 条件付きGO:「リスク緩和策を3つ実装できれば GO」のように条件化

一番やってはいけないのは「矛盾を見なかったことにして平均点で決める」こと。多面で見る意味がなくなります。

並列多数 vs 順次1件ずつ

5並列が本当にお得か?ここはよく誤解されます。「順次だと30分、並列だと6分」ぐらいの素朴な期待を持ちがちですが、実態は下のグラフのとおり。

5並列 vs 順次 の所要時間比較(体感ベンチ)
順次は線形加算。並列は最も重いSubagentに律速される(+統合オーバーヘッド)。
5並列の強み

壁時計時間が大幅に短い

  • メインAgentの待ち時間が重Subagent1個分に収束
  • 観点ごとにコンテキストが独立で汚れない
  • 人間が他作業と並行しやすい(非同期)
5並列の弱み

コストは同時に5倍

  • トークン消費は合算(課金は並列でも割引なし)
  • 途中経過で方針転換ができない
  • 不要な観点に気づく前に5件走ってしまう
観点順次1件ずつ5並列3並列×2バッチ
壁時計時間遅い(5倍)速い(最遅に律速)中くらい
トークン消費同じ同じ(同時に発生)同じ
途中軌道修正○ できる✕ できない△ バッチ間で可
観点の相互参照○ 可(累積)✕ 不可△ 2バッチ目のみ
初学者にお勧め○ まず順次から慣れてから中間解

結論は単純で:「観点が独立 & 急ぎ & ある程度プロンプト設計に自信あり → 5並列」「観点間に依存 or 途中で判断したい → 順次 or バッチ」。あなたの業務状況で選びましょう。

2026年1月時点の注意

2026年1月時点で、1メッセージ内の Agent call 数に公式制限は明記されていませんが、実運用では 5〜7並列 までがコンテキスト・レート制限の観点で安定します。10並列以上は失敗率が上がるので避け、必要なら2バッチに分割してください。

ハンズオン手順

ここまで読んだら、自分のテーマで実行します。「Bun」は私の例題なので、あなたの職場で今まさに検討中の何かを選んでください。候補の出し方:

  • 直近の会議で出た「X どうする?」 — 新ツール導入、SaaS乗り換え、組織変更、プロセス改善など、判断保留になっている問い

Exercise 1
入門

自分のテーマで5観点のprompt雛形を書き換える

本章A1〜A5の5つのpromptを、あなたが今検討中のテーマに置き換えて保存してください。実行は不要、まずは書き換えてファイル化するところまで。

  1. テーマを1行で言語化(例: 「社内DBをPostgreSQLからAmazon Auroraへ移行すべきか」)
  2. A1〜A5のpromptで「Bun」と書かれている箇所を全置換
  3. 前提(規模・チーム構成・予算)をA2の【前提】欄に差し替え
  4. ファイル research-{theme}.md として保存
  5. 読み返して「これでSubagentが単独で動けるか?」自己チェック
模範解答を見る

テーマ例:「社内CRMを Salesforce から HubSpot へ移行すべきか」の書き換え例(A1抜粋)

subagent_type: general-purpose
description: HubSpot 技術調査 (互換性/性能/機能)
prompt: |
  あなたは SaaS CRM に詳しい技術リサーチャーです。
  「Salesforce から HubSpot へ移行可能か」を技術観点から評価してください。

  【調査項目】
  1. 既存 Salesforce カスタムオブジェクト/フローの移植性
  2. API レート制限と連携ツール対応状況(2026年1月時点)
  3. 過去のメジャーアップデート安定性
  4. 外部SaaS(Slack/GAS/MAツール)との統合
  5. カスタムレポート/ダッシュボード移行の難易度

  【成果物形式/制約】
  (本章A1と同じテンプレ)

ポイントは「テーマ名の置換」だけでなく「前提」「調査項目」も業界特化に書き直すこと。ここを手抜きするとSubagentがBun向けの一般論を返してきます。A2〜A5も同様に、CRM特有の軸(ライセンス課金体系、営業部門の教育、競合 Zoho/Dynamics との比較、顧客データ漏洩リスク)で書き換えます。保存先は research-crm-migration.md。

実際に5並列を回して統合サマリまで作る

Exercise 1で作ったprompt集を、実際にClaude Codeで並列実行し、統合サマリまで作成してください。

  1. research-{theme}.md を保存
  2. Claude Codeに「このファイルを読み、5つのAgent callを1メッセージで並列に実行して」と指示
  3. 5レポートが戻ってきたら、本章「結果の集約」節の統合promptをメインAgentに投げる
  4. 出力されたTL;DR+スコア表を summary-{theme}.md に保存
  5. 同僚1人に見せて「このサマリで判断できる?」フィードバックをもらう

# HubSpot 移行判断サマリ (2026-01-15)

## TL;DR
- 結論: 条件付きGO
- 最大の根拠: 年間ライセンス費 -38% かつ API連携数が現状ツールを全カバー
- 最大のリスク: 営業部20人の教育期間2ヶ月の受注機会損失

## 観点別スコア
| 観点 | スコア | 一言 |
|---|---|---|
| 技術    | 4/5 | API互換性は十分、カスタム移植に工数 |
| コスト  | 5/5 | 3年TCO -2,400万円 |
| 学習    | 3/5 | 営業部の教育2ヶ月が要 |
| 代替案  | 3/5 | Zoho/Dynamics より優位だが Salesforce据置も有力 |
| リスク  | 3/5 | 顧客データ移行時の漏洩リスクH×M |
| 総合   | 18/25 | GO ライン |

## 意思決定の分岐点
- GO条件: ①移行期の並行稼働3ヶ月 ②教育予算200万円確保 ③PoC部門先行
- NO-GO材料: 既存カスタムフロー30本以上が再現不可 / 営業部の強硬反対

## 次のアクション(24時間以内)
1. 情シスにカスタムフロー棚卸しを依頼
2. HubSpot日本法人に移行支援見積もりを打診
3. 営業部長に教育期間の受注影響ヒアリング

重要なのは「戻ってきた5レポートを鵜呑みにしない」こと。数値の根拠、出典URLを1件ずつ自分で軽く確認するだけで、生成AI特有のハルシネーションの9割は弾けます。あとはこのサマリをそのまま決裁資料の1枚目に貼って上司に持っていく。ここまで通して 普通なら丸一日かかる調査が30-45分 で終わります。
