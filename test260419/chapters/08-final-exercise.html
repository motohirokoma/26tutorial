最終演習：自分のワークフローに3つ組み込む

業務から3つ抽出→Subagent化→効果測定 — 7章の総仕上げ

所要時間: 35分 Phase: Practice 難易度: 応用

7章の学びを「自分の業務」に落とす

ここまでの7章で、Subagent の概念・パラメータ・タイプの使い分け・並列設計・プロンプト術・リサーチ5並列・レビュー分割並列を学んできました。ただし知識は使わなければ腐ります。この最終章で 自分の業務に3つ組み込むところまで行きましょう。

目標は「2週間のうちに3つの Subagent ワークフローを自分の業務で動かし、効果を数値で測定できている」状態です。ゼロから「候補抽出→プロンプト設計→実行→測定」までを1枚の表にまとめるのが本演習のゴール。

Tips

完璧な3つを選ぶ必要はありません。まずノートに5分で10個書き出し、その中から「頻度が高い×並列にしやすい×定型化できる」の3軸で3つに絞る。先に動かしてから、後で磨けばいいという割り切りが肝心です。

CONCEPT × DESIGN × EXECUTE の3軸を束ねたものが、あなた専用の運用テンプレートになる。

問題: 自分のワークフローに Subagent を3つ組み込む

以下の前提条件で、1枚の運用設計書を作ってください。フォーマットはスプレッドシート・Notion・Markdown、何でも構いません。

前提条件

項目内容
対象者Claude Code を日常業務で1週間以上使っている人
期間設計1日 + 運用2週間(実運用で検証)
個数3つ(欲張らない)
選定基準週1回以上発生する定型業務 / 独立タスクに分解可能 / 時間かコストが痛い
タイプExplore / general-purpose / 自作どれでも可(Ch03の判定フロー適用)
コンテキスト隔離原則 isolation 有効(長いリサーチ系は必須)
測定指標所要時間 / 完了率 / 満足度(5段階) の3つを最低限記録
検証環境Claude Code CLI (2026年1月時点)

求めるアウトプット (5点セット)

  • 棚卸しマトリクス — 自分の繰り返し業務を10個リストアップし、頻度×並列性×定型度で評価
  • 3つの候補カード — 名前・目的・起動トリガー・使う Agent タイプ・並列数
  • プロンプトテンプレ(3種) — Ch05の self-contained 原則を満たす。コピペで即起動できる粒度
  • 実行フロー図 — トリガー→並列起動→集約→出力の流れを Mermaid 1枚で
  • 測定指標と2週間ログ — 時間/完了率/満足度を数字で。Before/After の差分を出す

Note

「3つ全部うまく行く」ことを目指さないで構いません。3つのうち1つが残れば成功、2つ残れば合格、3つ残れば上出来。むしろ「やめるべき1つ」を見つけるのがこの演習の大きな収穫です。

段階ヒント (行き詰まったら開く)

いきなり全部開くと「答え写し」になります。まず自分で1つ書いてみて、詰まったヒントだけ開いてください。5つのヒントは 棚卸し → 分割 → プロンプト → 集約 → 測定 の順に対応しています。

もっと詳しく: ヒント1 — 候補の見つけ方 (Ch01 / Ch03)

Subagent 化の良い候補には5つのシグナルがあります:

  • 週1回以上発生する — 1度きりなら手でやったほうが速い
  • 独立タスクに分解できる — 5観点 / 5URL / 5ファイル など
  • 結果の形式が決まっている — 箇条書き・表・pros-cons など定型
  • コンテキストが重い — 主エージェントに余計な履歴を残したくない
  • 時間かコストが痛い — 1回30分以上か、認知負荷が高い

逆に悪い候補は: ①判断が対話的に変わる / ②中間状態の共有が必須 / ③1ステップで終わる軽タスク / ④厳密な再現性が必要(Agent は毎回少しゆらぐ)。この4つは手で書くか Slash Command にするほうが総じて速いです。

Tips

「候補を10個書き出せ」と言われて6個しか出ない人は、1日のタイムライン(9時→10時→…)を紙に書き、各時間帯に何をやっているかを埋めてみる。「メールトリアージ」「会議の準備メモ」「PRレビュー」「週報」など、無意識でやってる定型業務が必ず何個か出てきます。

もっと詳しく: ヒント2 — 分割単位の決め方 (Ch04 / Ch07)

並列分割のパターンは4つ(Ch04・Ch07で扱いました)。候補業務に当てはめて1つ選びます:

パターン分割軸並列数の目安合う業務の例
観点分割評価の側面(性能/可読性/安全性...)3-5PRレビュー / 企画書レビュー
対象分割同じ処理を複数対象に3-8複数URL調査 / 複数ファイル要約
候補分割複数解の並列生成3-5コピー3案 / 設計案3案
時系列分割時期・フェーズ別2-4過去半年の月次レポート比較

迷ったら 「結果が互いを参照しなくて完結するか?」だけ自問する。Yesなら並列、Noなら直列で十分。並列は 3-5がコストと効果のスイートスポット(Ch04)。

もっと詳しく: ヒント3 — プロンプトテンプレ (Ch05)

Ch05 の self-contained 原則に沿ったテンプレート骨格:

# 目的
[何を成果としたいか1行]

# 背景
[agent は会話履歴を見れない前提で、最低限の文脈]

# 入力
[ファイルパス or URL or 調査対象]

# 期待する形式
[箇条書き / 表 / pros-cons / コードdiff など具体的に]

# 制約
- 時間: [何分以内]
- 長さ: [何語以内]
- [その他の禁止事項]

# 完了報告
- 成果物のパス: [必須]
- 検証済みか: [Yes/No + 根拠]
- 所要時間: [実測]

手薄な例は「PRをレビューして」だけ。練り込んだ例は「本プロンプトに記載の観点3つ(性能/可読性/テスト)に限定し、優先度 H/M/L 付きの箇条書きで、完了報告はパス・検証済み可否・所要時間の3点を必須」まで書ききったもの。曖昧さを残すと揺らぐのが Subagent の特性です。

もっと詳しく: ヒント4 — 結果集約の仕方 (Ch06 / Ch07)

並列実行の結果をどう束ねるか、3パターン:

  1. 要約+優先度付け — 主エージェントが各 Subagent の出力を読み、重要度の高い項目 Top N を抽出して1枚のレポートに
  2. マージテーブル — 観点別の結果を行に、対象別の結果を列にして1つの表に統合(Ch07のPRレビュー方式)
  3. 投票・コンセンサス — 候補並列(3案生成)の場合、各案を別 Agent に採点させ、最高得点の案を採用

集約のコツは 形式を先に決めること。各 Subagent のプロンプトに「この表のこの列を埋める形で返せ」と指示すれば、集約時のマージコストがゼロに近くなります。

Warning

Subagent の戻り値は summary で返ることが多く、細部が落ちます。細部が必要な場合はプロンプトで「成果物ファイルに書いて、サマリにはパスだけ返せ」と指示するのが定石(Ch02)。

もっと詳しく: ヒント5 — 効果測定(時間削減/品質向上)

最小の測定セットは3指標:

所要時間
Before(手動分) vs After(Subagent分)
完了率
期待出力を一発で得られた割合(%)
満足度
5段階(1:ひどい〜5:最高)で自己評価

記録は 1行/回で十分。「2026-04-20 リサーチ 22分 完了◯ 満足度4 メモ: 代替案2の調査が浅い」のような形式。スプレッドシート1枚で2週間運用し、合計40行程度の生データが溜まれば、次の改善点は自然と見えます。

測定で見るべき本当の指標は「自分が再びそれを使いたいと思うか」。満足度4以上が 70% 超えなら定着候補、50%未満ならプロンプト再設計またはやめる判断。

考え方の解説

この3つに絞った理由は 3つの設計判断です。

  1. 週当たり頻度の分散を作る — 週2-3回(調査)/ 週5-8回(レビュー)/ 週1-2回(執筆) のように頻度をバラしておくと、運用のリズムが偏らない。どれかが今週使えなくても他が回る設計。
  2. observation vs creation の両方を持つ — 調査・レビューは「読む Subagent」、記事下書きは「書く Subagent」。役割が違うので プロンプト設計スキルが2方向に磨かれる。
  3. 成功体験の早い勝ち筋を1つ入れる — PRレビュー3観点は 最も導入コストが低く効果が見えやすい(Before/After が毎日取れる)。最初の2週間のモチベーション維持に効く。
Insight

MK 自身の2週間で最も効いたのは 新技術調査5並列でした。「90分→25分」の圧縮より、「1つに偏らない視点」が得られたこと(リスク観点が必ず独立 Agent で入るので、盲点が埋まる)。量より質の改善が本命だったと後から気づきました。

この3つを自分の業務で動かせる人は、もう「Subagent を使える人」から「Subagent でワークフローを設計できる人」に変わっています。

応用課題: 2週間後にさらに3つ追加する計画

最初の3つが2週間動けば、次は「追加する3つ」を選ぶフェーズです。測定データを踏まえて設計するのが、1周目との違い。

応用 1
応用

次の3つを選ぶレビュー会議を1人で30分でやる

2週間の計測データ(時間/完了率/満足度)を見ながら、次の3つを選ぶ判断を 30分の自己レビューで済ませてください。最終アウトプットは「追加3つの候補カード」と「やめる1つの決定」です。

  1. 現在の3つのうち、満足度3以下の週が半分を超えたものを「やめる」判断
  2. 棚卸しで上位4〜6位だったものから、次の3候補を選ぶ(2周目のスコア基準で見直す)
  3. 新しい3つは 1周目の強み(観点分割が得意 / プロンプト短め が得意 等)を生かせる形で設計
模範解答を見る

MK の2周目(2026年1月時点で仮運用中)の例:

  • やめる1つ: 「記事下書き3案並列」(満足度3.1/5、案の重複が多く集約コストが想定以上)
  • 追加3つ:
    1. 議事録からToDo抽出3並列(担当別の粒度で分割、週3-5回)
    2. メールトリアージ観点分割(返信要/情報のみ/削除、毎朝1回)
    3. 既存コードの依存関係マッピング(ファイル×3並列、月2-3回の調査タスク)
  • 設計の変化: 1周目で「候補並列(複数案生成)」は思ったほど効かなかったので、2周目は 観点分割 × 対象分割に振り切る
Insight

2周目で一番効くのは「やめる判断を怖がらない」こと。1周目で作り上げたものを捨てるのは心理的コストが高いが、データが示しているならやめる一択。捨てた業務は手作業 or Slash Command に戻すのが正解で、Subagent 一辺倒にしないバランスが熟練の証です。

修了のことば

おつかれさまでした。8章・4時間の旅、そしてこの演習をやり遂げたあなたは、もう「Subagent を知っている」側ではなく「Subagent で自分の業務を再設計できる」側の人間です。概念(Ch01-04)・設計(Ch05)・実践(Ch06-07)を束ね、最後に自分の仕事に刻み込むところまでやりきりました。

並列実行は、時間を買い戻す行為ではありません。自分の判断の質を、複数視点で補強する行為です。 — 本チュートリアルの総括
