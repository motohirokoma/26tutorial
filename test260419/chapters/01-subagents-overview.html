Subagentsとは

なぜ・いつ使うか、通常ツールとの違い、全体像

所要時間: 30分 Phase: Basics 難易度: 入門
この章のゴール

Subagent(=サブエージェント)を「自分の分身タスクランナー」として直感できるようになる。通常の Read や Bash ツールと何が違うのか、どんな場面で使うと時間が3-5倍速くなるのか、逆に使ってはいけない場面はどこか——ここを30分で押さえて、Ch02以降の「構文」の学習を自分事にします。読むより試すを合言葉に、コード片は全部コピペで動かせる形で載せています。

バージョン注記

本章の記述は 2026年1月時点 の Claude Code (Agent SDK/CLI) 挙動に基づきます。Agent の subagent_type ラインナップやパラメータ名は進化するため、コマンドが通らない時は 公式ドキュメント を必ず確認してください。

Subagent とは何か — 一言で

Subagent は、いまあなたと話している Claude(=メインエージェント)が、自分の分身として別文脈で起動する子 Claude です。子は独立した会話履歴を持ち、終わったら 要約1つ を親に返して消える。親は要約だけ読んで判断を続ける——これが基本モデルです。

メイン vs Subagent の関係
Subagent は親から見ると「一度だけ返事が返ってくる、黙々と働く関数呼び出し」に近い。

もう少し正確に書くと、Subagent は Agent tool を通じて起動され、独立したメッセージ履歴 / 独立したツール権限 / 独立したコンテキストウィンドウ を持ちます。親が触った会話履歴は引き継がれず、渡した prompt だけが出発点です。

今すぐ試す(30秒)

手元の Claude Code を開いて、チャットにこう打ってみてください。

このリポジトリの README.md を要約して、プロジェクトの目的だけ3行で返すタスクを
Agent tool で general-purpose として起動してください。

メインの Claude が 自分で README を読みに行く のと、子を起動して読ませ要約だけ受け取る のでは、返ってくる文量とメインのコンテキスト消費量が明確に違うのが体感できます。

なぜ Subagent が必要か — 3つの動機

単に「AIに依頼する」だけなら、普段の 通常ツール で十分です。わざわざ子を起動する価値は 3つの動機 に集約されます。

  • コンテキスト保護 — 大きなログ全文 / 巨大ファイル群 / 長いリサーチ結果を、親の履歴に入れずに 子に探索させ、要約だけを受け取れる。親のコンテキストウィンドウを節約できる。
  • 並列化 — 独立タスクは1メッセージで複数子を同時起動できる。5並列リサーチなら、順次だと15分→並列で3分、といった時間短縮が可能。
  • 役割分担 — 専門Agent(レビューア・エクスプローラー・特定ワークフロー)に任せて、親は「監督・統合」に専念できる。プロンプトのスコープも狭くなるので品質が上がる。
コンテキスト消費の比較(イメージ)
親が直接読むと履歴に全文が残る。子に読ませて要約を返させると、親には要約サイズだけが加算される。
もっと詳しく: なぜ並列が効くのか(同時性と課金の関係)

1メッセージ内で複数の Agent tool call を書くと、Claude Code ランタイムは 内部的にそれらを並行実行 します。待ち時間は「最も遅い子」で決まるので、独立なタスクほど時間短縮の効果が大きい。一方、各子はそれぞれ独立に課金されるので、並列数 × 各子のトークン消費 が合計コストになります。Ch04で「並列 vs 直列 / バッチング戦略」を具体的に扱います。

通常ツール vs Subagent — 決定的な違い

Read・Write・Bash・Grep などの通常ツールと Subagent は、同じ「タスク実行の手段」に見えて、知能レベルと独立性 が根本的に違います。

観点通常ツール(Read/Bash/Grep等)Subagent(Agent tool)
知能なし(決まった動作)あり(LLMが内部で判断・試行)
文脈親の履歴に結果が入る独立した履歴・結果は要約のみ返る
複数ターン1往復で終わる子の内部で複数ターン実行
ツール使用単一ツールのみ子がさらに Read/Grep/Bash 等を呼べる
並列起動基本 N/A1メッセージで多数起動可
コストほぼゼロ(親のトークンのみ)子のトークン × 子の数
向いてる用途明確な単一操作(1ファイル読む等)探索的・判断を伴う複数手順タスク
通常ツールで足りる例

Subagent は過剰

  • package.json を1つ読みたいだけ
  • ディレクトリ一覧を見たいだけ
  • 特定の文字列を grep で1回探すだけ
  • 既に確定した1行を書き換えるだけ
Subagent が光る例

判断 or 探索が混ざる

  • 「このリポジトリの認証の実装箇所を全部洗い出して」
  • 「5つの技術選定候補を並列で調査して比較表」
  • 「このPRをセキュリティ観点でレビューして」
  • 「独立な3タスクを同時に走らせたい」
今すぐ試す(判別クイズ)

次のうち Subagent 向き はどれでしょう? 心の中で答えてから下の折りたたみを開いてください。

  1. README を1回読み上げる
  2. src 配下の全テストファイルの失敗原因を仕分けする
  3. git status を1回打つ
  4. 3つの OSS ライブラリを技術/コスト/ライセンスで比較
答えを見る

2 と 4 が Subagent 向き(探索 + 判断 + 複数手順)。1 と 3 は通常ツールで十分。

6つのシーンを具体例で

1. リサーチ並列
「新技術Xを導入すべきか」を技術/コスト/学習/代替/リスクの5側面で同時調査 → Ch06で実践
2. 並列生成
記事冒頭の候補を3案同時に生成。親はトーンを比較して選ぶだけ
3. レビュー分割
大きなPRをセキュリティ/性能/可読性/テスト/型の5観点で並列レビュー → Ch07
4. 候補比較
OSSライブラリ3つを並列で触らせて「壊さない順」を返させる
5. 隔離実験
破壊的コマンド(DB migration等)を子の worktree 内で試し、親は要約で判断
6. 大量ログ分析
10MBのCIログを子に読ませ「失敗の根本原因3つ」だけ返させる。親の履歴に生ログを入れない
今すぐ試す(3並列の最小例)

あなたのメインエージェントに、下記のように指示してみてください。Ch02で詳しく扱う前の「体験としてまず動かす」のが狙いです。

次の3つを、1メッセージで Agent tool の general-purpose として並列起動してください。
  A: このリポジトリで使われているテストフレームワーク名と設定ファイル名を返す
  B: README で「install」と書かれている手順を1つにまとめて返す
  C: package.json の scripts を一覧化して各コマンドの意図を1行で推測

各子は50語以内のサマリで返してください。

全部で数秒〜十数秒、一度に3本の成果 が返ってきます。これが並列の体感です。

呼び出しの JSON イメージは下記のような形になります(Ch02で各パラメータを詳説)。

{
  "tool": "Agent",
  "input": {
    "subagent_type": "general-purpose",
    "description": "find test framework",
    "prompt": "このリポジトリで使われているテストフレームワーク名と\n設定ファイル名を1つ調べ、50語以内のサマリで返してください。"
  }
}

制約と4つの落とし穴

便利な Subagent にも、使う前に知っておきたい4つの落とし穴があります。どれも「知っていれば避けられる」類のものです。

4つの落とし穴マップ
4つとも対策は単純。Ch04-Ch05で各項目に対応するテクニックを学びます。

落とし穴ごとの対策一覧

落とし穴具体的な症状対策
コスト爆発軽い気持ちで10並列 → 通常の10倍トークン消費並列度の上限を決める(例: 5並列まで)。Ch04参照
要約性の制約子が生成した長文を親でそのまま使いたかったのに要約しか返らない「成果物はファイルに書く+パスだけ返す」運用。Ch05参照
状態非保持親が忘れたら子も分からない。履歴を暗黙に期待して失敗プロンプトを self-contained に書く。Ch05の原則
依存並列化「設計→実装→テスト」を並列実行して破綻独立性を判定してから並列化。Ch04の冒頭
注意

もう一つ地味な落とし穴として、子はデフォルトでは親と同じリポジトリを見ていること があります。破壊的な作業を子に任せる時は isolation: "worktree" を指定して別作業ツリーで動かすのが安全。Ch02で構文を扱います。

もっと詳しく: この章を学んだ後、どこから手を出すのが効率的?

人によって効く入口が違います。(A) 調査業務が多い人 → Ch01→Ch02→Ch06(リサーチ並列)を先に触ると効果を即体感。(B) レビュー業務が多い人 → Ch01→Ch02→Ch07(レビュー分割)。(C) 実装が中心の人 → 素直に順番通りで、Ch04(並列設計)とCh05(プロンプト術)で差が付きます。どの順でも Ch01 と Ch02 は土台なのでスキップしないこと。

Exercise 1
入門

Subagentを1つ起動してみる(体感)

自分の Claude Code 環境で、任意のリポジトリに対し以下を実行してください。実行結果のサマリを200字以内で記録します。

  1. 適当なディレクトリを開き、Claude Code を起動
  2. プロンプトで「このリポジトリの主な依存ライブラリを5つ選び、用途を1行で説明するタスクを general-purpose Subagent として起動してください」と依頼
  3. 返ってきたサマリの長さ・親の会話履歴の汚染度を観察
  4. 同じ問いを Subagent を使わず直接依頼した場合と比較
模範解答を見る

期待される観察結果と、起動に使える JSON イメージは次のようになります。

{
  "tool": "Agent",
  "input": {
    "subagent_type": "general-purpose",
    "description": "list top deps",
    "prompt": "このリポジトリのトップレベル依存を5つ選び、\n各ライブラリの用途を日本語1行で説明。\n最後に出力をサマリとしてまとめてください(200字以内)。"
  }
}

観察ポイント:(1) 親の履歴にはサマリ文だけが追加され、package.json の全文や node_modules の探索ログは入ってこない。(2) 一方 Subagent なしで直接依頼すると、親が Read を繰り返して履歴が大きく膨らむ。これが「コンテキスト保護」の実感。

Exercise 2
応用

自分の業務から「Subagent向きタスク」を3つ抽出する

Section4の6ユースケースと落とし穴表を参考に、自分の業務 からSubagent化すると効きそうなタスクを3つ挙げ、それぞれで次を明記してください。抽象語禁止(「効率化」だけはNG)。

  1. タスク名と、独立に分割できるか(並列か直列か)
  2. 親に返してほしい 最終サマリの形式(行数・項目・ファイル出力の有無)
  3. 失敗時のリスク(コスト / 情報漏れ / 依存ミス)と対策を1行
模範解答を見る

例: MK自身の「チュートリアル制作ワークフロー」のケース

【1. 章別HTML生成(並列)】
  - 8章を3-4並列のバッチで生成
  - 各子は summary 50語 + 成果物ファイルパスを返す
  - リスク: コスト → 並列上限4で抑制

【2. 視覚化要素の棚卸し(並列)】
  - 章ごとにSVG/Mermaid/Chart.js/表の使用数を数える子を並列起動
  - 親には "章名,道具,個数" のCSV形式で返却
  - リスク: 漏れ → リスト取得はmainで先に確定して子に渡す

【3. デプロイ前の構文チェック(直列)】
  - 各章の Mermaid 構文 → Chart.js 初期化 → リンク切れ の順
  - 前段の結果を次段が参照するので直列
  - リスク: 依存ミス → あえて並列化しない

ポイント: 自分の業務を最初から「並列化可/不可」「要約形式」で言語化する癖をつけると、Ch05〜Ch08の実践がスムーズになります。最終章 Ch08 では ここで挙げた3つを実際に組み込む ところまでやります。
