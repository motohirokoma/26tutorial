並列実行の設計

独立性判定・並列 vs 直列・バッチングの戦略

所要時間: 30分 Phase: Basics 難易度: 中級
この章のゴール

Subagentの真価は並列実行にあります。1メッセージ内で3〜5個のSubagentを同時に起動すれば、リサーチ・レビュー・多案比較の所要時間を一気に数倍圧縮できます。本章では「何を並列化すべきか(独立性判定)」「どう投げるか(1メッセージ複数call)」「どこで止めるか(バッチング戦略)」「失敗とコストの制御」までを一本のワークフローにまとめます。本章の記述は2026年1月時点の情報です。

なぜ並列化するのか: 時間 vs コストの構造

Subagentを直列で呼ぶと、各タスク時間が足し算になります。並列にすると、実時間は最も遅いタスクの時間で済みます。3タスクが各90秒なら、直列は270秒、並列は90秒。3倍の高速化です。

直列 vs 並列 タイムチャート(3タスク × 90秒)
並列化の効果はタスク数にほぼ比例。ただしトークン課金は直列と同じ(同時並行=同じ量を一気に消費)。

並列化の判断軸: 独立性(Independence)

並列化できるかどうかの唯一の判断基準は「独立性」です。タスクAの結果をタスクBが使うなら並列化できません。逆に「5つの異なる切り口で同じ問いを調べる」「ファイルごとにレビューする」などは典型的な並列向き。

「独立」かつ「書き換え競合なし」かつ「十分に重い」の3条件が揃ったときだけ並列化する。

独立/非独立の典型例

独立(並列OK)

結果が互いに不要

  • 5つのライブラリの性能をそれぞれ調査
  • ディレクトリ配下のファイルを個別レビュー
  • 同じ仕様から3案の設計書を並列ドラフト
  • 観点別(セキュリティ/性能/可読性)のレビュー
非独立(直列必須)

順番に依存

  • 調査→設計→実装→テスト(パイプライン型)
  • DB移行スクリプトの順次適用
  • 前タスクの出力を次タスクの入力に
  • 同一ファイルへの複数編集(競合)
今すぐ試す: 独立性チェック

自分が今週やっている仕事を3つ挙げて、上の判定フローに当てはめてみてください。「独立・競合なし・30秒以上」が揃うものが並列化候補です。多くの人は3つ中1-2つは当てはまるはずです。

1メッセージ内で複数Agent call(並列起動の作法)

Claude Codeの並列化は「同じアシスタントメッセージ内で Agentツールを複数回連続呼び出しする」ことで実現します。別メッセージに分けると直列になるので注意。

同時呼び出しの構造(3 Agent calls)

// 1つのアシスタントメッセージが tool_use ブロックを3つ含む → 3つとも並列実行される
[
  { "type": "tool_use", "name": "Agent", "input": {
      "subagent_type": "Explore",
      "description": "Next.js 15の性能を調査",
      "prompt": "Next.js 15の主要性能改善(RSC, Turbopack)を..." } },
  { "type": "tool_use", "name": "Agent", "input": {
      "subagent_type": "Explore",
      "description": "Remix v3の性能を調査",
      "prompt": "Remix v3(2025年リリース)の性能特性を..." } },
  { "type": "tool_use", "name": "Agent", "input": {
      "subagent_type": "Explore",
      "description": "Astro 5の性能を調査",
      "prompt": "Astro 5のSSR/Islands性能を..." } }
]
重要: 「同じメッセージ内」の意味

ここでいう「メッセージ」はアシスタント(Claude)の1ターンの出力を指します。ユーザーから見ると1回の応答の中で複数のtool_useが並ぶだけ。ユーザーは「Subagentを3つ並列で起動する」と依頼するだけでよく、並列化自体はClaudeが自動的に1メッセージ内にまとめてくれます。

手順: 3独立タスクを並列で投げる

  • 1. タスクを列挙 — 3つの独立した問いを紙に書き出す。依存関係があれば除外。
  • 2. 各promptを自己完結に — Subagentは会話履歴を見ない前提。背景/目的/期待形式を毎回書く。
  • 3. 「3つ並列で」と依頼 — 「同時に」「parallel」を明示すると確実。
  • 4. 結果を集約 — 3つのsummaryが返ったらメインAgentで統合レポート化。

バッチング戦略: 全並列 vs 3-5ずつに分割

「10タスクあるから10並列で!」は必ずしも正解ではありません。並列数が増えるほど、結果集約が破綻しやすく、1件失敗の影響も大きい。実用の落とし所はバッチングで3〜5個ずつ処理することです。

全並列 (N>5)

最速だがリスク大

  • 実時間は最短(最長タスクに等しい)
  • 同時課金で一気にトークン消費
  • 1件失敗で再実行判断が面倒
  • 集約メッセージが長大化 → 精度低下
  • 適用: 独立性100%、タスクが軽量、失敗許容
3-5並列バッチ

実用の最適解

  • 1バッチの集約が扱いやすいサイズ
  • 1バッチ失敗時に局所的に再実行可能
  • コスト/進捗を可視化しやすい
  • メインAgentのコンテキスト圧迫を抑制
  • 適用: 中〜重タスク、品質重視、継続運用
完全直列 (N=1)

遅いが確実

  • 前タスクの結果を次に渡せる
  • 失敗箇所で止めやすい
  • 時間は足し算で最遅
  • 適用: パイプライン依存があるとき
並列数別: 実時間・コスト・失敗リスクの推移(タスク10個想定)
実時間は並列数を上げるほど短縮されるが、失敗リスクと集約負荷が急増。3-5並列が時間・品質・コスト管理のバランス良点。
経験則: 3-5-10の法則

独立タスクが3つまで → 全並列、5つまで → 全並列も許容、10個以上 → 必ず3-5ずつバッチング。これを覚えるだけで事故の8割を回避できます。

失敗時挙動と再実行の設計

並列Subagentの1つが失敗しても他は独立に完遂されるのが基本挙動です。失敗したものだけを後追いで再実行するパターンが現実解。

失敗パターン症状対処
タイムアウトAgentが長時間返らないpromptを短く/範囲を限定して再実行
結果がスカスカsummaryが空 or 「未発見」のみprompt不足 → 背景と検索手段を追記
tool権限エラーExploreで編集しようとした等subagent_typeを見直す(general-purposeへ)
部分的成功5並列中2件だけ失敗失敗分のみ再並列投入(バッチ再実行)
今すぐ試す: 失敗後の再並列

一度並列リサーチを実行し、結果が薄いSubagentが出たら、それらだけを抽出して「この2つだけ、範囲を狭めて再度並列で」と依頼。1メッセージ内なので再び並列実行されます。この「部分再実行」が並列ワークフロー最大の強み。

コスト管理: 並列=同時課金という現実

並列化で縮むのは時間だけ、トークン量は直列と同じです。しかも同じ瞬間に一気に消費されるため、Claude Max等の利用枠に短時間で負荷がかかります。2026年1月時点、Claude Code(Max契約)では使用制限ウィンドウが5時間単位で計算されるため、並列数が多いほどウィンドウ上限に早く到達します。

並列数とトークン消費の推移(5時間ウィンドウ視点)
同じ10タスクでも「10並列一括」はウィンドウを10分で使い切る。「3並列×3バッチ」なら1時間かけて分散消費できる。

コスト抑制の3つのテクニック

  • promptを短く書く — 各Subagentに渡す背景情報は必要最小限に。500字 vs 2000字でトークン4倍。
  • 結果はsummaryで受ける — raw出力をそのまま展開せず、summary(要約)で受け取ればメインのコンテキストが軽い。
  • バッチ間に間隔を入れる — 10タスクを3並列×3+1にする、バッチ間に他の作業を挟むとウィンドウ負荷が均される。
もっと詳しく: Subagentのコストはメインの何倍か

Subagentは独立コンテキストを持つため、呼び出しごとに「システムプロンプト+与えられたprompt+作業トークン」を新たに消費します。メインAgentが同じ内容を直接処理するのに比べ、固定オーバーヘッド分だけ割高になります(目安: Subagent 1回あたり5-10k tokens追加)。

ただしメインのコンテキストが汚れないという強みがあり、長いセッションではむしろトータル消費が抑えられるケースも多い。短いワンショットタスクには向かず、「30秒以上の調査」「ファイル複数読む」のような重めの仕事で効いてきます。

ハンズオン: 3独立タスクを並列で投げる

ここまでの知識を一度に実行してみるのがこの章のメインイベント。以下の3タスクを1メッセージで並列投入します。

今すぐ試す: 並列リサーチ実習

Claude Codeを開いて、下記プロンプトをそのまま貼り付けてください。3つのSubagentが同時起動し、90秒前後で全結果が返ってきます。

以下の3つを独立したSubagent(subagent_type: Explore)で
**同時に並列**起動して調査してください。完了後にメインAgentで
3本のsummaryを200字以内で統合レポート化してください。

【タスク1】 React Server Components (RSC) の長所と短所を3点ずつ
【タスク2】 TanStack Start の特徴とRSCとの違いを3点
【タスク3】 Solid.js の性能特性と主な採用事例を3点

制約:
- 各Subagentのpromptは自己完結で書くこと
- 結果はsummary(要約)として返すこと
- 2026年1月時点の情報であることを明記

期待される挙動

  • 1メッセージ内に3つのtool_useが並ぶ(アシスタント応答で確認可能)。
  • 実時間は最長タスクに収束。直列なら3倍かかっていたはず。
  • メインが集約レポートを生成。3本のsummaryを1本の意思決定用文書に統合。
Exercise 1
初級

独立性の判定

以下の4タスクを「並列可/不可」に分類し、理由を1行で書いてください。

  1. A: 「ライブラリX, Y, Zの性能を調査する」
  2. B: 「仕様書を読む → 設計書を書く → コードを書く」
  3. C: 「同じ関数を3パターンで書き換える候補を出す」
  4. D: 「src/配下の全.tsファイルに型エラーがないか確認する」
模範解答を見る
  • A → 並列可: 3ライブラリは独立、同じ読み取り中心、重い調査なので典型的な並列向き。
  • B → 並列不可: 仕様→設計→実装は明確なパイプライン。前段の出力が次の入力なので直列必須。
  • C → 並列可: 3案は互いに独立(むしろ多様性を出すために別コンテキストで生成したい)。各Subagentに異なる制約を与えて並列実行。
  • D → 並列可(バッチング推奨): ファイル単位で独立。ただしファイル数が10+あるなら3-5ずつのバッチングが賢明。

判定軸は「結果の依存」「書き換え競合」「タスクの重さ」。この3つで迷わず振り分けられるようになれば実務に応用できます。

Exercise 2
応用

自分の業務で3独立タスクを並列投入する

今日・明日やる予定の仕事から、並列化できる3タスクを抽出して実際にClaude Codeへ投入してください。

  1. 候補を5つリストアップ
  2. 独立性判定フローでフィルタ(3つに絞る)
  3. 各タスクのprompt(自己完結)を書く
  4. 「3つ並列で起動して」と明示して投入
  5. 実時間と結果品質を記録(直列で想像した場合との比較)
模範解答を見る

回答例: MK(筆者)の実運用から

  • タスク1: 「次の学習サイトで使うチャート系ライブラリ比較(Chart.js / D3 / ECharts)」
  • タスク2: 「Claude Code の subagents 機能について2026年の最新アップデート調査」
  • タスク3: 「類似チュートリアルサイト(日本語)を3つ検索してUI/UXの差分を抽出」

3つとも独立(互いの結果を参照しない)、書き換え競合なし(読み取り中心)、30秒以上(調査タスク)。Explore agentを3並列で起動し、実時間は約2分、直列なら6分以上かかっていた計算。

気づき: promptを雑に書くと1つだけ結果が薄くなる。その場合は該当1件のみ再並列で投げ直す(= バッチ再実行)と全体の品質が揃う。
