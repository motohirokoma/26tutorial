Subagentプロンプト術

self-contained な指示、コンテキスト引き渡し、品質制御

所要時間: 45分 Phase: Practice 難易度: 中級
この章のゴール

Subagent の成否は「起動時に渡すプロンプト」で9割決まります。Subagent は会話履歴を見ないため、指示が単体で成立している(self-contained)ことが絶対条件。本章では必須5要素・コンテキスト渡し・品質制御の3軸で、曖昧な依頼を「再現性ある依頼」に書き換える技法を身につけます。本章の記述は2026年1月時点。

原則: self-contained

Subagent は呼び出し側(メイン Agent)の会話履歴・ファイル操作履歴・直前の発言を一切見られません。プロンプトに書かれた情報と自力で取得できるファイルだけがすべて。これをself-contained原則と呼びます。

メイン Agent と Subagent のコンテキスト境界
Subagent に渡るのは「prompt 文字列1つ」だけ。メイン側の空気は読めない。

したがってプロンプトは「新人に引き継ぐドキュメント」に近い密度が必要です。自分の頭の中にある前提をすべて書き出す覚悟を持ちましょう。

もっと詳しく: なぜ「履歴を渡す」設計ではないのか

Subagent の最大のメリットはコンテキスト汚染の隔離。メインの会話履歴を全部渡してしまうと、Subagent 側も長大な文脈を抱えてしまい、(1)コスト増、(2)lost-in-the-middle による精度低下、(3)意図しない過去情報の混入、が起きます。「必要な情報だけを厳選して渡す」制約が、結果的に指示の質を押し上げる設計になっています。

今すぐ試す: 文脈依存プロンプトを壊してみる

メイン Agent に以下の会話をさせた後、Agent tool を呼んでみてください。

User: src/auth.ts を読んで
Claude: (ファイルを読む)
User: 今読んだやつのテストを書いて(← Agent tool 経由で指示)

Subagent 側は「今読んだやつ」が何か分からず、ファイルを最初から探し回るか、推測で的外れなテストを書く羽目に。プロンプトに src/auth.ts のテストを tests/auth.test.ts に書いて と明示すれば回避できます。

Subagent プロンプト必須5要素

プロンプトは以下の5要素を全部含むと安定します。順不同ですが、1つでも欠けると品質が落ちる経験則です。

必須5要素チェックリスト(SVG)
「制約」だけは状況によって省略可。それ以外の4つは外さない。
  • 目的 — なぜやるのか。Subagent が細部で迷ったとき、これが判断材料になる。
  • 背景 — 関連ファイルパス・既存コードの方針・過去の経緯。ファイルパスを書いておけば Subagent は自力で読む。
  • 期待する出力形式 — 「Markdown レポート 300字」「Python ファイル1本」「JSON 配列」など、型を決める。
  • 制約 — 「package.json は触らない」「依存追加禁止」「既存 API を壊さない」など。
  • 検証基準 — 「npm test がパスする」「grep でこの文字列が出る」など、合格条件を具体化。
もっと詳しく: 「目的」を書くと何が変わるのか

目的は Subagent の判断の拠り所。たとえば「関数をリファクタリングして」とだけ書くと、Subagent は速度優先か可読性優先か分からず迷子になる。「新人が読める可読性を上げたいのでリファクタリング」と書けば、Subagent は自動的に「変数名を長く」「コメント追加」「ネスト浅く」といった判断を自発的に行えます。目的は北極星指示として機能します。

Before / After: 悪例と良例を並べる

百聞は一見に如かず。同じ依頼を最悪の書き方→改善版で対比します。

ケース1: リサーチ依頼

Before (悪い例)

「Next.js 15 について調べて」

  • 目的不明: 何のための調査?
  • 範囲不明: パフォーマンス? 新機能? 移行コスト?
  • 形式不明: 一言? 3ページ?
  • 検証基準なし: 何が揃えば合格?
After (良い例)

目的・背景・形式・検証すべて明記

目的: 社内 Next.js 14 プロジェクトを 15 に
  上げるべきか意思決定するため。

背景: 現状は Next.js 14.2.3 + App Router。
  CI は GitHub Actions、デプロイ先は Vercel。
  該当 PR の 仕様: docs/nextjs15-rfc.md

調査観点:
 1. 破壊的変更の件数と影響範囲
 2. 移行所要時間の見積もり
 3. 得られる主な利点(3つ)
 4. 他社の移行事例(2件以上)

出力形式: Markdown レポート / 800字以内 /
  見出しは観点そのまま / 最後に 3段階の
  「移行推奨度(A/B/C)」と理由を1行

検証: 公式 release notes と2社以上の
  事例記事 URL を引用付きで提示

ケース2: コード改修依頼

Before (悪い例)

「ログイン処理を修正しといて」

  • 対象ファイル不明
  • バグなのか機能追加なのか不明
  • 完了条件なし
  • テストの扱い不明
After (良い例)

パス・症状・期待・検証をすべて指定

目的: 誤ったパスワードを3回入力すると
  アカウントロックされる仕様を実装する
  (ユーザーの安全要件)

対象ファイル:
 - src/auth/login.ts (現状の実装)
 - src/auth/login.test.ts (ここにテスト追加)
 - docs/auth-spec.md (要件詳細、必ず読むこと)

やること:
 1. failedAttempts を Redis で管理
 2. 3回連続失敗で lockedUntil に +15分
 3. ロック中は LOGIN_LOCKED エラーを返す

やらないこと:
 - 既存の signIn API シグネチャは変えない
 - 他ファイルには触らない

検証基準:
 - npm run test src/auth がすべてパス
 - 新テスト 3 件以上 (成功/失敗/ロック中)
 - ESLint エラー 0

完了報告: 変更ファイルパスと、追加した
  テスト名を列挙。50語以内。

ケース3: コードレビュー依頼

Before (悪い例)

「この PR レビューして」

  • PR の実体がどこにあるか分からない
  • 観点が無限定 → 表面的な指摘だけ
  • 粒度が不明 → 1行コメントで終わる可能性
After (良い例)

観点・深さ・形式・優先度まで指定

目的: PR #1234 を本番マージする前に
  セキュリティ観点で重大な見落としが
  無いかを検証する。

対象: git diff main...feature/payment
  (または diff 全文をファイル経由)

観点 (セキュリティのみ):
 1. 入力バリデーション
 2. 認可チェック (IDOR含む)
 3. SQL/OS コマンドインジェクション
 4. 機密情報のログ出力

除外観点: 可読性 / パフォーマンス /
  スタイル(別レビューで扱う)

出力形式: Markdown 表
  | 重大度 | ファイル:行 | 指摘 | 修正案 |
  重大度は Critical / High / Medium のみ。
  Low は書かない。

合格基準: Critical が 0 かつ High が
  2件以下なら approve を提案

コンテキストを渡す2つの作法

長い仕様書・大量の参考資料を渡したいとき、プロンプトに全部貼るとプロンプトが肥大化して可読性が落ちます。ベストはファイル経由です。

悪例→良例への変換フロー
graph LR
  A["長文コピペ prompt
(5000字)"] --> B{"肥大化
読みにくい"}
  B --> C["一時ファイルに書き出す
docs/task-spec.md"]
  C --> D["prompt は
「docs/task-spec.md を読んで
それに沿って実行」と短く"]
  D --> E["Subagent が
Read ツールで自力取得"]
  E --> F["プロンプトは150字
情報は完全に届く"]
「仕様ファイルを先に書く → パスだけ渡す」が定石。

作法A: ファイル経由(推奨)

# Step 1: メイン側で仕様を書き出す
docs/tasks/refactor-auth-spec.md を作成:
  - 対象ファイル一覧
  - 変更方針
  - 制約事項
  - 検証手順

# Step 2: Subagent に短いプロンプトで投げる
prompt:
「docs/tasks/refactor-auth-spec.md を必ず
 最初に読んで、そこに書かれた方針通りに
 src/auth/ 配下をリファクタリング。
 完了後、変更ファイルパスと追加テスト名を
 50語以内で報告。」

作法B: 埋め込み(短い情報のみ)

# 50-200字程度の情報なら直接 prompt に
prompt:
「DB スキーマは以下:
 users (id BIGINT PK, email VARCHAR UNIQUE,
   locked_until TIMESTAMP NULL)
 login_attempts (user_id FK, failed_at TIMESTAMP)
 このスキーマ前提で、3回失敗で15分ロックする
 SQL マイグレーションを migrations/ に追加。」
使い分けの閾値

目安: prompt が 300字を超えるなら、情報部分はファイルに切り出す。ファイル経由は (1) 後で再利用しやすい、(2) 改版差分を git で追える、(3) 人間のレビューにも使える、という副次的メリットもあります。

品質制御: 完了報告の型を縛る

Subagent の戻り値はメイン Agent のコンテキストを消費します。だらだら長い報告は並列実行時に致命的。以下3つの「型縛り」で品質を上げます。

テク1: 文字数制限

完了報告の字数別・メインコンテキスト消費(目安)
5並列で各2000字の報告 = 合計1万字相当。対して各150字なら750字で収まる。
# 悪い例: 長大レポートが返ってきて context 枯渇
「調べて報告して」
→ Subagent が 3000字の丁寧な分析を返す
→ 5並列 = 1.5万字を main が抱える

# 良い例: 字数上限を明記
「調査結果を 150字以内で要約。
 詳細は docs/research-result-01.md に書き出し、
 報告には『詳細: docs/research-result-01.md』の
 一行を必ず含めること」

テク2: 成果物パス必須化

「レポートは必ずファイルに書き出し、パスを報告に含める」と書くと、メイン Agent は後で詳細を読みたくなったときファイルを開けば良くなります。

完了報告フォーマット(必須):
 - 成果物パス: (絶対パス or リポジトリ相対)
 - 要点: 50字×3行以内
 - 次アクション: あれば1行

テク3: 検証明記

「やったこと」ではなく「検証済みであること」を報告させる。Subagent は聞かれないと自発的に検証しない場合があります。

完了前に必ず以下を実行し、結果を報告に含める:
 1. npm test の最終出力(PASS / FAIL)
 2. npm run build の exit code
 3. 対象ファイルに対する grep で、
    変更が意図通り入っているか自己確認
いずれかが FAIL の場合、完了と報告しない。
(= 再試行または原因を報告)
もっと詳しく: 報告文字数が長くなりすぎるパターン

Subagent は「親切であろうとして」裏側の判断を全部語りがち。丁寧バイアスと呼ばれる挙動で、「目的/プロセス/結果/注意点/次ステップ...」とテンプレ的に書いてしまう。対策は「報告は決定済みファクトのみ。思考過程は書かない。」と明示すること。

テンプレート: 迷わず書けるひな形

上記をまとめた汎用テンプレートです。自分の環境にコピペして使ってください。

プロンプト構造(7ブロック)
全部書くと冗長に見えるが、チェックリストとして走査するだけで漏れが激減する。
今すぐ試す: テンプレートをコピーして1本書く

以下のひな形をエディタに貼り、自分のタスクで穴を埋めて Agent tool に渡してみましょう。

【目的】___(1文で)___
【背景】
 - 関連ファイル: path/to/file.ts, docs/spec.md
 - 現状: ___
【やること】
 1. ___
 2. ___
【やらないこと】
 - ___
【出力形式】___(型と字数)___
【検証基準】
 - ___ が成功すること
 - ___ を確認すること
【完了報告】
 - 成果物パス: (必須)
 - 要点: 50字×3行以内
 - 検証ログ: PASS/FAIL

アンチパターン5選

現場で繰り返し見る失敗パターンを症状 → 原因 → 対策で整理します。

症状原因対策
Subagent が「ファイルが見つかりません」と返す 相対パスで書いた / プロジェクトルートが不明 リポジトリ相対の絶対的なパスで書く。必要ならプロンプト冒頭で cwd を明記
修正したはずのバグが残っている 検証基準なし。Subagent が検証せず完了宣言 「npm test をパスすること」を検証基準に明記
期待と全く違うものが返る 目的が書かれていない。Subagent が意図を推測 【目的】行を必ず先頭に。なぜやるかを1文で
完了報告が長すぎて context 枯渇 字数制限なし。詳細をそのまま返す設計 「150字以内、詳細はファイル」と明記
本来触るべきでないファイルまで編集 やらないことが書かれていない 【やらないこと】ブロックで具体ファイル・挙動を禁止
もっと詳しく: 「目的」と「やらないこと」は矛盾しない

初学者は「目的を書けば Subagent が正しく動く」と考え、禁止事項を軽視しがち。実際は逆で、目的はポジティブ制約、やらないことはネガティブ制約として相補的に機能します。目的だけだと「過剰に親切」で余計な変更が入り、禁止だけだと「消極的すぎ」で必要な仕事まで止まる。両方書いてバランスを取ります。

Exercise 1
中級

曖昧タスクを self-contained に書き直す

以下の曖昧なタスクを、必須5要素(+やらないこと+完了報告)をすべて含む self-contained プロンプトに書き直してください。自分の環境/プロジェクトが無ければ架空の設定で構いません。

「API の速度遅いから調べて直して」
  1. このままだと Subagent が何を返すか想像してみる
  2. 目的/背景/やること/やらないこと/出力形式/検証基準/完了報告の7ブロックで書き直す
  3. 実際に Agent tool に投げて、返ってきた結果の「解像度の違い」を体感する
模範解答を見る 
【目的】/api/users エンドポイントのレスポンスが p95 で 1500ms
      超えており、ユーザーのダッシュボード表示が遅い。
      500ms 未満に改善して UX 問題を解消する。

【背景】
 - 対象: src/api/users.ts (GET /api/users ハンドラ)
 - 関連: src/db/userRepo.ts (DB アクセス層)
 - 計測: monitoring/recent-perf.md にダッシュボードの直近ログ
 - 現状: N+1 クエリの疑いあり(未検証)

【やること】
 1. userRepo.findMany の呼び出し箇所を特定し
    N+1 の有無を実際のクエリログで確認
 2. N+1 があれば JOIN か batch 取得に改修
 3. 既存の単体テストを壊さず、回帰テストを追加

【やらないこと】
 - DB スキーマは変更しない
 - API レスポンス形式(JSON 構造)は変更しない
 - 認証周り(src/auth/*)には触らない

【出力形式】
 - 変更ファイルのパッチ適用
 - 変更サマリ (Markdown, 150字以内)

【検証基準】
 - npm test が全パス
 - ローカルで GET /api/users を
   curl で叩いて 500ms 未満(3回計測の平均)

【完了報告】
 - 変更ファイルパス
 - 修正前後の計測値(ms)
 - 残課題(あれば)
 全体 80字以内

ポイント: 「速度遅いから調べて直して」から、このレベルまで落とし込めば Subagent は迷いません。最初は書くのがだるく感じますが、1度書けば次回からコピペで再利用できます。

Exercise 2
応用

完了報告を字数制約で締める

あなたが最近 Subagent (または同僚) に依頼した調査タスクを1つ思い出し、「完了報告フォーマット」を次の3点を含む形で書き直してください。

  1. 報告本文は150字以内
  2. 詳細は別ファイルに書き出し、そのパスを必ず報告に含める
  3. 検証が通ったか/通らなかったかをPASS/FAIL の明示で書かせる

書き直した完了報告フォーマットを、同じタスクに対する Agent 呼び出しで使い、「報告字数」と「メイン側で詳細を読む必要が出た回数」を記録してください。

模範解答を見る

回答例(調査タスク「React 19 への移行調査」):

【完了報告フォーマット】
 本文は以下の3行テンプレを埋めるのみ(各50字以内):
 - 結論: (移行推奨度 A/B/C と1行理由)
 - 重大な発見: (あれば1つ、なければ「なし」)
 - 詳細レポート: docs/research/react19-migration.md
 上記以外は書かない。思考過程や感想は不要。
 詳細は必ずファイルに書き出し、本文にはパスのみ。

【検証】
 - 引用元 URL が3つ以上あること(ファイル内に記載)
 - 公式 release notes を少なくとも1件含む
 検証結果を本文末尾に「検証: PASS」または
 「検証: FAIL (理由)」の1行で記す

計測結果例: 字数 2400字 → 180字(約93%減)。メイン側が詳細ファイルを読んだのは10回中2回だけで、多くの場合は結論の3行だけで判断できた。並列5本走らせたときのメインコンテキスト消費が特に劇的に改善します。
