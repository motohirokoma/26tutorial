実践：コードレビューを分割並列化

ファイル/観点ごとに並列実行、最後にまとめる

所要時間: 30min Phase: Practice 難易度: 中級
本章について

本章の記述は2026年1月時点の情報です。大きなPRを1人で読むのは苦痛ですが、Subagent 5体に観点別で並列レビューさせると、30分の作業が5分で終わります。自分のコードを題材に手を動かしながら進めましょう。

1. なぜ分割並列化か

大きなPRを1体のAgentに丸投げすると、注意が分散して全観点が薄くなります。「セキュリティも見て、パフォーマンスも見て、可読性も見て…」と詰め込むと、認知負荷が上がり、各観点が中途半端になります。観点別に分けて5並列で走らせると、各Subagentは1観点に集中でき、質が上がります。

観点分割の構造 — 1体多観点 vs 5体1観点
左: 1体で5観点 → 薄い。右: 5体が1観点ずつ並列 → 深い + 最後にメインAgentがmerge。

観点分割が効くのは「観点同士が独立」だからです。セキュリティの指摘はパフォーマンスの指摘と干渉しません。独立性があるタスクは並列化の最有力候補です。

もっと詳しく: なぜ1体で5観点だと品質が落ちるのか

Subagentも人間と同じく、プロンプトに書かれた要件を順序づけて処理します。「セキュリティ、パフォーマンス、可読性、テスト、型」の5観点を並べると、前半の観点に重点が置かれ、後半は流されがちです。さらに、1体のAgentが持つコンテキストは有限で、大PRを全観点で精査するとトークン圧迫で末端の指摘が雑になります。5体に分けると各体がフルコンテキストを1観点に使えます。

2. 観点分割 vs ファイル分割

並列化の切り口は大きく2つあります。観点分割(全ファイルをセキュリティ観点で見る、次に全ファイルをパフォーマンス観点で見る…)と、ファイル分割(ファイルAを全観点で見る、ファイルBを全観点で見る…)です。どちらが適切かはPRの性質で決まります。

観点分割(推奨)

全ファイル × 1観点 / Agent

  • 観点ごとの一貫した基準で指摘
  • クロスファイルの問題を発見しやすい
  • 大きなPR(5ファイル以上)に向く
  • mergeが「観点別の要約」に自然に整う
ファイル分割

1ファイル × 全観点 / Agent

  • ファイル単位の完結レビューに向く
  • ファイル間で観点の深さがブレやすい
  • 小〜中PR(3ファイル以下)に向く
  • クロスファイルの問題を見落としがち

迷ったら観点分割を選びましょう。観点別の基準が揃うので、統合レポートが読みやすくなります。ファイル分割は「monorepoの独立パッケージを横断するPR」のような特殊ケースで使います。

PR規模ファイル数推奨並列数
小1-2ファイル分割 or 単体レビュー1-2
中3-8観点分割3-5
大9+観点分割(必須)5
超大(monorepo横断)20+ファイル分割 × 観点分割の2段5 × N

3. 5観点の実プロンプト

観点ごとのSubagentプロンプトは「共通構造 + 観点固有の差分」で作ります。共通部分は「PRの diff を読む、指摘は重要度別に、50語以内で報告」。差分は各観点のチェックリストです。以下の5本を全部使えるようにコピペ可能な形で掲載します。

共通テンプレート

あなたは{OBSERVATION}観点のシニアレビュワーです。
対象: {PR_PATH} の diff（または指定ファイル群）
タスク: 下記チェックリストで重要な問題のみ指摘してください。
チェックリスト:
{CHECKLIST}
出力形式:
- 重要度(Critical/Major/Minor)
- ファイル:行番号
- 問題の要約(1文)
- 修正提案(コード or 1文)
完了報告: 50語以内、件数と最重要1件の要約。

観点別プロンプト5本

セキュリティ

{OBSERVATION}=セキュリティ / {CHECKLIST}:

- SQL/コマンドインジェクションの可能性
- XSS・CSRF・認可抜け
- シークレット/APIキーのハードコード
- 入力バリデーション不足
- 依存ライブラリの既知脆弱性(package.json/requirements.txt の変更)
- 認証バイパスのロジック欠陥
パフォーマンス

{OBSERVATION}=パフォーマンス / {CHECKLIST}:

- N+1クエリ(ループ内のDBコール)
- 不要なリレンダリング / 大きなuseEffect依存配列
- O(n²)以上のアルゴリズムで n が大きくなりうる箇所
- メモリリーク(イベントリスナ・setInterval の解除漏れ)
- 不要な同期処理(awaitで直列化)
- キャッシュ可能なのにされていない計算
可読性

{OBSERVATION}=可読性 / {CHECKLIST}:

- 関数が30行超 or ネストが4段超
- 変数名が意図を表していない(tmp, data, x)
- マジックナンバー / マジック文字列
- コメントと実装の乖離
- 複雑な条件式(ド・モルガンで簡約可能)
- 責務が混在している関数(分割余地)
テスト

{OBSERVATION}=テスト / {CHECKLIST}:

- 新規/変更コードに対応するテストの有無
- エッジケース(空配列・null・境界値)のカバー
- 正常系のみで異常系テストなし
- テストが実装と1:1すぎて意味が薄い
- モックが本物の挙動を偽装しすぎている
- 非決定的なテスト(時間・乱数・外部API依存)
型

{OBSERVATION}=型 / {CHECKLIST}:

- any / @ts-ignore / # type: ignore の使用
- unknownを受けて型ガードなしにキャスト
- 戻り値の型アノテーション欠落(公開関数)
- union型の不完全な網羅(exhaustive check なし)
- Optional/nullable の扱い不備(! の多用)
- ジェネリクスで表現できる重複型定義

この5本を1メッセージ内で同時起動すれば、5体が並列でレビューします。4章で学んだ並列実行の知識がそのまま活きます。

観点ごとのプロンプト差分
観点重視する箇所見落としがちな例
セキュリティ入力境界・権限・秘匿情報エラーメッセージに含まれる内部パス
パフォーマンスループ・I/O・データ量深いコンポーネント木での再レンダリング
可読性命名・長さ・責務正しいが読みにくい三項演算子の連鎖
テスト網羅・意味性実装を書き写しただけのテスト
型any・網羅性・nullableライブラリ境界の型定義の甘さ
もっと詳しく: 観点を増やしたい場合(アクセシビリティ・i18n・ログ)

Webフロントならアクセシビリティ(aria属性、キーボード操作、コントラスト)、国際化が必要ならi18n(ハードコード文字列、日付フォーマット)、運用系ならログ・可観測性(構造化ログ、メトリクス、トレースID)を観点として足せます。ただし並列は5観点程度が体感上限。7観点以上になるならフェーズ分割(P0観点→P1観点)を検討。

4. 結果集約フロー

5体の指摘が返ってきたら、メインAgentがmerge + 優先順位付けします。単純に並べるだけではレビュイが消化不良を起こすので、severity別に並べ替えしてから渡します。

5並列レビュー → merge の全体フロー
graph TD
  PR["PR diff"] --> Main["Main Agent"]
  Main --> S["Security Agent"]
  Main --> P["Performance Agent"]
  Main --> R["Readability Agent"]
  Main --> T["Test Agent"]
  Main --> Ty["Type Agent"]
  S --> Merge["Merge & 優先順位付け"]
  P --> Merge
  R --> Merge
  T --> Merge
  Ty --> Merge
  Merge --> Crit["Critical"]
  Merge --> Maj["Major"]
  Merge --> Min["Minor"]
  Crit --> Out["最終レビューコメント"]
  Maj --> Out
  Min --> Out
5 Subagentが並列で返し、メインAgentがseverity順に整形する。

優先順位付けのルール

  • Critical — セキュリティ穴・本番で壊れる可能性。即修正、マージブロック。
  • Major — パフォーマンス劣化・テスト不足・型の甘さ。今PR内で修正推奨。
  • Minor — 可読性・命名・軽微なリファクタ。別PRでもよい、または放置可。
  • Noise — 「好みの問題」「重複指摘」はmerge段階で捨てる。

merge時に重複指摘が出るのは健全なシグナルです(そこは本当に問題)。ただしレビューコメントとしては1つに統合して提示しましょう。

5. 時間短縮の実測

観点分割の並列化でどれくらい速くなるか、同じPR(約800行、6ファイル)で実測した結果を示します。測定条件は2026年1月のClaude Code環境。

レビュー所要時間の比較(分)
5並列は1体直列の約1/5。人間手作業より7-8倍速い。
5min
5並列レビュー所要
↓ 直列の1/5
18件
平均指摘数
↑ 単体レビューより約2倍
92%
有効指摘率
↑ ノイズ低い

並列化のコストは同時課金 × Agent数なので、5並列は単体の約5倍のトークン消費になります。ただし人間の時間単価を考えれば、大抵のケースでペイします。

6. ハンズオン: 自分のコードを3観点で並列レビュー

手を動かしましょう。自分の最新PR or ローカルの変更を3観点(セキュリティ/パフォーマンス/型)で並列レビューします。5観点フルはコストが重いので、まずは3観点で感覚を掴んでください。

1
題材を選ぶ
直近の自作PRか、未マージのブランチを用意
2
プロンプト準備
§3の5本から3本コピー
3
並列起動
1メッセージで3つのAgent呼び出し
4
merge
結果をseverity順に整形
実行コマンド例(Claude Code CLI)

Claude Codeを起動して、以下を1メッセージで送ります(擬似コード):

# 事前に対象を確認
$ git diff main...HEAD --stat
$ git diff main...HEAD > /tmp/review-target.diff

# Claude Codeに以下を1メッセージで依頼:
「/tmp/review-target.diff を以下3観点で並列レビューして。
 - Agent 1 (general-purpose): セキュリティ観点 + §3セキュリティチェックリスト
 - Agent 2 (general-purpose): パフォーマンス観点 + §3パフォチェックリスト
 - Agent 3 (general-purpose): 型観点 + §3型チェックリスト
 全員Critical/Major/Minorで分類し、50語以内で完了報告。
 あなた(メイン)は結果をseverity順にmergeして、重複は1件に統合してほしい。」

ポイント: Agent呼び出しは同一メッセージ内で複数tool_useにすること。順次起動すると直列になります。

今すぐ試す

上記の手順を今このタブを開いたまま別窓で実行してみてください。5分で終わります。大規模PRでなくて構いません、10-50行の変更でOK。観点別にどんな指摘が返るか、その質感を体験するのが目的です。

7. よくある落とし穴

  • diff を渡さず PR URL だけ渡す — SubagentはネットアクセスやGitHub APIを叩けないケースが多い。ローカルdiffをファイルで渡す。
  • 「完了報告50語以内」を書き忘れる — Subagentが長文を返し、メインのコンテキストを圧迫する。必ず出力制約を書く。
  • mergeを機械的に連結してしまう — 重複指摘が並び、ノイズが増す。severity順+重複統合が鉄則。
  • 常に5並列する — 小PRは単体で十分。規模に応じた並列数を選ぶ。
Exercise 1
入門

プロンプト差分を作る

§3の共通テンプレートを使って、アクセシビリティ観点のチェックリストを5項目作成し、{OBSERVATION}と{CHECKLIST}を埋めて完成プロンプトを書いてください。

  1. 共通テンプレートをコピー
  2. アクセシビリティで見るべき項目を5つ書き出す(aria、キーボード、コントラスト…)
  3. {PR_PATH}は /tmp/review-target.diff 等の具体パスに置換
模範解答を見る 
あなたはアクセシビリティ観点のシニアレビュワーです。
対象: /tmp/review-target.diff
タスク: 下記チェックリストで重要な問題のみ指摘してください。
チェックリスト:
- interactive 要素の aria-label / role 不足
- キーボードのみでの操作不可能な箇所
- フォーカス順序が視覚順序と乖離
- color contrast が WCAG AA(4.5:1)未満
- alt text 欠落、もしくは無意味な alt(画像ファイル名)
出力形式:
- 重要度(Critical/Major/Minor)
- ファイル:行番号
- 問題の要約(1文)
- 修正提案(コード or 1文)
完了報告: 50語以内、件数と最重要1件の要約。

ポイントは「見るべきか迷う境界」に項目を置くこと(例: コントラスト、alt text)。曖昧な観点(「使いやすさ」)は避けてチェックできる項目に落とす。

Exercise 2
応用

自分のPRを3観点で並列レビューする

実際に自分のローカル変更を3観点で並列レビューさせ、結果をmergeまで通します。

  1. 手元のブランチで git diff main...HEAD > /tmp/my-review.diff
  2. §3のセキュリティ/パフォーマンス/可読性の3プロンプトを準備
  3. Claude Codeに1メッセージで3 Agent並列起動を依頼
  4. 結果をCritical→Major→Minor順に並べ替え、重複統合
  5. 最終レビューコメントを自分のPR本文に貼り付けられる形で出力
模範解答を見る

想定される出力形(メインAgentのmerge後):

# レビュー結果 (3観点並列・18件)

## Critical (2件)
- [Security] src/api/login.ts:42
  入力バリデーション前のDBクエリ → SQLi可能性
  → prepared statement に変更
- [Security] .env.example:3
  本物っぽいAPIキー形式がコミットされている
  → ダミー値に置換

## Major (5件)
- [Performance] src/list.tsx:18 N+1 クエリ
- [Performance] src/list.tsx:24 useEffect依存配列欠落
- [Readability] src/auth.ts:120 関数45行、責務混在
- ...

## Minor (11件)
- [Readability] 命名: tmp → ...
- ...

省察: 直列で同じことをやると15-25分。並列で5分で終わった。最も価値があったのは「重複指摘が1箇所に集中したこと」 = そこが真の問題だった、というシグナル。
