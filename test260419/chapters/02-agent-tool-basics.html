Agentツールの基本

subagent_type・description・prompt・isolation パラメータを押さえる

所要時間: 30分 Phase: Basics 難易度: 入門
この章のゴール

Ch01でSubagentの概念を掴んだ今、Agentツールの具体的な呼び出し方を身体で覚えます。6つのパラメータ(subagent_type・description・prompt・isolation・model・run_in_background)の役割と書き分け、戻り値の扱い、よくある書き忘れの落とし穴まで、この章の末尾には「自分で1つ Agent を動かせる」状態を目指します。

2026年1月時点の注記

本章の記述は2026年1月時点の Claude Code / Claude Agent SDK 仕様を基にしています。Agent 関連 API は活発に更新されるため、パラメータ名やモデル名の最新は公式ドキュメントを必ず確認してください。

Agent tool の基本構文 ― まずは最小例

Claude Code での Subagent 起動は、内部的には Agent という名前のツール呼び出しとして実行されます。モデルはユーザーのメッセージを読み、「ここは Subagent に任せるべき」と判断したら自動で Agent ツールを tool_use ブロックで呼び出します。我々が意識するのは「どう指示を書けば良い Subagent 呼び出しが生成されるか」という上位レイヤーですが、内部構造を知っていると指示の精度が桁違いに上がるので、まず1回だけ覗いておきましょう。

最小の Agent 呼び出しは次のような JSON 構造になります。

{
  "type": "tool_use",
  "name": "Agent",
  "input": {
    "subagent_type": "Explore",
    "description": "auth middleware 調査",
    "prompt": "このリポジトリの `src/middleware/` 以下から認証関連ミドルウェアを列挙し、各ファイルの責務を1行で要約してください。ファイルパスは絶対パスで。"
  }
}

必須は subagent_type / description / prompt の3つだけ。他は任意です。「これだけ?」と思うかもしれませんが、その通りで、パラメータの少なさ=自由度の高さ。3行で書けるから気軽に使える、そして prompt 次第でどうにでも化ける、という設計思想です。

Agent 呼び出しの6パラメータ俯瞰図
3必須 + 3任意。最初は上段だけ覚えれば動く。

Claude Code の CLI で普通に Agent を呼ぶ時、ユーザーが直接 JSON を書くことはありません。モデルが自動生成します。ですが「こういうフィールドが埋められる」と知っておくと、日本語で指示する時も「Explore で auth 系を調査して」のようにsubagent_type と prompt を意識した言い回しができて、望むような呼び出しを引き出しやすくなります。

6つのパラメータを1つずつ

各パラメータの役割と注意点を表で一覧してから、よく迷うものだけ深掘りします。

パラメータ必須型役割よくある失敗
subagent_type必須string起動する Subagent の種類。Explore / general-purpose などtypo / 存在しない type 指定
description必須stringUI 表示用の3-5単語の短いラベル長すぎる / prompt と混同
prompt必須stringSubagent への実際の指示文(長め可)短すぎ / 背景情報なし
isolation任意string作業空間の分離モード(worktree 等)不要なのに指定 / 目的との不整合
model任意string使用モデル(Opus/Sonnet/Haiku)存在しないモデル名
run_in_background任意booleantrue で非同期化、後で結果を取得戻り値の扱いを忘れる

subagent_type ― どの Subagent を呼ぶか

一番よく使うのは Explore(リサーチ・調査特化、編集権限なし)と general-purpose(コード編集含むフルパワー)の2択。他にも Plan や code-reviewer 等、環境によってユーザーが定義した専門 Subagent が追加されている場合もあります。詳しくは Ch03 で扱います。

description ― 極短いラベル

UI で「今どの Subagent が走っているか」を表示するためだけに使われるラベル。3-5単語(短い英語)が理想。ここに長文を書く人が多いですが、無意味。prompt に詳細を書きます。

Bad

description が長すぎる

"description": "Explore the auth middleware files in the src directory and summarize each of them in one line..."

これは prompt に書くべき内容。description の枠に入れても UI が崩れるだけで Subagent の挙動は変わりません。

Good

description は短く

"description": "auth middleware 調査"

UI に表示された時に一目で「何が走っているか」がわかる長さ。日本語でもOK、3-5単語で。

prompt ― 本体となる指示文

Agent 呼び出しの9割はここで決まると言っても過言ではありません。Subagent は呼び出し元の会話履歴を見ないので、prompt だけで必要な情報・文脈・期待する成果物形式をすべて伝える必要があります(self-containedな指示)。詳しくは Ch05 で扱いますが、最低限以下4要素を含めるのが鉄則です。

  • 目的 — 何を調べる/作るのか。1文で明示。
  • 背景/コンテキスト — 関連ファイルパス、参照仕様、制約条件。
  • 期待する成果物の形式 — 「マークダウン表で」「1行要約×5件」等。
  • 完了条件/検証基準 — 「全ファイル網羅」「テスト合格」等。

isolation モード ― 分離の段階を選ぶ

isolation は Subagent の作業空間を本流から切り離すかを制御する任意パラメータ。典型値は "worktree"。使わないなら省略でOKで、ほとんどの読み取り専用タスク(Explore)では不要です。

モードファイル編集作業空間ユースケース
(省略)メイン作業ツリーで直接共有読み取り専用 / 単純な編集 / リサーチ
worktree別の git worktree で独立分離複数案を並列試作 / 失敗してもメインを汚さない実験
通常モード vs worktree モードの違い
worktree は「別の作業部屋」を git のパワーで実現する仕組み。
もっと詳しく: worktree が効くのはどんな時か

git worktreeは1つのリポジトリから複数の作業ツリーを同時に開く機能。Subagent がそこで編集しても他の Subagent・メイン側に干渉しません。

worktree が特に効くシーン:

  • 同じ機能に対して3案並列試作→後で比較して採用する案を決める
  • 「壊れるかもしれない」リスキーな変更を安全に試す
  • 並列 Subagent がそれぞれ独立したファイル編集を行う(Ch04 の並列実行で再登場)

不要なケース: 読み取り専用の調査 / 1つだけ走らせる単発編集 / 変更が小さくぶつからないと確信できる場合。指定するとオーバーヘッドが増えるだけです。

今すぐ試す ― isolation の指定方法

Claude Code に「この機能の実装案を3つ、worktree で並列試作して」と依頼すると、モデルが自動で3つの Agent 呼び出しを isolation="worktree" 付きで発行します。手で JSON を書く必要はありません。

// モデルが自動生成する呼び出しの模式図
{
  "name": "Agent",
  "input": {
    "subagent_type": "general-purpose",
    "description": "案A 試作",
    "prompt": "src/auth.ts にJWT実装を追加してください。方針A(自前実装)で。",
    "isolation": "worktree"
  }
}
// 同じメッセージ内でもう2つ並列呼び出し(案B: ライブラリ使用 / 案C: 外部サービス)

model と run_in_background ― コスト/速度の調整弁

残る2つの任意パラメータはコスト・速度・待ち方の調整弁です。

model ― モデルをSubagentごとに変える

呼び出し元が Opus でも、Subagent は Sonnet や Haiku で動かせるのがポイント。軽いタスク(ファイル列挙・grep・短いまとめ)を Haiku/Sonnet に逃がせばコストが1桁安くなります。重要な判断は Opus のまま、足回りの作業だけ安いモデルへ、の使い分けが実用的。

モデル別のコスト比(1Mトークン相対)

run_in_background ― 投げて先に進む

run_in_background: true を指定すると Agent がバックグラウンドで走り続け、結果を待たずにメインの会話は次へ進めます。後で別のツールで結果取得。長時間タスク(ビルド・広範な grep・大量リサーチ)で有効。

通常モード

同期実行(デフォルト)

  • Agent 完了まで待つ
  • メインの処理はブロック
  • 1分前後のタスク向き
  • 結果は即戻り値で手に入る
background

run_in_background: true

  • Agent は裏で動き続ける
  • メインは即次のステップへ
  • ビルド・長期リサーチ向き
  • 後で結果を取りに行く必要あり
使い分けの指針

1分以内に返ってくる見込みなら通常モード。5分以上かかりそう / 結果を見るのは後でいいなら background。両者を混在させて「重い1本は background、軽い5本は同期並列」という構成も実用的です。

戻り値の扱い ― サマリ、生データ、そして保存

Agent 呼び出しの戻り値は、Subagent が最終メッセージとして出力したテキストが1本返ってくる仕組みです。途中の思考プロセスやツール使用ログは返りません。つまり「Subagent から呼び出し元が受け取れるのは、最後のまとめ1本だけ」。ここを理解していないと、「10分もかけて走らせたのに要点が戻ってこない」という悲劇になります。

Agent 実行フローと戻り値
sequenceDiagram
  participant M as Main Agent
  participant S as Subagent
  participant T as Tools(Read/Grep...)
  M->>S: Agent(subagent_type, prompt)
  S->>T: Read file
  T-->>S: 内容
  S->>T: Grep pattern
  T-->>S: 結果
  S->>S: 中間推論(戻らない)
  S-->>M: 最終メッセージ(サマリ)
  M->>M: サマリを基に次の判断
途中のツール利用・思考は全て Subagent 内で閉じ、呼び出し元には最終テキストだけ返る。

サマリ vs 生データ ― どっちで受け取るか

prompt で指示すれば戻り値の形は自由にコントロールできます。

  • サマリ(要約)受け取り — 「3行でまとめて」「結論だけ」と指示。メインが判断材料として使うケース。
  • 生データ受け取り — 「該当ファイルの中身を全文」「JSON形式で構造化して」と指示。後続処理で使うケース。
  • ファイルに書き出し — 「`/tmp/result.md` に書いて、戻りではパスだけ返して」と指示。大量結果を戻り値でなくファイル経由で渡すテクニック。
もっと詳しく: 大きな結果はファイル経由が鉄則

Subagent の戻り値はテキスト1本で、トークン上限があります。100ファイル分のレビュー結果のような大きな成果物を戻り値に詰めるのは無理。そこで定番の指示が:

...結果は `/tmp/review-{観点名}.md` に書き出してください。
戻りメッセージでは
  - 成果物のパス
  - 発見した重要度high の issue 数
  - 1行サマリ
の3点だけを返してください。

これで Subagent は大きな成果をファイルに残し、呼び出し元には要点だけを渡せます。メインはパスを受け取って Read で必要部分だけ読みに行く、という流れ。Ch06/Ch07 の実践章で繰り返し使うパターンです。

戻り値は次にメインAgent が読むコンテキストそのもの。長く詰めるほどメイン側のトークン消費が増えるので、「要点だけ上げてもらう・詳細はファイルに」が省エネで最もスケールします。この設計はコンテキスト保護の観点でも Subagent を使う本来の意義と合致します。

プロンプト例 ― そのままコピペOK

Explore Subagent を起動して、このリポジトリの構造調査をお願いします。

# 目的
このプロジェクトのエントリーポイントと主要なディレクトリ構成を把握する。

# 背景
私はこのリポジトリを初めて触ります。まだファイル配置を知りません。

# 調査してほしいこと
1. package.json(または Cargo.toml / go.mod 等)の main/scripts を確認
2. src/ 配下の主要ディレクトリを列挙、各ディレクトリの責務を1行で
3. README/docs があれば概要部分を読んで「このプロジェクトは何?」を3行で

# 期待する成果物の形式
マークダウンで以下3セクション:
- ## エントリーポイント
- ## ディレクトリ構造(表形式、5-10行)
- ## プロジェクト概要(3行)

# 完了条件
上記3セクションが埋まっていること。不明な部分は「不明」と正直に書いてOK。
この指示文のポイント

「目的・背景・調査内容・成果物形式・完了条件」の5要素がきれいに揃っているのが肝。Claude Code のモデルはこの構造をそのまま Agent の prompt に流し込むので、Subagent 側で迷いません。逆に「調査して」の1行だけだと、Subagent は何を・どこまで・どう返せばいいか判断できず空振りします。

実行中に画面で起こること

  1. メインの Claude Code が Agent ツール呼び出しを発行(UI に「auth middleware 調査」等の description が表示される)
  2. Subagent が独立コンテキストで Read / Grep / Glob を連発
  3. 中間ログは UI 上で開閉可能(気になる時だけ詳細を覗く)
  4. 最終メッセージが呼び出し元に戻る → メインが受け取って次のステップへ

よくある書き忘れ ― 5つの地雷

ここまでの内容を試すと、ほぼ確実にどこかでつまずきます。初心者が連続でハマるパターンを5つだけ先に潰しておきましょう。

書き忘れ上位5件(失敗頻度の体感値)
  1. description を長く書きすぎ ― UI ラベルなのに prompt 本文を入れてしまう。短い3-5単語に。
  2. prompt に背景情報がない ― Subagent は会話履歴を見ないのに、「上のファイル」「さっきの方針」等の参照で書く。必ず self-contained に。
  3. 期待する成果物の形式を書き忘れ ― 「調べて」とだけ頼むと、箇条書き・文章・表・JSON のどれが来るか運次第。「表形式で」と指定する。
  4. worktree を無駄に指定 ― 読み取りだけの Explore に worktree は無意味(書かない)。逆に並列編集で worktree を付け忘れて衝突するのも頻出。
  5. background にして結果取りに行くのを忘れる ― 投げっぱなしで放置→セッション終了時に未取得で消える、の悲劇。background の後は必ず「結果確認」のステップを計画に入れる。
MK流チェックリスト

Agent を1本投げる前に唱える3行の呪文: 「description 5単語以内? prompt に4要素全部入った? 戻り値の形、指定した?」。これだけで8割の事故を防げます。

自分のリポジトリで Explore を1回動かす

手元の任意のリポジトリ(小さなOSSでもOK)で Claude Code を起動し、上のハンズオンのプロンプトをそのままコピペして実行。戻ってきたサマリを読んで以下を確認してください。

  1. 期待通り「## エントリーポイント」「## ディレクトリ構造」「## プロジェクト概要」の3セクションが揃っているか
  2. UI 上で Subagent の description(英語3-5単語)が何と表示されたか
  3. 戻ってきた文章のトークン量がメインのコンテキストを圧迫していないか(長すぎるなら「3行で」「表は5行以内」を指示して縮める)
模範解答を見る

確認ポイントと典型的な戻り値:

## エントリーポイント
- package.json の `main`: dist/index.js
- scripts: `npm run dev` で起動

## ディレクトリ構造
| dir        | 責務                          |
|------------|-------------------------------|
| src/api    | HTTPエンドポイント定義         |
| src/db     | Prisma モデル / マイグレーション |
| src/utils  | 共通ヘルパー                   |

## プロジェクト概要
このリポジトリはECサイトのBFF層で、フロントエンドと決済APIを仲介する。
Node.js(Express)+ TypeScript + Prisma(PostgreSQL)。
テストは vitest、CIは GitHub Actions。

観察ポイント:

  • description は「repo structure scan」のような英語短句が多い → 期待通り
  • 戻り値が300〜600トークン程度で収まっていれば上出来。1000超なら「3行で」等の制約を追加
  • 不明点を「不明」と正直に書いてくれている = self-contained 指示が効いている証拠

うまくいかなかった場合の典型原因は「prompt の完了条件が曖昧」。「この3セクションが埋まれば終わり」とゴールを明示すると精度が上がります。

Exercise 2
応用

悪い Agent 呼び出しを直す

以下は「良くない」Agent 呼び出しの例です。5つの地雷のうちいくつが踏まれているかを特定し、改善版を書いてください。

{
  "subagent_type": "Explore",
  "description": "このリポジトリのauth周りのファイルを全部読んで中身をまとめてください。できれば危険そうな箇所も教えてほしいです。",
  "prompt": "上記の通り調べて。",
  "isolation": "worktree",
  "run_in_background": true
}
  1. 何が間違っているかを地雷リスト(本章の5つ)と対応させる
  2. description・prompt を書き直し、不要な任意パラメータを削除
  3. 改善版 JSON を提示
模範解答を見る

踏んでいる地雷(5つ中4つ):

  • ① description が長すぎ(prompt の内容が入っている)
  • ② prompt が「上記の通り」と参照している(self-contained 違反、Subagent は description も会話履歴も見ない)
  • ③ 成果物の形式指定なし(「まとめて」だけ)
  • ④ Explore(読み取り専用)に worktree は無意味
  • ⑤ run_in_background: true だが後続で結果を取る計画が示されていない(呼び出し側の問題だがリスク)

改善版:

{
  "subagent_type": "Explore",
  "description": "auth 脆弱性調査",
  "prompt": "このリポジトリの `src/auth/` 配下の全ファイルを読み、以下をマークダウン形式で報告してください。\n\n# 目的\n認証ロジックの全体像把握と潜在的セキュリティリスクの洗い出し。\n\n# 調査項目\n1. auth ディレクトリ配下のファイル一覧(絶対パス)\n2. 各ファイルの責務を1行で\n3. 気になるセキュリティ観点(平文保存・弱いハッシュ・タイミング攻撃等)を list で\n\n# 期待する成果物\n## ファイル一覧(表)\n## 責務サマリ(箇条書き)\n## リスク候補(箇条書き、見つからなければ『該当なし』)\n\n# 完了条件\n上記3セクションが埋まっていること。ファイル読み取りのみ、編集禁止。"
}

変更点まとめ:

  • description を「auth 脆弱性調査」の3単語に圧縮
  • prompt を self-contained に書き直し、4要素+完了条件を明示
  • isolation(worktree)を削除 ― Explore は読み取り専用なので不要
  • run_in_background を削除 ― この規模なら同期実行で1分以内に返る

これで 「3パラメータだけのシンプルかつ確実に動く呼び出し」 に整います。シンプルさこそ安定性。
