仮説思考と論点思考

アウトプットから逆算する思考法。網羅的に分析するのではなく、最初に「結論候補」を仮置きして高速で検証する、コンサル中級以上の必須スキル。60分で「速く深く」考える型を身につけます。

所要時間: 60分 Phase: Advanced 難易度: ★★★

1. 仮説思考とは

仮説思考は、「最初に答えの当たりをつけてから、その当たりを検証する」 というアプローチ。網羅的に情報を集めてから結論を出す「網羅思考」の対極にあります。

網羅思考 vs 仮説思考 — 時間配分の違い
情報収集に時間をかけすぎず、仮説を立ててから「検証に必要な情報だけ」を集めるのが速さの源泉。

仮説思考の最大のメリットは スピード。コンサル業界で「仮説思考なしには案件が回らない」と言われるのは、限られた時間で結論を出す必要があるからです。

2. 良い仮説の3条件

  • 具体的 — 「○○すれば××になる」と言い切れる。曖昧な仮説は検証できない。
  • 検証可能 — Yes/No か数値で答えられる。「組織風土が悪い」より「離職理由Top3に上司不満が含まれる」。
  • 反証可能 — 否定される条件が明確。「絶対そうだ」と思える仮説は実は使えない(検証する意味がない)。
悪い仮説良い仮説
市場は厳しい市場成長率は前年比 -2% で2年連続マイナス
競合に負けている主要競合A社のシェアが過去1年で15%→25%に拡大
業務が非効率営業1人あたり週12時間が「資料作成」に消費されている

3. 仮説検証ループ

仮説検証の高速ループ
graph LR
  Start([問題・論点]) --> H["仮説立案"]
  H --> P["最小検証計画"]
  P --> V["検証実行"]
  V --> J{"仮説支持?"}
  J -- Yes --> N["次の仮説 / 詳細化"]
  J -- No --> R["仮説修正"]
  R --> H
  N --> Final([結論])
「修正前提」がコツ。1サイクル数時間〜数日で回す。

1サイクルを高速で回すコツは 「最小検証」。本格調査の前に、社内の有識者1人に5分聞く・既存データを30分眺める、で仮説の方向性は7割確認できます。

4. 論点思考とは

論点思考は仮説思考と双子の概念。「与えられた問い」をそのまま解くのではなく、本当に解くべき問いを設定し直す思考法です。

論点思考の核心

多くのプロジェクトは「問いの設定ミス」で失敗する。「価格を下げるべきか」を解いているうちは答えが出ないが、「なぜシェアが下がっているか」に問いを変えると一気に進む。

浅い論点

「広告予算をどこに振るべきか?」

  • 戦術レベルの問い
  • 大局を変えない
  • 答えても短期的
深い論点

「当社の主要顧客像はそもそも変わったのではないか?」

  • 戦略レベルの問い
  • 大局を変える
  • 答えると長期的に効く

5. 論点を見抜く4視点

「これは本当に解くべき論点か?」を判断する視点が4つあります。

視点問いの例外れていると...
意思決定者の視座誰の何の判断を助ける論点?誰も判断に使わない論点
時間軸の適切さ今、答える価値がある?3年早い or 半年遅い
因果の根本性症状ではなく原因に届く?対症療法に終わる
独自性自社が答える意味がある?業界共通の答えで終わる

「症状」と「原因」を区別する

症状を追うほど結論が遠くなる

同じ時間を投じても、症状を追うか原因に届くかで結果は4-5倍違う。論点設定の差はここに出ます。

6. 仮説思考と論点思考の連携

仮説思考と論点思考は 相補関係 にあります。論点思考で「解くべき問い」を設定し、仮説思考で「その答えの当たり」をつける。

ピラミッド化
もっと詳しく: コンサルが「20分で資料骨格を作る」秘訣

コンサル熟練者は、新案件のキックオフから20分後には資料の骨格(空のスライド10-15枚)を書いています。これが可能なのは:

  1. 論点: クライアントの課題を5分で「本当に解くべき1つの論点」に絞る
  2. 仮説: その論点への答えを「過去案件・業界知識」から仮置きする
  3. 骨格: 仮説を支えるキーメッセージ3本をピラミッド化し、空スライドのタイトルにする

残りの数週間は 「骨格を埋めるための検証作業」。これが「アウトプットドリブン」と呼ばれる進め方です。

7. 仮説思考の罠

  • 確証バイアス — 自分の仮説に合う事実だけ集めてしまう。意識的に反例を探す。
  • 仮説への固執 — 仮説が外れているのに変えられない。仮説は「修正前提」と心得る。
  • 過去経験への過信 — 「前のプロジェクトでこうだった」が必ずしも正しくない。新しい状況には新しい仮説。
最大の罠

「仮説が当たっていた!」と喜ぶのは罠。仮説思考は「速く検証して進む」ためのツールであり、「最初から正解を当てる」ためのものではない。修正回数の多さは恥ではなく、検証ループが回っている証拠。

優れたコンサルタントは、仮説を立てる速さより、仮説を捨てる速さで決まる。 — 内田和成
Exercise 1
入門

悪い仮説を「良い仮説」に書き換える

次の仮説をそれぞれ「具体・検証可能・反証可能」な仮説に書き換えてください。

  1. 「最近の若手は意欲が低い」
  2. 「商品力が弱いから売れない」
  3. 「Web集客は限界が来た」
模範解答を見る
  1. 「入社3年以内の社員のエンゲージメントスコアが、5年前の同期と比べて20pt以上低下している」(検証: スコア比較)
  2. 「直近12ヶ月の失注理由Top3に『機能不足』が30%以上含まれる」(検証: 失注分析)
  3. 「Web経由のリード獲得CPAが、前年同月比で50%以上悪化している」(検証: マーケログ)

共通するコツは 数値・期間・対象を具体化する こと。これにより仮説が検証可能になります。

Exercise 2
応用

1時間で論点 → 仮説 → 検証 → 結論を回す

あなたの今の業務課題を1つ選び、次のフローを実際に1時間で1サイクル回してください。

  1. 10分: 論点設定 — 「今、本当に解くべき問い」を1つに絞る
  2. 10分: 仮説立案 — その論点への答えを2-3個書く
  3. 30分: 最小検証 — 既存データを見る/同僚に5分質問/過去資料確認
  4. 10分: 結論 — 仮説を支持/修正/棄却し、現時点での結論を1文で書く

1サイクル後、「修正したか/しなかったか」を振り返る(修正している方が健全)。

模範解答を見る

例(マーケチームの「広告ROI悪化」を1時間で考える):

10分: 論点設定
  当初の問い: 「広告予算をどう振り分けるか?」
  → 修正: 「そもそも当社の主要顧客像は変わったのではないか?」

10分: 仮説立案
  H1: 主要顧客が30代→40代にシフトしている
  H2: 購入決定要因が「価格」→「品質」に変化
  H3: 既存広告メッセージが時代と合わなくなった

30分: 最小検証
  - 過去3年の購入者年齢データ確認 → H1: 38歳から43歳に変化(支持)
  - NPS自由記述5件読み返し → H2: 「品質」「長持ち」コメント増(部分支持)
  - 競合A社の最新広告チェック → H3: ファミリー層・品質訴求にシフト(支持)

10分: 結論
  論点「主要顧客像の変化」は YES。
  打ち手は「予算配分」より先に「広告メッセージ刷新」が優先。
  次のサイクルで広告クリエイティブ案を3本作って検証する。

当初の論点(広告予算配分)を解いていたら永遠に答えが出なかった。論点設定で10分使うことで、検証30分が劇的に効く例です。
