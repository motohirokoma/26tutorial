イシューツリー

「答えるべき問い」を分解する技術。ロジックツリー(Ch.04)の進化形で、仮説検証を回す現場での主役ツール。60分で「論点設計力」を養います。

所要時間: 60分 Phase: Core 難易度: ★★★

1. ロジックツリーとイシューツリーの違い

ロジックツリーは「対象の分解」、イシューツリーは「論点(イシュー)の分解」。同じ木構造でも、節点が 「対象」ではなく「問い」 になります。

ロジックツリー vs イシューツリー
同じテーマでも、節点を「名詞」にすればロジックツリー、「疑問文」にすればイシューツリー。

イシューツリーは 「分解 + 答え出し」がセット なので、検証可能な仮説を引き出しやすい。コンサルが現場で書くツリーは、ほぼ全部イシューツリー型です。

2. メインイシューとサブイシュー

イシューツリーの頂点を メインイシュー、その分解を サブイシュー と呼びます。

メイン / サブイシューの実例
graph TD
  M["メイン: 当社は新興市場Xに参入すべきか?"]
  M --> S1["サブ1: 市場は魅力的か?"]
  M --> S2["サブ2: 当社に勝ち筋はあるか?"]
  M --> S3["サブ3: 投資が回収できる規模か?"]
  S1 --> S1a["市場規模・成長率は?"]
  S1 --> S1b["顧客ニーズは存在?"]
  S2 --> S2a["技術優位性?"]
  S2 --> S2b["参入障壁を超えられる?"]
  S3 --> S3a["初期投資額は?"]
  S3 --> S3b["回収期間は?"]
サブイシュー全部に答えると、メインイシューに自動的に答えられる。これが理想形。

サブイシューを設計する原則は 「サブイシューに全部 Yes / 全部 No が出たら、メインも自動的に決まる」 こと。ここをクリアすると、検証作業がすごく楽になります。

3. 仮説検証ループの設計

イシューツリーは「描いて終わり」ではなく 「描いて検証してまた描き直す」 ループで使います。

1
論点提示
メインイシュー設定
2
仮説生成
サブイシューの暫定回答
3
検証計画
必要データ・分析を特定
4
検証実行
分析・インタビュー
5
ツリー更新
結果をフィードバック

多くの場合、Step 4 の検証で「想定と違う」が発覚し、ツリーを修正することになります。修正できるツリー設計 がイシューツリーの神髄です。

仮説検証ループの相互作用
sequenceDiagram
  participant T as ツリー
  participant H as 仮説
  participant D as データ
  T->>H: サブイシューに仮説をぶつける
  H->>D: 仮説検証のためデータ収集
  D-->>H: 結果を返す
  H-->>T: 仮説が支持/反証
  T->>T: 必要に応じツリー更新
  Note over T,D: 1ループ目で完璧を目指さない
「仮説→データ→修正」を高速で回す。1サイクル1-2日が理想。

4. 良いイシューの条件 — 「イシュー度」

論点には「答える価値」の高低があります。元コンサルタントの安宅和人氏は 「イシュー度 × 解の質」 を提唱しました。イシュー度が低い問いに答えても、誰の役にも立ちません。

イシュー度 × 解の質 マトリクス
右上を狙え。左下や左上で時間を使うのは「犬の道」。

イシュー度を上げる3つの観点

  • 意思決定インパクト — この問いに答えると、何が変わる?規模は?
  • タイミング — 今、答えを出す価値がある?(半年後では遅い、3年早い、はNG)
  • 独自性 — 自分(自社)が答える意味がある?他人がもう答えている問いか?

5. サブイシュー設計の3パターン

サブイシューに分解する型は3つに分けられます。

パターン典型形使う場面
Yes/No 直列「Aは○? → はいなら次へ → Bは○?」判断ロジックが明確な時
条件並列「Aが○ かつ Bが○ かつ Cが○ なら ...」複数条件で結論が決まる時
選択肢比較「案A vs 案B vs 案C を◯軸で比較」意思決定の選択肢比較

例: 新興市場参入の判断

条件並列型のイシューツリー
graph TD
  M["新興市場Xに参入すべき?"]
  M --> Q1{"市場は魅力的?"}
  M --> Q2{"勝ち筋はある?"}
  M --> Q3{"回収できる?"}
  Q1 -- Yes --> Q1Y["継続"]
  Q1 -- No --> Stop1["参入NG"]
  Q2 -- Yes --> Q2Y["継続"]
  Q2 -- No --> Stop2["参入NG"]
  Q3 -- Yes --> Go["参入する"]
  Q3 -- No --> Stop3["参入NG"]
「全 Yes でのみ参入」のロジック。1つでも No なら結論は決まる。
サブイシューの順序

「明らかに崩れそうな仮説」から先に検証する。最初の Q1 で No なら、Q2/Q3 を調べる必要がない。これが クリティカルパス 思考。

6. サブイシュー検証の進捗管理

サブイシューごとに「仮説」「検証状況」「結論」を管理すると、進捗が一目で見えます。

サブイシュー仮説検証状況暫定結論
市場は魅力的?10兆円規模、成長率15%市場調査完了Yes
勝ち筋はある?当社技術が3社のみ保有競合分析中Tentative Yes
回収できる?5年で投資回収可未着手—

サブイシュー検証進捗

サブ1: 市場魅力100%
サブ2: 勝ち筋60%
サブ3: 投資回収10%

サブ3の検証進捗が遅れているため要注意。サブ1は完了。サブ2は順調。

もっと詳しく: 「ストーリーライン」とのつながり

イシューツリーで設計したサブイシュー群は、最終的なプレゼン資料の 「ストーリーライン」 になります。サブイシューの順序 = スライドの順序、と覚えておくと、資料化が劇的に楽になります。

逆に言えば、イシューツリーが弱い = 資料が弱い。コンサル熟練者ほど、イシュー設計に多くの時間を投じます。

Exercise 1
入門

「ロジックツリー」を「イシューツリー」に変換

次のロジックツリーをイシューツリーに変換してください(各ノードを疑問文に)。

採用が進まない
  ├─ 母集団が少ない
  │   ├─ 求人媒体
  │   └─ リファラル
  └─ 歩留まりが悪い
      ├─ 書類選考
      └─ 面接
模範解答を見る 
メイン: 採用を加速するにはどうすべき?
  ├─ サブ1: 母集団は十分?
  │   ├─ 求人媒体は適切?
  │   └─ リファラルは活発?
  └─ サブ2: 歩留まりは適正?
      ├─ 書類選考の通過率は?
      └─ 面接の受諾率は?

各ノードを Yes/No か数値で答えられる形にすると、自動的にイシューツリーになります。「母集団」→「母集団は十分?」のように、名詞→疑問文への変換がコツ。

Exercise 2
応用

あなたのプロジェクトでイシューツリーを設計

あなたが今関わっているプロジェクト・課題で、メインイシューを1つ立て、サブイシューを3〜5個設計してください。

  1. メインイシュー(疑問文で1文):
  2. サブイシュー A:
  3. サブイシュー B:
  4. サブイシュー C:
  5. 各サブの「現時点の仮説」と「検証アクション」を書く
  6. クリティカルパス: 最初に検証すべきサブはどれ?(理由も)
模範解答を見る

例(SaaSプロダクトの新機能リリース):

メイン: 新機能Xをリリースして良いか?

サブA: ユーザーニーズは存在するか?
  仮説: 既存顧客の30%が要望
  検証: 顧客アンケート + 利用ログ分析
  優先度: ★★★ (これが No なら全終了)

サブB: 技術的に実現可能か?
  仮説: 既存基盤で2ヶ月で開発可能
  検証: 開発リードへのヒアリング
  優先度: ★★

サブC: 既存機能に悪影響を与えないか?
  仮説: 干渉する機能なし
  検証: 影響範囲調査
  優先度: ★

クリティカルパス: A → B → C
理由: A が No なら全部止まる。B も技術リスクで止まる可能性。
C は対策可能なので最後でOK。

クリティカルパスは 「壊れたら全部崩れる順」 で並べると、最短で結論に届けます。これがコンサル流の検証設計。
