ビジネスドキュメント作成

社内文書・提案書・報告書。AIが下書きしても、"構造を設計する目" が無ければ通りません。

所要時間: 30min Phase: Core 難易度: 初級

社内文書には3タイプしかない

「文書が書けない」と悩む新卒はたくさんいます。でも実は、会社で書く文書の9割は 3タイプ に集約されるんです。まずはこの地図を持つことから始めましょう!

タイプ読み手構造長さ目安
報告書上司・関係者事実 + 所感(気づき)A4で1〜2枚
企画書意思決定者提案 + 根拠(数字/背景)A4で3〜10枚
議事録参加者 + 不参加者合意事項 + 論点 + ToDoA4で1枚

この3タイプは、読み手も目的も違うので、書き方のルールも違います。「ひとつのフォーマット」で全部書こうとする新卒が、一番苦労してしまうのです。

3ドキュメントのメタファ — "誰に何を渡すか"
読み手と目的が違う = 構造が違う。この当たり前を最初に押さえましょう。
この章のゴール

AIに文書の下書きを任せつつも、構造を設計する目 を身につけます。読み手が「1行で理解できる」文書を、最初の3ヶ月 で書けるようになりましょう!

構造化の原則 — タイトル・見出し・段落

どの文書でも共通する「3階層の構造」があります。タイトル → 見出し → 段落。この順番で設計しましょう。

文書の基本レイアウト(色分け)
ピンク濃 = タイトル、ピンク中 = H2、ピンク薄 = H3、灰 = 段落。階層が見える文書は、読み手の負担が半減します。

3つのルールで十分です。

  • タイトルは「動詞 + 結論」 — 「〜について」で止めないでください。「〜を提案する / 〜を報告する / 〜を実施したい」まで書き切りましょう
  • 見出し階層は2〜3階層まで — それ以上深くすると、人間は読めません。深くなりそうなら、章を分けましょう
  • 段落は1トピック、5〜7行 — 1段落に2つの話題を混ぜないでください。長い段落は分割しましょう
悪い見出し

意味が取れない

  • 「新ツールについて」
  • 「先日の件」
  • 「現状と課題」
  • 「まとめ」

読んでも何が書いてあるか想像できません

良い見出し

読まずに分かる

  • 「Slack導入で週3時間削減できる」
  • 「4/10 A社打ち合わせで3件の宿題発生」
  • 「月次工数がここ半年で1.4倍に増加」
  • 「結論: 今期中に試験導入を提案」

見出しだけで文書の骨格が読めます

もっと詳しく: なぜタイトルに動詞を入れるのか

忙しい上司は、メールの一覧を眺めて タイトルだけで読む/後回しを判定します。「〜について」と書かれた文書は、中身を開くまで判断できないので、後回しされる確率が高いのです。

一方、「Slack導入を提案します」とあれば、上司は 「読んで意思決定する必要がある文書だ」 と一瞬で理解できます。タイトルは読み手への シグナル なんです。

具体例です。

  • × 「競合調査について」 → ○ 「競合A社との価格差を15%以内に抑える案(ご相談)」
  • × 「来期の方針」 → ○ 「来期: 新規顧客獲得を3軸で進める計画(ご承認依頼)」
  • × 「4月活動報告」 → ○ 「4月活動報告: 計画比105%達成、5月は○○に注力」

タイトルの末尾に「(ご相談)」「(ご報告)」「(ご承認依頼)」と アクション種別 を添えるだけでも、読み手の処理速度は上がります。

結論ファースト — ドキュメント版

Ch08で学んだ 結論ファースト は、文書ではさらに重要になります。なぜなら 読み手はまず冒頭の3行しか読まない からです。

その「冒頭3行」が エグゼクティブサマリ です。英語圏では「Executive Summary」と呼ばれ、欧米企業ではどんな文書にも必ず付けられます。日本企業でも徐々に標準化しつつあります。

サマリの3行構成

① 何を提案 / 報告するか(1行) → ② その背景・根拠(1行) → ③ 求めるアクション(1行)。この3行で、読み手は読む・読まない・誰に回すかを判断できます。

読み手が文書を読む動線は、実は ほぼ決まっています。

sequenceDiagram participant R as 読み手(上司) participant D as 文書 R->>D: 冒頭3行(サマリ)を読む D-->>R: 結論を返す Note over R: ここで判断:読む? 読まない? 誰かに回す? R->>D: 必要なら見出しだけざっと読む D-->>R: 構造を返す Note over R: 骨格を把握 R->>D: 気になった章だけ詳細を読む D-->>R: 根拠・数字を返す Note over R: 意思決定 or フィードバック

この動線に合わせて書きましょう。つまり、冒頭 = サマリ、中盤 = 見出しで骨格、後半 = 詳細と根拠 です。順番を逆にしてはいけません。

1
サマリ
冒頭3行で結論と依頼事項
2
結論
詳細な主張をもう一段具体化
3
根拠
数字・事例・背景の説明
4
補足
別案・想定質問・参考資料
もっと詳しく: 結論ファーストを崩していい例外

原則は「結論ファースト」ですが、以下の場合は例外的に 背景 → 結論 の順にする方が読みやすくなります。

  • 社内の歴史が絡む提案 — 「過去このプロジェクトはA案で失敗した。そのため今回はB案を提案します」のように、背景を先に置かないと結論が理解できないケース
  • 反対意見が予想される提案 — いきなり結論を出すと感情的反発を招いてしまいます。「現状の課題」から入って合意を作る方が通りやすいです
  • 謝罪・トラブル報告 — 「事実 → 原因 → 対応」の順が基本です。いきなり「再発防止策はこうです」と書くと、事実を隠しているように見えてしまいます

ただし例外はあくまで例外です。迷ったら結論ファーストでいきましょう!

AI下書きと、自分の仕上げ

AIは文書作成の最強の相棒です。ただし 任せていい領域と、絶対に自分でやる領域 を分けないと、Ch01で扱ったAさん現象に陥ってしまいます。

文書作成工数の理想的な配分

AIは下書きで40%を担います。しかし、数字・固有名詞・意思決定 は人間がやりましょう。この3つは、間違えたら信用を失う領域だからです。

  • 数字はAIに書かせない — 売上・工数・日付・人数は自分で確認しましょう。AIが生成した数字は ハルシネーション の温床です
  • 固有名詞は自分で入れる — 人名・会社名・製品名。AIに任せると誤記が起きます。後で差し込みましょう
  • 意思決定の「言い切り」は人間 — 「〜を提案します」「〜と判断します」。これは責任が伴うので、必ず自分の言葉でお願いします
  • 必ず1回は声に出して読む — AIが書いた文章は 「音として不自然」 なことがあります。読み上げると違和感が見つかります。これは新卒が守るべき鉄則です
  • 提出前の最終チェック5項目 — タイトル/サマリ/数字/固有名詞/依頼事項。この5点を確認する癖をつけましょう
公式ガイド

ビジネス文書の基本構成と標準書式については、経済産業省・総務省がまとめた公的な整理があります。ルールを押さえたい新卒は、まず目を通しておく価値がありますよ! 経済産業省 公式サイト →

もっと詳しく: AIに渡すプロンプトの型

新卒がAIに「報告書を書いて」と丸投げするので、出てくる文章が使い物にならないのです。プロンプトには 最低限4つの情報 を入れましょう。

  1. 読み手: 「上司(部長)に提出する」「取引先のA社に送る」
  2. 目的: 「承認を得たい」「意思決定を求めたい」「状況を共有したい」
  3. 材料: 事実・数字・背景(自分で持っている情報を全部渡す)
  4. 形式: 「A4で1枚」「見出しは3つまで」「エグゼクティブサマリ3行つき」

この4つを渡すと、AIの下書きの質が格段に上がります。「AIが使えない」と嘆く新卒の9割は、この渡し方を知らないだけなのです。

新卒が書く「最初の1枚」

入社後3ヶ月、あなたが書くことになる文書はほぼ決まっています。いきなり企画書を書かされることは少ないんです。まずは 日報・週報・引き継ぎ書。この3つのフォーマットを早く盗みましょう!

入社 〜 1ヶ月
業務日報
その日やったこと、気づき、翌日の予定。1日10分で書きましょう。先輩のフォーマットを真似するだけでOKです
1 〜 2ヶ月
週報
週の成果・課題・来週のアクション。数字を入れ始めましょう。ここで"結論ファースト"を練習します
2 〜 3ヶ月
議事録
会議で任されます。合意・論点・ToDoの3点セット。Ch07の内容を実践しましょう
3 〜 6ヶ月
引き継ぎ書・簡単な報告書
先輩のテンプレを下敷きに、自分の文書を作り始めましょう
鉄則: 「書く」より「フォーマットを盗む」

最初の3ヶ月は、新しいフォーマットを発明しない でください。先輩・上司の過去文書を見て、その型を 型 ごと真似しましょう。これが一番速く、一番失敗が少ないです。型を盗んでから、半年後に自分の色を出せばOKです!

文書は、あなたの代わりに会議室に残り続けます。あなたが退社した後も、あなたの文書は上司のデスクで読まれているのです。つまり文書は、あなたの分身。分身が雑だと、あなたも雑に見えてしまいます。分身が誠実だと、あなたも誠実に見えますよ!

冗長な報告書を "結論ファースト" に再構成する

以下は、新卒Aさんが書いた報告書の冒頭です。結論ファースト + エグゼクティブサマリ3行 の形に書き直してください。

件名: 先日の件について

お疲れ様です。4月10日に実施しましたA社との打ち合わせについて
報告させていただきます。当日はA社の山田部長と佐藤課長がご参加されて、
弊社からは私と先輩の田中さんが出席しました。打ち合わせは14時から16時
まで行われ、来期のシステム更新の件で幅広く議論しました。話題は多岐に
わたり、予算の話や、スケジュールの話、それから追加機能の話などが出ました。
結果、予算を現状の1.2倍程度に増やすことをA社側が検討してくださることに
なり、また追加機能3件についてはこちらで見積もりを出すことになりました。
以上、取り急ぎご報告まで。

書き直し例です。

件名: 4/10 A社打ち合わせ報告 — 予算1.2倍の可能性、追加機能3件の見積もり依頼あり(ご確認)

【エグゼクティブサマリ】
・A社が来期予算を現状の1.2倍へ増額する方向で検討に入った
・追加機能3件について、弊社側で見積もり提出が必要(納期: 4/末まで)
・次回打ち合わせは5/15を想定、それまでに資料準備が必要

【詳細】
1. 打ち合わせ概要
  日時: 4/10(水) 14:00-16:00
  参加: A社 山田部長・佐藤課長 / 弊社 田中・(自分)

2. 決定事項
  2-1. 予算: 現状比1.2倍での検討をA社が持ち帰り
  2-2. 追加機能: 3件(○○・○○・○○)の見積もりを弊社で作成

3. ToDo
  ・見積もり作成(担当: 自分、期限: 4/末)
  ・次回打ち合わせ調整(担当: 田中、期限: 4/20)

改善ポイント:

  • 件名に「動詞 + 結論 + アクション種別」を入れました
  • エグゼクティブサマリ3行で、上司が冒頭だけで全体を理解できます
  • 本文は見出しで骨格を見せ、数字・固有名詞・ToDoを明示しました
  • 冗長な「〜について」「〜させていただきます」を削りました
Exercise 2
応用

「新しい社内ツール導入」の企画書の骨組みを設計する

あなたは入社1年目。部内で「新しいチャットツールを導入したい」と提案することになりました。企画書の タイトル + 見出し階層(H2/H3)だけ を設計してください。本文は書かなくてOK、骨組みだけで大丈夫です。

  1. タイトルは「動詞 + 結論」で
  2. H2は4〜5個
  3. 必要ならH3を1〜2階層分入れる
  4. 読み手は「部長(決裁者)」を想定
模範解答を見る

骨組み例です(あくまで1例)。

【タイトル】
Slack導入で週3時間の工数削減を提案します(ご承認依頼)

【エグゼクティブサマリ】(3行)
・現状のメール中心の連絡で、1人あたり週3時間の無駄が発生
・Slack導入で年間コスト60万 / 削減効果は年間360万(ROI 6倍)
・3ヶ月の試験導入を提案、判断は3ヶ月後の実績で

1. 現状の課題
  1-1. メール確認にかかる時間の実測
  1-2. 意思決定の遅延事例

2. 提案 — Slack試験導入
  2-1. 対象範囲(まず○○部 10名)
  2-2. 導入スケジュール(3ヶ月)

3. 投資対効果
  3-1. コスト試算(ライセンス + 教育)
  3-2. 削減効果試算
  3-3. ROI比較

4. リスクと対応策
  4-1. 情報漏洩リスク
  4-2. 既存ツールとの並走期間

5. 依頼事項
  5-1. 試験導入のご承認
  5-2. 初期予算 60万円の確保

ポイント:

  • 見出しだけで文書全体の物語が読めます(現状 → 提案 → 効果 → リスク → 依頼)
  • H2は5個、H3は最大3階層で止めています
  • 最後に「依頼事項」を独立したH2にすることで、決裁者のアクションが明確になります
  • 数字(週3時間 / 60万 / 360万 / 6倍)をサマリに先出しして、読む価値を冒頭で示しています
