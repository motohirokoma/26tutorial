ロジカルシンキング入門

AIに賢い質問を投げられる人は、人にも賢い質問を投げられる。ロジカル思考はAI時代の"母国語"。

所要時間: 60min Phase: Core 難易度: 初級

ロジカルシンキングは "伝わる技術"

「ロジカルシンキング」と聞くと、冷たい・理屈っぽい・頭でっかちな印象を持つ方が多いです。でも、先に結論を言わせてください。ロジカルシンキングとは、相手が一度で理解できるように伝える技術のことです。冷たさではなく、優しさの形と言ってもいいでしょう!

「え、優しさ?」と思った方、正しい反応です。あなたが2回聞き返さなくていいように、上司が頭を使わなくて済むように、取引先が3秒で意思決定できるように — そのために 伝える側が構造を整えます。これがロジカルシンキングの本質です。

曖昧な伝え方 vs 構造化された伝え方
ロジカル = 冷たい、ではありません。ロジカル = 相手の脳を楽にする、です。

さらに、AI時代においてこのスキルは 単なるビジネスマナー以上の意味 を持ちます。ChatGPTに賢い質問を投げられる人は、上司にも賢い質問を投げられます。プロンプトを書く行為は、ロジカルシンキングそのものです。つまり —

この章のゴール

PREP・MECE・ピラミッド・So What / Why Soという 4つの基本道具 を使って、「何を話すか」より「どう構造化するか」で差をつけられるようになります。60分の中で、AI時代の"母国語"を一気にインストールしましょう!

結論ファースト / PREP法

最初の武器は、議論の余地なく PREP法 です。Point(結論) → Reason(理由) → Example(例) → Point(結論)の順で話します。たったこれだけで、伝達効率は2倍以上になります!

日本の学校教育では「起承転結」や「序論・本論・結論」を叩き込まれます。これは 読み物としては美しい のですが、ビジネス会話としては最悪です。結論が最後に来る = 相手は最後まで宙ぶらりん、に置かれてしまいます。忙しい上司ほど、最初の30秒で "使える情報か?" を判断しています。

最初と最後に結論を置く「サンドイッチ構造」。相手の記憶に残りやすいです。
Before

悪い伝え方(起承転結)

「今週、A社とB社の両方にヒアリングしまして、A社は納期に余裕があるものの価格が高くて、B社は価格が安いのですが実績がなく、それで色々悩んだ結果、A社でいこうかなと…」

聞き手の脳内: 「で、結論は?途中の説明、頭に残らない…」

After

PREP式

(P) A社を選ぶべきです。
(R) 実績とリスクのバランスが最適だからです。
(E) B社は価格が2割安いものの、類似案件の実績がゼロ。A社は3件の類似実績あり。
(P) よってA社で進めたく、承認をお願いします。

聞き手の脳内: 「OK、承認」で終了。30秒。

もっと詳しく: PREPが効く心理的理由

PREPが有効なのは、人間の認知特性と一致しているからです:

  • 初頭効果 — 最初に聞いた情報は記憶に強く残ります。結論を最初に置けば、その後の説明が"その結論の補強"として整理されます
  • 親近効果 — 最後に聞いた情報も記憶に残ります。だからPで締めて繰り返します
  • 認知負荷の軽減 — 結論を知っていれば、相手は「この情報を結論と紐付けるだけ」で済みます。逆に結論を知らない状態で聞くと、相手は全情報を記憶保持しないといけません

つまりPREPは、相手の脳内メモリを節約する 思いやり の型でもあります。

MECE — モレなくダブりなく

次の武器は MECE(ミーシー)です。「Mutually Exclusive and Collectively Exhaustive」の頭文字で、モレなく・ダブりなく分類するという意味です。マッキンゼーのコンサル界隈で生まれ、今や世界中のビジネスパーソンの共通言語になっています。

例えば「売上が落ちた」と言われた時。新卒は「頑張ります!」と答えがちですが、論理的な人は まず分解します。売上 = 客数 × 単価。客数が減ったのか、単価が下がったのか。両方か、片方か。どちらを直すかで打ち手は180度違います。

ロジックツリーの例です。左から右へ、徐々に具体的に分解していきます。
MECEな分類 vs そうでない分類
MECEは "見栄え" ではなく "分析可能性" のためにやります。
覚えておくMECEの切り口

迷ったら以下を使いましょう: (1)時系列(過去/現在/未来)、(2)要素分解(売上=客数×単価)、(3)二項対立(内/外、質/量)、(4)プロセス(認知→検討→購入→継続)。切り口を先に決めるとMECEになりやすいです!

ピラミッド原則

バーバラ・ミントが提唱した ピラミッド原則 は、PREPを大規模化したものだと思えばOKです。頂点に1つの結論、その下に3つの根拠、さらにその下に各根拠を支える具体データ — こういう三角形で主張を構造化します。

ピラミッド原則の構造
頂点=主張、中段=根拠(3つ前後)、底辺=具体。この三層が揃うと納得度が跳ね上がります!

プレゼン、報告、メール、提案書、企画書 — ビジネスで「相手を動かす」文書は、ほぼすべてこのピラミッド構造を持っています。つまり、ピラミッド原則を理解する = ビジネス文書全般のOSを理解する と言ってもいいでしょう。

階層役割粒度文字数の目安
頂点(L1)結論・メインメッセージ抽象度: 最高1〜2文(40〜80字)
中段(L2)結論を支える根拠(3つ前後)抽象度: 中各1〜2文(計150字前後)
底辺(L3)根拠を支える具体・データ・事例抽象度: 最低各2〜4文(計400字〜)
もっと詳しく: なぜ根拠は "3つ" が多いのか

ピラミッドの中段、つまり「根拠」が3つに揃うケースが多いのには理由があります:

  1. マジックナンバー3 — 人間が一度に記憶できる項目数は4±1と言われます。3つは覚えやすく、2つだと物足りず、4つ以上になると忘れます
  2. 弁証法的安定 — "A・B・C"の三点支持は、二点(対立に見える)や四点以上(散漫)より構造的に安定します
  3. MECEを作りやすい — 2つだと対立軸に偏り、5つだと重複が出やすいです。3つは自然にMECEになります

ただし、本当に2つしかない時に無理やり3つ目をひねり出すのは逆効果です。根拠の数より、MECEの質 が先です!

So What? / Why So? — 思考の往復運動

ここまでの道具(PREP・MECE・ピラミッド)は"並べる"ための型でした。次は"深める"ための型、So What? と Why So? です。

やり方はシンプルです。事実やデータを見て「So What?(だから何?)」と自問する → 一段上の示唆に上がります。主張を見て「Why So?(なぜそう言える?)」と自問する → 一段下の根拠に降ります。この2つを往復すると、ピラミッドの各階層がカチッと整います!

So What? / Why So? の往復
データ→示唆(So What?)、示唆→データ(Why So?)。この往復運動が論理の筋肉を作ります。
実務での使い方

会議中に「それはSo Whatで言うと?」と問うと、議論が一気に深まります。逆に誰かが壮大な主張をしたら「Why Soですか?」と問えば、根拠の薄さを暴けます。この2語は、ロジカル界の"魔法の呪文"です!

もっと詳しく: 演繹と帰納 — 論理の2つの方向

So What / Why Soの裏には、古典的な2つの論理形式があります:

  • 演繹(えんえき) — 一般ルール → 個別事例。「若者はSNSに敏感」+「うちの客は若者中心」→「うちはSNS対策が必要」。Why Soに近いです
  • 帰納(きのう) — 個別事例を集める → 一般原則。「A社で売れた」「B社で売れた」「C社で売れた」→「この商品は中小企業に受ける」。So Whatに近いです

名前を覚える必要はありません。ただ、ビジネスの主張には 必ずこの2方向のどちらかの論理形式がある と知っておくと、相手の主張の弱点を見抜きやすくなります。

AI時代のロジカルシンキング

ここまで学んだ4つの道具は、AI時代においてさらに重みを増します。なぜでしょうか。ロジカルシンキングは、AIに賢く仕事をさせる言語そのものだからです。

AIに「売上を上げて」と聞いても、まともな答えは返ってきません。でも「当社の売上低下要因をMECEで分解し、各要因にWhy Soで3段階まで掘ってほしい」と聞けば、A4数ページ分の分析が数秒で返ってきます。プロンプトを書く行為 = ピラミッドを組む行為 だと気づけた瞬間、あなたのAI活用力は一段跳ね上がります!

ロジカルシンキング5軸: Before(学生)→ After(1年目でこの章を習得後)
  • AIへの指示は結論と条件を先に — 「○○について、以下の3観点で比較して、300字で結論を先に書いて」。PREPをプロンプトに埋め込みます
  • AIの出力をMECEでチェック — AIは"それらしく"答えを出します。あなたが「これ、漏れてない?重複してない?」と検証する側に回りましょう
  • So What? で鵜呑みを防ぐ — AIが出したデータに「だから何?」と問いかける習慣を!AIは答えは出せても、示唆は出しにくいです
ここが分岐点

AIが普及すればするほど、論理の"検証者" としての人間の価値が上がります。AIが書いた企画書・AIが出した分析・AIが下した判断 — これらを鵜呑みにしない構造的な目を持つ人が、職場で最も重宝されます。その目を作るのが、この章の4道具です!

論理とは、相手の脳を楽にする優しさの形です。そしてAIとは、論理が伝わる相手の、もう一つの呼び名です。あなたがロジカルに話せるようになる = 人間にもAIにも、同じ深さで仕事を任せられるようになる、ということです。
Exercise 1
入門

曖昧な報告をPREP法で書き直す

以下は、新卒Aさんの上司への報告メッセージです。これをPREP法で200字以内に書き直してください。

「お疲れさまです。なんか売上が落ちた気がしてて、
いろいろあるっぽいです。B社さんの件とか、広告の件とか、
あとサイトも重くなってるって話もあって、なんか全部絡んで
ますよね。どうしたらいいですかね、みたいな感じなんですけど…」

制約: (1) Point / Reason / Example / Pointの構造を明示します、(2) 数字か固有名詞を最低1つ入れます、(3) 最後に "判断を仰ぐ事項" を書きましょう。

模範解答を見る

模範解答例です(約190字):

【結論(P)】 先月の売上が前月比-12%で、主因は客数減と見られます。対応方針をご相談させてください。

【理由(R)】 単価は横ばい、客数が-15%だったためです。

【例(E)】 特にB社経由の新規が先月から半減(30→14件)。広告費削減とサイト表示速度低下が影響している可能性があります。

【結論再提示(P)】 広告費復活 or サイト改修、どちらを優先するか本日中にご判断いただきたいです。

ポイント: 元の文は「気がする」「〜っぽい」だらけで、上司は判断できません。PREP化すると (a)数字、(b)原因仮説、(c)判断を仰ぐ事項 が浮かび、30秒で意思決定が可能になります!

Exercise 2
応用

「社内システムをAIツールに置き換えるべきか」をロジックツリーでMECE分解する

あなたの会社が現在使っている社内システム(例: 経費精算・勤怠・CRMなど、何でも良いです)をAIツールに置き換えるかどうか、論理的に判断するためのロジックツリーを書いてください。

制約: (1) 頂点は Yes / No を決める論点、(2) L2は MECE な判断軸を3つ前後、(3) L3は各軸の具体チェック項目を各2つ以上、(4) 最終的に自分の結論(Yes or No)と一言の理由を書きましょう。

模範解答を見る

回答例です(経費精算システムを題材):

【論点】 経費精算システムをAIツールに置き換えるべきか?

├─ (L2-1) 効果: 置き換えで何が改善するか
│   ├─ 入力工数の削減率(現行30分 → AI化後?)
│   └─ 月次締め作業の短縮(現行3日 → AI化後?)
│
├─ (L2-2) リスク: 置き換えで失うもの
│   ├─ 既存データの移行コスト(履歴5年分)
│   ├─ 監査対応(AIの判定根拠の説明可能性)
│   └─ セキュリティ(経費データは個人情報を含むか)
│
└─ (L2-3) コスト: 置き換えの総額
    ├─ 初期費用(導入・移行・研修)
    └─ ランニング費用 vs 現行費用の差分

【結論】 Noを推奨。理由: 効果は明確だが、監査対応の
説明可能性リスクが現時点で未解決のため。
1年後にAIの監査対応事例が出揃った段階で再検討。

ポイント: (a) 効果・リスク・コストはMECEの代表的な3軸です、(b) L3に数字が入ると急に"使える"ツリーになります、(c) 結論に「なぜ今Noなのか・いつYesにするのか」まで書けると上司の質問を先回りできます!
