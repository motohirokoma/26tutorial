1. 課題ニュース記事(架空)

2026年4月15日 — Tech Crunch JP(架空)

米 AcmeAI、700億パラメータの新LLM「Zephyr-70B」を公開
Claude 3.5 Sonnet に匹敵する性能、オープンウェイトで提供

米 AcmeAI は4月14日、新しい大規模言語モデル「Zephyr-70B」を発表した。パラメータ数は700億、Decoder-only Transformer構造を採用し、Llama 3 と同様にオープンウェイトで Hugging Face 上に公開された。

事前学習には 15Tトークン(うちコード3T、日本語0.5T、英語10T、その他1.5T)を使用。Chinchilla 比で見ても データ量が極めて多い 設計となっており、推論コストが低い小型モデルとして実用性能を最大化する戦略を取った。

Fine-tuning には SFT 12万件 + DPO を採用。RLHF ではなく DPO を選んだ理由として、AcmeAI は「学習の安定性とコスト面での優位性」を挙げている。

ベンチマーク結果は MMLU 84.2、HumanEval 76.5、GSM8K 91.0。同規模の Llama 3 70B Instruct を全項目で上回り、Claude 3.5 Sonnet と同等水準とされる。コンテキスト長は 128K、Multi-Head Attention の代替として Grouped Query Attention (GQA) を採用、推論時のメモリ消費を1/4に削減した。

ライセンスは独自の「AcmeAI Open License v1」で、月間アクティブユーザ7億未満の商用利用は無料。ハルシネーション対策として、推論時に RAG パイプライン を組み合わせる事例が紹介された。

4. 模範解答

Q1: 「70B / 15T」の意味

70B = 70 Billion = 700億パラメータ。モデルが持つ重み(数値)の総数。「賢さの容量」の目安。

15T = 15 Trillion = 15兆トークン。事前学習に使ったテキストデータの量。

Chinchilla 法則の最適比は P : D = 1 : 20。70B × 20 = 1.4T が「最適データ量」。15T はその 約11倍 で、明らかに 過剰。これは Llama 3 と同じ戦略で、「推論時に小さく安く、訓練時に大量データで性能を絞り出す」設計思想。Chinchilla 最適より過剰なデータを与えると、収益逓減はあるが性能はまだ上がる、というトレードオフを取っている。

Q2: Decoder-only Transformer

Decoder-only = 入力を 左から右に 一方向で処理し、各位置で次のトークンを予測する Transformer 構造。因果マスク(Causal Mask)で「自分より後ろのトークン」を見えないようにする。

3種類の比較:

構造特徴代表モデル得意
Decoder-only左→右で生成GPT, Claude, Llama, Zephyr対話・生成・補完
Encoder-only双方向で理解BERT, RoBERTa分類・抽出・検索
Encoder-Decoder入力理解 → 出力生成T5, BART翻訳・要約(古典)

現代の対話型LLMはほぼすべて Decoder-only。生成と理解を 1つの構造 で兼ねられる柔軟性が勝った。

Q3: SFT + DPO とRLHFの代替性

SFT = Supervised Fine-Tuning(教師ありFine-tuning)。事前学習後の base モデルに、「Q→理想A」の人間作成ペア を与えて指示に従えるようにする段階。

DPO = Direct Preference Optimization。RLHFと同じく「人間の好みデータ(2つの回答どっちが良い?)」を使うが、報酬モデルもPPOも要らない。好みデータから直接モデルを最適化する。

なぜRLHFの代替になるか:

  • RLHF = (1) 報酬モデル訓練 + (2) PPOで強化学習。2段構え。安定性が悪く、ハイパラチューニングが難しい
  • DPO = 1段で済む。実装が簡単、訓練が安定、コストが低い
  • 性能的にはほぼ同等(2024-2025年時点での研究結果)

AcmeAI が DPO を選んだのは、現在の 業界トレンド に沿った合理的判断。Llama 3 後期、Mistral, DeepSeek などもDPO系を多用している。

Q4: コンテキスト長128K

単位はトークン。128K = 131,072 トークン。日本語なら約6万文字、英語なら約10万単語(本1冊分)。一度のリクエストで入力できる(およびモデル内部で同時に保持できる)テキストの最大量。

メリット:

  • 長い文書(契約書、技術ドキュメント、小説)を一度に処理できる
  • 長い対話履歴を保持できる(会話の文脈を忘れにくい)
  • RAGで多くの文書を一度に渡せる

デメリット:

  • 計算量がトークン数の 2乗 で増える(Ch04参照)。コストと遅延が大きい
  • 「Lost in the middle」現象 — 中央付近の情報を見落としやすい
  • メモリ消費が大きい。GQA など緩和技術が必要

記事の 「GQAでメモリ消費を1/4に」 はこれを緩和する工夫の一例。

Q5: RAGとハルシネーション対策

RAG = Retrieval-Augmented Generation(検索拡張生成)。仕組み:

  1. ユーザの質問が来る
  2. 関連する文書を 外部DB から検索(Web、社内ドキュメント、技術マニュアルなど)
  3. 取得した文書をプロンプトに 添付 してLLMに渡す
  4. LLMは その文書を根拠に 回答を生成、引用元も付けられる

なぜハルシネーションを減らせるか:

  • LLMの内部知識(訓練データ)に頼らず、その場で取得した一次情報 を使うから
  • 「自分が知らないことを生成で埋める」必要がない → 嘘が出にくい
  • 引用元のURL/ページが付くので、ユーザが 事実検証できる
  • 学習締切以降の 最新情報 も扱える

現代の業務利用LLMはほぼ確実にRAGかWeb検索が組み込まれている。Perplexity, ChatGPT検索, Claude with web, Notion AI などすべてこの系統。
