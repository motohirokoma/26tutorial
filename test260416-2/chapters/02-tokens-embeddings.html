言葉を数字に変える 〜トークンと埋め込み〜

LLMは本質的に「数字を計算する機械」です。言葉はそのままでは食べさせられません。この章では、文章がどうやって数字に変換され、どう「意味を持つ数字」になるかを見ます。記事でよく出る「トークン」「埋め込み(Embedding)」「コンテキスト長 128K」の正体が、ここで全部つながります。

所要時間: 30分 Phase: Basics 難易度: 入門

1. なぜ「数字に変える」必要があるのか

LLMの中身は 巨大な行列の掛け算と足し算の山 です。コンピュータは 0.34 や 1.7 しか計算できません。「猫」も「Hello」も、まずは数字にする必要があります。

そこで2段階の変換が走ります:

2. ステップ1: トークン化

トークン化とは、文章を「LLMが知っている語彙単位」に区切ること。これは 単語そのもの でも 文字 でもなく、その中間の サブワード です。

具体例: 英語の場合

「tokenization」という単語は、英語の GPT 系では多くの場合 token + ization の2つに分かれます。なぜか? 未知の単語にも対応するため です。tokenize も tokenizer も tokenization も、共通の token を再利用できる。

具体例: 日本語の場合

日本語は 1文字 = 1トークン ぐらいの粒度になりがち(欧文に最適化されたBPEだと、漢字1字に2-3トークン消費することも)。だから「日本語はトークンを食う」と言われます。

テキスト大体のトークン数(GPT-4 系)1トークンあたり
"Hello, world!"4≒ 3.5文字
「こんにちは、世界!」10≒ 1文字
"The quick brown fox jumps over the lazy dog"9≒ 単語1個
「素早い茶色の狐が怠惰な犬を飛び越える」23≒ 0.7文字
実用Tips

API の 料金は入力+出力のトークン数 でかかります。日本語は英語の 2-3倍 トークンを食うので、コストが跳ねやすい。長文プロンプトは英語で書く方が安いという「テクニック」が成立する根拠がここです。

もっと詳しく: BPE(Byte-Pair Encoding)って結局何?

BPE は元々データ圧縮アルゴリズムとして1994年に発明されました。「最も頻繁に出る2文字ペア」を1つの新しい記号に置き換える、を繰り返します。

例: 大量の英語コーパスでは th が頻出 → 1トークンに昇格。the も頻出 → 1トークンに昇格。一方 xq はめったに出ない → 別々のままに留まる。

結果として、頻出語は 1トークン、レア語や新語は サブワードに分解 される。これで「未知語ゼロ」「語彙数を抑える」「形態論を捉える」を全部一度に達成できます。

1トークン ≒ 何文字?

主要言語の「1トークンあたりの文字数」(GPT-4 概算)

3. ステップ2: 埋め込み(Embedding)

トークンIDは 342、19 のような 意味のない通し番号。これをそのまま使ってもLLMは賢くなれません。「ID 342 と ID 19 が似てる単語かどうか」が分からないからです。

そこで 埋め込み(Embedding) の登場。各トークンIDを 数百〜数千次元のベクトル に変換します。このベクトル空間では、「意味が似ているトークンは近くに集まる」 という魔法が起きます。

埋め込み空間のイメージ(2次元に投影)
実際は数百〜数千次元。意味の似た単語が「方向」と「位置」で近くに集まる。

有名な不思議: 「王 - 男 + 女 = 女王」

埋め込みベクトルには 意味の引き算ができる という不思議な性質があります。Word2Vec の論文(2013)で示された 有名すぎる例:

vec("King") - vec("Man") + vec("Woman") ≒ vec("Queen") — ベクトル空間上で「性別の差分」を引いて足すと別の単語になる

つまり、ベクトル空間の中に 「性別の方向」「王族度の方向」 が存在する。これが「意味を数字で表現できる」という主張の根拠です。

4. ベクトルの「次元数」とモデルサイズ

埋め込みベクトルが何次元あるかは、モデル設計の 大きな選択。次元が多いほど表現力は上がるが、計算量とメモリも比例して増えます。

モデル埋め込み次元意味するところ
Word2Vec (2013)300当時はこれで十分だった
GPT-2 small768BERT-base と同じ
GPT-3 (175B)12,2881単語に1.2万個の数字
Llama 3 70B8,192多くは数千〜1万次元の範囲

5. コンテキスト長: 128K って結局何?

記事で「コンテキスト長 128K」「2M トークン対応」と書かれているこの数字。これは 「一度に入力できるトークン数」 のこと。

2K
GPT-3 (2020)
128K
GPT-4 Turbo
200K
Claude 3
2M
Gemini 2.0 Pro

128K トークン ≒ 英語の本 約1冊分(日本語だと1/2〜1/3)。長くなるほど、長い文書を一気に読ませて要約させたり、長い対話履歴を覚えさせたりできる。一方で 計算コストはトークン数の2乗 で増えます(Ch04 で扱う Attention の特性)。

  • 感覚として: 1万トークン ≒ 5,000日本語文字 ≒ 短編小説1本
  • 128Kトークン: ≒ 6万日本語文字 ≒ 新書1冊分
  • 2Mトークン: ≒ 100万日本語文字 ≒ 長編小説4-5冊分
  • 注意: 「入力できる」≠「全部活用できる」。途中の情報を見落とす(Lost in the middle)現象が知られている

6. まとめ

この章のキーポイント

① 文章は トークン化 でサブワード単位の ID列 になる
② 各IDは 埋め込み で数百〜数千次元の ベクトル になる
③ 意味の似たトークンはベクトル空間で近くに集まる
④ コンテキスト長 = 1回に入力できるトークン数 = モデルの「視野の広さ」

Exercise 1
入門

OpenAI Tokenizer で自分の文章を「数える」

OpenAI が提供する公式トークナイザビューアで、英語と日本語が 同じ文の長さでも別物 になることを確認します。

  1. https://platform.openai.com/tokenizer を開く
  2. "The quick brown fox jumps over the lazy dog" を入れてトークン数を確認
  3. 「素早い茶色の狐が怠惰な犬を飛び越える」を入れて確認
  4. 2つを比べて、何倍違うか計算する
模範解答を見る

結果(GPT-4o系トークナイザ概算):

  • 英語版: 約 9 トークン
  • 日本語版: 約 17-23 トークン(モデルによる、新しいGPT-4o系は日本語効率が改善されている)

日本語は 2倍以上 トークンを食うことが多いです。これがAPI課金時の 隠れたコスト になります。

応用Tip: GPT-4o の Tokenizer は GPT-3.5 のものより日本語効率が良いです。古いモデルから乗り換えるだけでコストが下がる、という現象も。

Exercise 2
応用

「埋め込みの距離」を擬似的に体感する

埋め込みベクトル空間で「どれくらい近いか」をLLMに直接質問できます。あなたが普段使うLLMに次のプロンプトを投げて、結果を比較してみてください。

  1. 使えるLLM(ChatGPT・Claude・Geminiなど)を1つ開く
  2. こう聞く: 「『京都』に意味的に近い単語を、日本の地名以外で、近い順に5つ挙げて」
  3. 次に「『京都』に近い単語を、抽象的な概念だけで5つ挙げて」
  4. 2つの結果を比較して、なぜ違うかを考察
模範解答を見る

例えば1つ目の結果は「古都」「奈良」「金閣寺」「茶道」「八坂神社」のように 場所・歴史・観光 軸が出ます。2つ目は「伝統」「侘び寂び」「四季」「精神性」「美意識」のように 抽象軸 が出ます。

面白いポイント: 同じ「京都」というトークンの埋め込みベクトルに、これだけ 多面的な「方向」 が含まれている、ということです。LLMはプロンプトの文脈に応じて、この多次元のベクトルから 必要な軸を引き出して 答えを生成しています。

これは 静的な単語埋め込み(Word2Vec)から 文脈依存の埋め込み(Transformer時代のもの)への進化を体感する練習でもあります。Ch04 の Attention で「文脈で意味が変わる」仕組みが分かるとさらに面白いです。
