Hookイベント一覧

Pre/PostToolUse・Stop・UserPromptSubmit・SessionStart 等、発火タイミングと入出力を完全把握

所要時間: 35分 Phase: Basics 難易度: 初級
この章のゴール

Ch01で「Hookとは何か」を掴んだ今、Claude Code が発火してくれるイベントの全種類を1つずつ押さえます。主要な6イベント(PreToolUse / PostToolUse / Stop / UserPromptSubmit / SessionStart / SessionEnd)について、いつ発火するか・何が stdin で渡ってくるか・どう出力するかを体系的に整理。最後に最小の PreToolUse Hook を実際に仕込んで発火を確認します。2026年1月時点の仕様に基づきます。

2026年1月時点の注記

本章の記述は2026年1月時点の Claude Code Hooks 仕様に基づきます。新しいイベント(例: Notification, PreCompact)が追加されたり stdin JSON のフィールドが増える可能性があるため、最新は公式 Hooks ドキュメントで必ず確認してください。

セキュリティ警告

Hook は Claude Code が自動で任意のシェルコマンドを実行する仕組みです。他人の settings.json をコピペする前に、その中で何が実行されるか必ず目を通してください。rm -rf / curl | sh / 秘密情報の echo などが紛れ込んでいても、発火時にあなたのマシンで実行されます。本章でも繰り返し注意します。

主要6イベントの全体像 ― セッション寿命の中での位置取り

Claude Code のセッションは「開始 → ユーザ入力 → モデル応答(ツール連打) → 完了 → 終了」のライフサイクルをぐるぐる回ります。Hook はその各節目に割って入るフックです。まずは全体像を1枚の図で押さえましょう。

6つの主要イベントの発火位置
セッションを横軸にして6イベントを置くと、それぞれの立ち位置が一目で掴める。

特に重要なのが、PreToolUse と PostToolUse は1セッション中に何度でも発火すること。Claude がツールを10回使えば、それぞれ Pre/Post が10回ずつ呼ばれます。一方 SessionStart/SessionEnd はセッションにつき1回、Stop はモデルが応答を完了する度に1回(継続対話なら複数回)発火します。

6つとも覚える必要はありません。最初は PreToolUse / PostToolUse / Stop の3つで9割の用途は賄えます。残り3つは「ユーザ入力の段階で何かしたい」「セッション境界で何かしたい」と思った時に引っ張り出せば十分。

全イベント一覧 ― 発火タイミング・入出力・頻度を1枚で

ここまでの6つに加え、環境により使える補助イベントも含めた完全表を置いておきます。現場で迷ったらここに戻ってきてください。

イベント名 発火タイミング 1セッション頻度 stdin に入るもの ブロック可否 典型ユースケース
PreToolUse ツール実行の直前 ツール回数分(多い) tool_name / tool_input(JSON) 可(exit 2) フォーマット・危険コマンドブロック・Lint前処理
PostToolUse ツール実行の直後 ツール回数分(多い) tool_name / tool_input / tool_response 不可(結果は既に出ている) 書き出し後の自動整形・通知音・ログ記録

2026年1月時点の覚え方

「Pre は阻止、Post は観測、Stop は締め、Session は境界」。この4語で8割迷わなくなります。exit code 2 でブロックできるのは Pre 系(PreToolUse / UserPromptSubmit)だけ、という対称性も合わせて覚えると強力。

発火シーケンス ― 1ターンの中で何が起きるか

表だけでは頭に残りにくいので、「ユーザが1発指示 → Claudeがファイル編集1回 → 完了」という最小ターンをシーケンス図で追いましょう。各イベントの厳密な位置関係が見えます。

1ターンの発火シーケンス(編集タスクの例)
sequenceDiagram
  participant U as User
  participant CC as Claude Code
  participant H as Hook(shell)
  participant M as Model
  participant T as Tool(Edit)

  Note over CC: セッション開始
  CC->>H: SessionStart 発火
  H-->>CC: "git pull 実行" 等

  U->>CC: "auth.ts を修正して"
  CC->>H: UserPromptSubmit 発火
  H-->>CC: "PIIマスク済み prompt"
  CC->>M: 強化された prompt を送信
  M-->>CC: "Edit ツール呼ぶ" 判断

  CC->>H: PreToolUse 発火(tool_name=Edit)
  H-->>CC: exit 0(通過) または exit 2(ブロック)
  CC->>T: Edit を実行
  T-->>CC: 編集結果
  CC->>H: PostToolUse 発火(tool_response 付き)
  H-->>CC: ログ記録完了

  M-->>CC: "応答完了" 判断
  CC->>H: Stop 発火
  H-->>CC: "git diff 表示・Slack通知"
  CC-->>U: 最終応答

  Note over CC: セッション終了
  CC->>H: SessionEnd 発火
1往復の会話で Hook は最低6回呼ばれる。ツール数が増えれば Pre/Post の数も線形に増える。

注目すべきはHook が毎回同期的にブロックすること。Claude Code は Hook の終了を待ってから次に進みます。つまり PreToolUse Hook が5秒かかるコマンドを仕込むと、ツール実行のたびに5秒待たされる。Hook は速くあれが鉄則です(通常は100ms以内、長くても数秒)。

もっと詳しく: なぜ同期ブロックなのか / 非同期にしたい時は?

Hook が同期的なのは「Claude の次のアクションに影響を与えられる」という価値があるから。PreToolUse が遅延なしで走り、exit 2 を返せば、Claude はそのツールを使わずに方針を立て直せる。これを非同期にすると、すでに危険コマンドが実行された後に止めても手遅れです。

どうしても重い処理を Hook でやりたい時の逃し方:

  • バックグラウンド化: nohup heavy-task.sh & で投げて Hook 自体は即時 exit 0
  • 条件早期return: 最初に「対象ファイルか?」を判定して、対象外なら即 exit 0
  • キャッシュ: 同じ引数なら前回結果を使う(jq で stdin ハッシュ化)

実運用では「Hook は判定だけ、重い処理は別ワーカーに投げる」設計が安定します。

入力コンテキスト ― stdin に何が流れてくるか

Hook として起動されたシェルコマンドは、標準入力(stdin)に JSON が流し込まれるという約束になっています。イベントごとに形は違いますが、「hook_event_name」「session_id」「transcript_path」は共通で入ります。残りはイベント固有フィールド。

PreToolUse の stdin 例

{
  "hook_event_name": "PreToolUse",
  "session_id": "a1b2c3d4-....",
  "transcript_path": "/Users/you/.claude/projects/xxx/transcript.jsonl",
  "cwd": "/Users/you/repo",
  "tool_name": "Edit",
  "tool_input": {
    "file_path": "/Users/you/repo/src/auth.ts",
    "old_string": "password",
    "new_string": "passwd"
  }
}

PostToolUse の stdin 例(tool_response が追加)

{
  "hook_event_name": "PostToolUse",
  "session_id": "a1b2c3d4-....",
  "tool_name": "Edit",
  "tool_input": { "file_path": "...", "old_string": "...", "new_string": "..." },
  "tool_response": {
    "success": true,
    "filePath": "/Users/you/repo/src/auth.ts"
  }
}

UserPromptSubmit の stdin 例

{
  "hook_event_name": "UserPromptSubmit",
  "session_id": "a1b2c3d4-....",
  "prompt": "APIキー sk-xxxx を使って叩いて"
}

SessionStart の stdin 例

{
  "hook_event_name": "SessionStart",
  "session_id": "a1b2c3d4-....",
  "source": "startup"
}
stdin を読むのは jq が最速

シェルスクリプトで JSON を捌くなら jq が事実上の標準。tool_name=$(jq -r '.tool_name') で一発取り出し。Python/Node派なら python3 -c 'import sys,json;d=json.load(sys.stdin);...' 等でもOK。Hook内で他のコマンドに渡す値は必ずクォート(空白・改行混入でバグる)。

共通フィールド
hook_event_name / session_id / transcript_path / cwd
Pre/Post 固有
tool_name / tool_input / tool_response(Postのみ)
UserPromptSubmit
prompt(ユーザ入力テキスト)
SessionStart
source(startup / resume / clear)
もっと詳しく: 環境変数 $CLAUDE_PROJECT_DIR も使える

stdin の JSON 以外に、Hook 実行時の環境変数でも情報が降ってきます。代表例が $CLAUDE_PROJECT_DIR(Claude Code がそのプロジェクトのルートと認識しているディレクトリの絶対パス)。settings.json の command 内から $CLAUDE_PROJECT_DIR/scripts/my-hook.sh と書いておけば、どこから Claude Code を起動してもプロジェクト基点でスクリプトが呼べます。

他にも $CLAUDE_HOOK_EVENT(hook_event_name と同値、stdin を読まずに早期分岐したい時に便利)などが利用可能。全リストは公式ドキュメントで確認してください(2026年1月時点で追加中の機能)。

出力方法 ― stderr と exit code でClaudeに話しかける

Hook から Claude Code に結果を返す主要な手段は2つ: exit code と stderr の文字列です。stdout は通常ログ(人間向け)に使い、Claudeの挙動を変える方法は exit code と stderr 経由が基本です。

exit 0

通常終了(通過)

  • Claude は何も気にせず処理続行
  • stdout/stderr は UI/ログに記録
  • ユーザにも Claude にも影響なし
  • 大半の Hook はこれ
exit 2

ブロック(Pre系で有効)

  • Claude に「このツール実行は拒否された」と伝わる
  • stderr の内容が Claude にフィードバックされる
  • Claude は代替案を考え直す
  • 危険コマンド阻止・フォーマット強制
exit 1 他

警告・失敗(ブロックはしない)

  • Claude には直接影響しない
  • ユーザに stderr が見える
  • Hook の内部エラー扱い
  • デバッグ出力に

典型コード: exit 2 でブロック

#!/usr/bin/env bash
# block-rm-rf.sh — Bash ツールで rm -rf / 系を阻止

set -euo pipefail
CMD=$(jq -r '.tool_input.command // ""')

if echo "$CMD" | grep -qE 'rm\s+-rf?\s+/'; then
  echo "[block] 危険コマンド検出: $CMD" >&2
  exit 2   # Claude にブロック通知
fi

exit 0

典型コード: stderr で警告だけ(ブロックしない)

#!/usr/bin/env bash
# warn-large-file.sh — 1MB超のファイル編集時に警告

FILE=$(jq -r '.tool_input.file_path // ""')
[ -z "$FILE" ] && exit 0

SIZE=$(wc -c < "$FILE" 2>/dev/null || echo 0)
if [ "$SIZE" -gt 1048576 ]; then
  echo "[warn] 1MB超のファイルを編集中: $FILE" >&2
fi

exit 0
セキュリティ警告(再掲)

exit 2 でブロックできる=それだけ権限がある、ということ。逆に言えば、信頼できないHookを入れると Claude の挙動を勝手に改変できる。他人のdotfilesから settings.json をコピーする時は、command 欄のスクリプト中身を必ず目視レビューしてください。

イベント→使い道 ― 「何をしたいか」から引ける逆引き

現場で先に決まるのは「何を自動化したいか」の方です。その要求から「どのイベントに仕掛けるか」を即座に選べるよう、逆引きマップを用意しました。

  • 危険コマンドをブロックしたい → PreToolUse(Bashツールを matcher で絞る)+ exit 2
  • Write/Edit後にPrettier で整形したい → PostToolUse(Write|Edit を matcher)
  • 作業完了を通知したい → Stop + macOS osascript / Slack Webhook
  • ユーザ入力のPIIを隠したい → UserPromptSubmit(正規表現置換)
  • セッション冒頭で git pull したい → SessionStart(source=startup 時のみ)
  • 会話履歴をバックアップしたい → PreCompact(圧縮前に transcript_path をコピー)
  • 作業ログを jsonl に蓄積したい → PostToolUse or Stop で append

どのイベントが一番よく使われるか

筆者が運用している複数の settings.json を集計すると、PreToolUse と PostToolUse で全Hookの7-8割を占めます。「ツール実行前後」が最も旨味が大きいから当然の結果。

実運用での Hook イベント採用率(MK集計・2026/1時点)

matcher との組み合わせ ― 「全Edit」か「特定拡張子だけ」か

PreToolUse / PostToolUse は matcher(正規表現)でツール名を絞れます。matcher を書かないと全ツールで発火するので、ログが暴走しがち。目的のツールだけに絞るのが作法です。

matcher 正規表現マッチするツール典型用途
"Edit"Edit のみ既存ファイルの差分編集だけを見張る
"Write|Edit"Write と Editファイル書き込み全般を整形対象に
"Bash"Bashシェル実行を監査・ブロック
".*" または省略全ツール全ツールログ(デバッグ用・本番は重い)
"WebFetch|WebSearch"外部通信ツール外部アクセスの監査

ここまで来ると「matcher の書き方」「イベントとの組み合わせ」でほぼ任意の自動化が設計できます。次の章(Ch03)で settings.json の完全な書式を学べば、自作 Hook を丸ごと書けるようになります。本章ではその一歩手前、最小の Hook を1本だけ動かして発火を目視確認するところまでやりましょう。

ハンズオン ― 最小 PreToolUse Hook を仕込んで発火確認

実際に自分のマシンで動かしてみます。「Edit ツールが呼ばれるたびに、ファイルパスをログに残す」だけの超シンプル Hook。ブロックはせず、ただ発火したことを観測します。

1
設定ファイルを開く
~/.claude/settings.json
2
hooks 節を追加
PreToolUse / Edit
3
Claude Code で Edit を依頼
どんなファイルでもOK
4
ログファイルを確認
/tmp/hook-ch02.log に追記されてる?

Step 2: 貼り付ける settings.json(そのままコピペ可)

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit",
        "hooks": [
          {
            "type": "command",
            "command": "jq -r '\"[\\(.hook_event_name)] \\(.tool_name) -> \\(.tool_input.file_path // \"(no file)\")\"' >> /tmp/hook-ch02.log; exit 0"
          }
        ]
      }
    ]
  }
}

このスクリプトは stdin の JSON を jq で整形して /tmp/hook-ch02.log に append し、exit 0 で終了します。Claude の挙動には一切影響せず、純粋に観測のみ。

Step 3: 試す指示(Claude Code 内で)

何でもいいので手元のテキストファイル(例: /tmp/test.txt)を
作って、Edit ツールで「hello」から「hi」に書き換えてください。

Step 4: ログを見る

$ cat /tmp/hook-ch02.log
[PreToolUse] Edit -> /tmp/test.txt
発火しなかった時のチェックリスト

(1) settings.json の JSON 構文が有効か(jq . < ~/.claude/settings.json で検証)
(2) Claude Code を再起動したか(settings.json は起動時に読むので編集後は再起動)
(3) matcher の綴りは Edit で合っているか(大文字小文字厳密)
(4) jq が PATH にあるか(which jq)
詳しい診断は Ch07(トラブルシュート)で扱います。

もっと詳しく: Bash の <,>,& エスケープに注意

JSON 内で command に長いシェル式を書く時、>>(append)や <(入力リダイレクト)はJSON文字列としてそのまま書けますが、HTMLドキュメントにコピペして貼るときは HTML 実体参照(&gt; 等)に誤変換されがち。エディタに貼る時は必ずプレーンテキスト貼り付けを使い、> < & が生のまま入っているか確認してください。

また、シェルの特殊文字($, `, \, ")を command に書く時は JSON エスケープ(\\ や \")が必要。長くなるなら外部スクリプトファイルに切り出して "command": "/path/to/my-hook.sh"と参照する形の方が圧倒的にラク。これはCh03で詳しく扱います。

これで「Hook イベントは本当に発火する」という事実を自分の目で確認できました。抽象的な仕組みが身体感覚に落ちた瞬間から、Hookは急に身近なツールになります。この感覚を持ったままCh03に進めば settings.json の文法はスッと入ってきます。

入門

ハンズオンを自分の環境で動かして観察

本章のハンズオン(PreToolUse + Edit のログ記録)をそのままコピーして動かし、以下を確認してください。

  1. /tmp/hook-ch02.log に1行追記されているか
  2. matcher を "Edit" から "Write|Edit" に変更し、Writeツール実行時にも発火するか試す
  3. さらに matcher を "Bash" に変更して、Claude に ls を実行してもらうと tool_input.file_path は (no file) になる。この挙動の理由を考察してください(ヒント: tool_input のキーはツールごとに違う)
模範解答を見る

期待される観察結果:

  • 1: ログには [PreToolUse] Edit -> /tmp/test.txt の形で1行。Edit が複数回呼ばれれば複数行
  • 2: matcher を "Write|Edit" にすると、両方のツールで発火。Write の場合は tool_input.file_path が Edit と同じように入るので -> の後ろにパスが出る
  • 3: Bash ツールの tool_input は { "command": "ls", "description": "..." } という形で file_path フィールドを持たない。jq の .tool_input.file_path // "(no file)" は null 時のデフォルト値なので (no file) になる。Bash をログに残すなら .tool_input.command を参照する方が適切

改善したBash用ログHook:

{
  "matcher": "Bash",
  "hooks": [{
    "type": "command",
    "command": "jq -r '\"[\\(.hook_event_name)] Bash: \\(.tool_input.command)\"' >> /tmp/hook-ch02-bash.log; exit 0"
  }]
}

このようにツールごとに tool_input の構造が違うことを理解すると、matcher と jq パスの選び方が自然に身につきます。

Exercise 2
応用

PreToolUse で「rm -rf /」をブロックする Hook を書く

本章の「典型コード: exit 2 でブロック」を参考に、Bash ツールで rm -rf / 系のコマンドを実行しようとした時だけ阻止する Hook を仕上げてください。

  1. スクリプトファイルを ~/.claude/hooks/block-rm-rf.sh として作成し、実行権限(chmod +x)を付与
  2. ~/.claude/settings.json の PreToolUse に matcher "Bash" で登録
  3. Claude Code で「rm -rf /tmp/test を実行してください」と言った時と、「rm -rf / を実行してください」と言った時の挙動の違いを観察
  4. (応用) /tmp/ 配下はブロック対象から外す除外ルールを追加
模範解答を見る

Step 1: ~/.claude/hooks/block-rm-rf.sh

#!/usr/bin/env bash
# block-rm-rf.sh — rm -rf / 系を検出してブロック
# 2026/01 時点の PreToolUse stdin 仕様に基づく

set -euo pipefail

CMD=$(jq -r '.tool_input.command // ""')

# /tmp 配下なら許可(応用)
if echo "$CMD" | grep -qE 'rm\s+-[rRf]+\s+/tmp(/|$)'; then
  exit 0
fi

# ルート・ホーム・重要ディレクトリ直下を狙う rm -rf を阻止
if echo "$CMD" | grep -qE 'rm\s+-[rRf]+\s+(/|~|\$HOME|/etc|/usr|/var)(\s|$|/)'; then
  echo "[BLOCK] 危険な rm -rf を阻止しました: $CMD" >&2
  echo "必要なら削除対象を具体パスに絞り込んでください。" >&2
  exit 2
fi

exit 0

Step 2: ~/.claude/settings.json

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          { "type": "command", "command": "~/.claude/hooks/block-rm-rf.sh" }
        ]
      }
    ]
  }
}

Step 3: 実行して観察

  • rm -rf /tmp/test → 許可ルールに該当、exit 0 で通過、Claude はそのまま実行
  • rm -rf / → ブロックルールに該当、exit 2 で阻止。stderr の内容が Claude にフィードバックされ、Claude が「代替方法を考えます」と路線変更してくれる

セキュリティ上の注意:

  • このHookは完璧ではありません。rm -rf /home/... やシンボリックリンク、環境変数展開(rm -rf "$UNSET_VAR/")等は別途検出が必要
  • 真の防御は Claude Code の permissions 設定(deny リスト)と組み合わせること。Hook だけで全てを守らない
  • 他人のHookをコピペする時は command 欄のスクリプト内容を必ず自分の目で確認(本章で繰り返し警告している通り)

この演習が終わると、「PreToolUse + matcher + exit 2」の三点セットでClaude Codeの挙動を自在に制御できる実感が掴めます。Ch03では settings.json の構造を、Ch04では10種のレシピを学んで、この力を現場で使える武器に仕上げていきます。
