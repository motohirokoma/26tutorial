最終演習：頻出作業3つをHook化

手で繰り返している作業から3つ選んで自動化 — 7章の総仕上げ

所要時間: 30分 Phase: Practice 難易度: 応用

7章の学びを「自分の手作業」に落とす

ここまで、Hook の概念・イベント・settings.json 構文・レシピ集・Lint/Prettier 自動化・Stop サマリ・トラブルとセキュリティを学んできました。知識は使わなければ腐ります。この最終章では 「手で繰り返している作業を3つ Hook 化し、効果を数値で測定できている」状態まで持っていきます。

目標は 「1週間以内に3つの Hook を自分の settings.json に常駐させ、Before/After の手間削減を記録している」こと。抽出 → settings.json 実装 → 動作検証 → 測定までを1枚の設計書にまとめるのが本演習のゴールです。

Tips

完璧な3つを選ぶ必要はありません。まず5分で10個書き出し、その中から「頻度が高い × 定型化できる × 副作用が小さい」の3軸で3つに絞ります。完璧より「まず動かす」が正解です。

段階ヒント (行き詰まったら開く)

いきなり全部開くと「答え写し」になります。まず自分で1つ書いてみて、詰まったヒントだけ開いてください。5つのヒントは 作業の見つけ方 → イベント選定 → matcher 設計 → テスト方法 → セキュリティ確認 の順に対応しています。

もっと詳しく: ヒント1 — 繰り返し作業の見つけ方 (Ch01 / Ch04)

Hook 化の良い候補には5つのシグナルがあります:

  • 週3回以上発生する — たまにしかないなら手でやったほうが学習コストが安い
  • コマンドが1行で書ける — prettier --write / pytest -q / osascript -e ... など
  • Claude に毎回頼んでいる — 「フォーマットして」「テストして」と何度も指示している作業
  • 入出力が決まっている — 入力はツール名とパス、出力は成功/失敗だけで良い
  • 失敗しても切り戻せる — 書き換えた後に Git で差分を戻せる範囲

逆に悪い候補は: ①対話的な判断が必要 / ②パラメータが毎回変わる / ③実行に30秒以上かかる(Claude の応答が詰まる) / ④失敗が高コスト(本番DB への書き込み等)。これらは Slash Command / CI/CD に任せるのが正解です。

Tips

候補が10個出ない人は、直近5セッションの会話ログを見返すのがお勧め。「これフォーマットして」「テスト通して」「コミットメッセージ考えて」のような同じ依頼が必ず何個か見つかります。それが Hook 化の種です。

もっと詳しく: ヒント2 — イベントの選び方 (Ch02)

主要5イベントの使い分け(Ch02の復習):

イベント発火タイミングブロック可否典型的な用途
PreToolUseツール実行の直前exit 2 で可危険コマンドの拒否 / Write前の整形
PostToolUseツール実行の直後不可書き出し後の Prettier / Lint / テスト
StopClaudeが応答を終えた時不可通知 / git status / jsonl ログ
UserPromptSubmitユーザ入力送信直後exit 2 で可PII マスク / 禁止語検知
SessionStartセッション開始時不可git pull / 環境変数ロード / 初期通知

迷ったら 「書き出しの前か後か」で切り分け。前に整形したい → PreToolUse(Claude に整形済みを見せる)、後で検証したい → PostToolUse(書き出された実体を検証)。作業終了時の通知 → Stop の3択でほぼ回ります。

もっと詳しく: ヒント3 — matcher の設計(Ch03)

matcher は tool 名を正規表現でマッチするフィールド。書き方のコツ:

  • 複数ツールに同じ処理 → 交替: "matcher": "^(Edit|Write|MultiEdit)$"
  • 全ツール対象 → 空文字または省略: "matcher": ""(PreToolUse全般に発火)
  • Bash のみ → 完全一致: "matcher": "^Bash$"
  • MCP ツール名の接頭辞マッチ → 部分正規表現: "matcher": "^mcp__playwright__"

よくあるミス: "Edit" と書いて Edit / MultiEdit / Write を全部拾うつもりが、完全一致にならず "Edit" を含むツール名にまで発火して想定外の動作をするケース。必ず ^ と $ で囲むのを習慣化するとハマりません。

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "^(Edit|Write|MultiEdit)$",
        "hooks": [
          { "type": "command", "command": "prettier --write \"$CLAUDE_PROJECT_DIR\"" }
        ]
      }
    ]
  }
}

環境変数 $CLAUDE_PROJECT_DIR はプロジェクトルートを指す。Hook 内から相対パスを使うと呼び出し元の cwd に依存して壊れやすいので、明示するのが安全です。

もっと詳しく: ヒント4 — テスト方法(Ch07)

Hook の動作確認は 3段階で進めると迷いません:

  1. コマンド単体で動くか — まず対象コマンドをシェルに貼って動かす。「Prettier に PATH 通ってないので動かない」などの大半はここで潰れる。
  2. claude --debug で発火するか — デバッグモードで Hook 呼び出しのログが出る。--debug でログ確認が最速の切り分け。
  3. 実ワークフローで1日触るか — 小さい PR や手元ブランチで実運用。本番ブランチでいきなり試さない。

それでも動かない場合の典型原因:

PATH問題
Hook はログインシェルと異なる環境で動くため /usr/local/bin 等が通っていない
matcher誤り
^/$忘れ、大文字小文字ミス、ツール名の認識違い
stdin未読
Hook に渡る JSON を jq で取り出す処理を忘れる
exit code
ブロックしたいのに exit 1 にしている(ブロックは exit 2)
もっと詳しく: ヒント5 — セキュリティ確認(Ch07)

Hook は 任意のシェルコマンドを自動実行する仕組みです。ゆえに自分で書いた3つでも、採用前に以下の5点チェックを通してください:

  1. 破壊的操作が含まれていないか — rm -rf / git reset --hard / DROP TABLE 等は絶対にNG。「そんなわけない」と思ったHookでも1度は目検する。
  2. 秘密情報がログに出ていないか — API Key、.env、認証トークンを stderr/stdout に流していないか。set -x のトレース出力も注意。
  3. 外部通信は必要最低限か — curl で外部に POST する Hook は URL を一度ドメインで確認。Slack 通知なら webhook URL を環境変数化する。
  4. 文字列展開で注入されないか — ファイル名や Claude の出力を eval したり、そのまま bash -c "..." に渡さない。必ず変数にクオートする。
  5. 他人由来の Hook は精査してから採用 — チームで共有された settings.json、ブログで拾ったレシピは 必ず中身を読んでから入れる。Hook は「Skillと違ってシェルに直結」している。
Warning

特に危険なのは Stop イベントの curl + eval パターン(外部から取得した内容を実行)。これは RCE に直結します。便利そうに見えても使わない勇気が肝心です。

2. 採用した3つの Hook

#1

Prettier 自動化

  • 目的: 書き出し直後の整形
  • イベント: PostToolUse
  • matcher: ^(Edit|Write|MultiEdit)$
  • 頻度: 日20-40回
  • 短縮: 3秒/回 → 0秒
#2

コミット前テスト

  • 目的: git commit 前の安全ネット
  • イベント: PreToolUse
  • matcher: ^Bash$ + git commit検出
  • 頻度: 日5-8回
  • ブロック: テスト失敗で exit 2
#3

Stop 時 macOS 通知

  • 目的: 長時間作業の終了気付き
  • イベント: Stop
  • matcher: 空(全セッション)
  • 頻度: 日10-15回
  • 効果: 放置ロスが月3時間減

3. settings.json 完全版

~/.claude/settings.json に以下を追加(既存の hooks があればマージ):

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "^(Edit|Write|MultiEdit)$",
        "hooks": [
          {
            "type": "command",
            "command": "cd \"$CLAUDE_PROJECT_DIR\" && npx --no-install prettier --write --log-level=warn \"$(jq -r '.tool_input.file_path // empty')\" 2>&1 || true"
          }
        ]
      }
    ],
    "PreToolUse": [
      {
        "matcher": "^Bash$",
        "hooks": [
          {
            "type": "command",
            "command": "~/.claude/hooks/pre-commit-test.sh"
          }
        ]
      }
    ],
    "Stop": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "~/.claude/hooks/stop-notify.sh"
          }
        ]
      }
    ]
  }
}

#1 Prettier自動化 のポイント

  • jq で Hook の stdin JSON から tool_input.file_path を取り出し、対象ファイルだけ整形。全プロジェクト整形は重すぎるので避ける。
  • --no-install で Prettier 未導入プロジェクトでは黙ってスキップ。|| true で失敗しても Claude の処理を止めない。
  • --log-level=warn で余計な stdout を出さない(Claude の応答欄が騒がしくなる原因)。

#2 コミット前テストの実体(pre-commit-test.sh)

#!/usr/bin/env bash
set -euo pipefail

# stdin から tool_input.command を取り出す
INPUT=$(cat)
CMD=$(echo "$INPUT" | jq -r '.tool_input.command // ""')

# git commit を含む場合だけ発火
if [[ "$CMD" == *"git commit"* ]]; then
  cd "$CLAUDE_PROJECT_DIR"
  if [ -f "package.json" ] && jq -e '.scripts.test' package.json > /dev/null; then
    echo "[hook] pre-commit test running..." >&2
    if ! npm test --silent; then
      echo "[hook] tests failed, blocking commit" >&2
      exit 2  # Claudeにブロックを伝える
    fi
  fi
fi
exit 0

#3 Stop 通知の実体(stop-notify.sh)

#!/usr/bin/env bash
set -euo pipefail

INPUT=$(cat)
# 直近のユーザプロンプト(最初の40文字)を取得
LAST_PROMPT=$(echo "$INPUT" | jq -r '.last_user_message // "Claude finished"' | head -c 40)

# macOS通知 (osascript)
osascript -e "display notification \"$LAST_PROMPT\" with title \"Claude Code\" sound name \"Glass\""

# git status を軽くログ
cd "$CLAUDE_PROJECT_DIR" 2>/dev/null || exit 0
CHANGED=$(git status --porcelain 2>/dev/null | wc -l | tr -d ' ')
echo "[hook] stop: $CHANGED changed files" >&2

考え方の解説

この3つに絞った理由は 3つの設計判断です。

  1. イベントを3種に分散させる — PreToolUse / PostToolUse / Stop を1つずつ取ることで、Hook 同士が干渉しない。複数のPostToolUseを積みすぎると発火順と副作用で詰まる。
  2. 「守り」と「攻め」のバランス — Prettier は攻め(自動整形で時間節約)、pre-commit は守り(失敗の阻止)、Stop 通知は体験改善。違う性質を混ぜると相乗効果が出る。
  3. 切り戻しが1行で済む — どれも settings.json の該当ブロックを削れば即無効化。Git hooks と違い commit に絡まないので、実験コストが極小。
Insight

MK 自身の1週間で一番効いたのは、実は Stop 通知でした。削減工数では Prettier に負けますが、長時間作業の放置ロス(Claudeが終わってるのに気づかない)が圧倒的に減った。体感の改善 > 数字の改善という良い例です。

この3つを自分の settings.json に常駐させられる人は、もう「Hook を知っている」側ではなく「Hook で自分の開発環境を設計できる」側の人間です。

MK の場合、2026年1月時点で仮運用している役割分担は次の通り:

  • 応用1: チーム共有Hookパック — プロジェクト .claude/settings.json にコミット。個人設定を上書きしない範囲に絞り、Prettier と pre-commit test のみ共有(Stop通知は個人嗜好なので除外)。README に「無効化したい時は .claude/settings.local.json に {"hooks": {}} を置けばOK」と明記。

Insight

Hook と CI/CD の使い分け原則は「Hook は1秒、CI は1分以上かけて良い」という時間軸。Hook で重い処理を回すと Claude の応答体験が壊れる。遅い処理は迷わず CI、速い処理で 即座に気づきたいものだけ Hook に置く、という線引きで大抵うまく行きます。

修了のことば

おつかれさまでした。8章・4時間の旅、そしてこの演習をやり遂げたあなたは、もう「Hook を知っている」側ではなく「Hook で自分の開発環境を再設計できる」側の人間です。概念(Ch01-04)・実装(Ch05-06)・運用(Ch07)を束ね、最後に自分の settings.json に刻み込むところまでやりきりました。

Note

本章および本チュートリアル全体の記述は 2026年1月時点の Claude Code 仕様に基づきます。hooks セクションのスキーマ・イベント名・環境変数は変わる可能性があります。必ず 公式 Hooks ドキュメントで最新を確認してから運用してください。Hook は任意のシェルを実行するため、仕様変更時の切り戻しも念頭に置いてください。
