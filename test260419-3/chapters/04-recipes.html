Hookレシピ集

フォーマット/テスト/通知/ブロック — 10のすぐ使える例

所要時間: 40分 Phase: Basics 難易度: 中級
この章のゴール

Ch01-03で学んだ理屈を実働する10個のレシピに落とし込みます。各レシピは「いつ発火するか」「settings.jsonスニペット」「シェルスクリプト実体」の3点セットで提示。コピペして10分で1つ動かせる粒度に揃えました。最後にハンズオンで3つ選んで自環境に導入します。本章の記述は2026年1月時点の情報です。Hookは任意のシェルコマンドを実行する機能なので、他人が配布するスニペットは必ず中身を読んでから入れてください。

前提: Ch03 settings.jsonの書き方でmatcher・command・exit codeの基本を理解していること。未読なら先に戻ってください。

10レシピの全体像: どのイベントに何を仕込むか

まずは10個のレシピをイベント×目的のマトリクスで俯瞰します。どれから導入するか決める判断材料に。

10レシピのイベント分布(色=目的カテゴリ)
PreToolUseで「やる前に止める/整える」、PostToolUseで「やった後に検査/通知」、Stopで「終わりに集約」というのが基本構図。

レシピ1: Edit前にPrettier(PreToolUse)

ClaudeがEdit/Writeで書き換えようとする直前に、対象ファイルをPrettierで整形します。Claudeは整形済みの状態をベースに書き換えるので、差分ノイズが劇的に減ります。

settings.json

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          { "type": "command", "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/prettier-pre.sh" }
        ]
      }
    ]
  }
}

.claude/hooks/prettier-pre.sh

#!/usr/bin/env bash
# Edit/Write対象ファイルをPrettierで事前整形
set -euo pipefail
INPUT=$(cat)
FILE=$(echo "$INPUT" | jq -r '.tool_input.file_path // empty')
[ -z "$FILE" ] && exit 0
[ ! -f "$FILE" ] && exit 0
case "$FILE" in
  *.js|*.jsx|*.ts|*.tsx|*.json|*.css|*.md)
    npx --no-install prettier --write "$FILE" 2>/dev/null || true
    echo "prettier: formatted $FILE" >&2
    ;;
esac
exit 0

レシピ2: Write後にESLint(PostToolUse)

Write/Editが完了した直後にESLintを走らせ、構文/ルール違反があればClaudeにstderr経由でフィードバックします。Claudeはその内容を読んで自動修正を試みます。

settings.json

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          { "type": "command", "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/eslint-post.sh" }
        ]
      }
    ]
  }
}

.claude/hooks/eslint-post.sh

#!/usr/bin/env bash
# Write/Edit後のファイルをESLintで検査し、違反はstderr→Claudeへ返す
set -uo pipefail
INPUT=$(cat)
FILE=$(echo "$INPUT" | jq -r '.tool_input.file_path // empty')
[ -z "$FILE" ] && exit 0
case "$FILE" in
  *.js|*.jsx|*.ts|*.tsx) ;;
  *) exit 0 ;;
esac
OUTPUT=$(npx --no-install eslint --no-color "$FILE" 2>&1 || true)
if [ -n "$OUTPUT" ]; then
  echo "[eslint] $FILE に問題があります:" >&2
  echo "$OUTPUT" >&2
fi
exit 0

レシピ3: Bash実行前に危険コマンドをブロック(PreToolUse)

Bashツールの実行直前にコマンド文字列を検査し、危険コマンドパターンに一致したらexit 2でブロックします。これがHookの防御機能の真骨頂。

警告: exit code 2 の意味

PreToolUseフックがexit code 2で終了すると、Claude Codeはそのツール呼び出しを実行せずに停止し、stderrの内容をClaudeに返します。exit 1は単なる警告(継続)なので混同しないこと。2026年1月時点の仕様。

settings.json

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          { "type": "command", "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/bash-guard.sh" }
        ]
      }
    ]
  }
}

.claude/hooks/bash-guard.sh

#!/usr/bin/env bash
# 危険Bashコマンドをブロック(exit 2)
set -uo pipefail
INPUT=$(cat)
CMD=$(echo "$INPUT" | jq -r '.tool_input.command // empty')

# 危険パターン(必要に応じて追加)
PATTERNS=(
  'rm[[:space:]]+-rf?[[:space:]]+/($|[[:space:]])'
  'rm[[:space:]]+-rf?[[:space:]]+\$HOME'
  'rm[[:space:]]+-rf?[[:space:]]+~'
  'chmod[[:space:]]+-R?[[:space:]]*777'
  'mkfs\.'
  'dd[[:space:]]+if=.*of=/dev/'
  ':\(\)\{.*\}:'
  'curl[[:space:]]+.*\|[[:space:]]*(sh|bash)'
  'wget[[:space:]]+.*\|[[:space:]]*(sh|bash)'
)

for p in "${PATTERNS[@]}"; do
  if [[ "$CMD" =~ $p ]]; then
    echo "[bash-guard] 危険コマンドを検出しました: $CMD" >&2
    echo "[bash-guard] パターン: $p" >&2
    echo "[bash-guard] 実行を中止しました。必要ならユーザーが手動実行してください。" >&2
    exit 2
  fi
done
exit 0

レシピ4: Stop時にgit status表示(Stop)

Claudeの応答が終わるタイミング(Stop)でgit statusとgit diff --statを表示します。「結局どのファイルが変わったのか」を毎回確認せずに済みます。

settings.json

{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          { "type": "command", "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/stop-gitstatus.sh" }
        ]
      }
    ]
  }
}

.claude/hooks/stop-gitstatus.sh

#!/usr/bin/env bash
# Stop時にgit status / git diff --stat を表示
set -uo pipefail
cd "$CLAUDE_PROJECT_DIR" 2>/dev/null || exit 0
[ ! -d .git ] && exit 0
echo "" >&2
echo "===== git status =====" >&2
git status -s 2>&2 || true
echo "" >&2
echo "===== git diff --stat =====" >&2
git diff --stat 2>&2 || true
echo "" >&2
exit 0

レシピ5: PreToolUse で注意喚起(ログ出力)

ブロックはしないが実行を可視化したいツール(Write・Bash等)について、実行直前にコマンド内容をログ記録します。後から「Claudeは何をしたのか」を監査できます。

settings.json

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash|Write|Edit",
        "hooks": [
          { "type": "command", "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/audit-log.sh" }
        ]
      }
    ]
  }
}

.claude/hooks/audit-log.sh

#!/usr/bin/env bash
# 実行ツール内容を監査ログに記録
set -uo pipefail
INPUT=$(cat)
LOG_DIR="$CLAUDE_PROJECT_DIR/.claude/audit"
mkdir -p "$LOG_DIR"
LOG_FILE="$LOG_DIR/$(date +%Y-%m-%d).log"
TS=$(date +"%Y-%m-%dT%H:%M:%S%z")
TOOL=$(echo "$INPUT" | jq -r '.tool_name // "?"')
SUMMARY=$(echo "$INPUT" | jq -c '.tool_input | {file_path, command}' 2>/dev/null || echo '{}')
echo "{\"ts\":\"$TS\",\"tool\":\"$TOOL\",\"input\":$SUMMARY}" >> "$LOG_FILE"
exit 0

レシピ6: UserPromptSubmitでPIIマスク

ユーザーが入力したプロンプトをClaudeへ渡す前に検査し、メール/電話番号/クレカ等のPIIらしき文字列を[REDACTED]にマスクします。stdoutに加工済みプロンプトを出力するとClaudeはそれを見ます。

settings.json

{
  "hooks": {
    "UserPromptSubmit": [
      {
        "hooks": [
          { "type": "command", "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/pii-mask.sh" }
        ]
      }
    ]
  }
}

.claude/hooks/pii-mask.sh

#!/usr/bin/env bash
# PIIらしき文字列をマスクしてstdoutに返す
set -uo pipefail
INPUT=$(cat)
PROMPT=$(echo "$INPUT" | jq -r '.prompt // empty')
MASKED=$(echo "$PROMPT" \
  | sed -E 's/[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}/[EMAIL_REDACTED]/g' \
  | sed -E 's/\b0[0-9]{1,4}-[0-9]{1,4}-[0-9]{3,4}\b/[PHONE_REDACTED]/g' \
  | sed -E 's/\b[0-9]{4}[[:space:]-]?[0-9]{4}[[:space:]-]?[0-9]{4}[[:space:]-]?[0-9]{4}\b/[CARD_REDACTED]/g')
# 加工済みをstdoutに(Claudeはこれを見る)
jq -n --arg p "$MASKED" '{prompt: $p}'
exit 0
注意: マスクは万能ではない

正規表現によるマスクは完全性を保証しません。住所・氏名・社内ID等はパターン化が難しく漏れがち。機密業務では「そもそも機密を入力しない」を第一線に、マスクは第二線の保険と考えてください。

レシピ7: PostToolUseで通知音(macOS)

長時間タスク後に「終わった」と気づきたいとき、PostToolUseでafplayを鳴らします。Write・Edit・Bashなど重い処理の直後に短い音を鳴らす使い方が人気。

settings.json

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          { "type": "command", "command": "afplay /System/Library/Sounds/Pop.aiff &" }
        ]
      }
    ]
  }
}

シンプル版(スクリプト不要)

上記のようにcommandに1行埋め込めばスクリプトファイル不要。&でバックグラウンド実行にしておけば、鳴り終わりを待たずに処理が進みます。

Linuxの場合

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          { "type": "command", "command": "paplay /usr/share/sounds/freedesktop/stereo/complete.oga &" }
        ]
      }
    ]
  }
}

レシピ8: SessionStartでgit pull

Claude Codeを起動した瞬間(SessionStart)にgit pullを実行し、常に最新リモートから作業を開始します。チーム開発で古いブランチで作業してしまう事故が減ります。

settings.json

{
  "hooks": {
    "SessionStart": [
      {
        "hooks": [
          { "type": "command", "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/session-pull.sh" }
        ]
      }
    ]
  }
}

.claude/hooks/session-pull.sh

#!/usr/bin/env bash
# セッション開始時にgit pull(mainブランチ限定)
set -uo pipefail
cd "$CLAUDE_PROJECT_DIR" 2>/dev/null || exit 0
[ ! -d .git ] && exit 0
BRANCH=$(git symbolic-ref --short HEAD 2>/dev/null || echo "")
case "$BRANCH" in
  main|master|develop)
    echo "[session-start] git pull on $BRANCH" >&2
    git pull --ff-only 2>&2 || echo "[session-start] pull失敗: 手動確認してください" >&2
    ;;
  *)
    echo "[session-start] ブランチ $BRANCH: pullをスキップ" >&2
    ;;
esac
exit 0
注意: 強制pullは禁物

git pull --ff-onlyにしているのはマージコミットを勝手に作らないため。未コミット変更があると失敗しますが、それは意図通り(壊れる前に止まる方が良い)。

レシピ9: Stop時にSlack通知

長時間作業が終わったらSlackに「終わったよ」と通知。Slack Incoming WebhookのURLを環境変数で持たせる運用が安全です。

settings.json

{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          { "type": "command", "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/stop-slack.sh" }
        ]
      }
    ]
  }
}

.claude/hooks/stop-slack.sh

#!/usr/bin/env bash
# Claude作業終了をSlackに通知
set -uo pipefail
[ -z "${SLACK_WEBHOOK_URL:-}" ] && exit 0
cd "$CLAUDE_PROJECT_DIR" 2>/dev/null || true
CHANGED=$(git diff --stat 2>/dev/null | tail -1 || echo "変更なし")
HOST=$(hostname -s)
PAYLOAD=$(jq -n \
  --arg text ":white_check_mark: Claude Code 作業終了 @ *$HOST*" \
  --arg changed "$CHANGED" \
  '{text: $text, blocks: [
    {type:"section", text:{type:"mrkdwn", text: $text}},
    {type:"context", elements:[{type:"mrkdwn", text: $changed}]}
  ]}')
curl -sS -X POST -H 'Content-Type: application/json' \
  --data "$PAYLOAD" "$SLACK_WEBHOOK_URL" >/dev/null || true
exit 0
Webhook URLの管理

SlackのWebhook URLはシェルのrcファイル(~/.zshrc等)にexport SLACK_WEBHOOK_URL=...で入れ、settings.jsonやリポジトリにはコミットしないこと。Claude Codeは親シェルの環境変数を引き継ぎます。

レシピ10: Stop時にサマリをjsonl保存

Stop時に1行JSON(jsonl)として作業サマリを蓄積。あとから「先月何をやったか」をjqで集計できます。

settings.json

{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          { "type": "command", "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/stop-jsonl.sh" }
        ]
      }
    ]
  }
}

.claude/hooks/stop-jsonl.sh

#!/usr/bin/env bash
# Stop時に1行JSONでサマリを追記
set -uo pipefail
cd "$CLAUDE_PROJECT_DIR" 2>/dev/null || exit 0
LOG_DIR="$CLAUDE_PROJECT_DIR/.claude/sessions"
mkdir -p "$LOG_DIR"
LOG_FILE="$LOG_DIR/$(date +%Y-%m).jsonl"
FILES_CHANGED=$(git diff --name-only 2>/dev/null | wc -l | tr -d ' ')
LINES_ADDED=$(git diff --numstat 2>/dev/null | awk '{s+=$1} END {print s+0}')
LINES_REMOVED=$(git diff --numstat 2>/dev/null | awk '{s+=$2} END {print s+0}')
BRANCH=$(git symbolic-ref --short HEAD 2>/dev/null || echo "?")
jq -nc \
  --arg ts "$(date +%Y-%m-%dT%H:%M:%S%z)" \
  --arg branch "$BRANCH" \
  --argjson files "${FILES_CHANGED:-0}" \
  --argjson add "${LINES_ADDED:-0}" \
  --argjson rm "${LINES_REMOVED:-0}" \
  '{ts: $ts, branch: $branch, files_changed: $files, lines_added: $add, lines_removed: $rm}' \
  >> "$LOG_FILE"
exit 0

集計例

# 今月のファイル変更数合計
jq -s 'map(.files_changed) | add' .claude/sessions/$(date +%Y-%m).jsonl

# ブランチ別の作業回数
jq -r .branch .claude/sessions/*.jsonl | sort | uniq -c | sort -rn

個別レシピを組み合わせて「定番セット」を作りましょう。以下はWeb開発者向け標準構成の例です。

「Web開発者向け標準セット」の発火シーケンス

導入効果(実測感覚)

メリット

  • 整形差分ノイズが激減(PR差分が本質的変更だけに)
  • 危険コマンド事故の未然防止
  • 長時間作業の完了を見逃さない
  • jsonlログで自分の生産性が数値化できる
注意点

トレードオフ

  • Hookが増えるほど1操作の待ち時間が伸びる
  • スクリプトの不具合で作業が止まるリスク
  • 誰かの書いたhookを入れる前に必ずレビュー
  • シェル/コマンド依存(OS間移植性に注意)

Hookの実行コスト比較(ms単位で積み上がる)

Hookはツール呼び出し毎に発火するため、1個あたり100ms程度でも10個重なれば1秒の体感遅延になります。各レシピの典型実行時間を計測した目安は以下。

各レシピの実行時間目安(中央値・ミリ秒)
ESLintとSlack通知は重め。重い処理は&でバックグラウンド化するか、matcherで対象ファイルを絞ることで体感を改善できる。

実行順とフロー

PreToolUse → ツール実行 → PostToolUse の流れ
graph LR
  A["ユーザー入力"] --> B["UserPromptSubmit
(PIIマスク等)"]
  B --> C["Claude思考"]
  C --> D{"ツール呼ぶ?"}
  D -- Yes --> E["PreToolUse発火
(整形/ガード)"]
  E --> F{"exit 2?"}
  F -- Yes --> X["ブロック(ツール未実行)"]
  F -- No --> G["ツール実行"]
  G --> H["PostToolUse発火
(Lint/通知音)"]
  H --> C
  D -- No --> I["応答生成"]
  I --> J["Stop発火
(git status/Slack/jsonl)"]
Stopは応答が完全に終わったタイミングで1回だけ発火。複数のhooks配列はすべて順次実行される。
もっと詳しく: hooks配列内の実行順序

1つのイベント/matcherに対して複数のhooksを並べた場合、配列の上から順に直列実行されます。前のhookがexit 2で失敗すると後続のhookは実行されないため、「ガード系は先に・通知系は後に」という順序を意識すると安全。

並列にしたい場合は各commandの末尾に&を付ければバックグラウンド化できますが、stderrの順序が崩れるので通知系にだけ使うのが無難です。

もっと詳しく: matcher正規表現のコツ

matcherはツール名の正規表現です。"Edit|Write"で両方マッチ、".*"や空文字で全マッチ。"Bash"のように完全一致っぽく書いても実際はregexなので、意図せずMyBashにもマッチしてしまう可能性あり(現時点の実装では心配不要ですが、将来的に増える可能性を考えて"^Bash$"と書く流派もあります)。

matcherはイベント単位で複数書けるので、「Bash用のガード」と「Edit用のフォーマット」を別ブロックに分ける設計が保守しやすいです。

もっと詳しく: セキュリティチェックリスト(hook導入前)
  1. コマンドの中身を1行ずつ読む(他人のsnippetは必ず)
  2. rm・curl | sh・環境変数エクスポートを含むならアラート
  3. 書き込み先のパスが相対ではなく$CLAUDE_PROJECT_DIR配下か
  4. Webhook URLやトークンをハードコードしていないか
  5. 失敗時にset -eで全体を止めるべきか|| trueで黙って流すべきか判断する

ハンズオン: 3つ選んで自環境に導入する

10レシピ全部は一気に多すぎるので、3つだけ選んで入れるのがおすすめです。以下の「初学者向け推奨3点セット」から始めると失敗しにくい。

今すぐ試す: 推奨3点セット導入
  1. レシピ4(Stop時git status) — 作業の終わりにファイル変更一覧を出す、副作用ゼロ
  2. レシピ7(PostToolUse通知音) — 終わりに気づける、1行で完結
  3. レシピ3(Bash危険コマンドブロック) — rm -rf系の事故防止、効果絶大

この3つなら10分で導入でき、追加の外部サービス(Slack・PII辞書等)も不要です。

手順

  • 1. ディレクトリ作成 — mkdir -p .claude/hooks をプロジェクトルートで。
  • 2. スクリプト配置 — 上記レシピ3/4のシェルスクリプトを.claude/hooks/に配置。chmod +xを忘れずに。
  • 3. settings.json編集 — .claude/settings.jsonにレシピ3・4・7の設定をマージ。
  • 4. 動作確認 — Claudeに「適当なファイルを作って」と依頼→PostToolUseの音+Stopのgit statusが出ることを確認。
  • 5. ブロックテスト — Claudeに「rm -rf /tmp/testを実行して」と依頼。bash-guardがブロックすれば成功。

3点セット統合版 settings.json

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          { "type": "command", "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/bash-guard.sh" }
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Bash|Write|Edit",
        "hooks": [
          { "type": "command", "command": "afplay /System/Library/Sounds/Pop.aiff &" }
        ]
      }
    ],
    "Stop": [
      {
        "hooks": [
          { "type": "command", "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/stop-gitstatus.sh" }
        ]
      }
    ]
  }
}
Exercise 1
初級

レシピの分類

以下6つのシナリオについて、使うべきイベント(PreToolUse/PostToolUse/Stop/UserPromptSubmit/SessionStart)を答えてください。

  1. A: 「書き込み前にコードをフォーマットしたい」
  2. B: 「作業が終わったらSlackに通知したい」
  3. C: 「プロンプトに社内URLが含まれたらマスクしたい」
  4. D: 「Claude起動時に最新リモートをpullしたい」
  5. E: 「書き込み後にユニットテストを流したい」
  6. F: 「Claudeがrm -rfを実行しようとしたら止めたい」
模範解答を見る
  • A → PreToolUse: Write/Edit直前なのでPre。matcherは"Edit|Write"。
  • B → Stop: 全作業終了のタイミング。1回だけ発火。
  • C → UserPromptSubmit: Claudeに渡る前に加工する必要がある。
  • D → SessionStart: Claude Code起動時に1回のみ。
  • E → PostToolUse: Write/Edit後なのでPost。ただしテストが長いと体感遅延が大きいのでStopに移す選択肢もあり。
  • F → PreToolUse (matcher: Bash): 実行前に止める必要があるのでPre+exit 2。

「前に止めるか後で検査するか」「全体の終わりか各ツール単位か」という軸で判別できれば十分。

Exercise 2
応用

自分のプロジェクトで3レシピを動かす

以下の5項目を実施し、実際にHookが発火した証拠(スクショor貼り付けテキスト)を残してください。

  1. プロジェクトを1つ選ぶ(Git管理下・書き込み可能)
  2. 10レシピから3つを選び、理由(目的)を1行で書く
  3. .claude/hooks/にスクリプトを配置してchmod +x
  4. .claude/settings.jsonにHook設定をマージ
  5. Claudeに適当な作業を依頼し、3つのHookが発火することを確認
模範解答を見る

回答例: MK(筆者)の学習サイトリポジトリで導入

  • 選定: レシピ1(Prettier)、レシピ4(Stop時git status)、レシピ10(jsonlログ)
  • 理由: 「HTML/CSS差分ノイズ排除」「作業終わりの可視化」「毎日の作業量記録」
  • 確認: Claudeに「index.htmlを少し編集して」と依頼→PreToolUseでPrettierが動いた痕跡(stderrログ)→Stopでgit statusが出る→.claude/sessions/2026-01.jsonlに1行追記

気づき: afplay の通知音を入れると会議中に鳴って困る。平日昼は音を消すようtest $(date +%H) -ge 18のガードをかけた。「自分の環境に合わせて微調整する」がHook活用の本質。

もう一つの落とし穴: chmod +xを忘れると静かに失敗する。実行権限エラーはデフォルトでは表示されにくいので、発火しないときはまずls -l .claude/hooks/を確認すること。
