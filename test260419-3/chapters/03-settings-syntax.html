settings.jsonの書き方

基本構造・matcher・command・stdout/exit codeの扱い

所要時間: 30分 Phase: Basics 難易度: 初級
この章のゴール

Hookを動かすには必ず settings.json に定義を書きます。この章では3つのスコープ(user/project/local)・Hooksセクションの構造(イベント→matcher→hooks)・matcherの正規表現・commandの書き方・stdout/stderr/exit codeの規約・Hookスクリプトで使える環境変数を押さえ、最後に自分で設定を書いて発火させるハンズオンまでやります。本章の記述は2026年1月時点の情報です。

セキュリティ注意

Hookは 任意のシェルコマンド を自動実行します。他人から渡された settings.json をそのまま読み込むとコードインジェクションになります。Ch07のセキュリティ章と合わせて、本章の例も内容を理解してからコピペしてください。

3つのスコープ — どこに書くか

settings.json は3つの場所に置けます。同じキーが複数に存在すると狭いスコープが優先されます。つまり「全体に適用したいならuser、プロジェクトで共有したいならproject、自分だけ・gitに入れたくないならlocal」という棲み分けです。

スコープの階層 — 狭いほど強い
3つが同時に定義されると、local > project > user の順で勝つ。
スコープパス適用範囲gitに入れる?主な用途
user~/.claude/settings.json全プロジェクト対象外(個人PC)自分のマシン共通Hook(通知・音・全体ログ)
project.claude/settings.jsonそのリポジトリだけ入れる(チーム共有)プロジェクト固有Hook(Lint・テスト)
local.claude/settings.local.jsonそのリポジトリ&自分だけ入れない(.gitignore)個人の実験・APIキー・Webhook URL等
優先順位のイメージ

同じイベント(例: PreToolUse)が3スコープすべてに書かれていると、3つすべてが発火します(上書きではなくマージ)。ただし permissions のような単一値キーは local > project > user の順で勝ちます。Hookはマージ・それ以外は上書きと覚えると実務で困りません。

もっと詳しく: どこに書くか迷ったら

判断フロー: (1) 他のマシンでも欲しい? → No なら local。Yes なら (2)へ。(2) チーム全員に効かせたい? → Yes なら project(gitコミット)。No なら user(自マシン全リポジトリ)。

特に Slack Webhook URL や個人APIキーのような シークレット を含むHookは必ず localに置き、.gitignore に .claude/settings.local.json を追加してください。

Hooksセクションの構造 — 3階層

Hookの定義は必ず "hooks" キーの下に書きます。構造はイベント名 → matcher配列 → hook配列の3階層。これだけ押さえれば、どんなHookも読めるようになります。

Hooksセクションの3階層ツリー
graph TD
  R["hooks (root)"] --> E1["PreToolUse (event)"]
  R --> E2["PostToolUse (event)"]
  R --> E3["Stop (event)"]
  E1 --> M1["matcher: Edit|Write (regex)"]
  E1 --> M2["matcher: Bash"]
  M1 --> H1["hook[0]: prettier --write"]
  M1 --> H2["hook[1]: eslint --fix"]
  M2 --> H3["hook[0]: danger-cmd-check.sh"]
  E3 --> MS["matcher: (空=常時)"]
  MS --> HS["hook[0]: summary.sh"]
イベント名の下に複数matcher、各matcherに複数hook。この3階層で何が起きても書ける。

最小サンプル — これが読めれば全部読める

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "echo 'about to edit' >&2"
          }
        ]
      }
    ]
  }
}
  • 第1階層: "hooks" の直下はイベント名(PreToolUse/PostToolUse/Stop/UserPromptSubmit/SessionStart/SessionEnd)。各イベントの値は配列。
  • 第2階層: matcher はツール名のマッチング正規表現。"Edit|Write" なら両方で発火。省略か空文字ならそのイベントの全発火で動く。
  • 第3階層: "hooks"(同名でややこしい)配列には {type: "command", command: "..."} が並ぶ。複数書けば上から順に実行。

matcher — 発火対象を絞る正規表現

matcher はツール名に対する正規表現です。「このツールが呼ばれた時だけHookを走らせたい」を表現します。完全一致ではなく部分一致なので注意。

matcher の値マッチするツール典型用途
"Edit"Edit のみ(ただし "MultiEdit" にも当たる。"^Edit$"で厳密に)編集Hook
"Edit|Write|MultiEdit"3つのいずれかファイル書き込み全般
"Bash"Bashツール呼び出しコマンド実行前チェック
"^(Edit|Write)$"厳密に Edit か Write のみ他と誤マッチしたくない時
"" または省略そのイベントの全発火Stop/SessionStartなどツール不要イベント
".*"全ツールPreToolUse全体で共通処理
regex 落とし穴 TOP3

(1) "Edit" は MultiEdit にも一致する(部分一致のため)。完全一致したければ "^Edit$"。
(2) パイプ | は OR。括弧でくくるのが無難: "^(Edit|Write)$"。
(3) JSON文字列なので バックスラッシュは二重: "\\.ts$" のように書く(matcherではあまり使わないが、commandに正規表現を渡す時に注意)。

もっと詳しく: matcherは「ツール名以外」にも拡張されつつある

2026年1月時点、matcherの主用途はツール名マッチです。ただしイベントによっては対象が異なります(UserPromptSubmit ではプロンプトにマッチさせる話が議論中、SessionStart は source にマッチなど)。最新仕様は公式docを参照。迷ったらツール名だけで考えるのが実務で安全です。

command — シェル1行 or スクリプトpath

command にはシェルコマンドの文字列を書きます。短い処理なら1行で書き切り、長ければ外部スクリプトに切り出すのが定石です。

1行で書き切る

短くて使い捨て

{
  "type": "command",
  "command": "date +%s >> ~/claude-hook.log"
}

向いてる: ログ追記・echo・通知1行。

外部スクリプト

複雑・再利用

{
  "type": "command",
  "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/lint.sh"
}

向いてる: 条件分岐・複数コマンド・チーム共有。

command のベストプラクティス

  • 絶対パス or 環境変数で書く — $CLAUDE_PROJECT_DIR/.claude/hooks/xxx.sh。相対パスはcwdに依存するので避ける。
  • スクリプトには実行権限 — chmod +x .claude/hooks/lint.sh を忘れない。実行できないと Permission denied で失敗する。
  • 長時間処理はタイムアウト設定 — "timeout": 30(秒)を hook オブジェクトに指定。指定なしは既定60秒で強制終了。
  • 依存コマンドは前提を明示 — which jq >/dev/null || exit 0 のように、入ってないツールがあってもHook自体は失敗させないガードを入れる。

stdout / stderr / exit code の規約

Hookが終了した時のexit code と stderr によって、Claude Codeの挙動が変わります。ここを理解すると「警告だけ出す」「処理を止める」を使い分けられるようになります。

exit code → Claudeの振る舞い
exit 2 だけが特別。0は成功、2以外の非0はユーザ向け警告で続行。
exit code意味stdout の行方stderr の行方Claudeの挙動
0成功transcript(ユーザ表示)ユーザ表示(参考)続行
2ブロックユーザ表示Claudeに戻るツール実行を中断
1 や他の非0警告ユーザ表示ユーザ表示のみ続行(Claude不伝達)

PreToolUseで「危険コマンドを止める」実例

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "cat | jq -r '.tool_input.command' | grep -qE '^rm -rf /' && { echo 'Dangerous rm -rf detected' >&2; exit 2; } || exit 0"
          }
        ]
      }
    ]
  }
}

cat | jq でstdinに流れてくるJSONから tool_input.command を取り出し、rm -rf / で始まるなら exit 2 でブロック、stderrにエラー理由を書きます。Claudeはstderrを読んで「やめます」と判断します。

JSONで詳細制御(高度)

stdoutに{"decision": "block", "reason": "..."} のようなJSONを出すと、exit codeより細かい制御ができます(continue / stopReason / suppressOutput 等)。まずはexit 0/2だけで十分実用になります。深掘りはCh04のレシピ集で。

Hookで使える環境変数

Hookスクリプトの中では、Claude Codeが用意する環境変数が参照できます。これを使うとスクリプトがポータブルになり、チーム共有しても壊れません。

$CLAUDE_PROJECT_DIR
プロジェクトルートの絶対パス。最頻出。スクリプトpathの基準に使う。
$CLAUDE_SESSION_ID
現セッションID。ログをセッション単位に分けたい時に使う。
$CLAUDE_EVENT
発火イベント名(PreToolUse等)。1つのスクリプトを複数イベントで共有する時に分岐用。
$CLAUDE_TOOL
発火元のツール名(Edit等)。ツール名でさらに条件分岐したい時。
標準変数
$HOME/$PATH/$USER等のシェル変数も通常通り使える。

典型的な使い方 — ログをプロジェクト配下に貯める

#!/usr/bin/env bash
# .claude/hooks/log-edit.sh
LOG_DIR="$CLAUDE_PROJECT_DIR/.claude/logs"
mkdir -p "$LOG_DIR"
echo "[$(date -Iseconds)] $CLAUDE_EVENT $CLAUDE_TOOL session=$CLAUDE_SESSION_ID" \
  >> "$LOG_DIR/hooks.log"
exit 0
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write|MultiEdit",
        "hooks": [
          {
            "type": "command",
            "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/log-edit.sh"
          }
        ]
      }
    ]
  }
}

このスクリプトはどのマシンに持っていっても動きます。$CLAUDE_PROJECT_DIR に頼ることで、cwdや絶対パスのブレを吸収できるのがポイント。

もっと詳しく: stdin経由で流れてくるJSON

環境変数だけでなく、HookにはstdinからJSONオブジェクトが流れ込みます。中身は {session_id, transcript_path, cwd, hook_event_name, tool_name, tool_input, ...}。環境変数に無い情報(例: tool_input.file_path)はここからjqで取り出してください。例: FILE=$(jq -r '.tool_input.file_path')。

ハンズオン — 書いて発火させる

読むだけでは身につきません。自分の手で設定を書き、claudeを実行して発火を観察します。所要5分、任意のディレクトリで実行できます。

1
準備
試し用ディレクトリ作成
2
書く
.claude/settings.json 作成
3
実行
claude で編集指示
4
確認
ログ出力を確認

Step 1: 試し用ディレクトリを作る

mkdir -p ~/hook-test/.claude
cd ~/hook-test

Step 2: .claude/settings.json を作る

cat > .claude/settings.json <<'JSON'
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "echo \"[HOOK FIRED] event=$CLAUDE_EVENT tool=$CLAUDE_TOOL at=$(date +%T)\" >> /tmp/hook-test.log"
          }
        ]
      }
    ],
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "echo \"[STOP] session=$CLAUDE_SESSION_ID ended at=$(date +%T)\" >> /tmp/hook-test.log"
          }
        ]
      }
    ]
  }
}
JSON

Step 3: Claudeで編集を頼む

claude "hello.txt に 'world' と書いてください"

Step 4: ログを確認

cat /tmp/hook-test.log
# [HOOK FIRED] event=PreToolUse tool=Write at=14:32:10
# [STOP] session=xxxxxxxx-xxxx-xxxx ended at=14:32:13
ログが出なかった時

(1) .claude/settings.json のJSON構文エラー — jq . .claude/settings.json で検証。
(2) matcherのミス — 実際に呼ばれたツール名をClaudeの出力から確認(Write or Edit or MultiEdit)。
(3) claude --debug で実行ログ詳細。Ch07のトラブルシュート章で深掘りします。

ここまで完了すれば、settings.jsonは怖くないという感覚が身についているはずです。Ch04ではこの知識を土台に、すぐ使える10個のレシピを紹介します。

実際のプロジェクトで、3スコープにどれくらいの割合でHookが置かれているかの肌感覚を可視化しておきます。

Exercise 1
入門

次のsettings.jsonの「何が起きるか」を読み解く

以下の設定を読んで、Claude Codeが Edit と Bash を呼んだ時にそれぞれ何が起きるか、日本語で説明してください。

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "^Edit$",
        "hooks": [
          { "type": "command", "command": "echo 'pre-edit' >&2; exit 0" }
        ]
      },
      {
        "matcher": "Bash",
        "hooks": [
          { "type": "command", "command": "echo 'no bash allowed' >&2; exit 2" }
        ]
      }
    ]
  }
}
模範解答を見る

Edit呼び出し時: 1つ目のmatcher ^Edit$ が厳密一致で発動。pre-edit をstderrに出し、exit 0 で成功。Claudeはそのまま編集を実行する。stderrは参考表示でユーザーに見える。

Bash呼び出し時: 2つ目のmatcher Bash が発動。no bash allowed をstderrに出し、exit 2 でブロック。Claudeはstderrを読んで「Bashが禁止されている」と理解し、別の手段を探すか処理を中断する。

ポイント: ^Edit$ にすると MultiEdit には当たらない(厳密一致)。Bash だけは部分一致なので BashOutput があれば当たるが、2026年1月時点ではBashのみ。

Exercise 2
応用

自分のリポジトリに3種類のHookを配置し、発火を確認する

任意のGitリポジトリで、3スコープすべてに1つずつHookを置いて動かします。

  1. ~/.claude/settings.json に Stopイベント で osascript -e 'display notification "Claude done"' (macOS) か notify-send (Linux) を書く。全リポジトリで終了通知が飛ぶ
  2. 対象リポジトリの .claude/settings.json に PreToolUse: Edit|Write で $CLAUDE_PROJECT_DIR/.claude/hooks/log.sh を呼ぶ設定と、その log.sh を配置(実行権限をchmod +x)
  3. .claude/settings.local.json に自分だけの PostToolUse: Bash で $CLAUDE_PROJECT_DIR/.claude/bash-audit.log への追記Hookを置く
  4. .gitignore に .claude/settings.local.json と .claude/*.log を追加
  5. claude で編集+Bash実行を伴う指示を1つ投げ、3つすべてのHookが発火したか確認
模範解答を見る

user scope (~/.claude/settings.json):

{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          { "type": "command", "command": "osascript -e 'display notification \"Claude done\" with title \"Claude Code\"'" }
        ]
      }
    ]
  }
}

project scope (.claude/settings.json):

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write|MultiEdit",
        "hooks": [
          { "type": "command", "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/log.sh" }
        ]
      }
    ]
  }
}

.claude/hooks/log.sh(chmod +xを忘れず):

#!/usr/bin/env bash
echo "[$(date -Iseconds)] $CLAUDE_TOOL on $(pwd)" \
  >> "$CLAUDE_PROJECT_DIR/.claude/edit-audit.log"
exit 0

local scope (.claude/settings.local.json):

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          { "type": "command", "command": "cat | jq -r '.tool_input.command' >> \"$CLAUDE_PROJECT_DIR/.claude/bash-audit.log\"" }
        ]
      }
    ]
  }
}

観察ポイント: Claude実行後に (1) Stop通知が飛ぶ (2) edit-audit.log に編集履歴 (3) bash-audit.log にBash履歴、が残っていればすべて正常。.gitignoreでsettings.local.jsonと*.logが除外されていればチームに漏れません。この配置が実運用の標準形です。

※ claude --debug を付けて実行すると、マージされた最終settingsと発火ログが見られます。Ch07のトラブルシュートで詳しく扱います。
