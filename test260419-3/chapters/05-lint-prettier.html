実践：Lint/Prettier を PreToolUse で

ESLint/Prettier/Black/gofmt/rubocop による自動フォーマット — 言語別セットアップと競合回避

所要時間: 30分 Phase: Practice 難易度: 中級
この章のゴール

Claude Code が書き出すコードを、毎回手で整形し直す作業から解放されます。本章では PreToolUse(Write/Edit 直前)と PostToolUse(Write/Edit 直後)の使い分け、Prettier / ESLint / Black / gofmt / rubocop の 言語別セットアップ、そして「Claude が書いた直後に勝手に再フォーマットして差分が騒がしくなる」問題の競合回避を扱います。ハンズオンで TypeScript プロジェクトに Prettier hook を実装し、動作確認まで一気に到達します。本章の記述は2026年1月時点。

PreToolUse と PostToolUse の使い分け

フォーマット系の hook は、発火タイミングで役割が別物になります。最初に両者のシーケンスを図解します。

Pre / Post の発火順序(Write ツール呼び出し時)
sequenceDiagram
  autonumber
  participant U as User
  participant C as Claude
  participant H1 as PreToolUse hook
  participant FS as ファイルシステム
  participant H2 as PostToolUse hook
  U->>C: "login.ts を直して"
  C->>H1: Write ツール呼び出し(仮)
  H1-->>H1: 書き込み予定の内容を整形
  H1->>FS: 整形済みコード書き込み
  Note over H1,FS: ここで実ファイルに書く
  FS-->>H2: Write 完了を通知
  H2-->>H2: 整形再実行 / Lint 検査
  H2-->>C: stderr でフィードバック
  C->>U: 完了報告
PreToolUse は「書く直前の介入」、PostToolUse は「書いた直後の後処理」。どちらを選ぶかで設計が変わる。

用語を整理します。PreToolUseはツール実行の直前に割り込みます。PostToolUseはツール実行の直後に呼ばれます。

観点PreToolUse で整形PostToolUse で整形
Claude が見るコード 整形後 (後続の Edit がキレイな前提で動く) 整形前 (Claude が書いたままの形)
実装の素直さ tool_input を書き換える必要あり(やや複雑) ファイル書き込み後に cmd 実行でシンプル
差分のきれいさ 最終状態が整形済みで安定 Claude の出力 → 再整形で勝手に変わる
失敗時の影響 整形失敗で書き込みブロック(安全側) ファイルは既に書かれている(ロールバック必要)
推奨用途 「書く前に標準化」したいフォーマッタ 「後から検査」したい Lint / 型チェック
もっと詳しく: なぜ PostToolUse フォーマッタは差分が騒がしくなるのか

Claude は自分のコンテキストに残っているコードを基準に次の Edit を計画します。PostToolUse でファイルだけ書き換えると、Claude の内部モデルとディスクが乖離し、次の Edit で「どこを変えるか」指示が通りにくくなります。例えば Claude は const x =1;(空白崩れ) のつもりで次を指示するが、実際のファイルは const x = 1; になっており、Edit の old_string が見つからず失敗する、といった事故が起きます。PreToolUse で整形すれば Claude もキレイな形で認識するため、この食い違いが起きません。

競合を避ける設計マトリクス

ツール別に「どちらを使うべきか」を整理したのが以下のチャートです。読者のプロジェクトで迷ったら、この図を1分で見るだけで判断できます。

ツール別推奨マトリクス
冪等(何度実行しても結果が同じ)かつ高速なものは PreToolUse。時間のかかる型検査や全体 Lint は hook ではなく CI に任せる。
  • 冪等かつ高速 (Prettier / Black / gofmt) — PreToolUse で書き込み前に整形。ファイル全体を渡しても冪等性が保証されている。
  • 自動修正あり Linter (ESLint --fix / rubocop -a) — PostToolUse で検査だけ回し、修正の提案は stderr 経由で Claude に見せる。自動書き換えは差分を騒がしくしやすい。
  • 低速(tsc full / pytest) — hook に入れると UX が破綻。CI 側で担当し、hook は「変更ファイルだけ軽く」にとどめる。
セキュリティ警告

Hook は任意のシェルコマンドを実行します。チーム共有の .claude/settings.json に他人が悪意あるコマンドを追加すると、あなたの環境で勝手に実行されます。settings.json を git から pull したら必ず diff を目視レビューしてから claude を起動してください。settings の優先順位は settings.local.json > settings.json。

言語別 settings.json セットアップ

ここから実戦です。各言語ごとにそのままコピペで動く settings.json と、補助シェルスクリプトを載せます。matcher の正規表現・exit code の意味を省略しないので、自分の環境に合わせて微調整してください。

TypeScript / JavaScript: Prettier + ESLint

// .claude/settings.json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "bash .claude/scripts/prettier-hook.sh"
          }
        ]
      }
    ]
  }
}
#!/usr/bin/env bash
# .claude/scripts/prettier-hook.sh — 直後整形スクリプト
# stdin から tool 情報(JSON)が渡ってくる
set -euo pipefail

# jq で Write/Edit の対象ファイルパスを取り出す
payload="$(cat)"
file="$(echo "$payload" | jq -r '.tool_input.file_path // empty')"

# 対象外は素通し
if [[ -z "$file" ]]; then exit 0; fi
case "$file" in
  *.ts|*.tsx|*.js|*.jsx|*.json|*.md|*.css) ;;
  *) exit 0 ;;
esac

# Prettier が無ければ warn だけ(環境ごとの事故防止)
if ! command -v npx >/dev/null 2>&1; then
  echo "npx not found; skip prettier" >&2
  exit 0
fi

# 整形を実行。差分があれば stderr に通知(ブロックはしない)
if npx --no-install prettier --write "$file" 2>/dev/null; then
  echo "[prettier] formatted: $file" >&2
  exit 0
fi
echo "[prettier] failed: $file (check .prettierrc)" >&2
exit 1   # 1 = Claude に警告として見せる。ブロックはしない

Python: Black + ruff

// .claude/settings.json (抜粋)
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          { "type": "command",
            "command": "python3 .claude/scripts/black_hook.py" }
        ]
      }
    ]
  }
}
#!/usr/bin/env python3
# .claude/scripts/black_hook.py
import json, sys, subprocess, pathlib
data = json.load(sys.stdin)
fp = data.get("tool_input", {}).get("file_path", "")
if not fp.endswith(".py"):
    sys.exit(0)
path = pathlib.Path(fp)
if not path.exists():
    sys.exit(0)
# Black で整形(line-length はプロジェクト設定に従う)
r = subprocess.run(["black", "--quiet", str(path)],
                   capture_output=True, text=True)
if r.returncode != 0:
    print(f"[black] error: {r.stderr}", file=sys.stderr)
    sys.exit(1)
# 追加で ruff --fix をかけたい場合のみ
subprocess.run(["ruff", "check", "--fix", str(path)],
               capture_output=True)
print(f"[black] formatted: {fp}", file=sys.stderr)
sys.exit(0)

Go: gofmt / goimports

// .claude/settings.json (抜粋)
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "bash -c 'f=$(jq -r .tool_input.file_path); [[ \"$f\" == *.go ]] && goimports -w \"$f\" && gofmt -w \"$f\" || exit 0'"
          }
        ]
      }
    ]
  }
}

Go は gofmt がスタイル論争を終わらせているため、PostToolUse で安全に書き換えできます。goimports を先に通すと import 整理も済みます。

Ruby: rubocop

// .claude/settings.json (抜粋)
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "bash .claude/scripts/rubocop-hook.sh"
          }
        ]
      }
    ]
  }
}
#!/usr/bin/env bash
# .claude/scripts/rubocop-hook.sh
set -euo pipefail
file="$(jq -r '.tool_input.file_path // empty')"
[[ "$file" == *.rb ]] || exit 0
# -a = auto-correct、-A は激しめなので意図的に使わない
if bundle exec rubocop -a --force-exclusion "$file" >/dev/null 2>&1; then
  echo "[rubocop] fixed: $file" >&2
  exit 0
fi
echo "[rubocop] remaining offenses in: $file" >&2
exit 1
もっと詳しく: exit code の設計ポリシー

Claude Code の hook はexit code に意味がある。0=成功で素通し、1=警告(stderr が Claude に表示される)、2=ブロック(後続のツール実行を止める)。整形系は「失敗しても人間が手で直せる」ので 1 推奨。対して「危険コマンドのブロック」や「秘密鍵検出」のような security hook は 2 で強制停止します。フォーマッタを 2 にすると、些細な構文エラーで Claude が先に進めなくなり UX が崩壊するので注意。exit code の規約。

競合回避：差分が騒がしい問題

PostToolUse で勝手に書き換えると、Claude 自身が想定した差分と実ファイル差分がズレます。これを可視化したのが下のチャートです。

hook 構成別・差分ノイズの実測(10タスク平均)
PreToolUse で整形すると Claude が整形後コードを認識するため、次の Edit で old_string 不一致が激減する。

回避策は3つ。順にチェックしてください。

  • (1) 対象ファイルを matcher で絞る — .md(ドキュメント)まで Prettier が触ると想定外の改行変更が入る。拡張子で明示的に絞る。
  • (2) editorconfig と設定ファイルを揃える — Claude も .prettierrc / .eslintrc を参照するので、プロジェクトに置いてあれば整形基準が一致し差分が減る。
  • (3) 整形失敗をブロックにしない — exit 1 で warn 止まり。2 にすると構文エラー途中で Claude が止まって UX 破綻。
Before (競合あり)

PostToolUse で .md も含めて強制 Prettier

// 問題設定
"matcher": "Write|Edit",
"command": "prettier --write ."  // 全部対象
  • Claude が書いた直後に .md の改行が変わる
  • 次の Edit で old_string が見つからず失敗
  • 「なぜか直らない」が連発
After (競合回避)

拡張子を絞り、対象ファイルのみ整形

// 修正後
"matcher": "Write|Edit",
"command": "bash .claude/scripts/prettier-hook.sh"
// 中で拡張子 *.ts|*.tsx|*.js|*.jsx|*.json|*.css
// に絞り込み、.md は対象外
// prettier は --write "$file" で1ファイル限定

中級

自分の主要言語で整形 hook を1本書く

自分がよく使う言語を1つ選び、PostToolUse で整形を回す .claude/settings.json とシェルスクリプトを書いてください。以下を必ず含めること。

  1. matcher は Write|Edit
  2. シェルスクリプト内で拡張子チェック(対象外は素通し)
  3. 整形コマンドが失敗したら exit 1 (ブロックではなく警告)
  4. stderr に [formatter] formatted: ファイル名 を出力
模範解答を見る

回答例(Python プロジェクトで Black):

// .claude/settings.json
{
  "hooks": {
    "PostToolUse": [
      { "matcher": "Write|Edit",
        "hooks": [
          { "type": "command",
            "command": "bash .claude/scripts/black-hook.sh" }
        ] }
    ]
  }
}
#!/usr/bin/env bash
# .claude/scripts/black-hook.sh
set -euo pipefail
file="$(jq -r '.tool_input.file_path // empty')"
[[ -z "$file" ]] && exit 0
case "$file" in
  *.py) ;;
  *) exit 0 ;;
esac
if ! command -v black >/dev/null 2>&1; then
  echo "black not installed; skip" >&2
  exit 0
fi
if black --quiet "$file"; then
  echo "[black] formatted: $file" >&2
  exit 0
fi
echo "[black] failed: $file" >&2
exit 1   # 警告止まり(ブロックしない)

ポイント: (1) command -v で先に存在確認しないと hook が壊れて Claude 全体が動かなくなる、(2) 対象外ファイルは必ず素通し(exit 0)、(3) exit 2 は絶対に使わない(フォーマット失敗で Claude が止まる)。

応用

ESLint --fix を安全に混ぜる

Exercise 1 の整形 hook に、追加で eslint --fix も走らせる構成を書いてください。以下の条件を満たすこと。

  1. Prettier → ESLint --fix の順で実行する
  2. ESLint が 残存 error を報告したら exit 1(警告)、error がなければ exit 0
  3. ESLint が未インストールなら静かにスキップ
  4. 対象外拡張子(例: .md)は ESLint をかけない
模範解答を見る 
#!/usr/bin/env bash
# .claude/scripts/ts-format-hook.sh
set -euo pipefail
file="$(jq -r '.tool_input.file_path // empty')"
[[ -z "$file" ]] && exit 0

case "$file" in
  *.ts|*.tsx|*.js|*.jsx) ;;
  *.json|*.css|*.md)
    # Prettier だけかけて ESLint はスキップ
    npx --no-install prettier --write "$file" 2>/dev/null || true
    exit 0 ;;
  *) exit 0 ;;
esac

# 1. Prettier
if ! npx --no-install prettier --write "$file" 2>/dev/null; then
  echo "[prettier] failed: $file" >&2
fi

# 2. ESLint --fix (存在確認してから)
if npx --no-install eslint --version >/dev/null 2>&1; then
  if out=$(npx --no-install eslint --fix "$file" 2>&1); then
    echo "[eslint] clean: $file" >&2
    exit 0
  else
    echo "$out" >&2
    echo "[eslint] remaining errors in: $file" >&2
    exit 1
  fi
fi
exit 0

ポイント: (a) npx --no-install で勝手なダウンロードを防ぐ、(b) Prettier 失敗は警告のみで ESLint 側に進む、(c) ESLint の残存 error だけ exit 1 で Claude に見せる、(d) .md は Prettier だけ走らせて ESLint は対象外、(e) どのケースでも exit 2 は使わない(ユーザー体験が崩れる)。
