Hooksとは

なぜ必要か、使い所、自動化の威力

所要時間: 25分 Phase: Basics 難易度: 入門
この章のゴール

Hooks(=フックス)という仕組みを、「Claude Codeの動作の隙間に自分のシェルコマンドを差し込める」という直感で掴めるようになる。なぜ必要か、どんな場面で威力を発揮するか、他の自動化(Skill / Git hooks / CI)とどう棲み分けるか、そしてどんな危険があるか——25分で押さえて、Ch02以降でイベントや構文の細部に入っていきます。

バージョン注記

本章の記述は 2026年1月時点 の Claude Code における Hooks 実装に基づきます。設定キー名・イベント名・入力JSON形式は進化します。動かない時は Claude Code Hooks 公式ドキュメント を最優先で確認してください。

セキュリティ警告

Hooks は あなたの環境で任意のシェルコマンドを実行する 仕組みです。他人の settings.json を無検証で取り込んだり、curl | bash 相当のコマンドを設定したりすると、リポジトリを閲覧しただけで任意コード実行に至る可能性があります。本章でも繰り返し注意しますが、読めないコマンドをHookに貼らない を金科玉条にしてください。

Hook とは何か — 一言で

Hook とは、Claude Code が何かをするタイミング(ツールを使う前後、停止時、セッション開始時など)に合わせて、あなたが指定したシェルコマンドを自動実行する仕組み です。settings.json の hooks セクションにイベント名・対象マッチャ・コマンドを書くだけで機能します。

Hook が差し込まれるイメージ
Hook は「Hで示した隙間」に差し込まれるシェルコマンド。Claude本体を改造せず挙動を拡張できる。

もう少し厳密に言えば、Hook は イベント ごとに登録され、イベント発火時に matcher で絞り込まれた対象のみコマンドが起動します。コマンドは標準入力で stdin からイベント情報を受け取り、終了コードや標準出力でClaudeに結果を伝え返せるのが特徴です。

ひとことメタファ

ブラウザの拡張機能に似ています。「ページ表示前にスクリプトを差し込む」「送信前にフォーム値を変換する」といった割り込み点を使う発想です。Claude Code 側が「ここで割り込んでいいよ」と公開している点に、自分のシェルを繋げる——それだけ。

なぜ必要か — 「毎回頼む」から「勝手にやる」へ

Hook を入れる最大の動機は 「毎回お願いする工数ゼロ」 です。Claude に「保存前に Prettier かけて」「Bash 前に危険コマンドか確認して」と毎回口頭でお願いする代わりに、Hook として一度仕込めば以後自動。プロンプトも短くなり、忘却事故もなくなる。

Before(毎回頼む)

人間が覚えて指示

  • 「保存したら ESLint かけて」と毎回付ける
  • 「Bash 打つ前に確認して」と毎回念押し
  • 忘れると rm -rf 事故の温床
  • プロンプトが長くなってノイズ化
After(Hookで自動化)

仕組みで担保

  • PostToolUse(Write) で自動で ESLint
  • PreToolUse(Bash) で危険パターンをブロック
  • 忘れる可能性ゼロ、環境ごと設定
  • プロンプトは本質のみに集中
生産性ゲイン(Hook前後のタスク所要時間イメージ)
典型的な日常作業5種で、Hookありの場合の手作業相当時間の減少。実測値の目安であり環境依存。
もっと詳しく: なぜ Claude 本体に「毎回頼む」ではダメなのか

Claude 本体に指示することは統計的な遵守にすぎず、長い会話の途中で忘れる / コンテキストから外れる / プロンプトが希釈されるといった事故が起きます。Hook は 決定的(deterministic) に動く機構で、Claude の判断に依存しないため、保証 レベルが違います。「LLM は説得、Hook は強制」と覚えると整理しやすい。

具体的な使用シーン 6例

Hook の旨みが出るシーンは無数にありますが、代表的な6つを並べます。自分の日常作業で「また言うの…」と感じる作業は、ほぼ全部 Hook 化候補です。

  • 1. 自動フォーマット — PostToolUse(Write/Edit後)で Prettier / Black / gofmt を走らせる。Claude が書いた直後に整形が揃い、差分が綺麗に
  • 2. Lint / 型チェック — PostToolUse で ESLint / mypy / tsc を走らせ、エラーをstderrで返す。Claudeが自分で直しにかかる
  • 3. 危険コマンドのブロック — PreToolUse(Bash)で rm -rf / sudo などをパターン検出し exit 2 でブロック
  • 4. 作業終了通知 — Stop イベントで macOS 通知・Slack webhook・音声再生。長時間作業の気付きに
  • 5. 作業ログ / 監査 — すべての PostToolUse を jsonl に追記。後から「あの変更どこで入った?」を検索可能に
  • 6. 起動時の前準備 — SessionStart で git pull / npm i を自動、UserPromptSubmit で毎回プロジェクト状態を添付

Ch02で各イベントの発火タイミングを詳説。いまは「だいたいどこに何を仕込むか」の土地勘がつけばOK。

実際の settings.json はこんな形になります(Ch03で詳しく分解します)。まずは「ああ、こんなJSONなんだ」くらいで読んでください。

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "bash -c 'jq -r .tool_input.command | grep -E \"rm -rf|sudo\" && exit 2; exit 0'"
          }
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "bash -c 'file=$(jq -r .tool_input.file_path); [ -f \"$file\" ] && npx prettier -w \"$file\" 2>/dev/null || true'"
          }
        ]
      }
    ],
    "Stop": [
      {
        "matcher": "",
        "hooks": [
          { "type": "command", "command": "osascript -e 'display notification \"Claude stopped\" with title \"Claude Code\"'" }
        ]
      }
    ]
  }
}
今すぐ試す(30秒)

上記の Stop フックだけ、~/.claude/settings.json にコピペしてみてください(macOS前提)。Claude Codeセッションを立ち上げて、何か雑談して止めれば、ネイティブ通知が飛びます。仕組みが自分の環境で動いた体験を先に取るのが、このチュートリアルの肝です。

他の自動化との棲み分け — Skill / Git hooks / CI/CD

「自動化」と言っても手段は複数あり、似て非なる道具が並びます。Hook がちょうどいい領域 はどこか、4つの道具を軸で整理します。

仕組み トリガー 実行場所 主な用途 Claudeの挙動に介入?
Claude Code Hooks Claude内部のイベント(ツール前後など) 手元のシェル Claudeの挙動補強・ガードレール する(強制可)
Skill Claudeの判断(自発的) Claudeの実行環境 ドメイン知識・手順書の配布 する(確率的)
Git hooks git操作(commit/push等) 手元のシェル コード品質・コミット規約 しない
CI/CD PR/push(クラウド側) サーバー チーム基準の最終チェック しない
4つの自動化の守備範囲
Hooks は「Claudeが何か動いた瞬間」を捕まえる、最も手元に近いガードレール。CIは最も遅いが最も強固な共有基準。
棲み分けのコツ

Hook=Claude経由で行う作業の即時品質担保。Skill=Claudeが自発的に参照する手順書。Git hooks=手動コミット時の最終防衛線。CI=チームの最終承認。この4層で 深さ方向に多重化 するのが強い。Hookで弾いたミスは「CIまで行かず消える」ので、チーム全体のフィードバックループが短くなります。

もっと詳しく: Skill と Hook の決定的な違い

Skill は Claude が見て判断して使うかもしれないもの。Hook は Claude の意思と無関係に強制的に発火するもの。例えば「commit前に必ずテスト」を徹底したい時、Skill では「必ず」を保証できないが、Hookなら事実として必ず動く。逆に「このテーマで文章を書く時のルール集」のような柔軟な指針はSkillが向きます。「ルール=Hook、作法=Skill」がざっくりの目安です。

危険性と注意 — 任意コード実行である事実

便利さの裏側に、Hook は 「あなたの端末で任意のシェルコマンドを自動実行する」 という強力かつ危険な性質を持ちます。以下の4つのリスクは常に意識してください。

4つのリスク領域
Ch07で診断テクと対処法を、Ch04で各レシピに沿った対策を扱います。
絶対やってはいけない事

(1) SNS / Zenn / GitHubで他人が晒している settings.json を読まずにコピーしない。(2) command に curl ... | sh のようなリモート取得+即実行を書かない。(3) Hook本体スクリプトを書き込み可能な共有ディレクトリに置かない(書き換えられると任意コード実行)。(4) secrets(APIキー等)を Hook 内で echo しない(ログ化される)。

settings.json は user / project / local の3スコープがあり、projectスコープのHookは他人のコミット経由であなたの端末で実行されうることを強く覚えてください。レビュー不在で git pull して Claude を起動 → 初回発火で任意コード実行、は現実の脅威モデルです。

Exercise 1
入門

最小の Stop Hook を仕込んでみる

セッション終了時に通知を飛ばす最小 Hook を、自分の ~/.claude/settings.json(user scope)に追加して動作確認してください。

  1. ~/.claude/settings.json を開く(なければ作成)
  2. 下記の Stop Hook セクションを追加
  3. Claude Code を起動して雑談 → 終了操作で通知が飛ぶことを確認
  4. macOS以外(Linux/Windows)の人は osascript を各OSの通知コマンドに読み替え
模範解答を見る

最小構成はこれだけ。macOS の osascript を使います。

{
  "hooks": {
    "Stop": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "osascript -e 'display notification \"Session ended\" with title \"Claude Code\" sound name \"Glass\"'"
          }
        ]
      }
    ]
  }
}

Linux なら notify-send "Claude Code" "Session ended"、Windows(PowerShell) なら powershell -Command "[console]::beep(800,300)" などで代替可。動かない時は Ch07 のトラブルシュートへ。確認ポイント: (1) 通知が1回だけ飛ぶ。(2) Hookのcommandで誤タイポするとClaude終了時に自分のシェルにエラーが見えるので、そのログを見て直す癖をつける。

Exercise 2
応用

自分の業務から Hook 候補を3つ挙げる

Section 3 の6ユースケースを参考に、あなたが普段 Claude に毎回お願いしていること から Hook 化候補を3つ挙げ、各項目で下記を明記してください。抽象語(効率化・品質向上)だけで終わらせないこと。

  1. どのイベントに仕込むか(PreToolUse / PostToolUse / Stop / UserPromptSubmit / SessionStart のどれか)
  2. matcher(対象ツール名や正規表現)
  3. command の概要(実行するシェル)と期待する exit code の意味
  4. 最悪シナリオ(暴発するとどうなるか)と緩和策
模範解答を見る

例: TypeScript プロジェクト保守者のケース

【1. 保存時に Prettier】
  - event: PostToolUse
  - matcher: "Write|Edit"
  - command: npx prettier -w "$(jq -r .tool_input.file_path)"
  - exit code: 0で続行、非0でも続行(警告のみ)
  - 最悪: 巨大ファイルで遅延 → タイムアウト設定で緩和

【2. Bash 実行前の rm -rf ガード】
  - event: PreToolUse
  - matcher: "Bash"
  - command: jq -r .tool_input.command | grep -qE "rm -rf|sudo" && exit 2 || exit 0
  - exit code: 2で実行ブロック(Claudeに再考させる)
  - 最悪: 正当な rm まで弾く → 許可リストで例外運用

【3. Stop 時に git diff サマリを Slack 通知】
  - event: Stop
  - matcher: ""
  - command: git diff --stat | curl -X POST -d @- $SLACK_WEBHOOK
  - exit code: 0で固定、失敗はログのみ
  - 最悪: secretsがdiffに混入してSlackに流れる
    → .gitignore徹底 + grepで .env 検出したら送らない

ポイント: イベント選びとmatcherの設計が最初の山です。ここまで言語化できれば Ch03-Ch04 のレシピ集の理解速度が段違いになります。最終章 Ch08 では ここで挙げた3つを実装 するところまでやります。
