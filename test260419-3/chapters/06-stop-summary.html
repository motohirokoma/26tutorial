実践：Stop時にサマリ/通知

作業終了時のログ集約・Slack/macOS通知

所要時間: 30分 Phase: Practice 難易度: 中級
この章のゴール

長時間の Claude Code 作業が終わった時、PCの前に戻って気付く のでは遅い。本章では Stopイベント を使って、作業終了時に「何が変わったか(git diff --stat)」を取得し、macOS通知 / Slack Webhook / ブラウザ通知 の3経路で自分に届ける仕組みを構築する。さらに jsonl に作業履歴を貯めて、後から振り返れるようにする。本章の記述は 2026年1月時点 の情報です。

なぜ「終わった通知」が要るのか

Claude Code に「この機能を実装して、テストも通して、リファクタしてPR本文も書いておいて」と投げると、複数ターンにわたって10〜30分沈黙することがあります。あなたはその間に別ウィンドウでドキュメントを読み、Slackで雑談し、気付けば30分経って「…あ、Claude何してたっけ」となる。戻ってくるタイミングを失う のが生産性最大の敵です。

Stop通知なし vs あり
放置時間の合計は想像以上。通知1本で1日2時間近く戻ってくる。

Stopイベントは「Claudeが応答を終えた瞬間」に発火します。ここで git diff --stat や変更ファイル一覧を集めて、Slackに流せば、別の場所にいても「終わったよ」が届く。これが本章で組む仕組みです。

Stopイベントの発火タイミングと入力

Stopは メインAgentの応答完了時 に1回だけ発火します。SubagentStop と混同しないこと。Subagentが5つ並列で回って5回発火するのは SubagentStop の仕事で、Stop は最上位Agentのみ。

1セッションのイベント発火順
sequenceDiagram
  participant U as あなた
  participant C as Claude Code
  participant H as Stop Hook
  U->>C: 「〜を実装して」
  C->>C: Edit / Bash / Read (複数ツール実行)
  C-->>U: 応答完了
  C->>H: Stop イベント発火
  H->>H: git diff --stat 取得
  H->>H: Slack Webhook POST
  H-->>U: 通知が届く
応答完了 → Stop発火 → スクリプト実行 → 通知 の順。数秒で全部終わる。

Hookコマンドの stdin には JSON が渡されます。中身は session_id、transcript_path、cwd、hook_event_name などです。stop_hook_active フラグもあり、これが true なら「既にStop Hook経由で再起動している」ことを示すので、無限ループ防止 に使います。

{
  "session_id": "abc123...",
  "transcript_path": "/Users/you/.claude/projects/.../abc123.jsonl",
  "cwd": "/Users/you/work/my-project",
  "hook_event_name": "Stop",
  "stop_hook_active": false
}
セキュリティ警告

Hookは 任意のシェルコマンドを自動実行する仕組み です。stdin で渡される cwd や transcript_path をそのままシェル展開に使うとコマンドインジェクションの穴になります。必ずJSONパース → 変数代入 → クオート済みで使う(下の実装例はすべてこのパターン)。

サマリ取得: git diff --stat + α

「終わった」だけでなく「何が終わったか」を一緒に通知するのが本当に役立つ仕組み。git diff --stat を中核に、以下の4点セットを集めます:

  • 変更統計 — git diff --stat HEAD で foo.ts | 12 ++++-- の一覧
  • ファイル数と行数 — git diff --shortstat HEAD で 3 files changed, 42 insertions(+), 8 deletions(-)
  • 未追跡ファイル — git ls-files --others --exclude-standard で新規作成ファイル
  • 作業時間 — Stopイベントのタイムスタンプから推定(jsonlログと突合)

まずサマリだけ生成するスクリプトを作ります。~/.claude/hooks/stop-summary.sh に保存:

#!/usr/bin/env bash
# ~/.claude/hooks/stop-summary.sh
# Stopイベントで呼ばれる。stdinのJSONからcwdを抜いて差分サマリを作る。
set -euo pipefail

# stdinのJSONをパース (jq必須)
INPUT="$(cat)"
CWD="$(echo "$INPUT" | jq -r '.cwd // empty')"
SESSION="$(echo "$INPUT" | jq -r '.session_id // "unknown"')"

# cwd が git リポジトリでなければ静かに終了
[ -z "$CWD" ] && exit 0
cd "$CWD" || exit 0
git rev-parse --git-dir >/dev/null 2>&1 || exit 0

# サマリ組み立て
SHORT="$(git diff --shortstat HEAD 2>/dev/null || echo 'no changes')"
STAT="$(git diff --stat HEAD 2>/dev/null | head -20 || true)"
UNTRACKED="$(git ls-files --others --exclude-standard | head -10 || true)"
BRANCH="$(git rev-parse --abbrev-ref HEAD 2>/dev/null || echo '?')"

# 標準出力に結果を返す (通知スクリプトがこれを受け取る)
cat <<EOF
[$BRANCH] $SHORT
--- changed ---
$STAT
--- untracked ---
$UNTRACKED
EOF

このスクリプトを chmod +x で実行可能にして、~/.claude/settings.json の Hooks セクションに登録します:

{
  "hooks": {
    "Stop": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "~/.claude/hooks/stop-summary.sh | tee -a ~/.claude/logs/stop.log"
          }
        ]
      }
    ]
  }
}

matcher は Stop には不要(tool名でマッチングする PreToolUse/PostToolUse と違い、Stop はイベント自体が1種類)。tee -a で ~/.claude/logs/stop.log に追記しつつ Claude Code 側にも表示、という両立が効きます。

もっと詳しく: 巨大差分で通知が長くなりすぎる時

リファクタで50ファイル変えたりすると、git diff --stat の出力も長くなります。Slackやトースト通知では 1500文字を超えると読みにくいので、以下で丸める:

  1. git diff --stat | head -20 で上位20行に制限(上の例で実装済み)
  2. --shortstat だけを通知本文にし、詳細は添付またはリンクに
  3. ファイル数が20以上なら本文に「他N件」と要約だけ書く分岐を足す

MKの運用では 本文は shortstat 1行 + 変更ファイルTOP5 で、詳細は後述の jsonl ログに全部貯める、という切り分けが最も読みやすいです。

通知手段3種の比較

通知をどこに飛ばすか、は環境とスタイルで選びます。本章では macOS通知 / Slack Webhook / ブラウザ通知 の3経路を扱います。全部組み合わせても良いし、1本だけでも良い。

手段到達性リッチ度導入コストうるささ向いている人
macOS osascriptPCの前にいれば確実文字+音0分(標準搭載)中(通知センター)1台で作業/Mac利用者
Slack Webhookスマホ・他端末でも届くBlock Kit でリッチ5分(Webhook URL取得)低(専用channelで隔離可)離席が多い/チーム共有したい
ブラウザ通知ブラウザ開いてる時のみタイトル+本文+アイコン10分(ローカル受信用ページ)中ブラウザ中心で作業する人
Linux notify-sendPCの前にいれば確実文字+アイコン0-5分中Linux利用者(参考)
メール/SMTP確実だが遅延ありHTML可15分+低(後読み)非同期レビュー派(参考)

手段1: macOS osascript

Macの標準通知。osascript は macOS にデフォルト搭載なので 追加インストール0。音も鳴らせる。

#!/usr/bin/env bash
# ~/.claude/hooks/notify-mac.sh
set -euo pipefail
SUMMARY="$(~/.claude/hooks/stop-summary.sh)"
TITLE="Claude Code 完了"
# ダブルクオート内のダブルクオートをエスケープ
SAFE="$(printf '%s' "$SUMMARY" | sed 's/"/\\"/g' | head -c 200)"
osascript -e "display notification \"$SAFE\" with title \"$TITLE\" sound name \"Glass\""

音は Glass / Hero / Ping / Pop / Purr / Sosumi / Submarine 等を指定可能。うるさければ sound name を外せば無音通知になります。

手段2: Slack Incoming Webhook

スマホに届くのが最大の利点。席を立っていても「終わった」が分かる。手順: Slack の Admin画面 → Apps → Incoming Webhooks を追加 → 投稿先channelを選ぶ → URL発行。URLは漏洩すると誰でも投稿できるので ~/.zshrc や .env に環境変数で退避し、絶対にGitに入れない。

#!/usr/bin/env bash
# ~/.claude/hooks/notify-slack.sh
set -euo pipefail

: "${SLACK_WEBHOOK_URL:?SLACK_WEBHOOK_URL not set}"

SUMMARY="$(~/.claude/hooks/stop-summary.sh)"
# JSONエスケープを jq に任せる
PAYLOAD="$(jq -n --arg text "$SUMMARY" '{
  text: "Claude Code 作業完了",
  blocks: [
    { "type": "header", "text": { "type": "plain_text", "text": "✅ Claude Code 完了" } },
    { "type": "section", "text": { "type": "mrkdwn", "text": "```\($text)```" } }
  ]
}')"

curl -sS -X POST -H 'Content-Type: application/json' \
  --data "$PAYLOAD" "$SLACK_WEBHOOK_URL" > /dev/null

settings.json 側は環境変数 SLACK_WEBHOOK_URL を参照します。Hook実行環境は ログインシェルの初期化ファイルを読まない 場合があるので、コマンド内で明示的に source するのが安全:

{
  "hooks": {
    "Stop": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "source ~/.claude/env && ~/.claude/hooks/notify-slack.sh"
          }
        ]
      }
    ]
  }
}

~/.claude/env に export SLACK_WEBHOOK_URL="https://hooks.slack.com/services/XXX/YYY/ZZZ" の1行を書いて chmod 600 しておくのがMKの運用。

手段3: ブラウザ通知

ブラウザが開いている時に Notification API で通知を出す。ローカルにWebSocketサーバを立てる方式と、ポーリングで localhost からJSONを取りに行く方式があり、ここではシンプルに ファイル更新 → ブラウザ拡張でtoast の構成を推奨。詳細実装は長くなるので MDN Notification API を参照してください。

Slack推し

離席・別端末・チーム共有に強い

  • スマホ/iPadでも受信できる
  • 専用channelで履歴が溜まる
  • Block Kitでサマリを構造化できる
Slack注意点

Webhook URL漏洩に注意

  • URLはGit/Qiita等に絶対入れない
  • 通知が多いとchannelが荒れる
  • オフラインだと遅延して届く

jsonl で作業履歴を貯める

通知は「今」届けば価値があるが、履歴は蓄積すると別の価値 が生まれます。1週間溜めると「自分は何に時間使っているか」が見える。jsonl(1行1JSON)形式なら jq で切り刻めて集計にも便利。

Stop → jsonl追記 → 後日集計の流れ
graph LR
  S["Stop イベント"] --> G["git diff --stat"]
  G --> J["JSON1行化"]
  J --> F["~/.claude/logs/sessions.jsonl"]
  F --> Q["jq で集計"]
  Q --> R1["週次ダッシュボード"]
  Q --> R2["ファイル別頻度"]
  Q --> R3["セッション時間分布"]
リアルタイム通知と別経路で永続ログを作ると振り返りに効く。

jsonl追記スクリプト:

#!/usr/bin/env bash
# ~/.claude/hooks/log-session.sh
set -euo pipefail

INPUT="$(cat)"
CWD="$(echo "$INPUT" | jq -r '.cwd // empty')"
SESSION="$(echo "$INPUT" | jq -r '.session_id // "unknown"')"
NOW="$(date -u +%Y-%m-%dT%H:%M:%SZ)"

# サマリ集計 (git repoなら)
FILES=0
INS=0
DEL=0
BRANCH=""
if cd "$CWD" 2>/dev/null && git rev-parse --git-dir >/dev/null 2>&1; then
  BRANCH="$(git rev-parse --abbrev-ref HEAD 2>/dev/null || echo '')"
  STATS="$(git diff --shortstat HEAD 2>/dev/null || echo '')"
  FILES="$(echo "$STATS" | grep -oE '[0-9]+ files? changed' | grep -oE '[0-9]+' || echo 0)"
  INS="$(echo "$STATS" | grep -oE '[0-9]+ insertion' | grep -oE '[0-9]+' || echo 0)"
  DEL="$(echo "$STATS" | grep -oE '[0-9]+ deletion' | grep -oE '[0-9]+' || echo 0)"
fi

# 1行JSONとして追記
mkdir -p ~/.claude/logs
jq -nc \
  --arg ts "$NOW" --arg session "$SESSION" --arg cwd "$CWD" \
  --arg branch "$BRANCH" --argjson files "${FILES:-0}" \
  --argjson ins "${INS:-0}" --argjson del "${DEL:-0}" \
  '{ts:$ts, session:$session, cwd:$cwd, branch:$branch, files:$files, insertions:$ins, deletions:$del}' \
  >> ~/.claude/logs/sessions.jsonl

settings.json で Stop に複数Hook登録するとチェーンできます:

{
  "hooks": {
    "Stop": [
      {
        "matcher": "",
        "hooks": [
          { "type": "command", "command": "~/.claude/hooks/log-session.sh" },
          { "type": "command", "command": "source ~/.claude/env && ~/.claude/hooks/notify-slack.sh" }
        ]
      }
    ]
  }
}

この順にすると「記録が先、通知が後」。記録は必ず残り、通知失敗しても履歴は保全される。1週間後の集計はこうやります:

# 今日の合計変更行数
jq -s 'map(select(.ts | startswith("2026-04-18"))) | map(.insertions + .deletions) | add' ~/.claude/logs/sessions.jsonl

# セッション数 TOP プロジェクト
jq -s 'group_by(.cwd) | map({cwd:.[0].cwd, count:length}) | sort_by(-.count) | .[:5]' ~/.claude/logs/sessions.jsonl
1週間のjsonlから見えたもの(MKの実データ例)
Claude Codeでの変更行数を曜日別に集計。火/木に集中、土日は少なめ。

セキュリティ再確認

Hookは 任意コマンドを自動実行 します。本章のスクリプトは全て公開コードを元にしていますが、自分で追記する場合は必ず set -euo pipefail を入れ、stdin由来の値は必ずjqでパースしてから変数経由で使うこと。eval や裸の $VAR 展開は禁止です。Webhook URLは絶対にGitにコミットしない。

もっと詳しく: 通知が多すぎて逆に邪魔になる時

Claude Code を1日10回以上使う人は、全部のStopで通知が飛ぶとうるさい。以下の条件で抑制する実装が有用:

  1. 作業時間しきい値: セッション開始→Stopが5分未満なら通知しない(sessions.jsonlを引いて判定)
  2. 変更が無ければ通知しない: git diff --shortstat が空なら早期exit
  3. 時間帯制限: 深夜/休日はmacOS通知のみ、Slackには飛ばさない
  4. プロジェクト限定: cwd が特定のディレクトリ配下の時だけ通知

MKは「5分以上 かつ 変更あり」で抑えると、1日5-7本に落ち着いて丁度よくなりました。

Exercise 1
入門

macOS通知のみでStop Hookを1本仕込む

Slackはまだ設定しない前提で、最小構成の「作業完了トースト通知」を動かします。

  1. ~/.claude/hooks/notify-mac.sh を本章のサンプル通り作成して chmod +x
  2. ~/.claude/settings.json の hooks.Stop に1エントリ追加
  3. Claude Code で適当なディレクトリに cd し「helloと出力して」と依頼
  4. 応答完了後、macOSの右上に通知が出ることを確認
  5. 出ない場合はシステム設定 → 通知 → スクリプトエディタ の通知許可を確認
模範解答を見る

~/.claude/settings.json に以下を追加(既存ファイルがあればマージ):

{
  "hooks": {
    "Stop": [
      {
        "matcher": "",
        "hooks": [
          { "type": "command", "command": "~/.claude/hooks/notify-mac.sh" }
        ]
      }
    ]
  }
}

これだけで動きます。確認のコツは最初は通知スクリプトの中身を osascript -e 'display notification "test" with title "hi"' に単純化 してmacOS側の権限問題を切り分けること。いきなり stop-summary.sh を噛ますと、git非リポジトリで早期exitして「動かない」ように見えることがあります。確認が取れたら本番の notify-mac.sh に戻す、が安全な順序です。

Exercise 2
応用

Slack通知 + jsonl記録 + 抑制ロジックを全部乗せる

実運用に耐えるフル構成を自PCに構築します。所要15分。

  1. Slack管理画面で Incoming Webhook を取得し ~/.claude/env に退避(chmod 600)
  2. 本章の stop-summary.sh / notify-slack.sh / log-session.sh を配置
  3. notify-slack.sh の冒頭に「変更0ならexit 0」の分岐を追加
  4. settings.json の Stop に log-session.sh → notify-slack.sh の順で2つチェーン
  5. 適当なリポジトリで軽い編集依頼を出し、Slackに通知が届くこと・sessions.jsonl に1行追加されることを確認
  6. 同じリポジトリで「出力だけの指示」(ファイル変更なし)を出し、Slack通知が出ないことを確認
模範解答を見る

抑制ロジック入り notify-slack.sh の冒頭はこう書きます:

#!/usr/bin/env bash
set -euo pipefail
: "${SLACK_WEBHOOK_URL:?SLACK_WEBHOOK_URL not set}"

# 変更が無ければ通知しない (stop-summary.shの出力で判定)
SUMMARY="$(~/.claude/hooks/stop-summary.sh)"
if echo "$SUMMARY" | grep -q 'no changes'; then
  exit 0
fi

# 本文を200文字で丸める(Slack可読性)
SHORT="$(printf '%s' "$SUMMARY" | head -c 1500)"
PAYLOAD="$(jq -n --arg text "$SHORT" '{
  text: "Claude Code 作業完了",
  blocks: [
    { "type": "header", "text": { "type": "plain_text", "text": "✅ Claude Code 完了" } },
    { "type": "section", "text": { "type": "mrkdwn", "text": "```\($text)```" } }
  ]
}')"
curl -sS -X POST -H 'Content-Type: application/json' \
  --data "$PAYLOAD" "$SLACK_WEBHOOK_URL" > /dev/null

settings.json の完成形:

{
  "hooks": {
    "Stop": [
      {
        "matcher": "",
        "hooks": [
          { "type": "command", "command": "~/.claude/hooks/log-session.sh" },
          { "type": "command", "command": "source ~/.claude/env && ~/.claude/hooks/notify-slack.sh" }
        ]
      }
    ]
  }
}

検証ポイントは3つ: (1) jsonl は「変更なし」でも追記される(履歴の欠損防止) (2) Slack は「変更なし」時に飛ばない (3) Webhook URL は env ファイル側に退避されて settings.json には残らない。これでチームに settings.json を共有しても事故りません。1週間運用すると sessions.jsonl が 30-50行 溜まり、冒頭の jq ワンライナーで自分の働き方が数字で見えてきます。
