最終演習：主戦場ツールをMCP化

自分の業務ツールを1つ選び、Claude Codeから操作できるようにする統合課題

所要時間: 25分 Phase: Practice 難易度: 応用
本章の前提

本章の記述は 2026年1月時点。MCPサーバー公式カタログ・SDK APIは進化が早いので、着手前に modelcontextprotocol.io と Claude Code公式ドキュメント を必ず最新版で確認してください。

7章の学びをここに束ねる

ここまでの7章で、MCPの思想(Ch01)、.mcp.jsonとスコープ(Ch02)、公式サーバーカタログ(Ch03)、GitHub(Ch04)・Filesystem/他(Ch05)の実践、カスタムサーバー自作(Ch06)、そしてトラブルシュート(Ch07)を順に積んできました。この最終章では、そのすべてを 1つの「自分の主戦場」に向けて統合します。

目標は 「日常で使う業務ツールのうち1つを選び、Claude Code から操作できるようにする」こと。公式サーバーがあればそれを繋ぎ、なければカスタムMCPを自作します。設計→認証→実装→動作確認まで一気通貫で通す演習です。

Tips

この演習の肝は「完璧なMCPサーバーを作る」ことではなく、「自分の手のツールにClaudeが届く」という成功体験を1つ作ることです。最初のユースケースは3つで十分。まず動かして、そこから育てましょう。

問題: 主戦場ツールをMCP化する設計書を書く

以下の前提条件で、ゼロから1つの設計書を作ってください。テキストファイルでもObsidianノートでも構いません。

段階ヒント (行き詰まったら開く)

いきなり全部開くと「答え写し」になります。まず自分で1回設計してみて、詰まった項目だけ開いてください。

もっと詳しく: ヒント1 — ツール選定の4基準 (Ch01 / Ch03)

「週5以上触る」だけでは選べない時の追加基準:

  1. 使用頻度 — 1日3回以上触る(低頻度ならROIが出ない)
  2. APIの存在と品質 — 公式REST/GraphQL APIがあり、rate limitが緩い
  3. 自動化余地 — 「検索・集計・転記」など定型操作がある(新規創作寄りは向かない)
  4. 認証の入手しやすさ — 自分の権限で token/API keyを発行できる

候補例と評価(1=低、5=高):

ツール頻度API自動化余地認証MCP適性
Obsidian (ローカルVault)5ファイルI/O5不要★★★★★
Notion5REST公式4Integration Token★★★★★
Jira5REST公式4API Token★★★★
Linear5GraphQL5API Key★★★★★
Figma3REST(read寄り)3Personal Token★★★
社内独自Web5なし/不安定2社内SSO★★
Tips

最初の一台は Obsidianローカルか Linearが最適です。Obsidianは認証不要・Linearは公式GraphQLが秀逸。Notionも定番ですが、database schemaが複雑なので2台目以降が無難です。

もっと詳しく: ヒント3 — 公式がない時のカスタム設計 (Ch06)

Ch06の最小ひな型を土台に、3つのツール関数を足す構成が最短です。

設計テンプレート(1ツールあたり):

- name: 英数ケバブケース (例 "obsidian_search_notes")
- description: 1行で「何をする/何が返る」
- inputSchema:
    type: object
    properties:
      query:       { type: string, description: "検索文字列" }
      vault_path:  { type: string, description: "Vaultのルートパス" }
      max_results: { type: integer, default: 10 }
    required: [query, vault_path]

3ツールの型(推奨):

種類例(Obsidian)例(Linear)例(Notion)
Readsearch_noteslist_issuesquery_database
Writeappend_daily_notecreate_issueappend_page_block
集計/要約tag_overviewsprint_summarypage_stats
Warning

write系ツールには 必ず "dry_run" パラメータを入れてください。Claudeが誤って本番に書き込む事故を防ぎます。初期はdefault: trueにしておくくらいが安全です(Ch07)。

もっと詳しく: ヒント4 — 認証の準備と渡し方 (Ch02 / Ch04)

認証情報は 3重に分離して管理:

  1. 発行元で最小権限 — Linearなら read + issue:write だけ、Notionなら特定DBのみ共有
  2. OSの秘密領域 — macOS Keychain / 1Password CLI / .env(gitignore)
  3. 子プロセス環境変数として渡す — .mcp.jsonのenvで${VAR}参照

もっと詳しく: ヒント5 — 初期3ユースケースの決め方

「何でもできるMCP」を作るのは失敗の元。最初は3つに絞る。選ぶ基準:

  1. 毎日やる読み取り 1つ(朝一で状況把握する系)
  2. 週1〜数回やる書き込み 1つ(定型テンプレで追記する系)
  3. 月1でやる集計/棚卸し 1つ(普段面倒だが価値があるもの)

Obsidianでの例:

Read(毎日)
search_notes(キーワード検索)
Write(週2)
append_daily_note(Dailyに追記)
集計(月1)
tag_overview(タグごとのノート数)

Linear での例:

Read(毎日)
list_issues(自分のアサイン一覧)
Write(週2)
create_issue(定型テンプレで起票)
集計(スプリント末)
sprint_summary(完了率・遅延)
Insight

「3つ絞り」はただの制約ではなく、MCPサーバーを「育てる対象」にするための設計原則。最初から10個あるサーバーはメンテされず、3個のサーバーは1ヶ月後に5個に成長します。

3. サーバー実装(TypeScript・最小構成)

// obsidian-mcp/src/server.ts (Ch06のひな型を拡張)
import { Server } from "@modelcontextprotocol/sdk/server/index.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import {
  ListToolsRequestSchema,
  CallToolRequestSchema,
} from "@modelcontextprotocol/sdk/types.js";
import fs from "node:fs/promises";
import path from "node:path";

const VAULT = process.env.OBSIDIAN_VAULT_PATH!;
if (!VAULT) throw new Error("OBSIDIAN_VAULT_PATH is required");

const server = new Server(
  { name: "obsidian-mcp", version: "0.1.0" },
  { capabilities: { tools: {} } }
);

server.setRequestHandler(ListToolsRequestSchema, async () => ({
  tools: [
    {
      name: "search_notes",
      description: "Vault内のMarkdownをキーワード検索する",
      inputSchema: {
        type: "object",
        properties: {
          query: { type: "string" },
          max_results: { type: "integer", default: 10 },
        },
        required: ["query"],
      },
    },
    {
      name: "append_daily_note",
      description: "今日のDaily Noteにテキストを追記する",
      inputSchema: {
        type: "object",
        properties: {
          text: { type: "string" },
          dry_run: { type: "boolean", default: true },
        },
        required: ["text"],
      },
    },
    {
      name: "tag_overview",
      description: "Vault全体のタグ別件数を集計",
      inputSchema: {
        type: "object",
        properties: { top_n: { type: "integer", default: 20 } },
      },
    },
  ],
}));

async function walk(dir: string): Promise<string[]> {
  const out: string[] = [];
  for (const e of await fs.readdir(dir, { withFileTypes: true })) {
    const p = path.join(dir, e.name);
    if (e.isDirectory()) out.push(...(await walk(p)));
    else if (e.name.endsWith(".md")) out.push(p);
  }
  return out;
}

server.setRequestHandler(CallToolRequestSchema, async (req) => {
  const { name, arguments: args } = req.params;
  if (name === "search_notes") {
    const files = await walk(VAULT);
    const q = String(args.query).toLowerCase();
    const hits: { file: string; excerpt: string }[] = [];
    for (const f of files) {
      const body = await fs.readFile(f, "utf8");
      const i = body.toLowerCase().indexOf(q);
      if (i >= 0) hits.push({ file: path.relative(VAULT, f), excerpt: body.slice(Math.max(0, i - 40), i + 80) });
      if (hits.length >= (args.max_results ?? 10)) break;
    }
    return { content: [{ type: "text", text: JSON.stringify(hits, null, 2) }] };
  }
  if (name === "append_daily_note") {
    const today = new Date().toISOString().slice(0, 10);
    const fp = path.join(VAULT, "Daily", `${today}.md`);
    const line = `\n- ${new Date().toLocaleTimeString()} ${args.text}\n`;
    if (args.dry_run !== false) {
      return { content: [{ type: "text", text: `[dry_run] would append to ${fp}:\n${line}` }] };
    }
    await fs.mkdir(path.dirname(fp), { recursive: true });
    await fs.appendFile(fp, line);
    return { content: [{ type: "text", text: `appended to ${fp}` }] };
  }
  if (name === "tag_overview") {
    const files = await walk(VAULT);
    const counts = new Map<string, number>();
    for (const f of files) {
      const body = await fs.readFile(f, "utf8");
      for (const m of body.matchAll(/(?:^|\s)#([\w\/-]+)/g)) {
        counts.set(m[1], (counts.get(m[1]) ?? 0) + 1);
      }
    }
    const top = [...counts.entries()].sort((a, b) => b[1] - a[1]).slice(0, args.top_n ?? 20);
    return { content: [{ type: "text", text: JSON.stringify(Object.fromEntries(top), null, 2) }] };
  }
  throw new Error(`unknown tool: ${name}`);
});

const transport = new StdioServerTransport();
await server.connect(transport);

8. 考え方の解説

このモデル解答の肝は 3つの設計判断です。

  1. あえて「Obsidianローカル」を選ぶ — 認証ゼロ・完全ローカル完結で、Ch07のトラブル切り分けが最も楽。初めての自作MCPで環境要因の障害を踏まない。
  2. 3ツール絞り+dry_run既定 — 書き込み系の事故を構造的に防ぎつつ、育てる余白を残す。「完璧を作らない」設計。
  3. パスも環境変数化 — チームに配る時に.mcp.jsonの編集不要、各自のVaultに向くだけで動く。project スコープ運用との相性が最高。
Warning

本モデルは Vaultに秘密情報を書いていない前提です。社外秘を含むVaultでは、search_notesのexcerpt長を短くする・特定タグ(#private)を含むファイルは除外する等の リダクションを実装してください(Ch07のセキュリティ注意)。

この設計書が描けたなら、あなたは「他人が作ったMCPを繋ぐ人」から「自分の仕事に合わせてMCPを設計できる人」に変わっています。

応用課題: 条件を変えて計画を拡張する

モデルが解けたら、次は「条件を変えたら何がどう変わるか」を体感しましょう。2つ目のツールを加えたり、チームに配布するだけで、設計は別物になります。

模範解答を見る

(1) 配布形式の選び方

  • gitリポ(社内GitHub): 最も楽。projectスコープの .mcp.json も一緒に commit できる。初手はこれ
  • npm private registry: 複数プロジェクト横断で使うなら。バージョン管理が効く
  • zip配布: 非エンジニアや閉鎖環境だけ。自動更新できないので最終手段

(2) README必須5項目

  1. 前提環境(Node.jsのバージョン、OS、Claude Codeバージョン)
  2. インストール手順(git clone → npm i → npx tsc)
  3. 環境変数の一覧と取得方法(OBSIDIAN_VAULT_PATH など)
  4. .mcp.jsonへの追加例(コピペ可能な形)
  5. 動作確認手順(claude mcp list で connected が出る画面)
