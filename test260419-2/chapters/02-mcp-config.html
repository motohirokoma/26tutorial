Claude CodeにMCPを追加

.mcp.jsonの書き方、スコープ(user/project/local)、認証

所要時間: 40分 Phase: Basics 難易度: 初級
この章のゴール

Ch01でMCPの概念を掴んだあなたは、次に「実際にClaude Codeへサーバーを接続する」フェーズに入ります。本章では .mcp.json の構造、スコープ、stdio transport と http transport の違い、認証トークンの扱い、そして claude mcp CLIを使った実地操作までを身につけます。最終ハンズオンでは filesystem サーバーを自分のマシンに追加します。

本章の記述時点

以下の内容は2026年1月時点のMCP仕様とClaude Code CLI挙動に基づきます。MCP仕様・公式サーバー・CLIコマンドは頻繁に更新されるため、最新情報は modelcontextprotocol.io と docs.claude.com/claude-code を参照してください。

.mcp.json の全体像

MCPサーバーをClaude Codeに認識させる方法はいくつかありますが、根幹となるのはJSONファイルにサーバー定義を書くことです。Claude Codeは起動時に複数の場所からJSON設定を読み込み、マージして使います。まずは最小形を見てみましょう。

{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/Users/koma/Desktop"]
    }
  }
}

わずか数行ですが、これがMCP設定の原子単位です。キー mcpServers の下に、任意の名前(ここでは filesystem)を付けたサーバー定義が1つ以上並びます。各サーバー定義は「どう起動するか」を記述するオブジェクトです。

.mcp.json の構造(stdio transport の場合)
最小構成は command + args の2キー。env と transport は必要に応じて追加。

フィールドは以下の通りです:

  • command — サーバープロセスを起動する実行コマンド。npx / node / python / uv / 自作バイナリパスなど。
  • args — commandに渡す引数の配列。パッケージ名・フラグ・対象ディレクトリパスなど。
  • env — サーバープロセスに渡す環境変数(トークンやAPIキーなど)。省略可。
  • transport — "stdio"(デフォルト)か "http"。http時は別構造になる(次節)。
  • disabled / timeout — 一時無効化や起動タイムアウト秒数(optional)。

初心者の落とし穴: JSON末尾のカンマ。サーバーを2つ以上並べるとき「最後のオブジェクトの後にカンマを付けない」というJSON構文を忘れがちです。Claude Codeは起動時に「Invalid JSON at line X」と出て静かに失敗します。不安ならエディタで .mcp.json を開き、赤い波線が出ていないか確認してから保存しましょう。

スコープ — user / project / local

MCP設定を「どのファイルに書くか」で、適用範囲と共有相手が変わります。Claude Codeは3つのスコープを提供します。

① user

ユーザー全体

  • ~/.claude.json(ホーム)
  • 全プロジェクトで使える
  • 自分のマシンでだけ有効
  • git共有されない
② project

プロジェクト共有

  • <repo>/.mcp.json
  • そのプロジェクトでだけ有効
  • リポジトリにコミットしてチーム共有
  • トークンは書かない(envで渡す)
③ local

プロジェクト×自分だけ

  • <repo>/.claude/settings.local.json
  • そのプロジェクトだけ・自分だけ
  • .gitignore対象(共有されない)
  • 個人トークンはここ
3スコープの読み込み順とマージ

同名のサーバーが複数スコープに存在する場合、local > project > user の優先順で上書き。

具体的なファイル配置例

# ~/.claude.json           ← user scope (全プロジェクトで常時有効)
{
  "mcpServers": {
    "filesystem-desktop": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/Users/koma/Desktop"]
    }
  }
}

# ~/myrepo/.mcp.json       ← project scope (チーム共有・git管理)
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": { "GITHUB_PERSONAL_ACCESS_TOKEN": "${GITHUB_PAT}" }
    }
  }
}

# ~/myrepo/.claude/settings.local.json  ← local scope (自分のトークン)
{
  "env": {
    "GITHUB_PAT": "ghp_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx"
  }
}
スコープ選択の原則

どこでも使うサーバー → user(例: 自分のDesktopを読むfilesystem)。チームで共有するサーバー定義 → project(例: このリポで使うGitHub MCP)。個人のトークン・認証情報 → local。プロジェクトの .mcp.json に生のトークンを書いて git push するのは最悪の事故パターンです。必ずlocal scopeかシェルの環境変数経由で渡しましょう。

もっと詳しく: 環境変数参照 ${VAR} 記法

.mcp.json の文字列値では ${VAR_NAME} という形でシェル環境変数を参照できます。これにより「JSON本体に秘密を書かず、シェル or local scopeで注入する」という分離が可能になります。例:

"env": {
  "GITHUB_PERSONAL_ACCESS_TOKEN": "${GITHUB_PAT}",
  "SLACK_BOT_TOKEN": "${SLACK_TOKEN:-}"
}

${VAR:-default} の書式でデフォルト値も指定できます(シェルのパラメータ展開と同じ)。チーム配布時は「このrepoを使うには GITHUB_PAT を設定してください」とREADMEに書くだけで済みます。

stdio と http — 2つのトランスポート

MCPサーバーとClaude Codeの通信路には大きく2種類があります。どちらを使うかは「サーバーをローカルで起動するのか/リモートに立てるのか」で決まります。

項目stdio transporthttp transport
通信方式標準入出力(stdin/stdout)HTTP POST + SSE
サーバーの場所ローカルプロセス(都度起動)ローカル or リモートURL
認証環境変数で渡すBearer token / OAuth
起動/終了Claude Codeが自動管理常駐サービス側で管理
向いている用途個人ツール・ローカル操作・CLI系サーバーSaaS連携・社内共有・マネージド
設定の難易度低(command+args)中(URL+認証設計)
stdio と http の通信フロー
stdioは「サーバーをClaude Codeが起動する」、httpは「既に立っているサーバーに接続する」。

http transport の設定例は次のようになります:

{
  "mcpServers": {
    "my-remote-api": {
      "transport": "http",
      "url": "https://mcp.example.com/api",
      "headers": {
        "Authorization": "Bearer ${MY_REMOTE_TOKEN}"
      }
    }
  }
}

command/argsの代わりに url と headers を指定します。2026年1月時点では stdioが圧倒的多数派 ですが、Cloudflare Workers MCPやリモートSaaSの浸透により、Remote MCPは急増しています。

初学者向けの選び方: 迷ったらstdioを選びましょう。本章でも以降の章でも、特記なき限り stdio で進めます。httpはOAuth対応サーバーや企業SaaSと連携する時に登場しますが、Ch04〜05の範囲では不要です。

具体的なサーバー追加例(stdio)

代表的なMCPサーバーを .mcp.json に追加する実例を見ていきます。以下はすべて 2026年1月時点で動作する 設定です(パッケージ名の変更があれば公式READMEを優先)。

例1: filesystem — ローカルファイルを読み書き

{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-filesystem",
        "/Users/koma/Documents",
        "/Users/koma/Desktop/claude"
      ]
    }
  }
}

argsの末尾に読み書きを許可するディレクトリを複数指定できます。指定外のパスへのアクセスはサーバー側で拒否されるため、サンドボックスとして機能します。

例2: github — Issue/PR操作

{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "${GITHUB_PAT}"
      }
    }
  }
}

GitHub Personal Access Token(PAT) が必要です。repo / read:org など必要最小のscopeだけ付与しましょう(詳細はCh04)。

例3: fetch — URLを取得して要約

{
  "mcpServers": {
    "fetch": {
      "command": "uvx",
      "args": ["mcp-server-fetch"]
    }
  }
}

Python製のMCPサーバーは uvx(uv の実行専用コマンド) で動かすのがデフォルトです。pipインストール不要で都度ダウンロードして起動します。

例4: 複数サーバーを同時登録

{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/Users/koma/Desktop"]
    },
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": { "GITHUB_PERSONAL_ACCESS_TOKEN": "${GITHUB_PAT}" }
    },
    "fetch": {
      "command": "uvx",
      "args": ["mcp-server-fetch"]
    }
  }
}

キー名(サーバーID)は任意の英小文字+ハイフン。Claude Code上では @filesystem / @github のように参照されるので、自分が覚えやすい名前を付けましょう。

もっと詳しく: npx -y の意味と挙動

npx -y <package> は「このnpmパッケージをグローバルインストールせずに都度ダウンロードして実行し、確認プロンプトをスキップする」という意味です。-y がないと初回実行時に「このパッケージを取得していい?(y/N)」とstdinで聞かれ、Claude Codeからは応答できずハングします。必ず -y を付けるのが鉄則。なお毎回ネットから取得するわけではなく、npxがローカルにキャッシュします。

認証 — token をどこに置くか

GitHub / Slack / Notion / DB系など、認証が必要なMCPサーバーは多数あります。原則は「生のトークンを .mcp.json(project scope) に書かない」ことです。代わりに以下3つの手段のどれかを使います。

  1. シェル環境変数: ~/.zshrc や ~/.bashrc に export GITHUB_PAT=ghp_... と書き、.mcp.jsonでは ${GITHUB_PAT} で参照。
  2. local scope: .claude/settings.local.json の env に書く。.gitignore されるので共有されない。
  3. OS keychain連携: 対応サーバーなら OS keyring から取得(例: op run で1Password注入)。
トークン漏洩リスクの比較
警告: 生トークンpushの事故

.mcp.json(project scope) に "GITHUB_PERSONAL_ACCESS_TOKEN": "ghp_xxxxx" と直書きしてコミットし、GitHubにpushするとGitHub側が自動検知してトークンを即時revokeします(secret scanning)。しかし検知まで数秒〜数分あり、その間にbot/攻撃者が読み取って悪用する事例が報告されています。revoke通知メールが来た時点で「すでに使われた可能性」を疑い、即ログ監査を実施してください。

OAuth系サーバー(http transport)

Notion / Linear / Atlassian などの新しめのMCPサーバーは、tokenではなくOAuth 2.0で認証します。この場合 .mcp.json には url だけ書き、Claude Code起動時にブラウザが開いて認可フローが走ります。

{
  "mcpServers": {
    "notion": {
      "transport": "http",
      "url": "https://mcp.notion.com/mcp"
    }
  }
}

認可後のトークンはClaude Code本体が OS keychain に保存するため、JSONには一切秘密が残りません。OAuth対応はRemote MCPの強み。

実務ルール: 「PATは3ヶ月ごとに再発行」「リポジトリscopeは最小」「チーム共有にはOAuth or machine userを使う」の3つを守れば、多くの事故は防げます。PATのローテーションは面倒ですが、流出した時の被害規模を考えると安い保険です。

claude mcp CLI — 手でJSONを触らずに管理

Claude Code CLI には claude mcp サブコマンド群があり、.mcp.jsonを直接編集せずにサーバーを追加・一覧・削除できます。初心者はこちらから入るのが楽です。

add — サーバー追加

# 基本形 (user scope に追加)
claude mcp add filesystem -s user -- npx -y @modelcontextprotocol/server-filesystem /Users/koma/Desktop

# project scope に追加(repo の .mcp.json に書き込む)
claude mcp add github -s project -e GITHUB_PERSONAL_ACCESS_TOKEN=${GITHUB_PAT} \
  -- npx -y @modelcontextprotocol/server-github

# http transport のサーバーを追加
claude mcp add notion -s user --transport http --url https://mcp.notion.com/mcp

-- 以降が実行コマンド+引数。-s でスコープ(user/project/local)、-e で環境変数、--transport でトランスポート種別を指定します。

list — 登録一覧

$ claude mcp list
filesystem   stdio  npx -y @modelcontextprotocol/server-filesystem /Users/koma/Desktop    [user]    ✔ connected
github       stdio  npx -y @modelcontextprotocol/server-github                            [project] ✔ connected
notion       http   https://mcp.notion.com/mcp                                             [user]    ✔ connected
fetch        stdio  uvx mcp-server-fetch                                                   [user]    ✘ failed (exit 127)

接続状態がひと目で分かります。✘ failed のサーバーがあれば、claude mcp get <name> で詳細・stderrログを確認できます。

remove — 削除

claude mcp remove fetch -s user

スコープを明示しないと「どのスコープから消すか曖昧」なエラーになります。必ず -s を付けましょう。

もっと詳しく: CLI追加後に .mcp.json を確認

claude mcp add は実体として「指定スコープのJSONファイルに mcpServers エントリを追記する」だけです。追加後に以下で確認できます:

# user scope
cat ~/.claude.json | jq '.mcpServers'

# project scope
cat ./.mcp.json | jq '.'

# local scope
cat ./.claude/settings.local.json | jq '.mcpServers'

「CLIで作ったJSON」を読むことで、逆にJSON手書きのお手本になります。最初の数回はCLI→JSON確認→理解、の流れがおすすめです。

この章のまとめ

スコープ別の「書く項目」分布(推奨)

要点は4つに集約されます:

  • .mcp.json の最小形は command + args の2フィールド。
  • スコープは user / project / local の3段階。秘密は必ず project scope の外。
  • stdio がデフォルト、http はリモート/OAuth用。迷ったら stdio。
  • claude mcp CLI(add/list/remove)を使うとJSON編集ミスを避けられる。
MCP設定でハマったら、まず claude mcp list。それで9割のミスは見える。 — Claude Code公式ドキュメントの実質的な鉄則
Exercise 1
入門

自分用の .mcp.json を組み立てる

次の要件を満たすuser scope設定(~/.claude.json)のJSONを書き、実際に claude mcp list で3つとも ✔ connected になるまで調整してください。

  1. filesystem: 自分のDesktopと、任意の作業ディレクトリ2つを許可
  2. fetch: Python版(uvx mcp-server-fetch)
  3. github: PAT環境変数経由(${GITHUB_PAT}で参照)
模範解答を見る

~/.claude.json(抜粋):

{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-filesystem",
        "/Users/koma/Desktop",
        "/Users/koma/Desktop/claude"
      ]
    },
    "fetch": {
      "command": "uvx",
      "args": ["mcp-server-fetch"]

    "github": {
      "command": "npx",

Exercise 2
応用

スコープを使い分けてチーム配布を設計する

あなたは5人チームのテックリード。チーム全員が使うプロジェクトでGitHub MCPを導入したい。ただし個人PATは人それぞれ。以下をどう設計するか、3つのファイル配置と内容を説明してください。

  1. リポジトリにコミットする .mcp.json(project scope)
  2. 各メンバーが自分で作る .claude/settings.local.json(local scope)
  3. READMEに書くべきセットアップ指示
模範解答を見る

①リポジトリ ./.mcp.json(commit対象):

{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "${GITHUB_PAT}"
      }
    }

トークン本体は書かず、環境変数参照のみ。これによりJSONはチーム共通資産として安全にコミットできる。

② 各メンバーの ./.claude/settings.local.json(.gitignore済み):

{
  "env": {
    "GITHUB_PAT": "ghp_メンバー個人のトークンをここに"
  }
}

local scopeはgit管理対象外なので、各自が自分のPATをここに書く。誤commitを防ぐため .gitignore に .claude/settings.local.json が入っていることを確認。

③ README.md のセットアップ節:

## MCP Setup
1. GitHubで Personal Access Token を発行(scope: repo, read:org, read:user)
2. `.claude/settings.local.json` を作成し、GITHUB_PAT を設定
   (テンプレ: .claude/settings.local.example.json)
3. `claude mcp list` で github ✔ connected を確認
4. トラブル時は `claude mcp get github` でエラー確認

判断の軸: 「チームで共有する定義」→ project、「個人ごとに違う秘密」→ local、「全部のプロジェクトで使う個人ツール」→ user。この三分法を守るだけで、PATがGitHubに流出する事故は防げる。応用として、企業利用ならmachine userのOAuth + Remote MCPに移行する手もある(個人PATを完全排除)。
