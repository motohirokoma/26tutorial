Phase 2: Practice > Chapter 04

実践：GitHub MCP

Issue/PR/Actions をClaude Codeから操作する

所要時間: 40分 Phase: Practice 難易度: 中級
この章のゴール

GitHub MCP サーバーを Claude Code に追加し、Issue 一覧・PR レビュー・Release notes 生成などを自然言語で実行できる状態を作ります。認証(PAT)のセットアップから、実用シナリオ3つ、権限管理のベストプラクティスまでを一気通貫で体験。最後のハンズオンでは自分のリポジトリの issue を取得して分類させるところまで到達します。

2026年1月時点

本章の記述は2026年1月時点の GitHub MCP サーバー(公式: github-mcp-server)の仕様に基づきます。MCP 仕様・サーバー実装は変化が速いので、コマンドや設定で迷ったら必ず modelcontextprotocol/servers と docs.claude.com/en/docs/claude-code の最新ドキュメントを参照してください。

GitHub MCP サーバーの全体像

GitHub MCP サーバーは、Claude Code から GitHub の GitHub API を「ツール」として呼び出せるようにする中継サーバーです。Claude Code は自然言語の依頼を受け、必要に応じてこのサーバーの tool を呼び出し、結果を要約してユーザーに返します。

Claude Code ⇔ GitHub MCP ⇔ GitHub の3層構造
graph LR
  A["あなた
(自然言語)"] --> B["Claude Code
(MCPクライアント)"]
  B -->|"tool call"| C["github-mcp-server
(stdio / http)"]
  C -->|"REST/GraphQL"| D["GitHub API"]
  D -->|"JSON"| C
  C -->|"tool result"| B
  B --> A
Claude Code はコードを書く以外にも、この図の B の役割で「どの tool を呼ぶか」を判断する。GitHub API に直接アクセスするのはあなたの PC 上で動くローカルな MCP サーバー。

できること/できないこと

できる

自然言語で GitHub を操作

  • Issue: 一覧、作成、コメント、ラベル付け、クローズ
  • PR: 一覧、diff取得、コメント、レビュー投稿、マージ
  • Actions: workflow 一覧、run 状態、log 取得、再実行
  • repo検索: コード検索、commit検索、ユーザー検索
  • ファイル操作: 読み取り、作成、更新(コミット経由)
できない / 要注意

境界と制約

  • 大量一括削除など破壊的操作は API 側で制限あり
  • Organization 管理(SSO 設定等)はscope 不足で失敗しがち
  • Private repo は PAT の repo scope が必須
  • GitHub Enterprise Server は別途 endpoint 設定が必要
  • API レート制限に到達すると一時停止
もっと詳しく: なぜ Claude Code から直接 curl しないのか

Claude Code は内蔵の Bash ツール経由で curl を叩けば、理論上 GitHub API を直接呼べる。しかしそれには毎回「URL とヘッダーを文字列で組み立てる」必要があり、エラー処理・ページネーション・パースを毎回モデルが再発明することになる。MCP サーバーは「tool 名 + 引数の JSON」というスキーマを提供するので、モデルは正しい形で呼びやすく、失敗時のリトライも構造化される。これが MCP の最大の価値。

セットアップ：PAT 発行と .mcp.json

GitHub MCP サーバーを使うには2つだけ用意します: (1) PAT(認証用トークン) と (2) .mcp.json の設定。ここが最初の山ですが、一度やれば以降は使い回せます。

Step 1: PAT を発行する

GitHub の Fine-grained PAT を推奨。古い Classic PAT より安全に scope を絞れます。

PAT 発行の4ステップ
graph TD
  S1["1. GitHub > Settings > Developer settings"] --> S2["2. Personal access tokens > Fine-grained tokens"]
  S2 --> S3["3. Repository access = 対象リポのみ選択"]
  S3 --> S4["4. Permissions = Issues/Pull requests/Contents等だけ付与"]
  S4 --> S5["ghp_... を安全にコピーして保管"]
「Public repositories (read-only)」プリセットから始めて、必要な書き込み権限だけ追加するのが安全。
最低限付与する Permissions(シナリオ別)

読み取り専用で試す: Contents=Read, Issues=Read, Pull requests=Read, Metadata=Read
Issue/PRにコメントも書く: 上記 + Issues=Read & Write, Pull requests=Read & Write
Actions を触る: 上記 + Actions=Read(必要なら Write)
最初はすべて Read から始めて、困ったら Write を足す。

Step 2: .mcp.json に登録

プロジェクトルートに .mcp.json を作成し、以下を記述します。GITHUB_PERSONAL_ACCESS_TOKEN は環境変数から読むのが鉄則(JSONに直書きしない)。

.mcp.json 最小構成(2026年1月時点)
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "${env:GITHUB_TOKEN}"
      }
    }
  }
}
シェルに export GITHUB_TOKEN=ghp_xxxxx を書いておけば、MCP 起動時に展開される。macOS なら ~/.zshrc に。

Step 3: CLI で登録＆起動確認

.mcp.json を手書きする代わりに、claude mcp add でも登録できます。両方覚えておくと便利。

セットアップの実コマンド
# 1) 環境変数に PAT を登録(.zshrcに追記)
echo 'export GITHUB_TOKEN=ghp_xxxxxxxxxxxxxxxxxxxx' >> ~/.zshrc
source ~/.zshrc

# 2) プロジェクトに GitHub MCP を追加(project scope)
claude mcp add github \
  --scope project \
  --env GITHUB_PERSONAL_ACCESS_TOKEN=\${GITHUB_TOKEN} \
  -- npx -y @modelcontextprotocol/server-github

# 3) 登録の確認
claude mcp list
# => github   stdio   running

# 4) Claude Code 起動 → "/mcp" で tool 一覧を確認
claude
> /mcp
/mcp スラッシュコマンドで、認識された tool 一覧と接続状態が見える。赤マークが付いたら Ch07「トラブルシュート」を参照。

提供される tool 一覧とカテゴリ

GitHub MCP サーバーは2026年1月時点で概ね40+ の tool を提供します。全部を覚える必要はなく、「どのカテゴリに何があるか」だけ押さえれば十分。モデルは自動的に正しい tool を選びます。

カテゴリ代表的な tool典型用途
Issuelist_issues, create_issue, update_issue, add_issue_comment棚卸し、優先度付け、担当割
Pull Requestlist_pull_requests, get_pull_request_diff, create_pr_review, merge_pull_requestレビュー、マージ、差分確認
Actionslist_workflow_runs, get_workflow_run_logs, rerun_workflow_runCI失敗調査、再実行
Reposearch_repositories, get_file_contents, create_or_update_fileコード検索、ファイル編集
Searchsearch_code, search_issues, search_users横断検索、ユーザー発見
Releaselist_releases, create_release, get_commitRelease notes 生成
tool カテゴリの「重さ」分布(Claude Code 使用頻度の体感)
Issue と PR 周りの tool が圧倒的に使われる。最初はこの2カテゴリだけ触れれば9割カバーできる。
もっと詳しく: tool 名を覚える必要はある?

結論: 覚えなくていい。Claude Code は「このリポのopen issueを出して」と自然言語で書けば、list_issues(state="open") を自動的に選んでくれる。ただし曖昧な依頼(例「PRを良い感じにして」)は判断に迷うので、用途を書いた具体的なプロンプトのほうが成功率が上がる。「どの tool を使うべきか」をあなたが知っていると、プロンプトを書きやすくなるのが副次効果。

実用シナリオ1: Issue 棚卸し

3ヶ月放置されている open issue が30件…という状況、ありますよね。Claude Code にラベル/古さ/本文から優先度を推定させて、一覧を整理します。

プロンプト例

Claude Code への依頼プロンプト
@github motohirokoma/my-app リポジトリの open な issue を全部取得し、
以下の観点でマークダウンの表に整理してください。

- 列: #番号 / タイトル / labels / 作成日(相対) / 優先度推定 / 推奨アクション
- 優先度は P1(本番影響) / P2(機能改善) / P3(軽微) の3段階
- 推奨アクションは "今すぐ対応" / "次スプリント" / "triage不要(close候補)"
- bug ラベルと stale(90日以上更新なし) を優先的にP1/close候補に振り分け
- 最後に「今週やるべきTOP3」を箇条書きで

レポート作成後、分類結果を元の issue にコメントとして書き込むのは
まだしないでください(確認後に指示します)。
「まだしないで」の一行が重要。破壊的アクション(コメント・クローズ)は必ず確認ステップを挟む。

期待される動作

Issue 棚卸しの処理フロー
sequenceDiagram
  participant U as あなた
  participant C as Claude Code
  participant M as github MCP
  participant G as GitHub
  U->>C: 棚卸しプロンプト

1回の list_issues 呼び出しで済むケースが多い。30件超のリポは自動でページネーションが走る。
Tips: ページネーションを意識する

issue 数が100件超のリポでは、Claude Code が自動で page=1,2,3... と取りに行きます。遅いと感じたらプロンプトで「最新30件だけ」と絞るのが手。API レート(5000req/h)を無駄に消費しないコツでもあります。

実用シナリオ2: PR 自動レビュー

チームメンバーが出した PR に一次レビュー(typo/命名/テスト漏れ)を入れる作業は自動化の恩恵が大きい領域。Claude Code は diff を読んで、具体的な行番号にコメント案を作ってくれます。

プロンプト例(diff 取得 → レビュー生成)

PR レビュープロンプト
@github motohirokoma/my-app の PR #142 を以下の手順でレビューして。

1. get_pull_request_diff で差分を取得
2. 以下の観点でチェック:
   - 命名規則(camelCase / kebab-case の統一)
   - null安全性(option型の扱い漏れ)
   - テスト: 追加・変更された関数に対する test ケースが足りているか
   - ドキュメントコメント(JSDoc/docstring)の有無
   - 明らかな typo

3. 見つかった指摘を以下のフォーマットで出力:
   [ファイル名:行番号] 重要度(must/should/nit) 指摘内容 提案コード

4. 「このPRをLGTMにできるか」を最後に yes/no で判断

ただし GitHub への実コメント投稿はまだしないこと。
出力を私が確認してからコメント投稿を指示します。
must/should/nit の3段階は、GitHub文化でよく使われる重要度ラベル。モデルに判断基準を渡すと指摘がぶれにくい。

レビューコメントを実際に投稿する

出力を確認したら、続けて次のプロンプトで投稿できます。

レビュー投稿のプロンプト
先ほどの指摘のうち、must と should だけを PR #142 に
create_pr_review(event="COMMENT") で投稿してください。

- 各指摘は行コメント(review comment)として当該ファイル・行に紐づける
- 全体コメントには「Claude Code 一次レビュー。最終判断は人間が行ってください」と明記
- event は APPROVE ではなく COMMENT に(自動承認しない)
event=APPROVE を自動で発行しないのが最重要ポイント。AIは一次レビュー止まり、承認は必ず人間。

実用シナリオ3: Release notes 生成

前回リリースから今日までに merged された PR を集め、Release notes のドラフトを作る作業も数分で終わります。

Release notes 生成プロンプト
@github motohirokoma/my-app の v1.3.0 に向けた Release notes ドラフトを作って。

1. list_releases で前回タグ(v1.2.0)の公開日を取得
2. list_pull_requests で state=closed & merged=true を
   v1.2.0以降のものだけ取得(base=main)
3. PR のラベル・タイトル・本文から Keep a Changelog 形式で分類:
   - Added / Changed / Deprecated / Removed / Fixed / Security
4. 各項目は「- PR #番号 タイトル (@author)」の行で
5. 最後に「Breaking changes」サマリを別段落で

出力は Markdown で。create_release はまだ呼ばないで、
ドラフトを私に見せるだけで止めてください。
Keep a Changelog 形式を明示すると、モデルの出力が安定する。あとは人間が微調整して公開するだけ。

リリースごとのPR件数トレンド(例)

直近6リリースのmerged PR数(架空のデータ)
Claude Code でこういうグラフ化も「list_releases + list_pull_requests の結果を CSV にして」と頼めば出してくれる。
もっと詳しく: GitHub 自動生成Release notesとの違い

GitHub には標準で「Generate release notes」ボタンがあり、labelベースで自動分類してくれる。ではなぜ MCP 経由で作るのか? 答えはカスタム分類。「Breaking changesだけ強調」「日本語で要約」「社内向け補足を追加」などの要件を、プロンプトで自由に指定できる。チーム固有のフォーマット(例: 「影響範囲」列を足す)にも即対応できるのがMCPの強み。

権限管理のベストプラクティス

GitHub MCP は強力なぶん、PAT が漏れたら即時にリポジトリへ書き込まれるリスクがあります。最小権限原則を徹底しましょう。

PAT設計の5原則

  • (1) Fine-grained PAT を使う — Classic PATは全リポ・全scopeを横断できて危険。必ず Fine-grained で対象リポを絞る。
  • (2) 有効期限を短く — 最長でも90日。常用でも30日ごとに更新する運用が安全。
  • (3) 環境変数で注入 — .mcp.json や git 管理下に PAT を書かない。${env:GITHUB_TOKEN} で参照。
  • (4) 用途別に複数トークン — 「読み取り専用PAT」と「書き込み可能PAT」を分ける。破壊的操作のときだけ書き込みPATに切り替える運用も有効。
  • (5) リポ側で branch protection — main へのforce pushを禁じ、PR経由必須に。MCP 経由で事故っても被害が限定される。
PAT 権限マトリクス(最小構成 → 段階的に広げる)
左から「読むだけ」→「コメントも書く」→「PR マージまで」と段階的に権限を広げる。Organization 管理が必要になったら、そこは別の管理者用PATに分離すべき。
Tips: PAT 漏洩に気づいたら

GitHub の Settings > Developer settings > Personal access tokens から即座に revoke(失効)できます。git log に誤って commit した場合は git filter-repo で履歴から消した上で revoke し、必ず新PATを発行。GitHubは漏れたトークンを検知すると自動でrevoke することもあります(secret scanning)。

権限ミスのよくあるパターン

症状原因対処
403 Forbiddenscopeが足りない / Fine-grainedで対象リポ未選択PAT編集画面で必要な permission を Read/Write に
404 Not Foundprivate リポなのに repo scopeなし(Classic) / repo未選択(Fine)リポ選択 or scope追加
401 UnauthorizedPAT期限切れ / 環境変数未設定PAT再発行、echo $GITHUB_TOKENで確認
tool呼べるが sensitive操作だけ失敗org側で SAML SSO未認証 / IP制限PAT画面で "Configure SSO" を実行

ハンズオン: 自リポの Issue を分類する

ここまでの内容を、あなた自身のリポジトリで実行します。所要15分。

準備

ハンズオンの前提チェックリスト
# 前提 1: 自分の GitHub リポジトリ(open issueが5件以上あるもの)
#         なければ本記事のコメント欄用に新規リポを作ってもOK

# 前提 2: Fine-grained PAT を発行済み
#         - Repository: 対象の1リポだけ選択
#         - Permissions: Contents=Read, Issues=Read, Metadata=Read

# 前提 3: 環境変数にPATを登録
export GITHUB_TOKEN=ghp_xxxxxxxxxxxxxxxxxxxx

# 前提 4: Claude Code に GitHub MCP を登録
claude mcp add github \
  --scope project \
  --env GITHUB_PERSONAL_ACCESS_TOKEN=\${GITHUB_TOKEN} \
  -- npx -y @modelcontextprotocol/server-github

# 前提 5: 確認
claude mcp list | grep github
grepで github が出ればOK。出ない場合は Ch02 の設定確認に戻る。

実行プロンプト(コピペ可)

ハンズオン本体プロンプト
@github <あなたのGitHubユーザー名>/<リポ名> の open issue を
すべて取得し、以下の4カテゴリに分類してください。

カテゴリ:
  A) Bug (実害が出ている不具合)
  B) Feature (機能追加・改善要望)
  C) Docs (ドキュメント・コメント改善)
  D) Chore (依存更新・リファクタ等、利用者影響なし)

出力:
  1) カテゴリごとの件数サマリ(表)
  2) 各カテゴリの一覧(#番号 タイトル 作成日)
  3) 「今週やるなら」TOP3とその理由
  4) 分類できなかった issue があれば「要トリアージ」として別枠に

制約:
  - issue への書き込み(コメント・ラベル付け・close)はしない
  - 最新50件だけで良い
< > の部分を自分のユーザー名とリポ名に置き換えて実行。最後の「書き込みしない」が安全ネット。

Exercise 1
入門

安全なPATを自分用に設計する

あなたが今日 MCP 用の Fine-grained PAT を発行するとしたら、以下を書き出してみましょう。

  1. 対象 Repository(1つ、または少数)
  2. 必要な Permissions(Read/Write の別)
  3. 有効期限(日数)
  4. 環境変数名(例: GITHUB_MCP_TOKEN)
  5. 保管場所(~/.zshrc / password manager / CI secret 等)
模範解答を見る

回答例(読み取り中心で始める設計):

  • Repository: motohirokoma/my-app の1つだけ
  • Permissions: Contents=Read, Issues=Read & Write, Pull requests=Read & Write, Metadata=Read, Actions=Read
  • 有効期限: 60日
  • 環境変数名: GITHUB_MCP_TOKEN(普段のgit用PATと別名)
  • 保管場所: ~/.zshrc に export、物理バックアップは1Password に保管

ポイントは「普段のgit CLIと別トークンにする」こと。MCP用が漏れても普段の開発用に影響が出ない。もう一段進むなら、書き込み用をさらに別PAT(短期)にして「普段は読み取りPAT、必要な時だけ書き込みPATに切り替え」運用が理想。

Exercise 2
応用

PR ワンストップレビューのプロンプトを設計する

「PR番号を渡すと、(1)diff取得→(2)一次レビュー→(3)指摘のMarkdown出力→(4)確認後にコメント投稿」を一本で回すプロンプトを設計してください。以下の要件を満たすこと。

  1. 自動承認(APPROVE)は絶対にしない
  2. 指摘は must/should/nit の3段階に分ける
  3. 投稿前に人間の確認ステップを必ず挟む
  4. レビュー本文に「AIによる一次レビュー」と明記する
模範解答を見る

回答例プロンプト:

@github motohirokoma/my-app の PR #{番号} を以下の手順でレビューして。

Step 1: get_pull_request_diff で差分取得
Step 2: 以下の観点でレビュー(指摘は must/should/nit に分類)
  - 命名統一 / null安全性 / テスト不足 / ドキュメント / typo / セキュリティ
Step 3: [ファイル:行] 重要度 内容 の Markdown表で出力
Step 4: 私が "OK" と返すまで GitHub への書き込みはしない
Step 5: OK後、create_pr_review(event="COMMENT") で投稿
  - 本文冒頭に "[Claude Code 一次レビュー] 最終判断は人間が行います" を明記
  - event は APPROVE/REQUEST_CHANGES ではなく必ず COMMENT

ポイント: (a) Step4 の human-in-the-loop、(b) event=COMMENT 固定、(c) 本文の明示的な AI表記。特に(c)はレビュイー(PR作成者)への誠実さとして重要。自動レビューに混乱させられないように、AI生成であることを明記する文化を持ち込む。
