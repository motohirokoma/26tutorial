実践：Filesystem・他MCP

外部ツリーを扱う / DB / Slack / スクレイピング などの実用例

所要時間: 30分 Phase: Practice 難易度: 中級

Ch04でGitHub MCPを主戦場にした。今度はそれ以外の日常よく使う4ジャンル — Filesystem MCP、DB系MCP、Slack MCP、Fetch/Puppeteer MCP — を駆け足でまわす。Ch03のカタログに出てきたサーバー群を、実際にどう動かすかに焦点を絞った実践章だ。

前提

Ch02の .mcp.json 基礎と、Ch04のシェル実行フロー(claude mcp list / トークン管理)がわかっていること。本章は「繋ぎ方」より「使いどころ」の比率が高い。

4つのMCPをどう位置づけるか

ひとまず本章で扱う4ジャンルを、「アクセス先」と「副作用の重さ」で整理する。Claude Codeのツール呼び出し承認UIでどれだけ慎重にならないといけないか、の肌感になる。

4ジャンルの位置関係
右下ほど外界への影響が大きい。右下サーバーは承認設定を厳しめに。
ジャンル代表サーバー主な用途副作用推奨スコープ

本章のサーバー名・パッケージ名は2026年1月時点のもの。公式サーバー集は modelcontextprotocol/servers が一次情報。パッケージ名は変わることがあるので都度README確認を。

もっと詳しく: 「公式」「準公式」「サードパーティ」の見分け方

同じ名前のMCPサーバーが複数あることがある(例: slack-mcp と server-slack)。判断基準は (1) GitHub organization が modelcontextprotocol か、(2) @modelcontextprotocol/ スコープのnpm/PyPIパッケージか、(3) awesome-mcp-servers リポジトリで Official タグが付いているか、の3点。自社コード・DBに触れるものは公式 or 自分で監査できるOSSだけに絞るのが安全。

Filesystem MCP: 作業dir外のファイルを扱う

Claude Codeには組み込みのRead/Edit/Writeがあるが、それらは起動時の作業ディレクトリ配下しか扱えない。「~/Documents の資料を要約したい」「別リポの設定を覗きたい」といった場面で詰まる。ルート許可モデルで安全にこれを解放するのがFilesystem MCPだ。

設定は1行追加するだけ。引数に渡したパスだけが読み書き可能な領域になる。

// ~/.claude/.mcp.json  (user scope)
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-filesystem",
        "/Users/koma/Documents",
        "/Users/koma/Desktop/claude"
      ]
    }
  }
}

引数のパス以外には絶対にアクセスできない。Claude Codeの組み込みRead/Writeとは独立したツールセット(filesystem__read_file, filesystem__write_file, filesystem__list_directory, filesystem__search_filesなど)が生える。

組み込みツール vs Filesystem MCPの守備範囲
graph LR
  U["あなた"] --> CC["Claude Code"]
  CC --> Builtin["組み込み Read/Write"]
  CC --> FSMCP["filesystem MCP"]
  Builtin --> WD["作業ディレクトリ内
/cwd 配下のみ"]
  FSMCP --> R1["~/Documents"]
  FSMCP --> R2["~/Desktop/claude"]
  FSMCP -.禁止.-> X["/etc /root 等"]
組み込みはcwd、MCPは明示許可したパス。2系統が共存する。

できること・できないこと

  • read_file / list_directory — 許可パス配下のファイル閲覧・ディレクトリ走査
  • search_files — ファイル名の部分一致検索(中身検索ではない点に注意)
  • write_file / edit_file — 書込も可能。要・承認UI
  • move_file — 名前変更・ディレクトリ移動
  • できない — 許可外パス、シンボリックリンクの外、プロセス実行
Tips

引数には絶対パスを渡す。~ は展開されないのでフルパス化が必須(/Users/yourname/Documents)。Windowsは C:\\Users\\... でバックスラッシュ2個。

もっと詳しく: なぜ「組み込みで十分」にしないのか

Claude Codeの組み込みRead/Writeを全ディレクトリに開放すると、プロジェクトのcwd外にうっかり書き込んでしまう事故が起きる。MCP経由にしておくと (1) 許可パスがJSON1箇所に集中する、(2) ツール名が filesystem__* と明示されるので承認時に「あ、外ディレクトリ触ろうとしてる」と気づきやすい、(3) 別プロジェクトで設定を使い回せる、という3つのメリットが出る。明示性はMCPの最大の価値。

DB系MCP: スキーマ検査と安全なクエリ

DBに生で psql を叩かせるのはリスクが高い。DB系MCPは読取専用 + スキーマ自動取得を前提にしたインターフェイスを提供する。代表はPostgres公式サーバー。

// .mcp.json  (project scope)
{
  "mcpServers": {
    "postgres": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-postgres",
        "postgresql://readonly:PASS@localhost:5432/myapp"
      ]
    }
  }
}

起動するとpostgres__query(SELECTのみ)とpostgres__list_tablesが生える。Claude Codeは初手でスキーマを自動で読んでからクエリを組み立てるので、テーブル名をこちらから教える手間がない。

DB MCPの典型ワークフロー
sequenceDiagram
  participant U as あなた
  participant C as Claude Code
  participant M as Postgres MCP
  participant D as DB
  U->>C: "先月の売上Top10を"
  C->>M: list_tables
  M->>D: information_schema問合せ
  D-->>M: tables
  M-->>C: スキーマ返却
  C->>M: query(SELECT ...)
  M->>D: 読取SQL
  D-->>M: rows
  M-->>C: 結果
  C-->>U: 整形して回答
「スキーマ取得→適切なSELECT」が自動で走る。ユーザーはSQLを1行も書かない。
安全設計

DB接続は専用のreadonlyロールを作って渡す(CREATE ROLE readonly LOGIN PASSWORD '...'; GRANT SELECT ON ALL TABLES IN SCHEMA public TO readonly;)。書込権限のあるroleを渡さないのが鉄則。

  • 探索 — 「このテーブルのカラム構成は?」で即答える
  • 集計 — 「月次のアクティブユーザー推移」をSQL書かずに
  • 差分調査 — 「本番と開発でテーブル定義が違う箇所」を2接続で比較
もっと詳しく: Postgres以外の選択肢

2026年1月時点、公式or準公式MCPが存在する主なDB: PostgreSQL(公式)、SQLite(公式)、MySQL(サードパーティ複数)、BigQuery(サードパーティ)、Snowflake(サードパーティ)。MongoDB/Redis もコミュニティ実装あり。サードパーティは必ず監査してから使う。自社DB接続情報が外部に漏れるリスクに注意。

Slack MCP: チャンネル要約と投稿

Slack MCPはBot Tokenを使ってチャンネル読書きする。要約用途ならchannels:history/channels:readだけのBotで十分で、投稿を許すならchat:writeを追加する。

// .mcp.json
{
  "mcpServers": {
    "slack": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-slack"],
      "env": {
        "SLACK_BOT_TOKEN": "xoxb-********",
        "SLACK_TEAM_ID": "T0123456"
      }
    }
  }
}

主な提供ツール: slack__list_channels, slack__post_message, slack__reply_to_thread, slack__get_channel_history, slack__add_reaction, slack__get_users。

投稿系ツールは permissionMode: "askOnce" や Claude Code のdeny設定で事故を防ぐ。「一度許可したら同セッション中はスキップ」は投稿ではオフにしておくのが安全。

Fetch / Puppeteer MCP: Webを読む

Web情報を扱うMCPは2系統。Fetch系は軽量HTTPクライアント、Puppeteer系はフルブラウザ。用途で使い分ける。

用途FetchPuppeteer
静的HTMLの取得◎(軽量・速い)○(オーバースペック)
SPA/JS描画ページ×(空のHTMLが返る)◎
スクリーンショット×◎
フォーム送信・クリック×◎
起動コスト数百ms数秒(Chromium起動)
 
// 両方いれる場合
{
  "mcpServers": {
    "fetch": {
      "command": "uvx",
      "args": ["mcp-server-fetch"]
    },
    "puppeteer": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-puppeteer"]
    }
  }
}

スクリーンショットはbase64で大きい。Claude Codeのコンテキストを一気に食う。全画面ではなく要素指定スクショ(screenshot(selector))で絞るか、ファイル保存して後で参照させる運用が賢い。

もっと詳しく: robots.txt と利用規約

MCPだからといってスクレイピングの倫理が緩むわけではない。robots.txt の尊重、利用規約の確認、API があるならAPIを優先、過剰リクエストの回避は通常のクローラーと同じ。FetchやPuppeteerで大量に叩くと、サービス側から見ればDoS攻撃と区別がつかない。レート制御は呼び出し元(あなたとClaude Code)の責任。

ハンズオン: ~/Documents を要約する

ここからは手を動かす。Filesystem MCPで ~/Documents にアクセスし、ファイル構造と主要ドキュメントを要約させる。Claude Codeがインストール済みで、Node.js 18+ がある前提。

  • 手順1: ~/.claude/.mcp.json に filesystem サーバーを1つ追加(本章の設定例をそのまま使ってOK、ただしパスは自分の環境に書き換え)
  • 手順2: シェルで claude mcp list を実行し、filesystem: running が出ることを確認
  • 手順3: Claude Codeで ~/Documents の直下ファイル・フォルダを一覧させる(プロンプト: 「filesystem MCPで~/Documentsのトップをlist_directoryし、主要ディレクトリを3行で要約して」)
  • 手順4: 次に「サイズの大きい.mdファイルTop3をsearch_filesで見つけ、それぞれread_fileして200字で要約」を投げる
  • 手順5: 最後に「~/Documents外のファイルを読もうとしてみて」と指示し、拒否されることを確認(安全境界のチェック)
想定つまづき

(1) ~が展開されない → フルパス記述、(2) claude mcp listでエラー → npxのキャッシュクリア npx clear-npx-cache、(3) 大量ファイルで遅い → list_directoryは深さ1固定、再帰は自分で指示。

# 動作確認コマンド例
claude mcp list
# -> filesystem: running

# Claude Codeセッション内でのプロンプト例

Exercise 1
入門

~/Documents のファイル構造を要約する

上のハンズオンをそのまま実行し、手順5(拒否確認)までやりきる。

  1. 自分の環境用に.mcp.jsonを書いて保存する
  2. claude mcp listでrunningを確認
  3. Claude Codeで「~/Documents配下の主要ディレクトリTop5を役割別に要約」させる
  4. 「/etc/passwdを読んで」と投げて拒否されることを確認
模範解答を見る

想定される応答の形(例):

~/Documents 配下の主なサブディレクトリ:
- work/        : 業務関連PDF・Word中心(計 412 files)
- personal/    : 個人メモ・Markdown(計 87 files)
- screenshots/ : 画像中心・サイズ大(計 1.2GB)
- archive/     : 古い資料・zip多数(計 340 files)
- receipts/    : 領収書PDF・年別(計 210 files)

手順5で/etc/passwdを指示した際の期待応答: 「filesystem MCPの許可パス外のためアクセスできません」。これが出ればルート許可モデルが正しく効いている。

チェックポイント: (a) 応答が具体的ファイル数を含む、(b) ディレクトリ役割を人間が読める日本語で分類、(c) 禁止パスは明確に拒否。この3つが揃えばOK。

Exercise 2
応用

Fetch MCPで技術ブログ記事をMarkdown化する

Fetchを足して、Web記事→要約のフローを組む。

  1. .mcp.jsonに mcp-server-fetch を追加(uvx必須。未導入なら pip install uv)
  2. 好きな技術ブログ記事のURLを1つ用意
  3. Claude Codeで「このURLをfetchしてMarkdownで取得→300字で要約→~/Documents/notes/にYYYYMMDD-title.mdとして保存」を指示
  4. filesystem MCPの書込承認UIが出ることを確認し、保存される
模範解答を見る

完成する.mcp.json(抜粋):

{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem",
               "/Users/you/Documents"]
    },
    "fetch": {
      "command": "uvx",
      "args": ["mcp-server-fetch"]
    }
  }
}

期待されるClaude Codeの動き:

  1. fetch__fetch(url) → Markdown本文を受け取る
  2. 内部で300字に要約
  3. filesystem__write_file("/Users/you/Documents/notes/20260118-xxx.md", content) をツール呼び出し
  4. 書込承認UIが表示 → ユーザーがYes → 保存完了

応用ポイント: Fetchが返すのはサイト側のHTML→Markdown変換。JSで後から生成される内容は取れないので、SPA記事ならpuppeteerに切り替える判断ができれば◎。この「Fetchで試す→ダメならPuppeteer」の分岐を自力で回せれば本章の目的達成だ。
