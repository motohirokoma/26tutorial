公式MCPサーバー一覧

GitHub/Filesystem/Slack/Notion他の定番カタログと用途

所要時間: 30分 Phase: Basics 難易度: 初級
この章のゴール

MCPの仕組みが分かったら、次は「**実際に何を繋げるか**」です。本章では公式MCPサーバーと、準公式サーバー・コミュニティサーバーを**用途別カタログ**として一望し、自分の業務に連携できそうなサーバーを2つ選べる状態を目指します。Ch04以降の実践の「下ごしらえ」です。

2026年1月時点の情報

本章の記述は**2026年1月時点**のものです。MCPエコシステムは拡大途中で、サーバー一覧は毎月のように増えます。最新情報は modelcontextprotocol/servers と awesome-mcp-servers を都度チェックしてください。

MCPサーバーの3階層 ― Official / Partner / Community

MCPサーバーは「誰が作ったか」で3つの階層に分けられます。階層が変わると**信頼性・更新頻度・サポート窓口**がガラッと変わるので、最初に頭に入れておくと選定がぐっと楽になります。

MCPサーバーの信頼性ピラミッド
上層ほど「壊れにくい・サポートあり」。下層ほど「種類は豊富・保証なし」。

業務で使うなら原則Official または Partnerから選ぶのが鉄則。Communityサーバーは**便利だがトークンをそのまま渡せば全権限を握られる**可能性があるので、コードを読んで判断できる人以外は手を出さない方が安全です。

階層管理者信頼性更新頻度業務利用
Official (公式)Anthropic / MCP org◎ 最高週〜月◎ 推奨
Partner (準公式)サービスベンダー自身 (GitHub社/Notion社/Slack社等)○ 高い月〜四半期○ 推奨
Community (コミュニティ)個人・OSS貢献者△ コード確認必須ピンキリ△ 非機微な用途のみ
注意: 「公式」の定義ブレ

ネット上で「公式MCPサーバー」と書かれていても、**Anthropic公式**とは限りません。サービスベンダー自身が配布する Partner も「公式」と自称します。本章では**Anthropic / MCP org製のみOfficial**、**ベンダー自社配布はPartner**、**それ以外はCommunity**と区別して扱います。

`servers-archived` について

Anthropicは2025年に一部のリファレンス実装を**`servers-archived`** へ移管しました。動作はしますが、**メンテナンスはコミュニティ・Partnerに委譲**されています。本番運用ではPartner版(存在する場合)を優先しましょう。

主要サーバーの「できる / できない」

サーバー名だけ見ても「何を任せられるか」はイメージしづらいので、5つの定番についてできること / できないことを明示しておきます。「できない」を知ることが選定のショートカットです。

  • Filesystem — できる: 許可パス配下のread/write/list/search/move。 できない: 許可外パス・シンボリックリンクを辿った外部アクセス・chmod等の権限操作。
  • GitHub — できる: Issue/PR/Actions/Repo/検索/レビュー投稿。 できない: Billing操作・Enterprise admin操作・PAT権限外のリポアクセス。
  • Slack — できる: チャンネル一覧・メッセージ送受信・スレッド取得。 できない: ユーザー作成・ワークスペース設定変更・DMの全履歴取得(権限次第)。
  • Postgres — できる: スキーマ検査・読み取り専用クエリ。 できない: INSERT/UPDATE/DELETE(既定では無効)・DDL。
  • Fetch — できる: 公開URLのGET→HTML→Markdown化。 できない: POSTフォーム送信・Cookie認証セッション維持・JSが必要なページの完全レンダリング(その場合はPuppeteerへ)。
  • Notion — できる: ページ検索・作成・ブロック追加/更新・DB検索。 できない: ワークスペース管理・権限共有の一括変更・未公開のInternal API。
もっと詳しく: Postgresが「読み取り専用」になっている理由

公式Postgres MCPサーバーは**明示的に`readOnly`トランザクションでクエリを実行**します。理由は単純で、LLMにDELETE文を打たせて本番DBを飛ばす事故を防ぐため。Claude Codeが「テーブルを綺麗にしますね」と善意でTRUNCATEを走らせる可能性がゼロでない以上、最初から書き込み口を塞いでおく設計になっています。書き込みたい場合は**専用の別MCPサーバー(例: 書き込み許可を明示した社内ラッパー)を自作**するか、Claude Codeに承認モードを強制して1クエリずつ目視確認する運用に寄せるのが現実解です。

選び方 ― 主戦場別の推奨セット

「全部入れる」は間違い。MCPサーバーは**使うほど起動時間が伸び、プロンプトのツール説明スペースも食います**。自分の主戦場に応じて3〜5本に絞るのが定石です。

セキュリティスコープ ― 「最小権限」を貫く

MCPサーバーを入れる=AIに鍵を渡すです。便利の裏側で、鍵を広く渡すほど事故の爆発半径も広がります。原則は常に最小権限原則(Principle of Least Privilege)。

権限のレイヤー(どこで絞るか)
4層すべてで絞る。1層だけに頼らない。
  • トークンは用途専用で発行 — GitHub PATは「claude-code-read-only」のような名前で、必要なスコープだけ付ける。
  • Filesystemは許可パスを限定 — `~/` や `/` を指定しない。`~/projects/my-app` のような具体パスに。
  • Communityサーバーはコードを読む — `stdio` で起動するコマンドの中身を確認。サプライチェーン攻撃を避ける。
  • 本番DBへの接続は避ける — 読み取り専用レプリカか、開発用スナップショットDBに向ける。
  • トークンは期限付き — 90日などで切れる設定に。漏洩時の被害期間を限定する。
実話系の警告: "便利"と"事故"は隣り合わせ

2025年にコミュニティで報告された事例として、**ローカルで動かしていた自作MCPサーバーが環境変数経由でAPIキーをログに吐き続けていた**、**Filesystem サーバーを`/`で起動して`.ssh/id_rsa`までClaudeが読める状態だった**、といった事故が複数あります。「ちょっと試す」の瞬間に鍵が漏れるのがMCPの怖さ。最小権限 × 監査ログ × 短命トークンの3点セットを常に意識しましょう。

「業務情報を扱うClaude Codeに、業務ツールを繋げる」この組み合わせはMCPの本領発揮ですが、同時に**情シスや上長との事前合意**を得る作業を忘れないでください。許可の取り方自体が業務スキルの1つです。

もっと詳しく: 「信頼できるサーバー」の見抜き方

GitHubリポジトリを開いたら、次の5点をチェック:

  1. Star数 — 100未満ならまだ実験段階。業務用途なら500+を目安に
  2. 最終コミット日 — 3ヶ月以上更新がないと仕様追従が怪しい
  3. Issueの回答率 — メンテナが反応しているか
  4. 認証の実装 — トークンを環境変数で受け取るか、ハードコードしていないか
  5. 依存関係 — `package.json` / `pyproject.toml` の依存が多すぎないか。少ないほど攻撃面が狭い

この5点で大半のコミュニティサーバーは選別できます。「有名なCEOが作った」「Twitterでバズった」は信頼性の指標として**最弱**なので鵜呑みにしないでください。

ハンズオン ― 自分の業務に関連しそうなサーバーを2つ選ぶ

ここまで読んだら、あとは**手を動かすだけ**。以下のステップで、自分の業務に直結するMCPサーバーを2つピックアップしてみましょう。Ch04以降の実践パートで実際にそれを動かします。

1
主戦場を1つ決める
コード / ドキュメント / データ / 連絡
2
カタログから2本仮選定
本章Section 2の表を参照
3
READMEを3分読む
何ができる/できないを確認
4
Ch04以降で追加
.mcp.jsonに書いて動かす

Claude Code の `.mcp.json` に追加する時の**テンプレート**を以下に置きます。2つ選んだサーバー名に置き換えてコピペしてみてください。

{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-filesystem",
        "/Users/YOUR_NAME/projects"
      ]
    },
    "github": {
      "command": "docker",
      "args": [
        "run", "-i", "--rm",
        "-e", "GITHUB_PERSONAL_ACCESS_TOKEN",
        "ghcr.io/github/github-mcp-server"
      ],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "ghp_xxx..."
      }
    }
  }
}
次章への橋渡し

選んだ2つはメモしておきましょう。Ch04では**GitHub MCP**を徹底的に使い倒します。Ch05では**Filesystem/Postgres/Slack/Fetch**を実案件で動かします。自分が選んだサーバーがそこに含まれていれば、学習が二度楽しくなります。

MCPは「何と繋ぐか」ですべてが決まる。ツールの数ではなく、自分の業務にどれだけ深く刺さるかが価値。 — Anthropicエンジニアリングブログより意訳
Exercise 1
入門

あなたの業務に関連するMCPサーバーを2つ選ぶ

本章Section 2のカタログ表を見ながら、以下を紙かテキストエディタに書き出してください。

  1. 自分の「主戦場」を1つ書く(例: バックエンド開発 / SaaS運用 / ライティング / データ分析)
  2. カタログから該当する**分野**を特定し、MCPサーバーを2つピックアップ
  3. それぞれについて、GitHubリポジトリのREADMEを開いて、「できること」を3つずつ書き出す
  4. それぞれの「階層」(Official/Partner/Community)をメモ
  5. 繋ぎたい順で番号を振る(Ch04・Ch05での優先順位になる)
模範解答を見る

解答例(主戦場: バックエンド開発):

主戦場: Python/TypeScriptのバックエンド開発

サーバー1: GitHub (Partner, github社公式)
  できること:
    - Issueの一覧・作成・更新
    - PR の差分取得・レビューコメント投稿
    - GitHub Actions のworkflow run 状態確認
  繋ぎたい優先度: ①(最優先)

サーバー2: Postgres (Official, servers-archived)
  できること:
    - スキーマ情報の取得(テーブル/カラム一覧)
    - 読み取り専用SELECTクエリ
    - EXPLAIN による実行計画確認
  繋ぎたい優先度: ②

ポイント: "できること"は抽象論ではなく**READMEに実際に書かれたツール名**を転記するのがコツ。Ch04・Ch05でそのまま使えるメモになる。「両方Officialから選ぶ」よりも、Partner/Officialを組み合わせて**主戦場の幅をカバー**する方が学習効率が高い。

Exercise 2
応用

セキュリティスコープ設計

Exercise 1で選んだ2つのサーバーに対して、最小権限原則を満たすスコープ設計を詳細に書いてください。以下の観点を含めること。

  1. トークン/APIキーのスコープ(例: GitHub PATならどのscopeか)
  2. MCPサーバー引数での制限(例: 許可パス、接続先DB)
  3. 機微情報に触れるリスク(何が漏れたら一番まずいか)
  4. そのリスクを抑える3つの対策
  5. 漏洩時の初動(どうやって検知し、どう止めるか)
模範解答を見る

解答例(GitHub + Postgres):

【GitHub MCP】
1) PATスコープ: repo:read, issue:write のみ
   (admin:org / workflow:write は付けない)
2) サーバー引数: 環境変数で PAT を注入、
   .mcp.json に直書きしない(env 分離)
3) 漏洩リスク: private repoコードの流出、
   機密Issueの内容漏洩
4) 対策:
   - PATに90日の有効期限を設定
   - GitHubの Fine-grained PAT を使い
     対象リポジトリを限定
   - Claude Codeで承認モードを有効化し
     書き込み系ツールは都度確認
5) 漏洩時初動:
   - GitHub Settings > PAT で即revoke
   - git log で直近のアクセスを確認
   - 必要なら該当リポの全commitを
     audit logで確認

【Postgres MCP】
1) DB接続情報: read_only ロールのユーザで接続、
   対象schemaも public のみに絞る
2) サーバー引数: 接続先は**ステージングDB**、
   本番DBには絶対に向けない
3) 漏洩リスク: 顧客個人情報(PII)の流出
4) 対策:
   - 本番データはマスク済みスナップショットのみ
   - SSH tunnel経由で直接接続は避ける
   - クエリログをアプリ側で収集
5) 漏洩時初動:
   - DBユーザを即DROP USER
   - 接続ログから流出範囲を特定
   - セキュリティ担当に即エスカレ

判断の軸: ①鍵の権限を「業務に必要な最小」に絞る ②鍵の生存期間を短く ③何かあった時の「止め方」を事前に決めておく。③を書けるかが**プロとアマの境界線**。運用開始前にRunbookとしてドキュメント化しておくと、実際の事故時に慌てずに済む。
