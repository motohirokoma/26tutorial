MCPとは

Model Context Protocol の思想と全体像

所要時間: 25分 Phase: Basics 難易度: 入門
この章のゴール

MCP (Model Context Protocol) が何を解決するプロトコルなのかを、自分の言葉で説明できるようになること。「N×M問題」「3概念(Tools / Resources / Prompts)」「Client-Serverアーキテクチャ」の3本柱を押さえて、Ch02以降の設定ファイル操作が「なぜその形なのか」腑に落ちる状態にします。本章の記述は2026年1月時点の情報に基づきます。MCP仕様や公式サーバー一覧は変化が速いので、実作業前に modelcontextprotocol.io と Claude Code docs を確認する習慣を付けましょう。

MCP (Model Context Protocol) とは

MCP (Model Context Protocol) は、AIアシスタントが外部のツールやデータに接続する方法を標準化するオープンプロトコルです。Anthropicが2024年11月に仕様を公開し、2026年1月時点ではClaude Code・Claude Desktop・Zed・Cursor などの主要AIクライアントが対応しています。

ざっくり言えば、MCPは「AIのUSB-C端子」です。どのノートPCでも同じUSB-Cケーブルで周辺機器が繋がるように、どのAIクライアントでも同じMCPサーバーに繋げば、Slack / GitHub / Postgres / ローカルファイル などを操作できる——その共通規格を目指しています。

MCP is an open protocol that standardizes how applications provide context to LLMs. Think of MCP like a USB-C port for AI applications. — Model Context Protocol 公式サイト (modelcontextprotocol.io)

MCP登場前後で何が変わったか

  • 統合方法の統一 — 「このLLM用 × このツール用」の個別実装が不要に。1つ書けばどこでも繋がる。
  • ユーザ承認のフロー標準化 — どのツール呼び出しにどんな権限が要るか、クライアント側で一貫して扱える。
  • ローカル実行が基本 — サーバーは自分のマシンで動き、データは外部クラウドを経由せずに扱える(stdio transport)。
  • エコシステムの爆発的増加 — 2026年1月時点で公式 + コミュニティ合計 1,500 以上のMCPサーバーが公開されている。

この章では、「なぜこの規格が必要だったのか(N×M問題)」「どういう構造になっているのか(3概念 + アーキテクチャ)」「似た技術とどう違うのか」を順に見ていきます。

なぜ必要か:N×Mの統合問題

MCPを理解する一番の近道は、「もしMCPが無かったら、今どれだけの個別実装が必要になるか」を想像することです。

N×M問題とは、ソフトウェア統合でよく出てくる古典的課題です。N個の消費者(AIクライアント)と M個の提供者(ツール)を結びつけるとき、個別にコネクタを書くと N × M 本のコードが必要になります。中間にプロトコル層を1つ挟むと、必要なのは N + M 本に激減する——これがMCPの存在意義です。

Before MCP: N×M 本の個別コネクタ
クライアントが増えるとコネクタは掛け算で増殖する。新しいツール1つ足すだけで N 本追加。
After MCP: N + M 本で済む
MCPというバスを1つ挟むだけで、追加コストが乗算から加算に変わる。

数字で見るスケーラビリティ

クライアント × ツール数と必要コネクタ数
個別実装(青)は二次関数的に、MCP経由(橙)は線形に増える。10x10 なら 100 vs 20。
もっと詳しく: N×M問題はソフトウェア業界の「お決まり」

似た構造の問題は歴史上何度も現れています。たとえば ODBC はアプリとデータベースの N×M を解き、LSP (Language Server Protocol) はエディタと言語処理系の N×M を解きました。MCPはその系譜の最新版と言えます。

面白いのは、LSPの設計者が MCP 仕様の参考文献として繰り返し引用されていることです。JSON-RPC の使い方、capabilities のネゴシエーション、stdio transport などは LSP とよく似ています。LSPを触ったことがあれば、MCPのRFCも驚くほど早く読めます。

MCPの3つの概念:Tools / Resources / Prompts

MCPサーバーがクライアントに提供できる能力は、Tools / Resources / Prompts の3種類に整理されています。この3つの違いを最初に押さえておくと、Ch02以降で公式サーバーのドキュメントを読むのが格段に楽になります。

3概念の関係:誰が主語か
Toolsは「AIが判断して呼ぶ」、Resourcesは「参照用データ」、Promptsは「ユーザ起点のテンプレ」。

それぞれの役割と具体例

概念主語(誰が発火)何ができるGitHub MCPでの例
Toolsモデル(AI)外部に作用する操作。引数を渡して実行し、結果を返すcreate_issue, merge_pr, search_code
Resourcesアプリ / ユーザ読み取り専用の文脈データをAIに渡すファイル内容、repo の README、issue の本文
Promptsユーザプリセット化されたプロンプトテンプレート(スラッシュコマンドとして見えることが多い)/review-diff, /release-notes

Tools (一番よく使う) の中身をもう少し

大半のMCPサーバーが主に公開するのは Tools です。各ツールは name / description / inputSchema (JSON Schema) の3点セットで宣言され、Claude Codeはこれを見て「今このツールを呼ぶべきか」を判断します。

たとえば GitHub MCPサーバーが list_issues ツールを宣言すると、Claude Codeからは以下のように見えます。

{
  "name": "list_issues",
  "description": "List issues in a GitHub repository",
  "inputSchema": {
    "type": "object",
    "properties": {
      "owner":  { "type": "string" },
      "repo":   { "type": "string" },
      "state":  { "type": "string", "enum": ["open","closed","all"] }
    },
    "required": ["owner", "repo"]
  }
}

この宣言だけで、Claude Code は「ユーザが『最近のopen issue見せて』と言ったらこのツールを呼ぼう」と自律的に判断できるようになります。

2026年1月時点のTips

公式ドキュメント上、Resources と Prompts はオプショナルです。実際のコミュニティサーバーの多くは Tools のみ実装しており、Resources/Prompts までサポートするのは公式リファレンス実装が中心です。Ch06 でカスタムサーバーを書く際も、まずはTools 1個から始めるのが定石です。

アーキテクチャ:Client ⇔ Server の対話

MCPは Client-Server アーキテクチャ です。Host(Claude Code等のAIアプリ)が Client を介して Server と通信します。

Host / Client / Server の関係

  User["👤 ユーザ"] --> Host["Host
(Claude Code)"]
  Host --> C1["Client A"]
  Host --> C2["Client B"]
  Host --> C3["Client C"]
  C1 -->|JSON-RPC| S1["Server
GitHub MCP"]
  C2 -->|JSON-RPC| S2["Server
Filesystem MCP"]
  C3 -->|JSON-RPC| S3["Server
自作MCP"]
  S1 --> API1["GitHub API"]
  S2 --> FS["ローカルFS"]
  S3 --> Biz["社内システム"]
Host 1つにつき複数のClient/Serverペア。各サーバーは独立プロセスで、stdio か HTTP で喋る。

通信フロー(stdio transportの場合)

起動から初回ツール呼び出しまで
sequenceDiagram
  participant H as Host (Claude Code)
  participant C as Client
  participant S as Server (MCPサーバー)
  H->>S: プロセス起動 (stdio)
  C->>S: initialize (capabilities交換)
  S-->>C: initialize result
  C->>S: tools/list
  S-->>C: ツール一覧を返す
  Note over H,S: ここまで接続確立
  H->>C: ユーザがClaude Codeに質問
  C->>S: tools/call (list_issues, {owner,repo})
  S-->>C: JSON結果
  C-->>H: AIに結果を渡す
起動時にcapabilityをネゴシエーションし、その後は必要な時だけtools/callで呼び出す。

Transport(通信方式)は主に2種類

Transport用途典型例
stdioローカル実行。標準入出力で JSON-RPC をやりとり自作MCPサーバー、Filesystem、ローカルのGit操作
HTTP (Streamable)リモートサーバー。HTTPでJSON-RPC + SSEでストリーミングSaaS提供のMCPエンドポイント、社内リモートサーバー
もっと詳しく: SSE transport は非推奨へ

MCPの初期仕様には HTTP + SSE(Server-Sent Events)ベースの transport もありましたが、2025年のspec更新で Streamable HTTP に統合・再設計されました。2026年1月時点では、新規サーバーは stdio か http(Streamable) を選ぶのが推奨で、旧SSE単独のサーバーを見かけたら更新検討のサインです。最新動向は MCP specification を確認してください。

ちょっと動かしてみる:最小のハンズオン

Ch02 で詳しく扱いますが、イメージを先に掴むためにコマンドだけ先出しします。Claude Code のインストール済みであれば、1コマンドで公式Filesystemサーバーを追加できます(コピペで試せます)。

# 公式 Filesystem MCPサーバーを user scope で追加
claude mcp add filesystem -s user \
  -- npx -y @modelcontextprotocol/server-filesystem /Users/you/Documents

# 登録済みサーバーを確認
claude mcp list

# 出力例:
# filesystem: npx -y @modelcontextprotocol/server-filesystem /Users/you/Documents (stdio)
#   Status: ✓ Connected
#   Tools:  read_file, write_file, list_directory, ...

このあと Claude Code を起動して「~/Documents の中にあるMarkdownファイル一覧を出して」と頼むと、モデルが list_directory ツールを自律的に呼び出し、結果を整形して返します。「AI が自分で外部ツールを呼ぶ」という体験を、最短5分で味わえます。

注意

Filesystemサーバーに渡すパスは、AIが読み書きしてよい範囲です。/ や $HOME をそのまま渡さないでください。最初は ~/Documents/mcp-test のような専用ディレクトリを切るのが安全です。セキュリティ詳細は Ch07 で扱います。

他の連携手段との比較

「外部ツールにAIを繋ぐ」手段は MCP 以外にも複数あります。Claude Codeユーザが実務で出会う選択肢を整理しておきましょう。

方式何をする強み弱み使いどころ
MCP AIクライアント<->ツールを標準プロトコルで接続 共通規格、ローカル実行、再利用性高 サーバーを別途用意する必要あり 永続的に使うツール連携、複数クライアントで使い回したい時
Function Calling (API) Claude API等で関数定義をモデルに渡す 自分のアプリ内で完結、柔軟 アプリごとに実装、クライアント横断で再利用不可 自作のAI組み込みアプリを書く時
Claude Code 内蔵ツール Read / Edit / Bash / Grep 等 設定不要、即使える、権限モデルが統一 拡張できない、カスタムなし ファイル編集・シェル実行など汎用タスク
独自API連携 シェルスクリプト+curlでREST呼び出し 最速で試せる、依存ゼロ 認証・エラー処理を毎回書く、再利用困難 一度きりのスクリプト、PoC
Subagent / Hooks Claude Codeの拡張機構でイベント発火 Claude Code固有の挙動拡張に強い 他クライアントでは使えない プロンプト前後の自動処理、特定タスクの専門エージェント化

同じタスク「GitHubのissueを一覧する」を4方式で書くと

MCP方式

再利用性 ◎

  • GitHub MCPサーバーを追加
  • 「open issueを見せて」と自然文
  • Cursorや他クライアントでも同じサーバーで動く
内蔵Bash

即席 ◎

  • gh issue list --state open
  • 1回きりなら最速
  • 毎回コマンド文を指示する必要あり
独自API + curl

学習コスト高

  • 認証ヘッダ、pagination、rate limit を自前で
  • 成功しても再利用が効かない
  • 長期運用には向かない

判断フロー:MCP か、それ以外か

「その連携、MCPでやる?」判定フロー

Exercise 1
入門

N×M問題を自分のケースで描く

あなたが今使っている(あるいは使いたい)AIクライアントと外部ツールを列挙し、本章で見た「個別実装」と「MCP経由」の両方の図を描いてみましょう。紙でも、テキストでもOKです。

  1. 自分が触っているAIクライアントを2-3個書き出す(例: Claude Code, ChatGPT, Cursor)
  2. 日常的に使いたい外部ツールを3-5個書き出す(例: GitHub, Slack, Notion, Google Drive, Postgres)
  3. 個別実装だと何本のコネクタが必要か、MCP経由だと何本になるかを計算する
  4. 削減率を「N+M / N×M」として出す
模範解答を見る

例: Mさんのケース(クライアント3、ツール5)

クライアント: Claude Code / Cursor / Claude Desktop  (N=3)
ツール:       GitHub / Slack / Notion / Postgres / Filesystem  (M=5)

個別実装: 3 × 5 = 15本のコネクタ
MCP経由:  3 + 5 = 8本(各クライアントが1つMCPクライアントを持ち、
                     各ツール1つがMCPサーバーを持つ)

削減率: 8 / 15 ≒ 53%  (つまり約半分)

さらに、ツールをもう1つ追加したら:
  個別実装は +3本(クライアント数分)
  MCP経由は  +1本 のみ

ポイント: 数字で書くと「なぜこの規格が産業標準になろうとしているのか」が腑に落ちます。ツール数が増えるほどMCPの効果は非線形に伸びます。

Exercise 2
応用

最小のMCPサーバーを発見して、やりとりを観察する

実際に Filesystem MCPサーバーを追加して、Claude Code が「どのツール」を「どの引数で」呼んだかを観察してみましょう。Ch02 先取りの準備運動です。

  1. 専用テストディレクトリを作る: mkdir -p ~/Documents/mcp-test && echo "hello" > ~/Documents/mcp-test/hi.md
  2. 公式Filesystemサーバーを追加: claude mcp add fs-test -s user -- npx -y @modelcontextprotocol/server-filesystem ~/Documents/mcp-test
  3. claude mcp list で Status が ✓ Connected になることを確認
  4. Claude Codeを起動して「~/Documents/mcp-test にあるファイルを一覧して、各ファイルの内容を要約して」と依頼
  5. どのツール(list_directory / read_file 等)が何回呼ばれたかを、Claude Codeの出力で確認
  6. 終わったら claude mcp remove fs-test -s user で片付ける
模範解答を見る

想定される観察結果:

# 期待されるClaude Codeの動き
1. list_directory({ path: "/Users/you/Documents/mcp-test" })
     → ["hi.md"] を取得
2. read_file({ path: "/Users/you/Documents/mcp-test/hi.md" })
     → "hello" を取得
3. モデルが結果を整形して「hi.md があり、内容は "hello" です」と回答

# claude mcp list の出力例
fs-test: npx -y @modelcontextprotocol/server-filesystem /Users/you/Documents/mcp-test (stdio)
  Status: ✓ Connected
  Tools:  read_file, write_file, list_directory, move_file, search_files, ...

解説: MCPサーバーが公開するツールは起動時に Claude Code が tools/list で問い合わせて一覧化しています。自然文での質問を、モデルが 自律的に適切なツールコール列に翻訳しているのが確認できれば、MCPの本質は掴めています。

補足: もし Status が Connected にならない場合は、Node.js が入っているか(node -v)、npx が動くかを確認してください。Ch07 でトラブルシュートを扱います。
