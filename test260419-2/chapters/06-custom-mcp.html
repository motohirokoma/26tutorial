カスタムMCPサーバー入門

最小構成で自作、TypeScript/Pythonひな型

所要時間: 40分 Phase: Practice 難易度: 中級
この章のゴール

公式MCPサーバーに無い機能は自分で作れる。この章ではMCP SDKを使い、echoツールだけの最小サーバーを TypeScript / Python の両方で実装し、Claude Codeに登録して動かします。完了すると「自作MCPをClaude Codeから呼ぶ」感覚が身につきます。

本章の前提(2026年1月時点)

記述は2026年1月時点のMCP SDKバージョン(TypeScript SDK 1.x / Python SDK 1.x)に基づきます。APIは安定してきていますが、最新の仕様は必ず modelcontextprotocol.io と各SDKのREADMEで確認してください。

なぜ自作するのか ― 公式では届かない場所

Ch03〜05で公式MCPサーバー(GitHub / Filesystem / Slack など)を一通り触りました。多くの業務はこれで回ります。しかし「自分たちの社内ツール」「独自ワークフロー」「ニッチなSaaS」は公式カタログに存在しません。ここで登場するのがカスタムMCPサーバーです。

公式MCPで届く範囲 vs カスタムMCPで届く範囲
公式は幅広い定番ツール、カスタムは「自分たちの業務」に特化したツールを埋める。

自作を選ぶ典型シーン

  • 社内ツール連携 — 社内勤怠API、独自CRM、オンプレのチケット管理など、公式MCPがあり得ないもの。
  • 独自ワークフローの集約 — 「このボタン1つで Jira+Slack+GitHub を同時に動かす」のような合成操作を、MCPツール1個として包む。
  • 独自DB・独自ファイル形式 — Obsidian Vault、自分用Notion代替、社内Wiki等。スキーマを知っているのは自分だけなので、専用ツールを書くのが最速。
  • 安全な抽象化 — 「危険な操作は絶対にさせない」ラッパ。たとえばDBのSELECTだけ通す。公式MCPが広すぎる時の安全策。

逆に公式で済むなら自作しないのが原則です。メンテコストはゼロではないので、「公式サーバーに欲しい機能があるか → 足りない部分だけ薄く自作する」が賢い判断。自作したMCPが負担になってくると、Claude Code環境自体の安定性も落ちます。

MCP SDK ― TypeScript / Python の2択

MCPサーバーは任意の言語で書けますが、2026年1月時点で公式SDKが提供されているのはTypeScript / Python / Kotlin / Java / Swift / C#など。中でも日常のツール作成で最も使われているのが TypeScript と Python です。この章ではこの2つを扱います。

最小構成とツールSchema ― 核は3フィールド

MCPサーバーが提供する1個のツールは、ツールSchemaという定義で表されます。核となるフィールドはたった3つ。

name
ツールの識別子 (例: echo)。半角英数字と記号のみ。
description
LLMが「いつ使うべきか」を判断する文。丁寧に書くほど正確に呼ばれる。
inputSchema
JSON Schema形式での引数定義。型・必須フィールド・説明を含む。

echoツールのSchema(具体例)

{
  "name": "echo",
  "description": "Echo back the provided text. Useful for testing MCP connectivity.",
  "inputSchema": {
    "type": "object",
    "properties": {
      "text": {
        "type": "string",
        "description": "The text to echo back."
      }
    },
    "required": ["text"]
  }
}
description の書き方は精度に直結

descriptionはLLMだけが読む短い説明書。「何をするか」「いつ使うべきか」「いつ使わないべきか」を英語で具体的に書くと誤呼び出しが減ります。日本語でも動きますが、ツール選択の精度は英語が一段高めです(モデルの学習偏り)。

もっと詳しく: inputSchemaで使える型と制約

MCPのinputSchemaはJSON Schema Draft 2020-12互換。よく使う要素:

  • type: string / number / integer / boolean / array / object
  • enum: 許可する値リスト (例: "enum": ["low", "medium", "high"])
  • minimum / maximum: 数値の範囲
  • pattern: 文字列の正規表現
  • items: 配列の要素型
  • required: 必須フィールドの配列

ここを厳しく書くほど、LLMが間違った引数を渡す事故が減ります。特にenumは強力 ― モデルが勝手に新しい値を作り出すのを防ぎます。

stdio通信の基本 ― 1行1メッセージ

MCPの最もポピュラーなトランスポートがstdio。仕組みは驚くほどシンプルで、Claude CodeがサーバープロセスをspawnしてstdinにJSONを書き、stdoutから1行ずつJSONを読むだけです。

stdio通信のライフサイクル
sequenceDiagram
  participant C as Claude Code
  participant S as MCP Server (子プロセス)
  C->>S: spawn (node/python を起動)
  C->>S: stdin ← initialize
  S-->>C: stdout → server info
  C->>S: stdin ← tools/list
  S-->>C: stdout → [echo]
  Note over C,S: ここから通常稼働
  C->>S: stdin ← tools/call echo {"text":"hi"}
  S-->>C: stdout → "hi"
  C->>S: 終了シグナル
  S-->>C: プロセス終了
初期化 → ツール列挙 → 呼び出し の3段階。いずれも1行のJSON-RPCメッセージ。
絶対守るルール: stdoutは汚さない

stdioサーバーでは stdoutはプロトコル通信専用。console.log / print でstdoutに書き込むとJSONが壊れて即死します。デバッグ出力は必ずstderr(console.error / sys.stderr)へ。SDKはstderrを汚してもOKに設計されています。

ハンズオン①: TypeScriptでechoサーバー

ではハンズオンに入ります。まずはTypeScript SDKでechoツール1個だけのサーバーを作ります。30分で動くところまで行きます。

1
プロジェクト初期化
npm init + SDK導入
2
server.ts 実装
SDKでツール登録
3
ビルド&実行確認
手動でJSONを流して動作検証
4
Claude Codeに登録
.mcp.jsonに書いて再起動

Step 1: プロジェクト初期化

mkdir mcp-echo-ts && cd mcp-echo-ts
npm init -y
npm install @modelcontextprotocol/sdk zod
npm install -D typescript tsx @types/node
npx tsc --init --target es2022 --module node16 --moduleResolution node16 --outDir dist --rootDir src

zodは引数のランタイム検証用。SDK内部でもZodを使っているので親和性抜群です。

Step 2: src/server.ts を書く

// src/server.ts
import { Server } from "@modelcontextprotocol/sdk/server/index.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import {
  CallToolRequestSchema,
  ListToolsRequestSchema,
} from "@modelcontextprotocol/sdk/types.js";

// 1. サーバーインスタンス
const server = new Server(
  { name: "mcp-echo-ts", version: "0.1.0" },
  { capabilities: { tools: {} } }
);

// 2. ツールのカタログを返すハンドラ
server.setRequestHandler(ListToolsRequestSchema, async () => {
  return {
    tools: [
      {
        name: "echo",
        description:
          "Echo back the provided text. Useful for testing MCP connectivity.",
        inputSchema: {
          type: "object",
          properties: {
            text: {
              type: "string",
              description: "The text to echo back.",
            },
          },
          required: ["text"],
        },
      },
    ],
  };
});

// 3. ツール呼び出しハンドラ
server.setRequestHandler(CallToolRequestSchema, async (request) => {
  const { name, arguments: args } = request.params;
  if (name === "echo") {
    const text = String(args?.text ?? "");
    // ※ stdoutは絶対に汚さない。ログはstderrへ
    console.error(`[echo] received: ${text}`);
    return {
      content: [{ type: "text", text: `echo: ${text}` }],
    };
  }
  throw new Error(`Unknown tool: ${name}`);
});

// 4. stdioトランスポートで起動
const transport = new StdioServerTransport();
await server.connect(transport);
console.error("mcp-echo-ts running on stdio");

Step 3: 手動動作確認

# tsxで直接実行できる(ビルド不要)
npx tsx src/server.ts

# 別のターミナルから動作確認する場合は、Inspector(後述)がおすすめ
# Inspector不使用の簡易テスト: サーバーを起動したターミナルに直接JSONを貼る
# (Ctrl+D で終了)

起動すると stderr に mcp-echo-ts running on stdio と出ます。何も出ないまま入力待ちになれば成功。

Step 4: Claude Codeに登録

プロジェクトのルートに.mcp.jsonを作成 (user scopeなら~/.claude/mcp.json)。

{
  "mcpServers": {
    "echo-ts": {
      "command": "npx",
      "args": ["tsx", "/absolute/path/to/mcp-echo-ts/src/server.ts"]
    }
  }
}

保存後、Claude Codeを再起動。claude mcp list で echo-ts が出れば成功。Claude Codeに「echoツールを使って "hello from MK" を返して」と頼むと、実行して結果が返ってきます。

ハンズオン②: Pythonでechoサーバー

同じechoツールをPython SDKで作ります。TypeScript版が動いているなら、こちらは15分で終わります。

Step 1: セットアップ

mkdir mcp-echo-py && cd mcp-echo-py
python -m venv .venv
source .venv/bin/activate     # Windowsは .venv\Scripts\activate
pip install "mcp[cli]"

mcp[cli]はサーバー本体 + Inspector等の開発補助CLIをまとめてインストールします。

Step 2: server.py を書く

# server.py
import sys
from mcp.server.fastmcp import FastMCP

# FastMCP は低ボイラープレートのハイレベルAPI
mcp = FastMCP("mcp-echo-py")

@mcp.tool()
def echo(text: str) -> str:
    """Echo back the provided text. Useful for testing MCP connectivity.

    Args:
        text: The text to echo back.
    """
    # ※ stdoutは絶対に汚さない。ログはstderrへ
    print(f"[echo] received: {text}", file=sys.stderr)
    return f"echo: {text}"

if __name__ == "__main__":
    # stdioトランスポートで起動
    mcp.run()

注目ポイント: Pythonではdocstringが description に、型ヒントが inputSchema に自動変換されます。TypeScript版で20行だったコードが数行になりました。

Step 3: 動作確認とClaude Code登録

# 直接起動テスト
python server.py
# → stderrに起動メッセージ、stdinで待機

# Inspectorで対話的にテスト(後述)
npx @modelcontextprotocol/inspector python server.py

.mcp.jsonの例:

{
  "mcpServers": {
    "echo-py": {
      "command": "/absolute/path/to/mcp-echo-py/.venv/bin/python",
      "args": ["/absolute/path/to/mcp-echo-py/server.py"]
    }
  }
}
venvのPythonパスが肝

commandにはvenv内のpython絶対パスを指定。システムのpythonを指すとmcpパッケージが見つからずサーバーが落ちます。Ch07の「トラブルシュート」でもこのパターンを扱います。

TypeScript版とPython版を両方試すのを強くおすすめします。同じ概念が違うシンタックスで見えると、「MCPの本質はstdioで動くJSON-RPCサーバー」という構造感がはっきり身につきます。

デバッグ戦略 ― 3つの武器

MCPサーバーは子プロセスなのでデバッグしにくい印象がありますが、実は武器が揃っています。

武器①

MCP Inspector

  • npx @modelcontextprotocol/inspector ...
  • GUIでツール一覧・呼び出しが可能
  • Claude Code不要でテストできる
武器②

stderr ログ

  • console.error / print(file=sys.stderr)
  • Claude Code起動ログに混ざって見える
  • 本番運用中でも消さない
武器③

手動JSON-RPC

  • サーバー起動 → stdinに直接JSONを貼る
  • プロトコルレベルで疑うとき用
  • 最終手段、でも本質が見える

MCP Inspector の使い方

# TypeScript版サーバーをInspectorで開く
npx @modelcontextprotocol/inspector npx tsx src/server.ts

# Python版
npx @modelcontextprotocol/inspector python server.py

ブラウザが開き、左側にツール一覧、右側にリクエスト/レスポンスのログが出ます。Claude Codeに繋ぐ前に必ず Inspector でツールが見えることを確認すれば、原因切り分けが爆速になります。

デバッグ時の問題→武器 対応表

よくあるハマりどころ

  1. stdoutにconsole.logが残っている — サーバーが突然死する。console.errorに置換。
  2. 絶対パスではなく相対パス — Claude Codeはcwdが違う。.mcp.jsonのargsは必ず絶対パス。
  3. venvのPythonを使っていない — ModuleNotFoundError: mcp。絶対パスでvenv内pythonを指す。
  4. SDKバージョン不一致 — TypeScript SDK 1.x系 と 0.x系はAPIが違う。package.jsonを確認。
  5. 再起動忘れ — .mcp.jsonを変えたらClaude Code自体を再起動。

  1. TypeScript版かPython版のどちらかを選ぶ(好みで)
  2. 本文の手順通りにセットアップ・実装
  3. MCP Inspector で echoツールが見えることを確認

模範解答を見る

期待される最終状態:

  • claude mcp list に自作サーバー名が出る
  • Claude Codeで「echoツールで X を返して」と頼むと、echo: X が返る
  • stderr(Claude Codeの起動ログ)に[echo] received: Xが出る

もし動かない場合の確認順:

  1. Inspector単体で動くか ― 動かないならサーバー側コードのバグ
  2. .mcp.jsonのcommand/argsが絶対パスか ― 相対パスは落ちる
  3. Claude Codeを完全に再起動したか ― キャッシュが残る
  4. stdout汚染がないか ― console.log / 生のprintを探してconsole.error / print(..., file=sys.stderr)に

この順で切り分ければ、ほとんどの初回トラブルは10分以内に解決します。

Exercise 2
応用

独自ツールを1つ追加する

echoサーバーに自分の独自ツールを1つ追加してください。題材は何でもOKですが、以下のどれかをおすすめします。

  1. random_quote: 引数なし、ランダムに名言を1つ返す
  2. word_count: text: stringを受け取り、単語数と文字数を返す
  3. now: timezone: stringを受け取り、その地域の現在時刻を返す
  4. 独自アイデア: 自分が日常で「あったら便利」と思う関数

追加後、Inspectorで新ツールが見えること、Claude Codeから呼べることを確認してください。

模範解答を見る

Python版で word_count を追加する例:

# server.py に追記
@mcp.tool()
def word_count(text: str) -> dict:
    """Count words and characters in the provided text.

    Args:
        text: The text to analyze.
    """
    words = len(text.split())
    chars = len(text)
    chars_no_space = len(text.replace(" ", ""))
    return {
        "words": words,
        "chars": chars,
        "chars_no_space": chars_no_space,
    }

Python版は型ヒント + docstringだけでSchema自動生成。returnはdictやstrなら自動でテキスト化されます。

TypeScript版で word_count を追加する例:

// ListToolsRequestSchema のtools配列に追加
{
  name: "word_count",
  description: "Count words and characters in the provided text.",
  inputSchema: {
    type: "object",
    properties: {
      text: { type: "string", description: "The text to analyze." },
    },
    required: ["text"],
  },
},

// CallToolRequestSchema ハンドラに分岐追加
if (name === "word_count") {
  const text = String(args?.text ?? "");
  const result = {
    words: text.split(/\s+/).filter(Boolean).length,
    chars: text.length,
    chars_no_space: text.replace(/\s/g, "").length,
  };
  return {
    content: [{ type: "text", text: JSON.stringify(result) }],
  };
}

確認ポイント:

  • Inspector の左ツール一覧に2個(echoとword_count)出る
  • descriptionが英語で具体的 ― LLMは「テキストの単語数を数える時に使うツール」と正確に判断できる
  • Claude Codeに「この段落の単語数をword_countで数えて」と頼むと、適切に引数がセットされて呼ばれる

発展: 独自ツールが動いたら、Ch08(最終演習)で「自分の主戦場ツール」をMCP化する題材として、同じ形式でどんどんツールを足していけます。MCPサーバーの本質は「ツール関数を並べるだけ」であることが、このExerciseで腑に落ちるはずです。
