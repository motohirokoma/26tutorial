失敗パターン10：ここで詰まる

先人が踏んだ地雷を10個先に踏んで学ぶ。症状→原因→対処を一気に整理する章。

所要時間: 60分 Phase: Advanced 難易度: 中級
この章のゴール

Claude Code を1週間以上使うと、ほぼ全員が同じ10個の失敗を必ず踏みます。Advanced 段階に入った今こそ、この地雷一覧を先に見て先に避けるのが最も費用対効果が高い。Ch01〜Ch21 で学んだ武器を、暴発させない技術に仕上げる60分です。

バージョン注記

本章の記述は2026年1月時点の Claude Code (Agent SDK / CLI) 挙動に基づきます。/compact /clear Hooks MCP の挙動は進化するため、コマンドが通らない時は 公式ドキュメント → を確認してください。

10失敗の全体像 — まず一覧で俯瞰する

失敗は大きく3層に分かれます。対話の失敗(1-4)、運用の失敗(5-7)、自動化の失敗(8-10)。下の表でfrequency と severity を見て、まず自分が直近に踏んだものにチェックを入れてみてください。

#失敗パターン層頻度重さ該当した?
1ベタな頼み方対話★★★★□
2大きすぎる依頼対話★★★★★□
3diffを見ない対話★★★★★★□
4テストを緩めさせる対話★★★★★□
5/compact /clear 使いすぎ/なさすぎ運用★★★★★□
6長時間1セッション運用★★★★★□
7Plan なしの危険操作運用★★★★★□
8Bash 盲信自動化★★★★★□
9MCP 権限広すぎ自動化★★★★★□
10Hooks 無限ループ自動化★★★★□
ここがプロの境目

他人の失敗から学ぶのが最速。自分で全部踏んで覚えるのは高くつく——特に「diffを見ない」「MCP権限広すぎ」「Hooks無限ループ」は本番DB/本番リポに被害が及ぶ級。パワーユーザーは必ず「他人がどこで詰まったか」を先に読んで、同じ穴を掘らない。
試し方： 上の表の「該当した?」列を今すぐ自分で□→■に塗ってみる。1つでも当てはまったら、このあとのそのセクションだけ先に読む。

失敗が発生する3つの層
層が深くなるほど被害が大きい。自動化層の失敗は本番に届く。

対話層の失敗 1〜4

Fail 1: ベタな頼み方

症状

Claude の返答が薄い・的外れ

  • 「いい感じにして」「なんかおかしい直して」
  • 返答が一般論だけ、自分のコードに刺さらない
  • 何度も「もっと具体的にすると…」と前置きされる
対処

3点セットを渡す

  • 具体性: どのファイル/関数か
  • 制約: 使って良いライブラリ/禁止事項
  • 成功条件: テストが通る/出力が○○

原因: 「頼む=自然言語で書く」という思考の範囲で止まっている。Claude は成功条件が無いと「もっと良くできるのでは」と無限に書き換え続ける。Ch02 の3点セットに戻って、テンプレ化することで解決する。

Fail 2: 大きすぎる依頼

症状: 「認証まわり全部リファクタして」の1発依頼 → 返ってきた diff が500行超、何が変わったか追えない。

依頼サイズと制御可能性

原因: context pollution が起きて、Claude が各所でほどほどに妥協した解を返す。結果、どこも完成していない diff ができる。

対処: 依頼を15-30分で終わる単位に切る。TodoWrite で先にリスト化(Ch05)→ 1項目ずつ依頼 → 都度 commit(Ch07)。

もっと詳しく: 大きな依頼をどう分割するか

分割の型は「縦切り」と「横切り」の2つ。

  • 縦切り (vertical slice): ユーザー動線1本を UI→API→DB まで通す。「ログイン画面だけ」「パスワードリセットだけ」
  • 横切り (horizontal slice): 1レイヤーだけ全部直す。「全エンドポイントにバリデーション追加」「全 model に tests 追加」

不明なら縦切り優先。動くものが早く手に入り、確認が早い。

Fail 3: diff を見ない

症状: Claude が「完了しました」と言った → そのまま commit → 本番で壊れる。

実例

あるチームでは、Claude が「型エラーを直した」と報告 → 内部では as any を15箇所追加していた。3ヶ月後に別のバグの温床になった。diff を見ていれば5秒で防げた。

原因: Claude の「完了しました」という自然言語の報告を信じてしまう癖。自然言語は解像度が粗い。真実は git diff にしか書かれていない。

対処: 依頼のたびに

git diff --stat  # 変更ファイル数と行数をまず把握
git diff         # 実際の変更を全部読む
git diff -- ':(exclude)package-lock.json'  # lock除外で本体だけ

これをルーティン化する。Hook で「commit 前に diff を表示」を自動化すれば、意志力ゼロでできる(Ch18)。

Fail 4: テストを緩めさせる

症状: 「テスト通して」と頼む → テストが通った → よく見るとテスト側が書き換わって通っただけ。

テスト改変の起こり方
graph LR
  A["テスト失敗"] --> B{"Claude の判断"}
  B -->|"正攻法"| C["実装を直す"]
  B -->|"楽な道"| D["assertion を緩める"]
  D --> E["テストが通る"]
  E --> F["仕様が壊れる"]
  C --> G["バグが直る"]
楽な道に逃げさせない指示が必要。

原因: 「テストを通す」というゴールだけ渡すと、Claude は最短経路を選ぶ。それが assertion 改変のこともある。

対処: 依頼文に必ず入れる一文:

テストは絶対に変更しないでください。テストを通すためにテスト側を書き換えた場合、
その修正は無効です。実装側だけ修正してください。

運用層の失敗 5〜7

Fail 5: /compact /clear 使いすぎ/なさすぎ

/compact と /clear の判断ミスには2種類あります。

タイプ症状原因対処
使いすぎ 作業途中で文脈が薄くなり、Claude が「どこまでやったっけ」状態に 不安で5分おきに /compact context bar が 70% 超えてから /compact で十分
なさすぎ context 満杯警告 → 次の依頼が途中で切れる / 応答が遅い 「進んでいるから止めたくない」病 切れ目(機能完成・commit後)で /clear を入れる癖を付ける
/compact と /clear の使い分けフロー
graph TD
  A["context 70%超え?"] -->|"Yes"| B{"作業継続したい?"}
  A -->|"No"| Z["そのまま続ける"]
  B -->|"Yes"| C["/compact で要約圧縮"]
  B -->|"No"| D{"機能の切れ目?"}
  D -->|"Yes"| E["/clear で白紙から"]
  D -->|"No"| C
「満杯前に compact、切れ目で clear」が基本ルール。

Fail 6: 長時間1セッション

症状: 4時間ぶっ通しで同じセッション → 後半、Claude の出力が明らかに劣化 / 前半の指示を忘れる / ハルシネーション増。

原因: context window は有限。長く使うほど、序盤のメッセージは圧縮された影になり、Claude の「いま扱っている作業」の解像度が落ちる。

セッション長と出力品質の相関(目安)

対処: 90分ルール。90分/1セッションで区切り、機能完成ごとに /clear。CLAUDE.md に重要情報を書いておけば文脈はすぐ戻る(Ch06)。

Fail 7: Plan なしの危険操作

症状: 「DB マイグレーションやっといて」→ 実行してから「あれ、rollback どうすれば…」

公式ドキュメント必読

ここは公式ドキュメントにも書いていますし、実務で間違えると致命的なので必ずマスターしてください。Plan モードと ExitPlanMode の挙動は Claude Code の安全機構の中核です。Claude Code 公式 →

対処: Plan モードをデフォルト起動するレベルの依頼の基準を決める:

  • データを変更する — DB migration / バッチジョブ / 一括ファイル変換
  • 履歴を書き換える — git rebase / force push / 履歴削除
  • 外部へ送信する — デプロイ / API 呼び出し / メール送信
  • 削除系 — rm -rf / DROP TABLE / git reset --hard

自動化層の失敗 8〜10

Fail 8: Bash 盲信

症状: Claude の出す Bash コマンドを確認せず実行 → rm -rf ~/Desktop/work で3日分の作業消失。

原因: Claude の Bash 生成精度は高いが、パスの間違い/変数展開の勘違いが混じる時がある。特に環境変数・ワイルドカードは事故源。

危険

盲信パターン

rm -rf $DIR/*
find . -name "*.log" -delete

$DIR が空なら rm -rf /* に化ける。

安全

確認→実行

ls "$DIR"           # 中身確認
[ -n "$DIR" ] || exit 1  # 空ガード
rm -rf "$DIR"/*

クォート必須、ガード必須。

Fail 9: MCP 権限広すぎ

症状: Slack MCP に全チャンネル write 権限 → Claude が誤って #general に下書きメッセージ送信 → 全社に通知。

原因: MCP scope 設定時に「とりあえず全権限」にすると、Claude は躊躇せず使う。権限があれば使う、が MCP の原則。

MCP 権限の段階

対処: 最小権限の原則。まず read-only で試す → 必要になった時だけ scope を拡張 → 本番リソースと dev リソースは必ず別 MCP で分離。

もっと詳しく: MCP 権限設計のチェックリスト
  • ✅ 初期接続は必ず read-only scope から
  • ✅ write 権限を与える時、resource ID で絞れるなら絞る(「特定DB」「特定チャンネル」のみ)
  • ✅ 本番/開発で別MCP接続・別トークン
  • ✅ .mcp.json を git にコミットする時、トークンが直書きされていないか最終確認
  • ✅ 半年に1回、使ってない scope を削る棚卸し

Fail 10: Hooks 無限ループ

症状: PostToolUse hook で「Prettier 整形」を仕掛けた → Prettier の実行が Write を発生 → また hook が発火 → ∞。

Hooks 無限ループの発生経路
sequenceDiagram
  participant U as User
  participant C as Claude
  participant H as Hook
  participant T as Tool
  U->>C: ファイル書いて
  C->>T: Write 実行
  T-->>H: PostToolUse 発火
  H->>T: Prettier (=Write) 実行
  T-->>H: PostToolUse 発火
  H->>T: Prettier 再実行
  Note over H,T: ∞
整形系 hook は自分自身を再発火しやすい典型パターン。

原因: Hook が自分の発火条件を自分で再生成してしまう設計ミス。特に「Write 後に整形」「Edit 後に lint fix」系は要注意。

対処:

  1. 無限ループ防止フラグをhook内に置く(環境変数 CLAUDE_HOOK_DEPTH を見て深さ1で抜ける等)
  2. 整形コマンドを --write モードではなくチェックのみで走らせる
  3. hook 対象を特定の tool のみに絞る(matcher で tool_name 指定)
もっと詳しく: Hooks の安全な書き方テンプレ 
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "[ \"$CLAUDE_HOOK_DEPTH\" -gt 0 ] && exit 0; CLAUDE_HOOK_DEPTH=1 prettier --check \"$CLAUDE_FILE_PATH\""
          }
        ]
      }
    ]
  }
}

ポイント: (1) matcher で対象tool絞る (2) 環境変数で再帰検知 (3) --check で副作用回避。

10失敗のうち、自分が1度でもやったものを紙/メモにリストアップしてください。そのリストがあなた専用の地雷マップになります。次の1週間、Claude を使うたびに「これは Fail N を踏みそうじゃないか?」と自問する習慣を付けましょう。

失敗しない習慣の作り方 — 4ステップ

知識として10個を暗記しても、行動が変わらなければ意味がない。習慣化の4ステップ:

1
リストアップ
自分が踏んだ失敗を書き出す
2
テンプレ化
回避フレーズをCLAUDE.mdに
3
Hook化
機械的に防げる部分を自動化
4
月次レビュー
毎月1日、失敗ログを振り返る

特に 2. テンプレ化 が効きます。CLAUDE.md に以下のような一文を入れておく:

# 失敗回避ルール (Ch22 より)
- 依頼は15-30分で終わる粒度に切ること
- テストは絶対に書き換えないこと。テスト改変は無効
- 危険操作 (DB/deploy/rm) は必ず Plan モードから
- Bash 削除系は ls で中身確認してから
- commit 前に必ず git diff を読むこと

これで、Claude は毎回このルールを読んでから動きます。ルール違反の失敗を Claude 側でも踏みにくくなる。

失敗は知識として持っていても意味がない。行動が変わる仕組みに落とし込んではじめて価値になる。 — Ch22の要点

Exercise 2
応用

Hook 無限ループのデバッグ

あなたが作成した PostToolUse hook が無限ループに陥りました。以下の設定から原因を特定し、安全な設定に書き直してください。

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "*",
        "hooks": [
          { "type": "command", "command": "prettier --write ." }
        ]
      }
    ]
  }
}
  1. 何が問題か、理由を2つ以上挙げる
  2. 修正版の JSON を書く
  3. 修正後の動作を1行で説明する
模範解答を見る

問題点:

  1. matcher: "*" で全ツール発火 → Read/Grep 後にも走る無駄発火
  2. prettier --write . で全ファイルを書き換える → その Write が再び PostToolUse を発火させ無限ループ
  3. 再帰検知フラグなし
  4. 対象が . (全体) で範囲が広すぎる

修正版:

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "[ \"$CLAUDE_HOOK_DEPTH\" -gt 0 ] && exit 0; CLAUDE_HOOK_DEPTH=1 prettier --check \"$CLAUDE_FILE_PATH\" || true"
          }
        ]
      }
    ]
  }
}

動作: Write/Edit のみ発火、対象ファイルだけ --check(書き換えなし)、再帰深度1で抜けるので無限ループしない。
