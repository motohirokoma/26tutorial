プロンプト設計の型

5つの型を暗記して、毎回ゼロから考えない

所要時間: 60分 Phase: Core 難易度: 中級
この章のゴール

Claude Code への依頼は、大きく 5つの型(新規作成 / 修正 / 調査 / レビュー / デバッグ)に収まります。この章では型ごとのテンプレート・よくある反例・実際のコピペで使えるプロンプト例を用意しました。暗記して体に染み込ませるのがゴール。60分で、毎回頭を白紙から捻る必要がなくなります。

バージョン注記

本章の記述は 2026年1月時点 の Claude Code の挙動・用語に基づきます。スラッシュコマンドや Agent 周りは進化が速いので、コマンドが通らない時は 公式ドキュメント を確認してください。型そのもの(5分類)は1年単位では変わりません。

なぜ「型」なのか — 毎回ゼロから考えない

Ch02 で学んだ 3点セット(具体性・制約・成功条件)は全プロンプトの土台です。しかし毎回ゼロから「具体性は…制約は…成功条件は…」と組み立てていたら、1リクエストに3分かかって全然捗りません。そこで テンプレート を用意して、型に穴埋めする運用にします。

素人

毎回ゼロから文を捻る

  • 1プロンプトに2-3分かかる
  • 書き忘れ(制約・成功条件)が多発
  • 「なんか期待と違う」で再依頼の往復
  • 型の揺れでClaudeも精度にばらつき
パワーユーザー

5型のうち1つに当てはめる

  • 1プロンプトに20-30秒
  • 穴埋めなので欠落が起きにくい
  • 一発で意図通りの返答率が上がる
  • 型=共通言語なのでチームで再現可能

ポイントは すべての依頼を型に分類する習慣 です。「今から何を頼もうとしているか?」の最初の判断が 5択 に収まると気付いたら、あとはその型のテンプレをただ埋めるだけ。脳みそのワーキングメモリが空くので、「型に沿う → 型に沿わない例外を見つける」というプロの動き方に移れます。

5つの型と、依頼の入り口
依頼が来たら、まずこの5型のどれに入るかを1秒で判定。あとは型のテンプレを埋めるだけ。

5型の俯瞰 — 「この依頼はどれ?」を秒で判定する

いきなり個別の型を見るのではなく、まず俯瞰比較を頭に入れます。判定の速さが、型運用のスピードを決めます。

型 典型的な依頼 主役パラメータ 主なツール 危険度
1. 新規作成 「○○を作って」 要件 / 参考ファイル / 成功条件 Write / Edit / Bash 中(暴走に注意)
2. 修正 「この挙動を変えて」 対象ファイル / 差分 / 制約 Edit / Bash 高(既存破壊)
3. 調査 「○○はどこで使われてる?」 探す対象 / スコープ / 出力形式 Grep / Glob / Read 低(読むだけ)
4. レビュー 「このコードをチェックして」 対象 / 観点 / 重大度基準 Read / Grep 低(指摘のみ)
5. デバッグ 「このバグを直して」 再現手順 / 期待 / 観察 / 制約 Read / Edit / Bash 高(表層塞ぎ)

注目してほしいのは「主役パラメータ」列です。型ごとに「絶対に書かないとダメな項目」は決まっている。逆に言えば、この列を暗記しておけば、プロンプトに何を書くべきかで迷いません。

公式ドキュメント必読

Claude Code の「効果的なプロンプト設計」のベースは公式ドキュメントにも体系化されています。型を身につけた後、1度は通読して自分の型と照らし合わせてください。Claude Code 公式 →

型①:新規作成 — ゼロから書かせる

使う場面:まだ存在しない機能・スクリプト・テスト・ドキュメントをゼロから書いてもらう時。スキャフォールディング や小物CLIの生成に一番向いています。

新規作成プロンプトの4要素
graph LR
  A["1. 目的"] --> B["2. 要件 & 制約"]
  B --> C["3. 参考ファイル"]
  C --> D["4. 成功条件"]
  D --> E["Claude が実装"]
1-4を1メッセージに畳み込めば、ほぼ一発で期待通りが出る。

テンプレート(暗記推奨)

【目的】{何のため / どんなユースケース}
【要件】
- {機能A}
- {機能B}
【制約】
- 言語: {Python 3.11 / TypeScript strict / ...}
- 依存: {標準ライブラリのみ / 既存の X を使う}
- スタイル: {既存の {file} に合わせる}
【参考】
- 似た構造の既存ファイル: {path}
- 関連仕様: {URL or doc path}
【成功条件】
- {npm test が通る / "hello" と標準出力 / ...}
- 完了したら git diff を見せて停止

Before / After プロンプト例

Before(反例)

ベタ依頼

「CSV を JSON に変換するスクリプトを書いて」

  • 言語・入出力先・エラー処理が不明
  • どこまでで完成かの成功条件がない
  • 既存構造との整合が取れない
After(型どおり)

型①を埋めた依頼

【目的】運用チームの日次CSVをJSONに変換
【要件】
- 引数で入力CSVパス/出力JSONパスを受ける
- ヘッダ行をキーに
【制約】Python 3.11 / 標準ライブラリのみ
【参考】scripts/csv2tsv.py のCLI作法に合わせる
【成功条件】
- python convert.py sample.csv out.json で動く
- 終わったら diff だけ見せて停止
もっと詳しく:新規作成で暴走させないコツ
  • 成功条件を「止まる条件」として書く:「○○が出たら停止」「diff 見せて終わり」。これが無いとClaudeはどんどん機能を足してくる。
  • 参考ファイルを1つ指定する:既存コードのスタイルに寄せるだけで、レビュー時間が半分になる。
  • 大きな新規作成は Ch13 Plan モードと組み合わせる:いきなり書かせず、計画を出させる→同意→実装の3段。
今すぐ試す(90秒)

今の自分のプロジェクトで「作りたいけど作ってない小物」を1つ決めて、上のテンプレを埋めて Claude Code に投げてみてください(例:「当日作業ログを .md に書き出す Bash スクリプト」)。テンプレを埋めるだけで、**一発で動くコードが返ってくる確率**が自分でも驚くほど上がるのを体感できます。

型②:修正 — 既存コードを小さく変える

使う場面:既存挙動を少しだけ変える。メッセージ文言・パラメータ・条件分岐の追加など。最も事故が起きやすい型でもあります(既存破壊・余計な周辺修正)。

テンプレート

【対象】{file:line} または {関数名}
【現状】{今どうなっているか = 観察事実}
【変えたいこと】{どうしたいか、なぜか}
【変えてはいけない】
- {他の関数 / 他のファイル}
- {公開APIシグネチャ}
【制約】
- {追加依存なし / テストは現状維持 / ...}
【成功条件】
- {どのテストが引き続き通る / 差分は N 行以内}
- 終わったら git diff を見せて停止

ここで大事なのは 「変えてはいけない」 行。Claude Code は頼まれていない範囲まで親切心で直してしまうことがあります。これを明示的に禁止するのが、修正型で最も効く一行です。

修正型の依頼:書き方別「一発成功率」体感
MK実測の個人指標(2026/1 時点)。型に「変えてはいけない」行を1行足すだけで一発成功率が大きく変わる。

反例:よくある「抜け」

  • 対象の粒度が粗い — 「ログの出し方を変えて」だけだと全ファイル見に行って暴走する。必ずファイル/関数まで指定。
  • 「現状」を書かない — Claude 側が現状解釈で失敗する。grep 結果やconsole.logの抜粋を貼るだけで精度急上昇。
  • 「変えてはいけない」を書かない — 周辺コードをリファクタし始めたり、テスト側を書き換える事故の原因。
もっと詳しく:修正型 + Plan モードの合わせ技

1ファイル以内の修正なら型②だけで十分ですが、複数ファイルに波及しそうな修正は Plan モード(Ch13)と合わせると事故が減ります。「型②のテンプレを Plan モードに投げて、計画に同意したら実装」という2段運用が、パワーユーザーの典型パターンです。

型③:調査 — 事実を拾って報告させる

使う場面:リポジトリ内の何かを探す・まとめる・仕様を把握する。書き換えさせないのが肝。調査依頼で Claude にコードを触らせるとコンテキストだけ食われて損です。

テンプレート

【知りたいこと】{○○がどこで / どう使われているか}
【スコープ】
- 対象ディレクトリ: {src/ 配下のみ}
- 除外: {tests/ , node_modules/}
【出力形式】
- 箇条書き: {file:line — 1行要約}
- 最後に「全体傾向 3行」
【禁止】
- 書き換えない(Read/Grep/Glob のみ使う)
- 要約で原文を改変しない

調査型の失敗の9割は 「出力形式を指定しない」 こと。形式を指定しないと、Claude は長文の説明を返してきて、あなたはそれを読む時間で消耗します。箇条書き・表・3行サマリなど、欲しい形を先に指定しましょう。

1
知りたいこと
Yes/No で答えられるレベルに絞る
2
スコープ
対象ディレクトリと除外を明示
3
出力形式
箇条書き・表・3行サマリ など
4
書き換え禁止
Read/Grep/Glob のみ許可
Before

ざっくり質問

「このコードベースで認証周りはどうなってる?」

→ 長文の説明 + 勝手にリファクタ提案まで。時間ロス。

After

型③で依頼

【知りたい】認証ミドルウェアの適用箇所
【スコープ】src/ のみ、tests除外
【出力】- file:line — 1行要約 の箇条書き
最後に全体傾向を3行
【禁止】コード変更・新ファイル作成

→ 20秒で読める報告が返ってくる。

型④:レビュー — 観点を指定して辛口にチェックさせる

使う場面:自分が書いたコードや差分、他人のPRを Claude に読ませて指摘を出させる。観点を指定しないと「ざっくり良いと思います」みたいな無味乾燥なコメントしか返ってきません(Ch09で深掘り)。

テンプレート

【対象】{file:line 範囲 / git diff 範囲 / PR URL}
【観点】(複数指定可)
- [ ] セキュリティ(SQLi / XSS / 権限)
- [ ] パフォーマンス(N+1 / 無駄ループ)
- [ ] 可読性(命名 / 関数長)
- [ ] テスト(カバレッジ / 境界値)
- [ ] 設計(責務分離 / 副作用)
【重大度基準】
- critical: 本番で即バグ
- major: 将来確実に詰む
- minor: 改善余地
【出力形式】
- 観点別 × 重大度順で箇条書き
- 各指摘に「なぜ問題か」「どう直すか」2行
【禁止】コードを書き換えない(指摘のみ)
観点指定の有無で出てくる指摘数
5観点を明示すると、指摘の厚みが各方向に広がる(著者実測・参考値)。
ここがプロの境目

型を2〜3個暗記する。毎回ゼロから考えない。特に「新規作成」「修正」「レビュー」の3つを暗記しておくと、依頼の8割はこれで回る。Claude の返答ばらつきが減り、1日に捌けるタスク数が変わる。
試し方: 今日の自分の頻出タスクを1つ選んで、このページの該当型テンプレを埋め直して再依頼してみる(例:「いつもの週次レポート作成」を型①で書き直す)。

型⑤:デバッグ — 再現→仮説→修正

使う場面:バグを直させる。表層だけ塞がせないのが命。Ch12で深掘りしますが、ここでは最低限の型だけ押さえます。

テンプレート

【バグの概要】1行で {何が起きているか}
【再現手順】
1. {手順1}
2. {手順2}
3. {ここでエラー}
【期待動作】{こうなってほしい}
【観察結果】{実際のメッセージ・ログ・スタック}
【すでに試したこと】{○○を変えたが効果なし}
【制約】
- assertion を緩めない / テストを書き換えない
- 根本原因を先に仮説3つ挙げてから手を入れる
【成功条件】
- 再現手順で発生しないこと
- 既存テストが全て通ること
- 修正 diff + 原因仮説を見せて停止
デバッグ依頼が暴走する典型パターン
graph TD
  A["バグ報告"] --> B{"仮説を立てた?"}
  B -- "No" --> C["表層を塞ぐ patch"]
  C --> D["別バグが再発"]
  D --> A
  B -- "Yes / 3個挙げた" --> E["根本原因に到達"]
  E --> F["再現が消える"]
仮説3つを先に挙げさせるだけで、無限ループを抜ける。
もっと詳しく:デバッグ型でテストを守るフレーズ

Claude Code は「テストを通すためにテストを書き換える」ことがあります(Ch11 で詳説)。これを防ぐには、デバッグ型のプロンプトに 必ず 「assertion を緩めない」「テストを書き換えない」を入れる。Ch11 の先取りですが、型⑤にはこの行を常駐させるのが安全です。

全型に共通する3つのコツ

3
コツ①:成功条件を最後に置く
Claudeが「止まる位置」を知る
2
コツ②:参考ファイルを先に渡す
スタイル合わせが無料になる
1
コツ③:禁止を1行書く
暴走の8割がここで止まる
  • 成功条件を最後に置く — 「diff を見せて停止」「npm test が緑なら完了」。これが型全共通の最重要フィールド。
  • 参考ファイルを1つだけ指定 — 「scripts/hello.py のCLI作法に合わせる」の一文で、既存コードとの統一感が勝手に揃う。
  • 禁止事項を1行書く — 「他のファイルは触らない」「テストを変えない」「READMEは書き換えない」など、目的以外の変更を封じる1行。
型別:主役パラメータ早見表
型目的現状参考禁止成功条件
①新規作成必須—強く推奨推奨必須
②修正必須必須任意必須必須
③調査必須——必須(書換禁止)必須(出力形式)
④レビュー必須(観点)任意任意必須(書換禁止)必須(重大度基準)
⑤デバッグ必須必須(再現/期待/観察)任意必須(テスト保護)必須
空欄(—)の箇所は省略OK。必須だけ埋めれば型は成立する。

暗記戦略:2〜3個まず体に染み込ませる

5型を一気に暗記しようとするのは逆効果。まず自分の頻度の高い2〜3型だけを マッスルメモリー まで落とします。残りは「必要になった時に辞書引き」でOK。

エンジニア系
②修正 / ⑤デバッグ / ④レビュー
PM/企画系
③調査 / ①新規作成
学習/リサーチ
③調査 / ①新規作成
運用/SRE
⑤デバッグ / ②修正

自分の職種で頻度が高い2-3型だけを CLAUDE.md または自分のメモに貼り付けて、毎日見る所に置くのが一番早いです。1週間で体に入ります。

今すぐ試す(2分)

自分の職種の主要2型を上のテンプレから選び、プロジェクトの CLAUDE.md 末尾に「## 頼み方テンプレ」として貼り付ける。次から「型①で」「型②で」と呼び出せば、Claude側も型に沿った返答をしやすくなります。

プロの依頼は、「考えたことがある」文章に見える。素人の依頼は、「いま考えている」文章に見える。型は前者への近道。 — MK

演習

Exercise 1
入門

自分の頻出タスクを型①〜⑤で分類

過去1週間に Claude Code に投げた依頼を5件思い出して、それぞれ5型のどれだったかラベリングしてください。分類できないものがあれば、なぜ分類できないか1文で言語化。

  1. 過去の依頼を5件メモに列挙
  2. 各依頼の型を①〜⑤でラベル付け
  3. 分類不能なものは「なぜ?」を1文で書く
模範解答を見る

MK の場合の例:

1. 「このエラーを直して」    → ⑤デバッグ
2. 「ユーザ削除APIを作って」 → ①新規作成
3. 「認証はどこで?」         → ③調査
4. 「このPRをレビューして」   → ④レビュー
5. 「挨拶文を '拝啓' に変えて」→ ②修正

分類不能の典型:「リファクタして」。これは型②修正と型①新規作成のハイブリッド。その場合は分解するのが王道。先に③調査で「何を」→ ②修正を複数回、のように分けると精度が上がる(Ch10で詳説)。

Exercise 2
応用

型にはめて Before/After を作る

あなたの日常で一番ベタに頼みがちなタスクを1つ選び、(a)今まで通りのベタプロンプトで Claude Code に投げる、(b)型①〜⑤のうち該当する型でテンプレを埋めて投げる、の2通りを並べて、返答の質を比較してみてください。

  1. タスク選定(例:「今日のタスクログ集計スクリプトを書いて」)
  2. ベタ版を投げる → 結果をメモ
  3. /clear して同じ内容を型に沿って投げる → 結果をメモ
  4. 返答の「一発精度」「無駄な説明の有無」「自分のレビュー時間」を比較
模範解答を見る

お題:「今日分の git log を拾って .md に整形するスクリプトを作って」

(a) ベタ版:

今日分の git log を拾って .md に整形するスクリプトを作って

→ 言語が bash か python か分からず、出力先も想像で決まる。既存スクリプトのスタイルに寄せてくれない。

(b) 型①で:

【目的】日次作業ログを chatops 投稿用に .md に整形
【要件】
- 今日の git log --since=midnight を対象
- コミットタイトル + 短いボディ を箇条書きに
- 出力: ./logs/YYYY-MM-DD.md
【制約】
- Bash 単体(python 使わない)
- 追加依存なし
【参考】scripts/daily_reports.sh のCLI作法
【成功条件】
- ./logs/ に当日の .md が出力される
- 終わったら diff を見せて停止
