Bashツールと安全な使い方

強力なシェル実行と、取り返しのつかない事故を防ぐための作法

所要時間: 45分 Phase: Basics 難易度: 初級
この章のゴール

Claude Code の Bash ツールは、あなたの端末と同じ権限でコマンドを実行する最も強力で最も危険なツールです。この章の45分で、(1)実行モデルを掴む、(2)destructive action の見分け方を覚える、(3)「確認 → 実行」を分ける型を身体に染み込ませる——ここまでを目指します。読むより手を動かす時間を長く。

バージョン注記

本章の記述は 2026年1月時点 の Claude Code (CLI / Agent SDK) の挙動に基づきます。permission mode の名称や挙動は進化するため、違和感を感じたら 公式ドキュメント で最新を確認してください。

1. Bash ツールの実行モデル

まず押さえるべき事実は1つ。Claude Code の Bash ツールは、あなたが今いるシェルと同じ権限・同じ作業ディレクトリ・同じ環境変数でコマンドを走らせます。つまり「Claude が動かしたコマンド」は「あなたが自分の手で叩いたコマンド」と等価です。これを忘れると rm -rf 一発で週末が消えます。

「Claude が動かす」= 「あなたが動かす」
サンドボックスではない。Claude が動かした結果は自分で叩いたのと全く同じ。

もう1つ重要な性質は、Bash コールごとに状態がリセットされることです。cd /tmp しても、次の Bash コールでは元の CWD に戻っています。export FOO=1 も同じく消えます。だから Claude は絶対パスを使ったり、cd /foo && bar のように1行でチェーンしたりします。

公式ドキュメント必読

ここは公式ドキュメントにも書いていますし、実務で間違えると致命的なので必ずマスターしてください。特に permission mode と設定ファイル settings.json の許可ルールは、誤設定すると rm が確認なしで走ります。Claude Code 公式: Security & Permissions →

今すぐ試す: Bash の「状態リセット」を体感

Claude に以下のようにお願いしてみてください。

/tmp に移動して "hello" と書かれた test.txt を作って。
そのあと、別のBashコールで同じファイルを cat して中身を確認して。

Claude は cd /tmp && echo hello > test.txt と cat /tmp/test.txt のように、2コール目でも絶対パスで参照しているはずです。この挙動が身体に入ると、後のデバッグで詰まりません。

2. permission mode と許可ルール

Claude Code には4つの permission mode があり、危険度別に「どこまで自動で走らせるか」を決めます。迷ったらdefaultから抜け出さないこと。

モードファイル編集Bash 実行向いてる場面
default毎回確認毎回確認通常作業・初学者
acceptEdits自動承認毎回確認大量編集で承認が面倒な時
bypassPermissions自動承認自動承認使い捨て環境・CI・自分の母艦では避ける
plan禁止(計画のみ)読取のみ許可設計・調査フェーズ

settings.json の permissions.allow / permissions.deny に個別のコマンドパターンを書くと、モードに関係なく「これは常に確認なし」「これは絶対ブロック」が作れます。deny の方が優先です。

{
  "permissions": {
    "allow": [
      "Bash(npm test:*)",
      "Bash(git status)",
      "Bash(git diff:*)"
    ],
    "deny": [
      "Bash(rm -rf:*)",
      "Bash(git push --force:*)",
      "Bash(git reset --hard:*)"
    ]
  }
}
ここがプロの境目

危ない操作は「確認 → 実行」を分ける。いきなり rm -rf ./build と頼むのではなく、先に ls ./build で中身を確認してから削除する。git reset --hard も同じで、まず git stash を検討するか git log --oneline -5 で戻り先を目視してから。
試し方： 次に削除を頼む時、プロンプトの1行目に「まず対象を ls で見せて、私が承認したら削除」と書き足す。この1行で90%の事故は防げます。

3. コマンドの危険度マトリクス

頭の中で常に動かしたい四象限。「復元可能?」×「影響範囲」の2軸で考えます。右上(復元不能 × 広範囲)は原則 plan モードで議論してから実行。

Bashコマンド 危険度マトリクス
右下(赤)は 必ず確認→実行の2ステップ。plan モードで議論してから default に戻って実行。

もう少し具体的に、「この操作が失敗したら何を失うか?」の観点でも整理しておきましょう。

destructive command 一覧と「失うもの」
もっと詳しく: なぜ git push --force が「致命傷」側なのか

ローカルで git reset --hard した時点では、git reflog に履歴が残っているので60-90日は戻せる可能性があります。しかし git push --force でリモートを上書きすると、他メンバーが pull していないコミットは消失し、CI に残ったビルド成果物・リリースタグも参照が切れます。個人リポでは --force-with-lease を、共有リポでは原則禁止(PR 経由)が鉄則です。

本章の後半でフロー化しますが、「force系は事前に git log --oneline @{u}.. で相手のコミットを確認」「PR化して同僚の目を通す」「タグを打ってから実行」の3つが現実的な防波堤です。

4. 「確認 → 実行」を分けるフロー

これが本章のコアです。ドライラン を挟む癖をつければ、事故は激減します。

安全なBash依頼のデフォルトフロー
graph LR
  A([依頼を思いつく]) --> B{"destructive?"}
  B -- No --> E[そのまま実行]
  B -- Yes --> C["対象を ls / diff / log で確認"]
  C --> D{"想定と一致?"}
  D -- Yes --> F["実行 (必要なら stash/tag で保険)"]
  D -- No --> G[依頼を修正]
  G --> C
Yes分岐では必ず「確認ステップ」を挟む。この1手間が未来の自分を救う。

実際の頼み方を Before/After で見てみます。

Before

"build フォルダ消して"

  • Claude はいきなり rm -rf build
  • 実は別プロジェクトの build だった
  • 2時間の作業が消える
After

"build の中身を ls してから削除"

  • まず ls -la build/ で一覧
  • 想定外のファイルがあれば中断
  • OKなら rm -rf build/

今すぐ試す: 3ステップお守りプロンプト

次に重めの掃除・リセットを頼む時は、この型をコピペして使ってください。

以下の順で進めてください。各ステップで私の承認を待つこと。

1. 対象を読み取り専用で見せる (ls / git status / cat など)
2. 実行する予定のコマンドを箇条書きで提示(まだ実行しない)
3. 私が「GO」と返したら実行、1コマンドずつ結果を見せる

5. git 系の定番地雷と回避策

Claude に一番多く頼むのはおそらく git 操作です。ここだけは特別に覚えましょう。

1
reset --hard の前に
git stash か git tag backup-YYYYMMDD を打つ
2
push --force の前に
git log --oneline @{u}.. で相手のコミット確認
3
rebase の前に
作業ブランチを別名でコピー(git branch backup HEAD)
4
branch -D の前に
git branch --contains HEAD でどのブランチに属するか確認
  • force系は必ず --force-with-lease — リモートが自分の知っている状態と一致する時だけ上書きする、事故防止機構。
  • git reflog を味方にする — reset や rebase で消したつもりのコミットも、reflog から git reset --hard HEAD@{3} のように戻せる(60-90日有効)。
  • 危ない作業は別ブランチ or worktree — Ch21で詳説。失敗したらブランチごと削除して終わり。
  • 本番環境の git pull は禁止 — 「main に戻しました」と軽く言う前に、CI/CDパイプライン経由かを必ず確認。
もっと詳しく: --force-with-lease の挙動

git push --force は「リモートの状態を無視して上書き」しますが、--force-with-lease は「自分が最後に fetch した時点のリモートと、現在のリモートが一致している場合のみ上書き」します。つまり他人のコミットを知らずに踏むと拒否されるため、個人リポやPR運用の自分ブランチでは常にこちらを使うのが安全です。

Claude に頼む時も「push するなら必ず --force-with-lease」と CLAUDE.md に書いておくと、毎回聞かずに済みます(Ch06参照)。

6. 事故った時の対処フロー

それでも事故は起きます。大事なのは「追加で壊さないこと」。以下の順で冷静に。

症状まずやる次にやるやってはいけない
rm でファイル消した作業を止めるエディタの未保存バッファ / タイムマシン / IDE履歴を探す上書き保存・git commit
git reset --hard した落ち着くgit reflog → git reset --hard HEAD@{N}新しい commit を重ねる
push --force 事故チームに即共有他メンバーの最新 local から強制 push で復元黙って pull し直す
DB を誤更新アプリを止める最新バックアップから復元 / PITR を発動本番で直接SQL調整
Tips: Claude に「止まって」と言う

作業中に「あれ、これまずい?」と感じた瞬間、Esc キーで即停止できます。Ctrl+C だと実行中のコマンドを止めるだけで次の行動は続くので、Esc のほうが確実。落ち着いてから状況を整理して再開してください。

Exercise 1
入門

自分の settings.json に deny ルールを足す

もっとも致命的な3コマンドを、あなたの環境で 常にブロック するよう設定しましょう。

  1. ~/.claude/settings.json を開く(なければ作る)
  2. permissions.deny に rm -rf:* / git push --force:* / git reset --hard:* の3パターンを追加
  3. Claude Code を再起動して、「プロジェクトの build フォルダを rm -rf して」と頼んでみる
  4. ブロックされる挙動を確認する
模範解答を見る

~/.claude/settings.json に以下のように追記します(既存の他キーは残す)。

{
  "permissions": {
    "deny": [
      "Bash(rm -rf:*)",
      "Bash(git push --force:*)",
      "Bash(git reset --hard:*)"
    ]
  }
}

再起動後、Claude に削除系を頼むと「このコマンドは deny ルールでブロックされています」と返ってきて実行されません。本当に必要な時だけルールを一時的に外す、という運用に切り替えると事故ゼロに近づきます。--force-with-lease は通したいので、Bash(git push --force:*) は --force パターンにマッチさせ、--force-with-lease は明示的に allow に入れておくのがおすすめ。

Exercise 2
応用

「確認→実行」プロンプトテンプレを自作する

自分の日常ワークフロー(例: リリース準備・キャッシュクリア・DB マイグレーション) を1つ選び、3ステップのお守りプロンプトをカスタマイズして、実際にClaudeに走らせてみましょう。

  1. 対象ワークフローを1つ選ぶ
  2. 「読取ステップ / 計画提示ステップ / 実行ステップ」の3段に分解
  3. 各段で Claude が使うべきコマンドを明記
  4. Claudeに走らせ、途中で「待った」を入れて修正できるか確認
模範解答を見る

例「npm のキャッシュと node_modules を綺麗にする」フローの場合:

node_modules まわりを掃除したい。以下の順で進めて。

1. まず現状を確認:
   - du -sh node_modules
   - ls -la | head
   - git status

2. 実行予定のコマンドを箇条書きで提示(まだ走らせない):
   - rm -rf node_modules
   - npm cache clean --force
   - npm install

3. 私が「GO」と返したら1コマンドずつ実行し、各コマンド後に結果を見せて。
   node_modules の削除前に、pwd が期待通りか必ず show して。

ポイントは 「計画提示で止まる」 を明示すること。これを書かないと、Claude は "賢く" 一気に進めてしまいます。また「pwd を見せて」の1行で、想定外ディレクトリで rm を打つ事故が消えます。慣れてきたら、このテンプレを CLAUDE.md や Skill(Ch20) に外出しすると、毎回コピペする必要がなくなります。
