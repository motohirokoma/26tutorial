MCP と Agents

外部システム (Slack/Notion/DBなど) とClaudeをつなぐ標準プロトコル MCP、巨大タスクを分業するサブエージェント。1人のClaudeが、組織のAIになる章。

所要時間: 1h Phase 3: Advanced 難易度: 応用

MCP — AI界の「USB-C」

MCP (Model Context Protocol) は、AIモデルと外部システムをつなぐ標準プロトコル。これまでツール毎にバラバラだった連携を、1つの仕様にまとめました。

MCP のメタファ
どのサービスも「MCPサーバー」を立てれば、Claudeから同じ作法で扱える。
なぜMCPが重要か

以前は「Slack連携するには独自スクリプト、Notionは別の独自スクリプト...」と毎回車輪の再発明でした。MCPで仕様が共通化され、誰かが書いた連携を即流用できるようになっています。

MCP の3要素

  • MCPサーバー — 外部システム側に立てるAdapter。「Slackとは何ができる」をAIに伝える
  • MCPクライアント — Claude Code 側の機能。サーバーをリストアップして、必要に応じて呼ぶ
  • JSON-RPC プロトコル — クライアント↔サーバー間の共通言語。仕様書はオープン

MCPサーバーの追加 — 3つの導入経路

MCPサーバーを Claude Code に追加する方法は3つ。手軽さで使い分けます。

経路特徴例
claude mcp add CLI1コマンドで追加。ローカル設定に書き込まれるclaude mcp add github npx @modelcontextprotocol/server-github
.mcp.json プロジェクト同梱チーム全員で同じMCP構成を共有リポジトリ直下に .mcp.json
desktop拡張UIから追加・OAuth対応サービスが楽Linear / GitHub / Notion 等

.mcp.json の例

{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/Users/me/Documents"]
    },
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": { "GITHUB_TOKEN": "ghp_..." }
    },
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres", "postgres://localhost/mydb"]
    }
  }
}
MCPサーバー追加から利用まで
graph LR
  A[claude mcp add] --> B[サーバー起動]
  B --> C[ツール一覧公開]
  C --> D[Claudeが認識]
  D --> E[ユーザー指示]
  E --> F[該当MCPツール呼出]
  F --> G[結果をClaudeへ]
1度入れたら、あとはClaudeが自動でMCPツールを使う。
権限とセキュリティ

MCPサーバーは認証情報(APIトークン等)を持つことが多いです。環境変数で渡し、.mcp.json 自体には書かない。第三者製サーバーは公式以外なら必ずソース確認。

MCP の典型的な活用例

読み取り系

情報取得

  • 「先週のSlackで議論された機能を要約して」
  • 「Linearの今週オープンなIssue一覧」
  • 「DB の users テーブルから過去30日の登録数」
書き込み系

アクション

  • 「このバグをLinearにIssueとして登録」
  • 「READMEを更新してNotionの仕様書も同期」
  • 「このPRをGitHubでドラフトとして作成」

サブエージェント — Claudeが自分の代わりに別のClaudeを呼ぶ

大きなタスクを分業するための仕組みがサブエージェント。メインClaudeが「コードレビュー担当」「テスト書き担当」など独立した別Claudeを派生させ、結果だけ持ち帰ります。

サブエージェントの構造
graph TD
  M[メインClaude]
  M -->|"PR分析"| A1["Sub: code-reviewer"]
  M -->|"テスト追加"| A2["Sub: test-writer"]
  M -->|"ドキュメント更新"| A3["Sub: doc-writer"]
  A1 -->|要点のみ| M
  A2 -->|テスト結果| M
  A3 -->|更新差分| M
各サブは独立したコンテキストを持つ。メインのコンテキストを汚さない。

なぜサブエージェントを使うか

  • コンテキスト節約 — 巨大ログをサブで処理、メインには結論だけ返す
  • 専門化 — レビュー特化のシステムプロンプトを持つサブを別個に運用
  • 並列処理 — 5つのファイルを5つのサブで同時にチェック

サブエージェント定義の例

サブエージェントは .claude/agents/<name>.md に定義します。

---
name: code-reviewer
description: PR や変更diffをレビューする専門エージェント。指摘は「Critical / Major / Minor」に分類して返す。
tools: Read, Grep, Bash(git diff:*)
---

# コードレビュー担当

## 役割
- 変更されたファイルの品質を評価
- セキュリティ・性能・保守性の3軸でチェック
- 各指摘に重要度ラベルを必ず付ける

## 出力フォーマット
### Critical
- ...
### Major
- ...
### Minor
- ...
もっと詳しく: メインClaudeとサブの呼び出し作法

メインClaudeはユーザーの依頼を見て、「これは code-reviewer 案件だ」と判断するとサブを派生させます。明示的に呼ぶには「code-reviewer エージェントを使ってこのPRを分析して」のように指示すればOK。

サブが返すのはテキスト要約のみ(ファイル本体ではない)。なので並列で10個サブを動かしてもメインClaudeのコンテキストは肥大化しません。

Exercise 1
入門

filesystem MCPサーバーを追加して、Documentsを分析させる

公式の filesystem MCPサーバーを追加し、自分の Documents フォルダの構成を Claude に分析させてみましょう。

  1. ターミナルで claude mcp add fs npx -y @modelcontextprotocol/server-filesystem ~/Documents を実行
  2. claude mcp list で追加されたか確認
  3. Claudeに「Documents の中で容量が大きいフォルダ Top 5 を教えて」と聞く
模範解答を見る

Claudeが MCP の list_directory や get_file_info ツールを順番に呼んで集計します。出力イメージ:

Documents の容量Top 5:
1. Photos        — 12.3 GB
2. Backups       —  4.1 GB
3. Projects      —  2.8 GB
4. Downloads     —  1.6 GB
5. ScreenRecords —  0.9 GB

動かない時の確認: claude mcp list で 状態 connected になっているか / 引数のパスが存在しているか / Claudeセッションを再起動したか。

Exercise 2
応用

code-reviewer サブエージェント + GitHub MCP で PR レビュー自動化

サブエージェントで「PR をレビューする専門家」を定義し、GitHub MCPと組み合わせて、PR番号だけ渡せばレビューコメントが自動生成される仕組みを作ってください。

  1. ./.claude/agents/code-reviewer.md を作成 (上の例を参考)
  2. GitHub MCP を claude mcp add で追加 (要 GITHUB_TOKEN)
  3. Claudeに「#42 の PR を code-reviewer エージェントでレビューして」と依頼
  4. サブが diff を取得 → 分析 → メインに要約 → ユーザーに提示、までの流れを確認
模範解答を見る

動作確認のポイント:

  1. サブが tools: Read, Grep, Bash(git diff:*) 以外を使おうとして拒否されないか (allow設定)
  2. GitHub MCP の get_pull_request / get_pull_request_diff が呼ばれているか
  3. レスポンスが Critical / Major / Minor の3区分で構造化されているか

うまく行けばこんな出力に:

### Critical
- src/auth.ts:42 - SQLインジェクションの可能性。プレースホルダ使用必須

### Major
- src/api.ts:101 - エラー時に500を返しているが、原因のスタックトレースをログ出力していない

### Minor
- README.md - インストール手順の Node バージョンが18のまま (現在20以上推奨)

応用: 全PRに自動でこのレビューを走らせるには、PostToolUse Hook で gh pr view 実行後にサブを呼ぶように設定可能。
