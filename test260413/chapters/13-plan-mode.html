Planモードの使い所

大きな変更は「計画への同意」から入る。ロールバック保険としての Plan

所要時間: 30分 Phase: Core 難易度: 中級
この章のゴール

Claude Code の Plan モードが「何のために存在するのか」を腹落ちさせて、自分の業務でいつ発動するかを即断できる状態になることです。大きな変更・複数ファイル横断・取り返しのつかない操作は、すべて Plan から入るのがパワーユーザーの基本姿勢。30分で判断フローまで落とし込みます。

2026年1月時点の情報

本章の記述は 2026年1月時点 の Claude Code の Plan モード仕様 (ExitPlanMode ツール含む) に基づきます。UI のキーバインドやモード切替の細部は変わる可能性があるため、動作が異なる場合は 公式ドキュメント を確認してください。

Plan モードとは — 一言で

Plan モードは、Claude が読み取り専用で情報を集め、「これから何をやるつもりか」を計画として提示するだけのモードです。あなたが計画に OK を出さない限り、ファイルは1行も書き換わらず、Bash も実行されません。

通常モードでは Claude は「考える → 書く → 実行する」を一気通貫でやります。Plan モードはその真ん中に同意ゲートを差し込む仕組み。いわばdry-runと契約書レビューを合体させたようなものです。

通常モード vs Plan モード
Plan モードの本質は「実行前に一度止まる」こと。壊れてから戻すより、壊れる前に止めるほうが圧倒的に安い。

ExitPlanMode の意味 — 「同意ゲート」の正体

Plan モード中に Claude が計画をまとめ終わると、内部で ExitPlanMode というツールを呼びます。これが実質的に「ここまでが計画です、進めていいですか?」という承認プロンプトを出す仕組み。あなたが承認するまで、実行系ツール(Write / Edit / Bash)は一切走りません。

公式ドキュメント必読

Plan モードの挙動、特にどのツールがブロックされるかは公式ドキュメントにも明記されていて、ここを勘違いすると「Plan 中なのに書き換わった」という事故につながります。実務で使う前に必ず一次情報を見てください。Claude Code 公式 →

ユーザ側の行動は2択です:

Approve

計画に同意する

  • そのまま実行フェーズへ
  • 計画通りに Edit / Write が走る
  • 計画にないことはやらない前提
Reject

差し戻す

  • 何も書き換わらない(安全に戻る)
  • 追加要望を伝える
  • Claude は計画を更新して再度 ExitPlanMode

ここで重要なのは、Reject のコストがほぼゼロである点です。通常モードで「ちょっと違った、戻して」を発動したら、ファイル差分を確認し git reset や git checkout -- を叩く必要がある。Plan モードなら単に「No」だけでいい。これが計画への同意=ロールバックの保険という発想です。

Plan に含めるべき5要素

同意ゲートが機能するのは、計画が読みやすく、判断しやすい時だけ。Claude に Plan を作らせるなら、5つの要素が揃っているかを毎回チェックしてください。

  • 変更するファイルの列挙 — どのパスに触るか、具体的に。src/** のような曖昧な範囲は不可
  • 各ファイルで何をやるか — 「追加」「削除」「関数名のリネーム」などアクション動詞で
  • 成功条件(done criteria) — テストが通る / ビルドが成功する / 特定の挙動が確認できる、など判定可能な形で
  • リスクと未解決の前提 — 「この API 仕様が想定と違ったら計画変更」など、変動要因を先に出す
  • スコープ外の明示 — 「今回はリファクタしない」「DB マイグレーションは別回」など、やらないことを固定
もっと詳しく: 「スコープ外の明示」がなぜ最重要か

Claude は親切で働き者なので、計画中に「ついでにここも直しておきますね」をやりがちです。これが積もると PR が肥大化し、レビュアがキレ、ロールバック単位が崩れます。Plan モードで「やらないこと」を明記することで、同意した範囲の中だけで動く契約が結べる。スコープ外は Ch10(改修ワークフロー)とも直結する考え方です。

具体的には ## Out of Scope セクションを Plan 末尾に必ず持たせるようプロンプトで指示するのが確実です。

実際の Plan 例(リファクタ案件)

以下は「UserService の責務を分割する」依頼に対して Claude が返した Plan の形。この粒度・書式がテンプレートです。

# Plan: UserService の責務分割

## 背景
- UserService に認証・プロフィール更新・通知送信が混在
- テストが膨らみ、認証部分だけを差し替えたい需要がある

## 変更するファイル
1. src/services/UserService.ts — 認証ロジックを抽出、プロフィール系のみ残す
2. src/services/AuthService.ts — 新規作成、login/logout/verify を移動
3. src/services/UserService.test.ts — 認証系テストを AuthService.test.ts に移動
4. src/services/AuthService.test.ts — 新規作成、移動したテストを配置
5. src/index.ts — DI コンテナに AuthService を登録

## 成功条件
- `npm test` が全部 green
- `npm run build` がエラーなく通る
- UserService から `import { authenticate }` が消えている (grep 確認)

## リスクと前提
- UserService を参照している他モジュールに破壊的変更が波及する可能性
  → 事前に grep で呼び出し元を列挙(調査フェーズで実施済み: 3ファイル)
- Notification 送信は今回触らない(次回へ)

## Out of Scope
- 通知機能のリファクタ
- DB スキーマ変更
- API レスポンス形式の変更
今すぐ試す

Claude Code で自分のリポを開いて、次のように頼んでみてください。

Plan モードで、このリポの README.md と CONTRIBUTING.md を再整理する計画を作って。
変更ファイル・各ファイルのアクション・成功条件・Out of Scope を明記して。

出てきた Plan の5要素が揃っているかチェック。欠けていたら「Out of Scope が抜けてる」と追記要求するだけで、Plan の質が一段上がります。

使う・使わないの判定フロー

毎回 Plan を挟むと遅くなるし、軽い作業で Plan を挟むと逆に冗長です。次のフローを身体で覚えるのが目標。

Plan を使う/使わない 判定フロー
graph TD
  Start([依頼が来た]) --> Q1{"触るファイル数は?"}
  Q1 -- "3 以上" --> Plan["Plan モード必須"]
  Q1 -- "1-2" --> Q2{"取り返しのつかない操作を含む?"}
  Q2 -- "Yes - delete/migrate/force" --> Plan
  Q2 -- "No" --> Q3{"本番・共有コードに影響?"}
  Q3 -- "Yes" --> Plan
  Q3 -- "No" --> Q4{"初見のコードベース?"}
  Q4 -- "Yes" --> Plan
  Q4 -- "No" --> Direct["通常モードで直行OK"]
  Plan --> Approve["計画レビュー → 同意 → 実行"]
「3ファイル以上触る作業は Plan 必須」の1行ルールを徹底すると、事故の9割は事前に潰せる。

使うべき典型シーン / 使わないべきシーン

シーンPlan 推奨度理由
3ファイル以上のリファクタ★★★ 必須影響範囲が広く、レビュー単位が必要
DB マイグレーション / スキーマ変更★★★ 必須取り返しがつかない、データ損失リスク
rm -rf / git reset --hard 系★★★ 必須実行後のロールバック不可
新機能の初期実装★★☆ 推奨設計段階で方向修正できる
初めて触るコードベース★★☆ 推奨把握のために調査フェーズが必要
typo 1箇所修正☆☆☆ 不要直接やるのが速い、計画書の方が長くなる
README の文言微調整☆☆☆ 不要軽量な変更は直行で
ログ追加・console.log 系☆☆☆ 不要戻すのが容易、試行錯誤フェーズ
もっと詳しく: 「中間領域」の判断に迷った時

「2ファイルだけど片方がコアロジック」「新機能だけど既存部分には一切触らない」のような中間ケースで迷った時は、迷った時点で Plan を挟むを原則にしてください。理由は2つ:

  • Plan を挟むコストは 2-3分、やり直しコストは 30分〜数時間。非対称すぎる。
  • 「判断を保留する」こと自体が、自分のコードベース理解度が浅いサインなので、Plan の調査フェーズで埋まる。

慣れてくると「Plan いらないな」が即判断できるようになります。それまでは保険多めで動くのが上位20%への最短ルート。

なぜ Plan が「ロールバック保険」になるのか

Plan が保険として機能する本質は「同意した瞬間、以降の変更に責任の境界線が引かれる」点です。同意前の計画書は、あなたと Claude の間の契約。この契約に沿わない変更が起きたら差し戻せば済む、という姿勢が取れます。

Plan ありなしの手戻りコスト (相対値)

小さな変更では Plan の方がむしろ重い(左2本)。しかし変更が広がるほど Plan ありのコスト曲線は緩やかで、Plan なしは指数的に膨らむのがポイント。損益分岐点は経験的に「3ファイルあたり」です。

ここがプロの境目

大きな変更は Plan から入る。計画への同意がロールバックの保険になる。壊れてから git 操作で戻すより、壊れる前に Reject する方が桁違いに安い。責任の境界線が明示されるため、Claude に「計画外の提案」を抑制する効果もある。
試し方: 今日から「3ファイル以上触る作業は Plan 必須」を自分ルール化して、Claude Code を開いたら最初にプランモードで計画作ってと打つクセをつける。

よくある失敗と回避策

Plan を読まずに同意
変更ファイル数だけは必ず目視
同意後に「ついで」追加
範囲外は新セッションで
計画が抽象的すぎる
パス名と行動詞を要求
毎回 Plan で冗長
typoや1行修正は直行
もっと詳しく: 「Plan 後に追加要求」が一番やりがちな罠

Plan に同意したあと、実行中に「そういえばここも直して」と追加すると、Plan の同意範囲が崩れて保険が無効化します。追加したい時は2択:

  • (a) 一度完了させて commit、そのあと次の Plan を作る(推奨)
  • (b) 実行を止めて Plan を更新し、再度 ExitPlanMode で同意を取り直す

「口頭で途中追加」だけは避けること。これをやると Ch22(失敗パターン)の典型 #7 に落ちます。

Exercise 1
入門

Plan モードを発動して5要素をチェック

自分のリポジトリ(どれでも可)を開き、以下を Claude Code に頼んでみてください。

  1. Plan モードで「README.md と package.json のバージョン表記を揃える」計画を作らせる
  2. 返ってきた Plan に 5要素(変更ファイル/アクション/成功条件/リスク/Out of Scope)が揃っているかチェック
  3. 欠けている項目があれば補うよう追加指示
  4. 計画を承認して実行させる
模範解答を見る

典型的な初回 Plan は「Out of Scope」と「リスク」が抜けていることが多いです。追加指示は次のテンプレで:

Plan に以下を追加してください:
- Out of Scope セクション (CHANGELOG は今回触らない、など)
- リスクセクション (バージョン表記のフォーマットが2ファイルで違うなら事前に揃える必要があるか)

こうして Plan の質を一段上げた状態で同意する習慣が付きます。実務では「Out of Scope が書かれていない Plan は差し戻し」をルール化しているチームもあります。

Exercise 2
応用

「3ファイルルール」を自分の業務に組み込む

自分の最近の Claude Code セッションを振り返って、以下を書き出してください。

  1. 直近1週間で「3ファイル以上触った作業」を3つ列挙
  2. その中で Plan モードを使わなかったものにチェック
  3. 使わなかった結果「手戻り・やり直し」が発生した作業を特定
  4. 次回同じ作業が来たら Plan をどう書くか、テンプレ化してプロジェクトの CLAUDE.md に追記
模範解答を見る

CLAUDE.md に追記する形の例:

# Plan モード運用ルール (本プロジェクト)

## Plan 必須の作業
- 3ファイル以上を同時に触る
- DB マイグレーション、スキーマ変更
- 認証・権限系コードの変更
- 外部 API の契約に関わる変更

## Plan に必ず含める
1. 変更ファイル (パス列挙)
2. 各ファイルのアクション (追加/削除/リネーム等)
3. 成功条件 (テスト・ビルド・grep 確認)
4. リスクと前提
5. Out of Scope

CLAUDE.md に書いておくと、Claude が Plan を作る際に自動でこのルールに沿ってくれるため、毎回口頭で指示する必要がなくなります。Ch06(CLAUDE.md)との合わせ技がパワーユーザーの基本装備です。
