対話の原則：上手な頼み方

3点セット(具体性 / 制約 / 成功条件)で、Claudeの返答が別物になる

所要時間: 45分 Phase: Basics 難易度: 入門
この章のゴール

Claude Codeへの頼み方を「なんとなく」から「型」に変える。同じタスクでも、頼み方次第で返答の質は2倍〜10倍変わる。本章は5セット以上のBefore/Afterで「型」を身体にしみこませ、最後に自分のリアル業務で試すまでを45分で体験する。読んで終わりではなく、いまのターミナルでコピペして試すを合言葉に進めます。

バージョン注記

本章の記述は 2026年1月時点 の Claude Code (CLI) を前提としています。プロンプトのコツは長期的に安定していますが、スラッシュコマンドやモード名は変わることがあるため、動かない時は 公式ドキュメント で確認してください。

なぜ「頼み方」だけで結果が変わるのか

Claudeは、あなたの頭の中を読めません。プロンプトに書いていない情報はすべて空白で、その空白をClaudeはもっとも一般的な解釈で埋めます。空白が多いほど、あなたの意図から逸れた回答が返る。逆に、空白を埋める情報=コンテキストを明示すれば、Claudeは「あなたの意図」という特定の一点を狙って答えます。

下図は同じ「ログイン機能を作って」というお題が、情報量によってどれだけ結果のブレが変わるかを示したものです。

頼み方の情報量と、回答の「当たり範囲」の関係
情報を絞るほど、Claudeの「当たり範囲」が縮む。

ここで言いたいのはシンプルです：プロンプトとは、Claudeの探索範囲を絞るためのフィルタ。フィルタが緩いと望まない結果も通るし、キツすぎると動けない。ちょうど良い絞りが3点セットです。

3点セット — これだけで8割いける

本章で最初に覚えるべき型がこれ。毎回3つだけ書けばいい。

  • 具体性 (Concreteness) — 何を作る/直す/調べるのか、対象を名詞で指す。「ログイン画面」ではなく「src/pages/login.tsx のEmail入力フォーム」。
  • 制約 (Constraints) — 使う技術、避けるべきこと、性能・互換性・トーンなどの「枠」。「React 18で」「新しいライブラリは入れないで」。
  • 成功条件 (Done Criteria) — どうなったら終わりか。「npm testがグリーン」「git diffが10行以内」「ユーザーがEnterで送信できる」。
3点セットのベン図 — 交点に「意図通りの成果」がある
どれか1つ抜けるだけで、結果はずれ始める。
要素これを書かないと起きること書くためのヒント
具体性対象ファイルを推測で開き、関係ないところまで触るファイルパス/関数名/スクリーンショットを貼る
制約勝手に新ライブラリ追加、巨大リファクタ化「使ってよい技術」「触らない範囲」を明記
成功条件いつまでも編集し続ける／途中で止めるテスト名、コマンド、見た目の状態で定義
ここがプロの境目

成功条件 (done criteria) を渡すと、Claudeは「どこで止まればいいか」分かる。これを書かない初心者が一番多いが、ここの効果が一番大きい。「テストが通ったら終わり」「git diff表示して手を止めて」と一言添えるだけで、やりすぎも、やり残しも消える。
試し方： 次の頼み事の末尾に「npm testがすべてグリーンになったら報告して終わり」を付けてみる。

Before/After を5セット、体に刷り込む

ここが本章の中核。悪い頼み方と良い頼み方を並べて読むのが最速の学習です。全部「そのままコピペして試せる」形で書いています。自分のプロジェクトに置き換えて1つは必ず実行してください。

Before

セット1: バグ修正

ログインが動かない。直して。
  • 対象ファイルが分からない
  • 「動かない」の意味が不明
  • どこまで直すか分からない
After

セット1: バグ修正

src/pages/login.tsx のEmailフォーム
で、空欄送信時にエラーが出ない。
期待: 「Emailを入力してください」を
表示。React 18, 既存の useForm 使用。
成功条件: npm test の login.test.tsx
がパスしたら終わり。
Before

セット2: 新機能

ダークモード欲しい。
  • どの画面に？
  • CSS変数？Tailwind？
  • 切替方法は？
After

セット2: 新機能

全ページにダークモード切替を追加。
src/styles/theme.css のCSS変数を使い
 --bg, --text を切替。Header右上に
☀/🌙 ボタン。localStorage保存。
制約: 新ライブラリなし、Tailwind不可。
完了: リロード後も状態保持されること
を確認して終わり。
Before

セット3: 調査

このコードベース見てまとめて。
  • 規模不明で全読みされる
  • 何の観点でまとめるか不明
  • 長文が返ってきて使えない
After

セット3: 調査

src/ 配下の主要ディレクトリ構造を
Grepでエントリポイント (main/index)
を探してから把握。目的: 新人引継ぎ
資料。出力: Markdown 箇条書き30行
以内。制約: src/__tests__ は除外。
完了: Markdownを output.md に保存
して終わり。
Before

セット4: リファクタ

user.service.ts が長いので短くして。
  • 挙動保証なし
  • どこまで分けていいか不明
  • 関連ファイル全部書き換えられる可能性
After

セット4: リファクタ

src/services/user.service.ts (320行)
の責務を3つに分割: auth系/profile系
/preference系。まず Plan モードで
分割案を提示して承認を取ってから実装。
制約: public APIは変えない。テストは
既存のまま通る必要あり。完了: npm test
グリーン + git diff レビュー可能な
粒度で stage して終わり。
Before

セット5: レビュー

このPRレビューして。
  • 観点が広すぎて薄い指摘しか来ない
  • 優先順位が分からない
After

セット5: レビュー

gh pr diff 42 を取得してレビュー。
観点: (1) セキュリティ(認証・XSS)
(2) エラーハンドリング漏れ
(3) 命名の一貫性。
各観点で「Must/Should/Nice」に分類。
成功条件: Must 指摘ゼロなら approve、
あれば理由と修正案を添えて返す。
Before

セット6: 実験

TypeScript strict 化して。
  • 既存コードが大量にエラー化する
  • どこまで直すか不明で延々続く
After

セット6: 実験

tsconfig.json を strict: true にした
時の影響を試算してほしい。実装は
しない。手順: (1) 有効化 (2) tsc で
エラー件数カウント (3) 多い順にTop5
ファイル報告。完了: レポートを返し
たら tsconfig.json を元に戻して終わり。

どうでしょう。Afterは長く見えても読解コストはBeforeの1.5倍くらい。その代わり、返答の的中率は一気に上がり、追加のやり取り3-5往復が省ける。結局Afterの方が速いのです。

もっと詳しく: なぜAfterが速く感じないのに、結果は速いのか

人は「書く時間」に敏感ですが、「やり取りの往復回数」には鈍感です。Beforeは3秒で書けても、そこから「どのファイル？」「何が動かないの？」と3往復発生し、合計5分かかります。Afterは書くのに1分かかっても、返答は1発で着地する。目安はやり取り往復2回以下。3回以上になるなら3点セットのどれかが抜けています。

「なぜ」を1行添えると、Claudeは最適解を選ぶ

3点セットは「何を」「どう」「どこまで」の問いに答えますが、もう一つ強力な情報がある。「なぜ」です。目的が分かるとClaudeは手段を賢く選びます。

「なぜ」があると、ツール選択が自動で変わる
graph LR
  A["頼み事: ログ整形"] --> B{"なぜ?"}
  B -->|"顧客デモで5分後に見せる"| C["最小変更 / 既存ライブラリ流用"]
  B -->|"本番運用で毎日動かす"| D["エラーハンドリング強化 / テスト追加"]
  B -->|"自分の学習用に試したい"| E["シンプル実装 / コメント多め"]
同じ依頼でも、目的によってClaudeが選ぶ「正解」は変わる。

例えば「このAPIを高速化して」。なぜ高速化したいかで解が変わる：

「なぜ」Claudeが選びがちな解
レスポンスが2秒→500ms欲しい(UX)キャッシュ追加、N+1解消
コスト削減したい(月額)クエリ回数削減、不要呼び出し排除
ピーク時に落ちる(安定性)タイムアウト、リトライ、レート制限
学習目的(勉強)各改善案を比較レポート
Tips

書き方テンプレ: 「(目的)のため、(対象)を(制約)で(成功条件)まで」。1文でも強力。例:「本番の月額コスト削減のため、/api/search のN+1を解消し、既存テストが通ったら終わり」。

文脈の与え方 — パスを貼る vs 中身を貼る

文脈の渡し方には3段階あります。状況に応じて使い分ける。

1
パスだけ
Claudeに Read させる
2
抜粋貼付
関連部分だけペースト
3
検索指示
Grep/Globから探させる

判断基準は「ファイルが50行以内か、自分でもどこか分からないか」。小さくて明確ならパス指定、大きくて自分も分からないならGrep/GlobからClaudeに探させる。

ファイル共有の3段階フロー
graph TD
  Q["Claudeに頼みたい"] --> S1{"対象ファイルは?"}
  S1 -->|"明確"| A["パスだけ指定"]
  S1 -->|"不明"| S2{"規模は?"}
  S2 -->|"小さい"| B["関連部分を抜粋して貼る"]
  S2 -->|"大きい / 分からない"| C["Grep/Glob で探させる"]
  A --> OK["Claude が Read"]
  B --> OK
  C --> OK

「今すぐ試す」用のサンプルプロンプト：

# パスだけ (小さく明確)
src/utils/date.ts を読んで、
formatDate の戻り値を "YYYY-MM-DD" に統一して。

# 抜粋貼付 (関連部分だけ見せたい)
以下のエラーログを元に、原因仮説を3つ。
```
TypeError: cannot read 'map' of undefined
  at line 42, components/List.tsx
```

# 検索指示 (どこにあるか自分も分からない)
プロジェクト内で useAuth を呼んでいる箇所を
Grep で列挙し、引数パターンを3種類以下にまとめて報告して。
公式ドキュメント必読

ファイル操作とコンテキスト管理は、Claude Codeを効率よく使う上で最重要。全ファイル Read はコスト的に致命的なので、Grep/Glob ワークフローは必ずマスターしてください。Claude Code 公式 →

反復の型 — 1発で終わらせようとしない

「1発で完璧」を狙うと、プロンプトが膨らみ読むのも書くのも遅くなる。パワーユーザーは2-3ラウンドの対話で確度を上げる設計をします。

  • ラウンド1: 把握 — 「まず現状の構造を3点にまとめて。実装はしない」
  • ラウンド2: 計画 — 「この変更プランでいい？ファイル単位で教えて」(Planモードが有効)
  • ラウンド3: 実装 — 「ではその計画で進めて。テストグリーンで止まって」
1発完璧型 vs 3段階型 — 総所要時間の比較(体感値)

途中で路線を変えたい時の言い方も重要。「さっきの話は忘れて」より、「X についてだけ方針変更。Y はそのまま」と差分で指示するとClaudeのコンテキストが混乱しません。

もっと詳しく: やり直しが必要になった時の3つの選択肢
  1. 差分修正: 「AについてBに変更」。最も軽量、文脈保持。
  2. /compact: 長くなった会話を要約圧縮。必要な情報だけ残す。Ch14参照。
  3. /clear + 再投入: セッションを綺麗にリセット。路線が大幅に変わる時に。

「今のトピックと無関係になった」と感じたら迷わず /clear。古いコンテキストが残ると別の問題の原因になります。

ここまでで基礎パターンは揃いました。書いて→確かめて→差分で修正する。この3拍子がClaude Codeとの対話のリズムです。

よくある失敗 5つと回避

症状原因回避
思ったファイルと違うところを触られた具体性不足、ファイルパス未指定パス or Grep指示を添える
ライブラリが勝手に追加された制約未指定「新規依存追加禁止」を明記
延々と改修が続く成功条件なしテスト名 or コマンドで定義
テストが都合よく書き換えられた「テスト変更禁止」を言っていない「テストは触らず実装側で合わせて」と添える
長い会話で話が噛み合わなくなるコンテキスト汚染/compact か /clear
もっと詳しく: テストを緩められる問題 (Ch11の先取り)

「テスト通して」と頼むと、Claudeは「テストを通すためにテストを書き換える」という解を選ぶことがあります。実装にバグがあっても assertion を緩めれば通る——これは技術的には正解ですが、意味的にはアウト。

回避策はシンプル: 「テストを通すためにテストを変えない。実装側を直して」と毎回添える。Ch11 でTDDと合わせて深掘りします。

# 3点セット テンプレ
目的: {なぜ}
対象: {ファイルパス or 検索指示}

Exercise 1
入門

Before/After を自作する

以下の「悪い依頼」を、3点セット(具体性/制約/成功条件)入りの「良い依頼」に書き直してください。自分のプロジェクト前提で考えてOK。

  1. 「README 書いて」
  2. 「テスト足して」
  3. 「このバグ直して」
模範解答を見る

例(自分のWebアプリ前提):

1. README 書いて
→ 目的: 新しい協力者が5分でセットアップできる導線。
   対象: プロジェクトルートの README.md (現在空)。
   制約: 日本語、100行以内、絵文字なし。
   成功条件: セットアップ手順/使い方/コントリビュート
   の3セクションが埋まったら終わり。

2. テスト足して
→ 目的: ログイン周りのリグレッション検知。
   対象: src/pages/login.tsx の Email バリデーション。
   制約: Jest + React Testing Library、新規依存不可。
   成功条件: 3ケース(正常/空欄/不正形式)で
   npm test がグリーンなら終わり。

3. このバグ直して
→ 目的: ユーザー報告「送信ボタンが押せない」の解消。
   対象: src/components/SubmitButton.tsx, 再現手順は
   空入力+日本語IME変換中に Enter 押下。
   制約: Chrome/Safari両対応、他コンポーネントは触らない。
   成功条件: 手動再現で解消 + 既存テスト通過。

ポイント: 目的を先に書く、ファイルパスを具体的に、どうなったら終わりかを明記。3つが揃うだけでClaudeの返答は別物になります。

Exercise 2
応用

成功条件だけで止まる依頼を試す

「プロの境目」を体感する演習。以下の2通りで同じタスクを Claude Code に投げ、結果を比較してください。

  1. 自分のリポジトリで小さな機能追加 or バグ修正を選ぶ
  2. パターンA: 成功条件を書かずに依頼 → 何が起きるか観察
  3. パターンB: 同じ依頼に「npm testがグリーンになったら報告して終わり」を末尾に追加 → 比較
  4. 差分(Claudeの停止タイミング/出力量/追加質問の数)を3行でメモ
模範解答を見る

典型的な差分観察例:

  • 停止タイミング — A: 実装後もリファクタ提案やREADME追記を続ける / B: テスト通過で即停止
  • 出力量 — A: 冗長、関係ないファイルにまで手を伸ばす / B: 必要最小の変更で終わる
  • 追加質問 — A: 「これも直しますか？」が2-3回発生 / B: ゼロ、あってもエラー時のみ

実コードでの例:

# パターンB (成功条件あり)
src/utils/validate.ts の isEmail に TLD 制約を追加。
制約: 既存の他ユーティリティ関数は触らない。
成功条件: tests/validate.test.ts の全ケースがグリーンに
なったら git diff を表示して終わり。次の提案は不要。

「次の提案は不要」「 git diff を表示して終わり」のような停止シグナルを末尾に置くのがコツ。Claudeは「終わっていい」と明示されないと、善意で仕事を増やし続けます。
