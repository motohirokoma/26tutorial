コンテキスト管理

/compact と /clear を使い分け、長セッションを潰さないパワーユーザーの基本動作

所要時間: 45分 Phase: Core 難易度: 中級
この章のゴール

Claude Code のコンテキストウィンドウという有限資源をどう扱うかを、手を動かしながら体で覚えます。/compact で要約圧縮して続ける、/clear で思い切ってリセットする、CLAUDE.md や memory で「毎回必要な文脈」は常駐させる——この3つの使い分けができるだけで、長時間セッションの品質が一気に安定します。

2026年1月時点の情報

本章の記述は 2026年1月時点 の Claude Code の挙動に基づきます。/compact のアルゴリズムや context window の上限は機種・プランで異なり、今後アップデートで変わる可能性があります。スラッシュコマンドが期待どおり動かない時は 公式ドキュメント と claude --version をまず確認してください。

コンテキストウィンドウという「作業机」

Claude にはコンテキストウィンドウという有限の「作業机」があります。あなたの質問、Claude の返答、Read で読んだファイル、Bash の出力——すべてが机の上に積まれ続け、机からはみ出した瞬間にセッションは破綻します。

モデルごとの目安は 200K〜1M トークン。1トークン ≒ 日本語0.5〜1文字・英語0.75単語の粗い感覚で、200K トークン = 日本語で原稿用紙500枚前後。分厚い本1冊を机の上に開きっぱなしにできるサイズ、と思うと体感に近いです。

コンテキストウィンドウの中身(イメージ)
机の多くは「過去の会話」と「ツールが吐いた大量の出力」で埋まる。余白が尽きる前に圧縮するのが仕事。

実務上、一番太るのはツール出力です。Read で大きなファイルを丸ごと読んだ、Bash で npm test のログを取った、WebFetch で HTML を丸呑みした——こうした出力が積み重なり、読者が気付かないうちに「残り10%」になるのがあるあるのパターン。

公式ドキュメント必読

コンテキストウィンドウの正確な上限・/compact と /clear の正式な挙動・memory 機能のスコープは、モデルとプランで変わる上、誤解したまま使うと「圧縮したつもりが全部消えた」事故につながります。ここは公式ドキュメントに書いてあり、実務で間違えると致命的なので必ずマスターしてください。Claude Code 公式 →

使用率は「こう推移する」

新品の机は当然スカスカ。そこから会話とツール実行を重ねるたびに埋まっていき、どこかで天井に到達します。下のグラフは、実際の長時間セッションでありがちな推移パターン。70%が第一警報、90%が赤——この体内時計を持つのが今日のゴールです。

コンテキスト使用率の推移 (長セッション模擬・45分)
70% で /compact を実行すると、約30-40% まで縮んで再び伸びていく。無策だと100% で頭打ち → 応答劣化。

赤の実線(/compact を挟んだ方)を見てください。70%時点で圧縮を掛けた瞬間に 40%台まで下がって、続きが普通にこなせている。灰色点線(無策)は 100% に張り付いた後、返答の品質が崩れはじめるゾーンに入ります。

「いま何%か」を体感する3つのサイン

  • 返答が妙に短くなる / 省略が増える — 机が埋まって要約モードに入ったサイン。70%超え濃厚。
  • さっき話したはずの内容を忘れる — 古い履歴から押し出されている。80%以上を疑う。
  • ツール呼び出しが途中で止まる / タイムアウト — 残量ギリギリ。90%以上。即 /compact か /clear。

/compact と /clear — どっちを使う?

最重要の2つのスラッシュコマンド。混同されがちですが、目的がまったく違います。

/compact

要約して「続ける」

  • 過去の会話を Claude が要約して圧縮
  • 作業の流れ・意思決定・重要な結論は残す
  • タスクの途中で机を広げたい時に使う
  • 30-60% まで縮むのが典型
  • 満杯(90%+)になってからでは遅い
/clear

全部捨てて「仕切り直す」

  • 会話履歴を完全リセット(CLAUDE.mdは残る)
  • 過去の文脈は一切残らない
  • タスクの切れ目・別件に移る時に使う
  • 使用率は即5%前後に戻る
  • 誤爆するとやり直し不可能
ここがプロの境目

/compact は満杯前、/clear は切れ目で。タイミングを読むのがパワーユーザーの条件です。/compact は満杯 (>90%) になってからでは「圧縮前の情報を参照できない」事故が起きやすく、/clear はタスクを跨がずに使うと文脈ごと消えて二度手間になる。どちらも早すぎず遅すぎず。
試し方： 長時間セッションで context 70% になったら迷わず /compact。タスクが1つ終わって別件に入る前に /clear。これを1週間やるだけで事故が激減します。

判断フロー

/compact と /clear の分岐
graph TD

今すぐ試す(2分)

手元の Claude Code で、わざと大きなファイルを2-3個 Read させてから以下をそれぞれ実行してみてください。

# 1. まずは要約圧縮
/compact

# 2. もしくは完全リセット
/clear

/compact の後、Claude に「さっきまで何をしていた?」と聞くと、要約された文脈で答えが返ってきます。一方 /clear の後は「何も覚えていません」と返ってくるはず。この違いを体感するのが本章最大の投資です。

memory と CLAUDE.md — セッションを跨ぐ記憶

/clear で消えない記憶が2つあります。CLAUDE.md(プロジェクト/ユーザースコープの定常メモリ) と、memory です。

記憶場所スコープ/compact で/clear で使うもの
会話履歴このセッション要約される全消去いま話してる内容すべて
ツール出力このセッション要約される全消去Read/Bash/WebFetch の結果
CLAUDE.md (project)このリポジトリ残る残るプロジェクトのルール・構成
~/.claude/CLAUDE.md (user)全プロジェクト残る残る自分の好み・デフォルト作法
memoryユーザー横断 (*)残る残る長期的な嗜好・学習事項

(*) memory 機能の具体的スコープ・保存場所は 2026年1月時点でバージョン依存。公式ドキュメントで最新を確認すること。

使い分けの原則

  • 毎回必要な情報(プロジェクト固有のコマンド・ビルド手順など) → CLAUDE.md に書く。会話で毎回説明しない。
  • 自分のクセ・好み(TypeScript 派・日本語で返事して等) → ~/.claude/CLAUDE.md。どのプロジェクトでも効く。
  • この会話だけの一時的な文脈 → 会話でそのまま。/compact で要約、/clear で捨てる。
もっと詳しく: memory と CLAUDE.md はどう違う?

CLAUDE.md はあなたが書くマークダウンで、Claude は毎セッション読み直します。明示的・制御可能・diff で見える——「教える記憶」です。

対して memory はClaude が自動で覚えようとする機能で、ユーザーの好みや過去の判断を学習して保持する仕組み(2026年1月時点で段階的に展開中)。暗黙的・制御が難しい——「育つ記憶」に近い。

現実的な使い分け: ルール・仕様・コマンドは CLAUDE.md、嗜好・プロファイルは memory に任せる。memory の内容がズレてきたら、公式の手順で内容確認/削除できます(バージョンにより異なる)。公式 →

長セッションを潰さない5つのコツ

ここまでの /compact と /clear を「反応的」に使うだけでなく、そもそも机を太らせない予防策のほうが効きが大きい。実務で効く5つを紹介します。

1
Grep→Readの順
いきなり全文 Read しない。場所を絞ってから部分 Read。
2
offset/limit 活用
大きなファイルは必要な範囲だけ切り出す。
3
Subagent に逃がす
重い調査は子エージェントへ。要約だけ親に返る。
4
節目で結果を保存
重要な結論は .md や commit に外出ししてから /compact。
5
タスクを切る
45-60分を目安にいったん /clear して仕切り直す。

使い方マトリクス — 状況別ベスト

状況使用率推奨アクション理由
長い実装の途中、あと1-2ステップで完了70-85%/compact流れを保って最後まで走る
調査が終わって、実装フェーズに入る50-80%結論保存 → /clear頭を切り替えて品質を保つ
Read で 3000行級のコードを読んだ直後急に60%+Subagent に逃がすべきだった次回は最初から分業
90%超え + 重要結論未保存90%+まず結果を外出し → /compact圧縮で重要情報が削られるリスク回避
別プロジェクトに切り替え何%でも/clear混在の方が事故のもと
もっと詳しく: /compact に「何を残してほしいか」を指示できる

/compact は単独で打つと自動で要約されますが、事前に「次に必要な情報を1段落でまとめて」とお願いしてから打つと、要約の質がぐっと上がります。

# 圧縮直前プロンプトの例
いったん /compact で圧縮する前に、これまでの作業を次の形式で要約してください:
- 確定した仕様
- 残タスク
- 関わっているファイル
- 次に取るべき最初のアクション

そのうえで /compact を実行します。

こうするとメインに「要約済み要約」が残り、圧縮後も手戻りがほぼゼロになります。

もっと詳しく: /clear のあと何を復元すべきか

/clear 直後は完全に記憶ゼロなので、続きで同じタスクをやるつもりなら再起動用プロンプトを1つ用意しておくと神業的に効率化します。

# /clear 直後の再起動テンプレ
前回の続きです。今日のゴールは「XXXX」。参照すべきファイルは
 - path/to/A.ts
 - path/to/B.test.ts
現状はYYY までできていて、次のタスクはZZZです。
まず path/to/A.ts を Read して現状を確認してください。

毎回同じ形式で書けるように、この骨組みを自分の CLAUDE.md や Skill に入れておくとさらに速い。

今すぐ試す(3分)

自分の普段のプロジェクトで、以下の流れを1サイクル回してください。

  1. 大きめのファイル2-3個を Read させて、体感で机を太らせる
  2. Claude に「現状と残タスクを1段落で要約して」と依頼
  3. /compact を実行
  4. 圧縮後に「続きから再開してください」と伝える

うまく行くと、コンテキストは縮んだのに作業は途切れない感覚が得られます。これが本章の到達点です。

よくあるアンチパターン3選

NG 1

/compact を打ちすぎる

  • 30% 時点で連打する人がいる
  • 圧縮のたびに細部が削れる
  • 結果: 話が浅くなる
NG 2

満杯で初めて慌てる

  • 99% でツールが動かなくなって気付く
  • この時点では要約も質が落ちる
  • 結果: セッション破綻
NG 3

別件でも /clear しない

  • プロジェクトAの文脈にBが混ざる
  • Claude が違うリポの話を持ち出す
  • 結果: 誤った修正を提案される

この3つを避けるだけで、長セッションの9割は安定します。残りの1割は次章(Web検索・取得)で触れる「外部から情報を取ってくる量をコントロールする」系のテクニックでカバーします。

Exercises — 手を動かして定着させる

Exercise 1
入門

「要約保存 → /compact → 続行」を1サイクル回す

自分の普段のリポジトリで、次の流れを実際にやってみましょう。

  1. Claude Code を開き、大きめのファイル(1000行以上)を2つ Read させる
  2. 「ここまでの作業を『確定仕様 / 残タスク / 次のアクション』の3項目で要約して」と依頼
  3. 返ってきた要約を確認したら /compact を実行
  4. 「続きから再開してください」と伝えて、作業が途切れずに進むか確認
模範解答を見る

期待される挙動: /compact 後も、先ほど依頼した3項目の要約は「確定仕様」「残タスク」「次のアクション」として引き継がれる。大きなファイルの全文は消えているが、Claude は「どのファイルが関係しているか」を覚えているので、必要なら再 Read すれば続けられる。

# 圧縮前プロンプト例
ここまでの作業を次の形式で要約してください:
【確定仕様】このセッションで決まったこと
【残タスク】まだ手を付けていない作業
【次のアクション】再開時に最初にやるべきこと

要約が返ったら /compact を実行します。

ポイント: 圧縮の直前に「何を残したいか」を会話で明示すると、要約の質が上がる。/compact を無策で打つより、この1手間が結果を劇的に変えます。

Exercise 2
応用

CLAUDE.md に「圧縮運用ルール」を書き込む

自プロジェクトの CLAUDE.md(または ~/.claude/CLAUDE.md)に、本章の学びを1ブロック分追記してください。次回以降、Claude がこのルールを踏まえて動いてくれます。

  1. 70% を超えた時の挙動ルール
  2. タスクの切れ目で /clear を提案させる条件
  3. 圧縮前の要約フォーマット

書いたあと、適当な長セッションを動かしてみて、Claude 自身が「そろそろ要約しましょうか?」と提案してくるか観察しましょう。

模範解答を見る

CLAUDE.md に追加するブロック例:

## コンテキスト管理ルール

- context 使用率が 70% を超えたと判断したら、自発的に
  「そろそろ /compact を推奨します」と提案してください。
- タスクが完結し別件に移る時は、「/clear で仕切り直しますか?」と確認してください。
- /compact 直前には必ず次の3項目を返してから圧縮に進みます:
  【確定仕様】 / 【残タスク】 / 【次のアクション】
- 大きなファイルの全文 Read は避け、Grep→部分Read の順を基本にします。

効果: これを入れておくと、Claude 側から能動的に「圧縮しますか?」を提案してきます。自分が機械になって毎回判定する必要がなくなり、体内時計が CLAUDE.md に移った状態。これぞ Top 20% の作法です。

注意: ルールを増やしすぎると CLAUDE.md 自体が肥大化してコンテキストを食います。本章ルールは10-15行以内に収めるのが目安。Ch06 の「CLAUDE.md は最小限に」の原則を忘れずに。
