ファイル操作とコード編集

Claudeが裏で使っている「ツール」の種類と、それぞれの得意分野を理解しましょう。道具を知れば、指示がもっと的確になります。

所要時間: 1.5h Phase: Core 難易度: 入門〜中級

Claudeが使う6つのツール

Claude Codeは内部で 6つの基本ツール を持っています。それぞれの「得意/苦手」を知ると、結果が安定します。

R

Read

ファイルを読む。テキスト・PDF・画像対応

W

Write

新規ファイルを作る/全文置換する

E

Edit

既存ファイルの一部を差分編集する

G

Glob

パターンでファイル名を検索

F

Grep

ファイルの中身を検索(ripgrep)

B

Bash

シェルコマンドを実行

タスク → ツール選択フロー
graph TD
  A["タスク発生"] --> Q1{"何をしたい?"}
  Q1 -- "中身を読む" --> Read
  Q1 -- "ファイル名で探す" --> Glob
  Q1 -- "中身を検索" --> Grep
  Q1 -- "新規作成" --> Write
  Q1 -- "一部を変更" --> Edit
  Q1 -- "コマンド実行" --> Bash
Claudeは内部でこの判断を自動でやる。あなたは結果(差分・出力)を見て確認する。

ファイルを読む (Read)

Read ツールはファイルの中身を 行番号付き で読みます。Edit と組み合わせて「N行目を修正」のような精密操作ができる仕組みです。

# Claudeから見たRead結果(イメージ)
   1  function add(a, b) {
   2    return a + b;
   3  }
   4
   5  function sub(a, b) {
   6    return a - b;
   7  }
  • 得意: テキスト・JSON・YAML・PDF・画像(スクショ)
  • 苦手: 巨大ファイル(自動でページング、明示的にoffset/limit指定が無難)
  • Tip: @パス をプロンプトに書くと、Claudeが自動でReadする

ファイルを探す (Glob と Grep)

Glob

ファイル名で探す

  • パターン: **/*.ts src/**/test_*.py
  • 「テストファイルだけ全部」など
  • 結果は更新日時の新しい順
Grep

中身で探す

  • 正規表現対応(ripgrep)
  • 「`useEffect`を使ってる箇所全部」
  • 件数だけ・ファイル名だけ・本文付きを選べる
Glob 例
> src/ 配下の .test.ts を全部探して

Grep 例
> "TODO" を含む箇所をファイル名と行番号で出して

ファイルを書き換える (Edit)

Editは 古い文字列 → 新しい文字列 の精密置換です。「N行目を変えて」ではなく、「この文字列をこの文字列に」という形でClaudeが動きます。

なぜ差分編集なのか

ファイル全体を書き直すと、意図しない部分まで変わるリスクがある。Edit方式なら、変更箇所だけが正確に指定され、レビューもしやすい。

Editの動き(イメージ)
old: "function add(a, b) {\n  return a + b;\n}"
new: "function add(a: number, b: number): number {\n  return a + b;\n}"

新しいファイルを作る (Write)

Writeは新規作成または完全上書き。既存ファイルへの部分修正にWriteを使うのは避ける(Editを使う)のが鉄則です。

Writeの落とし穴

うっかり既存の重要ファイルにWriteすると、丸ごと置き換わってバージョン管理外の変更が消える。必ず git で差分を見てから commit する習慣を。

コード生成のコツ

良い「実装依頼」の構造
この4要素を最低限指定すると、Claudeの一発実装の精度が一段上がる。

リファクタリングパターン

典型的な依頼パターンと、Claudeが何を気にしてくれるかをまとめます。

パターン頼み方の例気をつける点
関数分割「foo を 3 つに分割。テストの挙動は変えずに」テスト先に走らせる
命名統一「getX を fetchX にリネーム。呼び出し元も全部」大量diff、PR分けたい
型導入「JSをTSに。anyを使わず最善型を推論」tsconfig 確認
抽象化「重複した3関数を共通ヘルパに」過剰抽象化に注意
削除「使われてないexportを全部消して」動的import要注意

複数ファイルにまたがる変更

Claudeは 計画 → 各ファイル編集 → 検証 を内部で組み立てます。あなたは「全体の意図」を1回伝えるだけでOK。

大きめリファクタの内部処理
sequenceDiagram
  participant U as User
  participant C as Claude
  participant FS as Files
  U->>C: "useApi を useFetch に統一"
  C->>FS: Grep "useApi"
  FS-->>C: 12箇所ヒット
  C->>FS: Read 各ファイル
  C->>FS: Edit 各ファイル
  C->>FS: Bash npm test
  FS-->>C: テスト結果
  C-->>U: "12ファイル変更、テストOK"
「全部やって」と頼むと、内部ではこの順で進む。途中の進捗報告は出してもらうこともできる。
もっと詳しく: 部分編集を強制したいとき

「Writeしないで、Editだけで」と明示的に伝えると、Claudeは差分編集に絞ります。長大なファイルを丸ごと書き直されたくないときに有効。

> babel.config.js は触らないで。
src/ 配下のみ Edit で部分修正してください。

実践例: Expressサーバーを段階的に作る

Step 1
骨格を作る
「Express + TypeScript の最小サーバを src/index.ts に作って」
Step 2
エンドポイント追加
「/users GET/POST を src/routes/users.ts に。Zodで入力検証」
Step 3
テスト追加
「Vitest+supertestで、各エンドポイントのテストを tests/users.test.ts に」
Step 4
実行確認
「npm test 走らせて、結果を見せて」
Exercise 1
入門

Glob/Grep を使い分ける

あなたのプロジェクト(またはOSSの適当なリポジトリ)で、以下の質問にClaudeで答えを出してみましょう。

  1. 「.test.ts ファイルは何個ありますか?」
  2. 「console.log はどこで何回呼ばれていますか?」
  3. 「直近に変更されたMarkdownファイル3つは?」
模範解答を見る

Claudeが内部で叩くツールは(おおよそ):

  1. Glob **/*.test.ts → 件数を数える
  2. Grep console\.log → ファイル別件数で集計
  3. Glob **/*.md → 結果は新しい順、上位3件

これらは「自分でripgrepを叩く」より速いことが多い。Claudeは結果を読み込んで「サマリ」も付けてくれます。

Exercise 2
応用

マルチファイルリファクタを依頼する

架空 or 自分のプロジェクトで、「特定の API 呼び出しをラッパー関数に置き換える」リファクタを依頼してみましょう。

  1. 「fetch(...) 直接呼びを全部 apiClient.get/post に置換」と頼む
  2. 変更前にClaudeに「対象ファイル一覧と置換方針」を出させる
  3. 方針を承認 → 一気に置換
  4. テストを走らせて緑になることを確認
模範解答を見る

頼み方:

> src/ 全体で生 fetch(...) を使ってる箇所をリストアップ → それを共通の apiClient に置換する計画を出して。
実行は私の承認後で。テストも合わせて走らせて。

ポイント:

  • Step を分ける: 「計画 → 承認 → 実行」が安全
  • 差分はgitで確認: git diff で全変更を眺める
  • テストで安全網: 振る舞いが変わってないかを最後に検証

大規模な変更でも、この型を覚えれば30分程度で終わります。
