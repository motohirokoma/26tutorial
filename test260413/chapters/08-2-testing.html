テスト & TDD

テストの自動生成、TDD、カバレッジ管理、CI/CD連携まで。テスト文化を定着させましょう。

所要時間: 1h Phase: Core 難易度: 中級

テストの自動生成

テスト作成は、Claudeが特に得意な領域。「この関数のユニットテストを書いて」と頼むだけで、正常系・異常系・境界値を一通り網羅してくれます。

> src/utils/calc.ts の addTax 関数について、Vitest で
正常系3件・異常系2件・境界値1件 のユニットテストを
tests/calc.test.ts に書いてください。
describe で関数ごとにまとめ、it 名は日本語で。
テストピラミッド
下が広い「ピラミッド」が理想。Claudeはどの層も書けるが、まずUnitから固めるのが王道。

言語/ランナー別テンプレ

言語定番ランナーClaudeへの依頼例
TypeScript/JSVitest, Jest「Vitest で it 名は日本語、describe でグルーピング」
Pythonpytest「pytest、parametrize で境界値テスト」
Gotesting「table-driven test、t.Parallel() 付き」
SwiftXCTest, Swift Testing「Swift Testing の @Test、struct でグルーピング」

テスト実行と失敗時の対応

Claudeに 「npm test 走らせて、失敗があれば原因特定 → 修正提案」と一発で頼めます。

> npm test を走らせて、失敗があれば
1) 失敗内容を要約
2) 原因の仮説
3) 修正案(コード)
を順に出してください。実行はまだしないで、私の承認後に。
「テストを通すために実装を歪める」NG

Claudeに「テストを通して」と言うだけだと、最悪の場合 テスト側を雑に直して通す ことがある。「テストの意図は変えず、実装を直して」と必ず添える。

TDD (テスト駆動開発) のやり方

Claude は TDD と非常に相性が良いです。Red → Green → Refactor サイクルを守らせます。

TDDサイクル
graph LR
  R["Red: 失敗するテスト"] --> G["Green: 通る最小実装"]
  G --> RF["Refactor: 整える"]
  RF --> R
3拍子。Claudeに「TDDで進めて」と言うと、各ステップで止まって相談してくれる。
> 機能: ユーザー名は3〜20文字、英数字のみ
TDDで進めて。
まず Red: 失敗するテストを5件書いて、私の承認後に Green に進む。
R

Red

「まだ実装がないので落ちる」テストを書く。仕様を固める段階。

G

Green

テストを通す最小の実装を書く。やりすぎ禁止。

F

Refactor

テスト緑のまま、コードを整える。重複削除・命名改善。

カバレッジを測る

カバレッジは「テストが何%のコードを実行したか」の数値。100%が偉いわけではないですが、主要なロジックパスが通っているかを見るのに使えます。

カバレッジ目標 (チーム別の目安)
チームの種類で適正値は違う。「目標100%」は時に有害(意味のない無理矢理テストが量産される)。
> npm run coverage を走らせて、
- 全体カバレッジ
- 80%未満のファイルTOP5
- そのうち「重要だがテストが薄い」ファイルを推薦
を出してください。

テスト品質の見方

  • 振る舞いをテスト — 実装の詳細(private関数など)はテストしない
  • 速いテスト — 全件を5秒以内に終わらせるのが目標
  • 独立性 — 順番に依存しない、グローバル状態を持たない
  • 失敗メッセージが読める — 「赤になったとき何が悪かったかすぐ分かる」が理想

CI/CDとの連携

Ch07-2 で扱ったActionsと組み合わせて、PRごとに自動テスト → カバレッジレポート投稿、までできます。

.github/workflows/ci.yml
- run: npm run coverage
- uses: actions/upload-artifact@v4
  with:
    name: coverage
    path: coverage/

PRコメントに自動投稿(Claudeを呼ぶ)
- run: |
    claude -p "coverage/coverage-summary.json を読み、
    主要ファイルのカバレッジを表で要約、
    前回比の差分も付けて" \
      | gh pr comment ${{ github.event.number }} -F -
もっと詳しく: テストを「ドキュメント」として書く

良いテストは仕様書になります。describe('ユーザー名バリデーション') / it('英数字のみ通す') / it('20文字を超えるとエラー') のように日本語で意図が読めるテスト名にすると、新人がテストを読むだけで仕様を把握できます。Claudeに「テスト名は日本語で、それ自体が仕様になるように」と頼むのがコツ。

Exercise 1
入門

関数1つに対するユニットテストを書く

適当な計算関数(add discount など)について、Claudeに6件のテスト(正常3、異常2、境界1)を生成してもらいましょう。

  1. 対象関数のあるファイルを @ 付きでClaudeに渡す
  2. テスト6件 + describe で構造化を依頼
  3. テスト名は日本語に
  4. npm test で全緑を確認
模範解答を見る 
import { describe, it, expect } from 'vitest';
import { discount } from '../src/utils/calc';

describe('discount', () => {
  it('10%引きを正しく計算する', () => {
    expect(discount(1000, 0.1)).toBe(900);
  });
  it('0%引きは元の値を返す', () => {
    expect(discount(1000, 0)).toBe(1000);
  });
  it('100%引きは0を返す', () => {
    expect(discount(1000, 1)).toBe(0);
  });
  it('負の割引率はエラー', () => {
    expect(() => discount(1000, -0.1)).toThrow();
  });
  it('1超の割引率はエラー', () => {
    expect(() => discount(1000, 1.1)).toThrow();
  });
  it('境界: 価格0なら結果は0', () => {
    expect(discount(0, 0.5)).toBe(0);
  });
});

テスト名を読むだけで仕様が分かる、というのが「良いテスト」の条件。

Exercise 2
応用

TDDで小機能を実装する

「メールアドレスのバリデーション関数」をTDDサイクルで実装。Claudeに各ステップで止まらせ、承認しながら進めましょう。

  1. 「TDDで進めて。Red → 私の承認 → Green → 承認 → Refactor」
  2. Red: テスト6件(正常3、異常3)
  3. 承認 → Green: 最小実装
  4. 承認 → Refactor: 命名改善・正規表現整理
  5. カバレッジ100%を確認
模範解答を見る

3往復で完成する流れ:

R: 6件のテスト(empty, no-@, no-domain, valid simple, valid plus tag, too-long)
   → 全部 fail を確認 → 承認

G: function isValidEmail(s: string): boolean {
     return /^[^@\s]+@[^@\s]+\.[^@\s]+$/.test(s) && s.length <= 254;
   }
   → 全件 green → 承認

R: 正規表現を定数に切り出し、JSDoc追加
   → green 維持 → 完了

TDDの利点: 「実装の意図」が常にテストに残る。後から触る人(未来の自分)が安心。
