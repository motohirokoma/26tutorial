最終演習：自社業務を1つClaude Code化

24時間の学びを統合し、自分の会社の業務を1つ動くSkillに変える — それがマスターへの道

所要時間: 90分 Phase: Projects 難易度: 応用 (最終章)
Note

本章の記述は 2026年1月時点の Claude Code / Agent SDK / Skills / Hooks / MCP / Scheduled Tasks の仕様に基づきます。パラメータ名・設定ファイル名は変わる可能性があります。運用導入前に必ず 公式ドキュメントで最新を確認してください。

24時間の学びを「1つの成果物」に束ねる

ここまで25章・24時間の旅を走ってきました。対話の原則、ファイル操作、Git連携、プロンプト設計、コードレビュー、リファクタ、TDD、Planモード、Subagents、MCP、Hooks、Scheduled Tasks、Skills、worktree、そして3つのプロジェクト。これが最後の章です。

最終演習のゴールはシンプルで大きい:
自分の会社(または日常)の業務を1つ選び、Claude Code 化された動く状態にし、チーム共有方法まで設計する。

ここがプロの境目

学んだ全てを統合して1つ作る — それがマスターへの道。読むだけの24時間と、成果物1つを出した24時間は、別物の経験になる。断片的な知識は1週間で忘れるが、自分の業務に刺さった Skill は10年使える。
試し方： 今日の自分の作業ログから「3回以上やった仕事」を1つだけ選び、この章の演習を最後まで進めてみる。

24時間の学びが「自社業務1つのSkill」に収束する
25章の知識は単独では使わない。1つの業務に全部束ねてこそ、パワーユーザーの仕事になる。

問題提示: 自社業務を1つ Claude Code 化する

あなたの会社・チームの業務から1つだけ選び、以下の4点セットを揃えてください。このチュートリアルの総決算です。

前提条件

項目内容
対象者Ch01-25 を一通り触った人(触ってないものが3つ以下)
期間90分(設計+実装+検証+共有設計を含む)
個数1つ(絶対に欲張らない)
選定基準週1回以上発生 / 繰り返し / 属人性が痛い / 時間かコストが痛い
形式Skill / Hook / Subagent / Scheduled Task のいずれか(複数組み合わせも可)
動作要件コマンド1行または1イベントで実際に動く状態(スライドだけは不可)
共有要件チームの他の人が同じ手順で再現できる配布方法を設計
検証環境Claude Code CLI (2026年1月時点)

求めるアウトプット (4点セット)

  • 業務選定メモ — 10候補の棚卸しマトリクス(頻度×時間×属人性)から1つに絞った判断
  • 形式判定と設計 — Skill/Hook/Subagent/Schedule のどれか、理由、設計メモ
  • 動く実装 — SKILL.md など実ファイル + 1回以上実行した実行ログ or スクリーンショット
  • チーム共有方法 — リポジトリURL or 社内配布手順 + 効果測定指標の決定
業務選定の3軸マトリクス(頻度 × 時間 × 属人性)
Aの「毎朝のIssue整理」は高頻度・中時間・中属人性で、今回の模範解答のテーマ。自分の業務もこの図にプロットしてみる。

段階ヒント(詰まったら順に開く)

いきなり全部開くと「答え写し」になります。まず自分の業務で5分考え、詰まったヒントだけ開いてください。5つのヒントは 業務選定 → 形式判定 → 設計 → 共有 → 測定 の順です。

もっと詳しく: ヒント1 — 業務選定(繰り返し頻度 × 時間 × 属人性)

まず10個書き出してから絞ります。書き出せない人は1週間の自分のカレンダーを眺めて、同じ形の会議・作業が並んでいる箇所を探してください。以下の3軸で評価:

  • 繰り返し頻度(週あたり): 週1回以下は後回し。週2以上が候補。
  • 1回あたり時間: 15分未満は効果が見えにくい。30分-2時間が狙い目。
  • 属人性(自分しかできないか): 属人性が高いほど Skill化の価値が大きい。

3軸を各10点で採点し、合計25点以上の業務が「最優先ゾーン」。迷ったら 「今週も来週もやる仕事」を1つ選べば外しません。

Tips

候補が出ない人は、Slack検索で自分の発言に "また" "いつもの" "毎週" が含まれる投稿を探してみる。繰り返し業務の金脈がそこに眠っています。

もっと詳しく: ヒント2 — Skill か Hook か Subagent か Scheduled Task か

選定した業務の起動トリガーと並列性で形式が決まります。

形式起動トリガー向いている業務参照章
Skill人間の指示(/skill-name)手順が長く、頻繁に呼び出す定型業務Ch20
Hookツール実行前後・Stop等のイベント「毎回自動でやる」系(commit前にPrettier等)Ch18
SubagentSkill/Hookの中から Task ツールで独立並列化できる観点・対象分割Ch16
Scheduled Taskcron(時刻・間隔)毎朝・毎週など定刻の放置業務Ch19

組み合わせ例:

  • 毎朝のIssue整理: Scheduled Task(毎朝9時)→ Skill を呼ぶ → 内部で Subagent 並列 → 完了後 Hook で Slack 通知
  • PR自動サマリ: Hook(PostToolUse on git push) → Skill → Subagent でレビュー
公式ドキュメント必読

Skills / Hooks / Scheduled Tasks の正確な設定ファイル名・スキーマは公式を必ずマスターしてください。ここを間違えると動かないだけで済みますが、実務のデバッグ時間が一気に膨らみます。Claude Code 公式 →

もっと詳しく: ヒント3 — プロンプト / SKILL.md の設計

Ch20 で学んだ SKILL.md の骨格を再掲します。最終演習はこれを埋めるだけ:

---
name: your-skill-name
description: この Skill をいつ使うか (Claude が自動発動を判断する基準)
---

# {Skill のタイトル}

## いつ使うか
- トリガー1
- トリガー2

## 入力
- {必要な情報・ファイル・URL}

## 手順
1. ステップ1
2. ステップ2
3. ステップ3

## 出力形式
- {期待するフォーマット}

## 注意点
- やってはいけないこと
- エッジケース

ポイントはdescription を「いつ使うか」に特化して書くこと。これが甘いと Claude が Skill を発動しないので、実務で使い物になりません(Ch20の最大の落とし穴)。

もっと詳しく: ヒント4 — チーム共有方法(リポジトリ・配布)

自分だけで動けば十分、ではありません。明日隣の席の人が同じ手順で動かせるまでが合格ラインです。配布パターン3つ:

  1. Git リポジトリ配布(推奨) — company-skills/ リポジトリに .claude/skills/your-skill/ を置き、チーム全員が clone & シンボリックリンクで ~/.claude/skills/ に繋ぐ。更新は git pull で即共有。
  2. 社内Wiki + zip配布(簡易) — Notion や Confluence に手順と SKILL.md を貼り、動作確認済みの zip を添付。小規模チームならこれで十分。
  3. Claude Code プロジェクト設定 — プロジェクトリポジトリ直下の .claude/ にコミット。そのリポで作業する人は自動的に使える。プロジェクト固有の Skill に最適。

配布時の必須3点セット: README(導入手順3行で済む粒度)/ SKILL.md(本体)/ サンプル実行ログ(期待動作のプリセット)。

Warning

Hook や MCP を含む Skill は権限スコープに注意。不用意に Bash(*) を許可すると、チーム全員の端末に広い権限が撒かれます。配布前に 権限スコープを最小化するレビューを入れてください。

もっと詳しく: ヒント5 — 効果測定(時間削減 / 品質向上)

作って終わり、は3週間で忘れます。数字で効果を残すのが最後の仕事です。最小セットは3指標:

時間削減
Before(手動分) → After(Skill分)
品質向上
抜け漏れ件数 / 再作業率(%)
定着度
チーム内の週間利用者数

「1人あたり1日10分の節約 × 10人 × 200営業日 = 年333時間」のように 年間換算で出すと、経営側への説明が強くなります。2週間で最低10回実行して数字を固定し、月次で見直しましょう。

測定で見るべき本当の指標は「自分とチームが、再びそれを使いたいと思うか」。数字が渋くても満足度が高ければ改善で勝てます。逆に数字が良くても誰も使わない Skill は廃止判断を。

模範解答: 「毎朝のIssue整理 Skill」 の完全実装

自分の業務で先に設計してみましたか? ここからは MK が実際に2026年1月に運用している 「毎朝のIssue整理 Skill」の完全実装例です。書き上げてから答え合わせにどうぞ。

模範解答を見る(SKILL.md 全文 + 使い方 + Slack通知Hook連携)

1. 業務選定の判断

MK の場合、棚卸しで出てきた10候補から「毎朝のIssue整理」を選びました。理由:

採用理由 #1

毎日発生する確実性

  • 頻度: 週5回(営業日毎朝)
  • 時間: 1回20-30分
  • 年間: 約100時間消費
採用理由 #2

形式が決まっている

  • 入力: GitHub Issue 一覧
  • 出力: Markdown レポート
  • 判断軸: 優先度 / 期限
採用理由 #3

属人性の解消価値

  • 「MKしか整理できない」を解消
  • 同じフォーマットで継続可
  • 休暇時も代替可能

4. SKILL.md 全文(実ファイル)

以下が ~/.claude/skills/morning-issues/SKILL.md の実物(2026年1月時点で稼働中のもの)。そのままコピペ可能です。

---
name: morning-issues
description: 毎朝のGitHub Issue整理。P0/P1/P2の優先度で並列に整理し、当日対応すべきタスクを Markdown レポートにまとめる。毎朝9:00の自動実行、または「Issue整理して」「今日のタスク整理」と依頼された時に使う。
---

# 毎朝のIssue整理 Skill

## いつ使うか
- 毎朝9:00(Scheduled Task 経由で自動起動)
- ユーザーが「Issue整理して」「今日のタスク整理」と依頼した時
- 週明けや休暇明けに溜まった Issue を一気に棚卸ししたい時

## 入力
- 対象リポジトリ: `~/.claude/skills/morning-issues/config.yml` に記載のリポジトリ一覧
- 対象ラベル: `priority/P0`, `priority/P1`, `priority/P2`(未ラベルは P2 扱い)
- アサイン条件: 自分 or チームメンバー(config.yml の team_members)

## 手順

### Step 1: Issue一覧取得(MCP GitHub)
MCP GitHub サーバー経由で、以下の条件で Issue を取得:
- state: open
- assignee: self + team_members
- sort: updated, direction: desc
- 件数上限: 50件

### Step 2: 優先度で3分割し Subagent 並列起動
Task ツールで3つの Explore Subagent を並列起動:

```
Task(subagent_type="Explore",
     description="P0優先度 Issue 整理",
     prompt="本プロンプトのP0セクションテンプレに従い...")
Task(subagent_type="Explore",
     description="P1優先度 Issue 整理", prompt="...")
Task(subagent_type="Explore",
     description="P2優先度 Issue 整理", prompt="...")
```

各 Subagent のプロンプトは `prompts/p{0,1,2}.md` を参照。

### Step 3: 集約してレポート生成
3つの結果を以下のフォーマットで 1 つの Markdown に集約:

```markdown
# Daily Issue Report YYYY-MM-DD

## 今日必ず対応 (P0)
- [#123] タイトル - 期限: 今日 - 担当: @xxx

## 今週中に着手 (P1)
- [#456] タイトル - 期限: 金曜 - 担当: @yyy

## バックログ (P2)
- [#789] タイトル - 次回レビュー: 来週月曜
```

### Step 4: 保存とサマリ
- `reports/YYYY-MM-DD.md` に保存
- 主エージェントの最終サマリ:「P0: N件, P1: M件, P2: K件。レポート: reports/YYYY-MM-DD.md」

## 出力形式
- Markdown ファイル: `reports/YYYY-MM-DD.md`
- stdout サマリ: 1行(P0/P1/P2 件数 + レポートパス)

## 注意点
- **Issue 本文の機密情報は Slack に貼らない**(タイトルと番号のみ通知)
- MCP GitHub の認証失敗時は、reports/YYYY-MM-DD.md に「取得失敗」と明記して Stop
- P0 が5件を超えた日は、サマリに「⚠️ P0過多」と付記する(ヘルス指標)
- Subagent は isolation=true 推奨(主コンテキストを汚さない)

## 配布
- リポジトリ: `github.com/motohirokoma/company-claude-skills`
- 導入手順: README の 3行で完了(clone → symlink → ~/.claude/settings.json に追記)

6. Slack通知 Hook 連携

Skill の Stop イベントにフックして、Slack #daily-issues に投稿します。~/.claude/settings.json の Hooks セクション:

{
  "hooks": {
    "Stop": [
      {
        "matcher": {
          "skill_name": "morning-issues"
        },
        "hooks": [
          {
            "type": "command",
            "command": "~/.claude/skills/morning-issues/notify-slack.sh"
          }
        ]
      }
    ]
  }
}

notify-slack.sh(実ファイルの抜粋):

#!/usr/bin/env bash
set -euo pipefail

REPORT_DIR="$HOME/projects/issues-reports/reports"
TODAY=$(date +%Y-%m-%d)
REPORT_PATH="$REPORT_DIR/$TODAY.md"

if [[ ! -f "$REPORT_PATH" ]]; then
  exit 0  # レポートが無い日はスキップ
fi

# P0/P1/P2 の件数を集計
P0=$(grep -c '^- \[#' "$REPORT_PATH" | head -1 || echo 0)

# SlackのWebhook URL(環境変数から)
SLACK_URL="${SLACK_DAILY_ISSUES_WEBHOOK:?SLACK_DAILY_ISSUES_WEBHOOK is not set}"

curl -sS -X POST -H 'Content-type: application/json' \
  --data "{\"text\":\"朝のIssue整理完了: ${TODAY} — レポート: ${REPORT_PATH}\"}" \
  "$SLACK_URL" > /dev/null
Tips

Webhook URL を Hook スクリプトに直書きしないのは鉄則。環境変数 or 社内の secret manager 経由で渡します。リポジトリに漏れた Webhook は全チーム宛に誰でも投稿できる爆弾になります(Ch22のMCP権限ミスと同じクラスの事故)。

Insight

MKが2週間運用して一番効いたのは「時間削減」ではなく、「P0見落としゼロ」でした。25分→6.5分はもちろん嬉しいけれど、「月1件の事故が消える」のインパクトは桁違いです。量より質で効くパターンが、属人性の高い業務ほど強く現れる、という学びが本物の収穫でした。

この1つを動かせれば、もう「読者」ではない

ここまで写経でも、改変でも、自分の業務適用でも構いません。動く SKILL.md が1つ手元にある人は、もう「Claude Code を学ぶ人」ではなく「Claude Code で組織の仕事を変える人」になっています。

応用課題

1つ動いたら、次の2つで組織スケールに展開します。

応用

2週間後に3つに拡張する計画を作る

最初の1つを2週間運用したら、同じリポジトリに2つ追加する計画を30分で作ってください。候補は棚卸しマトリクスの2-4位から。1周目の学びを設計に反映するのがポイント。

  1. 1周目で満足度4.0未満の Skill は改修 or 廃止を検討
  2. 2周目は 1つは「書く系」、1つは「調査系」のように役割を分散
  3. 共有リポジトリ company-claude-skills/ に CHANGELOG.md を作り、追加ごとに記録
応用 2
応用

チーム共有Skillパックをリポジトリ化する

自分の3つが揃ったら、チーム全員が貢献できる形にします。company-claude-skills/ を「Skill の共通流通所」として運用する設計を作ってください。

  1. リポジトリのContributing ガイド(新しい Skill を追加する手順・レビュー基準)を書く
  2. 月1のSkill棚卸し会を設定(効果測定の結果を見て、追加・改修・廃止を決める)
  3. 権限スコープのレビューチェックリスト(Hook・MCP 権限の最小化)
模範解答を見る(応用1 + 応用2)

応用1 の例: MK の2周目(2026年1月時点)

  • 継続: 毎朝のIssue整理(満足度4.6)
  • 追加1: PR自動サマリ投稿 Skill(Hook起動、PostToolUse on git push)
  • 追加2: 議事録からToDo抽出 Skill(Subagent×3: 担当別並列)
  • CHANGELOG.md に記録: 追加日 / バージョン / 設計判断 / 効果測定の結果

応用2 の例: Contributing ガイドの骨格

# Contributing to company-claude-skills

## 新Skillの追加手順
1. `skills/{skill-name}/SKILL.md` を作成(テンプレ: `docs/SKILL_TEMPLATE.md`)
2. `examples/` に期待出力サンプル1つ以上
3. README に1行追記
4. PR を作り、下記チェックリストを self-review

## レビューチェックリスト
- [ ] description は「いつ使うか」が明確
- [ ] 入力・出力のフォーマットが決まっている
- [ ] Hook / MCP の権限は最小化されている
- [ ] 環境変数/秘密情報はスクリプトに直書きしていない
- [ ] examples/ にサンプル出力がある
- [ ] 実行ログ or スクショを PR description に添付

## 月1 Skill棚卸し会(第1金曜 16:00-17:00)
- 全Skillの利用回数・満足度をレビュー
- 追加候補(応用1の2周目以降)をチームで投票
- 廃止候補(満足度4.0未満が2か月継続)を決める
Insight

応用2 の真の価値は「個人最適から組織最適への切り替え」です。1人のパワーユーザーが生む効果は年間数百時間ですが、10人のチームでSkillパックが共通資産になれば年間数千時間 — 桁が変わります。ここまで来ると Claude Code は「便利ツール」ではなく「組織のワークOS」の一部になります。

締めのことば: 会社全員が上位20%になる日を作りましょう

おつかれさまでした。24時間・26章の旅をやりきり、自社業務を1つ動く形にしたあなたは、もう間違いなく Claude Code 利用者の上位20%(パワーユーザー)です。おめでとうございます。

1人のパワーユーザーが年間数百時間を取り戻す。
10人のチームがパワーユーザーになれば、年間数千時間が組織に戻ってくる。
そして、会社全員が上位20%になる日 — そこからが本当のスタートです。 — 本チュートリアルの最終メッセージ

でも、あなた1人が上位20%になっただけでは、組織は変わりません。隣の席の人にも、このチュートリアルを勧めてください。あるいは今日作った SKILL.md を社内チャットに流してください。1人から3人、3人から10人、10人から会社全体へ — その伝播を止めないでください。

この運用設計書・SKILL.md・リポジトリは、作って終わりではなく、チームで磨き続けて初めて本当の資産になります。1か月後、半年後、1年後の会社の景色が、今日作った1つのSkillから始まる — そう信じて、今日を終えてください。

あなたの会社の全員が、Claude Code を当たり前に使いこなす日が来ますように。その日まで、1つずつ Skill を育てていきましょう。

Note

本章および本チュートリアル全体の記述は 2026年1月時点の Claude Code / Agent SDK / Skills / Hooks / MCP / Scheduled Tasks 仕様に基づきます。cron 形式・scheduled-tasks.json の場所・Hook イベント名は将来変わる可能性があります。必ず 公式ドキュメントで最新を確認してから運用してください。
