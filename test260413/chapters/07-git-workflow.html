Git × Claude Code 基本ワークフロー

コミット単位、ブランチ、diffレビュー、PR作成、ロールバックまで — Claudeと安全に開発する基礎

所要時間: 45分 Phase: Basics 難易度: 初級
この章のゴール

Claude Code に実装を任せる時、Git が安全網になります。この章では「1機能=1コミット」の粒度感、ブランチ作成と diff レビューのリズム、gh pr create での PR 作成、そして 失敗コミットからの戻し方までを45分で身につけます。これが無いまま大きな変更を依頼すると、壊れた時に戻せなくなる——Claude を本気で使うなら Git の型が先です。

バージョン注記

本章の記述は 2026年1月時点 の Claude Code と gh CLI (v2.x 系) の挙動に基づきます。gh pr create のオプションや PR テンプレート機能は進化するため、コマンドが通らない時は gh CLI 公式マニュアル を参照してください。

なぜ Git × Claude Code か — 3つの理由

Claude Code は便利ですが、時々ファイルを意図せぬ方向に書き換えます。大きな変更を1発で指示した時、テストが消えていた・別ファイルが壊れていた・意図と違う方針で書かれていた——そんな事故を回復可能にするのが Git の役割です。

1機能
1コミットの粒度
↑ ロールバック容易
30秒
diff レビュー所要時間
↑ 壊れてたら即気付く
3秒
git revert で復旧
↑ 小さなコミットの恩恵
  • 安全網 — コミットは「ここまでは動いていた」のセーブポイント。Claude が壊しても戻せる。
  • 実験場 — ブランチを切れば main を汚さずに試せる。失敗したらブランチごと捨てればよい。
  • 可視化 — git diff で「Claudeが何を変えたか」を人間が目で確認できる。盲信を防ぐ最後の砦。
ここがプロの境目

小さな commit を細かく積む。Claude に任せる時こそ「ここまでは動いていた」というロールバックポイントを残すのがパワーユーザーの型です。1機能 = 1コミットを死守すると、何か壊れても git revert 1発で戻せる。大きな変更を1コミットにまとめる癖は、Claude を使う時には最大の敵です。
試し方： 次に Claude に頼む時、「各論理的ステップごとに commit を切って」と明示してみる。

基本ワークフロー全体像

Claude Code で1つの機能を追加する時、下のフローを毎回回します。最初は意識的に、慣れたら無意識に。

Claude Code × Git 標準ワークフロー
graph TD
  A([機能依頼を受ける]) --> B["ブランチ作成
git checkout -b"]
  B --> C[Claudeに実装依頼]
  C --> D["git diff で
変更内容レビュー"]
  D --> E{意図通り?}
  E -- Yes --> F["git add + commit
1機能=1コミット"]
  E -- No --> G[修正を依頼 or 破棄]
  G --> C
  F --> H{"他に変更
必要?"}
  H -- Yes --> C
  H -- No --> I["git push + gh pr create"]
  I --> J([完了])
ブランチ→実装→diff レビュー→commit→PR のループ。各ステップで人間が介入するのがポイント。

このフローの肝は diff レビュー を commit 前に必ず挟むことです。Claudeの作業結果を盲信しない。これが無いと事故が確率で起きます。

ブランチ作成とコミット単位

ブランチを切る習慣を作ると、Claude に自由に書かせても main が守られます。命名は feat/◯◯, fix/◯◯, refactor/◯◯ の3種類を覚えておけば十分です。

今すぐ試す(コピペ): 現在のリポジトリで新しいブランチを切って、Claude に小さな変更を依頼する準備を整えます。

# main の最新を取得してからブランチを切る
git checkout main
git pull origin main
git checkout -b feat/add-user-greeting

# この状態で Claude に依頼する:
# 「src/greet.ts に getGreeting(name) 関数を追加して。
#  テストも同時に書いて、1機能=1コミットで commit して」

コミット単位の「ちょうど良い」粒度

粒度例良い?理由
細かすぎ1行ごとに commit×履歴が読めない、レビュー不能
ちょうど良い1機能 / 1バグ修正 / 1リファクタ○revert/cherry-pickの単位と一致
粗すぎ「今日の作業」で1コミット×部分ロールバック不可、PRが爆発
Claudeへの明示

Claude は放置すると1つのでかいコミットにまとめがちです。依頼時に明示的に 「関数追加 → テスト追加 → ドキュメント更新 の3コミットに分けて」と伝えると、その通りにやってくれます。

diff レビューと commit メッセージ

Claude が実装を終えたら、必ず自分の目で diff を見る。ここが手を抜くと、「なぜかテストが書き換わっていた」「別ファイルが消されていた」系の事故が起きます。

# ステージ前の変更を見る
git diff

# ステージ済みを含めて見る
git diff HEAD

# ファイル単位で確認
git status
git diff src/greet.ts

# 大きい変更なら名前だけ先にスキャン
git diff --stat

commit メッセージは Claude に書かせる

commit メッセージを考えるのは意外と面倒。Claude に任せましょう。コツは「diffを見せて、決まった形式で書いて」と頼むことです。

Before

ベタな依頼

  • 「commitして」
  • メッセージが update など無意味
  • 履歴を辿れない
After

型を伝える依頼

  • 「git diffを確認して、Conventional Commits 形式で日本語 commit メッセージを作って。
    形式: feat(greet): メッセージ」
  • 意図・範囲が明確な履歴
  • PR作成時にそのまま流用可

Conventional Commits の型:

feat
新機能追加
fix
バグ修正
refactor
挙動変えずに構造改善
test
テスト追加/修正
docs
ドキュメント
chore
雑務・設定

Claude への具体的な頼み方例(コピペ):

以下の git diff を見て、Conventional Commits 形式で
commit メッセージを1つ作ってください。

制約:
- 日本語。1行目は50文字以内、2行目以降は必要なら空行+詳細
- prefix: feat / fix / refactor / test / docs / chore
- scope は src 配下の主要ディレクトリ名
- 変更の「何を」「なぜ」を含める(「どう」は書かない)

```diff
$(git diff --staged)
```

PR作成 — gh pr create を使い倒す

gh CLI を使えば、PR 作成をターミナルから一発でできます。ブラウザを開かない分、Claude と対話している流れを止めません。

今すぐ試す:

# ブランチをリモートに push
git push -u origin feat/add-user-greeting

# PR作成(タイトルと本文をコマンドライン指定)
gh pr create \
  --title "feat(greet): getGreeting関数を追加" \
  --body "$(cat <<'EOF'
## Summary
- src/greet.ts に getGreeting(name) を追加
- 対応するテストを tests/greet.test.ts に追加

## Test plan
- [ ] npm test が全パス
- [ ] 空文字入力で 'Hello, !' にならないこと
EOF
)"

# ブラウザで確認したい時
gh pr view --web
公式ドキュメント必読

gh pr create は --draft/--reviewer/--label/--base など多数のフラグを持ちます。ここは公式ドキュメントで一度通読しておくと、以後の作業効率が段違い。gh pr create 公式マニュアル →

Claude に PR 本文も書かせる

コミットメッセージと同じ要領で、PR 本文も Claude に書かせます。複数コミットをまとめるPRなら git log を渡して要約させるのが定石:

以下の commit 群から PR 本文を Markdown で作ってください。

セクション:
## Summary (3-5行、何をしたか)
## Why (なぜ必要か)
## Test plan (チェックボックス形式)
## Notes (レビュアーへの申し送り。なければ省略)

```
$(git log main..HEAD --oneline)
```
もっと詳しく: PR テンプレートを .github/PULL_REQUEST_TEMPLATE.md で固定する

リポジトリ直下に .github/PULL_REQUEST_TEMPLATE.md を置くと、gh pr create 時に自動で本文がプリセットされます。チームで形式を揃えたい時、あるいは自分の脳内テンプレを外出ししたい時に便利:

# .github/PULL_REQUEST_TEMPLATE.md
## Summary
<!-- 変更の概要 3-5行 -->

## Why
<!-- なぜ必要か -->

## Test plan
- [ ] npm test 全パス
- [ ] 手動動作確認

## Notes
<!-- レビュアー向け補足(あれば) -->

Claude に「このテンプレに沿って埋めて」と頼めば、ブレのない PR が量産できます。

失敗 commit の戻し方 — 3つの道具

Claude が変なコードを commit した。あるいは自分が「これでOK」と commit した後に気づいた。そんな時、道具は3つあります。状況によって使い分けるのが肝心です。

ロールバック判断フロー
graph TD
  S([戻したい]) --> Q1{"push 済み?"}
  Q1 -- No --> Q2{"直前の
commit?"}
  Q1 -- Yes --> REV["git revert
(新しいcommitで打消し)"]
  Q2 -- Yes --> AMEND["git commit --amend
or git reset --soft HEAD~"]
  Q2 -- No --> RESET["git reset HEAD~N
(慎重に)"]
  REV --> DONE([完了])
  AMEND --> DONE
  RESET --> DONE
push済みなら revert 一択。ローカルだけなら reset か amend。
コマンド何が起きる使う場面注意
git commit --amend直前の commit を上書きメッセージ誤り / ファイル漏れ(未push)push済みは不可
git reset --soft HEAD~1commit を取消し、変更は staged で残る直前 commit を作り直したいローカルのみ
git reset --hard HEAD~1commit と変更を完全削除捨てたい(危険!)作業消失、reflogで復旧可
git revert <sha>打消し commit を新規作成push済み / チーム開発時履歴が汚れず安全

今すぐ試す(安全なシナリオ): 直前の commit を取消して、やり直す:

# シナリオ1: メッセージを直したいだけ
git commit --amend -m "feat(greet): getGreeting関数を追加（改訂版）"

# シナリオ2: ファイルを追加し忘れた
git add forgotten.ts
git commit --amend --no-edit

# シナリオ3: commitを取消してやり直したい(未push)
git reset --soft HEAD~1   # 変更はstagingに残る
# → ここでファイルを直すなり再編成するなり
git commit -m "fix: ..."

# シナリオ4: push済みを打消したい
git revert <commit-sha>
git push
危険: reset --hard と force push

git reset --hard は 未コミットの変更まで消します。実行前に必ず git status で確認。また push 済みブランチでの git push --force は他の人の作業を消す可能性があります。チーム開発では git revert を選択するのが安全です。

Claude に戻させるときの頼み方

Claude に「直前の commit を取消して、内容を見直してからやり直して」と頼むと、git reset --soft HEAD~1 + 再 commit の流れを自動で組んでくれます。ただし push 済みかどうかは Claude は分からないので、自分で状態を伝えるのがコツ:

直前の commit を取消したい。
- 状態: まだ push していない、ローカルのみ
- 目的: テストが弱い(境界値を見ていない)ので追加してから commit し直したい
- 制約: reset --hard は使わない(変更内容は保持したい)

上の条件で戻してから、テストを追加し、再度 commit してください。

パワーユーザーの1日フロー

ここまでを自然に回せるようになった時の、典型的な1日のリズムです。

1
朝
main を pull、ブランチを切る
2
午前
小機能単位で Claude に依頼 → diff レビュー → commit。2-3サイクル。
3
午後
テスト/動作確認 → 追加 commit → push → gh pr create
4
夕
レビュー対応 → 追加 commit → merge

1日で小さな PR を 1-3本 出すのがパワーユーザーの標準ペースです。大きな PR を週1本出すより、学習速度もレビューの質も圧倒的に高い。

コミット粒度ごとの「戻しやすさ」
もっと詳しく: git add -p で「部分的にステージ」を覚える

1ファイルに複数の論理的変更が混ざった時、git add -p(patch mode)を使うと、hunk 単位で stage するかを選べます:

git add -p src/greet.ts
# y (yes, stage this hunk)
# n (no, skip)
# s (split into smaller hunks)
# q (quit)

これで「このhunkは commit A に、あのhunkは commit B に」と分けられます。Claude がまとめて書いた変更を人間が後から分割する武器です。上位20%ユーザーは頻繁に使います。

もっと詳しく: CLAUDE.md に Git ルールを書く

Ch06 で学んだ CLAUDE.md に、以下のような Git ルールを追加しておくと、毎回指示する手間が消えます:

# CLAUDE.md 抜粋

## Git ルール
- 1機能 = 1コミットを厳守
- commit メッセージは Conventional Commits 形式、日本語
- push 前に必ず diff を見せて、ユーザ確認を取る
- reset --hard, push --force は提案しない(明示依頼時のみ)
- PR 本文は Summary / Why / Test plan の3節

これだけで毎セッションの指示量が半減します。

Exercise — 手を動かして定着させる

Exercise 1
入門

小さな機能を1コミットで PR まで出す

手元の任意の Git リポジトリで、以下を最初から最後まで通してやってみましょう。

  1. main を pull してから feat/hello-claude ブランチを作る
  2. Claude Code に「README.md の末尾に『Built with Claude Code』の1行を追加して」と依頼
  3. git diff で変更内容を自分の目で確認する
  4. Claude に「Conventional Commits 形式で日本語 commit メッセージを作って commit して」と頼む
  5. git push -u origin feat/hello-claude
  6. gh pr create で PR を作る(Summary / Test plan を含める)
模範解答を見る

実際のコマンド列:

git checkout main && git pull origin main
git checkout -b feat/hello-claude

# Claude に依頼: 「README.md の末尾に 'Built with Claude Code' を追加して」
# 変更後:
git diff
# → +Built with Claude Code が見えればOK

# Claude に依頼: 「git diff を確認して、Conventional Commits 形式の
# 日本語 commit メッセージで commit して」
# → Claude が git add README.md && git commit -m "docs: READMEにクレジットを追記" を実行

git push -u origin feat/hello-claude
gh pr create --title "docs: READMEにクレジットを追記" --body "$(cat <<'EOF'
## Summary
- README.md 末尾に 'Built with Claude Code' を追加

## Test plan
- [ ] プレビューで表示確認
EOF
)"

ポイント: 各ステップで「自分が次に何をすべきか」を意識すること。特に git diff のレビューを飛ばさないこと。これが今後の事故率に直結します。

Exercise 2
応用

「しくじった commit」を3通りの方法で戻す

失敗シナリオを自分で作り、3つのロールバック方法を1つのブランチで試します。

  1. 実験用ブランチ practice/rollback を作る
  2. ダミーファイル sample.txt を作り、「bad commit」というメッセージで commit する(ファイル内容は適当に3行)
  3. 未push状態で git commit --amend -m "fix: メッセージ修正" を試す
  4. さらに commit を重ねた後、git reset --soft HEAD~1 で戻して、再 commit
  5. 最後に push してから、直前の commit を git revert で打消す新しい commit を作る
  6. 各ステップで git log --oneline を確認し、履歴がどう変化したかメモする
模範解答を見る

全工程のコマンド例:

# 準備
git checkout -b practice/rollback
echo -e "line1\nline2\nline3" > sample.txt
git add sample.txt
git commit -m "bad commit"
git log --oneline   # bad commit が1つ見える

# [1] amend でメッセージ修正
git commit --amend -m "fix: メッセージ修正"
git log --oneline   # メッセージだけ変わる(SHAは変化)

# [2] もう1コミット重ねる
echo "line4" >> sample.txt
git add sample.txt && git commit -m "add line4"

# [3] soft reset で直前を取消、再commit
git reset --soft HEAD~1
git status   # line4の変更が staged に戻っている
git commit -m "feat: sample.txt に4行目を追加"

# [4] push してから revert
git push -u origin practice/rollback
git revert HEAD --no-edit
git push
git log --oneline   # 打消しcommitが1つ追加されているのが見える

学びのポイント: amend と reset --soft はSHAを書き換える(履歴改変)。push済みなら revert 一択で新しい commit を積む。SHA の変化を git log --oneline で確認する習慣が付けば、一人前です。
