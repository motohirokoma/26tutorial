Hooksで自動化

「毎回頼むこと」を settings.json で自動発火させる

所要時間: 60分 Phase: Advanced 難易度: 中級
この章のゴール

Hooks は、Claude Code の動作の「継ぎ目」に自分のスクリプトを差し込める仕組み。ファイル保存のたびに Prettier を走らせる、テストが通ったら通知を鳴らす、危ないコマンドは実行前に止める——といった自動化が settings.json 数行で実現します。
この章では、発想→構造→主要イベント→定番レシピ3つ→セキュリティ を60分で通し、終わるころには「毎日繰り返している手動作業」を1つ消せる状態にします。

2026年1月時点の情報

本章は 2026年1月時点 の Claude Code Hooks(PreToolUse / PostToolUse / Stop / Notification / SessionStart など)挙動に基づきます。イベント名や JSON スキーマは更新される可能性があるので、動かない時は 公式ドキュメント を見るのが最速です。

Hooksの発想 — 「頼まなくても動く」世界へ

Hooks は、Claude が「これからツールを使う瞬間」「使った直後」「応答を返し終わった瞬間」など特定の継ぎ目を フック(引っ掛けて) 、あなたのシェルコマンドを自動実行する仕組みです。Claude に毎回お願いしていた「保存したら Prettier かけて」を、Claude が忘れようが気絶しようが確実に実行してくれます。

依頼主導 vs Hook主導 — 自動化の境目
依頼を忘れても/Claudeが忘れても、Hook は静かに確実に動く。

ポイントは 「お願い」から「契約」への昇格。お願いは揺れる(Claude の機嫌、自分の疲労度)。契約は揺れない(settings.json に書いた瞬間に確定)。これが自動化の本質です。

ここがプロの境目

「毎回頼むこと」を1つ Hook 化するだけで、1日に5回は手間が消える。1日5回 × 20営業日 = 月100回。3秒の依頼でも累計5分、認知負荷で見ればもっと大きい。上位20%は「同じ依頼3回目」で即 Hook 化する。
試し方： 今日やった繰り返し手動作業(保存→lint・コミット前チェック・終了通知のどれか)を1つ選び、本章末の「Recipe 1」を丸コピして5分で体験する。

settings.json の構造を頭に入れる

Hooks は Claude Code の settings.json に書きます。配置場所は3階層:

配置パス用途Git管理
User(全プロジェクト共通)~/.claude/settings.json個人の通知音・ログ等対象外
Project(チーム共有).claude/settings.jsonPrettier/Lint/テスト等チーム標準コミットする
Local(個人オーバーライド).claude/settings.local.json一時的な実験・個人ツール.gitignore

Hooks の JSON 骨格はこう。イベント名 → matcher(ツール名パターン) → hooks 配列 → command の4層です。

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          { "type": "command", "command": "npx prettier --write $CLAUDE_FILE_PATHS" }
        ]
      }
    ]
  }
}
settings.json Hooks の構造
イベント名 → matcher → hooks[] → command の4層を覚える。

matcher の書き方

matcher は どのツールに反応するか を決める正規表現風の文字列。"Write|Edit" は OR、""(空) は全ツール、"Bash" なら Bash 実行のみ。ツール名を知らない時は、一旦 "" で全取りして $CLAUDE_TOOL_NAME をログに吐き出すのが早いです。

もっと詳しく: 使える環境変数一覧
環境変数意味例
$CLAUDE_TOOL_NAME実行されたツール名Write / Bash
$CLAUDE_FILE_PATHS対象ファイルパス(スペース区切り)src/app.ts src/index.ts
$CLAUDE_PROJECT_DIRプロジェクトのルート/Users/me/proj
$CLAUDE_SESSION_ID現在のセッションIDUUID 文字列

JSON で入力を受けたい時は、command の代わりに stdin 経由で tool input をパースする(jq との組み合わせが鉄板)。詳細は公式 hooks リファレンス参照。

公式ドキュメント必読

Hooks のイベント名・matcher 文法・stdin スキーマは、誤ると 無限ループや破壊的操作 を招きます。ここは必ず公式で正確な最新仕様を押さえてください。Claude Code 公式 Hooks リファレンス →

主要イベント — Pre/PostToolUse と Stop

全イベントのうち まず覚えるのは3つ だけで十分。これで9割の自動化が書けます。

1ターン内の Hook 発火フロー
graph LR
  U([ユーザー入力]) --> M[Claude 応答生成]
  M --> Pre["PreToolUse Hook"]
  Pre -->|許可| T[ツール実行]
  Pre -->|block| Stop1([中断])
  T --> Post["PostToolUse Hook"]
  Post --> M
  M --> End["Stop Hook"]
  End --> Done([次の入力待ち])
PreToolUse は「ツール実行の前」、PostToolUse は「直後」、Stop は「ターン終了時」。
イベント発火タイミング得意な用途注意
PreToolUseツール実行 直前危険コマンド検知・block / 入力補正exit 2 で blockできる
PostToolUseツール実行 直後Prettier / Lint / テスト / ログツールが失敗でも発火する設計に注意
StopClaude がターンを 終えた時通知音・Slack通知・git status長いスクリプトは非推奨
NotificationClaude が 注意を引きたい 時(パーミッション要求等)OS通知連携発火タイミングが特殊
SessionStartセッション 開始時環境情報注入・コンテキスト先出し出力が大きいとコスト増

exit code でフローを制御

PreToolUse の Hook コマンドが exit 2 で終了 すると、Claude Code はそのツール実行を block し、標準エラー出力を Claude に見せます。つまり「このコマンドは危険なのでやり直させる」ができる。これが Hooks の最大の武器の一つです。

  • exit 0 — 何も言わず通す(デフォルト成功)
  • exit 2 — PreToolUse では block、PostToolUse では stderr を Claude に feedback
  • その他の exit code — エラーとしてログ、実行は止めない

定番レシピ3つ — コピペで動く

まずは以下3本を .claude/settings.json に入れて、Claude Code を再起動。これだけで今日から体感が変わります。

Recipe 1: 保存時に Prettier 自動整形

今すぐ試す (5分)

ステップ: (1) プロジェクトルートに .claude/settings.json を作成 → (2) 下記を貼り付け → (3) Claude に「何か .ts ファイルを編集して」と頼む → (4) 保存後に整形されていれば成功。

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit|MultiEdit",
        "hooks": [
          {
            "type": "command",
            "command": "for f in $CLAUDE_FILE_PATHS; do case \"$f\" in *.ts|*.tsx|*.js|*.jsx|*.json|*.md) npx prettier --write \"$f\" 2>/dev/null ;; esac; done"
          }
        ]
      }
    ]
  }
}

ポイント: case 文で拡張子をガードするのがコツ。バイナリや画像まで Prettier に食わせると警告で画面が荒れます。

Recipe 2: Stop 時にテスト走らせて通知

{
  "hooks": {
    "Stop": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "cd $CLAUDE_PROJECT_DIR && npm test --silent && osascript -e 'display notification \"tests green\" with title \"Claude Code\" sound name \"Glass\"' || osascript -e 'display notification \"tests RED\" with title \"Claude Code\" sound name \"Basso\"'"
          }
        ]
      }
    ]
  }
}

ポイント: macOS の osascript で通知+音。Linux なら notify-send、WSL なら PowerShell の BurntToast に置き換え。テスト失敗でも Claude を止めない(情報提供のみ)のがポイント。

Recipe 3: PreToolUse で危険コマンドを止める

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "jq -r '.tool_input.command // empty' | grep -E '(rm -rf /|git push --force|:\\(\\)\\{|dd if=)' && { echo 'BLOCKED: destructive pattern' >&2; exit 2; } || exit 0"
          }
        ]
      }
    ]
  }
}

ポイント: stdin から JSON でツール入力が渡ってくるので jq で抽出。危険パターンにマッチしたら exit 2 で block + stderr で Claude にフィードバック。Claude は「このコマンドは止められた、別の手段を考えよう」と挙動を変えます。

どの手動作業を Hook 化するか

「繰り返し × 定型 × 低リスク」が Hook 化の3条件。下のレーダーで自分の依頼を採点して、80点以上を超えた作業から順に Hook 化しましょう。

Hook化候補スコアリング(例: 保存時Prettier)
Hook化による時間削減(週次シミュレーション)
もっと詳しく: Hook化してはいけない作業
  • デプロイ・本番反映 — Stop 発火で勝手にデプロイは絶対NG。必ず手動トリガーに
  • 金銭が動くAPI — LLM課金・外部SaaS課金系は Hook で連打すると破産
  • 判断が要る作業 — 「このPRレビューOKですか?」→ 人間が読むべき
  • 他人の環境への影響 — 共通DB更新・共有サーバ再起動等

セキュリティ — Hooks は刃物

⚠️ セキュリティ警告

Hooks は あなたのシェル権限で何でも実行できます。信用できない settings.json(他人の git clone、怪しい gist 貼り付け)を読み込むと、PCが乗っ取られます。必ず内容を目視し、初見の Hook は echo に置き換えてから試すこと。

  • 他人のsettings.jsonを読む前に目視 — git clone 時は .claude/settings.json を先に確認
  • シークレットを echo しない — env 丸ごと log すると .zsh_history に漏れる
  • 無限ループに注意 — PostToolUse で Write を呼ぶ Hook は自分を再発火させうる(Ch22 で詳述)
  • 長すぎるHookはタイムアウトで切る — timeout 30 ... を頭に付ける癖を
  • チーム共有はプロジェクトsettings.jsonのみ — settings.local.json は絶対コミットしない
Hook は契約。契約書を読まずにサインするな。コピペ前に1行ずつ何をするか言語化できることだけコピペする。 — 共有settings.jsonで社内事故を起こした人の言葉
もっと詳しく: Hook debugのベストプラクティス
  1. まず echo でドライラン — "command": "echo [PostToolUse] $CLAUDE_FILE_PATHS >> /tmp/claude-hooks.log"
  2. ログをtail — 別ターミナルで tail -f /tmp/claude-hooks.log
  3. 単発で手動テスト — CLAUDE_FILE_PATHS="test.ts" sh -c "コマンド"
  4. 本番コマンドに置換 — 動作確認してから Prettier なり何なりに差し替え

「いきなり本番ツールをHookに入れない」「必ず一度は log だけ吐かせる」——この2つで事故の9割は防げます。

Exercise 1
入門

Recipe 1 を動かす

自分のプロジェクトに Recipe 1(Prettier 自動整形)を導入し、実際に動作を確認してください。

  1. プロジェクトルートで mkdir -p .claude && touch .claude/settings.json
  2. Recipe 1 の JSON をコピペして保存
  3. Claude Code を再起動(必要なら /exit → 再起動)
  4. Claude に「sample.ts を作って。中身は整形されていない関数1つ」と頼む
  5. 生成されたファイルが Prettier で整形されていることを確認
模範解答を見る

以下の手順で5分以内に確認できます。

mkdir -p .claude
cat > .claude/settings.json <<'EOF'
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit|MultiEdit",
        "hooks": [
          { "type": "command",
            "command": "for f in $CLAUDE_FILE_PATHS; do case \"$f\" in *.ts|*.tsx|*.js|*.jsx|*.json|*.md) npx prettier --write \"$f\" 2>/dev/null ;; esac; done" }
        ]
      }
    ]
  }
}
EOF

確認ポイント: ファイル保存後に git status を見ると、同じファイルが Prettier の整形で変わっているはず。npx prettier を使うので初回は依存が取得されるまで1-2秒遅延します。

詰まった時: which npx で npx が PATH にあるか確認。モノレポ等で .prettierrc が無い時は prettier --check でエラーが出るので、事前に npm i -D prettier しておくと確実です。

Exercise 2
応用

自分専用Hookを1つ設計して実装する

今日あなたが Claude Code に 2回以上同じことを頼んだ 作業を1つ選び、Hook 化してください。下の手順に沿って進めます。

  1. 今日の依頼を書き出し、繰り返し3条件(繰り返し/定型/低リスク)で10点満点評価
  2. 最高得点の作業を PreToolUse / PostToolUse / Stop のどれで動かすか決める
  3. まず echo でドライラン Hook を書き、/tmp/claude-hooks.log に書き出す
  4. ログで発火タイミングが正しいことを確認してから、本番コマンドに置き換える
  5. 1週間運用してみて「誤発火」「忘れていたケース」があれば matcher を調整
模範解答を見る

例: 「commit 前に TypeScript の型チェック走らせて」を毎回頼んでいる人

→ これは PreToolUse の Bash + git commit パターンをフックするのが正解。

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "jq -r '.tool_input.command // empty' | grep -q 'git commit' && { cd $CLAUDE_PROJECT_DIR && npx tsc --noEmit || { echo 'tsc failed — commit blocked' >&2; exit 2; }; } ; exit 0"
          }
        ]
      }
    ]
  }
}

設計メモ:

  • matcher を Bash に絞る(Write/Editで発火させない)
  • jq で tool_input.command から git commit を検出
  • tsc 失敗時は exit 2 で block + stderr にメッセージ → Claude が「型を直してから commit しよう」と軌道修正してくれる
  • 最後の ; exit 0 は他の Bash コマンド(ls 等)を巻き込まないため

ドライラン版 (先にこれで発火を確認):

"command": "jq -r '.tool_input.command // empty' >> /tmp/claude-hooks.log; exit 0"

1週間運用して誤発火がなければ本番版に差し替える、が鉄板フロー。
