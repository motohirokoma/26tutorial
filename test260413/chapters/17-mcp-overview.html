MCPで主戦場ツールと繋ぐ

Claude Code を「自分の主戦場」へ接続し、生産性を桁で変える標準プロトコル

所要時間: 60分 Phase: Advanced 難易度: 中級
この章のゴール

MCP(Model Context Protocol)を「Claude と外部ツールを繋ぐ共通コネクタ規格」として直感できるようにし、.mcp.json に1つサーバーを書いて動かすところまで60分で到達します。GitHub / Slack / Google Drive / 社内DB — あなたの「主戦場ツール」が Claude Code の中から直接操作できるようになる、その入口です。

2026年1月時点の情報

本章の記述は 2026年1月時点 の Claude Code と MCP 仕様(プロトコルバージョン 2025-06-18 系)に基づきます。公式サーバーのパッケージ名や認可フロー(OAuth 2.1)は改訂が続いているため、実際に接続する際は MCP公式 / Claude Code公式 を必ず参照してください。

MCPとは — 「AIのUSB-C」

MCP(Model Context Protocol)は、Anthropicが公開したLLMと外部ツールを繋ぐ共通規格です。よく「AIのUSB-C」と呼ばれます — これまで各LLMクライアントが個別に作っていた「GitHub統合」「Slack統合」「Notion統合」を、1本の規格にまとめた、というのがポイント。

Claude Code は MCP クライアントとして動作し、MCPサーバーを .mcp.json に登録するだけで、そのサーバーが公開するツールがすべて Claude から使えるようになります。

MCPの全体像 — クライアントとサーバーの接続
Claude Code は一律のプロトコル(JSON-RPC 2.0)で各MCPサーバーと話す。サーバーは実サービス(GitHub / ファイルシステム / Slack 等)に専用の方法でアクセスする。
今すぐ試す(1分)

ターミナルで現在登録されているMCPサーバーを確認してみましょう。

claude mcp list

まだ何も入れていなければ空のはずです。この章の後半で最初の1つ(Filesystem MCP)を入れます。

なぜ必要なのか — N×M問題

MCPが生まれた背景は「N×M問題」です。5つのLLM(ChatGPT / Claude / Gemini / Cursor / ...)と100個のツール(GitHub / Slack / Jira / ...)を繋ごうとすると、単純計算で 5×100 = 500個のインテグレーションが必要になる。これが共通プロトコルMCPで 5 + 100 = 105個 に圧縮される、というのが発想の根っこです。

N×M(個別統合) vs N+M(MCP経由)
クライアント5 × ツール100 の場合、個別統合は500本、MCPなら105本。規模が増えるほど差が指数的に広がる。

現実的には「すべてのツールを一人で使うことはない」ので、個人レベルでは 主戦場ツール3-5個を MCP 化 するのが一番効きます。毎日触るもの、つまり あなたの"主戦場" を識別することが最初のステップです。

主戦場タイプ典型ツール代表的なMCPサーバー
コード管理GitHub / GitLab@modelcontextprotocol/server-github
コミュニケーションSlack / Discord@modelcontextprotocol/server-slack
ドキュメントGoogle Drive / Notion@modelcontextprotocol/server-gdrive / Notion公式
タスク管理Jira / LinearAtlassian公式 / Linear公式
ローカル資源ファイルシステム / SQLite@modelcontextprotocol/server-filesystem / server-sqlite
DB / データ基盤Postgres / BigQueryserver-postgres / ベンダー公式
ここがプロの境目

主戦場ツール(GitHub / Slack / Drive等)は、迷わず MCP 化する。「Claude Code で何でもやる」をやれるようになると、コピペ/貼付けの往復が消え、生産性が桁で変わる。「PRの一覧を見て説明して、コメント付けて」「Slackの#devチャンネルの今週の議論を要約」がチャット1行で完結する世界です。
試し方： 自分が今週もっともタブを切り替えたツール1つを思い出し、今週中に対応MCPサーバーを入れる。まずは1つ、読み取り専用スコープで十分。

.mcp.json の書き方 — 3スコープ

Claude Code での MCP 設定は、基本的に .mcp.json ファイルに書きます。スコープ(=どこに書くか)で読み込み範囲が決まるのが最重要ポイントです。

local (個人)

~/.claude/mcp.json

  • 自分のマシン全体で有効
  • API キー等を安心して書ける
  • チームには共有されない
project (共有)

./.mcp.json(リポジトリ直下)

  • プロジェクトごとのサーバー定義
  • Gitで共有可能、秘密情報は環境変数で
  • チーム全員が同じMCP構成になる
project.local (非共有)

./.mcp.local.json

  • 個人オーバーライド
  • .gitignore対象
  • 自分だけの実験用

.mcp.jsonの最小例(Filesystem MCP を追加):

{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-filesystem",
        "/Users/you/Desktop/claude"
      ]
    }
  }
}

この mcpServers オブジェクトのキー("filesystem")が、Claude 側から見たサーバー名になります。args の末尾に渡したパス(/Users/you/Desktop/claude)がアクセス許可される唯一のディレクトリ — このスコープ外には絶対に触れません。

今すぐ試す(5分) — Filesystem MCP 導入

一番簡単な Filesystem MCP を入れて動作確認します。

  1. プロジェクトルートで claude mcp add filesystem --scope project -- npx -y @modelcontextprotocol/server-filesystem $(pwd)
  2. claude mcp list で filesystem が ✓ になっているか確認
  3. Claude Codeを開いて「mcp__filesystem__list_directory を使って、このプロジェクトのルートを一覧化して」と依頼

ツール名は mcp__{サーバー名}__{ツール名} のパターンで自動展開されます。

もっと詳しく: スコープの優先順位と読み込み順

同じサーバー名が複数スコープに書かれている場合、project.local > project > local(user) の順で優先されます。チームは共通構成を .mcp.json に、個人APIキーは .mcp.local.json に書く、が基本パターン。また、env フィールドで環境変数を子プロセスに渡せます(後述GitHub MCP参照)。disabled: true を書くと無効化できます(一時オフ用)。

トランスポート — stdio vs HTTP

MCPサーバーとの通信手段は大きく2つあります。stdio と HTTP(Streamable HTTP + SSE)です。どちらを選ぶかで設定形が変わります。

観点stdioHTTP
サーバーの所在ローカル(自分のマシン)リモート(クラウド) or ローカル
起動方法Claude Code が子プロセスとして起動既に起動しているサーバーのURLを叩く
認証プロセス起動時に環境変数で渡すOAuth 2.1 / Bearer token
得意ファイル操作・ローカルDB・個人利用チーム共有サーバー・SaaS公式
苦手複数マシンで共有セットアップのハードル高

stdio形式の例(ローカル実行):

{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "${GITHUB_TOKEN}"
      }
    }
  }
}

HTTP形式の例(リモートサーバー接続):

{
  "mcpServers": {
    "company-wiki": {
      "type": "http",
      "url": "https://mcp.your-company.com/wiki",
      "headers": {
        "Authorization": "Bearer ${COMPANY_MCP_TOKEN}"
      }
    }
  }
}
どちらを選ぶか(判断フロー)
graph TD
  A[MCPサーバーを追加したい] --> Q1{"サーバーは自分の端末で動く?"}
  Q1 -- Yes --> S[stdio を選ぶ]
  Q1 -- No --> Q2{"公式がHTTPエンドポイント提供?"}
  Q2 -- Yes --> H["HTTP を選ぶ(OAuth推奨)"]
  Q2 -- No --> R["リモート用にstdio→HTTP ブリッジを立てる"]
  S --> End[完了]
  H --> End
  R --> End
ローカル用は stdio、リモート/SaaS は HTTP。公式サーバーが提供されていればまずそれを使う。
公式ドキュメント必読

ここは公式ドキュメントにも書いていますし、実務で間違えると致命的(権限広すぎで情報漏洩等)なので必ずマスターしてください。MCP Transports 仕様 → / Claude Code MCP docs →

実装例 — GitHub MCP

主戦場ツールの中でも、開発者にとってGitHubは間違いなく第1位。ここを MCP 化すると、Claude Code のチャットで gh コマンドを自分でゴリゴリ叩かなくても、PR一覧/Issue作成/コメント投稿までまとめて任せられます。

1
PAT発行
scopeはrepo,read:org最小限
2
環境変数
export GITHUB_TOKEN=...
3
設定追加
.mcp.local.jsonに記述
4
動作確認
claude mcp listで✓
5
実タスク投入
「今週のPRを要約して」

.mcp.local.json の中身(個人のトークンを含むので必ず .local 側へ):

{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "${GITHUB_TOKEN}"
      }
    }
  }
}

設定後、Claude Code にこう頼めます(実プロンプト例):

mcp__github__list_pull_requests を使って、
リポジトリ anthropics/claude-code の直近10件のPRを
タイトル・作成者・状態の表にまとめて。
今すぐ試す(10分)
  1. GitHubの Fine-grained PAT を発行(Repo access限定、必要最小scope)
  2. ~/.zshrc に export GITHUB_TOKEN=ghp_xxx を追記して source ~/.zshrc
  3. claude mcp add github --scope local -- npx -y @modelcontextprotocol/server-github
  4. Claude Code を開き直して「自分が今週開いたPRを一覧化して」と頼む

うまくいけば、mcp__github__search_issues 等のツール呼び出しが画面に出ます。

認証とスコープ — 事故る前に

MCPは強力な分、権限設計を間違えると一撃で事故るツールでもあります。失敗パターンはだいたい「スコープが広すぎる」「秘密情報が共有ファイルに漏れた」の2パターン。

  • トークンは最小スコープで — GitHubなら repo:read だけで足りる場面は多い。admin:*系は書かない
  • 秘密情報は .mcp.local.json または環境変数へ — .mcp.jsonに平文トークンを書いて git push したら終わり
  • Filesystem MCP は許可パスを狭く — $HOME 直下や / は絶対NG。プロジェクトルートのみ
  • 信頼できないサーバーを入れない — 公式 or 一次ベンダー or コード確認済みOSS のみ。怪しいnpmパッケージは中でなんでもできる
  • 破壊的ツール(delete/push/merge)は確認プロンプトを有効に — Claude Codeの --dangerously-skip-permissions と MCP は相性が悪い、絶対に併用しない
スコープの広さ vs リスク(イメージ)
"最小権限" は理想値。実務では開発用トークンでも repo を丸ごと与えがちだが、意識して絞ると事故が激減する。
もっと詳しく: OAuth 2.1 対応のHTTPサーバー

2025年の仕様改訂で、リモートMCPサーバーは OAuth 2.1 + PKCE による認可フローが推奨になりました。Claude Code は type: "http" サーバーの /.well-known/oauth-authorization-server を自動探索し、ブラウザで認可画面を開いてtoken交換します。PAT直書きより安全で、revokeもサービス側UIから一発。チーム導入時はHTTP+OAuthが本命です。

もっと詳しく: 失敗事例アンチパターン3つ

(1) PAT共有リポジトリにハードコード → 他人にMergeされた瞬間に露出。(2) Filesystem MCP で $HOME 許可 → .ssh/id_rsaまで読み放題。(3) 怪しいサードパーティMCPを npx で即実行 → 実は環境変数全送信のマルウェア。いずれも「最小スコープ + 公式ソースのみ」で回避できます。

MCPは便利の延長に「権限」という大きな扉を開く。開けた扉を常に意識するのがプロ。 — MCP運用の鉄則

Exercise 1
入門

Filesystem MCP を自プロジェクトに入れる

自分が普段触っている任意のプロジェクトディレクトリで Filesystem MCP を動かし、Claude Codeから「README の先頭10行を要約して」を実行してみる。

  1. プロジェクトルートで claude mcp add コマンドで Filesystem を追加
  2. claude mcp list で接続OKを確認
  3. Claude にREADMEの要約を依頼し、mcp__filesystem__ で始まるツール呼び出しが使われるか観察
模範解答を見る

以下のコマンドをプロジェクトルートで実行。

# 追加
claude mcp add filesystem --scope project -- npx -y @modelcontextprotocol/server-filesystem $(pwd)

# 確認
claude mcp list
# => filesystem: npx ... (✓)

# Claude Code内でのプロンプト
このプロジェクトの README.md の先頭10行を読んで、
内容を3行で要約してください。

途中で Claude が mcp__filesystem__read_text_file や mcp__filesystem__list_directory を呼んでいれば成功です。ツール呼び出しログは画面に表示されます。呼ばれていない場合は、手動で「MCPの filesystem ツールを使って」と指示を足すと確実。

Exercise 2
応用

自分の主戦場ツール1つをMCP化する計画を書く

あなたが今週一番タブを切り替えたツールを1つ選び、MCP化の実行計画を作る。実施は今週中に。

  1. ツール名と、MCP化したい代表ユースケース3つを列挙
  2. 対応するMCPサーバー(公式 or 一次ベンダー)を公式リストから探す
  3. 必要な権限スコープを最小限で洗い出す
  4. .mcp.json または .mcp.local.json の草案を書く(トークンはプレースホルダ)
  5. リスク(権限事故 / 秘密漏洩 / レート制限)を1-2行で書き留める
模範解答を見る

例: Slack を主戦場に選んだ場合。

# 選定
Tool: Slack
Usecases:
  - 今週の #dev チャンネルのスレッドを要約
  - 自分メンションの未読を一覧
  - 決定事項を Notion へ転記(後続MCPと組み合わせ)

# MCPサーバー
@modelcontextprotocol/server-slack (公式)

# 最小スコープ (Slack App設定)
channels:history, channels:read, users:read, chat:write (投稿する場合のみ)
admin系は不要

# .mcp.local.json 草案
{
  "mcpServers": {
    "slack": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-slack"],
      "env": {
        "SLACK_BOT_TOKEN": "${SLACK_BOT_TOKEN}",
        "SLACK_TEAM_ID": "${SLACK_TEAM_ID}"
      }
    }
  }
}

# リスク
- Bot tokenが漏れると自分のSlackアカウント権限で操作される
  => .mcp.local.json は必ず .gitignore
- chat:write を付けると誤投稿の事故可能性
  => まず読取専用で始める

このように「ユースケース → サーバー選定 → 最小スコープ → 設定草案 → リスク」で書くと、導入のPR/相談資料としてそのまま使えます。
