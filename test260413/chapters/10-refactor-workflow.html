既存コードの改修ワークフロー

理解 → 計画 → 実装 → 検証 — 4ステップで壊さず直す

所要時間: 60分 Phase: Core 難易度: 中級
この章のゴール

既存コードベースの改修を、いきなり書き換えさせず「理解 → 計画 → 実装 → 検証」の4ステップに分解して Claude に頼めるようになります。Explore subagent で現状を把握し、Plan モード で計画を固めてから、段階的に実装・テストで担保する流れを、実リポで手を動かしながら身につけます。

2026年1月時点の情報

本章の挙動・ツール名・コマンドは 2026年1月時点の Claude Code に基づきます。subagent_type のラインナップ、Plan モードのUI、TodoWriteの仕様は更新される可能性があるため、詰まったら 公式ドキュメント を確認してください。

なぜ改修は「いきなり書き換えさせる」と壊れるのか

新規作成と違い、既存コードの改修は地雷原です。関数は他の場所から呼ばれているし、テストが暗黙の仕様になっているし、コメントに書いていない「なぜそうなっているか」が積み重なっている。Claude に「ここをこう直して」とだけ頼むと、目の前の1行だけ綺麗に直して周辺を壊す——という事故が起こります。

これを防ぐための型が、今日学ぶ 4ステップ改修ワークフロー です。「書き換え」を一番最後に置く、ただそれだけの思想です。

4ステップ改修ワークフロー
各ステップの主役ツールは異なる。step 3 より前に3/4の時間を使うのが「プロの改修」。
ここがプロの境目

改修前に必ず「理解フェーズ」を置く。いきなり書き換えさせると、Claude は呼び出し元・テスト・仕様を見ずに「その1行」だけ綺麗にして周辺を壊す。理解に30%の時間を投下するだけで、差し戻し率が体感で半分以下になる。
試し方: 次に改修を頼むとき、最初のメッセージは必ず「このモジュールの責務を3行でまとめて、呼び出し元を列挙して」だけにする。

Step 1 — 理解:Explore subagent で現状を押さえる

最初のステップは「書き換える前に知る」こと。ポイントは メイン文脈を汚さず に情報を取ることです。ここで Explore subagent が主役になります。

通常の Read でファイルを全部読ませると、メインの コンテキスト がどんどん減る。Explore は子文脈で読んで、要約1つだけ親に返してくれるので、メインは疲れずに判断だけに集中できます。

理解フェーズで聞くべき5つの質問

  • 責務 — このモジュール/関数は「何をする」のが仕事? 3行で。
  • 呼び出し元 — どこから呼ばれている? Grep 結果のファイル一覧で。
  • 既存テスト — 何が assertion されている? テスト名を列挙して。
  • 副作用 — DB書き込み、ファイルIO、ネットワーク呼び出しは?
  • 暗黙の仕様 — コメントにないが守られている約束(例: null許容、空配列時の挙動)は?
今すぐ試す(3分)

自分のリポジトリで、これから改修したい関数を1つ選んで、Claude Code にこう頼んでください。

Agent tool を subagent_type="Explore" で起動して、以下を調べて要約だけ返して:
1. src/xxx/yyy.ts の関数 fooBar() の責務を3行で
2. fooBar() を呼んでいる箇所をファイル単位で列挙
3. fooBar() に対する既存テストを列挙(テスト名だけでOK)
4. 副作用(IO/DB/network)があればその種類を明記
5. コメントに書かれていないが守られていそうな暗黙仕様を2つ推測

返ってきた要約を、そのまま次のステップ(計画)の入力にします。ここで「全ファイルを読んでまとめました」ではなく Explore 子経由で要約だけ取るのが重要。

Explore vs 通常 Read の使い分け

場面Explore subagent通常 Read/Grep
ファイル数が5以上◎ 推奨× 文脈浪費
「要約だけ欲しい」◎ 要約が本業△ メイン文脈に全文が残る
1ファイルを細かく編集したい× 編集権限ない◎ 推奨
改修実行の直前△ 要約は粗い◎ 正確な行番号が必要
並列に複数観点で調査◎ 観点別に複数起動× 逐次実行になる
もっと詳しく: Explore 子が苦手なこと

Explore は「読む専門」のため、編集は一切できません。また、返すのは要約テキスト1つなので、正確な行番号・完全な関数本体が欲しいときはメインで改めて Read する必要があります。

運用的な目安として、「調査 → 要約 → それを元にメインで該当箇所だけ Read」の2段構えが最も文脈効率が良いです。Explore は地図を描く役、Read は目的地でシャベルを入れる役。

Step 2 — 計画:Plan モードで手順に分解する

理解が取れたら、書き始める前に計画を作ります。Plan モード を使うと、Claude は「編集せずに計画だけ提示」して、ユーザが ExitPlanMode で OK を出すまで書き換えません。

Plan モードで起きる3つのこと
graph LR
  A["1. 計画提示"] --> B["2. ユーザが読む"]
  B --> C{"OK?"}
  C -->|"はい"| D["ExitPlanMode → 実装開始"]
  C -->|"いいえ"| E["計画を修正"]
  E --> B
合意が取れてから手が動くので、ロールバックの保険になる。

良い Plan の条件 — 4つの型

悪い Plan

曖昧 / 粒度バラバラ

  • 「fooBar を直す」
  • 「テストも更新」
  • 「PR を出す」
良い Plan

段階的 / 検証可能

  • (1) fooBar に失敗する新テストを追加 — red
  • (2) fooBar の分岐を extract — 既存テスト緑維持
  • (3) 新分岐を実装 — 新テスト緑
  • (4) 重複を削除 — 全テスト緑 + diff レビュー

良い Plan は「各ステップ完了時点でリポジトリがビルド可能・テスト通過」であること。途中でコケても前ステップに戻れる「階段状」の計画です。

今すぐ試す(2分)

Step 1 で取った要約を貼り付けて、こう頼んでみてください。

上の要約をもとに、fooBar() の改修プランを Plan モードで作って。
条件:
- 各ステップ完了時点でテストが全部緑であること
- 1ステップ = 1コミット想定
- 最後のステップに「diff 全体を人間がレビュー」を必ず入れる
- 破壊的な置き換えは最後。途中は追加のみ。

Claude から ExitPlanMode で計画が提示されたら、読んで納得してから Approve。ここで違和感があれば修正要求を1-2回繰り返します。

公式ドキュメント必読

Plan モードの挙動・ExitPlanMode の UI は公式ドキュメントに根幹仕様があります。特に「Plan 中に編集されないツール一覧」は実務で間違えると致命的なので必ず確認してください。Claude Code 公式 →

もっと詳しく: Plan を使わない方が良い場面

Plan モードは「大きな変更」向けのツール。typo修正、ログ1行追加、README の誤字など1ファイルの軽微な編集では、逆に冗長で時間が溶けます。

目安: 編集が3ファイル以上 or 100行以上 or テスト修正を伴う → Plan 使う。それ以下 → 直接実装で OK。

Step 3 — 実装:段階的コミットで壊さず進める

計画に合意したら、ようやく書き始めます。肝は 1ステップ1コミット。途中で失敗しても、直前の緑の状態に戻れる「セーブポイント」を積み上げる発想です。

1
ブランチ
feature/refactor-xxx を切る
2
TodoWrite
Plan の各ステップをToDoに
3
実装→テスト
1ステップずつ進めて緑を確認
4
commit
緑の瞬間にすぐ commit
5
diff レビュー
自分の目で git diff を確認

実装段階の変化 — 時間配分の理想

60分改修の時間配分(理想 vs 初心者)

初心者は 実装に40分 溶かして残り20分で事故対応。上位20%は 前半に投資し、実装自体は短く終わる。実装が速いのではなく、実装前に事故要因を潰している。

TodoWrite で進捗を見失わない

Plan の各ステップを TodoWrite に落とすと、Claude 自身が「今どのステップか」を見失わなくなります。Ch05 で詳しく扱いましたが、改修では 絶対に 最初に立てるクセを付けてください。

# 良い TodoWrite 例(Plan から自動展開できる)
[
  { content: "fooBar の失敗テスト追加", status: "in_progress" },
  { content: "分岐ロジックを extractFoo() に抽出", status: "pending" },
  { content: "新挙動を extractFoo() に実装", status: "pending" },
  { content: "重複コード削除 + 全テスト", status: "pending" },
  { content: "git diff を人間がレビュー", status: "pending" }
]
Tips

commit メッセージは「何を変えたか」ではなく 「なぜ変えたか」。Plan の各ステップ名がそのままコミットメッセージの「何」になるので、「なぜ」の1行を足すだけで良いコミットログになります。

Step 4 — 検証:テストと diff で「壊れてない」を担保する

最後のステップは検証。改修の成功条件は「機能が増えた/変わった」かつ「それ以外は何も変わってない」こと。後者が抜けがちです。

3層の検証

層やること見つかる事故頼み方
自動テスト既存 + 新規テスト全部走らせる仕様破壊・回帰バグ「npm test を走らせて、失敗があれば原因を3行で」
diff レビューgit diff main を人間が読む意図しない編集・余計な削除「git diff main...HEAD を機能ブロックごとに要約して」
手動動作確認実際に動かす(UIなら画面、CLIなら実行)テストで拾えない UX 退行(人間が自分でやる)
今すぐ試す(4分)

検証ターンで Claude にこう頼んでみてください(テストがあるリポで)。

以下の順で実行して結果を報告して:
1. 現在のブランチでテスト全部走らせる (例: npm test or pytest)
2. main との diff を gh pr diff 風に出して、変更を機能ブロック単位で要約
3. 「意図に沿っていない変更」が1つでもあれば、その diff 断片だけ引用して指摘
4. 結論を 3行で: テスト OK か / diff に想定外はないか / マージ可否の意見

ここで「結論3行」を必ず求めるのがコツ。Claude が自分の仕事を自分で採点するモードになります。

検証で拾える典型事故

42%
テスト改変
↓ assertion を緩めた
28%
隠れた削除
↓ 使用中コードの消去
18%
スコープ膨張
↓ 頼んでない所も直した
12%
副作用漏れ
↓ ログ/メトリクス欠落

※ 筆者が観測した Claude Code 改修の事故内訳(2026/1サンプル)

もっと詳しく: Claude に「テストを緩めさせない」交渉術

最大の罠は、Claude が「テストを通すためにテストを変える」こと。最初の依頼時点で明示的に禁止するのが鉄則です:

重要: テストを通すために assertion や期待値を変更しないで。
テストが落ちたら、まずコードを直す。どうしてもテストが間違っている
と判断した場合のみ、理由を1段落で説明してから変更許可を求めて。

これを CLAUDE.md に書いておくとプロジェクト全体で効きます(Ch06参照)。詳細は Ch11(テスト駆動)で扱います。

実リポハンズオン:4ステップを15分でなぞる

ここまで抽象論で説明してきました。次は実リポで1周回してみます。自分のリポジトリが無ければ、下のシェル1行で練習用のミニ Python CLI を作ってください。

# 練習用リポ作成(コピペで動く)
mkdir -p ~/tmp/refactor-demo && cd ~/tmp/refactor-demo
git init -q
cat > greet.py << 'PY'
def greet(name, lang="en"):
    if lang == "en":
        return "Hello, " + name + "!"
    elif lang == "ja":
        return "こんにちは、" + name + "さん!"
    else:
        return "Hi, " + name
PY
cat > test_greet.py << 'PY'
from greet import greet
def test_en():  assert greet("Ada") == "Hello, Ada!"
def test_ja():  assert greet("凛", "ja") == "こんにちは、凛さん!"
def test_unk(): assert greet("Bob", "fr") == "Hi, Bob"
PY
git add -A && git commit -q -m "init: greet.py"
echo "done. now: claude"

この greet() を「f-string で書き換え + 新言語 fr を追加 + 辞書ベースにリファクタ」するのが今回のお題。あなたは Claude Code にこう頼みます(各ステップで別プロンプトを投げる):

Step 1 (5分)
理解
Agent tool で Explore 起動。greet.py の責務 / 呼び出し元 / 既存テストの assertion / 暗黙仕様 を要約して
Step 2 (3分)
計画
Plan モードで、(1) fr テスト追加 (2) 辞書化 (3) 新言語対応 (4) diffレビュー、の階段型プランを提示して
Step 3 (5分)
実装
ExitPlanMode で承認 → TodoWrite に4項目登録 → 1ステップずつ実装& pytest → 緑のたびに git commit
Step 4 (2分)
検証
pytest 全緑確認 → git diff main..HEAD を要約 → 意図外変更があれば指摘 → 結論3行
同じ改修を、いきなり「辞書化して fr 足して」と頼むのと、4ステップに分けて頼むのとでやってみてください。差し戻しの回数、コミットの粒度、diff の読みやすさ——全部違うはずです。 — この差が 60分 × 週5日 × 50週 で効いてきます

アンチパターン4選 — これをやると壊れる

改修の危険な道と安全な道
graph TD
  A["改修を依頼"] --> B{"理解を省いたか?"}
  B -->|"はい"| C["表面的な修正"]
  C --> D["呼び出し元が壊れる"]
  D --> E["差し戻し / 事故"]
  B -->|"いいえ"| F["責務を3行でまとめる"]
  F --> G["Plan で手順化"]
  G --> H["段階コミット + テスト"]
  H --> I["安全に完了"]
分岐は1箇所。「理解を省いたか?」の Yes / No で運命が分かれる。
  1. 「このファイル全部読んで直して」 — メイン文脈がパンクする。Explore 子で要約だけ取る。
  2. Plan 無しで大規模変更 — ロールバックポイントが無く、途中で迷子に。
  3. 1コミットに全部詰める — 事故ったときに戻る場所がない。1ステップ1コミットを死守。
  4. テスト落ちたら assertion を緩める — 最悪。CLAUDE.md に禁止ルールを書く。
リマインダ

4ステップの中で一番カットしたくなるのが Step 1(理解)。でもここを15分ケチると、Step 3 で45分失う。「急がば理解」を合言葉に。

Exercise 1
入門

自分のリポで「3行責務まとめ」

手持ちのリポジトリから、最近触った関数/クラスを1つ選び、書き換える前にClaude Code に以下を頼んでください。

  1. Agent tool を subagent_type="Explore" で起動
  2. 対象の「責務を3行」「呼び出し元のファイルリスト」「既存テスト一覧」「暗黙仕様の推測」を要約で返させる
  3. 返ってきた要約を読み、自分が知っている情報と 食い違う箇所 が1つでもあれば、それがこれまで見落としてきた「地雷」
模範解答を見る

プロンプト例(そのままコピペ可):

Agent tool を subagent_type="Explore" で起動して、以下だけを要約で返して。
対象: src/billing/invoice.ts の Invoice クラス

返してほしい内容:
1. このクラスの責務を「何を」「何のために」で3行
2. new Invoice(...) / Invoice.xxx を呼んでいるファイルの一覧(パスだけ)
3. Invoice に対する既存テスト(describe/it 名だけ)
4. コメントには書かれていないが、コードを読むと守られていそうな
   暗黙の約束(null 許容、空入力時の挙動など)を2つ推測

出力はマークダウン見出し付きで簡潔に。ソース引用は最小限で。

注目ポイント: (4) の「暗黙仕様の推測」が最も価値が高い。ここに「え、そんな約束あったの?」が1つでもあれば、あなたはそれを知らずに改修しようとしていたということ。Explore を挟んだ意味が出ています。

Exercise 2
応用

4ステップを貫通:段階的な実リファクタ

Section 6 の練習リポ(または自分のリポ)で、以下をすべて4ステップで実施してください。

  1. feature ブランチを切る
  2. Step 1: Explore で現状把握 → 要約を貰う
  3. Step 2: Plan モードで階段型プラン(各ステップ終了時にテスト緑)
  4. Step 3: TodoWrite で進捗追跡 + 1ステップ1コミット
  5. Step 4: 全テスト緑確認 + git diff main..HEAD の要約 + 結論3行

終わった後、commit 数とプランの手順数が一致している か、assertion が1つも緩められていない かをセルフチェック。

模範解答を見る

Section 6 の greet.py を題材にした場合の流れ全体(ターン別):

# ターン1(理解)
Agent tool で Explore 起動: greet.py の責務 / 呼び出し元 / 既存テストの
assertion / 暗黙仕様 を要約して返して。

# ターン2(計画)
上の要約を踏まえ、Plan モードで以下の条件に従う計画を作って:
- (1) test_fr を追加(現状 red)
- (2) 言語→挨拶 の辞書化(既存テスト全緑のまま)
- (3) fr 対応を辞書経由で実装(test_fr を green に)
- (4) 冗長コード削除 + 全テスト緑 + diff 人間レビュー
各ステップ完了時点で pytest が緑であること。

# ターン3(実装)
ExitPlanMode で承認。TodoWrite に4項目登録。
1ステップずつ実装 → pytest → 緑なら git commit -m "step N: xxx" を繰り返して。

# ターン4(検証)
pytest を全部走らせた結果 / git diff main..HEAD を機能単位で要約 /
テスト assertion が緩められた箇所の有無 / 結論3行(マージ可否の意見)
を報告して。

期待される出力: 4コミット、全テスト緑、diff には辞書化リファクタ+fr追加の2機能ブロックのみ、assertion は一切変更なし、結論は「マージ可、リスクなし」。

もし commit 数が1-2個に固まってたら、段階的コミットが身についていないサイン。次の改修から意識的に commit 単位を細かくしてください。
