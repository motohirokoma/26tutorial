コードレビュー依頼の作法

「観点」を指定して、浅いレビューから脱出する

所要時間: 45分 Phase: Core 難易度: 中級
この章のゴール

「レビューして」だけで止めるのをやめる章。セキュリティ・パフォーマンス・可読性・テスト・設計の 5観点 指定を身につけ、gh pr diff と組んで PR 全体 / 部分レビューを使い分けられるようにします。最初の5分で手を動かします。

バージョン注記

本章は 2026年1月時点 の Claude Code CLI (v2.x) と GitHub CLI (gh) v2.70+ の挙動で検証しています。フラグ名や出力形式は進化するため、挙動が違ったら Claude Code 公式 と gh CLI マニュアル を確認してください。

なぜ「観点」指定で劇的に変わるのか

「レビューして」だけの依頼は Claude にとって 無限の自由度。返ってくるのは「ネーミングが…」「コメント足したら…」という薄い一般論になりがち。逆に「セキュリティ観点だけで」と絞ると、該当分野の チェックリスト に沿って深く掘る挙動に切り替わります。

観点の有無で返ってくるレビューの質
観点を1つに絞るだけで、指摘の密度と深さが体感2-3倍になる。
ここがプロの境目

観点を指定してレビューを頼む。「レビューして」だけだと薄い指摘しか返らない。セキュリティ／パフォーマンス／可読性／テスト／設計——どれか1つ、または複数を明示的に渡すと、Claude は該当分野のチェックリストを走り切る。
試し方： 自分の直近PRに対して「セキュリティ観点だけでレビューして。入力検証・認可・秘密情報の3つを最優先で」と依頼してみる。

5つの観点 — 暗記する5枚のカード

まずは 観点(viewpoint) の全体像を押さえます。この5枚を頭に入れておくと、PRを見た瞬間「今回は設計と可読性」「これはN+1最優先」と即決できます。

5観点
覚えるべき切り口
全部の起点
2-3倍
指摘密度の向上
観点あり vs なし
45秒
1観点の追加コスト
再依頼で済む
観点主な問いClaudeが強い作業使う頻度
セキュリティ入力検証／認可／秘密情報／注入攻撃パターンマッチによる抜け発見PR毎
パフォーマンスN+1／アルゴリズム計算量／無駄な再計算ホットループ・クエリの指摘大きな変更時
可読性命名／関数長／ネスト／マジックナンバー機械的な基準で一貫適用PR毎
テスト境界値／エラーパス／モック妥当性抜けているケースの列挙テストを含むPR
設計責務分離／依存方向／公開APIの形類似コードとの比較新規モジュール時

観点別・コピペで動くプロンプト例

以下は claude CLI に直接投げて動く形です。対象ファイルは自分のリポで置き換えてください。

# 1. セキュリティ観点
claude "src/api/ 配下をセキュリティ観点だけでレビューして。
優先順位: (1)入力検証 (2)認可チェック (3)秘密情報の扱い。
各指摘は {ファイル:行} + 理由 + 修正提案の3点で返して。
該当なしの項目は『該当なし』と明記して。"
# 2. パフォーマンス観点
claude "src/services/order.ts をパフォーマンス観点でレビュー。
特に (a)DBクエリのN+1 (b)O(n^2) 以上のループ (c)同期I/Oの直列化 を探して。
『疑わしい』レベルでもいいから列挙して、計測方法も添えて。"
# 3. 可読性観点
claude "この diff を可読性だけで見て:
- 命名が意図を伝えているか
- 関数は1画面に収まるか（40行超を指摘）
- ネストが3段超えていないか
- マジックナンバーが残っていないか
機械的にチェックして、違反を全部出して。"
# 4. テスト観点
claude "このPRのテストを観点レビューして:
- 正常系 / 異常系 / 境界値の3分類でカバレッジを評価
- 抜けているケースを『シナリオ: 入力 → 期待動作』で5件以上提案
- モックが実装を固定しすぎていないか"
# 5. 設計観点
claude "新規モジュール src/payment/ を設計観点でレビュー:
- 責務分離（1モジュール1責務か）
- 依存方向（外→内、抽象→具象になっているか）
- 公開API（export している関数の粒度は適切か）
類似の src/user/ と比較して、一貫性の崩れも指摘して。"
もっと詳しく: 5観点を「一度に」頼むべきか？

結論: 一度に頼まない方が深くなる。観点を5つ束ねて一発で投げると、Claudeは平均的に各観点を薄く撫でる傾向があります。45分の章時間内なら、観点ごとに別プロンプトで3-5回呼び直す方が、累計トークンは増えても指摘の質が圧倒的に高い。

例外は「超小さなPR」(差分50行以下)。この場合は「セキュリティ・可読性・テストの3観点で」と束ねても質が落ちにくい。

また、Ch16(Subagents)で学ぶ Agent ツールを使えば、5観点を並列に走らせて最後に集約することも可能です。これが本章の先にある上位技。

PR全体レビュー vs 部分レビューの使い分け

1000行の巨大PR丸投げは最悪パターン。コンテキストが薄まり指摘密度が下がります。レビュー粒度 を意識的に切り替え、diff を分割して渡しましょう。

PRレビューの2段階フロー
graph LR
  A[PR到着] --> B{"差分行数?"}
  B -->|"< 200行"| C["全体レビュー 1回"]
  B -->|">= 200行"| D["全体 概要レビュー"]
  D --> E["重要ファイル抽出"]
  E --> F["ファイル単位 深掘り"]
  F --> G["観点別 個別レビュー"]
  C --> H["gh pr comment"]
  G --> H
まず全体で地形を把握 → 深掘りする場所を決める、が王道。

今すぐ試す：gh pr diff と組む3行コマンド

gh pr diff で PR の差分をテキストとして取り出し、Claude に渡します。自分のリポで番号を置き換えて実行してください。

# ① PR全体の概要レビュー (200行未満なら一発)
gh pr diff 123 | claude "この差分を可読性＋設計の2観点でレビュー。
大項目3つに絞って、各項目に代表箇所1つだけ挙げて。"

# ② 重要ファイルだけ部分レビュー
gh pr diff 123 -- src/auth/login.ts | claude "セキュリティ観点で精読して。
認可・入力検証・エラーハンドリングの3軸で指摘して。"

# ③ commit単位でレビュー
gh pr view 123 --json commits --jq '.commits[].oid' | \
  xargs -I{} sh -c 'git show {} | claude "このcommit単体を設計観点で。"'
公式ドキュメント必読

gh pr diff のパス指定・--name-only・--color=never などの組み合わせは、Claudeと組む時の生産性を桁で変えます。ここは公式を一度通読しておくと投資回収が早い。gh pr diff マニュアル →

粒度判定のスタッツ

全体レビュー向き

小さな一貫PR

  • 差分 200行以下
  • 1機能 / 1バグ修正
  • 同じレイヤに集中
  • テスト付き
部分レビュー必須

大きな混ざりPR

  • 差分 500行超
  • 複数機能・レイヤ横断
  • 自動生成ファイル混在
  • リファクタ＋機能追加

自コードを「批判的に」レビューさせる

Claudeにはデフォルトで 褒める圧 がかかっています。「LGTM！」で終わらせないために、批判モード に切り替える魔法の言葉を覚えましょう。

「このコードを厳しい先輩レビュアーとして見てください。褒めはいらない、問題点だけ列挙してください。妥協点も明示してください。指摘が5個未満なら『真面目に見ていない』と判断します。」 — 批判モード起動フレーズ

批判モードのチェックリスト

  • 役割を固定する — 「厳しい先輩」「セキュリティ監査人」「パフォーマンスエンジニア」など具体ロールを与える
  • 褒めを禁止する — 「良い点は書かなくていい」と明示するだけで出力の80%が問題点に変わる
  • 最低指摘数を指定する — 「最低5件」「10件探して」と数を渡すと、表面で止まらず深掘りする
  • トレードオフを語らせる — 「この設計で捨てている選択肢は？」と聞くと、妥協箇所が浮かぶ
  • シナリオ攻撃を依頼 — 「悪意のある入力で壊すシナリオを5つ挙げて」は特にセキュリティで強力

今すぐ試す：批判モード・フルテンプレ

claude "以下のコードを厳しい先輩レビュアーとして見てください。

【役割】10年目のシニアエンジニア、本番障害を何度も経験している
【観点】セキュリティ + パフォーマンス + エラーハンドリング
【ルール】
- 褒めは書かない
- 最低8件、問題点を列挙
- 各指摘に「最悪のシナリオ」を添える
- トレードオフで妥協した箇所も明示
- 深刻度 (Critical / High / Medium / Low) でタグ付け

【対象コード】
$(cat src/api/payment.ts)
"
Tip

自分が書いたコード（Claude に書かせたコードを含む）を投げる前に、「これはあなた自身が書いたものです」と伝えないのがコツ。Claudeは自分の出力を甘く評価する傾向がごくわずかにあります。「先輩エンジニアが書いた」「チームメイトが書いた」の体で投げると、より客観的に見てくれます。

指摘の深刻度と集計の流儀

指摘を受け取ったら、全部直す必要はない。深刻度で仕分けて、今回のPRで直すもの／別Issueにするもの／見送るものに分類します。Claudeにそのまま仕分けさせるのが一番早い。

深刻度レーダー(観点 × 指摘数)

直近PRの観点別レビュー結果(サンプル)

深刻度ドーナツ

指摘の深刻度配分(典型パターン)

claude "先ほどのレビュー結果を以下に仕分けて:

(A) このPRで必ず直す — Critical + High 全件
(B) 別Issueに切り出す — Medium で設計に関わるもの
(C) 今回は見送る — Low + 好み問題

各カテゴリごとに gh issue create コマンドも作って。

もっと詳しく: 「LGTM」で済ませていい時はあるか？

あります。以下の3条件すべて揃った時:

  1. 差分が 50行以下 かつ単純（typo修正、定数値の更新、ドキュメント追加など）
  2. テストが通っている（CIが緑）
  3. 本番影響度が低い（設定ファイルのコメント、READMEの誤字など）

これ以外では必ず最低1観点でClaudeを通す。5分の投資で1時間のトラブルを防げる。

よくある落とし穴と対処

症状原因対処
指摘が3件で止まる最低数の指定なし「最低8件」と明示
褒め言葉ばかり批判モード未起動役割＋「褒めない」宣言
同じ指摘が繰り返される文脈窓が飽和/compact か新セッション
抽象論だけ返る{ファイル:行} 指定忘れ「{file:line}形式で」と書く
diffが大きすぎて失敗PR全体を丸投げファイル単位に分割
実装と違う指摘旧リビジョンを参照gh pr diff で最新を渡す

レビューと修正を同じセッションで混ぜないのも予防策。「ついでに直して」と続けるとClaudeは指摘の検証を飛ばして書き換えに走り、レビューの独立性が失われます。(1)レビュー → (2)仕分け → (3)別セッションで修正 の3段階に分けるのが安全。(3) は Ch10 で深掘りします。

Exercise

Exercise 1
入門

セキュリティ観点だけで1PRを見る

自分の手元にある直近のPR(なければサンプルリポのgh pr listから1件選ぶ)に対して、セキュリティ観点のみでレビューさせてください。

  1. gh pr diff {番号} で差分を取得
  2. Claudeに「セキュリティ観点だけでレビュー。入力検証・認可・秘密情報の3軸で。各指摘は{ファイル:行}+理由+修正提案」と依頼
  3. 返ってきた指摘のうち、Critical 相当だと思うものを1つ抜き出して書き留める
模範解答を見る

以下のワンライナーが最短コース:

gh pr diff 42 | claude "この差分をセキュリティ観点だけでレビュー。
優先順位は (1)入力検証 (2)認可チェック (3)秘密情報の扱い。
各指摘は以下の形式で:
- {ファイル:行} Critical|High|Medium|Low
- 問題: ...
- 攻撃シナリオ: ...
- 修正案: ...

該当なしの観点は『該当なし』と明記。最低5件出して。"

返答例 (抜粋):

- src/api/login.ts:34 Critical
  問題: パスワードを平文のままログ出力している
  攻撃シナリオ: ログ収集基盤に侵入された攻撃者が全パスワードを取得
  修正案: logger.info の引数から password を除外、あるいは '[REDACTED]' 置換

実運用では、Criticalと判定された指摘だけはその日のうちに修正すること。別Issueに回すのは High 以下に留める。

Exercise 2
応用

自分のコードを3観点×批判モードで吊るし上げる

自分が最近書いた1ファイル(100-300行程度のもの)に対して、セキュリティ + パフォーマンス + 設計の3観点×批判モードで連続レビューを掛け、結果を Critical/High/Medium/Low で仕分けて gh issue create コマンドまで生成させてください。

  1. 観点ごとに別プロンプトでレビューを実行(3回)
  2. 最後に統合セッションで「3観点の指摘を深刻度でマージして仕分けして」と依頼
  3. Critical + High を gh issue create 用のコマンド列に変換させる
  4. 生成されたコマンドのうち1つを、ドライランで確認してから実際に実行してみる
模範解答を見る

3回のレビュー→統合→Issue化の流れを1セッションで回すサンプル:

# Step 1: 観点別レビューを3回
FILE=src/payment/charge.ts

claude "$(cat $FILE) を厳しい先輩としてセキュリティ観点だけで見て。最低5件、{ファイル:行}+深刻度+シナリオ+修正案で。"

claude "$(cat $FILE) をパフォーマンス観点だけで見て。N+1・計算量・同期I/Oの3軸で最低5件、深刻度タグ付きで。"

claude "$(cat $FILE) を設計観点だけで見て。責務分離・依存方向・公開APIの粒度の3軸で最低5件、深刻度タグ付きで。"

# Step 2: 統合仕分け
claude "上記3観点の指摘を統合して、Critical/High/Medium/Low で仕分けて。
重複する指摘は1件にマージ。最終出力は表形式で。"

# Step 3: Issue化コマンド生成
claude "Critical と High のみを対象に gh issue create コマンドを生成して。
タイトルは '[review] {深刻度}: {一行要約}'、body に 問題/シナリオ/修正案 を入れて。
各コマンドは echo で始まる dry-run バージョンも併記して。"

ポイント:

  • 観点を束ねず3回分けて呼ぶことで、各観点の掘り下げ密度を確保
  • 統合仕分けで重複マージをClaudeに任せる(人間でやると眠くなる作業)
  • Issue化コマンドは必ずdry-run版も併記させる。いきなり gh issue create を連発すると事故る

この流れを1Skill化しておくと、以後のPRレビューが10倍楽になります(Ch20でSkill化を学びます)。
