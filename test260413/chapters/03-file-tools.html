ファイル操作ツール4つ (+Glob/Grep)

Read / Write / Edit / Glob / Grep — 探してから読む、壊さず書き換える

所要時間: 60分 Phase: Basics 難易度: 初級
この章のゴール

Claude Code の日常使いで 9割を占める5つのツール ——Read / Write / Edit / Glob / Grep—— を、いつ・どの順で・どんなパラメータで使うかを体で覚える。60分で「探してから読む」の型が染み付くと、これ以降の章の指示が格段にスッと入ります。コピペで動く実プロンプトを 10 個以上載せています。読むより触る。

バージョン注記

本章の記述は 2026年1月時点 の Claude Code 挙動に基づきます。ツール名やパラメータ(offset/limit/replace_all 等)は公式で更新される可能性があるため、動作がおかしい時は 公式ドキュメント を確認してください。

5つのツール — 役割早見

まず全体像。ファイル系ツール は大きく「探す」(Glob/Grep)と「読む/書く/直す」(Read/Write/Edit)の2グループに分かれます。この分類が、章全体の基礎になります。

5ツールの守備範囲
左で場所を特定してから、右で操作する。この流れを崩すとコンテキストが溶ける。
ツール一言で主なパラメータ使うタイミング
Globファイル名パターンでリストpattern, path「○○ファイルが何個あるか」を知りたい時
Grep中身を正規表現で検索pattern, glob, output_mode「この文字列を含むファイルはどれ」を知りたい時
Read1ファイル読む(行番号付き)file_path, offset, limit場所を特定した後に該当1ファイルの中身を見る時
Edit既存ファイルを差分編集old_string, new_string, replace_all既存行を置き換える(読んだ後に使う)
Write新規作成 or 全書き換えfile_path, content新ファイル作成。既存は Edit 優先

使い分けフロー — 「探してから読む」が鉄則

初心者が最初にやらかすのは「とりあえず Read してください」と言って、数百行のファイルや 10 ファイルを丸ごと読ませてしまうパターン。これは コンテキストウィンドウ を食い潰す最悪の選択で、その後の会話の精度が一気に落ちます。正しい順序はこれ。

ファイル操作の正しい順序
graph TD
  Start([やりたいこと]) --> Q1{"ファイル名は既に分かっている?"}
  Q1 -- Yes --> Read["Read で1ファイル読む"]
  Q1 -- No --> Q2{"名前で絞れる?"}
  Q2 -- Yes --> Glob["Glob でパターン検索"]
  Q2 -- No --> Grep["Grep で中身検索"]
  Glob --> Pick["候補から1つ特定"]
  Grep --> Pick
  Pick --> Read
  Read --> Q3{"編集する?"}
  Q3 -- No --> Done([完了])
  Q3 -- "一部だけ" --> Edit["Edit で差分編集"]
  Q3 -- "全書き換え/新規" --> Write["Write"]
  Edit --> Done
  Write --> Done
Glob/Grep で場所を特定してから Read。Edit は Read の後でしか使えない(実装上の制約)。
ここがプロの境目

Grep/Glob で探してから Read。絶対に Read で全ファイル読ませない。コンテキスト浪費の最大の敵は「不要な全文読み込み」。探索は安く、読み込みは高い——この2つの違いを体感することで、長時間セッションでも精度が落ちない使い手になれる。
試し方: 手元のリポで このリポで 'TODO' を含むファイル一覧を Grep で出して、そのうち1つだけ Read してください。 と頼む。Claude が Grep → Read の順で動くか見る。

この「探す → 読む → 直す」の3ステップは、Ch10 (改修ワークフロー) や Ch12 (デバッグ) でも登場する基本骨格。今のうちに体に染み込ませましょう。

Glob / Grep — 探し方の型

Glob は Glob パターン でファイル名を絞る。Grep はファイルの中身を正規表現で検索する。両者は補完関係です。

Glob の実プロンプト例

# 最新のTSファイル全部見せて
src 配下の .ts ファイルを Glob で列挙してください。パターンは src/**/*.ts で。

# テストファイルだけ欲しい
リポジトリ全体から *.test.ts と *.spec.ts を Glob で出してください。

# 設定ファイルを探す
ルート直下の package.json / tsconfig*.json / .eslintrc* を Glob で一気に取得。

Grep の実プロンプト例

# TODO コメントの場所を全部出す
リポ全体から 'TODO|FIXME|HACK' を含む行を Grep してください。output_mode は content、-n で行番号付き。

# 特定関数の定義を探す
function calculateTotal が定義されている場所を Grep で。*.ts に絞ってください。

# import の依存関係を調べる
from ['"]lodash['"] を import している箇所を Grep で。output_mode は files_with_matches で充分です。
公式ドキュメント必読

Grep/Glob は内部で ripgrep を使っており、正規表現の記法と出力モードは公式に詳しく書かれています。特殊文字のエスケープを間違えると検索が 0 件になり、「ないんだな」と誤判断する事故につながるので必ずマスターしてください。Claude Code 公式 →

NG

Read から入る

  • 「src フォルダ全部読んで TODO 探して」
  • → 数千行を親コンテキストに展開
  • → その後の対話で肝心の情報を忘れる
OK

Grep から入る

  • 「src/ で TODO を Grep、hit した行だけ見せて」
  • → マッチ行 30 行くらいだけ返る
  • → そこから1ファイルだけ Read で掘る
もっと詳しく: Grep の output_mode 3モードの使い分け

Grep には出力モードが3つあります。用途ごとに選び分けるとコンテキスト消費が大きく変わる:

  • files_with_matches (デフォルト) — マッチしたファイルパスのみ返る。「このエラーメッセージはどのファイルで出る?」系の質問に最小コストで答えられる。
  • content — マッチ行そのものを返す。前後行も欲しい時は -A 3/-B 2/-C 2(After/Before/Context)を指定。
  • count — ファイルごとのマッチ数。「TODO が何個たまってる?」の棚卸しに便利。

迷ったら files_with_matches → content -C 2 → Read の3段階で深掘りするのが黄金パターン。

検索コスト比較(イメージ)
Glob はファイル名だけ返すので最安。Grep の files_with_matches もほぼ同等。Read で全文ロードすると桁違いに高い。
今すぐ試す(2分)

自分の作業リポジトリで Claude Code を開き、このまま打ってみてください:

このリポジトリで 'TODO' か 'FIXME' を含むファイルの一覧を Grep で出してください。
output_mode は files_with_matches、glob は '**/*.{js,ts,py,md}' に絞って。
その後、一番マッチ数が多そうなファイル1つだけを Read で見せてください。

Claude が Grep → 候補特定 → Read 1ファイル の順で動くのを確認。親コンテキストが軽いまま済む感触を体で覚えておきます。

Read — 大きなファイルは offset/limit で刻む

Read は指定ファイルを cat -n 形式(行番号付き) で返します。この行番号は後述の Edit で位置を特定する根拠になるので、Read せずに Edit はできないという制約があります。

基本の使い方

# ふつうに1ファイル読む
src/components/App.tsx を Read してください。

# 特定範囲だけ見たい(大きいファイル対策)
src/lib/huge-file.ts の 500 行目から 100 行だけ読んでください。offset=500, limit=100 で。
大きなファイルの扱い

デフォルトで Read は先頭 2000 行まで読みます。それを超える offset / limit を活用して部分読みするのがプロの動作。「全部は読まない」勇気を持ってください。

1
当たりをつける
Grep で行番号を取得
2
範囲を絞る
offset=該当行-20, limit=60
3
必要なら拡張
限定的に前後行を追加 Read
4
編集 / 完了
Edit に進むか報告する
# 実例: 5000行のログから該当箇所だけ見る
logs/server.log の 3200〜3250 行を Read してください。offset=3200, limit=50 で。

# 実例: テスト失敗箇所のスタックトレース付近を見たい
まず 'TypeError' を tests/ で Grep。ヒット行番号の ±15 行を Read してください。
もっと詳しく: PDF / 画像 / Jupyter notebook を Read する

Read は PDF・画像 (PNG/JPG)・Jupyter notebook (.ipynb) も受け付けます。ただし PDF は 10 ページを超えると pages パラメータ必須(例: pages: "1-5")。画像は Claude がマルチモーダルで直接「見る」ので、スクリーンショットを手渡しで相談するフローに便利です。notebook はコードセル + 出力 + マークダウンが全部返ってくるので、EDA の文脈共有が楽。

Edit — 差分編集と replace_all の落とし穴

Edit は old_string → new_string の1対1の置換。シンプルだけど、replace_all の扱いに罠があります。

基本ルール

  • Read が先 — Edit を使う前に同ファイルを Read しておかなければならない(行番号とファイル状態の整合性担保)。
  • old_string はユニークに — ファイル内で1箇所にしか一致しない文字列にする。複数ヒットするとエラーで止まる。
  • インデントを保つ — タブ/スペースを含めてそのままコピー。ずれると当たらない。
  • replace_all は変数名リネーム用 — 変数名・関数名など「全部変える」時のみ。UIテキストでは絶対使わない。

replace_all の罠 — Before / After

ケースBefore素朴な replace_all の結果意図した結果
変数名リネーム user → account const user = ... / コメント // fetch user data / user.name / 関数 parseUser 全部 account 化(コメントまで変わる、parseUser は parseAccount に) 変数のみ。関数名・コメントは残す
定数値 100 → 200 timeout: 100 / limit: 100 / 1000 1000 の中の 100 まで書き換わる(部分一致の事故) timeout の行だけ
コメント文「TODO:」を一括削除 コード + 自然文 自然文の TODO まで消える可能性 コメント行のみ
落とし穴 — replace_all の3大事故

(1) 部分一致事故: 100 で一括置換すると 1000・100% まで侵食。
(2) コメント侵食事故: 変数名リネームがコメント内の自然文や JSDoc まで書き換える。
(3) テスト壊し事故: テストファイル内の期待値 (assertion 文字列) まで変わってテストがグリーンになる錯覚。
対策: 事前に Grep でヒット件数と文脈を確認 → 必要なら old_string に周辺行も含めてユニークな文脈を作る → 疑わしい時は replace_all を使わず1箇所ずつ Edit する。

実プロンプト例

# 安全な1箇所置換
src/config.ts の timeout: 100 という1行を timeout: 200 に Edit してください。
old_string は周辺2行を含めて一意に。

# 安全な変数名リネーム(事前確認あり)
まず Grep で 'legacyUser' のヒット件数と出現場所を出してください。
それが全部「変数としての使用」である事を確認してから、Edit に replace_all=true で一括リネーム。
コメント内の 'legacyUser' があれば除外対象として報告を。

# 危険な依頼(NG例)
'user' を 'account' に replace_all で置換して。
→ NG: user は部分一致の宝庫。必ず Grep で文脈確認してから。
今すぐ試す(3分)

適当なリポで、自分のコード中の変数名を1つリネームさせてみてください。ただし順序を以下で指定:

(1) まず Grep で対象文字列のヒット件数を出す
(2) 件数と文脈を自分で確認
(3) OKなら Edit (replace_all=true) で一括置換
(4) 最後に Grep で「もう残っていない」事を確認

この 「Grep → 確認 → Edit → Grep 確認」 の 4 ステップが、replace_all 事故を根絶する唯一の型です。

Write — 新規 OK、既存ファイルは Edit 優先

Write は「指定パスに全文を書き込む」ツール。既存ファイルを指定するとそのまま上書きされます。だから鉄則は:

既存ファイルの修正は Edit、新規ファイルの作成は Write。 迷ったら Edit を優先する。

使い分けの判断

状況使うツール理由
新しい設定ファイルを置くWrite存在しないから差分編集できない
README に1段落追記Edit既存部分を壊さずピンポイント変更
大規模リファクタで関数全書き換えEdit(大きな old/new で)意図しない周辺破壊を避ける
生成スクリプトの出力ファイル更新Write毎回全文生成なので Edit 不要
テンプレートから新ファイル1個作成Writeそもそも元がない
Write を使う前のお作法

既存パスに対して Write を使うとツール側が「先に Read していないと拒否」してくるケースがあります(上書き事故防止のガード)。新規以外では、まず Read で現状確認 → 全書き換えの必然性を確認 → それでも Write、という流れが安全。

もっと詳しく: 「ドキュメントは勝手に作らない」原則

Claude Code のデフォルト動作として、ユーザが明示的に頼まない限り README や summary の .md ファイルを勝手に作らない、という方針が推奨されます。Write で新規 .md ファイルを生成する前に「これは本当に必要か」を1拍考える癖を。自分のリポに report.md・summary.md が散らばっていたら、この原則を伝えきれていないサインです(Ch06 の CLAUDE.md で明文化可能)。

統合ハンズオン — 5ツールの連携

実務では1タスクで5ツール全部を使うことが珍しくありません。典型的な「設定値を変えてテスト追加」の流れをトレースします。

Step 1 · Glob
関連ファイルを特定
config/**/*.ts を Glob。3 ファイルヒット。
Step 2 · Grep
該当定数の使用箇所を調査
MAX_RETRY を Grep (content, -C 2)。5 箇所で使用。
Step 3 · Read
定義元を 1 ファイルだけ読む
config/retry.ts を Read(offset/limit なし、80 行程度なので全読みOK)。
Step 4 · Edit
値を変更(周辺文脈付き old_string で)
export const MAX_RETRY = 3; → export const MAX_RETRY = 5;。replace_all なし(定数定義は1箇所のみなので不要)。
Step 5 · Write
新規テストファイル作成
tests/retry.test.ts を Write で新規作成。境界値テストを含む。
5
使ったツール種別
↑ 全ツール活用
~3%
消費コンテキスト(リポ全体比)
↑ 効率的
0
意図しない破壊
↑ 型通りの安全運用
# このタスクを Claude に頼む実プロンプト例
config/ 配下から MAX_RETRY という定数を使っている場所を確認したい。

手順:
1. config/**/*.ts を Glob で列挙
2. その中で 'MAX_RETRY' を Grep (content, -C 2)
3. 定義元 (多分 config/retry.ts) を Read
4. 値を 3 から 5 に Edit(old_string は2〜3行含めて一意に)
5. tests/retry.test.ts を Write で新規作成。境界値(0, 1, 5, 6)をカバー

各ステップの結果を都度見せてから次に進んでください。

失敗パターン 5つ — 症状と処方箋

症状原因処方箋
Claude が話してるうちに過去の文脈を忘れる Read で全ファイル読ませすぎ Grep から入る習慣。/compact で途中圧縮(Ch14)
Edit がエラーで止まる old_string が複数ヒット or 不一致 周辺2〜3行を含めて一意に。インデントをそのまま貼る
replace_all 後にテストが謎の緑色 テスト内期待値まで一括置換 事前 Grep で影響範囲確認。テストディレクトリはglobで除外
Grep でヒットしない 正規表現の特殊文字未エスケープ、. の扱い リテラル検索は \. \( 等エスケープ。まず簡易パターンで試す
Write で既存ファイルが消えた 新規のつもりが既存パスを指定 Write の前に ls or Glob で存在確認。既存なら Edit に切替
復旧の基本

Claude Code での操作は Git と組み合わせるとロールバックが簡単。重要な変更前は必ず git status と git diff を挟み、replace_all のような広範囲変更はこまめなコミットで守ります(Ch07 で詳しく)。

入門

自リポで「探してから読む」を実演する

あなたの手元のリポジトリ(どんな言語でも可)で、以下を Claude Code にやらせてください。Read を使わず Grep から始めることが条件。

  1. リポジトリ全体から TODO または FIXME を含む行を Grep(output_mode=content, -n)
  2. ヒット件数を集計してもらう
  3. 最も多いファイル1つだけを、該当行 ±10 行の範囲で Read(offset/limit 活用)
  4. 内容を3行で要約してもらう
模範解答を見る

Claude Code に送る実プロンプト:

このリポジトリで TODO と FIXME の棚卸しをしたいです。手順:

1. Grep で 'TODO|FIXME' を検索。output_mode=content、-n で行番号付き、
   glob は '**/*.{js,ts,py,md,go}' に絞ってください。

2. ファイルごとのヒット件数を table で報告してください(多い順)。

3. 一番ヒット数が多いファイルを特定したら、そのファイル全体は読まず、
   最初のヒット行を中心に offset=(ヒット行-10), limit=20 で Read してください。

4. 読んだ範囲の内容を3行で要約してください。

途中経過を逐次見せてから進めてください。

ポイント:

  • glob で対象拡張子を絞ると検索高速化&ノイズ削減(node_modules/.git はripgrepが自動除外)
  • -n(line number)を付けさせると、後の Read の offset 指定に直結
  • 「全体は読まず」と明示しないと、つい Read で全文を取りに行きがち
  • 「逐次見せて」と書くと、各ステップで止まって確認してくれる
Exercise 2
応用

安全な変数名リネームフローを設計する

自リポの適当な変数名(2〜3文字のものは NG、部分一致事故が起きやすいので 6 文字以上推奨)を別名にリネームするタスクを、replace_all 事故を起こさない 4 ステップで Claude に実行させてください。Grep の出力モード選択を必ず使い分けること。

  1. ヒット件数と出現ファイルをまず確認する(何モード?)
  2. 各出現箇所が「変数としての使用」か「コメント/文字列リテラル」かを確認する(何モード?)
  3. replace_all で一括置換するか、Edit を複数回に分けるかを判断する
  4. 置換後の残存チェックを行う
模範解答を見る

手順とモード選択:

(1) Grep で対象名を output_mode=count で検索
    → ファイルごとのヒット数が分かる(全体規模を把握)

(2) 同じ対象名を output_mode=content, -C 2 で検索
    → 前後2行の文脈付きで、変数か文字列リテラルかを目視判断できる

(3-A) 全箇所が「変数としての使用」で、テストや自然文に混ざっていない場合:
      → Edit を replace_all=true で一発。old_string は対象名そのもの

(3-B) コメントや文字列リテラルに混在している場合:
      → Edit を複数回に分ける。各回で old_string に周辺2〜3行の文脈を含め、
         ファイル内でユニークに特定できる形にする
      → テストファイルや docs/ 配下は意図的に除外

(4) 置換後、もう一度 Grep (count) で対象名を検索 → 0件になっているか確認
    もし想定より残っていたら、残った箇所を個別に Read して意図を確認

この型は Ch10(改修ワークフロー) や Ch22(失敗パターン10) にも繋がる基本骨格です。「変更前に測る・変更後に測る」の癖は、Claude Code のパワーユーザー全員が持っています。
