Project C：ゼロからWebアプリ→デプロイ

90分で「設計 → 実装 → 公開」を一気通貫。URL を誰かに送れる状態で終わる。

所要時間: 90min Phase: Projects 難易度: 応用
2026年1月時点の情報

本章は 2026年1月時点の Claude Code / GitHub Pages / 標準ブラウザ挙動に基づきます。GitHub Pages の UI や Actions 連携仕様は今後変わる可能性があるので、最終デプロイ時は GitHub Pages 公式 の最新手順を確認してください。

章の地図：90分の全体像

このチャプターは「読む章」ではなく「手を止めず90分走り切る章」です。終わった瞬間、https://あなたの名前.github.io/... という URL が手元にあります。まずは地図を頭に入れましょう。

90分の内訳と狙い
設計20分 → 実装50分 → 公開20分。配分を守るのが最大のコツ。

お題は 静的Webアプリ。以下2つからどちらかを選びましょう。迷ったら家計簿を推奨します(身近でUI設計しやすい)。

Option A

家計簿記録アプリ

  • 日付・カテゴリ・金額を入力
  • 合計を表示
  • localStorage で永続化
  • 月別の円グラフ(任意)
Option B

議事録整形ツール

  • 貼り付けたテキストを Markdown 整形
  • TODO / 決定事項を抽出表示
  • ボタン1発でコピー
  • 最新の整形結果を localStorage に保存
ここがプロの境目

「小さく作って公開する」を90分で体験する。機能が少なくても、公開経験の有無が次回以降の態度を根本から変える。動いてるものを世界に晒す緊張感は、ローカルで完成させた満足感とは別物。
試し方： 90分後、完成した URL を家族か同僚1人に LINE/Slack で送る(感想は問わない、URLを送ることが目的)。

Part 1：設計(20分)

「いきなり手を動かす」は禁止。90分で終わらない最大の理由は、作りながら仕様が揺れることです。紙1枚(もしくはテキストファイル1つ)で要件を固めてから実装に入ります。

1
要件定義(5分)
誰が何をしたいか1文で書く
2
UI設計(8分)
画面要素を箇条書き + 手描きワイヤー
3
技術選定(3分)
HTML/CSS/JSだけ。フレームワーク禁止
4
リポジトリ準備(4分)
GitHub に空リポジトリを作る

1-1. 要件定義:MVP は削ぎ落とす

最初の罠が MVP を欲張ること。90分でやるなら、機能は3つまで。以下のテンプレで要件を1ファイルに書きます。

# 家計簿アプリ (MVP)

## ゴール
日々の支出を1画面で入力・閲覧できる。閉じても消えない。

## 利用者
自分。スマホ/PC 両方で使いたい。

## 必須機能 (3つだけ)
1. 入力: 日付・カテゴリ・金額を追加
2. 一覧: 追加したものをリスト表示、合計も表示
3. 削除: 行ごとに削除ボタン

## 非機能
- localStorage に保存(タブ閉じても残る)
- モバイルで縦持ちで使える幅

## やらないこと (今回)
- ログイン / 複数デバイス同期
- 集計グラフ(余裕あれば追加)
- 編集(削除→再入力で済む)
Tip

「やらないこと」を明示するのが超重要。Claude Code は指示されれば何でも作るので、境界線を先に引くのが90分で終わらせる秘訣です。

1-2. UI設計:ワイヤーを Claude に清書させる

手描きで十分ですが、紙がなければ Claude に ASCII でワイヤーを生成させましょう。これで Part 2 の実装プロンプトが具体的になります。

ASCII artで描いて。ヘッダー / 入力フォーム / 一覧 / 合計 の
4ブロックで。スマホ縦持ちを想定、縦1カラム。

1-3. 技術選定:今回は意図的に「貧しく」作る

今回の技術スタック(あえて最小)
90分で公開まで行くなら、ビルド工程が要らない構成が最強。React は次回。
公式ドキュメント必読

GitHub Pages は静的ファイルしかホストしない。ビルド工程のあるフレームワークを使うなら Actions か別サービス(Vercel/Netlify)が必要です。今回はまず「そのまま開ける構成」で勝ち切る。GitHub Pages 公式 About →

1-4. リポジトリ準備

GitHub で新規リポジトリを作ります(名前は mk-kakeibo など好きに)。Public で OK、README.md は 後で。

# ローカルで
mkdir mk-kakeibo && cd mk-kakeibo
git init
git branch -M main

# GitHubで作ったリポジトリをリモートに
git remote add origin git@github.com:<あなたのGitHub名>/mk-kakeibo.git
もっと詳しく: なぜ今 git init するのか

Claude Code に実装を頼む「前」にリポジトリを作っておくのが Ch07 で学んだコツ。小さく commit を重ねられる土台を先に作ることで、Part 2 の実装中に「さっきのが動いてたので戻したい」という事態にも対応できる。Claude に git log --oneline させながら進めると精度が上がる。

逆に、動くものができた後で git init すると、commit 履歴が1行になってしまう。ロールバックの保険が効かない。

Part 2：実装(50分)

いよいよ手を動かすフェーズ。Claude Code に対する頼み方は Ch08 の「新規作成型」プロンプトそのまま。最小動作 → 見た目 → 永続化 の順で3段階に分けます。

実装フローとリアルタイム判断
graph TD
  A([Part 2 スタート]) --> S1["Step 1: 骨組み生成 15分"]

2-1. Step 1:骨組みを一気に生成(15分)

ここで Ch08 で学んだ3点セット(具体性・制約・成功条件)をフル活用。要件.md を添えて、Claude Code に以下のような実プロンプトを投げます。

# プロンプト例 (骨組み生成)
家計簿Webアプリの MVP を作って。以下の3ファイルだけ生成、それ以外作らない。

## 出力
- index.html: 1画面構成。form要素で日付/カテゴリ/金額入力
- style.css: 基本スタイルのみ(後で整える)
- app.js: 追加・削除・合計計算・localStorage保存/復元

## 制約
- フレームワーク禁止 (vanilla JS / ES2020)
- 外部CDNは使わない
- ファイルは3つだけ、index.html からの相対パスで読み込み
- localStorage のキーは "mk-kakeibo-v1"
- カテゴリは「食費/交通/日用品/交際/その他」の5択

## 成功条件
1. open index.html で真っ白な画面にならず、入力フォームが見える
2. 追加ボタンで一覧に反映される
3. 再読み込みしてもデータが残る
4. ブラウザのコンソールにエラーが出ない

TodoWrite で3ファイルを順に作り、完成したら ls -la で確認。
Tip: TodoWrite を強制する一言

「TodoWrite で〜」と書くだけで、Claude が自分で進捗管理するようになる。Ch05 の学び。90分中に何度も軌道修正するなら必須の1行です。

今すぐ試す:骨組みが動いたか5秒チェック

Claude の生成が終わったら、プロンプトを返すのではなく、ブラウザで index.html を開く。動かなければ Ch12 のバグ報告3点セットで再依頼します。

# macOS
open index.html

# もしくはシンプルな静的サーバで
python3 -m http.server 8000
# → http://localhost:8000 を開く
もっと詳しく: file:// で開く vs ローカルサーバで開く

open index.html は file:// プロトコルでブラウザが開く。今回のように localStorage だけで完結する構成なら問題なく動くが、fetch() で JSON を読むような拡張をすると CORS エラーが出る。その場合は python3 -m http.server か Node の npx serve で簡易サーバを立てる。

Part 3 の GitHub Pages は普通の HTTPS サーバなので、本番環境では file:// 問題は起きない。ローカルで怪しい動きをしても、Pages 公開後に直ることもある。

2-2. Step 2:見た目を整える(15分)

骨組みが動いたら次は見た目。テーマを先に1文で指定するのが Ch08 で学んだ型。曖昧に「綺麗にして」と頼むと、Claude は一般的すぎる見た目を返します。

# プロンプト例 (スタイル調整)
style.css を以下のテーマで書き直して。

## テーマ
ライト系ミニマル。アクセント色は #D97757 (やや橙)。背景 #fafafa。
カードは白 + zinc-200 ボーダー。角丸10px。

## 要件
- モバイル縦持ち(375px幅)でも横スクロールしない
- 入力フォームの各 input は幅100%、上下マージン12px
- 追加ボタンは アクセント色背景 + 白文字
- 一覧の各行は白カード、hover で薄グレー背景
- 削除ボタン [×] は 薄赤

## 成功条件
- iPhone Safari でも崩れない(viewport meta 確認)
- 合計の数字だけ font-size 24px 太字
ここがプロの境目

「動いたらすぐ git commit」を習慣化する。1機能=1コミット。理由:Step 3 でスタイルを壊した時、戻れる地点があるだけで心理的負荷が激減する。
試し方： Step 1 が動いた瞬間、Claude に git add . && git commit -m "feat: MVP skeleton works" を実行させる。

2-3. Step 3:永続化と仕上げ(15分)

localStorage 保存は Step 1 でほぼ組み込まれているはず。ここでは境界値のバグ潰しに集中します。以下は Claude に渡す「自己レビュー」プロンプト。Ch09 の流用。

# プロンプト例 (自己レビュー)
app.js を以下の5観点でレビューして、問題があれば修正PRを作って。
コードは改変せず、diff で提示してから apply で分けて。

1. 入力バリデーション (金額が空/負数/文字の時)
2. localStorage の JSON パース失敗時のフォールバック
3. XSS (カテゴリや金額に HTMLタグを入れた時)
4. 削除時の index ズレ (filter vs splice)
5. 合計計算の浮動小数点誤差

## 成功条件
- 各観点で「問題あり/なし」を明示
- 修正が必要なら最小 diff で示す
- 5観点すべて対処済みを確認後、git commit

2-4. 実装中のリアルタイム判断:詰まった時の逃げ道

症状原因候補逃げ道プロンプト
画面が真っ白 JS エラーで描画停止 / script タグの順序 「コンソールのエラーを貼るので原因仮説3つ出して」
動くけど汚い CSS指定が散漫 「CSS変数で色を4つに絞ってリファクタ」
localStorageが消える キー名typo / JSON.stringify漏れ 「現在のキー名と保存/読込のフローを図にして」
スマホで崩れる viewport meta / 固定幅 「375px幅でのみ崩れ。デベロッパーツールで再現手順つけるので直して」
90分に間に合わない 機能追加の欲張り 「要件.md の"やらないこと"に追記して、現状で commit してPart 3へ」
時間マネジメント

Part 2 開始 + 50分が経過したら、動いている状態で機能追加はやめて Part 3 へ。公開できない機能ゴージャスな実装より、公開済みのシンプル実装の方が学びが大きい。

実装時間のガイド(目安)

3-3. 動作確認

# 発行された URL を開く
open https://<あなたのGitHub名>.github.io/mk-kakeibo/

# 確認ポイント
# - トップが真っ白じゃないか
# - 追加→リロード→残ってるか
# - スマホから同じURLを開けるか (iPhoneで試す)
# - 初回反映は1-2分遅れることがある (ハードリロード: Cmd+Shift+R)
もっと詳しく: URL がトップで 404 になる時

よくある原因3つ:

  1. index.html がリポジトリのルートにない — app/index.html の位置だと見えない。ルート直下に置く
  2. Branch 選択が gh-pages になっている — main にしたのに有効化されない時はブランチ確認
  3. リポジトリが Private で Pages が有効化できない — Free プランでは Public のみ

それでも治らない時は Actions タブ → Pages の deploy ログを確認すると原因が書いてある。

3-4. 共有:これで終わり、ではなく、ここが本番

URL が動いたら、実際に誰かに送る。このステップを飛ばすと、このプロジェクトは単なる練習で終わる。送った瞬間に「人に見られてる」緊張感が発生し、次回以降の設計・実装の粒度が根本から変わります。

コードは公開されて、はじめて存在する。ローカルに置いたままのコードは、無いのと同じ。 — 公開を一度も体験していない人だけが「まだ完璧じゃない」と言う

発展課題(任意)

90分で公開まで辿り着けた人向け。どちらも「動いてる URL に機能を足す」系の課題です。

発展1: Firebase 連携でクラウド保存化

localStorage は端末1つに閉じるので、PC と スマホで同じデータを見られない。Firebase Firestore に繋げて複数端末共有化すると、一気にアプリの価値が上がる。

  1. Firebase Console で新規プロジェクトを作る(無料枠で十分)
  2. Web アプリを追加して firebaseConfig を取得、firebase-config.js に貼る
  3. Authentication で Google Sign-In を有効化
  4. Firestore を作成、ルールは users/{uid} の分離パスにする
  5. Claude に以下のプロンプトを投げる:
# プロンプト例 (Firebase連携)
既存の家計簿アプリを Firebase に繋げて。localStorage 保存は
バックアップとして残す(オフライン時も動くように)。

## 追加要件
- Google Sign-In ボタンをヘッダーに追加
- ログイン後は Firestore users/{uid}/entries に保存
- ログアウト時は localStorage だけで動く
- 既存データの移行は「ログイン時に1回だけアップロード」

## 成功条件
- 2台の端末で同じアカウントでログインして同期される
- firestore.rules を .rules ファイルに出力して提示

初回の実装は 60-90分を見ておく。Ch17(MCP)の Firebase MCP を使うとさらに短縮。

発展2: 段階的PRで機能追加する習慣

ここまで main 直接 commit だったと思いますが、PR ベース開発に切り替えると「自分のコードを自分でレビューする」体験ができる。Ch07/Ch09 を総動員する演習。

  1. git switch -c feat/monthly-chart で月別円グラフ機能のブランチを切る
  2. Claude Code に「月別集計を Chart.js で円グラフ表示。既存機能は壊さない」と依頼
  3. gh pr create で PR 作成(Claude に依頼可)
  4. Claude に 自分の PR を5観点でレビューさせる: 破壊変更 / 可読性 / パフォーマンス / XSS / テスタビリティ
  5. 指摘があれば修正 commit を追加、なければ merge
  6. 1-2分後、Pages に反映されることを確認

この流れを2-3回繰り返すと、仕事のWeb開発のリズムが掴めます。

発展3: Lighthouse スコアで品質可視化
