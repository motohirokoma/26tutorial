Project A：業務自動化ツールを作る

日次レポート生成CLIを90分で0→完成。Ch05/08/07/11/17/18を一気に統合する

所要時間: 90分 Phase: Projects 難易度: 応用
この章のゴール

この章は 読む章ではなく作る章 です。お題は「日次レポート生成CLI」——GitHubの Issue一覧 を取得し、ラベル別に整形して daily-report-YYYY-MM-DD.md を吐く小さなツールを、90分で0から完成+自動化まで持っていきます。途中で Ch05(TodoWrite) / Ch07(Git) / Ch08(プロンプト設計) / Ch11(TDD) / Ch17(MCP) / Ch18(Hooks) を 一気に統合。この章が終わると、自分のチームの反復業務を1つ自動化できる状態になります。

バージョン注記

本章の記述は 2026年1月時点 の Claude Code(Agent SDK/CLI)挙動に基づきます。gh CLI / GitHub MCP / Hooks の settings.json スキーマは進化します。コマンドが通らない時は 公式ドキュメント と gh CLI manual を確認してください。

ここがプロの境目

毎日の反復業務は、90分の初期投資で年間数百時間取り戻せる。1日10分の手作業×250営業日 = 年41時間。それを「書く・捨てる・自動化する」サイクルに乗せるのが、Claude Codeの本当の元を取る使い方です。
試し方： この章の完成物を Slack の #dev-tools に流し、「使いたい人いる?」と聞いてみる。1人でも乗ったら社内Skill化の合図。

完成物のアーキテクチャ
Hookで毎朝起動 → GitHub APIからIssue取得 → 整形して .md 出力。発展課題でSlack通知を足す。

Part 1 — 設計(15〜20分)

「ツールを作る」と聞くと、いきなり mkdir と npm init から入りたくなりますが、90分でしっかり完成させる鍵は 最初の15分を設計に使う こと。ここでは3ステップだけやります: 要件整理 → プロンプト設計 → TodoWrite。

1-1. 要件を「3つの箱」で整理する

要件を無限に膨らませないための実務的な型が、I/P/O フレーム です。今回の「日次レポートCLI」を当てはめるとこうなります。

箱中身決めること
InputGitHub Issue(open状態、直近24h更新)repo指定方法 / 日付境界
Processラベルで分類 → 整形対象ラベル / 並び順
Outputout/daily-report-YYYY-MM-DD.mdファイル名規則 / セクション順

この3箱だけ確定すれば、あとは Claude に丸投げしても軸がぶれません。「入力形式も出力形式も決まっている」状態は、AIコーディングの成功率を体感3倍にします。

1-2. 初手プロンプトを設計する(Ch08の型を使う)

Ch08で学んだ 新規作成プロンプトの型 に当てはめます。何を成功と呼ぶか(done criteria) を必ず書くのがコツ。

今すぐ試す: 初手プロンプト

以下をそのままコピペして、あなたのローカルで claude に流してみてください。

あなたはNode.js/TypeScriptに詳しいシニアエンジニアです。
「日次レポート生成CLI」を新規作成します。

# 目的
GitHubリポジトリのIssue(open, 直近24h更新)を取得し、
ラベル別に整形して out/daily-report-YYYY-MM-DD.md を出力する。

# 制約
- ランタイム: Node.js 20+ / TypeScript / tsx 実行
- Issue取得は gh CLI をchild_processで叩く方式(MCP版は後で差し替える)
- 対象ラベル: bug / feature / question / その他
- 出力ファイルは out/ 配下。既存なら上書き
- テストは vitest で書く(TDDで進める)

# 成功条件(done criteria)
1. `npm test` が全通る
2. `npx tsx src/index.ts --repo owner/name` で out/*.md が生成される
3. Markdown 内にラベル別4セクション + 件数サマリが含まれる
4. README.md に使い方が3行で書いてある

# 進め方
まず TodoWrite で作業計画を立ててください。
最初のステップは「package.json + vitest のセットアップ + 最初の失敗するテスト1本」です。
いきなり実装しないでください。

このプロンプトの急所は4つ:

  • 成功条件を4つ明記 — done criteria があると、Claude が勝手に機能を追加し続けるのを止められる
  • 段階の指示 — 「いきなり実装しない」「最初はテスト1本」と書くことで Ch11 の TDD モードに入らせる
  • 差し替え前提の記述 — 「MCP版は後で差し替える」で、最初はgh CLIの簡易実装、後で Ch17 の MCP に置き換える流れを設計段階で確保
  • TodoWrite 指定 — Ch05 の原則「3ステップ以上は TodoWrite」をここで発動させる

1-3. TodoWrite に投げ込ませる(Ch05)

初手プロンプトを投げると、Claude が最初に TodoWrite を叩いてくれる想定です。そこで出てくる Todo の典型例がこちら。

もっと詳しく: I/P/Oに When/Who/Why を足す

余裕があれば I/P/O に When(毎営業日08:30)/Who(チーム6名)/Why(朝会で3分把握) を足します。Whyが決まると、Markdownのセクション順が自然に決まる(「今日やるもの」先頭、「完了済み」は後ろ)。要件は README.md 冒頭6行に凝縮しておくと、将来のメンテ時に「何のために作ったか」が失われません。

2-2. Red → Green: groupByLabel のTDDサイクル(Ch11)

ここが一番「AIに頼んで差が出る」局面。テストを先に書かせて、実装を後で書かせる。Claudeに依頼する時のプロンプトはこうです。

依頼プロンプト(Red フェーズ) 
次のシグネチャで tests/groupByLabel.test.ts を書いてください。
実装はまだ書かないでください。テストが失敗することを確認するだけです。

型:
  type Issue = { number: number; title: string; labels: string[]; url: string };
  function groupByLabel(issues: Issue[]): {
    bug: Issue[]; feature: Issue[]; question: Issue[]; other: Issue[];
  };

テストケース(最低4つ):
1. 空配列 → 全カテゴリ空配列
2. labels=['bug'] のIssue → bug に入る
3. 複数ラベルで bug と feature 両方あるIssue → bug 優先(先に書かれた方)
4. 未知のラベルのみ → other に入る

注意: 後でテストを通すためにassertionを緩めないでください。
    期待値は今ここで確定します。

テストが書けたら npm test で 必ず赤(失敗)を確認 してから実装に進むのが鉄則。Claudeに「テストが落ちているのを確認できましたか? 出力を貼ってください」と明示的に聞くと、サボられません。

アンチパターン

「一気に書いて」

  • テストと実装を同時に書かせる
  • 結果: assertionが実装に合わせて書かれて、実はバグが通る
  • TDDの Red-Green-Refactor の意味が消える
推奨

「Red確認 → 実装」

  • まずテストだけ書かせて npm test で失敗を目視
  • その後「Greenにしてください、テストは触らないで」と依頼
  • テストを緩めようとしたら即ストップ

参考: groupByLabel の最終的な実装例はこんなサイズに収まります(20行弱)。

// src/groupByLabel.ts
import type { Issue } from "./types.js";

const PRIMARY = ["bug", "feature", "question"] as const;
type Bucket = typeof PRIMARY[number] | "other";

export function groupByLabel(issues: Issue[]) {
  const out: Record<Bucket, Issue[]> = { bug: [], feature: [], question: [], other: [] };
  for (const iss of issues) {
    const hit = PRIMARY.find(p => iss.labels.includes(p));
    (hit ? out[hit] : out.other).push(iss);
  }
  return out;
}

2-3. fetchIssues: gh CLI を child_process で叩く(15分)

まずは手っ取り早い gh CLI 版。後で MCP 版に差し替えます。

// src/fetchIssues.ts
import { execFileSync } from "node:child_process";
import type { Issue } from "./types.js";

export function fetchIssues(repo: string, sinceIso: string): Issue[] {
  // gh issue list --repo owner/name --state open --json number,title,labels,url
  //   --search "updated:>=SINCE"
  const raw = execFileSync("gh", [
    "issue", "list",
    "--repo", repo,
    "--state", "open",
    "--search", `updated:>=${sinceIso}`,
    "--json", "number,title,labels,url",
    "--limit", "200",
  ], { encoding: "utf8" });
  const parsed = JSON.parse(raw) as Array<{
    number: number; title: string; url: string;
    labels: { name: string }[];
  }>;
  return parsed.map(p => ({
    number: p.number, title: p.title, url: p.url,
    labels: p.labels.map(l => l.name),
  }));
}
公式ドキュメント必読

gh issue list の --json と --search フラグは挙動が細かく、間違えると意図しないIssueを取得します。実務で使う前に必ず一度マスターしてください。gh issue list manual →

2-5. CLIエントリ + 動作確認(10分)

// src/index.ts
import { writeFileSync, mkdirSync } from "node:fs";
import { fetchIssues } from "./fetchIssues.js";
import { groupByLabel } from "./groupByLabel.js";
import { renderMarkdown } from "./renderMarkdown.js";

const args = process.argv.slice(2);
const repo = args[args.indexOf("--repo") + 1];
if (!repo) { console.error("usage: tsx src/index.ts --repo owner/name"); process.exit(1); }

const today = new Date().toISOString().slice(0, 10);
const since = new Date(Date.now() - 24 * 3600e3).toISOString();

const issues = fetchIssues(repo, since);
const groups = groupByLabel(issues);
const md = renderMarkdown(groups, { date: today, repo, total: issues.length });

mkdirSync("out", { recursive: true });
const path = `out/daily-report-${today}.md`;
writeFileSync(path, md, "utf8");
console.log(`✓ wrote ${path} (${issues.length} issues)`);

2-7. Hooks で毎朝自動実行(Ch18 / 10分)

最後の仕上げ。Claude Code の Hooks 機能、もしくはOS標準の cron/launchd で自動化します。

// .claude/settings.json (プロジェクトローカル)
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "bash -c 'if [[ \"$CLAUDE_TOOL_INPUT\" == *\"git commit\"* ]]; then npx tsx src/index.ts --repo motohirokoma/26tutorial; fi'"
          }
        ]
      }
    ]
  }
}

上は「commit のたびに日次レポートを再生成」する例。毎朝決まった時間に動かしたいなら、素直にOSのスケジューラを使うのが確実です。

# crontab -e に追記(毎営業日 08:30)
30 8 * * 1-5 cd ~/work/daily-report && /usr/local/bin/npx tsx src/index.ts --repo motohirokoma/26tutorial >> cron.log 2>&1

# macOS なら launchd の .plist を ~/Library/LaunchAgents/ に置く方法もある
公式ドキュメント必読

Hooksは settings.json の構造ミスで無限ループに陥ると、Claude Code セッションが暴走します(Ch22の失敗パターン10の1つ)。matcher の範囲 と PostToolUse の再帰 は公式を必ずマスターしてから。Hooks 公式 →

もっと詳しく: Hook か cron か、どう選ぶか

判断基準はシンプル:

  • 「Claude Codeを使っている時に付随的に動いてほしい」(保存時にprettier・commit後にテスト等) → Hook
  • 「時刻で絶対に動いてほしい、人がClaude Codeを開いていなくても」(毎朝08:30のレポート等) → cron / launchd / Scheduled Tasks(Ch19)

今回のレポートは「毎朝必ず」なので本命は cron。Hookは副次的に「commitしたら最新版を再生成」で補助。両方を組み合わせるのがパワーユーザーの定番ワークフローです。

発展 1
応用

Slack に自動通知する

生成した Markdown の要約(件数サマリだけ)を、Slack の Incoming Webhook に POST する機能を足してください。

  1. SlackでWebhook URLを発行し、.env に SLACK_WEBHOOK_URL として保存
  2. src/notifySlack.ts を新規作成。fetch(url, {method:"POST", body:JSON.stringify({text}) })
  3. src/index.ts の末尾で呼び出し。環境変数が無ければスキップ
  4. 既存テストを壊さないこと(通知処理はモック可能な関数として分離)
ヒント・実装方針を見る

副作用(HTTPリクエスト)は 引数でinject できる形にすると、vitestで差し替えやすくなります。

// src/notifySlack.ts
export type Poster = (url: string, body: string) => Promise<void>;

export async function notifySlack(
  text: string,
  url: string | undefined,
  post: Poster = defaultPost,
) {
  if (!url) return;   // Webhook未設定ならno-op
  await post(url, JSON.stringify({ text }));
}

async function defaultPost(url: string, body: string) {
  const res = await fetch(url, {
    method: "POST",
    headers: { "content-type": "application/json" },
    body,
  });
  if (!res.ok) throw new Error(`Slack post failed: ${res.status}`);
}

テスト側では post にモック関数を渡せば、ネットワーク無しで検証できます。Webhook URLは絶対にリポジトリにコミットしないこと(.env は .gitignore 済みか再確認)。

# daily-report Skill

## 使い方
`/daily-report owner/repo` と呼ばれたら、以下を実行:

1. 対象リポで `npx tsx ~/tools/daily-report/src/index.ts --repo owner/repo` を実行
2. 生成された out/daily-report-YYYY-MM-DD.md を読んで要約を返す
