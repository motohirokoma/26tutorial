CLAUDE.md と設定管理

プロジェクトの記憶をClaudeに渡すCLAUDE.mdと、許可・モデル・サブエージェントを束ねるsettings.json。出力品質と安全性が一段階上がる、上級編の入口です。

所要時間: 1.5h Phase 3: Advanced 難易度: 応用

CLAUDE.md とは — Claudeへの「ブリーフィング資料」

CLAUDE.mdはプロジェクトルートに置く特別なファイル。Claudeは起動時に自動で読み込んで、そこに書かれたルール・コンテキストを毎ターン守ります。新メンバーに渡すオンボーディング資料を、Claude向けに書いておくイメージです。

CLAUDE.md の本質

「毎回口頭で説明していたこと」を文書化してClaudeの記憶にする仕組み。技術スタック、命名規則、よく使うコマンド、絶対NGな操作、をここに集約します。

CLAUDE.md のある/なしで何が変わるか
同じ指示でも、CLAUDE.md があるかどうかで結果が劇的に変わる。

3階層のCLAUDE.md — グローバル / プロジェクト / サブディレクトリ

CLAUDE.mdは1箇所だけではなく、3つのスコープで重ね書きできます。Claudeは下から上に読みあげて、近いものほど優先します。

設定の優先順位 (近いほど強い)
同じトピックがあれば、より近いCLAUDE.mdが上書きする。
レイヤ場所共有範囲典型的な内容
グローバル~/.claude/CLAUDE.md自分の全プロジェクト好みのスタイル / 共通の口調 / 個人の好みの言語
プロジェクト./CLAUDE.mdこのリポジトリのチーム全員技術スタック / コーディング規約 / 重要な前提
サブディレクトリfrontend/CLAUDE.md などそのディレクトリ配下のみその領域固有のルール (DBスキーマ / UI規約)
使い分けの原則

「自分だけ知ってればいい」ことはグローバル、「チームで共有すべき」ことはプロジェクト。プロジェクトCLAUDE.mdは git に commit するのが正解です。

良いCLAUDE.md のテンプレート

「全部書けばいい」わけではありません。Claudeは コンテキスト を毎ターン消費します。読まないと困ることに絞るのが鉄則。

  • 1. プロジェクト概要 (3行) — 何を作ってるか / 誰のためか / 主要技術
  • 2. 技術スタック — 言語 / FW / DB / デプロイ先 / バージョン
  • 3. ディレクトリ構造 — ざっくり何がどこにあるか
  • 4. よく使うコマンド — 起動 / テスト / リント / ビルド / デプロイ
  • 5. コーディング規約 — 命名 / コメント / インポート順 / フォーマッタ
  • 6. NG事項 — 触っちゃダメなファイル / 使っちゃダメなコマンド

サンプル: Web アプリの CLAUDE.md

# Project: Recipe Share

## 概要
ユーザーがレシピを投稿・検索できるWebアプリ。
ターゲット: 自炊初心者。要件は ./docs/spec.md 参照。

## 技術スタック
- Frontend: Next.js 14 (App Router) + TypeScript + Tailwind
- Backend: Next.js API Routes + Prisma + PostgreSQL
- Auth: NextAuth.js (Google OAuth)
- Deploy: Vercel (本番) / GitHub Actions (CI)

## ディレクトリ
- app/        — ページとAPI Route
- components/ — UI部品 (Atomic Design ではなく機能別)
- lib/        — DB接続・ユーティリティ
- prisma/     — schema.prisma とマイグレーション

## コマンド
- 開発:    npm run dev
- テスト:  npm test (Vitest)
- リント:  npm run lint (ESLint + Prettier)
- DB更新:  npx prisma migrate dev

## コーディング規約
- 関数コンポーネント + Hooks のみ (クラスは書かない)
- 1ファイル1コンポーネント、デフォルトエクスポート禁止 (named のみ)
- 日本語コメントOK、ただし関数名・変数名は英語
- API レスポンスは必ず { data, error } 形式

## NG
- prisma migrate reset は本番DBで絶対に使わない
- env ファイルをコミットしない
- master ブランチへの直接 push 禁止 (必ずPR経由)
200行を超えたら

CLAUDE.md が200行を超えたら、領域別に frontend/CLAUDE.md 等へ分割を検討。読み込みコストと優先順位の両面で有利になります。

もっと詳しく: アンチパターンと改善例

アンチパターン1: 当たり前の事だけ書く — 「TypeScriptを使う」「テストを書く」だけでは何も決まりません。具体的な選択 (Vitest なのか Jest なのか) を書く。

アンチパターン2: README をコピペ — README は人間向け、CLAUDE.md はClaude向け。「インストール手順」より「セッション内で守るべきルール」を優先。

アンチパターン3: 全部書く — 5000行のCLAUDE.mdは逆効果。必要な箇所は @docs/spec.md のように ファイル参照 で外出し。

settings.json — 許可・モデル・環境変数

CLAUDE.md は振る舞いを伝えるファイル。settings.json は 環境を設定するファイル。両者で役割が異なります。

設定ファイルの読み込みフロー
graph LR
  A[Claude起動] --> B["~/.claude/settings.json"]
  B --> C["./.claude/settings.json"]
  C --> D["./.claude/settings.local.json"]
  D --> E[マージ完了]
  E --> F[セッション開始]
右側ほど優先度が高く、上書きする。local は個人用なので gitignore 推奨。

主要な設定キー

キー用途例
permissions.allow確認なしで許可するツール"Bash(npm test:*)"
permissions.deny絶対に拒否するツール"Bash(rm -rf /:*)"
model使うモデル"claude-opus-4"
env環境変数{"DEBUG": "true"}
hooksイベントごとに実行するスクリプト(次章で詳説)

サンプル: プロジェクトの settings.json

{
  "permissions": {
    "allow": [
      "Bash(npm:*)",
      "Bash(git status)",
      "Bash(git diff:*)",
      "Bash(npx prisma:*)",
      "Read(*)",
      "Write(src/**)",
      "Edit(src/**)"
    ],
    "deny": [
      "Bash(rm -rf:*)",
      "Bash(npm publish:*)",
      "Write(.env*)",
      "Write(prisma/migrations/**)"
    ]
  },
  "model": "claude-sonnet-4",
  "env": {
    "NODE_ENV": "development"
  }
}
deny の力

deny は allow より強い。「念のため触らせない」ファイルは deny に明示しておくと、誤操作の事故を構造的に防げます。

どのファイルに何を書くか — 整理表

「許可をどこに書くべきか」迷ったら、この表を見てください。

書きたい内容場所理由

ありがち

個人設定をプロジェクトに

  • 口調や好みを ./CLAUDE.md に書く
  • 個人のAPIキーを commit してしまう
  • 結果: チームメンバーが混乱
正解

スコープを切る

  • 個人 → ~/.claude/
  • チーム → ./.claude/ (commit可)
  • 機密 → settings.local.json (gitignore)

commit 対象
プロジェクト版のみ

「Claude にCLAUDE.md を書かせる」テクニック

新規プロジェクトで一から書くのは大変。Claudeに /init コマンドを使わせるのが最速です。

/init を使った半自動セットアップ
sequenceDiagram
  participant U as ユーザー
  participant C as Claude
  U->>C: ["/init"]
  C->>C: リポジトリを走査 (package.json / README)
  C-->>U: CLAUDE.md ドラフト提示
  U->>C: 「DBはPostgreSQL、リント追加」
  C-->>U: 反映版を提示
  U->>C: 「OK、commit して」
/init はリポジトリの空気を読んで、たたき台を作ってくれる。
Exercise 1
入門

自分用グローバル CLAUDE.md を書く

~/.claude/CLAUDE.md を作成し、自分の好みを5行以内で記述してください。次のセッションでClaudeの返答が変わるか確認します。

  1. mkdir -p ~/.claude で作成
  2. ~/.claude/CLAUDE.md を編集
  3. 「日本語で」「コードはコメント付きで」など自分の好みを書く
  4. 新しくClaudeを起動して、設定が効いているか試す
模範解答を見る

例: 以下のような最小限の内容で十分効果が出ます。

# Personal Preferences

- 返答は日本語で。技術用語の英単語はそのままでOK。
- コードを書く時は、行ごとに何をしているかの短いコメントを付ける。
- 質問が曖昧な時は、勝手に進めずに最初に1問だけ確認する。
- 結論を最初に言ってから理由を述べる (PREP法)。

新しいセッションでClaudeに「FizzBuzzをPythonで書いて」と頼むと、上記の好みが反映された返答が返ってきます。

Exercise 2
応用

既存リポジトリに /init で CLAUDE.md を生成し、磨き込む

手元にある任意のリポジトリで、Claude に/init を使わせ、生成された CLAUDE.md を改善してください。

  1. 対象リポジトリで claude を起動、/init 実行
  2. 生成された CLAUDE.md を読み、不足/過剰を3点ずつリストアップ
  3. 修正版を編集して commit
  4. 同じプロンプト (例: 「テストを追加して」) を 修正前と修正後で実行し、出力の差を比較
模範解答を見る

典型的な不足: NG事項 (deny相当) と 命名規約 が抜けている事が多いです。逆に過剰なのは「インストール手順」「パッケージ一覧」など README にあるべき情報。

改善例の差分:

+ ## NG
+ - master へ直接 push しない
+ - .env をコミットしない
+ - 既存 migration ファイルの編集禁止 (新規追加で対応)
+
+ ## 命名規約
+ - コンポーネントは PascalCase、その他は camelCase
+ - テストファイルは *.test.ts (spec.ts ではない)

- ## インストール
- npm install して npm run dev

修正後は「テストを追加して」のような曖昧な指示でも、*.test.ts 命名を守って書いてくれるようになります。
