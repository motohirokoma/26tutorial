デバッグ手法

エラーが出たら焦らず、Claudeに相談しましょう。エラー報告から原因特定、修正までの流れを身につけます。

所要時間: 1h Phase: Core 難易度: 中級

デバッグの流れ

「エラーが出た!」その瞬間にやることは決まっています。パニックにならず、5ステップで対処。

デバッグの定型5ステップ
1
観測
何が起きたか正確に見る
2
再現
手順を確定させる
3
仮説
原因の候補を3つ
4
検証
仮説を1つずつ潰す
5
修正
最小パッチ + 再発防止テスト
Claudeはこの全ステップを伴走できる。「いきなり3」に飛ばすと外しがち、必ず1から。

エラーの伝え方

Claude に伝える情報量で、回答精度が大きく変わります。

悪い例

「エラー出た」

  • 原因の手がかり ゼロ
  • Claudeも当てずっぽう
  • 5往復しても解決しない
良い例

4要素で報告

  • 何をしたら:操作や入力
  • 何が起きた:メッセージ全文
  • 期待値:本来こうなるべき
  • 環境:OS/バージョン/Node
良い例(コピペ用テンプレ)
> ## 症状
npm run build したら以下のエラー。

(エラーメッセージ全文を貼る)

## 期待値
build が成功して dist/ に出力される

## 環境
macOS 14.5 / Node 20.10 / Vite 5.2

## 直近の変更
- vite.config.ts に alias を追加
- src/components/Header.tsx を新規作成

バグ調査を頼む

原因が分からないバグは、「いきなり修正して」より「まず仮説を3つ」が正解。

> 下記症状の原因仮説を、優先度順に3つ立ててください。
各仮説について、どう検証するか(コード/コマンド)も付けて。
最も濃い仮説から検証して、結果を見せて。
仮説検証の進め方
graph TD
  A["症状観測"] --> B["仮説1〜3"]
  B --> C{"検証1"}
  C -- 真 --> F["原因確定"]
  C -- 偽 --> D{"検証2"}
  D -- 真 --> F
  D -- 偽 --> E{"検証3"}
  E -- 真 --> F
  E -- 偽 --> G["再仮説 or 詳細化"]
  G --> B
  F --> H["修正案"]
「いきなり修正」より「順に潰す」が結局速い。

使えるClaude機能

  • Bashで再現コマンドを走らせる — エラー出力を生で見る
  • Grepで関連コードを横断検索 — 同じエラー文言が他でも出ていないか
  • git log -p で直近変更を遡る — 「いつから壊れたか」を二分探索
  • ログファイルを Read — logs/ や node_modules/.cache/ も覗く

よくあるバグパターン

症状典型的な原因確認順序
「動いてたのに急に動かない」依存パッケージの更新① git diff package-lock.json ② Node バージョン
「ローカルでは動く、CIで落ちる」環境差(env, OS, タイムゾーン)① CIログ ② 環境変数 ③ OS差
「たまに失敗するテスト」非同期・順序依存① テストの依存 ② mockの初期化 ③ シード値
「メモリ不足」無限ループ・巨大データ① プロファイラ ② 該当処理を縮小再現
「型エラー意味不明」パッケージの型定義変更① tsc --noEmit 単体実行 ② 型定義の最新ドキュメント
バグの「種類」と発生頻度の体感分布
体感的にはこのくらい。割合を知っておくと、最初の仮説の精度が上がる。
もっと詳しく: スタックトレースの読み方

スタックトレースは 下から上に読む のが基本(古い → 新しい)。「どこで爆発したか」より「呼び出し経路のどこが意外か」を見る。Claudeに丸ごと貼って 「このトレースを上から3行ずつ要約、原因の可能性を指摘」と頼むと、自分で読むより速い。

心構え

バグは「敵」ではなく「学習の機会」。原因が分かったら、同じバグを再発させないテストを必ず追加(Ch08-2 のTDDサイクル)。これが品質の積み上げになる。

Exercise 1
入門

エラー報告テンプレを使ってみる

適当に壊したコードのエラーを、本章の「4要素テンプレ」でClaudeに報告し、解決してみましょう。

  1. 動いている小さいプロジェクトで、わざと変な変更を入れる(例: import名を typo)
  2. エラー出力をコピー
  3. 4要素テンプレに沿って Claude に貼り付け
  4. 修正提案を受けて適用、テスト緑にする
模範解答を見る

たとえば import { Foo } from './foo' を import { Foo } from './fooo' にしただけでも、テンプレ通りに伝えるとClaudeが原因を1往復で当てます。

## 症状
npm run dev で以下のエラー:
Module not found: Can't resolve './fooo'

## 期待値
正常に dev サーバが立ち上がる

## 環境
macOS 14.5 / Node 22 / Next 14.2

## 直近の変更
src/index.tsx の import パスをいじった

Claudeの回答:「./fooo というファイルは存在しません。./foo のtypoでは? Edit で修正します」。これだけで30秒。

Exercise 2
応用

「いつから壊れたか」を二分探索する

git の履歴を使って、Claudeに「どのコミットでバグが入ったか」を特定させましょう。

  1. ある程度コミットの溜まったリポを用意
  2. 「現在 X というバグが起きている。git log --oneline -20 を見て、二分探索でバグ混入コミットを特定する手順を提案して」
  3. Claudeに git bisect 風に1コミットずつ checkout → 検証させる
  4. 原因コミット特定後、その差分を見て修正方針
模範解答を見る

本物の git bisect を使う:

git bisect start
git bisect bad         # 今は壊れてる
git bisect good v1.2.0 # v1.2.0 は動いてた
→ git が中間コミットを checkout

> いまの commit でバグ再現するか確認するコマンドを生成して。
再現したら git bisect bad、しなかったら good を実行する。

Claudeに自動で確認 → bisect コマンド を進めさせると、20コミットなら4〜5往復で犯人が見つかります。「バグが入った日」が分かれば、その日の他の変更との関連も見えてくる。
