worktreeで安全に実験

git worktree で本流から実験を切り離し、失敗してもゴミ箱に捨てられる構造を作る

所要時間: 30分 Phase: Advanced 難易度: 中級
この章のゴール

git worktree を「Claude Code の実験用サンドボックス」として即座に使えるようになる。Agent tool の isolation=worktree の使い所、危ない変更を本流から隔離する作法、失敗した時に worktree を削除すれば全部リセット できる安心感——この3点を30分で体に染み込ませます。主ブランチを壊す恐怖から解放されると、Claude に任せる仕事のスケールが一段上がります。

バージョン注記

本章の記述は 2026年1月時点 の Claude Code および git (2.45以降) の挙動に基づきます。Agent tool の isolation パラメータ名・既定値は進化するので、挙動が違う時は Claude Code 公式ドキュメント と git-worktree 公式 を突き合わせて確認してください。

git worktree とは — 一言で

git worktree は、1つのリポジトリに対して複数の作業ディレクトリを並行して持つ ための git 標準機能です。普段 main で作業している repo/ と同時に、repo-experiment/ という別フォルダに experiment ブランチをチェックアウトできる、という理解で十分です。

通常のclone vs worktree の構造
clone で2つ持つと履歴が離れる。worktree は .git 本体を共有するので、片方でのコミットは git fetch 不要で即もう片方から見える。

つまり worktree は 「同じリポジトリの別作業場」。ブランチ と似ていますが、物理的に別フォルダ なので、main で IDE を開いたまま、隣で experiment を走らせる、といった芸当ができます。

今すぐ試す(1分)

適当な git リポジトリで、experiment 用の worktree を1本生やしてみましょう。

# いまのリポジトリルートで実行
git worktree add ../myrepo-experiment -b experiment

# 別フォルダが出来たことを確認
ls ../myrepo-experiment

# 現在の worktree 一覧を表示
git worktree list

2つの worktree が並んでいるのが見えたら成功。main 側で開いているエディタを一切閉じずに、../myrepo-experiment に cd して自由に実験できます。

なぜ Claude Code と worktree は相性がいいのか

Claude Code は、あなたが Edit や Bash ツールの実行を承認すると、その作業ディレクトリのファイルを本当に書き換えます。rm -rf、git reset --hard、巨大リファクタなど、失敗した時の巻き戻しコストが大きい操作で、main ディレクトリに居るのは危険です。

判断のコアは「戻せなくなる系か」「main と同時に見たいか」の2問。
  • 隔離(isolation) — Claude が壊せるのは worktree フォルダのみ。main フォルダは一切触れられない。
  • 並行性 — main で動作確認しながら、worktree で Claude に大きな改修を任せられる。文脈切替なし。
  • 使い捨てしやすさ — 実験が失敗しても git worktree remove でフォルダごと消せる。気持ちがラク。
  • ブランチ整合 — worktree ごとに1ブランチ紐付け。同じブランチを二重チェックアウトできないので混線しない。
もっと詳しく: branch だけで十分では? worktree の必要性

単に git switch -c experiment で別ブランチに切り替えるだけでも隔離はできます。ただし 同じフォルダ の中身が実験ブランチの内容に置き換わるため、main を 同時に 見る・動かす・比較することができません。worktree は「main を一切止めずに実験」を実現する差分です。さらに Agent tool の isolation=worktree が選べるのは、worktree が fs 的に独立 しているからこそ。ブランチ切替だけでは Claude の破壊からの物理的な隔離にはなりません。

覚えるべき worktree 基本コマンド 5つ

実運用で使うのは 5つだけ。この表を写経して手元のリポで叩けば、worktree は完全に自分の道具になります。

コマンド何をするいつ使う
git worktree add <path> -b <branch>新規ブランチで worktree を作成実験を始める時(基本)
git worktree add <path> <existing-branch>既存ブランチを別フォルダで展開他人のPRをローカル確認など
git worktree list現在の全 worktree を一覧「何がどこにあるか」確認
git worktree remove <path>worktree を削除(フォルダも消える)実験失敗 or マージ完了後
git worktree prune既に消えたフォルダのメタ情報を整理手で rm -rf してしまった後始末
公式ドキュメント必読

git worktree のオプション(--detach、--lock、--force 等)は、本番作業で迷ったら必ず公式で裏を取ってください。特に --force は文字通り強制なので誤用で履歴を壊しかねません。git-worktree 公式 →

今すぐ試す(2分)ライフサイクル一周

worktree の作成 → 実験 → 削除を1分で体験します。

# 1. worktree を作る(experiment ブランチ新設)
git worktree add ../myrepo-exp -b experiment

# 2. そちらへ移動して、試しに何か壊す
cd ../myrepo-exp
echo "壊れてもOK" > DANGER.txt
rm -rf src/  # 大胆な変更を試す

# 3. 戻って、丸ごと捨てる
cd ../myrepo
git worktree remove ../myrepo-exp --force

# 4. main は無傷を確認
ls src/  # 元通り

この「作って、壊して、捨てる」のリズムが体に入れば、Claude への依頼もワンランク大胆にできます。

Agent tool の isolation=worktree

Claude Code の Agent tool には、起動する子エージェント(Subagent)の作業場所を worktree に隔離する オプションがあります(2026年1月時点)。子が触っていいのは worktree フォルダの中だけになり、main から物理的に切り離されます。

isolation なし

親と同じフォルダで動く

  • 子が Edit すると main のファイルが直接変わる
  • リファクタ失敗=main が壊れる
  • 複数子が同じファイルを触ると競合
isolation=worktree

子専用 worktree で動く

  • 子の変更は worktree の中だけに閉じる
  • 失敗したら worktree を捨てて終わり
  • 並列子もフォルダ別で干渉しない
isolation=worktree の動作イメージ
3つの子が同時に大改修をしても互いに干渉しない。main は無傷のまま「どの結果が一番良いか」を比較できる。
今すぐ試す(Claude へのプロンプト例)

次のような頼み方で、Claude に「worktree 隔離で実験」させられます。

Agent tool で subagent_type=general-purpose を使い、isolation=worktree で
別フォルダに experiment ブランチを切ってから、src/auth/ のリファクタを
実施してください。テストが通ったら差分のサマリだけ返してください。

オプション名が通らない版では、代わりに「まず git worktree add で別フォルダを作ってからそこで作業して」と 自然言語で依頼すれば同じ効果が出ます。

プロの境目と運用のコツ

ここがプロの境目

「戻せなくなる系の実験」は絶対 worktree で。大規模リファクタ・破壊的マイグレーション・依存パッケージ大更新・AIが自分でブランチ切る系——これらは main で走らせた瞬間、失敗時のリカバリ時間が跳ね上がります。worktree に隔離しておけば、最悪でも git worktree remove --force で5秒リセット。
試し方： 今週やろうとしているリファクタを、まず git worktree add ../repo-exp -b refactor してから始めてみる。

worktree で潰せる「戻せない系」リスクの頻度感
実務で実際に遭遇しがちな「戻せない事故」の頻度イメージ。各リスクは worktree 運用で概ね無害化できる。
5秒
worktree 削除の所要時間
↑ 失敗コストが体感ゼロ
0回
main が壊れる回数
↑ 実験が積極的になる
3+個
並行できる実験数
↑ A/B/C比較が可能
もっと詳しく: worktree の落とし穴と対処
  • 同じブランチを2つの worktree で開けない — git が拒否する。片方を remove してから再度 add。
  • 手で rm -rf したら幽霊が残る — git worktree list に消したはずの worktree が出続ける。git worktree prune で成仏。
  • node_modules 問題 — worktree 側でも npm install が必要。シンボリックリンクで共有する手もあるが、2026年1月時点は素直に別インストールが無難。
  • IDE のインデックス再構築 — VS Code 等で別フォルダを開くと初回インデックス時間がかかる。その代わり main のインデックスは汚れない。

実験ワークフローのテンプレ

worktree を使う「一連の流れ」をテンプレ化しておくと、毎回迷いなく実行できます。下は 30分〜数時間の実験に使える定型。

1
切る
git worktree add ../repo-exp -b exp-YYYYMMDD
2
依頼
Claude に worktree 内で作業させる
3
検証
worktree 内でテスト・動作確認
4
判定
採用 / 見送りを人間が決める
5
処分
merge か remove で片付け
判定結果次の操作コマンド例
採用 (main へ取り込む)main で merge → worktree 削除cd ../repo && git merge exp-20260118 && git worktree remove ../repo-exp
採用 (PR経由)push して PR 作成、PR merge後に削除git push -u origin exp-20260118 && gh pr create
見送りworktree 丸ごと削除、ブランチも消すgit worktree remove ../repo-exp --force && git branch -D exp-20260118
保留worktree はそのまま放置(履歴共有なので軽い)—
worktree は「失敗する権利」をあなたに返してくれる道具だ。安全に失敗できる場所がある人は、大胆な実験を続けられる。 — 本章のメッセージ
Exercise 1
入門

worktree のライフサイクルを一周する

手元の任意の git リポジトリで、以下を実行してください。

  1. git worktree add で experiment ブランチの worktree を作る
  2. worktree 内で任意のファイルを1つ削除・1つ追加する
  3. git worktree list で2つの worktree が並んでいることを確認
  4. git worktree remove --force で worktree を削除
  5. main 側が完全に無傷であることを git status で確認
模範解答を見る

以下をそのまま実行すればOKです(myrepo 部分は自分のリポ名に置換)。

# 1. 作成
cd ~/path/to/myrepo
git worktree add ../myrepo-exp -b experiment

# 2. 実験
cd ../myrepo-exp
rm README.md        # 削除テスト
echo "hello" > HELLO.txt  # 追加テスト

# 3. 一覧確認
git worktree list
# /Users/you/path/to/myrepo        abc1234 [main]
# /Users/you/path/to/myrepo-exp    abc1234 [experiment]

# 4. 削除
cd ../myrepo
git worktree remove ../myrepo-exp --force
git branch -D experiment  # ブランチも消す

# 5. main 無傷確認
git status  # nothing to commit, working tree clean
ls README.md  # ちゃんと存在

このリズムが体に入れば、以降の「戻せなくなる系」タスクで worktree を反射的に手が伸びるようになります。

Exercise 2
応用

Claude に worktree で大きめの改修を任せる

自分のプロジェクト(または任意の OSS)で、下記を Claude Code に依頼してください。

  1. まず人間側で git worktree add ../repo-exp -b refactor を実行
  2. Claude に「../repo-exp に cd してから、src/ 配下の関数名を ○○ 命名規則に揃えるリファクタをしてください。テストが通ったら差分のサマリを返してください」と依頼
  3. 返ってきた差分が期待通りなら main に merge、そうでなければ worktree を破棄
  4. どちらの結果になったか、なぜそう判断したかを2行で振り返る
模範解答を見る

推奨プロンプト(コピペ可):

次の作業を worktree 隔離で実行してください。

1. `cd ../repo-exp` で worktree に移動
2. `src/` 配下のユーティリティ関数名を snake_case → camelCase に統一
3. 既存テストを実行し、全部通ることを確認
4. 失敗したテストがあれば修正(ただしassertionは触らない)
5. 最後に `git diff --stat` の結果と、変更した関数の一覧(最大10件)を返す

なお main ディレクトリのファイルは一切触らないでください。

判定例: 差分が期待通り(関数名変更のみ、ロジック無変更)なら cd ../repo && git merge refactor で取り込み、git worktree remove ../repo-exp で片付ける。もし想定外の変更(コメント消失・ロジック改変)があれば git worktree remove --force で破棄。振り返り例: 「差分が --stat で8ファイル・改変ロジックなしと確認できたので採用。worktree 隔離のおかげで main ブランチのエディタを開いたまま差分を比較できた」。

この経験をすると、worktree なしで同じ事をやる気が起きなくなります。
