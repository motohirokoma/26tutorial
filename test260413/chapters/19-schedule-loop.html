Scheduled Tasks & /loop

「放っておいて、終わったら教えて」を自分の業務に持ち込む

所要時間: 45分 Phase: Advanced 難易度: 中級
この章のゴール

Claude Code の Scheduled Tasks(CronCreate) と /loop スキル を使い、あなたがパソコンの前に座っていない時間も Claude に働かせる発想を手に入れる。朝の Issue 整理・PR サマリ・CI 監視など、「定期的に人間が確認していた反復作業」を自動化して、結果が来たら判断だけする 働き方に切り替えます。

バージョン注記

本章の記述は 2026年1月時点 の Claude Code の schedule / loop スキル挙動に基づきます。トリガー名・cron 表記・UI 位置は進化するため、コマンドが通らない時は 公式ドキュメント を必ず確認してください。

3つの「放置運用」モード ― schedule / loop / dynamic

Claude Code で「自分が席にいない時間に仕事をさせる」手段は、2026年1月時点で大きく3つに分かれます。まずは Scheduled Task、次に対話セッション内で繰り返す /loop、そして間隔を Claude 自身に決めさせる dynamic モード です。

3つの放置運用モードの関係
「日次運用は Schedule」「数分ポーリングは interval /loop」「進捗次第の見張りは dynamic /loop」。

迷ったらこの 1 行で覚えてください: PC を閉じて寝てもいいなら Schedule、ターミナルを開いたままで十分なら /loop、進捗ペースが読めないなら dynamic。

今すぐ試す(30秒)

Claude Code を開いて、まず一度こう打って挙動を見てみましょう。

/loop 10s echo "ping $(date +%T)"

10秒ごとにプロンプトが再実行されるのが分かります。Ctrl-C で止められる のが確認できたら、すぐ止めてOK。これが最小の /loop です。

Scheduled Tasks(CronCreate)で毎朝のルーチンを飛ばす

最初に押さえるべきは CronCreate です。Claude に「毎朝9時に Issue を整理してSlackに投稿して」と頼むと、内部では cron 式(0 9 * * 1-5)付きの Trigger が生成されます。

公式ドキュメント必読

Scheduled Tasks はリモート実行・認証・料金が絡む機能なので、挙動の正本は公式です。ここは公式ドキュメントにも書いていますし、実務で間違えると無駄な実行・請求が発生するので必ずマスターしてください。Claude Code 公式 →

cron 式 ― 5フィールドで「いつ」を指定する

CronCreate に渡す時刻表現は、伝統的な cron の 分 時 日 月 曜日 5フィールドです。よく使う式を丸暗記すると早く書けます。

cron 式意味典型ユースケース
0 9 * * 1-5平日 09:00毎朝の Issue 整理 / デイリースタンドアップ準備
0 18 * * 1-5平日 18:00日次のPRサマリ / 明日の TODO 下書き
*/15 * * * *15分おきステージング環境のヘルスチェック
0 */2 * * *2時間おき長時間ビルドの進捗レポート
0 9 * * 1毎週月曜 09:00週次レビューのアジェンダ下書き
0 0 1 * *毎月1日 00:00月次コスト(/cost)集計
30 8 * * 1-5平日 08:30朝会前の議題生成
cron 5フィールドの読み方
左から順に「分・時・日・月・曜日」。迷ったら * * * * *(毎分)から始め、左端を絞る。

実プロンプト ― 毎朝の Issue 整理を Scheduled 化する

一番の典型例は「毎朝、自分の GitHub Issue を整理して Slack に流す」です。これを Claude Code に頼む時は、① 時刻 ② やること ③ 成果物の宛先 の3点セットで書きます。

平日の朝9時に、以下を毎日実行する Scheduled Task を作ってください。

やること:
1. gh issue list --assignee @me --state open で自分の Open Issue を取得
2. 期限切れ・今日中・今週・バックログ の4カテゴリに分類
3. 優先度 P0/P1 を先頭に並べ替え
4. Markdown で以下の形に整形:
   ## 🌅 今日の Issue サマリ (YYYY-MM-DD)
   ### 期限切れ (N件)
   - #123 タイトル (deadline)
   ...
5. Slack の #daily-issues チャンネルに投稿

時刻: 平日 09:00 (cron: 0 9 * * 1-5)
タイムゾーン: Asia/Tokyo
名前: daily-issue-triage

Claude は内部で schedule スキルを呼び、CronCreate を実行して、作成したトリガーの ID とプレビューを返してきます。初回は必ず ドライランを1回(「このトリガーを今すぐ1回だけ試し実行してください」)してから本番稼働させるのが鉄則です。

もっと詳しく: トリガー一覧・無効化・削除

作成済みのトリガーは Claude に「今あるScheduled Task一覧を見せて」と言えば取得できます(内部では ScheduleList)。不要になったら「daily-issue-triage を無効化してください」と頼むだけで止まります(ScheduleDisable)。

完全に削除したい時は「削除して」。コスト警告の観点で、業務から外した Scheduled Task は必ず disable か delete。動いたまま忘れると月末の請求で気付くことになります。

トリガーのログ(前回の実行結果・エラー)は「最後の3回の実行ログを見せて」で確認できます。CI 監視系は 週1でログ目視 するのを運用に組み込みましょう。

コスト警告

Scheduled Task は あなたが寝ている間も Claude が動いて課金される という性質上、無限ループ・不要な高頻度トリガー・誤った cron 式(* * * * * を意図せず毎分)を放置すると、月末に数万円規模の請求になり得ます。作った直後に必ず /cost で当日分を確認し、翌日も /cost をチェック。高頻度トリガーは最低頻度から始めて徐々に上げる のが安全。

/loop ― セッション内の繰り返し(interval vs dynamic)

/loop は Scheduled Task と違って いまのターミナルセッションの中で プロンプトを繰り返します。一番大きな違いは PC が起動していてセッションが開いていないと動かない こと。その代わり設定は一瞬で、プロンプトの途中経過を自分の目で見ながら 進められます。

# 固定間隔(interval)
/loop 5m /check-deploy-status

# Claudeが自分で間隔を決める(dynamic)
/loop /babysit-prs

前者は interval モード(5分ごと固定)、後者は dynamic モード(次回待機時間を Claude が宣言する)。

interval と dynamic の比較

観点interval (/loop 5m ...)dynamic (/loop ...)
次回起動までの時間固定 (5m / 1h など)Claudeが進捗を見て動的決定
向いている作業CI/デプロイ/テストなど終端が明確なポーリングPRレビュー伴走などいつ進むか読めない監視
無駄な実行起きやすい(「まだだった」を繰り返す)少ない(Claudeが待つ)
停止のタイミング人間が Ctrl-C / 完了宣言Claudeが「完了です」と自己判断しやすい
予測可能性高い(時計で読める)低い(何分後に動くか不定)
dynamic モードの判断ループ
graph LR
  Start([/loop 起動]) --> Do["タスク実行"]
  Do --> Q{"完了した?"}
  Q -- Yes --> End([終了報告])
  Q -- No --> Dec{"進捗あった?"}
  Dec -- あり --> Wait1["短め待機 (1-2分)"]
  Dec -- なし --> Wait2["長め待機 (10-30分)"]
  Wait1 --> Do
  Wait2 --> Do
進捗に応じて Claude が自分で待機時間を伸縮する。だから "自己判断型" と呼ばれる。
今すぐ試す(2分)

テストが終わるまで待ちたい時を模して試します。

/loop 30s bash -c 'sleep 5; echo "build running... $(date +%T)"'

30秒ごとに echo が出るのが見えます。ビルド完了ログが出るまで寝てOK の運用が、この数行の感覚です。

/loop の時間感覚 ― 何分おきが妥当か

ポーリング間隔の推奨レンジ(用途別)

対数スケールなのは、CI(1分)から定期業務(1日)までが 1000 倍スケールで広がるためです。「1分単位 /loop」は短すぎ、「1日単位 /loop」は長すぎ(それは Schedule の領域)。/loop の妥当レンジは概ね 30秒〜1時間 と覚えておけば大外しはしません。

放置運用プロンプトの設計原則 ― 4つの絶対要素

対面セッションのプロンプトと違い、放置運用プロンプト は「人間がその場にいない」を前提に書く必要があります。手直しできない以上、曖昧さはそのまま事故になる。

  • 終了条件を明示する — 「○○になったら終わり」。これがないと dynamic /loop は永久ループする。
  • 通知の宛先を決める — Slack? メール? ファイル? 「見える形で残す」先を1つに絞る。
  • 危険操作を禁止する — rm -rf、force push、本番DBへの write は明示禁止。
  • エラー時の振る舞いを決める — 「失敗したら通知だけして止まる」「3回リトライして諦める」等。
Bad

曖昧な放置プロンプト

  • 「毎朝Issue見て整理しておいて」
  • 終了条件なし → 際限なく動く
  • 通知先なし → 結果が消える
  • 失敗時の挙動未定義
Good

放置に耐えるプロンプト

  • 「9時に gh issue list → 分類 → Slack #daily-issues」
  • 終了条件:投稿が成功したら終了
  • 通知先:Slack チャンネル名まで指定
  • 失敗時:エラー内容を DM で自分に通知して停止

型:放置プロンプトの5パート構造

1
役割
「あなたは日次監視エージェント」
2
トリガー
cron or /loop 間隔
3
手順
番号付きで具体的に
4
終了条件
「○○したら終わり」
5
失敗時
通知先・リトライ回数
ここがプロの境目

「放っておいて、終わったら教えて」パターンを自分の業務に持ち込む。上位20%のパワーユーザーは、反復業務の8割を Schedule か /loop に寄せて、人間は 結果を見て判断するだけ の働き方に切り替えている。対面プロンプトを磨くのが「読解力」なら、放置プロンプトを磨くのは「委任力」で、これが効くのは年間数百時間が自分の時間に戻るから。
試し方: 自分の「毎朝のIssue整理(15分)」を1つだけ選び、平日09:00のScheduled Taskに移してSlack投稿まで繋げる。1週間運用してやめたくならなければ勝ち。

実例 ― CIチェック・PRサマリの /loop レシピ

レシピ1 ― CIチェックの完了待ち(interval /loop)

PR を出した後、CI グリーンになったらマージしたい。張り付いて gh pr checks を叩くのは時間泥棒なので、/loop に任せましょう。

/loop 2m あなたはCI監視エージェント。以下を毎回実行してください。

1. gh pr checks (ブランチ名) で現在のチェック一覧を取得
2. すべて success なら: Slack #my-notifs に「#PR-NNN CIグリーン」と投稿して終了(loop stop)
3. 1つでも failure があれば: 失敗したジョブ名とログ末尾20行を Slack DM で私に送って終了(loop stop)
4. まだ pending/in_progress があれば、何もせず次ターンを待つ

2時間経っても完了しなければタイムアウト扱いで通知して終了してください。

レシピ2 ― 日次PRサマリ(Scheduled Task)

平日18:00に以下を毎日実行するScheduled Taskを作ってください。
名前: daily-pr-digest, cron: 0 18 * * 1-5, TZ: Asia/Tokyo

1. gh pr list --author @me --state all --search "updated:>=今日"
2. 各PRについて、状態(Open/Merged/Closed)・レビュアーの反応・残タスクを3行で要約
3. 以下の形で Slack #daily-summary に投稿:

## 📬 今日のPRダイジェスト (YYYY-MM-DD)
### Merged (N件)
- #321 タイトル (reviewer: @xxx, 所要h)
### Open (N件)
- #322 タイトル (blocker: xxx, 次のアクション: xxx)

エラー時: 自分に DM で通知して停止、リトライしない

レシピ3 ― PRレビュー伴走(dynamic /loop)

/loop あなたはPR伴走エージェント。

対象: https://github.com/my-org/my-repo/pull/321

毎ターン:
- 新しいコミット/コメント/レビュー変化があれば要約して記録
- CI の状態が変わったら通知
- レビュアーからの指摘には自動で返信案を下書き(私の確認待ち)

完了条件: PRが Merged または Closed になった時点で終了報告して停止。
待機時間はあなたが判断してください(動きがあれば短く、無ければ長く)。
24時間 動きがなければ自動で停止してください。
もっと詳しく: レシピを Skill 化して再利用

毎回このプロンプトを打ち直すのはダルい。3回以上使うなら Skill 化 します(詳しくは次章 Ch20)。~/.claude/skills/ci-watch/SKILL.md に上記プロンプトを保存して呼び出し名 ci-watch を付ければ、以降は /loop 2m /ci-watch PR-321 だけで起動できます。

放置運用の本当の効果は「Scheduled Task × Skill」の組み合わせで発揮されます。この章の内容は Ch20 で再利用されるので、今のうちに「このレシピ、Skill にしたら便利そう」という嗅覚を育てておきましょう。

もっと詳しく: Hooks との違い・組み合わせ方

前章の Hooks(Ch18)は「Claude が何かしたら反応する」受動型。Schedule/loop は「時刻やポーリングで Claude が動く」能動型。この2つは競合せず、Schedule が起動 → Hook で Slack 通知 のように重ねて使えます。

例: 毎朝9時の Scheduled Task が完了 → Stop フックで通知音 → あなたが確認。これが「自動化の二段構え」です。

Exercise ― 手を動かして定着させる

Exercise 1
入門

/loop で date を5秒おきに打たせる

/loop の最小挙動を体で覚える課題です。

  1. Claude Code を起動し、/loop 5s date を実行
  2. 30秒以上観察し、5秒間隔で時刻が出るのを確認
  3. Ctrl-C で停止
  4. 次に /loop 5s bash -c 'echo "$(date +%T) progress"' を試し、メッセージ付きで出せることを確認
模範解答を見る

実行すると5秒ごとに現在時刻が表示されます。コマンド版の出力例:

14:21:10 progress
14:21:15 progress
14:21:20 progress
^C
/loop stopped.

ポイント: /loop 5s ... の「5s」は s / m / h が使えます。停止は Ctrl-C または Claude に「loopを止めて」と伝えるだけ。この感覚があれば、あとは中身のプロンプトを差し替えるだけで業務に応用できます。

Exercise 2
応用

自分の「毎朝15分の反復業務」を Scheduled Task 化

この章の本番課題。自分の実業務から1つ選んで、平日朝の Scheduled Task にする。

  1. 毎朝 10〜20分かけて繰り返している作業を1つ選ぶ(Issue整理 / 通知チェック / KPI確認 など)
  2. 放置プロンプト5パート構造(役割・トリガー・手順・終了条件・失敗時)で記述
  3. Claude に「これを Scheduled Task にして」と頼む
  4. 作成後すぐ「今1回だけ試しに実行して」とドライラン
  5. 出力を確認して、プロンプトを1-2箇所だけ微修正
  6. 翌日 /cost で当日分を確認し、暴走していないか検証
模範解答を見る

お題「毎朝の Open PR 確認」を Scheduled Task 化した例。

平日09:00のScheduled Taskを作ってください。
名前: morning-pr-check, cron: 0 9 * * 1-5, TZ: Asia/Tokyo

【役割】あなたは朝の PR 状況レポーター。
【手順】
1. gh pr list --author @me --state open --json number,title,reviewDecision,updatedAt
2. レビュー待ち / Change Requested / Approved の3つに分類
3. 各PRに「次に何すべきか」を1行で付ける (例: reviewer をつつく / 修正 / マージ)
4. Slack #my-morning に Markdown で投稿
【終了条件】Slack 投稿が成功したら終了
【失敗時】エラー内容を 私に DM 通知 して停止、リトライしない
【禁止】git push / close / merge などの破壊的操作は絶対に行わない

評価ポイント: ①終了条件が明確 ②通知先が1箇所に絞られている ③失敗時の挙動が定義済み ④破壊的操作の禁止が明記されている。この4つが揃っていれば、あなたが寝ている間に走らせても事故らない「放置耐性のあるプロンプト」です。

1週間運用してみて、毎朝これを見るのが楽しみになれば成功。もし惰性で見なくなるようなら、そのタスクは自動化の価値が低かったということ。消して別のタスクに切り替えましょう。
