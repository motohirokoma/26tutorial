Phase 3: Advanced > Chapter 10

Skills と Hooks

SkillsでClaudeに専門能力を追加し、Hooksで確定的な振る舞いを差し込む。「賢いだけ」から「自分の現場に最適化されたエージェント」へ。

所要時間: 1.5h Phase 3: Advanced 難易度: 応用

全体像 — 拡張ポイントの地図

Claude Codeを「拡張する」道具は4種類あります。前章のCLAUDE.md / settings.json と合わせて、まず全体像をつかみましょう。

Claude Code 拡張ポイント (ラフスケッチ)
中心はClaude本体。周りに「振る舞い」「能力」「環境」「自動化」が刺さるイメージ。
道具目的実体いつ動く
CLAUDE.md振る舞いを伝えるMarkdownテキスト毎ターン読まれる
settings.json環境/許可を決めるJSON設定起動時に1回
Skills能力を追加するフォルダ + SKILL.md必要だとClaudeが判断した時
Hooks確定的な処理を差し込むシェルコマンド特定イベント発生時
違いの要点

Skills は「取りに行ける知識」、Hooks は「強制的に挟む処理」。前者はClaude判断、後者はシステム判断。

Skills — 専門能力をフォルダで配る

Skills は、特定タスクの手順・テンプレ・参考データをひとまとめにしたフォルダ。Claudeはユーザーの依頼を読み、関係ありそうなSkillの説明文を見て必要な時だけ中身をロードします。

Skill フォルダの最小構成

~/.claude/skills/
└── pdf-parser/             ← Skill名 = フォルダ名
    ├── SKILL.md            ← 必須。Claudeが最初に読む説明書
    ├── templates/          ← 任意。テンプレファイル
    │   └── report.md
    └── scripts/            ← 任意。補助スクリプト
        └── extract.py

SKILL.md の構造

---
name: pdf-parser
description: PDFファイルからテキスト・表・画像を抽出する。pdfplumber を使う。
---

# PDF Parser スキル

## いつ使うか
- ユーザがPDFを「読んで」「要約して」「抽出して」と言った時
- ファイル拡張子が .pdf

## 手順
1. ファイルパスを ./scripts/extract.py に渡す
2. 出力 JSON を解析
3. 表があれば Markdown 表として整形
Skill が呼ばれる流れ
sequenceDiagram
  participant U as ユーザー
  participant C as Claude
  participant S as Skill
  U->>C: 「この請求書PDFを集計して」
  C->>C: 利用可能Skillの description を走査
  C->>S: SKILL.md を読み込み
  S-->>C: 手順 + scripts/extract.py を提示
  C->>S: extract.py を実行
  S-->>C: JSON 出力
  C-->>U: 集計結果
Skillはユーザーが指名しなくても、Claudeが自分で気づいて呼ぶ。
description の書き方が肝

Claudeは description を見てSkillを発火させるか判断します。ここに「このSkillはこういう時に使う」を具体的に書くほど、自動発火が正確になります。

場所共有範囲用途

具体的なSkill例 (3つの方向性)

  • 定型文書スキル — 議事録 / 障害報告 / RFC のフォーマットをテンプレ化。Claudeが正しい節構造で書く
  • 専門知識スキル — 「Stripe 連携の鉄則10条」「自社UI規約」など、頻出ドメイン知識をパッケージ化
  • 外部処理スキル — 補助スクリプト (Python等) を同梱。Claudeはそれを呼ぶだけで複雑処理を完了させる
もっと詳しく: SKILL.md の良い書き方 5原則
  1. 「いつ使うか」を冒頭に — Claudeが発火判断するため
  2. 手順は番号付き — 順序を守らせるため
  3. NG例も書く — 「この時は使わない」を明示
  4. テンプレは別ファイルに — SKILL.md 本体を肥大化させない
  5. 具体例を1つ入れる — Claudeはサンプルがあると精度が劇的に上がる

Hooks — Claudeのライフサイクルに割り込む

Hooks は settings.json に書く確定的な処理。Skillsと違いClaudeの判断は介在せず、特定イベントで必ず実行されます。

Hooks の発火イベント (タイムライン)
主要4イベント。他にもサブエージェント関連などが追加されつつある。

典型的なHookの設定

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          { "type": "command", "command": "npx prettier --write $CLAUDE_FILE_PATHS" }
        ]
      }
    ],
    "Stop": [
      {
        "hooks": [
          { "type": "command", "command": "say 'タスク完了'" }
        ]
      }
    ]
  }
}
Hookは"必ず"動く

Hookに無限ループや破壊的コマンドを書くと、Claudeを起動するたびに実行されます。動作確認は別タスクで行ってから本設定に入れる癖をつけましょう。

Skills と Hooks の使い分け

「これ、SkillとHookどっちで実装すべき?」と迷う場面が多発します。判断基準は「Claudeに考えさせるか」「機械的に決まるか」。

どちらを選ぶ?
graph TD
  Q{"その処理は毎回必ず実行?"}
  Q -- "Yes 必ず" --> H[Hooks]
  Q -- "No 状況次第" --> Q2{"高度な判断が必要?"}
  Q2 -- "Yes" --> S[Skills]
  Q2 -- "No 単純条件" --> H2["Hooks 条件付き"]
「必ず」「決定的」ならHooks、「内容を読んで判断」ならSkills。
Skills

判断付きの能力

  • PDFを読み解く
  • 議事録を整形する
  • Stripe連携コードを書く
Hooks

機械的な処理

  • 編集後にprettier実行
  • commit前にlint走らせる
  • 応答完了でSlack通知

合わせ技 — Skill + Hook

強力なのは両者の組み合わせ。「議事録Skill」がフォーマットを保ち、「PostTool Hook」が出力を即座にチームのesaへ投稿、といった具合です。

step 1
ユーザー入力
「先週の会議録音から議事録作って」
step 2
Skill 発火
議事録Skillが手順とテンプレを提供。Claudeが整形。
step 3
PostTool Hook
Write 直後に esa post CLIで自動投稿。
step 4
Stop Hook
完了通知をSlackへPOST。チームに即連携。
Exercise 1
入門

最小の Skill を作る

「英語の文章を3行に要約する」だけのSkillを作成してください。

  1. mkdir -p ~/.claude/skills/quick-summary
  2. ~/.claude/skills/quick-summary/SKILL.md を作る
  3. front matter (name, description) と本文 (いつ使うか / 手順) を書く
  4. Claudeに「これを要約して: <英文>」と頼んで、Skillが発火するか確認
模範解答を見る 
---
name: quick-summary
description: 英語のテキストを日本語で3行に要約する。長文や記事URLを渡された時に使う。
---

# Quick Summary

## いつ使うか
- ユーザが「要約して」「サマって」と言い、対象が英語テキスト or URL の時

## 手順
1. テキストの主題を1行で抽出
2. 主要な根拠を1行で抽出
3. 結論/示唆を1行で抽出
4. 出力フォーマット:
   - 主題: ...
   - 根拠: ...
   - 結論: ...

## NG
- 日本語テキストには使わない (別タスクとして処理)

Claudeで「次の英文を要約して: ...」と試すと、Skill 発火 → 3行構造で返ってくるはず。発火しない時は description をより具体的に書き直すのがコツ。

Exercise 2
応用

PostToolUse Hook で自動フォーマット + Slack通知

適当なJS/TSプロジェクトで、Claudeが Edit / Write を実行した直後に prettier を走らせ、さらにタスク完了時 (Stop) に Slack へ通知する Hook を組んでください。

  1. プロジェクトで ./.claude/settings.json を作成
  2. PostToolUse hook で npx prettier --write を編集対象に実行
  3. Stop hook で curl -X POST で Slack Incoming Webhook 叩く
  4. 新しいClaudeセッションで実際にファイル編集 → 自動フォーマット + 通知が来るか確認
模範解答を見る 
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "npx prettier --write $CLAUDE_FILE_PATHS 2>/dev/null || true"
          }
        ]
      }
    ],
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "curl -s -X POST -H 'Content-type: application/json' --data '{\"text\":\"Claude task done\"}' $SLACK_WEBHOOK_URL"
          }
        ]
      }
    ]
  },
  "env": {
    "SLACK_WEBHOOK_URL": "https://hooks.slack.com/services/XXX/YYY/ZZZ"
  }
}

ポイント: prettier が失敗してもClaudeを止めないように || true を付ける。SlackのWebhookは settings.local.json 側に置くとgitに上がらず安全。
