Skillsで自分のワークフローを作る

自分の「いつもの頼み事」を、名前付きの資産に変える

所要時間: 45分 Phase: Advanced 難易度: 中級
この章のゴール

Skill(=スキル)は、あなたが毎回書いている長めのプロンプト・手順・判定フローを、1つのMarkdownファイルに束ねた再利用可能なワークフローです。この章では「Skillとは何か→SKILL.mdの書き方→自分の頻出作業を1つSkill化→チームに配る」までを45分で通します。読むより作るが合言葉。章の最後には、あなたが今週3回以上繰り返している作業が1つ、Skill として ~/.claude/skills/ に置かれている状態を目指します。

2026年1月時点の情報

本章の記述は 2026年1月時点 の Claude Code / Agent SDK の Skill 仕様に基づきます。`~/.claude/skills/{name}/SKILL.md` 配置・frontmatter の `name` / `description` フィールド・Agent tool からの自動発火などは進化中のため、挙動に違和感があれば 公式ドキュメント を確認してください。

Skill とは — 一言で

Skill は、あなたの「いつもの頼み事」を名前付きの資産に変える仕組みです。毎回書いていた長いプロンプトを1ファイルに固定化し、Claude がユーザの依頼内容から自動で発火させてくれます。Hooks(Ch18)がイベント駆動、Schedule(Ch19)が時刻駆動なのに対し、Skillは意図駆動——「これをやって」の意図を読んで発火します。

Skill が置かれる階層と発火の流れ
User の依頼文に Skill の description が一致すれば Claude が自動でその Skill を呼び出す。明示的に /skill-name で呼ぶこともできる。

Skill の置き場所は2通り:ユーザ Skill(~/.claude/skills/{name}/SKILL.md)は全プロジェクト横断で効き、プロジェクト Skill(.claude/skills/{name}/SKILL.md)はそのリポジトリ内でのみ効きます。frontmatter に name と description を書くのが必須です。同じ名前のSkillがユーザとプロジェクトで重複した時はプロジェクト優先の挙動になります。

今すぐ試す(30秒)

既にインストールされている Skill を確認してみてください:

ls -la ~/.claude/skills/ 2>/dev/null
# 何が出てくるか見る。init / review / security-review などが並ぶはず

Claude Code には標準Skillが最初から同梱されています。/init や /review を使ったことがあるなら、それは Skill を呼んでいたということです。

Skill 化のサイン — 3回ルール

Skill を作る最大の壁は「作ること」ではなく「Skill にすべき作業を見つけること」です。パワーユーザーは共通して、ある簡単な判定基準を持っています。

ここがプロの境目

同じ頼み事を3回したら、Skill化のサイン。1回目は偶然、2回目は習慣の兆し、3回目は確実に繰り返される作業パターンです。毎回同じようなプロンプトを書いているなら、その5分の書き起こしが月に10-20回発生している——年間で数時間〜十数時間の無駄になります。
試し方： 今週のチャット履歴を見返し、3回以上繰り返した依頼を1つだけ選び、本章の手順でSkill化する。

Skill化すると効くパターン vs 効かないパターン

Skill 化が効く

定型で繰り返すもの

  • 毎朝のIssue整理・優先度付け
  • PR作成前のチェックリスト走査
  • 会議議事録からの課題抽出
  • リリースノートの自動生成
  • 学習サイトの章単位量産(make-tutorial)
Skill 化は過剰

1回きり・その都度違う

  • 新機能の設計(毎回要件が異なる)
  • 突発的なバグ調査
  • ひらめき系の議論
  • 3行で済む短い依頼(aliasで十分)

SKILL.md の構造 — frontmatter と本文

Skill の実体は1枚のMarkdownファイル、~/.claude/skills/{name}/SKILL.md。構造はシンプルにfrontmatter と本文の2ブロックだけです。frontmatter はYAML形式で、本文は普通のMarkdownが書けます。

SKILL.md の構造
frontmatter が Skill の「顔」、本文が「中身」。Claude は発火時に両方を読み込む。

frontmatter の2フィールド

フィールド必須役割書き方のコツ
name必須Skillの識別子(/name で明示呼出も可)ハイフン区切り、小文字、英数字(例: daily-issue-triage)
description必須Claudeが「このSkillを発火すべきか」判定する材料「いつ使う」+「何をする」を1-3文。トリガー語を複数含める
公式ドキュメント必読

Skill の frontmatter 仕様・発火の優先順位・ユーザSkillとプロジェクトSkillの上書き関係は、公式ドキュメントを必ずマスターしてください。ここを誤ると「書いたのに発火しない」「意図しないSkillが勝手に呼ばれる」が起きます。Claude Code 公式 →

description の書き方 — トリガーを網羅せよ

Claude は description を読んで、現在のユーザの依頼と意味的に一致すれば発火します。つまりトリガーとなる語彙を複数詰めるのが最大のコツです。単一フレーズだと発火漏れが多発します。

悪い description

発火漏れが多い

description: Issueを整理する
  • 短すぎて文脈がない
  • 類義語(タスク/チケット/バックログ)が入っていない
  • 「いつ使うか」が欠落
良い description

トリガー網羅

description: 毎朝のGitHub Issue整理。オープンIssueを優先度(P0-P3)で仕分けし、担当者と期限を提案する。「Issue整理」「朝タスク」「バックログ確認」「スプリント準備」時に使う。
  • いつ使うかが明確
  • 類義語を複数配置
  • アウトプットも匂わせる
もっと詳しく: description が長すぎたら逆効果になる

description は「Skillリストを眺めた Claude が『これだな』と思える最小文量」が理想。公式の組込みSkillを眺めると 1-3文 / 100-300字程度がほとんどです。500字を超えたら、それは description ではなく本文に書くべき情報です。Claude は全Skillの description を毎回読むため、冗長に書くと他Skillの判定精度が下がる副作用もあります。

本文の推奨構造

  • # スキル名 — H1 で Skill の名前(frontmatterと同じでOK)
  • ## いつ使うか — 発火してほしい場面を箇条書きで3-5個
  • ## 手順 — 実行ステップ。番号付きリスト、または ### サブ見出しで分割
  • ## 出力形式 — ユーザに何をどの形式で返すか明記
  • ## 注意事項 — 破壊操作の警告・確認タイミング・やってはいけないこと

SKILL.md 実例全文 — daily-issue-triage

頭で理解したら、実物を見るのが一番早い。ここでは「毎朝のIssue整理」Skillを丸ごと載せます。そのままコピーして ~/.claude/skills/daily-issue-triage/SKILL.md に置けば、明日の朝から動きます。

# Daily Issue Triage スキル

毎朝のGitHub Issue整理を自動化するスキル。オープンなIssueを走査し、P0-P3で仕分けた Markdown レポートを出力する。

## いつ使うか
- 朝会/スタンドアップ前にIssueを整理したい時
- 週次のバックログ確認時
- スプリント計画前のIssue優先度付け時
- 「Issue整理して」「今日のタスク何?」とユーザが聞いた時

## 前提
- `gh` CLI が認証済み(`gh auth status` で確認)
- 対象リポジトリのルートで実行されている

## 手順

### Step 1: オープンIssue取得
```bash
gh issue list --state open --limit 50 --json number,title,labels,assignees,createdAt
```

### Step 2: 優先度判定ルール
以下のルールで P0-P3 を付ける:
- **P0 (緊急)**: label に `bug` かつ `critical` / 作成から7日以上未対応
- **P1 (高)**: label に `bug` or `security` / 顧客からの報告
- **P2 (中)**: label に `enhancement` / 計画済み機能
- **P3 (低)**: label に `docs` `chore` / 改善提案

### Step 3: 担当者と期限の提案
- 担当者未設定のIssueには、labelから類推して推奨担当を付記
- P0は当日、P1は3日以内、P2は今週中、P3は今月中 を推奨期限とする

### Step 4: レポート出力
以下のMarkdownフォーマットで標準出力に返す:
```markdown
# Daily Issue Triage (YYYY-MM-DD)

## P0 緊急対応 (N件)
- #123 [タイトル] — 担当: @user / 期限: 今日
...

## P1 高 (N件)
...

## サマリ
- 総件数: N / P0-P3内訳: a/b/c/d
- 未担当: N件 / 期限超過: N件
```

## 出力形式
- 標準出力にMarkdown
- ユーザがファイル保存を望んだら `./triage-YYYY-MM-DD.md` に書き出す

## 注意事項
- Issue を **クローズしたり書き換えたりしない**(読み取り専用)
- 担当者の assign もユーザ承認なしに行わない
- `gh` が未認証なら、まずユーザに `gh auth login` を案内する
今すぐ試す(5分で設置)

上の実例を実際に設置してみましょう:

mkdir -p ~/.claude/skills/daily-issue-triage
# エディタで ~/.claude/skills/daily-issue-triage/SKILL.md を作り上記を貼る
# その後、適当なGitHubリポジトリで Claude Code を起動
claude
# チャットで: 「今日のIssue整理して」 と打つ
# → daily-issue-triage が発火するのを確認

発火しない場合は description のトリガー語を増やすか、/daily-issue-triage で明示呼び出しを試してください。

あなたの業務を1つ Skill化する — 5ステップ

理論はここまで。次はあなたの手で1つ Skill を作る工程です。45分の章の中で、ここが最重要の30分です。

1

2
手順書出

3

frontmatter+本文
4
設置・発火
skills/に置いて起動

各ステップの中身

1. 候補選定(5分)
今週のチャット履歴から「同じようなこと3回以上頼んだ」作業を1つ。迷ったら一番面倒だった作業を選ぶ
2. 手順書出(10分)
一度 Claude に通常依頼し、どういうプロンプトで・どの手順で成果に到達したかを箇条書きで記録
3. SKILL.md 執筆(10分)
実例(Section 4)を雛形にコピペ。name/description/## いつ使うか/## 手順を埋める
4. 設置・発火(3分)
~/.claude/skills/{name}/SKILL.md に保存 → 新しいチャットで自然文依頼 → 発火を確認
5. 改善(随時)
使いながら手順を追記・description のトリガー語を増やす・失敗パターンを注意事項に書き足す
今すぐ試す(15分・本章のハンズオン)

手を止めて、今 5ステップ を実行してください。候補が浮かばない時の例:

  • 「PR作成前に必ず走らせるチェック5項目」→ pre-pr-checklist Skill
  • 「Slack議事録から課題と次アクションを抽出」→ meeting-minutes-extract Skill
  • 「週次レポート作成(GitHub Activity要約)」→ weekly-report Skill
  • 「ブログ下書きの校正観点5つ走査」→ blog-review Skill

1つできれば、次からは10分で増産できるようになります。

Skill数が増えると時間がどう変わるか(イメージ)
Skill 0→5個で週あたりの定型作業時間が大きく減る。6個目以降は効果逓減するので「厳選5-10個」が現実解。

個人のSkillが育ってきたら、次はチーム展開です。Skillは単なるMarkdownファイルなので、gitリポジトリに置いて共有するだけで済みます。symlink を使えば社内共通Skillを ~/.claude/skills/ に自動反映できます。

もっと詳しく: 社内Skill Pack の symlink 運用例

社内リポ internal-skills に Skill 群を集めて、各メンバーが下記で同期する運用が軽量で強力:

# 初回
git clone git@github.com:yourcompany/internal-skills.git ~/work/internal-skills
for dir in ~/work/internal-skills/skills/*/; do
  name=$(basename "$dir")
  ln -sf "$dir" ~/.claude/skills/"$name"
done

# 更新時
cd ~/work/internal-skills && git pull
# symlinkのまま自動反映

PR レビューで Skill の品質を担保し、変更は全員に即行き渡る。社内のノウハウが書いた瞬間に全員のものになる状態が作れる。

個人の暗黙知を Skill に落とすと、それは翌日からチームの公共財になる。知識共有の最小単位として、Skill は会議や Wiki より圧倒的に軽い。
注意

チーム展開する Skill は破壊的操作を含めない(rm -rf / force push / DB 書換 等)のが鉄則。そういう作業は Hook(Ch18)で明示的な確認ステップを噛ませるか、Skill 本文で「必ずユーザ確認」と書いて実行は人間に残します。Skill は情報生成・整理・提案までで留め、破壊的実行は人間の判断を原則に。

落とし穴と対策

Skill 導入の初期に踏みやすい5つの落とし穴を整理します。知っていれば全部回避できます。

落とし穴症状対策
Skillが発火しない自然文で頼んでもSkillが使われないdescriptionのトリガー語を増やす//nameで明示呼出
意図しないSkillが発火関係ないSkillが勝手に呼ばれるdescription を限定的に書き直す・類似Skillを統合
Skillが多すぎて埋もれる20個超えて自分でも把握不能月1で棚卸し。使ってないSkillはアーカイブ
破壊操作を入れ込むSkillにいきなりDB書換を含めた破壊操作は手順にだけ書き、実行はユーザ確認必須と注意事項に記載
プロンプトが長大化SKILL.md が1000行超でメンテ不能本文を分割・scripts/やtemplates/サブディレクトリを活用
Skill運用の健康度チェック(レーダー)
自分のSkill運用を5軸で評価。弱い軸から改善する。

Exercise 1
入門

既存Skillの description を読む

Claude Code に同梱されている標準Skillの description を読み、どんなトリガー語が埋め込まれているかを観察してください。

  1. ls ~/.claude/skills/ で一覧を確認
  2. 3つ以上の SKILL.md を開き、frontmatter の description だけを抜粋
  3. 各Skillの description で「いつ使うか」「何をする」がどう表現されているかをメモ
  4. 自分なら何を足すかを書き出す
模範解答を見る

標準Skillの description の典型パターン:

# init
description: Initialize a new CLAUDE.md file with codebase documentation

# review
description: Review a pull request

# security-review
description: Complete a security review of the pending changes on the current branch

観察ポイント:(1) シンプルだが「何をする」が動詞で明確。(2) 発動条件(object)も書いてある(a pull request / the current branch)。(3) 日本語で書くなら類義語(PR/プルリク/変更/差分)を足すとより発火しやすい。自作Skillではさらに「いつ使う」のコンテキスト語(朝タスク/スプリント計画時など)を足すのがコツ。

例: 「PR作成前チェックリスト」Skill の最小実装

# Pre-PR Checklist スキル

PRを出す前の定型チェックを自動化する。

## いつ使うか
- PR作成を意識した時(「PR作る」「プルリク出す」等)
- commit が積み上がって push 直前
- リリース前の最終確認

## 手順
1. `git status` と `git diff --stat` で変更全体を確認
2. 言語に応じたテスト: `npm test` / `pytest` / `go test ./...`
3. Lint: `npm run lint` / `ruff` / `golangci-lint`
4. 型チェック: `tsc --noEmit` / `mypy` / `tsc`
5. commitメッセージが Conventional Commits になっているか確認
6. `.env` やAPIキーを含んでいないかgrepで検査

## 出力形式
各ステップの ✅ / ❌ を一覧化し、❌があれば詳細を添える。

## 注意事項
- 失敗項目を**自動で修正しない**(ユーザ判断)
- Secret らしき文字列を見つけたら必ずユーザに警告

運用ポイント: 最初は手順を少なく始め、失敗経験(「あのチェック忘れてPRで指摘された」)が出るたびに1行ずつ足していく。Skillは生き物のように育てるのが長く使うコツです。チーム展開する時は、repoの.claude/skills/pre-pr-checklist/SKILL.mdに置けば、cloneした全員に即配布されます。
