CLAUDE.mdで環境を教える

Claude に「このプロジェクトで何をどう進めるか」を1枚のメモで伝える

所要時間: 45分 Phase: Basics 難易度: 初級
この章のゴール

CLAUDE.md は Claude Code がセッション開始時に自動で読む「あなた専用の指示書」です。ここに正しく書けば、毎回「このプロジェクトはnpmじゃなくてpnpmです」「PRは main ではなく develop に」といった前置きを省略できます。逆に書きすぎると毎ターン無駄なトークンを消費する諸刃の剣。この章で、書くべきこと / 書かないこと / 短く維持するコツ を体で覚えます。

バージョン注記

本章の記述は 2026年1月時点 の Claude Code 挙動に基づきます。CLAUDE.md の検索パス・優先順位・`@import` 構文は将来変更される可能性があるため、挙動が想定と違う時は公式メモリドキュメントを必ず確認してください。

CLAUDE.md とは何か — 常駐する「環境説明書」

CLAUDE.md は、Claude Code がセッション開始時・サブディレクトリ移動時に自動で読み込む Markdown ファイルです。ユーザーが毎回プロンプトで説明しなくても、Claude は「このプロジェクトの前提」を既に知った状態で会話を始められます。

イメージとしては、新入社員にデスクの横に貼ってある「このチームの決まりごと」メモ。毎朝繰り返し説明しなくていい代わりに、貼ってある内容は毎回目に入る ので、書きすぎは逆効果。ここが全てのポイントです。

CLAUDE.md が読まれるタイミング
CLAUDE.md は毎ターンの文脈に混ざる。だから短く保つ意味がある。
  • Markdownで書くだけ — 特別な構文は不要。見出しと箇条書きで十分
  • 毎ターン参照される — 1回読んで終わりではありません。長いと毎回コストに響く
  • スコープが階層化 — ユーザー / プロジェクト / サブディレクトリの3層を合成
  • Gitで共有できる — プロジェクト版は commit してチームで育てる

スコープ階層：3層で合成される

CLAUDE.md は 1ファイル=1役割 ではなく、スコープごとに複数層あります。下から順に「狭い→広い」、Claude は全部を読んで合成します。

CLAUDE.md の3層スコープ
全部が合成される。だから各層で書くべき粒度が違う。
層パス例Gitに入れる?書くべきこと
①ユーザー~/.claude/CLAUDE.md個人設定(入れない)「日本語で回答」「git commit署名」「コード説明は簡潔に」
②プロジェクト./CLAUDE.md入れる(チーム共有)ビルド手順、テストコマンド、ブランチ戦略、禁止事項
③サブディレクトリ./packages/api/CLAUDE.md入れるそのディレクトリ固有の規約 (例: `/api` は REST 命名規則)
公式ドキュメント必読

ここは公式ドキュメントにも書いていますし、実務で間違えると致命的なので必ずマスターしてください。特にユーザー版 vs プロジェクト版の分離を誤ると、個人の好みがチームの CLAUDE.md に混ざって git で共有されてしまう事故が起きます。Claude Code 公式：メモリとCLAUDE.md →

もっと詳しく: `@import` で他ファイルを取り込む

CLAUDE.md の本文中で @path/to/other.md と書くと、そのファイルの内容が取り込まれます (2026年1月時点)。長い規約を 1ファイルに詰め込まず、@docs/coding-style.md のように分割できます。

# CLAUDE.md
## コーディング規約
@docs/coding-style.md

## デプロイ手順
@docs/deploy.md

ただし import先も毎回読まれる ので、分割=軽くなる ではありません。あくまで管理上の分割と割り切ること。

書くべきこと / 書いちゃダメなこと

CLAUDE.md に書く情報の選別は「毎回プロンプトで繰り返し言うくらいなら」基準で決めます。1回きりの依頼に関係する内容は書かない。ノイズが増えると重要ルールが埋もれます。

書く

毎回効く環境情報

  • パッケージマネージャ (pnpm/bun/yarn)
  • テスト実行コマンド (pnpm test:unit)
  • ブランチ戦略 (main直push禁止、PRは develop 宛)
  • 禁止操作 (git push --force禁止、など)
  • コード規約の要点 (3-5行)
  • 使っているフレームワーク/バージョン
書かない

ノイズになるもの

  • 今日のタスクリスト (その場のプロンプトで)
  • API仕様全文 (必要な時に Read させる)
  • 長いチュートリアル的説明
  • 公式ドキュメントの丸コピー
  • 「丁寧に書いて」「がんばって」等の情緒
  • 履歴・経緯 (CHANGELOG.md の仕事)
ここがプロの境目

CLAUDE.md は最小限に。200行超えたら分割/削除を考える。Claude は毎ターン CLAUDE.md を読むので、長さ=コスト×全会話数。長いほど重要な指示が埋もれ、トークンも消費し、指示精度も下がる三重苦になります。
試し方： いま自分のプロジェクトの CLAUDE.md を開いて、各行ごとに「これ本当に毎回必要?」と問い、Yes と即答できない行は削除候補としてコメントアウト。

肥大化版 vs 引き締め版:具体の比較

同じプロジェクトで書いた2つの CLAUDE.md を比べます。Before は肥大化の典型、After は引き締まったプロ仕様。

肥大化した例 (247行)

とりあえず全部書いた

# MyProject CLAUDE.md
## プロジェクト概要
このプロジェクトは2022年に
山田さんが始めて、その後田中
さんが引き継ぎ、2024年に…
(経緯が50行続く)

## 全API仕様
### GET /users
- params: id, name, email
- returns: User object
(API一覧が80行続く)

## コーディングルール
- 変数名はcamelCase
- 関数は小さく書いて
- コメントは日本語で
- エラーハンドリング大事
- 読みやすさ重視で
- がんばって書いてください
(情緒30行)

## TODO
- [ ] ログイン機能
- [ ] パスワードリセット
(TODOリスト40行)

経緯・API全仕様・情緒的指示・TODOが混在。毎ターン全部読まれる。

引き締めた例 (38行)

毎回必要な情報だけ

## Commands

本当に毎回効く情報だけ。詳細は @import で「必要な時に」参照。

もっと詳しく: ユーザースコープ(~/.claude/CLAUDE.md)の良い例

個人の好みなので、プロジェクトに漏らしてはいけない情報はここに。

# ~/.claude/CLAUDE.md

## 言語
- 回答は日本語

## 出力の好み
- コード変更時は必ず diff を先に見せる
- 説明は結論→理由の順
- 絵文字は使わない

## Git
- commit message は日本語可
- Co-Authored-By 署名は残す

## 禁止操作 (全プロジェクト共通)
- `git config` の書き換え
- `--force-with-lease` 以外の force push

ポイントは「プロジェクト特有情報は書かない」。Next.js の話はプロジェクト CLAUDE.md へ。

短く維持するコツ — CLAUDE.md の肥満を防ぐ

CLAUDE.md は放置すると必ず肥大化します。「あ、これも書いておこう」が積み重なって200行を超えた瞬間、劣化が始まります。以下の4ステップをルーチン化しましょう。

1
計測
月1で `wc -l CLAUDE.md`
2
棚卸し
各行「毎回必要?」を問う
3
移動
詳細は @import か docs/ へ
4
削除
効いてない指示は消す

特に step 4 の削除 が最も効きます。「書いた指示がどれだけ守られているか」を Claude の実挙動で検証し、守られてない/毎回自分で言い直してる指示は書き方が悪いか、不要かのどちらか。

CLAUDE.md の行数と指示精度の関係(体感値)
200行を超えると、重要ルールが埋もれて精度が落ちる。経験則。
Tips：古い指示の見つけ方

Claude に「このプロジェクトの CLAUDE.md を読んで、矛盾・重複・時代遅れな行を指摘して」と頼むと、自分では気づかない古い記述が見つかります。半年に1回やるだけで肥満化を防げます。

今すぐ試す：自プロジェクトにCLAUDE.md を書く

読むだけでは身につきません。いま、自分のプロジェクトで手を動かします。15分以内で最初の1枚を書き切りましょう。完璧を狙わない、後で育てる前提でOK。

ハンズオン①：最小CLAUDE.md

自分のリポジトリのルートで以下を実行し、Claudeに自動生成させてから整えます:

cd ~/your-project
claude
> このプロジェクトを調査して、最小限の CLAUDE.md を提案して。
> ルールは「毎ターン必要な情報だけ」「60行以内」「Stack/Commands/Branch/禁止/Style の5見出し」。

Claudeが提案したら、自分の目で各行を「毎回必要?」と問い、Yesのみ残して保存します。

ハンズオン②：既存CLAUDE.mdの減量

既に CLAUDE.md があるプロジェクトなら、減量をお願いします:

> CLAUDE.md を読んで、「毎ターン必要か微妙な行」「TODO 的な一時情報」「長すぎる説明」をリストアップ。
> 削除 or 別ファイルへ移動(@import)の案を、diff形式で提示して。私が承認してから書き換えて。

ポイントは diff 提示→承認→書き換え のフロー。いきなり書き換えさせない(Ch04の安全思想)。

CLAUDE.md は「書く技術」より「書かない技術」。何を諦めて書かないかで品質が決まります。

健全度セルフチェック

いまの自分の CLAUDE.md が健全かを、下の6指標で評価します。3つ以上「△」「×」なら改修のサイン。

CLAUDE.md 健全度レーダー(自己評価の例)
項目は「短さ」「鮮度」「具体性」「スコープ分離」「実測効果」「再利用性」。
指標○(健全)△(黄信号)×(要改修)
行数~100行100-200行200行超
鮮度3ヶ月以内更新半年以内半年以上放置
具体性コマンド/数値明示一部抽象「丁寧に」だらけ
スコープ分離user/project分離一部混在全部project
TODO混入なし2-3個TODOリスト化
実測効果指示が効いてる半分効く無視されてる
もっと詳しく: CLAUDE.md が「効いてるか」の検証法

CLAUDE.md は書いて満足しがちですが、効果検証まで含めてプロの運用です。

  1. A/Bテスト: CLAUDE.md を一時的にリネーム(mv CLAUDE.md _CLAUDE.md)して同じプロンプトで頼む。挙動が変わらない=その指示は元々効いてない
  2. 指示の復唱テスト: 新規セッションで「CLAUDE.md に書かれている主要ルールを3つ挙げて」と質問。出てこない行は埋もれてる
  3. 違反検知: Claude が CLAUDE.md のルールに反した時、その場で「CLAUDE.md の N行目を見て」と指摘。繰り返し違反する指示は表現を書き直す

あなたの CLAUDE.md は、半年後の自分が読んで「これ毎回必要?」と即答できる状態を目指す。これが継続的に機能する唯一の基準です。

Exercise 1
入門

自プロジェクトに最小CLAUDE.mdを1枚書く

15分以内で、自分のGitリポジトリのルートに 60行以内の CLAUDE.md を配置します。

  1. リポジトリのルートに移動し claude を起動
  2. 「このリポジトリを調査して、60行以内のCLAUDE.md 草案を出して。見出しは Stack / Commands / Branch / 禁止 / Style」と依頼
  3. 提案を受け取ったら、各行を「毎回本当に必要?」で選別
  4. 確定版を ./CLAUDE.md として保存、git add CLAUDE.md && git commit -m "chore: add CLAUDE.md"
模範解答を見る

Next.js + pnpm プロジェクトの実例(42行):

# MyApp

## Stack
- Node 22 / pnpm 9
- Next.js 15 (App Router) + TypeScript strict
- Prisma 5 / PostgreSQL 16

## Commands
- dev: `pnpm dev`
- test: `pnpm test` (vitest)
- lint: `pnpm lint`
- typecheck: `pnpm tsc --noEmit`
- db:migrate: `pnpm prisma migrate dev`

## Branch
- main: 本番(直push禁止、PR必須)
- develop: 次期リリース
- PR宛先は develop、main への PR は週1リリース時のみ

## 禁止
- `git push --force` / `--force-with-lease` も事前相談
- `rm -rf` は先に `ls` で対象確認
- 本番DBへの直接接続

## Style
- TS strict, `any` は PR で指摘対象
- 関数は純粋関数を優先、副作用は明示
- エラーは握りつぶさず必ず握る
- 日本語コメントOK、変数名は英語

## 参照
- API詳細: @docs/api.md
- ADR: @docs/adr/

ポイントは「毎ターン使う情報だけ」「具体的なコマンド」「禁止を明示」。API仕様の全文は書かず @import で必要時に参照させる形にしています。

Exercise 2
応用

既存CLAUDE.mdを Claude に減量させる

200行を超える CLAUDE.md があるプロジェクトで、Claude と協働して減量します(ない場合はExercise 1の成果を倍に増やしてから減らす練習として)。

  1. まず wc -l CLAUDE.md で行数を記録
  2. Claude に「CLAUDE.md を診断して:(1)毎ターン不要候補 (2)重複 (3)古い指示 (4)@importに逃がせる詳細 の4観点でリスト化」と依頼
  3. 提案を diff 形式で提示させ、自分で承認/却下
  4. 確定後に再度 wc -l で行数を比較。100行以内になるのが目標
  5. 別セッションで「この CLAUDE.md の主要ルールを3つ挙げて」と試し、重要なルールが残っているか確認
模範解答を見る

減量依頼プロンプトの実例:

> CLAUDE.md を読み込んで、以下4観点で分類してください。結果は表で。
> 1. 毎ターン必要な指示(=残す)
> 2. たまに効けばよい指示(=@docs/guides/*.md に移動)
> 3. 一時情報・TODO(=削除 or 別ファイル)
> 4. 古い/時代遅れな記述(=削除候補)
>
> 各行について「残す/移動/削除」のいずれかと理由を示し、最後に diff パッチを提案。
> 私が承認するまで CLAUDE.md は書き換えないでください。

よくある減量パターン:

  • 「API一覧80行」→ @docs/api.md に退避、本体には「APIルール要点3行」だけ
  • 「プロジェクト経緯50行」→ README.md に移動、CLAUDE.md からは削除
  • 「TODOリスト40行」→ GitHub Issues に移動、CLAUDE.md からは削除
  • 「情緒30行(がんばって等)」→ 全削除

結果: 247行 → 58行。Claudeのレスポンスが早くなり、かつ本当に守ってほしいルール(禁止操作・ブランチ戦略)への遵守率が上がります。1ヶ月後に再測定すると効果が実感できます。
