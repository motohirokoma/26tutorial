GitHub連携: Issues & PR運用

GitHubのIssue管理とPull Request運用をClaudeに任せる方法。バグ報告から実装・PR作成まで一気通貫でこなせます。

所要時間: 1h Phase: Core 難易度: 中級

GitとGitHubって何が違うんだっけ

復習: Git=道具(ローカル)、GitHub=場所(クラウド)。Ch06で扱ったgit操作はローカル中心、ここからは GitHub上の Issue / PR / Actions を扱います。

普段の開発フローと Claude の出番
graph LR
  A["Issue 作成"] --> B["ブランチ作業"]
  B --> C["PR 作成"]
  C --> D["レビュー"]
  D --> E["マージ"]
  E --> F["Issue クローズ"]
  A -.Claude.-> A1["要件整理"]
  B -.Claude.-> B1["実装支援"]
  C -.Claude.-> C1["/pr 自動生成"]
  D -.Claude.-> D1["/review"]
各ステップにClaudeを差し込めると、レビュー前のセルフチェックが格段に速くなる。

gh CLI の準備

Claude が GitHub を扱うには gh CLI が必要です。

# macOS
brew install gh
# Linux
sudo apt install gh
# 認証
gh auth login

確認:

gh auth status
gh repo view  # 現在のリポを表示
Claudeへ伝えるとき

Claude は gh が入っていれば自動で使います。なければ「gh を入れて」と促されます。

Issueの操作

Issue の一覧・要約

> gh issue list で未対応Issueを取得して、優先度順に並び替えて要約して

新規 Issue を立てる

> 「ログイン後にリダイレクトされない」バグ報告のIssueを立てて。
再現手順は… ブラウザはChrome 120、影響: 全ユーザー、と書いて。
良いIssueの構成
この型に沿って書かれたIssueは、修正までの時間が半分以下になる。

Issue → 実装の起点

Issueを起点に作業するのがClaudeとの相性◎。

> Issue #87 を読んで、要件を整理してください。
合意できたら、実装計画を立てて、私の承認後にコーディング開始。

Pull Requestの実践運用

PR テンプレート

リポジトリのルートに .github/PULL_REQUEST_TEMPLATE.md を置くと、Claudeはそれに沿って書きます。

.github/PULL_REQUEST_TEMPLATE.md
## 概要

## 変更内容
-

## レビュアーへの注意点

## チェックリスト
- [ ] テスト追加
- [ ] ドキュメント更新

PR レビューを受ける/する

受ける側

あなたが PR 作成者

  • 「gh pr view 142 でレビューコメントを取得して、対応方針を出して」
  • 「指摘1番だけ修正して、コミットして」
  • 「修正完了をPRにコメントして」
する側

あなたが レビュアー

  • 「gh pr diff 142 を読んで /review
  • 「行コメントは gh pr review --comment で送って」
  • 「OKならapprove、修正必要ならrequest changes」

よくあるPR運用パターン

パターン A
小さいPRを高頻度
1機能=1PR、200行以内目安。レビューが速くマージ事故が起きにくい。
パターン B
Draft PR で会話
未完成でも gh pr create --draft。早期にチームと方向性をすり合わせる。
パターン C
Stacked PR
A → B → C と積む。各PRが独立にレビュー可能。Claudeはブランチ管理が得意。

マージ戦略

戦略履歴の見え方こういう時
Merge commitマージコミットが残る機能の塊感を残したい
Squash1コミットに圧縮履歴を綺麗に保ちたい
Rebase線形履歴1コミット=1論理変更にこだわるチーム
もっと詳しく: PR間の依存関係を伝える

PR本文に Depends on #143 や Closes #87 と書くと、GitHubが自動でリンク。Claudeに 「PRに `Closes #87` を追加して」と頼めば即書き加えてくれます。Issueのクローズ漏れも減ります。

Exercise 1
入門

Issue から PR まで一気通貫

適当なリポで「軽微なtypo修正」のIssueを立て、それを起点に実装→PR作成までやってみましょう。

  1. 「README.md に typo があるのでIssue立てて」
  2. 「Issue #N を起点にブランチ作って、修正実装」
  3. 「テスト無しで /pr。 PR本文に Closes #N
  4. 「gh pr view でリンクが付いてるか確認」
模範解答を見る

Claudeにこう頼む流れ:

> README の "Quick Start" セクションに "thier" → "their" のtypo。
Issueタイトル "fix: README typo on Quick Start"、本文に該当行 を含めて gh issue create で立てて。

<Issue #45 作成>

> Issue #45 起点。fix/readme-typo ブランチで修正、/pr で PR 作成。 本文に "Closes #45" を入れて。

<PR #46 作成>

マージすると Issue が自動クローズ。これだけで「日常のtypo修正」が3分で完了します。

Exercise 2
応用

PRレビュー対応をフルサイクルで

あなた(or Claude)がPRを作って、別キャラとしてClaudeにレビューさせ、指摘を受けて修正、再レビュー → approveまで運用シミュレーション。

  1. 適当な実装PRを作る
  2. 「シニアエンジニアの視点で /review
  3. 「High優先度の指摘だけ対応するパッチを作って push」
  4. 「再度 /review してapprove判定」
  5. squash mergeでクローズ
模範解答を見る

会話例(抜粋):

> /review で、シニアの目線で。指摘はHigh/Mid/Lowタグつきで。
<指摘 6件>

> High だけ修正、テストも更新して push。
<3 ファイル修正、push>

> 再度 /review。修正で残った懸念があれば指摘、なければ approve判定の文言で締めて。
<残懸念なし、approve>

> gh pr merge 142 --squash で merge。Issueが自動クローズしたか確認。

1人開発でも「自分でレビュー、自分で対応」のサイクルを意図的に回すと、品質が一段安定します。
