CLAUDE.md (本章固有部分)

# Tasq

## API 設計原則
- すべて /api/* 配下、レスポンスは { data, error } 形式固定
- 認証必須エンドポイントは getServerSession で uid を取得し、必ず uid でWHERE
- 楽観ロック: updatedAt を返り値に含め、PATCHは前回updatedAtを必須とする

## DB 命名
- テーブル: 単数 PascalCase (User / Task)
- カラム: camelCase
- 外部キー: <参照先>Id (例: userId)

## NG
- prisma migrate reset を本番DBで叩かない
- API ルートで認証チェックを忘れない
- console.log を本番ビルドに残さない (Lint で落とす)

Step 2 — DBスキーマ設計

NextAuth が User/Account/Session を要求するので、それに Task を追加。

// prisma/schema.prisma
datasource db { provider = "postgresql"; url = env("DATABASE_URL") }
generator client { provider = "prisma-client-js" }

model User {
  id    String  @id @default(cuid())
  email String  @unique
  name  String?
  image String?
  tasks Task[]
  accounts Account[]
  sessions Session[]
  createdAt DateTime @default(now())
}

model Task {
  id         String    @id @default(cuid())
  title      String
  notes      String?
  done       Boolean   @default(false)
  dueAt      DateTime?
  priority   Int       @default(0)   // 0 = none, 1 = low, 2 = mid, 3 = high
  userId     String
  user       User      @relation(fields: [userId], references: [id], onDelete: Cascade)
  createdAt  DateTime  @default(now())
  updatedAt  DateTime  @updatedAt
  @@index([userId, done, dueAt])
}

// (Account / Session / VerificationToken は NextAuth テンプレ通り)

Step 3 — API ルートを Claude に書かせる

Claudeへの指示は1エンドポイントずつ。CLAUDE.mdに API設計原則を書いておけば、毎回明示しなくて済みます。

app/api/tasks/route.ts に GET (一覧取得 user限定) と POST (作成) を実装して。GET は ?view=today|upcoming|inbox|done でフィルタ。レスポンスは { data, error }。 — Claudeへのプロンプト例

生成される API ルート

// app/api/tasks/route.ts
import { getServerSession } from 'next-auth';
import { authOptions } from '@/lib/auth';
import { prisma } from '@/lib/db';
import { NextResponse } from 'next/server';

export async function GET(req: Request) {
  const session = await getServerSession(authOptions);
  if (!session?.user?.id) return NextResponse.json({ error: 'unauth' }, { status: 401 });

  const { searchParams } = new URL(req.url);
  const view = searchParams.get('view') ?? 'today';
  const userId = session.user.id;

  const now = new Date();
  const todayEnd = new Date(now); todayEnd.setHours(23,59,59,999);

  const where: any = { userId };
  if (view === 'today')    Object.assign(where, { done: false, dueAt: { lte: todayEnd } });
  if (view === 'upcoming') Object.assign(where, { done: false, dueAt: { gt: todayEnd } });
  if (view === 'inbox')    Object.assign(where, { done: false, dueAt: null });
  if (view === 'done')     Object.assign(where, { done: true });

  const data = await prisma.task.findMany({
    where,
    orderBy: [{ priority: 'desc' }, { dueAt: 'asc' }, { createdAt: 'desc' }],
  });
  return NextResponse.json({ data });
}

export async function POST(req: Request) {
  const session = await getServerSession(authOptions);
  if (!session?.user?.id) return NextResponse.json({ error: 'unauth' }, { status: 401 });
  const body = await req.json();
  const data = await prisma.task.create({
    data: { ...body, userId: session.user.id },
  });
  return NextResponse.json({ data }, { status: 201 });
}
セキュリティの肝

常に userId でWHEREすること。ID直打ちされたら他人のタスクを見れるバグ (BOLA) が頻発します。CLAUDE.mdに書いてあれば、Claudeも自動で守ってくれます。

Step 4 — Google OAuth 認証

OAuth ログインフロー
sequenceDiagram
  participant U as User
  participant N as Next.js
  participant G as Google
  participant DB as Postgres
  U->>N: "Sign in with Google"
  N->>G: 認可リクエスト
  G->>U: 同意画面
  U->>G: 同意
  G-->>N: 認可コード
  N->>G: トークン交換
  G-->>N: id_token (email/name)
  N->>DB: User upsert
  N-->>U: セッションCookie
NextAuth が定形フローを全部隠蔽。書くのは設定3項目。
// lib/auth.ts
import GoogleProvider from 'next-auth/providers/google';
import { PrismaAdapter } from '@auth/prisma-adapter';
import { prisma } from './db';

export const authOptions = {
  adapter: PrismaAdapter(prisma),
  providers: [
    GoogleProvider({
      clientId: process.env.GOOGLE_CLIENT_ID!,
      clientSecret: process.env.GOOGLE_CLIENT_SECRET!,
    }),
  ],
  callbacks: {
    async session({ session, user }) {
      if (session.user) (session.user as any).id = user.id;
      return session;
    },
  },
};
.env の管理

GOOGLE_CLIENT_ID / GOOGLE_CLIENT_SECRET / NEXTAUTH_SECRET / DATABASE_URL は .env.local へ。絶対に commit しない (Ch09 の deny で守られている前提)。

Exercise 1
入門

curl で Tasq APIを叩く

本章の手順でローカル起動した状態で、curlを使って3つのAPI呼び出しを成功させてください。

  1. ブラウザで Google ログイン → DevTools の Application タブから Cookie の next-auth.session-token をコピー
  2. curlで作成: curl -X POST http://localhost:3000/api/tasks -H "Cookie: next-auth.session-token=<...>" -H "Content-Type: application/json" -d '{"title":"テスト"}'
  3. curlで一覧: curl 'http://localhost:3000/api/tasks?view=today' -H "Cookie: ..."
  4. curlで完了: curl -X PATCH http://localhost:3000/api/tasks/<id> -H "Cookie:..." -d '{"done":true}'
模範解答を見る

うまくいかない時のチェック:

  • Cookie が古い → 再ログインして取り直し
  • 401 → getServerSession がCookieを認識しない (Hostヘッダ確認)
  • 500 → サーバ側コンソールに Prisma エラーが出ているはず

Exercise 2
応用

BOLA 脆弱性を Claude にテストさせる

「他人のタスクを更新できないこと」を保証する自動テストを Vitest で書いてください。Phase 2 のテスト章 (Ch08-2) の知識を活用します。

  1. tests/api/tasks.test.ts を作成
  2. 2人のユーザー (Alice, Bob) を seed
  3. Alice のタスクを Bob のセッションで PATCH しようとする
  4. 404 か 403 が返ることを assert
  5. npm test で通すまでデバッグ
模範解答を見る 
// tests/api/tasks.test.ts
import { describe, it, expect, beforeEach } from 'vitest';
import { prisma } from '@/lib/db';
import { PATCH } from '@/app/api/tasks/[id]/route';

describe('PATCH /api/tasks/[id]', () => {
  let aliceId: string, bobId: string, taskId: string;

  beforeEach(async () => {
    await prisma.task.deleteMany();
    await prisma.user.deleteMany();
    const a = await prisma.user.create({ data: { email: 'a@x.com' } });
    const b = await prisma.user.create({ data: { email: 'b@x.com' } });
    const t = await prisma.task.create({ data: { title: 'A only', userId: a.id } });
    aliceId = a.id; bobId = b.id; taskId = t.id;
  });

  it('Bob cannot update Alice task', async () => {
    const res = await PATCH(
      new Request(`http://x/api/tasks/${taskId}`, {
        method: 'PATCH', body: JSON.stringify({ done: true }),
      }),
      { params: { id: taskId } },
      bobId  // モックされたsession user id
    );
    expect(res.status).toBe(404);
  });
});

セッションのモック方法はテストフレームワーク次第。NextAuthをまるごとモックするか、API側を userId 引数で抽象化しておくと書きやすいです。
