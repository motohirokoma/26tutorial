Subagentsで並列思考

独立タスクを同時に飛ばして、時間を1/5に圧縮する

所要時間: 60分 Phase: Advanced 難易度: 中級
この章のゴール

Claude CodeのAgent tool(subagent呼び出し)の基本を押さえ、「独立したタスクは並列に飛ばす」という思考様式を身につけます。5観点レビュー(セキュリティ/パフォーマンス/可読性/テスト/設計)を同時並列で走らせるハンズオンを通して、直列処理では絶対に味わえない時短の快感を体験しましょう。ここがPhase 3 Advanced の入口、かつ上位20%の入場券です。

バージョン注記

本章の記述は2026年1月時点のClaude Code (Agent SDK/CLI) 挙動に基づきます。subagent_typeのラインナップやAgent tool のパラメータ名は進化するため、コマンドが通らない時は公式ドキュメントを必ず確認してください。

Subagentとは — 分身を送り出すイメージ

Subagentは、いまあなたと会話しているClaude(メインエージェント)が、Agent toolを通して起動する分身です。子は独立した会話履歴(context window)を持ち、タスクを終えたら要約1つを親に返して消えます。親は要約だけ読んで続きを判断する——これが基本モデルです。

メインエージェント vs Subagent の関係
親は子の会話履歴を見ない。子が返す「要約」だけを材料に次の判断を行う。

通常のRead/Grep/Bashツールが「手の延長」(親が指示してすぐ結果が返る)だとすれば、Subagentは「脳の複製」(独立に考えて結論だけ持ち帰る)です。この違いが、独立な複数タスクで桁違いの時短を生みます。

もっと詳しく: なぜcontextを分けるのか

メインClaudeの文脈はユーザとの対話履歴・既に読んだコード・TodoWriteなどで既に埋まりがち。そこにさらに「競合ライブラリ3本の仕様書」を詰め込むと、肝心の本題に割ける注意力が削れます。Subagentは独立contextで専念し、要約だけ親に返すことで、親のcontext窓を保護するのが最大の価値です。Ch14「コンテキスト管理」と合わせて読むと腑に落ちます。

Agent toolの呼び出し構造

Claude CodeにはAgentという内部ツールがあり、自然言語で「subagentを起動してほしい」と頼むと、Claudeがこれを使って子を立ち上げます。呼び出しの最小JSONはこんな形です:

{
  "tool": "Agent",
  "input": {
    "subagent_type": "general-purpose",
    "description": "セキュリティ観点のレビュー",
    "prompt": "リポジトリ直下の src/auth/ を読み、SQLインジェクション・XSS・認証バイパスの観点で\n気になる箇所を列挙してください。各指摘は【ファイル:行】【症状】【修正案】の3点セットで。\n最大10個、優先度順で。"
  }
}

この呼び出しで重要なのは3つのフィールドだけです:

フィールド役割書き方のコツ
subagent_type使うエージェントの種類general-purpose or Explore(後述)
description3-5語の短いラベルUIの実行ログで識別するため。日本語可
prompt子に渡す依頼文(全て)子は親の会話を見えない前提で、自己完結的に書く
公式ドキュメント必読

ここは公式ドキュメントにも書いていますし、実務で間違えると致命的(Agent呼び出しが成立しない／権限境界を誤解する)ので必ずマスターしてください。Claude Code 公式 - Sub-agents →

subagent_typeの使い分け

2026年1月時点、日常使いで覚えるべきタイプは2つだけです。

Explore

読むだけの調査特化

  • Read/Grep/Glob/WebFetch等読み取り系のみ
  • Write/Edit/Bashは不可(破壊リスクゼロ)
  • コードベース把握・仕様調査・類似事例収集に最適
  • 軽量で速い、並列に多数飛ばせる
general-purpose

書き換えもできる万能

  • ほぼ全ツールが使える(Edit/Write/Bashも)
  • タスクを完遂するまで子の中で試行錯誤
  • 独立した修正ブランチ・リファクタ・テスト追加に
  • 重い・長い・コスト高、数は絞る

迷ったらExplore。調査・把握・レビューなど「読むだけで結論が出る」タスクなら迷わずExplore。general-purposeはどうしても書き換えが要る時のリザーブ武器、くらいの感覚でちょうどいい。

もっと詳しく: カスタムsubagentとの関係

Claude Codeは.claude/agents/配下に自作のsubagent定義(Markdown + frontmatter)を置けます。例えばsecurity-reviewer.mdを作ればsubagent_type: "security-reviewer"として呼べるようになり、毎回長いプロンプトを書かずに済みます。これは専門編Subagents Masteryで深掘ります。

並列 vs 直列 — 体感で時短を掴む

ここが本章の核心。5観点レビューを直列でやるか並列でやるか、でどれだけ壁時計時間が変わるかを数字で見ます。

壁時計時間:5観点レビュー(各観点90秒と仮定)
5倍速。5観点が互いに独立なので、並列の律速は「一番遅い1つ」。これが時短の最大レバー。
ここがプロの境目

「このタスク同士は独立か?」を真っ先に判定し、独立ならsubagentで並列に飛ばす。これが時短の最大レバーです。直列なら5分の仕事が、並列なら1分で終わる——この差は1日何度も効いてきます。
試し方: 今日これからやるタスクで「実は独立な3-5個」に割れるものを1つ探し、Subagentを並列で飛ばしてみてください。

「独立か?」を判定する3質問

並列化の肝は依存関係の見極め。以下のフローで「独立」と判定できるものだけ並列化します。

並列化の判定フロー
graph TD
  Start([タスクA,B,Cが手元にある]) --> Q1{"Bの結果がAの入力に必要?"}
  Q1 -- Yes --> Serial[直列で順番に実行]
  Q1 -- No --> Q2{"同じファイルを書き換える?"}
  Q2 -- Yes --> Serial
  Q2 -- No --> Q3{"結果を集約する明確な窓口がある?"}
  Q3 -- No --> Serial
  Q3 -- Yes --> Parallel["並列でsubagentを同時launch"]
3つ全てYes(独立・書き換え無し・集約先明確)なら並列。どれか1つでもNoなら直列。
タスクセット判定理由
5観点コードレビュー並列◯各観点はread-only、互いに独立
3ライブラリの比較調査並列◯独立調査、集約は親が表で統合
設計→実装→テストの3段直列✕前段の成果物が次段の入力
同一ファイルへの複数修正直列✕書き込み競合で破壊的
10ファイルへのPrettier適用並列◯ファイル単位で独立、集約はgit diff

コスト感 — トークンはどう消費されるか

並列は時短の王様ですが、トークン消費は直列とほぼ同じか、やや多い(各子が独立に自己紹介プロンプトを読むため)。壁時計を買うためにコストを少し払う、という感覚で合っています。

相対コスト:観点数による推定トークン・時間
5×
壁時計の短縮(5並列時)
↑ 時間回収
1.1×
トークン増(並列/直列比)
→ ほぼ誤差
~10本
実用的な同時並列の上限
→ 余裕
もっと詳しく: general-purposeを5並列する時の注意

Exploreは読むだけなので5-10並列しても怖くありませんが、general-purposeを5並列すると5つのエージェントが同時に思考・編集を始めるため、消費が大きい。まずExploreで地図を引いてから、書き換えフェーズは直列 or 2-3並列に留めるのが安全。cost感覚は/costで毎日確認する習慣を(Ch01参照)。

結果の集約 — 子の声を親がまとめる

並列で飛ばすと、各子から要約が戻ってきます。そこで親(メインClaude)の仕事は「整形と統合」。以下のパターンが鉄板:

  • 観点別比較表で集約 — 観点/指摘数/優先度/修正見積 の列で横串に並べる
  • 優先度順の統合リスト — 各子の最重要3件だけ抜き、全体で15件に絞る
  • 矛盾の検出 — 子どうしで結論が割れた箇所を親がハイライト
  • 次アクションの明示 — 「誰が・どこから手を付けるか」1行で閉める

集約時に親が子の会話履歴を欲しがったら、それは「要約が薄い」か「prompt設計が粗い」サインです。親が追いかけ質問をするくらいなら、最初からpromptに「各指摘は【ファイル:行】【症状】【修正案】の3点セットで返せ」と明記した方が早い。

今すぐ試す:5観点並列レビュー

所要10分。自分のリポジトリ(任意のもの、小さい方が見通しやすい)で並列レビューを体験してみましょう。

1
対象指定
レビュー対象ディレクトリを1つ決める
2
発注
下記プロンプトをそのまま貼る
3
観察
5並列が同時走行するログを見る
4
集約
親が統合した表を受け取る
次のタスクを、5つのExplore subagentを「並列」で同時に起動して実行してください。

対象: {自分のリポのディレクトリパス、例 src/}
観点(各観点1つの子に割り当て):
  1. セキュリティ(SQLi/XSS/認証/機密ログ漏洩)
  2. パフォーマンス(N+1/不要な再計算/メモリリーク)
  3. 可読性(命名/関数長/ネスト深さ)
  4. テスト(カバレッジの穴/境界値不足)
  5. 設計(単一責任/依存方向/重複)

各子の出力形式(統一):
  - 【ファイル:行】【症状】【修正案】の3点セットで最大5件、優先度順
  - それ以上見つからなければ「指摘なし」と明記

集約(親の仕事):
  - 5観点 × 優先度の比較表で提示
  - 最優先3件だけ、具体的な着手手順を追記
  - 観点間で矛盾する指摘があればハイライト
観察ポイント

実行中、Claude Codeは5つのAgent呼び出しを同時にlaunchします。ログ上で5つの「Running...」が並走するのが見えるはず。終了時間はほぼ揃い、律速は一番遅い1子。直列なら5倍かかる仕事が、壁時計で1/5になる瞬間を体感してください。

並列化のアンチパターン

やりがちNG

直列タスクを無理やり並列化

  • 「設計→実装→テスト」を3並列にしてしまう
  • 同ファイルを3子が同時Edit(最後が勝つ)
  • 子のpromptに「親の会話を前提」の文言
正しい型

独立タスクだけ並列

  • 独立3質問を同時に、結果だけ親で統合
  • 書き換えが絡むならファイル単位で排他
  • 子promptは自己完結で書く(状況を全て埋め込む)

子は親の会話を見ていない——この一点を忘れると、promptを「続き文脈ありき」で書いてしまって、子が意味不明な挙動をします。子に渡すpromptは、他人に初めてお願いするメールだと思って書きましょう。

もっと詳しく: 「独立っぽく見えて実は依存」の典型3パターン
  1. 同一ファイルの異なる関数の修正 — 書き込み競合。行番号がズレて互いに壊し合う。
  2. 片方が生成したファイルを相手が読む — 完全依存。直列または親が中継。
  3. 共有DBへの並行書き込み — トランザクションが競合。SQLなら1つのトランザクションで

判断に迷ったら「両方を同時に他人2人に頼んで、互いに知らなくても成立するか?」を自問してください。成立するなら独立、しないなら直列。

Exercise 1
入門

「独立か?」判定練習

以下5つのタスクセットから、並列化可能なものと直列のものを分類してください。各判定に1行の理由を添えてください。

  1. 社内3つのライブラリ(A/B/C)のREADMEを要約して比較
  2. ある関数をリファクタ → そのテストを追加 → PR本文を書く
  3. 5ファイルそれぞれにblackを実行してフォーマット
  4. Day1のログを整形し、その結果でDay2の異常を検出
  5. 3つの候補アルゴリズムの計算量を別々に調査
模範解答を見る
  1. 並列◯ — 独立調査、書き換え無し、集約は比較表
  2. 直列✕ — 成果物が次段の入力(リファクタ→テスト→PR本文)
  3. 並列◯ — ファイル単位で独立、書き換えも排他的
  4. 直列✕ — Day1の結果がDay2の検出入力
  5. 並列◯ — 独立調査の典型、集約は親が表で統合

判定ポイント: 「一方の結果が他方の入力か?」「同じファイルに書き込むか?」「集約先が明確か?」の3質問を通してYes/Yes/Noのどれかに該当したら直列、全てクリアなら並列。

Exercise 2
応用

自分のリポで5観点並列レビューを実行

Section 8のプロンプトテンプレを使い、自分の任意プロジェクトで実際に5並列レビューを走らせてください。そのあと以下を記録します。

  1. 壁時計時間(全子が揃うまで) vs もし直列だったらの見積時間
  2. 5子のうち、最も重い指摘を出したのはどれか(観点名)
  3. 矛盾する指摘はあったか、あったならどれ同士か
  4. 今後「毎週このパターンで回す」とした場合、カスタムsubagent化したい観点はどれか
模範解答を見る

記録例(小規模Node.jsリポで筆者が実施した場合):

  • 壁時計 120秒 / 直列見積 580秒 → 約4.8倍速
  • 最重指摘:「設計」(循環依存が2箇所)。次点は「テスト」(境界値不足)
  • 矛盾1件:「可読性」は関数分割を推奨、「パフォーマンス」は関数呼び出しオーバヘッドを指摘 → 親判断で可読性を優先
  • カスタム化候補:「セキュリティ」(毎週同じ観点・同じ出力形式でいい)。.claude/agents/security-reviewer.mdを作って、以後subagent_type: "security-reviewer"で一撃呼び出し

この記録を残すと、自分の弱点観点が見える化します。セキュリティ指摘が毎週出るなら、それはあなたの設計クセ。習慣化レビューが効く所以です。
