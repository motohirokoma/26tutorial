デバッグ：バグを直させる頼み方

「動かない」と投げるな。3点セットで渡し、仮説3つから始めさせる。

所要時間: 45min Phase: Core 難易度: 中級
2026年1月時点の情報

本章は 2026年1月時点の Claude Code (Opus 4.7) の挙動を前提に書いています。エラー再現能力やWeb取得仕様はアップデートで進化する可能性があるため、最新仕様は 公式ドキュメント で確認してください。

なぜ「動きません」では直らないのか

デバッグを頼む時にやりがちな悪手の筆頭が、「動きません、直してください」と投げること。再現環境 が無いままClaudeはコードを眺めて推測で直し始める。結果、関係ない場所を書き換えたり、真の原因ではなく症状だけ塞いだコードが返ってくる。

熟練のエンジニアは、バグ修正依頼に最低限これを渡す — 再現手順 / 期待動作 / 観察結果。3点セットと呼ぶ。これが揃うだけで、Claudeの命中率は体感で2〜3倍になる。

3点セット — バグ報告の基本形
この3点が揃って初めて、Claudeは「あてずっぽう」を卒業する。
Before(死亡パターン)

「ログインが動きません。直して。」

  • 再現手順なし
  • 期待動作なし
  • エラーログなし
After(3点セット)

「ログインで TypeError: Cannot read 'id'」

  • 再現: /login → email/password 送信
  • 期待: /home にリダイレクト
  • 観察: Console に TypeError、Network は200

エラーメッセージの正しい渡し方

観察結果の核は スタックトレース と エラーメッセージ全文。要約せず、丸ごと貼るのがコツ。「TypeErrorっぽいの」ではなく、最低10行コピペする。

  • スタックトレースは丸ごと — 「at xxx.js:42」の1行目だけでなく、最低5-10段までコピペ。どの関数から来たかが原因特定の鍵
  • ブラウザ系ならNetwork/Console両方 — Console のエラー + Network タブの失敗リクエスト(status/response body)の両方を渡す
  • サーバ系は前後20行も — エラー行だけでなく、その直前のログも文脈になる
  • 「いつから」情報も添える — 「昨日まで動いていた」「v1.3.2にアップデートしてから」— これだけで git log や diff から疑わしい変更を絞り込める
公式ドキュメント必読

Claude Code はログやエラー全文を大量に渡してもきちんと読み解きます。ただし巨大ログは コンテキストを食うので、関連部分に絞ってから貼る習慣を。Claude Code 公式 →

今すぐ試す #1：エラー貼り付けテンプレ

## バグ報告

### 再現手順
1. `npm run dev` でローカル起動
2. http://localhost:3000/dashboard を開く
3. 「保存」ボタンを押す

### 期待動作
- トースト "保存しました" が出る
- DB の records テーブルに新しい行が追加される

### 観察結果
- トーストは出ない
- Network タブで POST /api/records が 500 を返している
- サーバログ(最新20行を貼り付け)：

```
2026-01-18 10:32:11 [error] TypeError: Cannot read properties of undefined (reading 'id')
    at createRecord (src/api/records.ts:42:18)
    at async Object.handler (src/api/records.ts:15:12)
    at async /node_modules/next/...
```

### 追加情報
- 昨日まで動いていた
- 昨日のコミット: abc1234 (feature/user-auth マージ)

## 依頼
まず原因候補を3つ挙げてから、最も可能性の高いものから検証する手順を書いてください。
修正はそのあと頼みます。

仮説 → 検証 → 記録のループ

3点セットが揃ったら、いきなり修正させず 仮説→検証→記録のループを回す。これが本章で一番伝えたいこと。

デバッグの黄金フロー
graph LR
  A([3点セット]) --> B["仮説を3つ出す"]
  B --> C{"最有力から検証"}
  C -- "当たり" --> D["原因確定"]
  C -- "外れ" --> E["次の仮説へ"]
  E --> C
  D --> F["最小修正"]
  F --> G["再現手順で再テスト"]
  G --> H["学びを記録"]
「当てずっぽう」ではなく、候補を並べて一つずつ潰す。これが科学的デバッグ。
ここがプロの境目

「仮説3つ立ててから直して」と頼む。Claudeに直接修正をやらせる前に、原因候補を3つ挙げさせ、優先度と検証方法を添えさせる。こうするとClaudeは表層だけを塞がず、本質に向かう。「null チェック追加」「APIレスポンス型変更」「auth middleware漏れ」のような異なる抽象レイヤーの仮説が並ぶと、どこを直すべきか人間側も判断できる。
試し方： 次にバグ報告する時、プロンプト末尾に「まず原因候補を3つ挙げてから、最も可能性の高いものから修正」と1行だけ追加してみて。

今すぐ試す #2：仮説ドリブンプロンプト

以下のバグについて、いきなり修正せず：

1. 原因候補を3つ挙げてください（異なる抽象レイヤーで）
2. それぞれの「もしこれが原因なら何が観察されるはず」を書いてください
3. 最も検証が安いものから順に並べてください
4. 最優先の仮説を検証する Bash コマンド or Read する箇所を提示してください

（ここに3点セット）
もっと詳しく: なぜ「3つ」なのか — 1つや5つではダメか

1つだと直感に引きずられる。最初に頭に浮かんだ仮説だけで走ると、それが外れた時に方向転換の気力がなくなる。5つ以上だと雑になる。無理やり数を揃えようとしてあり得ない候補を混ぜる。

3つが「複数の選択肢を比較する」下限で、かつ「質を保ったまま出せる」上限。経験的にも研究的(意思決定研究)にも妥当なバランス。

Claudeにとっても「3つ出せ」という制約は強力で、思考の幅を強制的に広げる。1つだけだと最頻出の原因しか出さないが、3つと言うと構造的に違う層(DB / アプリ / 環境)から出してくれる。

典型例：window.load 系の落とし穴

本章の締めは window.load / DOMContentLoaded 系の頻出バグ。Claudeに Chart.js や Mermaid の図を書かせた時、「描画されない」という症状で必ず一度はハマる。

なぜ Chart.js が描画されないか
sequenceDiagram
  participant M as main 内 inline script
  participant L as body末尾の chart.js
  participant C as Chart クラス
  M->>C: new Chart(ctx, ...) 呼び出し
  Note over M,C: window.Chart はまだ未定義
  M-->>M: 早期return（図描画されず）
  L->>C: Chart クラス読み込み完了
  Note over L,C: もう誰も呼ばない
body末尾のライブラリ読み込みより、main内のinline scriptが先に実行される。典型的な時系列バグ。
症状誤った修正(表層)正しい修正(本質)
Chart.js の canvas が空白 if (!window.Chart) return; で早期return window.addEventListener("load", () => { ... }) で包む
Lucideアイコンが文字のまま アイコン名を疑う 動的挿入後に lucide.createIcons() を再実行
rough.js 図が出ない optionを変える SVG要素DOM挿入 → スクリプト即時実行 に変更
Mermaid が文字列のまま CSSを疑う ラベルの特殊文字を ["..."] でクォート
もっと詳しく: このパターンの修正コード例

Chart.js の例。Bad:

<script>
(function() {
  var ctx = document.getElementById('ch12-chart-1');
  if (!ctx || !window.Chart) return;  // ← ここで早期return、図描画されない
  new Chart(ctx, { ... });
})();
</script>

Good:

<script>
window.addEventListener("load", function() {
  var ctx = document.getElementById('ch12-chart-1');
  if (!ctx || !window.Chart) return;
  new Chart(ctx, { ... });
});
</script>

loadイベントは全リソース読み込み後に発火するので、その時点ではChart.jsは必ず利用可能。DOMContentLoadedでも不十分(body末尾scriptはまだ評価中の可能性)なのでloadを使う。

記録する：次のバグは過去のバグから学ぶ

Exercise 1
入門

3点セットを整える

次の雑な依頼文を、3点セットに書き直してください(Claudeに送る前に自分で整理する練習)。

「ログインボタン押しても何も起きないんだけど。直して。」
  1. 不足している情報は何か列挙する
  2. 仮に自分のプロジェクトで起きたと想定し、3点セット形式で書き直す
  3. 最後に「原因候補を3つ挙げて」と付け足す
模範解答を見る

不足情報: 環境(ブラウザ/OS)・再現手順・期待動作・観察結果(console/networkログ)・いつから発生したか。

書き直し例:

## 再現手順
1. Chrome 131 / macOS 15 で https://myapp.local を開く
2. email: test@example.com / password: password123 を入力
3. 「ログイン」ボタンをクリック

## 期待動作
- /dashboard にリダイレクト
- ヘッダーにユーザ名が出る

## 観察結果
- ボタンは押せた(disabled 外れる)
- URL は変わらず /login のまま
- Chrome Console: (エラーなし、警告もなし)
- Network タブ: POST /api/login は送られていない(リクエスト自体が発火していない)

## 追加
- 昨日 tailwind を v3 → v4 にアップデートした
- フォームは <form onSubmit={handleSubmit}> で書いている

## 依頼
まず原因候補を3つ挙げてから、最も検証が安いものから順に検証手順を提示してください。修正はそのあと頼みます。

ポイント: 「Networkにリクエストが飛んでない」は強いヒント(handleSubmitが呼ばれていない可能性)。この情報があるだけで仮説の質が跳ね上がる。

Exercise 2
応用

自分の環境で仮説3つ運用をやる

自分が今扱っているプロジェクトで、過去1週間で遭遇したバグを1つ思い出す(軽微でOK)。それを題材に、Claude Code でこのフローを実体験する。

  1. 3点セットでバグを書き起こす(再現/期待/観察)
  2. Claudeに「原因候補3つ + 検証順」を依頼(修正はまだ頼まない)
  3. 返ってきた3つの仮説を見て、自分の直感と合うもの/違うものを評価
  4. 最も安い検証から実行(コマンド実行/Read)
  5. 原因が確定したら「最小の修正を提案して」と依頼
  6. CLAUDE.md か docs/lessons.md に学びを1-3行で記録
模範解答を見る

このExerciseは「やってみる」が本体なので模範解答は流れの例示。

実例(筆者の体験):

  • バグ: Next.js プロジェクトで画像が本番だけ404
  • 3点セットで投げた: next.config.js の images設定、Vercel のBuild Logs、ローカル/本番の比較スクショ
  • Claudeの3仮説: (1)basePath設定ミス (2)画像パスの絶対/相対の不一致 (3)Vercelキャッシュ
  • 検証順: (3)キャッシュは再デプロイで即判定(一番安い) → 変わらず → (1)next.config.js Read → basePath設定あり、ローカルで再現できたので確定
  • 修正: 画像src を next/image で basePath を自動付加する形に統一(2箇所の変更)
  • 学び: docs/lessons.md に「basePath を設定したら全<img>を next/image 経由にする」と1行追記

学びのコツ: 1週間後にまた読むので、3秒で意味がわかる1行に研ぐ。「困った話」ではなく「今後のチェックリスト」として書く。
