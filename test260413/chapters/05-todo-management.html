TodoWriteで作業管理

3ステップ以上の仕事は「見える化」から始める

所要時間: 30分 Phase: Basics 難易度: 入門
この章のゴール

TodoWrite というツールを使って、Claude Code に「計画を立ててから実行してもらう」感覚を身につけます。これをやると長い作業でも途中で迷子にならないし、あなたも進捗を目で追えるようになる。30分で、TodoWriteがいつ効くか/いつ不要か/どの粒度で書くべきかを押さえて、今日から自分の仕事に持ち込めるようにしましょう。

バージョン注記

本章の記述は2026年1月時点の Claude Code 挙動に基づきます。TodoWriteの状態名(pending/in_progress/completed)やフィールド仕様は進化する可能性があるため、動作が想定と異なる場合は公式ドキュメントを確認してください。

TodoWriteとは — 一言で

TodoWriteは、Claude Code が自分の頭の中で「これから何をやるか」をリスト化して外に出すためのツールです。プロジェクト管理ツール(Jira, Notion 等)ではなく、その会話のためだけの一時的なチェックリストだと考えてください。作ったリストはUIに表示され、進捗がリアルタイムに更新されていきます。

TodoWriteの役割 — 頭の中→見える化
リストを「外に出す」ことで、人間も Claude も迷子にならない。

TodoWriteが効く理由は2つあります: (1)Claude自身の思考の足場になる(人間が付箋に書くと整理できるのと同じ)、(2)あなたがリアルタイムに進捗を追える(放置していても「今どこ?」が一目で分かる)。逆にこのツールが不要な場面もある——それは後述します。

いつ使う、いつ使わない

基準はシンプル: 3ステップ以上で、かつ順番や依存がある仕事なら必ず使う。それ以外は不要。紋切り型で「全てに書け」ではないので注意してください。

使うべき

TodoWriteが効く仕事

  • 複数ファイルの一括リファクタ
  • 新機能の実装(設計→実装→テスト→ドキュメント)
  • バグ調査(再現→仮説→検証→修正→テスト)
  • 5つの観点で順番にレビュー
  • プロジェクト初期設定(複数のsetup)
不要

書かなくていい仕事

  • 「このファイルを読んで」で済む単発作業
  • 1〜2ステップの軽い修正
  • 会話ベースの相談・ブレスト
  • 情報検索や質問への回答
  • 即レス系の「何がエラー?」
ここがプロの境目

3ステップ以上の仕事は必ず最初に TodoWrite を立てさせる。人間もClaudeも、全体像が見えた瞬間に精度が上がります。逆にこれを怠ると、途中で「何やってたんだっけ」になって手戻りが発生する。
試し方: 次のセッションで「今日の自分の予定を TodoWrite で立てて」とだけ頼んでみる。1分で体感できます。

3つの状態と遷移ルール

TodoWriteのタスクは常に以下3状態のどれかです。pending(未着手) → in_progress(作業中) → completed(完了)。原則として in_progress は常に1つだけ。複数並行すると、あなたも Claude も注意が拡散してしまいます。

状態遷移図
pending → in_progress → completed の一方通行。completedにしたら「もう触らない」が原則。

completed にするタイミングは「そのステップが本当に終わった時」です。テストが通っていない、検証が残っている状態で completed にしてはいけません。「done criteria」をタスクごとに明確にするのが鍵——これはCh02「対話の原則」の延長線上にあります。

状態意味このタイミングで更新原則
pending未着手タスク作成時のデフォルト並列で複数あってOK
in_progress作業中そのタスクに取り掛かる瞬間同時に1つだけ
completed完了done criteria を満たした瞬間逆戻りしない

粒度の決め方 — 大きすぎず、細かすぎず

TodoWriteの質はほぼ粒度で決まります。1タスクは15〜30分で完了できる大きさが理想。これより大きいと中で迷子になるし、これより細かいと管理コストが増えるだけ。

粒度の3段階
細かすぎ(単調)・粗すぎ(迷子)を避けた「ちょうどいい」ゾーンが15〜30分。

実例で見る — 同じ仕事、3つの粒度

お題: 「ログイン機能を作る」

粒度タスク例評価
細かすぎファイルを開く / 関数名を決める / import文を書く / ...❌ 管理コスト過多
ちょうど良い1. Login UI作成 / 2. 認証API実装 / 3. テスト追加 / 4. エラーハンドリング✅ 進捗が読める
粗すぎ1. ログイン機能を作る❌ 中で迷子
もっと詳しく: 粒度に迷った時の3つの質問
  1. このタスクを15分で終わらせられそう? — Noなら分割を検討
  2. タスク名を見ただけで完了判定できる? — 「○○を書く」より「○○が××を返すようにする」の方が良い
  3. もし途中で中断したら、再開しやすい? — タスクを閉じるポイント(ファイル保存/コミット等)で区切る

この3問を満たすと、自然と「15〜30分・完了条件明確・再開可能」な粒度になります。

実際のTodoWrite使用例

あなたが Claude Code にこう頼んだとします:

ユーザー認証機能にパスワードリセットを追加して。
メール送信 → トークン検証 → 新パスワード保存の流れで、
既存のテストも壊さないように。

Claude は内部で TodoWrite を呼び、次のようなリストを立てます(UIに表示される形):

1
既存認証コード把握
Grep/Readで関連ファイル調査
2
リセット要求API実装
トークン発行+メール送信
3
トークン検証API実装
in_progress
4
新パスワード保存
バリデーション + ハッシュ化
5
テスト追加&既存確認
pytest通し

これを眺めるだけで、「今ステップ3、残り2つ、あと30分ぐらい」と見積れます。放置運用(別作業しつつチラ見)でも安心できる理由です。

TodoWrite の呼び出しイメージ(内部APIの雰囲気)

// Claude が内部で呼ぶ(あなたが書くコードではない)
TodoWrite({
  todos: [
    { content: "既存認証コードを把握",       status: "completed",   activeForm: "把握中" },
    { content: "リセット要求API実装",        status: "completed",   activeForm: "実装中" },
    { content: "トークン検証API実装",        status: "in_progress", activeForm: "実装中" },
    { content: "新パスワード保存",           status: "pending",     activeForm: "実装中" },
    { content: "テスト追加&既存テスト確認", status: "pending",     activeForm: "検証中" }
  ]
})
Tips: activeForm とは

activeForm は「作業中」と表示するときの現在進行形ラベルです。タスク名が名詞句でも、進行中の時は動詞句にしたいので別フィールドになっています。ユーザが意識する必要はありませんが、「今XXX中」という表示が綺麗になる裏側の仕組みです。

迷子防止メカニズム — なぜTodoWriteは効くのか

長時間の作業で Claude が「何やってるんだっけ」になるのは、コンテキストウィンドウの末尾に近い情報に注意が引っ張られるから。リストを途中で見直せる形で置くと、「本来の目的」に定期的に引き戻せる。

迷子発生のメカニズム
graph LR
  A["長い作業開始"] --> B["途中で脱線
(エラー対処等)"]
  B --> C{"TodoWriteあり?"}
  C -- Yes --> D["リスト見直し"]
  D --> E["本筋に復帰"]
  C -- No --> F["脱線先で深掘り"]
  F --> G["本来の目的を喪失"]
脱線は避けられないが、復帰経路の有無で結果が変わる。

迷子パターン3つ

  • 脱線型 — エラーに遭遇して、その解決に没頭。気づいたら「何のために今これを?」となる
  • ループ型 — 同じ箇所を何度も直し続ける。全体像がないと「ここはもう手をつけた」の記憶が薄れる
  • 完了判定曖昧型 — 「だいたい動いてる」で止まる。done criteria をリストに書いておけば防げる
もっと詳しく: 長時間セッションでTodoWriteが特に光る場面

30分を超える作業、特に以下は TodoWrite なしだと事故率が跳ね上がります:

  • 複数ファイルにまたがるリファクタ(Ch10で詳説)
  • 「Red-Green-Refactor」のTDDループ(Ch11)
  • 5観点以上の並列レビュー(Ch09/Ch16)
  • バグ調査の「仮説3つ→順に検証」(Ch12)

これらの章で TodoWrite は当たり前に使われます。ここで慣れておくと後がラク。

今すぐ試す — TodoWriteで今日の予定を立てさせる

5分で体感できる超簡単なハンズオン。ターミナルで claude を起動して、以下をそのままコピペしてください:

今日の僕の予定をTodoWriteで立てて。

・午前: 昨日のPRレビュー、新機能の設計会議
・午後: 認証バグの修正、デプロイ準備、退勤前にSlack整理

各タスクの完了条件も書いて。

Claude は TodoWrite を呼び出して、5-6個のタスクをリスト化してくれます。表示されたリストを眺めて、以下を観察:

  • 粒度 — 各タスクが15〜30分で完了できる大きさか?
  • done criteria — 「終わった」と判定できる基準が各タスクにあるか?
  • 順序 — 依存関係のあるタスクが正しい順になっているか?

気に入らなければ「もっと細かく」「このタスクを2つに分けて」と指示して更新させる——これが TodoWrite の使い方の全てです。

TodoWriteは「パワーユーザーへの入口」

3+
ステップ以上なら必ず立てる
↑ 精度が跳ね上がる
15-30分
1タスクの理想粒度
↑ 迷子ゼロ
1つ
in_progress は常に
↑ 集中が守られる

TodoWrite は地味に見えますが、Ch10(改修ワークフロー)・Ch11(TDD)・Ch13(Plan モード)・Ch16(Subagents)・Ch23-25(Project群)で当たり前に使う土台です。ここで慣れておくと、後の章で「Plan モードと TodoWrite の違い」「Subagent に TodoWrite を立てさせる」等の高等技に自然につながります。

Exercise 1
入門

今日の自分のタスクを TodoWrite で立てさせる

Claude Code を起動して、今日の自分の予定(仕事・家事・なんでも)を TodoWrite で立てさせてください。以下を確認します。

  1. Claude に「今日の予定をTodoWriteで立てて。完了条件も書いて」と頼む
  2. 出力されたリストをチェック — 粒度は15-30分? done criteriaは明確?
  3. 気になる部分があれば「このタスクを2つに分けて」「もっと具体的に」と指示して修正させる
模範解答を見る

良いリストの例(朝10時からの想定):

☐ 10:00-10:30 PRレビュー (#234, #235) → 承認 or コメント付きで返す
☐ 10:30-11:30 新機能の設計会議準備 → 論点3つをメモ化
☐ 13:00-14:00 認証バグ修正 → 再現テスト追加+緑
☐ 14:00-14:30 デプロイ手順確認 → checklist完成
☐ 17:30-18:00 Slackメンション整理 → 未返信ゼロ

評価ポイント: 各タスクに(a)時間枠 (b)具体的な成果物・完了条件 がある。もし「バグ修正」だけならdone criteriaがなく迷子になりやすい。模範は「再現テスト追加+緑」で何をもって完了かが自明。

逆に悪い例は「仕事する」「メールを見る」のように粒度粗すぎ・完了条件不明なもの。

Exercise 2
応用

実プロジェクトで「3ステップ以上の改修」を TodoWrite させる

自分のコード(or 練習リポ)で、以下のような複数ステップを含む依頼を出し、Claude が自発的に TodoWrite を立てるか観察してください。

  1. お題例: 「READMEを更新して、最新のCLIオプションを反映し、セットアップ手順の誤字を直して、例コードを1つ追加して」(4ステップ)
  2. Claude が TodoWrite を立てたか確認(立てなかったら「TodoWriteで計画立てて」と追加指示)
  3. in_progress が常に1つだけか、completed が順番に増えるかを観察
  4. 完了後、リストの質を振り返り: done criteriaは? 粒度は? 脱線はなかった?
模範解答を見る

理想的な挙動:

  1. Claude が自発的に TodoWrite を呼ぶ (3-4ステップの依頼で通常は自動)
  2. タスクは例えば: (1)現状のREADME読解 (2)CLIオプション最新化 (3)誤字修正 (4)例コード追加 (5)git diff で最終確認
  3. 実行中は「進行中: トークン検証API実装」のようにUI表示が更新される
  4. 最後に全て completed になって作業完了

もし TodoWrite を立てなかった場合: Claude の判断で「2ステップ程度」と見積もった可能性があります。明示的に「計画を立ててから始めて」と頼むと、次回以降も立てる傾向が強まります。これはCh02「成功条件を渡す」の応用形です。

振り返りのポイント: 脱線があった場合、TodoWriteのおかげで「残りXX個」が見えることで早く復帰できたはず。逆にリストなしだと同じ時間でも達成率が低くなりやすい——これを体感するのがこの Exercise の目的です。
