Git連携とコードレビュー

コミット、PR作成、コードレビュー、マージコンフリクト解決まで。Gitまわりの面倒な作業をClaudeに任せましょう。

所要時間: 1.5h Phase: Core 難易度: 中級

Claude Code と Git の関係

Claude Codeは git コマンドを Bash ツール経由で叩けるので、git の操作はほぼ全部任せられます。ただし破壊的操作(reset --hard, push --force)は default モードで毎回確認されるので安心。

Claude が触れるGit領域
「左から右へ流す」流れ。Claude はどの段階の操作も /commit /pr などで自動化できる。

/commit - コミットメッセージを自動生成

/commit を叩くと、ステージング済みの差分をClaudeが読み、意味のある日本語/英語のメッセージを書いてくれます。

$ git add src/auth/
$ claude
> /commit

[Reading staged diff...]
提案されたコミットメッセージ:

  feat(auth): JWT検証ミドルウェアを追加

  - tokenUtils.ts に decode/verify を実装
  - middleware/auth.ts で全保護ルートに適用
  - 不正トークン時は 401 を返す

このメッセージで commit しますか? [y/n]
Conventional Commits

Claudeはデフォルトで feat: / fix: / refactor: 形式に寄せます。チームのスタイルが違うなら CLAUDE.md に書いておけば追従します。

/pr - Pull Request作成

/pr はブランチ作成 → push → PR本文生成 → 作成 を一気にやるマクロ的存在。

/pr の内部処理
graph LR
  A["/pr を実行"] --> B["差分を読む"]
  B --> C["タイトル+説明 生成"]
  C --> D{"ブランチは?"}
  D -- main --> E["新ブランチ作成"]
  D -- 既branch --> F["そのまま使う"]
  E --> G["push"]
  F --> G
  G --> H["gh pr create"]
  H --> I["PR URL表示"]
差分から要約した上で、push と PR 作成までこなす。

/review - コードレビュー

差分を見て、バグ・スタイル・セキュリティの観点から指摘してくれます。

> /review

[Reading PR diff...]

【セキュリティ】
- src/auth/login.ts:42  パスワード検証で timingSafeEqual を使うべき(現在は ===)

【バグ可能性】
- src/utils/date.ts:18  タイムゾーン未指定で new Date()

【スタイル】
- src/api.ts:全体  関数名が camelCase と snake_case 混在

総合: マイナーfix後にマージOK
人間レビュー

強み

  • 意図・背景の理解
  • チーム文化との整合
  • 長期戦略の判断
/review

強み

  • 網羅的・素早い
  • 細かいスタイル違反の指摘
  • セキュリティ既知パターン検出

両方使うのが正解。/review を一次レビュー、人間が二次レビューのパターンが効率的です。

ブランチ操作

# 普通の操作はそのまま頼める
> feature/login ブランチ作って、ここで作業します
> main を pull して、現在のブランチにrebaseして
> 直近3コミットを squash で1つにまとめて
force push の前に

rebase 後の git push --force は、共有ブランチでは事故の元。--force-with-lease を Claudeに頼みましょう。

diffの解説を頼む

レビューで「このdiff何やってるか分からん」という時は要約を頼みます。

> git diff main..feature/login を読んで、変更を3段階(影響大→小)で解説して
レビュアーの時間は最も貴重。diff の意図を 1 段階要約してから人間に渡すと、レビューの精度・速度ともに上がる。

マージコンフリクトの解決

コンフリクト発生時、Claudeに「両方の意図を理解してマージ」を依頼できます。

> main と feature をマージしたらコンフリクト。
両方の意図を理解した上で、適切な方向に解決して。
判断に迷う箇所は私に聞いて。
安全な解決の順序

① ブランチを保存(git branch backup-feature) ② Claudeに解決依頼 ③ git diff で確認 ④ テスト実行 ⑤ コミット。

全部繋げると: Gitワークフロー例

朝
main を最新に
「mainをpull、コンフリクトあれば見せて」
作業前
ブランチ切る
「feature/cart-bug-123 を作って」
作業中
適宜コミット
「いま追加した部分だけ /commit」
作業後
PR作成
「/pr で作って、テンプレに沿って書いて」
レビュー
/review
「自分のPRをセルフレビュー、フィックスもまとめて」

注意: 危険なGit操作

操作の取り返し可能性
右に行くほど取り返しがつかない。Claudeはdefaultで赤を毎回確認するので、慎重に判断してから [y]。
もっと詳しく: チーム共有ブランチでの「礼儀」

main や develop など共有ブランチに対する強制push・履歴書き換えは、他のメンバーのローカルを破壊します。Claudeに頼むときも「共有ブランチには force push しないで」と CLAUDE.md に明記しておくと安心。Ch09 で詳しく扱います。

Exercise 1
入門

/commit を体験する

適当なリポで以下を試してみましょう。

  1. 適当なファイルを編集して git add
  2. Claude起動 → /commit を実行
  3. 提案されたメッセージをそのまま採用してコミット
  4. git log -1 で結果を確認
模範解答を見る

うまくいくと、こんな感じのコミットができます。

feat(ui): プロフィールカードのアバター対応

- 既存の TextOnlyCard を Card にリネーム
- avatarUrl を optional で受け取り
- 未指定時はイニシャル円を表示

「箇条書き、種別タグ、なぜ・何を」が入った PR レビュアーが喜ぶスタイル。気に入らなければ 「もう少し短く」「もう1案」と頼めば即書き直してくれます。

Exercise 2
応用

セルフレビュー → 修正 → 再コミット を1セッションで

未マージのPR(なければ適当なdiffを作って)に対して、Claudeに自己レビューさせ、指摘を受けて自分で修正、再コミットまで一気にやってみましょう。

  1. 「現在のdiffを /review セルフレビュー」
  2. 「指摘されたうち、High/Mid だけ Claudeに修正させる」
  3. 「修正後のテストを実行 → /commit」
  4. 「PR descriptionも、修正内容を反映して書き直し」
模範解答を見る

会話例:

> /review
<指摘5件>

> High と Mid だけ修正してください。Low は次回まとめて。
<Edit 3ファイル>
<npm test → green>

> /commit
<新規コミット作成>

> PR description を修正内容を反映して書き直して。Markdown、5行以内。
<PR説明文の差分>

3〜4ターンで「セルフレビュー + 修正 + 再コミット + 説明更新」が完了。普段30分かかる作業が10分に短縮できる典型例です。
