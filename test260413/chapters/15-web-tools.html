Web検索・取得の活用

WebFetch と WebSearch を正しく使い分けて、自分の記憶の外へ出る

所要時間: 30分 Phase: Core 難易度: 中級
この章のゴール

「最新の公式仕様」「昨日リリースされたパッケージの挙動」「バージョン差分」——こういう情報は、Claude の学習データに載っていない、または古い可能性が高い。本章で WebFetch と WebSearch の使い分けを手に染み込ませて、「記憶で答える Claude」から「取ってきて答える Claude」 に運転席から切り替えられるようになります。
30分でこの差を体感できれば、回答品質は目に見えて上がります。

バージョン注記

本章の記述は 2026年1月時点 の Claude Code 挙動に基づきます。WebFetch / WebSearch の仕様・レート制限・許可ドメインの扱いは変わりうるため、動かない時はまず 公式ドキュメント を見てください(まさに本章の教えの実践です)。

WebFetch と WebSearch は別物 — 役割を1行で

まず混同を殺します。WebFetch は「URLを知っている前提で、そのページの中身を取ってくる」ツール。WebSearch は「URLを知らないので、まず検索して候補を出す」ツール。片方ずつ役割を固定すれば迷いません。

WebFetch vs WebSearch の役割分担
「探す→深掘る」の2段階。まず Search で候補を集め、確信があるURLだけ Fetch で全文読む。
観点WebSearchWebFetch
入力クエリ文字列 (例: "Claude Code WebFetch docs")URL (例: https://docs.claude.com/...)
出力ヒット一覧(タイトル+URL+短いスニペット)ページ本文を解析済みテキスト化
用途公式URLが分からない / 最新記事を探すURLが決まっている仕様書・PR・README を読む
想定コスト比較的軽い(一覧だけ)中〜重い(本文全文を取り込む)
先に使うほぼ常に Search が先Search の結果を1-2個選んで Fetch
弱点検索精度に依存・広告/ノイズが混ざるJS描画/認証必須サイトは取れないことがある
今すぐ試す(60秒)

Claude Code に以下の1行を投げてみてください。WebSearch → WebFetch の連鎖が自然に起こるはずです。

Claude Code の WebFetch ツールの公式仕様を、WebSearch で探して、
ヒットした docs.claude.com のページを WebFetch で読んで、
パラメータ名と戻り値の形式だけ要点を3行で教えて。

結果が自分の記憶と食い違ったら、取ってきた方が正しいと思ってください。それが本章で身につける第一の感覚です。

ここがプロの境目 — 記憶ではなく取得で答えさせる

初級者は「Claudeは物知りだから聞けば教えてくれる」と考えがちです。しかしLLMにはナレッジカットオフがあり、数カ月〜1年前までの情報しか記憶にありません。特にAPI仕様やSDKのバージョン差分、リリースされたばかりの機能は記憶ベースの答えが平気で嘘を言います。

ここがプロの境目

「最新仕様は WebFetch で取ってきて」と頼む。自分の記憶に頼らず、Claudeにも頼らせない。ナレッジカットオフ以降の情報は、依頼の中で毎回 WebFetch <公式URL> を明示するのが、上位20%の基本動作です。記憶で書かれた間違った関数名やdeprecated API を追いかけるより、1回の Fetch で済ませた方が圧倒的に速い。
試し方： 「X の最新仕様を、記憶ではなく WebFetch https://公式URL で確認してから答えて」 と書き換えるだけ。

Before(記憶依存)

「Claude Code の WebFetch の使い方教えて」

  • Claudeが記憶で答える
  • 半年前の古い仕様を返すリスク
  • 関数名 typo / パラメータ欠落
  • 実行すると動かない
After(取得依存)

「WebFetch https://docs.claude.com/... で公式を読んで、使い方を教えて」

  • 取得した本文を根拠に回答
  • 2026年1月時点の最新挙動
  • パラメータ名が正確
  • コピペで動くサンプルが出る

迷った時の判定フローと実コマンド

実務では「これ Search? Fetch?」と悩む時間がそのままロスです。以下の分岐を頭に入れて、即断できるようにします。

WebFetch / WebSearch 判定フロー
graph TD
  Start(["最新情報が欲しい"]) --> Q1{"公式URLを
知っているか?"}
  Q1 -- "Yes" --> F["WebFetch そのURL"]
  Q1 -- "No" --> S["WebSearch で
検索クエリを投げる"]
  S --> Q2{"公式ドメインが
ヒットしたか?"}
  Q2 -- "Yes" --> F
  Q2 -- "No" --> Q3{"信頼できる
情報源があるか?"}
  Q3 -- "Yes" --> F2["WebFetch
(ブログ等)"]
  Q3 -- "No" --> Ask["追加の検索語を考え直す"]
  Ask --> S
  F --> Done(["取得した本文で回答"])
  F2 --> Done
URLが既にあるなら迷わず WebFetch。無いなら必ず WebSearch が先。
  • URLが分かっている — 即 WebFetch https://docs.claude.com/...。検索は不要。
  • 「○○のドキュメント」しか分からない — まず WebSearch "Claude Code WebFetch documentation" で候補を出す。
  • バージョン差分の比較 — v1と v2 の CHANGELOG URL 2本をそれぞれ WebFetch し、並列で比較。
  • ニュース性のあるトピック — WebSearch に日付限定で (例: "2026") を含めて新しい情報だけ拾う。
今すぐ試す(2分) — バージョン差分の検証

自分が使っているパッケージ(例: react, next, vitest のどれか)で、こう頼んでみてください。

私のプロジェクトの package.json にある <パッケージ名> のバージョンは X.Y.Z。
最新の X.Y.Z を WebSearch → WebFetch で公式 changelog から確認して、
私のバージョンと最新との差分で注目すべき breaking change を3つだけ教えて。

これが「記憶ではなく取得」のリアル応用例です。返ってくる3点は、Claudeの記憶ではなく本物の changelog から抽出されたもので、信頼度が段違いです。

公式docs取得の定型フレーズ

「取ってきて」と頼む日本語は、型を決めておくと毎回迷いません。以下3つを暗記しましょう。

1
公式フェッチ型
WebFetch <URL> で公式を読んで、X について答えて
2
検索→フェッチ型
WebSearch で "Y" を探し、公式ドメインのURLを WebFetch して答えて
3
差分検証型
A と B の CHANGELOG を両方 WebFetch し、差分だけ一覧化して
公式ドキュメント必読

WebFetch / WebSearch のパラメータ名・取得サイズの上限・許可ドメインリストは実務で間違えると致命的なので、1回は公式を通読してください。Claude Code 公式 →

もっと詳しく: WebFetchの挙動の細かい話

WebFetch は対象ページを HTTP GET で取得し、HTMLをテキストに近い形へ解析してから prompt 引数と組み合わせてClaudeに渡します。JavaScript で動的に描画されるSPAの中身は取れないことが多く、その場合はAPIエンドポイント(例: /api/...)を直接叩くURLに変える、あるいはWARC化されたミラー(Wayback Machine等)のURLを使うと成功率が上がります。
また、取得はキャッシュされることがあるので、変わったばかりのページを再取得したい時は prompt の中で「キャッシュを無視して最新を取って」と明示すると有効です(モデルの判断で再フェッチを促す)。

もっと詳しく: WebSearchの精度を上げるコツ

検索精度はクエリの具体性で決まります。単に "Claude Code" と投げると数千件ヒットしてノイズ地獄。"Claude Code" WebFetch "subagent_type" site:docs.claude.com のように 固有名詞 + 絞り込み + site演算子 を組み合わせると一撃で公式に当たります。日本語サイトが欲しい時は site:zenn.dev や site:qiita.com を、英語の質問回答が欲しい時は site:stackoverflow.com を使い分け。

プライバシーと落とし穴

Web系ツールには独特の落とし穴があります。特に内部URLとPIIの扱いです。

落とし穴症状対処
認証必須ページログイン画面のHTMLだけ返ってくるAPI key付きエンドポイントを直接指定 / MCP化する(Ch17)
社内URLClaudeから到達不可本文をコピペで貼る or VPN経由の代替URLを使う
PII入りURLクエリ文字列に氏名・メールが入るURLをマスク(?email=***)してから渡す
レート制限同一ドメインの連続取得で失敗並列数を絞る / 間をあける / キャッシュ活用
取得前チェックリスト(30秒でできる)
Tips: 社内情報は基本「貼る」運用

社内のConfluenceやGoogle Docs、社内Wikiは Claude のWebFetchから原則到達できません。無理にURLを投げるより、該当部分のテキストをチャットに貼る方が10倍早い。PII を含む場合は XXX にマスクしてから貼るのを習慣化しましょう。

Exercise 1
入門

WebFetch を明示して公式を読ませる

Claude Code で、以下の指示を投げて結果を観察してください。

  1. まず Claude Code の WebFetch ツールの使い方を教えて と素で聞く
  2. 次に WebFetch https://docs.claude.com/en/docs/claude-code で公式を読んで、WebFetch ツールの使い方を教えて と頼む
  3. 2つの回答の「情報の鮮度・正確さ・根拠の出し方」を比較する
模範解答を見る

1回目(素聞き)は Claude のナレッジカットオフ時点の記憶で答えるため、パラメータ名や戻り値のフォーマットが微妙に古い/不正確になりがちです。2回目(WebFetch明示)は取得した公式本文を根拠に答えるため、「公式によると〜」という引用つきで、コピペで動くサンプルが返ってきます。

観察ポイント: 2回目の回答は具体的なバージョン注記や「2026-XX-XX時点の公式ページ」といった日付が付くことが多い。1回目は「たぶん」「だったはず」と曖昧語が多い。

# 良い頼み方のテンプレ
WebFetch <公式URL> を読んで、
- 関数シグネチャ
- パラメータの必須/任意
- 戻り値の型
の3点だけ、引用符付きで答えて。
Exercise 2
応用

自プロジェクトの依存ライブラリのバージョン差分を検証

自分の手元の package.json / requirements.txt / Gemfile いずれかを開いて、1つ選んだライブラリについて:

  1. 現在のバージョンを確認
  2. Claude Code に WebSearch で最新の安定版を調べ、公式changelogを WebFetch して、私のバージョンから最新までの breaking change を時系列で教えて と依頼
  3. 得られた回答を元に、アップグレードの価値があるかどうかの判断メモを3行で書く
模範解答を見る

この演習の目的は「記憶ではなく取得で answer する」を実体験することです。うまくいけば、以下のような回答が返ります(Reactの例)。

# 期待される回答の骨格
- 現在: react@18.2.0 / 最新: react@19.x.y (WebFetch 結果より)
- Breaking Changes (changelog 抜粋):
  1. Automatic batching の挙動変更 (18.0)
  2. useEffect のcleanup タイミング変更 (18.2)
  3. 新しい JSX Transform が必須に (19.0)
- 判断メモ:
  1. 影響面: ○○ コンポーネントで useEffect 多用 → 要レビュー
  2. 緊急度: セキュリティfix は minor で配布済み、急がなくてよい
  3. 推奨: 次sprint で v19 の compat layer 経由で段階移行

ここで大事なのは、「breaking change を3つ」と数を指定していること。数を指定しないと、Claudeはダラダラ長い回答を返して読む気が失せます。Ch08「プロンプト設計の型」で学んだ「制約を渡す」がここでも効きます。
