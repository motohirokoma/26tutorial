Project B：既存コードの大規模リファクタ

テスト追加 → リファクタ → 安全な置き換えを90分で体験する

所要時間: 90分 Phase: Projects 難易度: 応用
この章のゴール

サンプルリポ(あるいは自分のリポ)を舞台に、「壊れない大規模リファクタ」を90分で1周します。Ch10の改修ワークフロー・Ch11のTDD・Ch13のPlanモード・Ch16のSubagents・Ch21のworktree——ここまでの武器を全部同時に使う、**統合ハンズオン**です。読む時間は最小、手を動かす時間を最大に取ってください。

バージョン注記

本章の記述は 2026年1月時点 の Claude Code (Agent SDK/CLI) 挙動に基づきます。git worktree の挙動・Plan mode の UI・Agent tool の subagent_type ラインナップは進化するため、コマンドが通らない場合は 公式ドキュメント を確認してください。

全体像 — 90分の地図

このプロジェクトは3つのパートで構成されます。リファクタは「振る舞いを変えない改修」なので、先にテストで振る舞いを固定するのが鉄則です。

Project B の90分タイムライン
中央のPart 2(実装)が最重量。前後の設計・検証が「壊さない」ための盾になる。
  • Part 1: 設計(15-20min) — git worktree で作業空間を隔離し、Explore agent でコード把握、Planモードで変更計画に合意する。
  • Part 2: 実装(50-60min) — 先にテストを足して振る舞いを固定 → リファクタ実施 → テスト全通過を確認。
  • Part 3: 検証(15-20min) — Ch09のコードレビュー技でdiffを批判的に読み、本流マージの可否を自分で判断する。
ここがプロの境目

「テストを書いてから」のリファクタは絶対に壊れない。既存の振る舞いをテストで固定してしまえば、あとは内部をいくらいじってもテストが壊れなければ正解だと保証される。
試し方： 対象モジュールのgit log --oneline -n 20を見てテストカバレッジが薄い箇所を探し、リファクタ前に先にテストを2-3本書かせる。

題材の準備 — サンプル or 自リポ

リファクタ対象は2択です。どちらも90分で1周できる規模に絞ります。

選択対象向いている人注意
A. サンプルCh23で作った日次レポCLIツールまず型を覚えたい人手元に23-project-automationの成果物が必要
B. 自リポ自分の業務/個人プロジェクトの1ファイルいま困っているコードがある人事前にmainへpushしておく

どちらも 「300-600行の1ファイル」くらいの単位 を選ぶとちょうどいい。大きすぎると90分で回しきれず、小さすぎるとリファクタの効果が見えにくい。

公式ドキュメント必読

git worktree は1つのリポジトリから複数のワーキングツリーを生やす、Git標準機能です。本章でも多用するので挙動を誤解すると作業が別ブランチで迷子になります。公式を一読しておいてください。git-worktree 公式 →

Part 1: 設計 — worktree / Explore / Plan (15-20min)

最初の20分は 触らない時間 です。いきなり書き換えさせると、壊れる確率がぐっと上がる——Ch10で学んだ通り、改修ワークフローの頭2ステップを丁寧に踏みます。

1
worktree作成
本流を汚さない隔離空間
2
Explore agent
コード把握を子に任せる
3
Planモード
計画への合意

Step 1: worktree で作業空間を隔離する

worktreeを使うと、mainをそのまま手元に残したまま、別ディレクトリで別ブランチを触れます。リファクタが失敗しても、mainは無傷です。

今すぐ試す(2分)

対象リポジトリのルートでこのコマンドを実行:

# リポのルートで
git fetch origin
git worktree add ../myrepo-refactor -b refactor/ch24 origin/main
cd ../myrepo-refactor
pwd   # 新しい作業ディレクトリにいるはず

これで../myrepo-refactorに別ブランチrefactor/ch24がチェックアウトされました。元のディレクトリはmainのまま温存されます。

Step 2: Explore agent でコード把握

コード把握はメインのClaudeでやってもいいですが、ファイル数本以上読ませると親のコンテキストが汚れます。そこでExplore subagentに任せます。

実プロンプト(コピペ可) 
Agent tool で subagent_type="general-purpose" として以下を実行してください。

prompt:
「src/ 配下の report_generator.py を読み、以下を要約してください：
(1) モジュールの責務を1行で
(2) 公開関数とその入出力
(3) 内部で抱えている "長すぎる関数" "重複コード" "テストしづらい箇所" を3つまで
(4) リファクタ候補の優先度(高/中/低)を付ける
出力は markdown 箇条書きで 30行以内。」

子が要約だけ返してくるので、親は要約を読むだけで全体像を掴める。親のコンテキスト消費は最小限です。

Step 3: Planモードで計画に合意する

Explore の要約を見たら、Planモードに入ってリファクタ計画を立てさせます。大きな変更こそPlanから入る——これがロールバックの保険になります。

Planモードの流れ
graph LR
  A["Exploreの要約"] --> B["Planモード起動"]
  B --> C{"計画に納得?"}
  C -- Yes --> D["ExitPlanMode承認"]
  C -- No --> E["修正要求"]
  E --> B
  D --> F["実装フェーズへ"]
Planモードは「同意してから実装」を強制する。気に入らない計画は必ず差し戻す。
実プロンプト(Planモードで) 
/plan

report_generator.py に対して次のリファクタ計画を立ててください。

前提：
- 振る舞いは変えない(=同じ入力で同じ出力)
- 1ファイル内で完結(新しいファイルを作らない)
- テストは先に追加してから改修する

観点：
- 長すぎる関数(30行超)の分割候補
- 重複の括り出し
- I/O と純関数の分離
- 変数名の明瞭化

出力：
1. 変更前後の構造差分(ASCIIで)
2. リファクタ順序(テスト追加 → Step1 → Step2 → ...)
3. 各Stepの予想コミットメッセージ
もっと詳しく: Plan に含めるべき「成功条件」

Planには「どこで終わるか」を必ず入れてください。Ch02で学んだ通り、done criteriaがないとClaudeは止まるタイミングを見失います。例:

  • 全テストがpytest -qで緑
  • git diff main...HEAD --statで±20%以内の変化
  • 対象関数の最大長が30行以下

この3点セットを渡すと、Claudeは「リファクタ頑張り続ける」ループに入らず、きっぱり止まります。

Part 2: 実装 — Test → Refactor → Green (50-60min)

ここがプロジェクトの心臓部です。順序が命——先にテスト、次にリファクタ、最後に確認。順番を間違えると「何が壊れたか分からない」地獄に落ちます。

Before(いきなり改修) vs After(テスト先行)
テスト先行は「いま書き換えたこの瞬間」に壊れたかどうかを教えてくれる。

Step 4: テストを先に追加する(RED)

Ch11のTDDで学んだ通り、Red-Green-RefactorのRedから入ります。ただしリファクタの場合は既存コードが既にGreenに相当するので、「既存の振る舞いを固定するテスト」を足すのが目的です。

実プロンプト(コピペ可) 
report_generator.py のリファクタに先立ち、現状の振る舞いを固定するテストを追加してください。

要件：
1. 公開関数(generate_report, format_row, load_issues)それぞれに最低1本
2. 正常系 + 境界値(空リスト / 100件 / 特殊文字を含むタイトル)
3. テストは tests/test_report_generator.py に配置
4. アサーションは「既存コードが実際に返している値」を基準にする
5. テスト内の入力は fixture として切り出す
6. pytest -q で全緑になることを確認してから終了

禁止事項：
- 既存コードの挙動を変えない
- 「本当はこうあるべき」という理想で assertion を書かない(今の挙動を固定するのが目的)

この禁止事項を必ず入れるのが今回のコツ。理想を書かせるとテストが既存挙動と食い違い、その後のリファクタが永遠に赤いまま終わらなくなります。

Step 5: リファクタを実施する(REFACTOR)

テストが緑になったら、Planで合意した順序で1Stepずつ進めます。1Step終わるごとにpytestを走らせ、git commitで区切るのが最大の保険です。

Step 5-1
長すぎる関数を分割
generate_reportをcollect_issues / sort_issues / render_markdownの3つに分解。各関数を Claude に抜き出させる。テスト実行 → 緑 → コミット。
Step 5-2
重複を関数に括り出す
日付フォーマット処理が3箇所に散らばっていたら、format_dateヘルパに集約。テスト緑 → コミット。
Step 5-3
I/O と純関数を分離
ファイル書き込みはsave_reportに集め、残りは入力→出力の純関数化。テスタビリティが劇的に上がる。テスト緑 → コミット。
Step 5-4
変数名の明瞭化
d→due_date、tmp→formatted_rowなど。IDEのrenameで機械的に。テスト緑 → コミット。
実プロンプト(1Stepずつ) 
Plan の Step 5-1 を実施してください。

やること：generate_report を collect_issues / sort_issues / render_markdown に分割

制約：
- 1ステップで完結(他のリファクタは次以降)
- テストを絶対に緩めない(assertion を一切変えない)
- 実装後に pytest -q を必ず実行し、結果をそのまま貼る
- 緑なら git add -p で変更を確認し、「refactor: split generate_report into 3 funcs」でコミット
- 赤なら直ちに revert して原因を報告

Step 6: コンテキストが重くなったら /compact

50-60分も作業するとセッションが長くなります。Ch14で学んだ/compactを、リファクタの区切り(Step 5-2 終了時など)で1回入れるとちょうどいい。

Tips

/compactの前に「ここまでの差分と残タスクを3行で要約して」と頼むと、圧縮前に自分で確認できて安心です。

Step 6の視覚化：リファクタ前後のコード複雑度

関数ごとの行数(Before/After 想定値)

大きな関数1個が消えて、20行前後の小さな関数が5-6個になる。これが「小さく分ける」の成果です。

Part 3: 検証 — Review と Merge判断 (15-20min)

実装が終わってテストも緑——ですが、ここで本流に入れるかはまだ保留。最後の20分で自分の目と、Claudeの批判的レビューを通します。

Step 7: 差分を批判的にレビューさせる

Ch09の型通り、観点を指定してレビューを依頼します。「レビューして」だけでは薄い指摘しか返りません。

実プロンプト(コピペ可) 
git diff main...HEAD を読み、以下の観点で辛口レビューしてください。

観点(必ず全て触れること)：
1. 振る舞い保存 — 既存の挙動が本当に同じか(テストがカバーしていない分岐は?)
2. 可読性 — 関数名・変数名・コメントは意図を表しているか
3. 責務の分離 — I/Oと純関数の分離は徹底されているか
4. 過剰リファクタ — 不要な抽象化や早すぎる最適化がないか
5. テスト品質 — 追加したテストは本当に振る舞いを固定しているか(assertion が緩くないか)

出力形式：
- 各観点で「OK / 懸念 / NG」を1つずつ
- NG/懸念があれば具体的な行番号と修正案
- 最後に「main にマージして良いか(Yes/No + 理由)」を1行で

Step 8: マージ判断

Claudeのレビューと自分の判断を照らし合わせます。ここで 「Yesと言われても自分が納得していなければマージしない」 のが鉄則です。AIは誤って太鼓判を押すこともある。

マージしていい

安心サイン

  • 全テスト緑 + 追加テストあり
  • レビュー全観点 OK
  • 自分でdiffを全行読んで納得
  • 各コミットが小さく戻せる
保留すべき

警戒サイン

  • 「多分大丈夫」「たぶん同じ」がレビューにある
  • 1コミットが500行超
  • テストが assertion 緩めに書かれている
  • 自分がdiffの意味を説明できない
マージ手順(安心サイン全Yesの時) 
# 1. worktreeからpush
git push -u origin refactor/ch24

# 2. PR作成(Ch07のgh pr createパターン)
gh pr create --title "refactor: report_generator.py を責務分離" \
             --body "テスト追加 → 関数分割 → 重複括り出し。振る舞い変更なし。"

# 3. マージ後、worktreeを片付ける
cd ..
git worktree remove myrepo-refactor
git worktree list   # 消えたか確認
もっと詳しく: worktree を残す選択肢

worktree removeせずに残せば、次のリファクタ時にgit pullして再利用できます。チーム開発では「リファクタ専用worktree」をメンバー各自が持つ運用もアリ。ただしブランチが古くなるので、週に1回はmainに追随させるのを忘れずに。

発展課題 — もう一段深く

90分でPart1-3を回し終えた人は、次の2問で応用幅を広げてください。どちらも自分の実リポでやるのが理想です。

発展課題 1
応用

複数ファイル横断リファクタ

1ファイル内で閉じるリファクタに慣れたら、3-5ファイルにまたがるリファクタに挑戦しましょう。例: 共通ヘルパの抽出、ドメインモデルの切り出し、循環依存の解消など。

  1. Explore agent を2つ並列で飛ばし、モジュール関係図を作らせる(Ch16)
  2. Planモードで「どのファイルをどの順で触るか」を決める
  3. 各ファイルに先にテストを追加 → 1ファイルずつリファクタ → 毎回 pytest 緑を確認
  4. 最後に全ファイルのdiffを1つのPRにまとめる
模範解答を見る

成功の鍵は「並列 Explore」+「直列 Refactor」。

# 並列Explore(2つの観点で同時に飛ばす)
Agent tool 並列起動:
  agent1 prompt: "src/ 配下のモジュール間の import 関係を列挙して、循環依存があるか調べて"
  agent2 prompt: "src/ 配下で3回以上重複している処理を見つけて、括り出し候補をリストアップして"

# 直列Refactor(1ファイル1コミット)
1. shared/helpers.py を新設(空)
2. moduleA から重複処理を helpers.py に移動 → pytest → commit
3. moduleB から同様に移動 → pytest → commit
4. moduleC から同様に移動 → pytest → commit
5. 最終的に重複がないことを確認

並列と直列を使い分けるのが本章の肝。調査は並列、書き換えは直列。並列で書き換えるとマージ競合で破綻します。

発展課題 2
応用

チームメンバー用のworktreeハンドオフ手順書化

本章でやったworktree運用を、チームの他のメンバーが再現できるMarkdown手順書に落とし込んでください。

  1. docs/refactor-playbook.mdを作成
  2. 「前提 / 手順 / よくある失敗 / 戻し方」の4セクション
  3. コマンドは全てコピペで動く形で(環境変数やパスは$REPO_PATHのように変数化)
  4. Claudeに「新入社員が読んでも迷わないか」視点でレビューさせる
模範解答を見る

手順書の骨子例:

# Refactor Playbook (worktree運用)

## 前提
- git 2.30以上、Claude Code導入済み
- 対象ブランチは必ず `main` 由来の新規ブランチ

## 手順
1. `git fetch origin && git worktree add ../${REPO}-refactor -b refactor/$(date +%Y%m%d) origin/main`
2. `cd ../${REPO}-refactor`
3. Claude Codeを起動し、本章のPart 1-3に従う
4. PR作成 → マージ後に `git worktree remove` で掃除

## よくある失敗
- 元のディレクトリで作業してしまう → `pwd` で確認
- worktree掃除し忘れ → `git worktree list` で週次監査
- 古いworktreeでmain追随を忘れる → `git pull origin main` を週1で

## 戻し方
- PR前: `git worktree remove --force ../${REPO}-refactor`
- PR後: 通常の `git revert`

チーム展開する時は、「戻し方」セクションが最重要。何かあった時にパニックにならず戻せる手順が、心理的安全性を生みます。
