GitHub連携: Actions & チーム開発

CI/CDの自動化、リポジトリ運用、チーム開発のベストプラクティスまで。日々の開発をもっと楽にする仕組みを整えましょう。

所要時間: 1h Phase: Core 難易度: 中級

GitHub Actions の自動生成

GitHub Actions はYAMLファイルでビルド・テスト・デプロイを自動化する仕組み。Claudeに「テストが走るCIを作って」と言えば .github/workflows/ci.yml を1分で生成してくれます。

$ claude
> Node.js プロジェクト用に、PR時に lint + test を走らせるGitHub Actions を作って。
Node 20、キャッシュも有効化して。

[Writing .github/workflows/ci.yml...]

name: CI
on:
  pull_request:
  push:
    branches: [main]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm'
      - run: npm ci
      - run: npm run lint
      - run: npm test
CI ワークフローの全体像
graph LR
  A["push or PR"] --> B["Trigger workflow"]
  B --> C["Setup Node"]
  C --> D["Install"]
  D --> E["Lint"]
  E --> F["Test"]
  F --> G{"All green?"}
  G -- yes --> H["✓ Status check pass"]
  G -- no --> I["✗ PR にエラー表示"]
緑が揃わないとマージできない設定にしておけば、品質の最低ラインが守れる。

よく使うトリガー

トリガー意味
on: pushpushされた時
on: pull_requestPR作成・更新時
on: schedulecronで定期実行
on: workflow_dispatch手動起動できるボタンを生やす
on: releaseリリース作成時

マトリクスビルド

「Node 18/20/22 全部でテストしたい」みたいな並列実行も得意。

strategy:
  matrix:
    node: [18, 20, 22]
steps:
  - uses: actions/setup-node@v4
    with:
      node-version: ${{ matrix.node }}

Issue → 実装 → PR 一気通貫

ここまで覚えた要素を組み合わせると、Issue起点の実装サイクルが半自動化できます。

標準フロー
sequenceDiagram
  participant U as User
  participant C as Claude
  participant GH as GitHub
  U->>C: "Issue #87 を解決して"
  C->>GH: gh issue view 87
  GH-->>C: 詳細
  C->>C: 計画立案
  C->>U: 計画を提示
  U->>C: 承認
  C->>GH: ブランチ作成 + 実装 + push
  C->>GH: gh pr create (Closes #87)
  GH-->>U: PR URL
  U->>GH: レビュー → merge
  GH->>GH: Issue 自動クローズ
Issue → ブランチ → 実装 → PR → マージ → Closeを「対話1セッション」で。

1コマンドで起動するパターン

「Issueを解決するスクリプト」も書けます。

# shell エイリアスにしておく
alias solve-issue='claude -p "$(echo Issue \$1 を要件整理→計画→実装→/pr の流れで進めてください)"'

solve-issue 87

CIから Claude を呼ぶ(Headless)

-p モードと ANTHROPIC_API_KEY を組み合わせると、CI内でClaudeを動かせます。

# .github/workflows/auto-review.yml(イメージ)
name: Auto Review
on: pull_request
jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - run: npm install -g @anthropic-ai/claude-code
      - run: |
          claude -p "gh pr diff ${{ github.event.number }} を見てレビュー、結論をmarkdownで標準出力" \
            | gh pr comment ${{ github.event.number }} -F -
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
CIでClaudeを使う注意

① APIキーは必ず secrets に。② permission-mode をbypassにしない。③ コスト上限を設ける。

リポジトリ管理

ブランチ保護

main を守るルールを設定しましょう。

  • Require PR before merging — 直接pushを禁止
  • Require status checks — CIが緑のときだけマージ可
  • Require approving reviews — 1名以上のapprove必須
  • Restrict who can push — 特定チームのみ

チーム開発のベストプラクティス

「Claudeを使ってる人」と「使ってない人」が混在するチームでは、Claude生成の差分でも「人間レビュー」を挟むのが安全。

チームでルールにすべき5つ

  1. CLAUDE.md をリポジトリに含める(Ch09)
  2. danger系コマンドは default モードを強制(Ch10 の hooks)
  3. AI生成のコミットは Co-Authored-By: Claude で示す
  4. 本番デプロイは必ず人間が押す(自動化はステージングまで)
  5. API キーはCIの secrets に。ローカルは .env

「Claude無しでも動く」ことの大切さ

CLAUDE.mdの内容に頼り過ぎると、新人が claude なしで読み解けなくなる。READMEには 人間向けの導入 を別途用意しましょう。

もっと詳しく: Claude bot との共生

GitHub上のIssue/PRに「@claude」と書くと自動応答するbotを設定するチームも増えています。設定例(claude-code-action)を使えば、Issueにメンションするだけで実装PRを返してもらえる。チーム内で「botに任せていい範囲」「人間が見るべき範囲」を最初に決めておくと混乱がありません。

Exercise 1
入門

最小CIを生成して動かす

適当なリポで Claude に CI を作らせ、実際に走らせてみましょう。

  1. 「Node プロジェクトの PR時 lint+test の Actions を作って」
  2. commit + push
  3. 適当なPRを作る → Actions タブで結果を見る
  4. 失敗したら、ログを Claude に貼って「直して」
模範解答を見る

典型的な失敗例と直し方:

  • npm ci が失敗 → package-lock.json がコミットされていない。Claudeが git add package-lock.json を提案
  • Node バージョン不一致 → .nvmrc を追加 + node-version-file 指定
  • lint がない → npm run lint スクリプトを package.json に追加

2〜3往復で全テスト緑になればOK。

Exercise 2
応用

PR自動レビュー Bot を仕込む

Claude を CI から呼んで、PR にレビューコメントを自動投稿する仕組みを作ってみましょう。

  1. 新規 workflow auto-review.yml を生成依頼
  2. secrets.ANTHROPIC_API_KEY を設定
  3. テストPRを作って、botコメントが付くか確認
  4. 不要時は if: contains(github.event.pull_request.labels.*.name, 'ai-review') で発動条件を絞る
模範解答を見る

運用上のコツ:

  • 常に走らせない — ラベル付与時のみなど条件を絞る。コスト管理に直結
  • 長文回答は折りたたみ — <details> で囲んで投稿、概要だけ見えるように
  • 自分のbotコメントには反応しない — github.actor を見て自己ループ防止

仕上がるとPRレビューの一次対応(typo・命名・明らかなバグ)が無人化され、人間レビュアーは「設計・意図」に集中できます。
