Phase 2: Core > Chapter 11

テスト駆動での依頼

Red → Green → Refactor を Claude と回す。assertion は絶対に緩めさせない。

所要時間: 60min Phase: Core 難易度: 中級
2026年1月時点の情報

本章は2026年1月時点の Claude Code (Sonnet/Opus系) の挙動を前提としています。テスト生成の精度・プロンプト感度は改善が続いているため、assertion を緩められる症状は弱まっていく可能性がありますが、人間側が期待値を握る原則は変わりません。

なぜ Claude には TDD が効くのか

Claude Code に実装を頼むと、しばしば「それっぽいコード」が返ってきます。動きそうに見えて、境界値や例外ケースで崩れる。これを防ぐ最強の手段が TDD です。Claude に「このテストを通して」と渡せば、成功条件が数式のように定義され、曖昧さが消えます。

逆に言えば、テストを渡さずに「○○機能を作って」と頼むのは、採点基準なしにテスト答案を書かせるようなもの。Claude は親切なので「たぶんこういう意味だろう」と想像で埋めますが、それは高確率で外れます。

Red → Green → Refactor の3色サイクル
赤→緑→オレンジ。Claude は主に Green フェーズで働き、Red と Refactor の判断は人間が握る。

3色それぞれの「誰がやるか」を分けて覚えてください。Red は人間、Green はClaude、Refactor はClaude + 人間レビュー。役割分担を明確にするだけで事故が激減します。

書かせるテスト vs 自分で書くテスト

「テストは全部 Claude に書かせれば早いじゃん」と思った方、そこが最初の落とし穴です。採点者を受験者に選ばせたら、甘くなる。核心の assertion は自分で書き、周辺の定型文を Claude に任せるのが鉄則。

種類誰が書く理由例
核心テスト(仕様)人間期待値は仕様そのもの。外に出すと甘くなる「金額以上なら配送無料」
境界値テスト人間(指定) + Claude(展開)境界の定義は人間、N件展開は機械的0/1/最大-1/最大/最大+1
例外・エラー系Claude(生成) + 人間(レビュー)網羅は機械が強い、取捨選択は人間null/空配列/型違い
定型セットアップClaudebeforeEach / fixture は退屈な繰り返しmock DB・テストデータ
snapshot テストClaudeUIの現状記録。レビューだけすればOKReact component の DOM
ここがプロの境目

「テストを通すためにテストを変えない」とあらかじめ明示する。これを忘れると Claude は実装で詰まった瞬間、assertion を緩めたりテストをスキップしたりして「Green になった」と報告します。テストは仕様書であり、仕様を勝手に下げられたら無意味。
試し方： 依頼プロンプトの最後に テストの期待値は変更不可。通らないなら実装を直して、それでも無理なら止まって相談して。 と必ず書く。

2つの失敗パターンを可視化

やりがち (NG)

全部お任せ

  • 「テストも実装も書いて」と丸投げ
  • Claude が自分で書いた assertion を自分で通す
  • 採点者=受験者で合格率100%
  • 本番で境界値が落ちる
プロ (OK)

核心を握って任せる

  • 仕様の assertion は自分で3-5本書く
  • 「これを通して。期待値は変更不可」と渡す
  • Claude が実装する
  • 境界値展開と mock を追加で依頼

実例：送料計算関数を TDD で作る

例題:「注文金額が 5000円以上なら送料無料、未満なら 500円」という仕様。これを Red → Green → Refactor で回します。

今すぐ試す: Red フェーズ — 人間がテストを書く

まず自分でテストファイル shipping.test.js を書きます。これはClaudeに書かせない。仕様の境界(5000円ちょうど)を境界値として明示します。

// shipping.test.js — 人間が書く
import { calcShipping } from './shipping.js';

describe('calcShipping', () => {
  test('5000円ちょうどは送料無料', () => {
    expect(calcShipping(5000)).toBe(0);
  });
  test('4999円は送料500円', () => {
    expect(calcShipping(4999)).toBe(500);
  });
  test('5001円は送料無料', () => {
    expect(calcShipping(5001)).toBe(0);
  });
  test('0円(空カート)は500円', () => {
    expect(calcShipping(0)).toBe(500);
  });
  test('負の金額は例外', () => {
    expect(() => calcShipping(-1)).toThrow();
  });
});

ここがミソ。5000以上なのか5000超なのか、仕様が「境界でどっち側か」を assertion で凍結しています。この5本があれば、実装の曖昧さは消えます。

今すぐ試す: Green フェーズ — Claude に実装を頼む

テストを添付(もしくは Read させて)、以下のプロンプトを投げます。

@shipping.test.js のテストを全て通す calcShipping を
shipping.js に実装してください。

制約:
- テストの期待値は変更不可。テストファイルは触らないこと
- 通らないなら実装を直す。それでも無理なら止まって相談
- JSDoc も付けて

Claude は最小実装を返してきます:

// shipping.js — Claude 生成
/**
 * 注文金額から送料を計算
 * @param {number} amount 注文金額(円)
 * @returns {number} 送料(円)。5000円以上で0円、未満で500円
 * @throws {RangeError} amount が負の場合
 */
export function calcShipping(amount) {
  if (amount < 0) throw new RangeError('amount must be non-negative');
  return amount >= 5000 ? 0 : 500;
}
公式ドキュメント必読

テストファイルを Claude に添付する方法・context で渡す方法は公式ドキュメントで必ず押さえてください。ファイル参照の作法はプロジェクト全体のコスト効率に直結します。Claude Code 公式 →

テストが青のまま、以下を追加で頼みます:

Refactor フェーズに入ります。

テストのスコープと境界値の指定

Claude に境界値を「察して」もらえません。境界値分析は依頼プロンプトに明示的に書くと効きます。

テストスコープの4象限(入力 × 系統)
オレンジ=境界値が最重要。「5000円」のような仕様値は必ず ±1 でテスト。
もっと詳しく: 境界値をプロンプトで指定するテンプレ

以下をコピーして使ってください。「○○」を仕様値に置き換えるだけ。

テストを追加してください。境界値観点で下記を必ずカバー:

1. 仕様境界値 ○○ のちょうど、-1、+1 の3パターン
2. 型境界(null, undefined, 空配列, 空文字, 型違い)
3. 数値境界(0, 負数, Number.MAX_SAFE_INTEGER, NaN, Infinity)
4. コレクション境界(要素0, 1, 最大想定数, 最大+1)

ルール:
- 既存のテストは変更しない(追加のみ)
- describe で境界値カテゴリごとにまとめる
- 各 test に「なぜこれを確認するか」を1行コメント

「テスト改変」を防ぐ仕組み化

口頭で「テスト変えないで」と言うだけでは弱い。仕組みで防ぎます。以下4つをレイヤーとして積むと、assertion が緩められるリスクがほぼ消えます。

テスト保護の4層防御
graph TD
  A["依頼プロンプト
期待値は変更不可と明記"] --> B["CLAUDE.md
テスト改変禁止ルール"]
  B --> C["git diff 確認
テストファイル変更を検出"]
  C --> D["pre-commit hook
機械的ブロック"]
  D --> E["安全なTDDループ"]
上から下に進むほど強制力が上がる。最上位の「プロンプト」だけでは忘れる。

レイヤー1: プロンプト明記(毎回)

【厳守】
- テスト(*.test.js, __tests__/)は touch しないこと
- assertion 変更, skip, xdescribe, todo 化 すべて禁止
- 通らないなら実装を直すか、止まって報告

レイヤー2: CLAUDE.md に恒久ルール

# CLAUDE.md (抜粋)

## テスト方針
- テストファイルの期待値(expect/assert)は、**明示的に依頼された時のみ**変更可能
- 実装が通らない時、テストを緩めて通すのは禁止。必ず相談すること
- skip / xdescribe / .todo は新規追加禁止

これで毎回書かなくても、プロジェクト単位で効くようになります(CLAUDE.md は Ch06 参照)。

レイヤー3: git diff 後レビュー

実装が通ったと Claude が報告してきたら、必ず git diff -- '*.test.*' でテストの差分を確認。空なら OK、何か出たら警戒。

レイヤー4: pre-commit hook で機械的ブロック

もっと詳しく: テストファイル改変を検出する pre-commit フック例

.husky/pre-commit に以下を追加:

#!/bin/sh
CHANGED_TESTS=$(git diff --cached --name-only | grep -E '\.(test|spec)\.(js|ts|tsx)$')
if [ -n "$CHANGED_TESTS" ]; then
  echo "⚠️  テストファイルの変更が含まれています:"
  echo "$CHANGED_TESTS"
  echo "意図的な変更なら 'git commit --no-verify' で通してください"
  exit 1
fi

これで Claude が勝手にテストを触って commit しようとしても、フックで止まります。commit を分けて「人間が意図的にテスト変更 → --no-verify」「実装のみ commit は素通り」と運用できます。

効果の可視化：TDDで欠陥はどれだけ減るか

体感ではなく数字で見ておきましょう。筆者が社内プロジェクトで TDD を導入する前後で、本番検出バグがどう動いたかのサンプルです(サンプルサイズ小なので傾向として)。

TDD導入前後の本番検出バグ数(月次)

4月の導入月から緩やかに減り、3ヶ月で1/4以下に。TDDは予防医療です。治療より安い。

TDDで守られる5つの側面(レーダー)

プロンプト・テンプレ集(保存推奨)

ここまでの内容をテンプレ化しておきます。スニペット管理ツールに登録してください。

  • T1: Red テスト依頼 — 自分が書ききれない定型部分だけ任せる
  • T2: Green 実装依頼 — 期待値変更不可を必ず添える
  • T3: Refactor 依頼 — テスト青のまま・観点リスト付き
  • T4: 境界値展開 — 仕様値を±1で自動展開
もっと詳しく: T1-T4 のフルテンプレ

Refactor フェーズ。テストは全て青のまま、以下を整える:

既存テストは触らない。describe('境界値: ○○') でまとめる。

よくある落とし穴と回避策

落とし穴1

テストを書きすぎる

  • 全関数に3本ずつ → 保守コスト爆発
  • 実装の中身までテストして結合テストみたいになる

回避: 境界値と核心仕様に絞る。カバレッジ100%は目的ではありません。

回避1

仕様1つに assertion 3-5本

  • 境界値の両側 + 正常系1本
  • 足りないと感じたら追加すればいい
落とし穴2

mock しすぎ

  • Claude に頼むと何でも mock しがち
  • 実装が変わっても mock が追従せず偽Green
回避2

mock は境界だけ

  • DB/外部API/時刻 のみ mock
  • 自分で書いた純粋関数は mock しない
Exercise 1
入門

calcDiscount の Red を自分で書く

仕様:「会員ランクが gold なら20%OFF、silver なら10%OFF、それ以外は0%」。以下を行ってください。

  1. calcDiscount.test.js を自分で書く(最低5本)
  2. 「期待値変更不可」を明記したプロンプトで Claude に実装を依頼
  3. git diff -- '*.test.*' でテストが触られていないことを確認
模範解答を見る

テスト(人間が書く):

import { calcDiscount } from './calcDiscount.js';

describe('calcDiscount', () => {
  test('gold は 20% OFF', () => {
    expect(calcDiscount(1000, 'gold')).toBe(800);
  });
  test('silver は 10% OFF', () => {
    expect(calcDiscount(1000, 'silver')).toBe(900);
  });
  test('bronze は割引なし', () => {
    expect(calcDiscount(1000, 'bronze')).toBe(1000);
  });
  test('未指定ランクは割引なし', () => {
    expect(calcDiscount(1000, 'unknown')).toBe(1000);
  });
  test('負の金額は例外', () => {
    expect(() => calcDiscount(-100, 'gold')).toThrow();
  });
});

Claudeへの依頼プロンプト:

@calcDiscount.test.js の全テストを通す実装を calcDiscount.js に。
テスト変更禁止。通らなかったら実装を直す。無理なら止まって相談。

期待される実装:

const RATES = { gold: 0.2, silver: 0.1 };
export function calcDiscount(amount, rank) {
  if (amount < 0) throw new RangeError('amount must be non-negative');
  const rate = RATES[rank] ?? 0;
  return Math.round(amount * (1 - rate));
}

ポイント: Math.roundの有無で落ちるケースがあります。テストで 1000 * 0.8 = 800 のような整数になる値を選んで曖昧さを消しておくのがコツ。

Exercise 2
応用

既存バグをRed-firstで直す

自分の過去のプロジェクト(なければ小さな関数を1つ用意)で、1つバグを見つけて以下のフローで直してください。

  1. バグを再現するテストを1本書く(これが Red)
  2. CLAUDE.md にテスト保護ルールを追記
  3. Claude にバグ修正を依頼(テスト変更禁止を明記)
  4. pre-commit フックを追加してテスト改変を検出する仕組みを作る
  5. 修正前後で git log を追い、ロールバック可能性を確認する
模範解答を見る

ステップ1: 再現テスト

// parseDate.test.js
test('バグ再現: 2026-02-29 (閏年でない) は null を返すべき', () => {
  expect(parseDate('2026-02-29')).toBeNull();  // ← 今は new Date(...) が返ってしまう
});

ステップ2: CLAUDE.md 追記

# CLAUDE.md
## テスト保護
- *.test.js のassertionは明示的指示がない限り不変更
- 実装が通らない時はテストを緩めず、必ず止まって相談すること

ステップ3: 修正依頼

@parseDate.test.js に再現テストを追加しました。
これが通るよう @parseDate.js を修正して。
テストファイルは変更しない。失敗したら止まって相談。

ステップ4: pre-commit フック — 前述の husky パターンを採用。

ステップ5: git 確認 — git log --oneline -- parseDate.js と git log --oneline -- parseDate.test.js を別々に確認。テスト側の commit が「追加のみ」で変更が無ければ安全。

学び: バグを直すと同時に回帰テストが手に入ります。TDDは最初の投資より、2回目以降の安心感の方が大きいです。
