Pages を有効化 (Actions経由)

GitHub Pages の設定は Source: GitHub Actions を選ぶのが推奨。mainへのpushで自動的にビルドされ、別ブランチを介さず公開されます。

gh api --method POST /repos/<username>/portfolio/pages \
  -f "build_type=workflow"
GitHub Pages 設定画面のイメージ
Source = GitHub Actions にすると、自分でworkflowを書く前提になる。

Step B — workflow を Claude に書かせる

Claudeに丸投げOKな部分。Astroの公式が withastro/action を出しているので、これを使うように指示します。

.github/workflows/deploy.yml を作って。Astro公式の withastro/action@v3 を使い、main への push で GitHub Pages にデプロイする最小構成で。Node 20。 — Claudeへのプロンプト例

生成される deploy.yml

name: Deploy to GitHub Pages

on:
  push:
    branches: [main]
  workflow_dispatch:

permissions:
  contents: read
  pages: write
  id-token: write

concurrency:
  group: pages
  cancel-in-progress: true

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: withastro/action@v3
        with:
          node-version: 20

  deploy:
    needs: build
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - id: deployment
        uses: actions/deploy-pages@v4
base path に注意

https://user.github.io/portfolio/ のようにサブパスで公開する場合、astro.config.mjs に site と base を指定する必要があります。次節で対応。

astro.config.mjs を更新

// astro.config.mjs
import { defineConfig } from 'astro/config';
import tailwind from '@astrojs/tailwind';

export default defineConfig({
  site: 'https://<username>.github.io',
  base: '/portfolio',
  integrations: [tailwind()],
});

よくあるエラーと Claude への聞き方

Troubleshooting
404 が出る (公開URLにアクセスしても何も出ない)
base 設定漏れの典型。astro.config.mjs に base: '/portfolio' が入っているか、リンクは <a href={`${base}/blog`}> の形になっているかをClaudeに確認させる。
画像が壊れる
絶対パス /img/foo.png はサブパス公開で壊れる。Astroの <Image> コンポーネントか import.meta.env.BASE_URL 経由で組み立てる。
GitHub Actions が build で落ちる
Node バージョン不一致が9割。workflow.yml を node-version: 20 固定 + package.json の engines も統一。
Lighthouse が低い
画像が原因の事が多い。Astroの <Image> 必須化、loading="lazy"、WebP出力をClaudeに頼む。

仕上げ — カスタムドメインとアクセス計測

カスタムドメイン

  1. 取得済みドメインの DNS で CNAME レコードを <user>.github.io に向ける
  2. リポジトリ Settings > Pages > Custom domain に入力
  3. Enforce HTTPS にチェック (証明書は自動発行)
  4. public/CNAME ファイルを置く (内容はドメイン名のみ1行)

アクセス計測 (プライバシー優先)

Google Analyticsよりプライバシー重視の計測を推奨。Plausibleなら1行のscriptタグで完了。

<!-- src/layouts/Base.astro <head> 内 -->
<script defer data-domain="example.com" src="https://plausible.io/js/script.js"></script>

Exercise 1
入門

GitHub Pages にサイトを公開する

前章で作ったローカルサイトを、本章の手順で公開してください。最終的に https://<username>.github.io/portfolio/ でアクセスできるところまで。

  1. gh repo create portfolio --public --source=. --push
  2. .github/workflows/deploy.yml をClaudeに作らせる
  3. astro.config.mjs に site / base を追加
  4. Pages 設定で Source = GitHub Actions に
  5. push して2分待つ → 公開URLにアクセス
模範解答を見る

うまく行かない時のチェックリスト:

  1. gh run list で workflow が走ったか
  2. gh run view --log でビルドエラー有無
  3. Settings > Pages で「Your site is live at...」が表示されているか
  4. URLにアクセスして 404 が出るなら base の / 漏れを疑う (例: base: 'portfolio' ではなく '/portfolio')

解決後は npm run build && npm run preview でローカルでもサブパス挙動を確認できると、デプロイ前に検出可能になります。

Exercise 2
応用

PR プレビュー環境を追加する

main へのpushで本番デプロイ、PR では別URLでプレビューを見れるように workflow を拡張してください。Cloudflare Pagesかvercel preview相当の体験を、GitHub Actions だけで実現します。

  1. PR トリガー用の workflow 追加 (on: pull_request)
  2. ビルドした dist/ を actions/upload-artifact でPRにコメント添付
  3. もしくは peaceiris/actions-gh-pages でブランチ別パスへデプロイ
  4. PRマージで preview ブランチを掃除する step も追加
模範解答を見る

シンプル路線: Cloudflare Pages preview を使う方が現実的に楽です。Cloudflare 側で GitHubリポジトリを連携するだけで、PR毎にユニークURLが自動発行されます。

あえて GitHub Pages 縛りでやるなら:

name: PR Preview
on: pull_request
jobs:
  preview:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: withastro/action@v3
        with: { node-version: 20 }
      - uses: peaceiris/actions-gh-pages@v3
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
          publish_dir: ./dist
          destination_dir: pr-${{ github.event.number }}
      - uses: marocchino/sticky-pull-request-comment@v2
        with:
          message: |
            Preview: https://<user>.github.io/portfolio/pr-${{ github.event.number }}/

ただしAstroの base もPR番号で動的に変える必要があり、思ったより手間。実務的には Cloudflare/Vercel に任せて GitHub Pages は本番のみが多いです。
