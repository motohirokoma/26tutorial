プロンプト設計の基礎

同じタスクでも「指示の書き方」で結果は劇的に変わります。Claudeから欲しい答えを引き出すコツを身につけましょう。

所要時間: 1.5h Phase: Basics 難易度: 入門

指示の質 = 結果の質

Claudeはとても賢いですが、あなたの頭の中までは読めません。曖昧な指示には曖昧な答えしか返ってこない。料理を頼むとき「なんかいい感じで作って」と言うか「鶏むね肉400gで、塩麹漬けて、フライパンで皮目から」と言うかの差です。

指示の解像度と成果物の関係
指示が具体的になるほど、Claudeの一発回答精度は上がる。指示の手間 < 修正の手間。
「曖昧な30秒の指示」より「具体的な90秒の指示」のほうが、トータル時間は短い。 — LLM活用のラショナル原則

良い指示の4原則

良いプロンプトの4要素
この4つを意識するだけで、回答の精度が一段上がる。

悪い例 vs 良い例(ケーススタディ)

悪い例

「ログイン機能つくって」

  • 言語が不明
  • 認証方式が不明
  • DBが不明
  • セキュリティ要件が不明
良い例

具体化したプロンプト

  • 「Express + TypeScript で」
  • 「JWT 認証、bcryptでハッシュ」
  • 「PostgreSQL + Prisma の users テーブル」
  • 「テストはJestで、レート制限も追加」

具体例を見比べてみましょう。

ケース1: バグレポートを書いてほしい

❌ 悪い例
> このバグの報告書を書いて

✅ 良い例
> logs/error.log の最後の50行を読んで、
> 1) 何が起きたか 2) 影響範囲 3) 再現手順
> の3項目で、Markdown形式の障害報告を書いて。
> ターゲット読者は非エンジニアの上司。

ケース2: コードレビューを頼みたい

❌ 悪い例
> このPR見て

✅ 良い例
> PR #142 の差分を gh pr diff で取得して、
> シニアエンジニアの視点で
> - セキュリティ
> - パフォーマンス
> - 可読性
> の3軸でレビューしてください。指摘は重要度順、コード行の引用付きで。

段階的に依頼する

大きいタスクは一気に頼むと失敗しがち。「設計 → 実装 → テスト → ドキュメント」のように分割するとClaudeも人間も混乱しません。

段階分割パターン
graph LR
  A["全体を一気に頼む"] -.失敗しやすい.-> X["膨大な差分・ズレた実装"]
  B["1. 設計の確認"] --> C["2. 実装"]
  C --> D["3. テスト追加"]
  D --> E["4. リファクタ"]
  E --> F["5. ドキュメント"]
下のパスのほうが、各段階で軌道修正できる&レビューしやすい。

コンテキストの渡し方

Claudeに渡せる文脈は、おもに4種類。

  • ファイル参照 — @src/index.ts のように @ パスを指示文に書くと、Claudeは中身を読んでから返事する
  • 画像 — スクリーンショットやスケッチをドラッグ&ドロップ。「このUIに合わせて実装して」
  • URL — ドキュメントURLを貼ると、必要に応じてWebFetchで読みにいく
  • 会話履歴 — 同じセッション内の過去のやりとりは自動的に文脈になる
> @src/api/users.ts と @docs/api-spec.md を比較して、
仕様と実装のズレを箇条書きで指摘してください。

指示の「粒度」を調整する

結果がイマイチだったとき、「もっと具体的に」と指示し直すか「他の選択肢も提案して」と広げるか。状況で使い分けます。

状況指示の調整
選択肢を狭めたい「条件を満たす案を1つだけ。Aは不可」
網羅性が欲しい「他の選択肢も含めて3つ提案して、メリデメ表で」
もっと深く「なぜそうなるか、根拠も書いて」
もっと簡潔に「結論を1行 + 理由を3行で」
専門外の人向け「中学生にもわかる言葉で」

覚えておくと便利なプロンプトパターン

段取り出し

「実装する前に、まず作業手順だけ列挙して。承認後にコードを書いてください」

ロール指定

「あなたは10年経験のセキュリティエンジニアです。以下のコードを監査してください」

制約付き出力

「JSON形式で。キーは `summary` `risks` `next_steps` のみ。日本語で」

Few-Shot

「例: input=foo → output=Foo。同じ規則でbar, bazを変換して」

Self-Critique

「上の回答を、自分でレビューして弱点を3つ挙げて、改善版を出して」

テンプレート埋め

「## 概要 / ## 変更点 / ## テスト方法 のフォーマットで埋めて」

対話を重ねて磨く

1発で完璧な答えを出させるのは難しい。会話を3〜5往復するつもりで臨むと結果が安定します。

第1ターン
ざっくり投げる
「ユーザー登録機能の設計案を出して」
第2ターン
フィードバック
「2案目で。ただしOAuthは不要、メール認証だけでOK」
第3ターン
実装に進む
「OK、その案で実装開始。まずDBスキーマから」
第4ターン
仕上げ
「テストも追加して、READMEに使い方を1セクション足して」
もっと詳しく: 「思考を見せて」と頼む

難しい判断を任せるとき、「結論の前に、考えのプロセスを箇条書きで」と頼むと、Claudeはステップバイステップで推論してから答えます(Chain-of-Thought)。回答の質が一段上がる代わりに、レスポンスが少し長くなります。

Exercise 1
入門

悪いプロンプトを良いプロンプトに書き換える

下記の指示文を、「役割・タスク・制約・出力形式」の4要素を入れて書き直しましょう。

  1. 「README作って」
  2. 「テスト書いて」
  3. 「これ早くして」
模範解答を見る

例:

  1. README:「あなたは新規参加者向けのドキュメントを書く担当です。リポジトリの主要ファイルを読み、READMEを下記4セクション(概要・セットアップ・使い方・コントリビュート方法)で作成してください。日本語、Markdown、150〜200行程度で。」
  2. テスト:「src/calc.ts の addTax / discount の2関数について、Jest形式で正常系3件・異常系2件のユニットテストを tests/calc.test.ts に作成してください。テスト名は日本語、describe で関数ごとにまとめてください。」
  3. 性能改善:「server/handler.ts のレスポンス時間が遅い。Profilerは未導入。まず計測コード(console.time)を入れる手順を提案 → 実行結果を見てから改善してください。改善前後の数値を比較表で。」

共通点: 「動詞だけの指示」を「条件・形式付きの依頼」に変える。

Exercise 2
応用

あなた専用のプロンプト・テンプレを3つ作る

あなたが日常的によく頼む依頼(コードレビュー / バグ修正 / ドキュメント生成 など)について、再利用できるテンプレートを作りましょう。

  1. 頻出タスクを3つ選ぶ
  2. 各タスクについて「役割」「インプット」「制約」「出力形式」をテンプレ化
  3. そのテンプレを ~/.claude/CLAUDE.md または ./CLAUDE.md に保存(Ch09で詳述)
模範解答を見る

テンプレ例:

## レビュー依頼テンプレ
あなたは {LANG} の経験10年のシニアです。
以下のdiff(または `gh pr diff {NUM}`)を読んで、
- セキュリティ
- パフォーマンス
- 可読性 / 命名
- テスト網羅性
の4軸でレビューしてください。指摘は重要度(High/Mid/Low)タグ付き、行引用つき。
最後に「マージOK / 要修正 / 議論必要」の判定を1行で。

## バグ修正テンプレ
症状: {SYMPTOM}
再現手順: {REPRO}
期待値: {EXPECTED}
あなたは原因仮説を3つ立て、優先度順に検証して、
最小修正パッチ + テストで再現を防ぐ コードを提案してください。

## ドキュメント生成テンプレ
ターゲット読者: {AUDIENCE}
読了時間: {MIN}分
入力: {FILES}
構成: 概要 → 動機 → 使い方 → よくある質問
日本語、Markdown、コード例は最小限。

これを CLAUDE.md に書いておくと、毎回コピペ不要で「レビュー依頼テンプレで PR #142 を見て」のような呼び出しができるようになります。
