templatesディレクトリの使い方

再利用部品・プレースホルダ・パス設計 — Skillを「量産装置」に変える

所要時間: 30min Phase: Basics 難易度: 中級
この章で得るもの

templates/ の正しい置き方と、{{VAR}} プレースホルダの設計・置換の実装手順。章の終わりには、自分の Skill に templates を足して具体的な成果物を一貫生成できるようになります。2026年1月時点 の Claude Code 仕様準拠。

なぜ templates/ が必要か

Ch03 で学んだ 単一責任 と テンプレ分離 の原則を実装する場所が templates/ です。SKILL.md に長文のテンプレートをそのまま書くと、AIが毎回全文を読み込むコストが膨らみ、さらに差分修正も追いにくくなります。

SKILL.md だけ vs templates/ 分離
SKILL.md は「指示書」、templates/ は「成果物の型」。役割を分けると保守も発火も軽くなる。

このチュートリアル自体が make-tutorial という Skill で生成されています。~/.claude/skills/make-tutorial/templates/ には chapter-shell.html / index-shell.html / style.css / progress.js / firebase-config.js / firestore.rules の6本が置かれ、これが全チュートリアルの「正本」になります。

ディレクトリ構造とパス設計

Skill のルートは ~/.claude/skills/{name}/。templates/ はその直下に置く「慣習」で、Claude Code 本体が自動で読み込むわけではありません。SKILL.md から相対パスで明示的に参照するのが決まりです。

Skill ディレクトリ構造の推奨レイアウト
templates/ 以外に examples/ scripts/ も置けるが「必須は SKILL.md のみ」。

ユーザースコープ と プロジェクトスコープ のどちらでも templates/ の配置は同じです。絶対パスは書かないこと(他人の環境では動きません)。

置き場所用途チーム共有
~/.claude/skills/{name}/templates/個人用テンプレ×(自分だけ)
<project>/.claude/skills/{name}/templates/プロジェクト固有テンプレ○(Git管理可)
外部URL(GitHub raw 等)非推奨(self-contained 違反)×

プレースホルダ規約: {{VAR}}

プレースホルダ は 二重波括弧 + 大文字スネークケース が業界慣習です。make-tutorial の templates/ もすべてこの規則に従っています。

命名規則(MK推奨)

{{SLUG}} {{CLAWD_COLOR}} {{CH_TITLE}} のように 英大文字+アンダースコア。小文字や CamelCase と混ぜないこと。grep/sed で確実にヒットさせるため、単独文字の {{X}} は避ける(偶発一致のリスク)。

make-tutorial の chapter-shell.html で実際に使われているプレースホルダを、役割ごとに分類します。

カテゴリ例由来
識別子{{SLUG}} {{BRAND}} {{CH_NUM}}ユーザ入力
テーマ色{{CLAWD_COLOR}} {{CLAWD_LIGHT}} {{CLAWD_COLOR_URLENC}}Phase 3-1 選択
定数{{ACCENT_COLOR}} {{ACCENT_LIGHT}}固定値
集約結果{{NAV_LINKS}} {{SIDEBAR_SECTIONS}} {{MAIN_CONTENT}}Skill側で生成
計算値{{TOTAL_HOURS}} {{SECTION_COUNT}}章メタから集計

プレースホルダの デフォルト値 はテンプレ側では持たず、Skill 本文(AIへの指示)で吸収します。これにより、テンプレを単純な「穴」に保てます。

もっと詳しく: なぜ {{VAR}} なのか? 他の記号との比較

Jinja2 ({{ var }}) / Mustache ({{name}}) / ERB (<%= var %>) / Handlebars など多くのテンプレート言語が二重波括弧を採用しているため、「テンプレだとひと目でわかる」可読性が最大の理由です。Claude Code の公式Skillも全てこの形を採用しています。

一方、${VAR} はシェル変数と衝突しやすく、%VAR% は Windows 環境変数、${VAR} は YAML と衝突します。{{VAR}} は「自然な文章中では絶対に現れない」ため安全です。

なお、Claude Code 本体は {{VAR}} を自動では解釈しません。置換するのは Skill を実行するAI自身です。

置換の5パターン

1. 単純値

{{SLUG}} → "email-draft"

  • ユーザ入力をそのまま埋める
  • 特殊文字エスケープは Skill 側で
2. 派生値

{{CLAWD_LIGHT}}

  • {{CLAWD_COLOR}}から透明度付きに変換
  • Skillの計算ロジックが必要
3. ブロック

{{NAV_LINKS}}

  • HTML片を組み立てて挿入
  • ループで生成した文字列
4. 条件

{{PREREQ_LINK_BOX}}

  • 条件により空文字も入る
  • Ch01のみブロック展開
5. ネスト(非推奨)

{{OUTER_{{INNER}}}}

  • 置換順序の曖昧さ
  • まず {{INNER}} を解決してから

ハンズオン: 最小テンプレ+置換を実装

実際に手を動かしましょう。ここでは 日報下書き Skill を作り、templates/daily-report.md を {{VAR}} 付きで書いて AI に置換させます。

1
ディレクトリ作成
~/.claude/skills/daily-report/templates/ を作る
2
SKILL.md
frontmatter と指示を書く
3
テンプレ配置
daily-report.md にプレースホルダ
4
発火・確認
Claude Code で「今日の日報」と話しかける

ステップ1: ディレクトリ作成

mkdir -p ~/.claude/skills/daily-report/templates
cd ~/.claude/skills/daily-report

ステップ2: SKILL.md を書く

---
name: daily-report
description: 当日の作業ログから日報を下書きする。「日報書いて」「今日の日報」と言われた時や、作業終了時刻前後の雑談で発火。
---

# daily-report

## いつ使うか
- ユーザが「日報」「今日の報告」と言った時
- ユーザが「お疲れ様」的な終業言及をした時

## 入力
- 日付(デフォルト: 今日)
- 主要トピック(ユーザから引き出す。3-5項目)
- 困ったこと(任意)

## 手順
1. templates/daily-report.md を読み込む
2. 以下のプレースホルダを置換:
   - {{DATE}} → 日付 (例: 2026-04-18)
   - {{AUTHOR}} → ユーザ名
   - {{TOPICS_LIST}} → "- " プレフィックス付きの箇条書き
   - {{BLOCKERS}} → 困ったこと。なければ "特になし"
3. 置換後の本文を Markdown として表示
4. ユーザに「Slackに送る?」と聞く(送るなら別途手配)

ステップ3: templates/daily-report.md

# 日報 {{DATE}}

Author: {{AUTHOR}}

## 今日やったこと
{{TOPICS_LIST}}

## 困っていること
{{BLOCKERS}}

## 明日やること
(明日の自分へ: {{DATE}} の続きから)

ステップ4: 発火確認

Claude Code を起動して「日報書いて」と打つだけ。Skill が発火し、不足情報を聞いた上で、上のテンプレをベースにした日報が出力されます。

うまく発火しない時

Ch02で学んだとおり description が発火判定の鍵。"日報" だけでなく "report" "daily summary" も入れるとヒット率が上がります。

もっと詳しく: 置換処理を「人間が書くスクリプト」で補強する

AIに置換させるのが通常ですが、確実性が欲しい時は scripts/render.py を併置し、SKILL.md に「python scripts/render.py --date=... --topics=... を実行してその出力を使う」と書く手もあります。make-tutorial の progress.js はAI置換ですが、書き出し後に必ずプレビュー確認する運用で精度を担保しています。

簡易な Python 置換の例:

import sys, pathlib, re
tpl = pathlib.Path('templates/daily-report.md').read_text()
vars = dict(arg.split('=', 1) for arg in sys.argv[1:])
out = re.sub(r'\{\{(\w+)\}\}', lambda m: vars.get(m.group(1), ''), tpl)
print(out)

テンプレ更新と「配布済みSkill」の関係

ここが多くの人が踏む落とし穴です。templates/ を後から更新しても、すでに別のリポジトリや成果物として書き出された 配布済み成果物 には自動で反映されません。

テンプレ更新の影響範囲
graph LR
  T["templates/chapter-shell.html"] -->|更新| T2["templates/chapter-shell.html v2"]
  T2 -.->|"次回生成は v2"| N["新しい学習サイト"]
  T2 -.->|"既存には届かない"| O["既存サイトA"]
  T2 -.->|"既存には届かない"| O2["既存サイトB"]
  O -->|"手動パッチ"| P["v2適用済みサイトA"]
破線は「自動では流れない」経路。配布済みには手動で差分を当てるしかない。

make-tutorial の SKILL.md 末尾にはこのことが明記されています:

ユーザから「/make-tutorialのデザインを変更して」等の指示があった場合...
既に生成済みのサイト群(~/Desktop/claude/26tutorial/*)は自動追従しない旨をユーザに告知。必要なら追従対象を明示してもらい、差分パッチを各サブフォルダに手動適用してまとめて push。 — ~/.claude/skills/make-tutorial/SKILL.md

戦略: バージョニングと追従

templates 変更で影響が広がる度合い
  • バージョン表記 — templates/VERSION に "v1.3" 等を置き、生成時に成果物にも刻印する
  • 追従スクリプト — 既存全サイトに差分を当てる shell を scripts/upgrade.sh で用意
  • 破壊的変更は避ける — プレースホルダ削除や意味変更はメジャーバージョン昇格扱い
  • 追加は積極的に — 新しい {{VAR}} は「未定義なら空文字」ルールで後方互換が保てる

実例: make-tutorial の templates を読み解く

本チュートリアル自身の「正本」である ~/.claude/skills/make-tutorial/templates/ を覗いて、実戦の設計を学びます。

ファイル行数役割代表プレースホルダ
chapter-shell.html123章ページの骨格{{CH_TITLE}} {{MAIN_CONTENT}} {{NAV_LINKS}}
index-shell.html174トップページの骨格{{PHASE_SECTIONS}} {{PHASE_DISTRIBUTION_BARS}}
style.css1642デザイン一式(置換なし・そのままコピー)
progress.js394進捗ロジック{{SLUG}} {{CHAPTERS_ARRAY}} {{STORAGE_KEY}}
firebase-config.js55Firebase初期化(置換なし・そのままコピー)
firestore.rules13アクセス制御(初回だけユーザが貼る)

chapter-shell.html のプレースホルダ出現数

どのプレースホルダが多く埋まっているか

特徴的なのは テーマ色系が独立カテゴリとして切られている点です。{{CLAWD_COLOR}} / {{CLAWD_LIGHT}} / {{CLAWD_COLOR_URLENC}} の3本セットで、「1色決めれば全視覚要素に波及する」という設計になっています。favicon の data-URI には URLエンコード版が必要なため、%23D97757 のような別プレースホルダを用意する。派生値は全部 Skill 側で計算して埋めるのが鉄則。

もっと詳しく: なぜ style.css と firebase-config.js は置換なしなのか

全チュートリアルで挙動が同一な「不変のコード」は、プレースホルダ無しで そのままコピー します。テンプレ側で分岐を持たないので、更新時の追従は「ファイルを上書きするだけ」で済みます。

逆に progress.js は {{SLUG}} で Firestore パスを分離するため、各サイト固有の値を持たせざるを得ません。この「固有値」の数をいかに絞るかがテンプレ設計の肝です。make-tutorial では6つに収まっています。

章の間にある細やかな統一感(カニの色、h2の飾り、余白)は、すべて chapter-shell.html 1本で定義されています。あなたの Skill でも 「骨格ファイル1本」+「中身をはめるプレースホルダ」という構造を目指しましょう。

Exercise 1
入門

PRタイトル整形テンプレを書く

PRタイトルを一貫した形式で生成する Skill pr-title を想像してください。以下の3プレースホルダを含む templates/pr-title.md を作成してください。

  1. {{TYPE}}(feat / fix / chore)
  2. {{SCOPE}}(該当モジュール名、例: auth)
  3. {{SUMMARY}}(1行要約)

形式は Conventional Commits 準拠で。最終出力が「feat(auth): add OAuth flow」のようになる想定。

模範解答を見る

~/.claude/skills/pr-title/templates/pr-title.md:

{{TYPE}}({{SCOPE}}): {{SUMMARY}}

SKILL.md 側で次のように指示:

## 手順
1. templates/pr-title.md を読み込む
2. {{TYPE}} は feat/fix/chore/docs/refactor/test/perf から選択
3. {{SCOPE}} は変更ファイルのトップレベルディレクトリ名
4. {{SUMMARY}} は50文字以内、命令形、末尾ピリオドなし
5. 置換結果をそのまま提示

「テンプレは1行でいい」という点が肝。短くても役割分離の価値は十分にあります。

Exercise 2
応用

自分の既存Skillに templates/ を導入する

Ch03 までで作った自分の Skill(なければ Ch02 の hello Skill をベースに)を拡張し、以下を実施してください。

  1. templates/ を新設し、最低1本のテンプレファイルを配置(拡張子は任意)
  2. テンプレ内に {{VAR}} を最低3つ含める
  3. SKILL.md 本文から相対パス templates/xxx で参照を追加
  4. 発火させて、置換結果を確認
  5. 意図的にテンプレを更新し、次回生成に反映されることを確認

自分の業務ドメインに引き寄せて設計すること(例: 調査レポートの骨子、コードレビューコメントの型、1on1アジェンダ等)。

模範解答を見る

例: 1on1 アジェンダ Skill one-on-one。

~/.claude/skills/one-on-one/
├── SKILL.md
└── templates/
    └── agenda.md

templates/agenda.md:

# 1on1 Agenda: {{MANAGER}} × {{REPORT}}

Date: {{DATE}}
Duration: 30min

## Check-in (5min)
- 最近の気分
- 体調・睡眠

## 仕事 (15min)
- 今週の主要タスク
- ブロッカー

## キャリア (10min)
- 来期の目標
- 学びたいこと

## Action Items
- [ ] (記入)

SKILL.md で「マネージャ名・メンバー名・日付を聞いてから templates/agenda.md を読み込み置換」と指示。テンプレを更新 → 再発火 で出力に反映されることを確認すれば完了。

追加の気づきポイント: {{DATE}} のフォーマットを ISO-8601 で統一すると、後で Notion / Slack 貼り付け時にソートしやすい。
