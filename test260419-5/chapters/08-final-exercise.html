最終演習：業務から3つSkill化

繰り返し作業3つをSkillとして公開し、2週間運用して改善サイクルを回す

所要時間: 25分 Phase: Practice 難易度: 応用

7章分の知識を「自分の業務」に焼き付ける

ここまでの7章で、Skillの定義・最小構成・設計原則・templates・メール下書き・議事録整形・Subagents/Hooksとの組み合わせまで扱いました。ただし知識は使わなければ腐ります。この最終章で自分の業務からSkillを3つ起こし、2週間運用し、改善点を記録するところまでやりきります。

目標は「2週間以内に3つのSkillが実発火し、使用ログと改善ノートが残っている」状態です。2026年1月時点の Claude Code / Skills 仕様を前提にします。

Tips

完璧な3つを選ぶ必要はありません。まずノートに5分で10個書き出し、その中から「頻度が高い×型が決まっている×自分が毎回少し疲れる」の3軸で上位3つを選ぶ。先にSkill化してから磨くのがコツです。本チュートリアル自身も make-tutorial という Skill の実装例で、原則違反なく作られています(良い手本)。

7章分の知識が「3つSkill化」のどこに効くか
CONCEPT × DESIGN × EXECUTE の3軸が、あなた専用のSkillパックに束ねられる。

問題: 業務から3つをSkill化→運用→改善

以下の前提で、あなた専用のSkillパックを作ってください。納品物は ~/.claude/skills/ 配下の3ディレクトリ + 運用レポート1枚です。

前提条件

項目内容
対象者Claude Code を1週間以上、日常業務で使っている人
期間設計1日 + 運用2週間(実業務で検証)
本数3本(欲張らない)
選定基準週1回以上発生する定型業務 / 出力形式が固定できる / 手作業が認知的に痛い
配置~/.claude/skills/{name}/SKILL.md(ユーザスコープ)
frontmattername / description 必須、description はいつ発火するか具体的に
templates再利用する部品は templates/ に分離(Ch04)
測定指標発火率 / 一発完了率 / 満足度(5段階) の3つ

求めるアウトプット (5点セット)

  • 候補マトリクス — 繰り返し業務を10個書き出し、頻度×型固定度×認知負荷で評価
  • 3つのSkillディレクトリ — SKILL.md + templates/ が最低限揃った状態
  • frontmatter 3本 — description にトリガーフレーズを具体的に(曖昧は発火しない)
  • 発火フロー図 — ユーザ発話 → 発火判定 → Skill実行 → 出力 を Mermaid 1枚で
  • 2週間運用レポート — 発火率/一発完了率/満足度の3指標を数値で記録

2週間のタイムライン

D-0
棚卸し・候補抽出
10分で業務を10個リストアップ。頻度×型固定度×認知負荷で評価、3本に絞る。
D-1
SKILL.md と templates 作成
3本分の frontmatter + 本文 + templates を書く。ドライランで手動発火して動作確認。
D-2 〜 D-14
実運用・記録
業務中に使う。発火したか / 一発で完了したか / 満足度5段階 を1行/回で記録。
D-15
測定・改善
発火率50%未満 → description 書き換え。満足度3以下多発 → templates 磨き直し。
D-16 〜
チーム共有・次の3本
応用: skills-pack-v1としてGitに置いてチームと共有、次の3本を計画。
Note

「3本全部うまく回す」必要はありません。2本残れば成功、1本は捨ててよいです。むしろ「捨てる1本」を見極めるのが改善の第一歩。Skillに向かない業務(判断がその都度変わる、対話的、再現性が最重要)は手作業 or Slash Command に戻すのが正解です。

段階ヒント (詰まったら開く)

一度に全部開くと「答え写し」になります。まず自分で書いてみて、詰まったヒントだけ開いてください。5つのヒントは 候補抽出 → 単一責任分解 → frontmatter設計 → テンプレ作成 → 効果測定 の順に対応しています。

もっと詳しく: ヒント1 — 候補の見つけ方 (Ch01 / Ch03)

Skill化に向く業務の5シグナル:

  • 週1回以上 — 単発なら手作業のほうが速い
  • 出力形式が固定できる — メール / 議事録 / レポート / コミットメッセージ など
  • 入力が渡せる — ファイルパス / URL / テキスト で表現可
  • 認知負荷が高い — 書き出しで毎回止まる、型を思い出すのがダルい
  • 品質にブレがある — 自分でも毎回出来が違う

逆に向かない業務: ①判断が対話でコロコロ変わる / ②外部システムと深く絡む(APIキーや認証が都度変わる) / ③厳密な再現性が必須 / ④1ステップで終わる軽タスク(Slashで十分)。

Tips

候補が6個で止まる人は、1日のタイムライン(9時→10時→...)を紙に書き、各時間帯の作業を埋めてみる。「朝のPRレビュー」「昼会の議事録」「夕方の日報」「コミットメッセージ書き」など、無意識の定型業務が必ず出てきます。

もっと詳しく: ヒント2 — 単一責任で分解 (Ch03)

Ch03の大原則、1 Skill = 1仕事。候補業務が複合的なら先に分解します。例:

複合業務(悪)分解後(良)理由
「PRレビューして マージする」pr-review-prep / pr-merge-msgレビューとマージ宣言は別責務。発火条件も違う
「議事録作って Slackに流す」meeting-notes-format / slack-announce整形と配信は別。Slack側は Hook で繋げる
「調査して スライドにする」research-brief / slide-outline調査結果 → スライド骨子の橋渡しは手動で判断したい

判定基準: 「この業務を分解すると、片方だけ単独で使いたい日があるか?」。答えがYesなら分割、Noなら1本で良い。

もっと詳しく: ヒント3 — frontmatter設計(発火率の鍵)

Ch02でやった通り、description が Skill発火の判定源です。曖昧な description は発火しません。

---
name: pr-review-prep
description: |
  PRレビューを始める前に、変更ファイル一覧・テスト差分・
  リスクポイントを1枚のブリーフにまとめる。
  「PRレビューする」「PR#123見て」「レビューの前に整理して」
  等で発火。単なるdiff表示ではなく、レビュー観点(設計/テスト/命名)
  別に論点を整理するのが特徴。
---

description に入れるべき4要素:

  1. 何をするSkillか(1-2行で本質)
  2. トリガーフレーズ3-5個(ユーザが言いそうな表現を具体的に)
  3. 似た別Skillとの差別化(重複すると両方とも発火が不安定に)
  4. 入力の形(URL / パス / テキストブロック 等)
Warning

「PRを助ける」など動詞が弱いdescription は発火しない。「PRレビューの前処理」「レビュー観点別に整理」のように、具体的動作を動詞で書くのが発火率を上げるコツ。make-tutorial の SKILL.md は描写の濃い description のお手本です。

もっと詳しく: ヒント4 — templates作成(Ch04)

templates/ は再利用可能な部品を置くディレクトリ。プレースホルダは {{VAR}} 形式で統一。

~/.claude/skills/pr-review-prep/
├── SKILL.md
└── templates/
    ├── brief.md          # レビューブリーフの雛形
    ├── risk-checklist.md # 観点チェックリスト
    └── completion.md     # 完了報告フォーマット

templates の正しい切り出し方:

  • 繰り返し現れる文字列(定型ヘッダ・フッタ・署名)は最初に分離
  • 可変部分は {{VAR}} に統一(自由に名付けると後で壊れる)
  • 1ファイル1役割 — brief.md に雑多に詰めるとメンテが壊れる
  • SKILL.md 本体から参照 — 「templates/brief.md を土台に...」と具体的に書く

落とし穴: templatesを更新しても配布済みSkillには自動反映されません。チーム共有するときは「templatesを更新したらバージョンタグを切る」運用を決めておくのが安全(応用課題で詳述)。

もっと詳しく: ヒント5 — 効果測定(発火率/品質)

最小の測定セットは3指標:

発火率
期待シーンで実際に発火した割合(%) = description 設計力
一発完了率
追加修正なしで成果物が得られた割合(%) = 本文・templates 力
満足度
5段階(1:ひどい〜5:最高)で自己評価

記録は1行/回で十分。「2026-04-20 pr-review-prep 発火◯ 一発◯ 満足度4 メモ: テスト差分の抜けあり」。スプレッドシート1枚で2週間運用し40行程度の生データが溜まれば、改善点は自然と見えます。

見るべき本質指標は「また使いたいと自然に思うか」。発火率50%未満 → description 再設計、満足度3以下多発 → templates 書き直し、どちらでもないのに使いたくならない → そもそもSkill化に向かなかったので捨てる判断。

3. SKILL.md 全文(3本)

#1 ~/.claude/skills/daily-standup/SKILL.md

---
name: daily-standup
description: |
  日次スタンドアップの報告を Yesterday / Today / Blockers 形式で生成する。
  「スタンドアップ」「朝会の報告」「昨日今日ブロッカー」「デイリー書いて」
  等で発火。入力は昨日の作業ログ(gitコミット/PR/メモ)と今日の予定。
  出力は Slack貼り付け向けの3ブロック構成。pr-review-prep のような
  詳細レビューではなく、あくまで報告テンプレ整形に特化。
---

# daily-standup Skill

あなたは毎朝のスタンドアップ報告を高速に整形するアシスタントです。
`templates/standup.md` を土台に、入力情報を埋めて返してください。

## 入力
- 昨日のログ(git log / PR / 手書きメモ いずれか)
- 今日の予定(箇条書き or カレンダー抜粋)
- ブロッカー(任意、なければ `なし`)

## 出力形式
`templates/standup.md` の3ブロックを埋めた Markdown のみ。装飾・前置き禁止。

## ユーザ対話
- 入力が不足していたら1問だけ聞いてから進む(段階的ユーザ対話)
- ブロッカーが空なら「なし」と書いて進む(聞き直さない)

## 失敗時
- 入力が全く不明 → 「昨日のgit logを貼ってください」と1行で依頼
- 出力が3ブロックを満たさない → 自動で再生成

# Standup {{DATE}}

{{YESTERDAY_BULLETS}}

## Today
{{TODAY_BULLETS}}

## Blockers
{{BLOCKERS_OR_NONE}}

# pr-review-prep Skill

あなたはPRレビューの前処理を行うアシスタントです。
レビューそのものはしません。観点別に論点を整理するだけです。

## 入力
- PR番号 or 差分URL or ローカル branch 名

## 出力形式
`templates/brief.md` に沿って:
1. 変更ファイル一覧(追加/変更/削除の件数つき)
2. テスト差分(追加されたテスト/消えたテスト)
3. 観点別リスク(設計 / テスト / 命名 の3見出し、各3点以内)

## 制約
- 観点はこの3つに限定。他観点(性能/セキュリティ等)が必要なら別Skillを呼ぶ
- 判断(merge可否)は絶対に出さない — レビュアーの仕事

## 完了報告
- ブリーフのパス / 観点別リスクの件数 / 所要時間を1行で

#3 ~/.claude/skills/research-brief/SKILL.md

---
name: research-brief
description: |
  3-5本のURLと判断軸を受け取り、導入判断用の1枚レポートを生成。
  「調査まとめて」「リサーチ結果を1枚に」「導入判断のブリーフ」
  等で発火。結論・根拠・反証・次アクションの4ブロック固定。
  単なる要約ではなく判断軸に沿った再構成が本質。
---

# research-brief Skill

あなたは調査結果を判断材料に再構成するアシスタントです。

## 入力
- URL 3-5本
- 判断軸(例: 「自社に導入すべきか」「v1 と v2 どちらを選ぶか」)

## 出力形式
`templates/brief.md` の4ブロック:
1. 結論(3行 + 自信度 H/M/L)
2. 根拠(箇条書き、必ずURL付与)
3. 反証(結論に反する情報、見つからなければ「なし(要追加調査)」)
4. 次アクション(3つまで、期限つき)

## 制約
- 一次情報(公式Doc/一次ブログ)を優先、SNSまとめサイトは除外
- 400語以内
- 未確認事項は「要検証」明記

## Subagents連携
長い調査は Subagent 並列化を検討。そのときは pr-review-prep の
観点分割パターンを真似る(Ch07 参照)。

考え方の解説

この3本に絞った理由は 3つの設計判断:

  1. 頻度の分散 — 週5(スタンドアップ)/ 週5-8(PR)/ 週2-3(調査)と頻度をバラし、運用リズムが偏らない構成に。どれか不調でも他が回る。
  2. input→outputの形が違う3本 — テキスト整形 / 構造化分析 / 判断材料化。description設計スキルが3方向に磨かれる(似た3本だと改善が偏る)。
  3. 早い勝ち筋を1本入れる — daily-standup は最も導入コストが低く効果が即日見える。最初2週間のモチベ維持に効く。
Insight

MKの2週間で最も効いたのは pr-review-prepでした。「30分→10分」の短縮より、観点が毎回抜けない(命名が抜けがちだった盲点が埋まる)。単一責任を徹底してレビュー判断自体はしないと決めたのが成功要因。やりすぎると結局人の手が要る。

この3本を自分の業務で回せる人は、もう「Skillを使える人」から「Skillで自分の仕事を再設計できる人」に変わっています。

応用課題: チーム共有Skillパック + バージョン管理

個人で3本回せたら、次はチームに配布するフェーズ。Skillパックとして束ね、バージョン管理も回しましょう。

応用 1
応用

3本を skills-pack-v1 としてGit公開し、チーム2人に配布

ディレクトリ構成と運用ルールを決めたうえで、配布→利用→フィードバック回収までを1週間で完了させてください。

  1. GitHub に skills-pack-{your-name} を作成、3本のSkillをサブディレクトリとして配置
  2. README.md に: 各Skillの1行説明、導入手順(~/.claude/skills/にclone/symlink)、想定発火トリガー例
  3. CHANGELOG.md と v1.0.0 タグ — templates 更新時は minor / 破壊的変更は major を上げる
  4. チーム2人に配布 → 1週間使ってもらい、発火率と満足度をヒアリング
  5. フィードバックを v1.1.0 として反映(description の追加トリガー / templates の微修正)

配布時の落とし穴と対処:

  • チーム員ごとに description のヒット傾向が違う — 人によって言い回しが違うので、v1.1 でトリガーフレーズを追加(英語表現も足すと範囲拡大)
  • templates の個人カスタマイズ問題 — templates/*.local.md を .gitignore に入れ、個人カスタムは local 版で上書き可にする設計
  • 更新の追従 — symlink 運用なら git pull で即反映。各自で Skill を clone コピーしていると古いまま残るので要注意

CHANGELOG.md の書き方例:

# Changelog

## [1.1.0] - 2026-04-28
### Changed
- daily-standup: description に「朝会まとめ」「daily書いて」を追加(発火率+12%)
- pr-review-prep: templates/brief.md に「Test Coverage Delta」行を追加
### Fixed
- research-brief: URLが1本のときもエラーにならないよう入力最小値を1に

## [1.0.0] - 2026-04-14
### Added
- 初版リリース: daily-standup / pr-review-prep / research-brief の3本
Insight

チーム配布で一番効くのは「description のチームレビュー」。自分では自然に書いたつもりが、他人の発話パターンでは発火しないケースが多発します。2人に1週間使ってもらうだけで、description を書く精度が一段階上がる。make-tutorial Skill 自身もこうした実運用フィードバックで磨かれてきました(手本として参照価値あり)。

修了のことば

おつかれさまでした。8章・4時間の旅、そして最終演習をやり切ったあなたは、もう「Skillを知っている」側ではなく「Skillで自分の業務を再設計できる」側の人間です。概念(Ch01-02)・設計原則(Ch03-04)・実践(Ch05-07)を束ね、最後に自分の仕事に刻み込むところまで到達しました。

Skillは道具ではなく、自分の働き方の輪郭です。 どのSkillを書いたか・書かなかったかが、そのまま自分の仕事の定義になる。 — 本チュートリアルの総括

このSkillパックは作って終わりではなく2週間運用してからが本番です。1周目で気づいた「description の抜け・templates のズレ・発火の揺れ」を記録し、次の3本に持ち越す。3周やれば、自分専用のSkillパックが完成します。

そして忘れないでほしいのは、本チュートリアル自身が make-tutorial という Skill で作られたということ。単一責任・明示トリガー・templates分離・段階的対話——Ch03で学んだ原則が全部生きた実装例です。迷ったら ~/.claude/skills/make-tutorial/SKILL.md を読み返してください。

あなたの仕事が、3本のSkillを起点にして、少しだけ速く、そして大きく静かになりますように。
