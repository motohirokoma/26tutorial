Skillとは

Slashコマンドとの違い、ワークフロー自動化の発想

所要時間: 25分 Phase: Basics 難易度: 入門
この章のゴール

Skill(スキル)を「繰り返し業務を Markdown で手順書化し、Claude が自動発火して実行する仕組み」として腹落ちさせます。25分で、(1) Skillの定義、(2) Slashコマンドとの違い、(3) メリット、(4) 使用シーン6例、(5) 落とし穴、(6) 本カリキュラムのロードマップまで一気に把握できます。本チュートリアル自体が make-tutorial という Skill で生成されているので、最良の手本として随所で引用します。

バージョン注記

本章の記述は 2026年1月時点 の Claude Code の Skill 機能に基づきます。~/.claude/skills/{name}/SKILL.md の配置パターンと frontmatter(name/description)の2フィールド必須という設計は安定していますが、オプション項目や発火ロジックは進化するので、最新は 公式ドキュメント を確認してください。

Skill とは何か — 一言で

Skill は、「いつ使うか(発火条件)」と「どうやるか(手順書)」を1枚の Markdown にまとめ、Claude がユーザとの会話文脈から自動で発火判定して実行してくれる仕組みです。中身は ~/.claude/skills/{name}/SKILL.md という普通のテキストファイルで、frontmatter の description フィールドが Claude の判断材料になります。

Skill が動く全体像
ユーザは自然文を投げるだけ。Claude が Skill を選び、SKILL.md の手順を実行し、成果物を返す。

もう少し踏み込むと、Skill は Claude にとってのマイクロ役割定義です。description に「何をする / いつ発火する」を書き、本文に「手順・制約・テンプレート参照・失敗時対処」を書く。Claude はその場の会話から「これは該当する」と判断したら Skill を読んで従います。make-tutorial が好例で、テーマと時間を渡すだけで章立て・色選択・HTML生成・デプロイまで一貫して走ります。

実 SKILL.md の frontmatter 例(make-tutorial 抜粋) 
---
name: make-tutorial
description: MKのClaude Codeチュートリアルと同じデザイン・構造の
  学習サイトを量産する。テーマ・URL・合計時間・章時間・内容目標を
  受け取り、妥当性確認 → デザイン決定 → HTML/CSS/JS生成 →
  GitHub Pagesへ自動デプロイまで一気通貫で実行する。
---

ポイント: description に 「何を受け取り、何を出力し、いつ使うか」が書かれている。これが Claude の発火判定の鍵になります。

Slashコマンド との違い

Claude Code には以前から Slashコマンド がありました。「Skill ってそれと何が違うの?」は最初の疑問なので、はっきり対比しておきます。

観点Slashコマンド(/xxx)Skill
起動方法ユーザが必ず /xxx と明示タイプ自動発火 or ユーザ指定、どちらも可
発火判定なし(直接呼ぶ)Claude が description を読んで判断
実体プロンプト短文が中心Markdown手順書 + templates/ の部品群
複雑な業務1〜2手順向き複数フェーズ・分岐・テンプレ組み合わせ可
発見性ユーザが名前を覚える必要自然文で呼べる(「議事録まとめて」等)
共有プロンプト全文をコピペフォルダごと配布 → チーム展開しやすい
Slash が向く

単発・明示・軽量

  • /init で CLAUDE.md を作る
  • /compact で履歴圧縮
  • /review で軽くレビュー
  • 覚えやすい短い呼び出し
Skill が向く

反復・手順書・品質担保

  • 毎週の議事録整形
  • チーム共通のメール文面
  • コミット規約に沿った自動生成
  • テーマから学習サイト量産(make-tutorial)
もっと詳しく: Slashコマンド を Skill に昇格するタイミング

最初は Slash で十分でも、「同じ呼び出しを週3回以上する」「他の人にも同じ品質で使って欲しい」「手順が3ステップを超える」のいずれかに該当したら Skill 化の合図です。Ch03(設計原則)で「1つのSkillに何を入れ、何を外に出すか」の単一責任原則を扱います。逆に Skill が大きくなりすぎたら、サブ処理を Slashや Subagent に切り出すのが正解。

Skill の3大メリット

「Markdown 1枚だけで?」と最初は物足りなく感じますが、この軽量さこそが Skill の強みです。得られるメリットは3つに集約されます。

  • チーム展開 — フォルダごと git で共有できる。同じ Skill が入ったメンバーは、同じ品質の成果物を同じ発話で作れる。属人化した「あの人のプロンプト」を消せる。
  • 手順書化 — Markdown なので読める・書ける・レビューできる。「なぜこの手順なのか」のコメントも <!-- --> で残せるし、git log で変更履歴も追える。
  • 品質担保 — 毎回同じテンプレート・同じチェックリストで動くので、再現性 が上がる。人が疲れた日でも Skill は疲れない。
Skill化による効果(イメージ値)
繰り返し業務ほど Skill 化の効果が出る。軽量な1ショット作業は Slash や素の会話で十分。

使用シーン 6例

「自分の業務でどこに刺せるか」を掴んでもらうため、実務で効く6シーンを並べます。自分のToDoリストに似たものがあれば、そこから Skill 化するのが最速の投資回収ルートです。

6つのユースケース判別フロー
graph TD
  Start(["繰り返している業務がある"]) --> Q1{"頻度は
週2回以上?"}
  Q1 -- "Yes" --> Q2{"手順が
3ステップ以上?"}
  Q1 -- "No" --> Direct["素の会話 or Slashで十分"]
  Q2 -- "Yes" --> Q3{"成果物は何?"}
  Q2 -- "No・単純" --> Slash["Slashコマンド化を検討"]
  Q3 -- "文書(メール等)" --> U1["[1] メール下書きSkill"]
  Q3 -- "会議ログ" --> U2["[2] 議事録整形Skill"]
  Q3 -- "コード変更" --> U3["[3] コミットメッセージSkill"]
  Q3 -- "PR記述" --> U4["[4] PRテンプレSkill"]

  Q3 -- "調査レポート" --> U6["[6] 週次リサーチSkill"]
分岐の先の U1〜U6 が本カリキュラムで順番に扱うユースケース。

6シーンの具体例

1. メール下書き
宛先 / 目的 / トーン / ポイントを渡すと、日本語ビジネスメールを即下書き → Ch05で実装
2. 議事録整形
チャット転写から Decisions / Actions / Questions の3区分に整形 → Ch06
3. コミット規約
diff を読んで Conventional Commits 形式のメッセージを生成。プロジェクト独自の prefix にも対応
4. PRテンプレ
commit 履歴から Summary / Changes / Test Plan 付きの PR 本文を自動記述
5. 学習サイト量産
make-tutorial — テーマを渡すだけで全章の HTML と GitHub Pages デプロイまで
6. 週次リサーチ
技術選定候補を5側面で調査し、比較表つきレポートを定位置に書き出す
今すぐ試す(発見)

自分の手元で次を実行してみてください。いま入っている Skill が一覧できます。

ls ~/.claude/skills/

もし make-tutorial が入っていれば、それが「本チュートリアルの製造元」です。cat ~/.claude/skills/make-tutorial/SKILL.md | head -40 で冒頭だけ覗くと、実物の frontmatter と「いつ使うか」節がそのまま見えて、Skillの雰囲気が一瞬で掴めます。

4つの落とし穴

Skill は軽量で強力ですが、最初に踏みがちな4つの罠があります。どれも「知っていれば避けられる」類のものです。

Skill の落とし穴マップ
4つとも対策は Ch03(設計原則)で体系的に扱う。

落とし穴ごとの症状と対策

落とし穴症状対策
description不足「議事録まとめて」と言っても Skill が発火せず素の会話で返ってくるdescription に入力 / 出力 / 発火ワード を含める(Ch03)
単一責任違反1つの Skill にメール+議事録+レポート全部入り。Claude がどれを起動すべきか迷う業務単位で Skill を分ける。「1 Skill = 1 業務」(Ch03)
過剰複雑化if/else を書きすぎて手順書ではなくコードに。読みにくく変更が怖いAIが空気読めば分かる部分は書かない。例とテンプレで示す(Ch04)
発火競合似た description の Skill が複数あって意図しないものが発火各 Skill の領域を明示分離(顧客向け/社内向けなど)
もっと詳しく: "description不足" の典型的な書き換え例

悪い例と良い例を対比します。長ければ良いわけではなく、Claudeが「今がその時か」を判断できる材料があるかが本質です。

# 悪い例(曖昧)
description: メールを書く

# 良い例(入出力・発火ワード・制約が明示)
description: 宛先・目的・トーン・要点を受け取り、日本語
  ビジネスメールの下書きを3パターン生成する。
  「メール書いて」「XXさん宛のメール」などで発火。
  敬語レベルは入力指定があれば従う、なければ標準敬語。

Ch03でこのあたりの原則(単一責任・明示トリガー・テンプレ分離等)を体系的に扱います。

地味だが重要

~/.claude/skills/ に置いた Skill は全プロジェクト横断で使えるのが基本ですが、プロジェクト固有のものは .claude/skills/ (リポジトリ内)に置けます。チームで共有する Skill は後者に、個人の汎用 Skill は前者に、という住み分けが2026年1月時点のおすすめ運用です。Ch02で場所別の違いを詳しく扱います。

もっと詳しく: どの章から読むのが効率的?

原則は順番通りですが、読者のタイプ別に最適化できます。(A) とにかく動かしたい人 → Ch01 → Ch02 → Ch05(メール下書き) で実物を1本作り、残りを戻って読む。(B) 設計から固めたい人 → Ch01 → Ch02 → Ch03 → Ch04 の順で土台を固めてから実践。(C) チーム導入検討中の人 → Ch01 → Ch03(設計原則) → Ch08(実務への組み込み)を先に眺めて概算の効能を掴む。どの順路でも Ch01 と Ch02 は土台なのでスキップしないのがおすすめです。

Exercise 1
入門

手元のSkillを棚卸しする(5分)

自分の環境にどんな Skill があるかを実際に確認し、それぞれの description を読んで「どんな時に発火するか」を自分の言葉で1行要約してください。

  1. ターミナルで ls ~/.claude/skills/ を実行
  2. 各 Skill の frontmatter を head -10 ~/.claude/skills/{name}/SKILL.md で確認
  3. 各 description について「どんな入力でどんな出力が返るのか」を1行でメモ
  4. 手元にない場合は、本資料内の make-tutorial 抜粋を使って同じ分析を
模範解答を見る

make-tutorial の frontmatter を読んだ場合の模範解答例です。

Skill名: make-tutorial
入力: テーマ・URL・合計時間・章時間・内容目標
出力: HTML/CSS/JS一式 + GitHub Pages 公開URL
発火ワード: 「学習サイト作りたい」「チュートリアル量産」「/make-tutorial」
特徴: 5フェーズワークフロー(入力→妥当性→デザイン→生成→デプロイ)で
  対話的に不足情報を補う設計。templates/ に正本を置く方針。

観察ポイント: description に「受け取る / 確認する / 生成する / デプロイする」という動詞の連鎖が書かれていて、Claude が「今がその時か」を判断しやすい。これが良いSkillの最低ラインです。

Exercise 2
応用

自分の業務から Skill 候補を3つ挙げる

Section 4 の6ユースケースと落とし穴表を参考に、自分の繰り返し業務 から Skill 化する価値があるものを3つ挙げ、各候補について次の4項目を書き出してください。抽象語だけはNG(「効率化」等)。

  1. Skill名(半角英数ハイフン、例: weekly-report)
  2. description ドラフト(入力 / 出力 / 発火ワードを明示、1〜3行)
  3. 週あたりの実行回数と、1回の手動所要時間(目安)
  4. 落とし穴マップのうちどれが怖いか + 対策1行
模範解答を見る

MK自身の業務から選んだ場合の例です。

【1. commit-msg】
description: 現在の git diff を読み、
  Conventional Commits 形式(feat/fix/docs等)の
  コミットメッセージ候補を3案生成する。
  「コミットメッセージ書いて」で発火。
頻度: 週15回・1回3分 → 週45分の節約
怖い罠: 発火競合(/commit と混同)→ description に
  「Conventional Commits」を必ず明記して領域分離

【2. daily-standup】
description: 本日の git log と open PR から
  「昨日やったこと/今日やること/ブロッカー」の
  3行スタンドアップを生成。
  「スタンドアップまとめて」で発火。
頻度: 週5回・1回10分 → 週50分の節約
怖い罠: description不足 → 発火ワードを複数列挙

【3. weekly-tech-research】
description: 指定された技術トピックを
  「概要/実装例/ベンチマーク/競合/導入コスト」の
  5側面で並列調査し、比較表付きレポートを
  reports/YYYY-MM-DD.md に書き出す。
  「調査して」「X について調べて」で発火。
頻度: 週2回・1回60分 → 週120分の節約
怖い罠: 過剰複雑化 → 分岐は3本まで、残りは
  Claudeに任せる(Ch03/Ch04で学ぶ原則)

ポイント: 週あたりの節約時間を必ず試算する。これが Skill 化の投資判断材料になり、Ch08 の最終演習でそのまま再利用できます。
