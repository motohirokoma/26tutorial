Skillの最小構成

SKILL.md・frontmatter・呼び出し方

所要時間: 30分 Phase: Basics 難易度: 初級
この章のゴール

Skillの物理的な姿を理解する——ディレクトリはどこに置くのか、SKILL.md の中身は何でできているのか、frontmatter のname/description は何に使われるのか、そしてどうやって起動するのか。この30分で "hello" と挨拶するだけの最小Skill を自分の手で1つ作り、/hello で呼び出して動かすところまで到達します。ここが腑に落ちれば、Ch03以降の設計論・実践章は全部あなた自身のSkillで読めるようになります。

バージョン注記

本章の記述は 2026年1月時点 の Claude Code における Skill 機能の挙動に基づきます。~/.claude/skills/ のスキャンタイミング・frontmatter の拡張フィールド・自動発火のアルゴリズムは進化するため、手元で動かない時は 公式ドキュメント で最新仕様を確認してください。

Skill の物理配置 — どこに・何を置くか

Skill の正体は、たった1つの Markdownファイル(SKILL.md)が入った 名前付きフォルダ です。Claude Code は起動時にこのフォルダ群をスキャンして「使えるSkill一覧」を作ります。

最小構成 vs 実用構成
最小は「フォルダ+SKILL.md」だけ。templates/ や examples/ は必要になったら足す。

この SKILL.md こそ Skillの全て です。Claudeはこのファイルを発火時に本体を読み込んで指示書として実行します。つまりあなたは「誰か(Claude)に手順書を書いて渡す」だけでSkillが作れます。プログラムは不要です。

スコープ2種 — ユーザー / プロジェクト

配置場所は ユーザースコープ と プロジェクトスコープ の2種類。役割が違うので最初に切り分けます。

スコープ配置パス有効範囲向いてる用途
ユーザー~/.claude/skills/{name}/SKILL.mdそのマシンの全プロジェクトメール下書き・議事録整形・make-tutorial など横断的な業務
プロジェクト<repo>/.claude/skills/{name}/SKILL.mdそのリポジトリ内のみ特定プロダクトのPR規約・コミット規約・デプロイ手順
迷ったら

まずユーザースコープから始めるのが鉄則。自分ひとりで試して良かったSkillだけ、チーム共有のためプロジェクトスコープに昇格させる(gitで追跡する)、という流れがストレスが少ない。~/.claude/skills/ は他の人のリポジトリを壊さないサンドボックスとして使えます。

frontmatter — Skillの"看板"

SKILL.md の先頭には frontmatter を書きます。ここはSkillの"看板"で、Claudeはまずここだけを読んで「このSkillがいま呼ばれるべきか」を判定します。

---
name: hello
description: ユーザーに短く挨拶して雰囲気を和ませる。雑談の流れで「挨拶して」「hello」「自己紹介して」「元気?」などと言われた時に発火する。
---

# hello スキル

あなたはこのファイルを読んだら...(SKILL.md本体)

必須フィールドは2つだけ

name
Skillの識別子。半角英数+ハイフン。/skill-name の名前にもなる。フォルダ名と一致させるのが慣習
description
発火判定の鍵。Claudeはこの文字列を常時ロードして「ユーザー入力に合うか」を判定する。具体的であればあるほど正しく発火する
description が発火判定に使われる流れ
graph LR
  U["ユーザー入力
(例: 挨拶して)"] --> C{"Claudeが
判定"}
  C --> D["Skill一覧の
description を照合"]
  D --> M{"マッチ?"}
  M -- "Yes" --> L["SKILL.md本体を
ロード&実行"]
  M -- "No" --> N["通常の応答"]
description がスカスカだとマッチせず、せっかく書いたSkillが永遠に眠ることになる。

description の良い例 vs 悪い例

Bad

抽象的で発火しない

  • description: メールのスキル
  • description: 議事録
  • description: コード関係

→ 何をするのか・いつ呼ぶかが分からないのでClaudeが判定できない

Good

動詞・対象・トリガー語が明確

  • 「社外向けの日本語ビジネスメールを下書きする。"メール書いて" "xxさん宛のメール"と言われた時に発火」
  • 「会議の生ログから Decisions / Actions / Questions に整形。"議事録整形" "会議メモまとめて" で発火」

→ 目的・出力・発火ワードの3点セットが揃っている

鉄則

description は「何を・いつ・なぜ」を1-2文で書く。本チュートリアル自体を生成している make-tutorial の description が手本:「MKのClaude Codeチュートリアルと同じデザイン・構造の学習サイトを量産する。テーマ・URL・合計時間・章時間・内容目標を受け取り、妥当性確認 → デザイン決定 → HTML/CSS/JS生成 → GitHub Pagesへ自動デプロイまで一気通貫で実行する。」——何を(量産)・入力(5項目)・流れ(5フェーズ) がこの1文に詰まっている。

もっと詳しく: description はどれくらい長く書いていいか

短すぎると発火しない、長すぎると他Skillの判定を圧迫する——というトレードオフ。目安は 1-3文、100-300文字。発火ワードを入れたい時は「"キーワード1" "キーワード2" などと言われた時」のように具体例を引用符付きで並べるのが効きます。「何でも屋」Skillは作らない——単一責任(Ch03)に則って description を尖らせるほど、Claudeは正確に選んでくれます。

任意フィールド(将来拡張)

2026年1月時点、必須は name と description の2つだけ。バージョン表記や作者情報を付けたい時は、以下のような任意フィールドを追加しても無視されるだけで害はありません(将来的に意味を持つ可能性あり)。

---
name: hello
description: 雑談の流れで挨拶する最小Skill
version: 0.1.0
author: MK
tags: [greeting, minimal, demo]
---

SKILL.md本体 — AIへの指示書として書く

frontmatterの下は普通のMarkdownです。ただし読み手は人間ではなく Claude自身——「この指示書を受け取った将来の自分」向けのAI向け仕様書として書きます。

SKILL.md本体の典型的な章立て
graph TD
  A["1. 概要
(1-2行で何をするか)"] --> B["2. いつ使うか
(トリガー条件の明示)"]
  B --> C["3. 入力の受け取り方
(不足は対話で聞く)"]
  C --> D["4. 手順
(番号付きで具体的に)"]
  D --> E["5. 出力形式
(例を1つ付ける)"]
  E --> F["6. 失敗時の対処
(エラーハンドリング)"]
この6節を意識すると、誰が読んでも同じ結果が出る"手順書"になる。

"hello" Skill の SKILL.md 全文

これから作る ~/.claude/skills/hello/SKILL.md の完成形は、こんな感じです。

---
name: hello
description: ユーザーに短く挨拶する最小Skill。雑談の流れで「挨拶して」「hello」「自己紹介して」「元気?」などと言われた時に発火する。/hello でも明示呼び出し可能。
---

# hello スキル

あなたは軽快な挨拶担当です。このSkillが呼ばれたら、以下の手順で短く挨拶してください。

## いつ使うか
- ユーザーが「挨拶して」「hello」「こんにちは」「自己紹介」などと言った時
- ユーザーが `/hello` を実行した時
- 会話の雰囲気を和ませたい時

## 手順
1. 現在時刻に応じて「おはよう/こんにちは/こんばんは」を選ぶ(分からなければ「こんにちは」)
2. 自分の役割を1行で名乗る(例: "Claude Code で作業のお手伝いをしています")
3. ユーザーの名前が既に分かっていれば呼びかけに使う
4. 「今日は何をしますか?」で締める

## 出力形式
3〜5行の自然な日本語。絵文字は使わない。装飾なしの素朴な挨拶。

## 例
> こんにちは、MKさん。Claude Code で作業のお手伝いをしています。
> 今日もよろしくお願いします。
> 今日は何をしますか?

## 失敗時の対処
- ユーザー名が不明 → 呼びかけを省略して汎用の挨拶にする
- 時刻が取れない → 無難に「こんにちは」で始める
  • 「いつ使うか」を先頭で明示 — frontmatter の description を補強する位置付け。ユーザー目線のトリガーを列挙する
  • 手順は番号付きリスト — 「まず〜それから〜」の散文ではなく、Claudeが1項目ずつ確実に実行できるよう分解する
  • 出力形式の例を必ず1つ付ける — 「3〜5行」と書くだけでなく具体例を添えると、出力のブレが激減する
  • 失敗時の分岐を書く — 情報不足・エラー・想定外の入力を、手順と同じ精度で明文化する
もっと詳しく: "プロンプトを書く" との違い

チャットに1回投げるプロンプトと違い、SKILL.md は 毎回同じ入力で同じ結果 が出る必要があります。そのため:(1)気分で書かない、(2)抽象語を避ける、(3)例示を載せる、(4)分岐を明示する——この4点が必須になります。良いSKILL.mdを書く作業は、実は 業務手順の標準化 そのもの。人に引き継げないノウハウが、人にもAIにも渡せる手順書になります。

呼び出し方 — 明示 vs 自動発火

Skill は 2つの経路 で起動されます。これがSlashコマンドとの決定的な違いで、「呼ばれなくても勝手に出てくる」挙動を理解することが、Skill設計の腕を分けます。

2つの起動経路
同じSkillが2つの入口を持つ。片方だけを想定して書くと、もう片方で事故る。

1. 明示呼び出し: /skill-name

最も単純な経路。チャット入力で /hello と打てば、description の照合を飛ばしていきなり本体がロードされて実行されます。Slashコマンドと見た目が同じですが、中身はSKILL.mdなのでSlashよりリッチに書けます。

# 会話の例
You: /hello

Claude: こんにちは、MKさん。Claude Code で作業のお手伝いをしています。
        今日もよろしくお願いします。
        今日は何をしますか?

2. 自動発火: 会話の流れから

Claudeはユーザー発話のたびに、登録されているSkillの description リストを見て「この入力に合うSkillがあるか」を判定します。マッチすれば ユーザーがSkillの存在を知らなくても、本体がロードされて実行されます。

# 会話の例(/hello と打たなくても発火する)
You: ちょっと挨拶してくれる?

Claude: (hello Skillのdescriptionにマッチ → SKILL.md本体を実行)
        こんにちは、MKさん。Claude Code で作業のお手伝いをしています。
        ...
自動発火の注意

自動発火は便利な反面、description の書き方が悪いと想定外のタイミングで発火したり、逆に呼ぶべき場面で静かに無視されたりします。Ch03でこの発火精度を上げる「単一責任 × 具体的 description」の設計原則を扱います。

今すぐ試す(体感)

この章の終盤でhello Skill を作り終えたら、以下を2回ずつ試してください。

  1. チャットに /hello と打つ → 明示呼び出しで発火
  2. チャットに「ちょっと挨拶して」と打つ → 自動発火でSKILL.mdが読まれる
  3. チャットに「天気はどう?」と打つ → 関係ないので発火しない(=description が正しく尖っている証拠)

3つ目が「関係なくても発火してしまう」なら description が広すぎるサイン。Ch03で矯正します。

ハンズオン — 最小"hello" Skill を3分で作る

これまでの知識を統合して、実際に動くSkillを1つ作ります。所要3分。ターミナルと Claude Code があれば他は何も要りません。

1
フォルダ作成
~/.claude/skills/hello/
2
SKILL.md作成
frontmatter + 本体
3
Claude再起動
スキルを再スキャン
4
動作確認
/hello と自動発火両方

Step 1: フォルダを作る

mkdir -p ~/.claude/skills/hello

Step 2: SKILL.md を書く

以下の内容で ~/.claude/skills/hello/SKILL.md を作成します。エディタでもいいですし、ヒアドキュメントでも。

cat > ~/.claude/skills/hello/SKILL.md <<'EOF'
---
name: hello
description: ユーザーに短く挨拶する最小Skill。雑談の流れで「挨拶して」「hello」「自己紹介して」「元気?」などと言われた時に発火する。/hello でも明示呼び出し可能。
---

# hello スキル

あなたは軽快な挨拶担当です。このSkillが呼ばれたら、以下の手順で短く挨拶してください。

## いつ使うか
- ユーザーが「挨拶して」「hello」「こんにちは」「自己紹介」などと言った時
- ユーザーが /hello を実行した時

## 手順
1. 現在時刻に応じて「おはよう/こんにちは/こんばんは」を選ぶ(分からなければ「こんにちは」)
2. 自分の役割を1行で名乗る
3. ユーザー名が既に分かっていれば呼びかけに使う
4. 「今日は何をしますか?」で締める

## 出力形式
3〜5行の自然な日本語。絵文字は使わない。

## 失敗時の対処
- ユーザー名が不明 → 呼びかけを省略
- 時刻が取れない → 「こんにちは」で始める
EOF

Step 3: Claude Code を再起動

Claude Code を一度終了して、もう一度起動してください。起動時に ~/.claude/skills/ を再スキャンして、今作った hello が認識されます。

Step 4: 動作確認

新しい会話を開いて、以下を順に試してください。

Test 1
明示呼び出し
チャットに /hello と入力 → Skillが起動して3〜5行の挨拶が返る
Test 2
自動発火
チャットに「ちょっと挨拶して」と入力 → description にマッチ → 同じ挨拶が返る
Test 3
発火しない確認
「今日の天気は?」と入力 → 関係ないSkillは起動せず普通の応答
動作確認の観察ポイント(3種類の入力×発火率)
明示・関連・無関連の3種で、Skillが 意図通りのタイミングだけで発火するかチェックする。
うまく発火しない時のチェック

(1) ファイルパスが正確か — ls ~/.claude/skills/hello/SKILL.md で存在確認。(2) frontmatter が有効か — 先頭の --- が1文字目にあり、YAMLとして valid か。(3) Claude を再起動したか — 起動後のスキャンが必要。(4) description が尖っているか — 抽象的すぎると判定が迷う。

Exercise 1
入門

hello Skill を自分の環境で動かす

本章の Step 1〜4 を実際に実行して、以下を確認してください。

  1. ~/.claude/skills/hello/SKILL.md を作成
  2. Claude Code を再起動
  3. /hello で明示呼び出しが成功するか
  4. 「挨拶して」で自動発火が成功するか
  5. 「天気は?」で発火しないことを確認
模範解答を見る

全ステップ成功時のチェックリスト:

$ ls ~/.claude/skills/hello/SKILL.md
/Users/<name>/.claude/skills/hello/SKILL.md

$ head -3 ~/.claude/skills/hello/SKILL.md
---
name: hello
description: ユーザーに短く挨拶する最小Skill。...

Test 1 (明示): /hello → 3〜5行の挨拶が返る。
Test 2 (自動): 「挨拶して」→ 同じ挨拶パターンが返る(description 経由)。
Test 3 (無関連): 「天気は?」→ 通常応答。helloは起動しない。

もしTest 2が失敗する場合は description に「挨拶」「hello」などの発火ワードが入っているか確認。Test 3で誤発火する場合は description が広すぎるので、Ch03の原則で絞ります。

Exercise 2
応用

"goodbye" Skill を自力で作る(+スコープ実験)

hello のミラーで、別れの挨拶をする goodbye Skill を自力で作成してください。ただし以下の条件付きです。

  1. frontmatter の description を自分で考える(発火ワードを具体的に3つ以上含める)
  2. 本体には「いつ使うか / 手順 / 出力形式 / 失敗時の対処」の4節を必ず書く
  3. プロジェクトスコープに配置する(任意のテスト用リポジトリの <repo>/.claude/skills/goodbye/)
  4. そのリポジトリの中で自動発火するか確認
  5. 別の(関係ない)リポジトリに移動した時、発火しないことを確認
模範解答を見る

SKILL.md の例:

---
name: goodbye
description: 会話の締めくくりで短くお礼と別れの挨拶をする。「終わり」「お疲れ様」「また後で」「bye」「ありがとう」などと言われた時、または /goodbye で発火する。
---

# goodbye スキル

あなたは会話を気持ちよく締める役です。

## いつ使うか
- ユーザーが「終わり」「お疲れ様」「bye」「ありがとう」などと言った時
- /goodbye が実行された時

## 手順
1. その日の成果を1行で要約(会話履歴から)
2. 感謝の言葉を1行
3. 次回への引き継ぎがあれば1行
4. 「お疲れ様でした」で締める

## 出力形式
3〜4行の丁寧な日本語。

## 失敗時の対処
- 成果が曖昧 → 要約を省略して感謝のみ

配置と検証:

mkdir -p <repo>/.claude/skills/goodbye
# 上記SKILL.mdを保存
cd <repo>  # このリポジトリで Claude Code を起動

# Test A: 「お疲れ様」と打つ → 発火する
# Test B: cd ~/tmp (別の場所) で起動 → 発火しない

学び: プロジェクトスコープは「このリポジトリ専用の手順書」を閉じ込める用途に最適。横断的に使いたくなったら ~/.claude/skills/ に移して昇格させればいい。
