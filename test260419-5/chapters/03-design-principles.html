良いSkillの設計原則

単一責任・明示トリガー・テンプレート分離 — 6つの原則でSkillの質を一段上げる

所要時間: 30分 Phase: Basics 難易度: 中級
この章のゴール

Ch02で「とりあえず動くSkill」を作れるようになった次の階段として、「長く使われるSkill」に共通する6つの設計原則を内面化します。単一責任・明示トリガー(description)・テンプレ分離・self-contained・段階的ユーザ対話・失敗時対処明記——どれも抽象論ではなく、Before/After の具体コードで diff レベルの違い を示します。読み終わる頃には、自分の既存Skillを点検して「どこを直せば化けるか」が分かるようになります。

バージョン注記

本章の記述は 2026年1月時点 の Claude Code (Skills 機能) の挙動に基づきます。frontmatter のフィールド仕様や発火判定アルゴリズムは進化するため、原則が通らない場面が出てきたら 公式ドキュメントを必ず確認してください。本章の手本としては、この学習サイト自体を生成している ~/.claude/skills/make-tutorial/SKILL.md を頻繁に引用します。

なぜ「設計原則」が要るのか

Skillは最小構成なら SKILL.md 一枚で動きます。だからこそ 「動くSkill」と「使えるSkill」の差は書き方だけで決まる。放っておくと以下の3症状が必ず出ます。

  • 発火しない — 呼んだはずなのに Claude が別の動き。descriptionが曖昧で判定が外れている。
  • 肥大化して壊れる — 1つのSkillに機能を詰め込んだ結果、どの分岐も中途半端になる。
  • 属人化する — 作った本人しか使いこなせない。チーム共有しても誰も再利用しない。

これらはすべて 設計原則 の不在が原因です。逆に言えば、6つの原則さえ守ればSkillは壊れにくく、発火しやすく、他人にも渡せる資産になる。

6つの原則の関係図
6つの原則が「良いSkill」という中心を多角的に支える。どれか1つ欠けるとバランスが崩れる。

原則1・2 — 単一責任と明示トリガー

原則1. 単一責任(Single Responsibility)

1つのSkillは1つのことだけを徹底的にやる。複数の責任を持たせると、発火判定が曖昧になり、プロンプトも肥大化し、失敗モードが増えます。Unix哲学の "Do one thing and do it well" をそのまま持ち込むのが正解。

Bad

責任を詰め込みすぎ

  • office-helper: メール・議事録・PR作成・日報をまとめて担当
  • descriptionに「業務全般を補助」と書くしかなくなる
  • 発火判定が曖昧で、どの文脈でも呼ばれない or 呼ばれすぎる
Good

1つの責任

  • email-draft: 日本語ビジネスメール下書きのみ
  • meeting-notes: 生ログ→整形議事録のみ
  • pr-summary: git diff→PR本文のみ
  • それぞれdescriptionが具体的に書ける → 発火が確実

原則2. 明示トリガー(description)

Skill の description は、発火判定アルゴリズムに「このSkillはいつ呼ぶべきか」を伝える唯一の情報源です。「何ができるか」ではなく「どんな時に呼ばれるべきか」を主語にして書く。

description の書き方 Before/After
graph LR
  A["Before: 「メール作成を支援する」"] --> B{"発火判定"}
  C["After: 「ユーザが『メール書いて』『xxさんに送るメール』と言ったとき、宛先/目的/トーンを受け取り日本語ビジネスメール下書きを生成する」"] --> B
  B -- "曖昧" --> X["発火せず"]
  B -- "具体" --> Y["確実に発火"]
description の粒度が発火確率を直接決める。動詞＋トリガーフレーズ＋入出力が揃うと強い。

良いdescriptionの型は「いつ(トリガーフレーズ) + 入力 + 出力」の3点セット。本チュートリアルを作った make-tutorial Skillの description を見てみましょう。

---
name: make-tutorial
description: MKのClaude Codeチュートリアルと同じデザイン・構造の学習サイトを量産する。
  テーマ・URL・合計時間・章時間・内容目標を受け取り、妥当性確認 → デザイン決定
  → HTML/CSS/JS生成 → GitHub Pagesへ自動デプロイまで一気通貫で実行する。
---

「学習サイトを量産する」(動詞)「テーマ/URL/時間/目標を受け取り」(入力)「HTML生成→デプロイ」(出力)と3点が揃っているので、「学習サイト作りたい」「チュートリアル量産」といった発話に確実にヒットします。

Tips

descriptionに「ユーザが〇〇と言った時」のトリガーフレーズ例を2〜3個混ぜ込むと精度が跳ね上がります。make-tutorial は SKILL.md 本文の「いつ使うか」セクションにもフレーズを書いて補強している——これが実装例そのものとして学べる手本。

もっと詳しく: descriptionのNGパターン3つ

(1) 抽象すぎる: 「業務を支援する」「便利なツール」→ 発火判定が効かない。 (2) 機能リスト羅列: 「メールも議事録もPRも書ける」→ 単一責任原則違反のサイン。分割せよ。 (3) 長すぎる: 500字超えるとClaudeが要点を見失う。目安は100-250字、トリガー+入力+出力+要注意点でまとめる。

原則3・4 — テンプレ分離と self-contained

原則3. テンプレート分離

再利用可能な部品は SKILL.md に埋め込まず、templates/配下に独立ファイルとして置く。SKILL.md は「判断ロジックと手順」、templates は「部品」として責任を分離します。

テンプレ分離 Before/After のディレクトリ構造
SKILL.md 本体は薄く、可変部分(文面パターン)は templates/ に追い出す。これだけで保守性が段違い。

原則4. self-contained(自己完結)

Skillは外部依存を最小化し、発火から完了まで自力で完結できる形に書く。外部ネットワーク・DB接続・他Skillへの暗黙依存は、壊れる可能性を増やすし他環境で動かなくなる原因になります。

依存の種類Bad例Good例
外部API起動ごとに社内APIを叩いてテンプレ取得templates/ に固定コピーを置く、更新は手動Pull
環境変数未設定時に暗黙エラー必須env一覧をSKILL.md冒頭に明記、不足時は対話で聞く
他Skill呼び出しSkill Aが暗黙にSkill Bを期待必要な処理は自Skill内で完結。連携はSKILL.mdに明記
ツールgh/jq等の存在を無言で仮定必要ツール一覧+未導入時の対処をSKILL.mdに書く
make-tutorial の実例

make-tutorial Skill は ~/.claude/skills/make-tutorial/templates/ を「正本」として宣言し、全ての部品(chapter-shell.html, index-shell.html, style.css 等)をそこに同梱。外部APIは叩かず、Firebase も設定済みの共通プロジェクトを使うだけ——完全にself-contained。これが手本である所以。

原則5・6 — 段階的対話と失敗時対処

原則5. 段階的ユーザ対話(Progressive Elicitation)

必要な情報が揃っていない時、全部まとめて聞かず1問ずつ聞く。一度に10項目の入力フォームを突きつけるSkillは使われなくなります。

入力収集の Bad/Good パターン
graph TD
  Start(["Skill 発火"]) --> Q{"入力の揃い具合を
Skill が自己判定"}
  Q -- "全部揃ってる" --> Run["即実行"]
  Q -- "不足あり" --> Choice{"聞き方"}
  Choice -- "Bad: 全部一気に" --> Wall["ユーザ離脱
(認知負荷爆発)"]
  Choice -- "Good: 1問ずつ" --> Ask["1番重要な不足から
対話で埋める"]
  Ask --> Run
make-tutorial の Phase 3 は「3-1テーマ色 → 3-2ブランド → 3-3Ch00要否 ...」と1問ずつ対話する形式で、まさにこのパターンの模範例。

SKILL.md 本文には必ず「不足分は1問ずつ聞く」と明記し、デフォルト値やヒューリスティックがある項目はそれも記述します。make-tutorial は「到達目標と時間のバランス」を判定ヒューリスティック(上級=20h以上、中級=4-10h等)で評価し、過少/過大を検知したら対話で調整を提案する——これも段階的対話の一形態です。

原則6. 失敗時の対処明記

Skillが動く「正常系」だけ書くのは半分しかカバーしていないのと同じ。「何が起きたらどう対処するか」を SKILL.md の末尾に専用セクションとして書きます。

Bad

失敗時の記述なし

  • Claudeが勝手に推測して変な復旧を試みる
  • ユーザに「失敗しました」とだけ伝わる
  • 次に何をすべきか分からない
Good

明示的な失敗対処

  • SKILL.md に「エラー時の対処」セクション
  • 典型エラー3-5パターンと復旧手順をセットで
  • ユーザに具体的アクションを提示できる

make-tutorial はこのパターンを徹底していて、例えば

## エラー時の対処

- `~/Desktop/claude/26tutorial/{SLUG}` が既存 → 上書き確認をユーザに取る
- `gh` 未認証 → `gh auth login` を案内
- `git push` が rejected → まず `git pull --rebase` を試す

という具合に、典型3つの失敗モードと対応アクションをセットで宣言しています。Claudeはこの記述を根拠に、ユーザに具体的な次のアクションを提示できる——ユーザ体験の差は大きい。

もっと詳しく: 失敗分類は「予測可能/不可」で分ける

予測可能な失敗(環境不足・入力不備・権限不足・衝突)はSKILL.mdに列挙して対処を書く。一方 予測不可な失敗(ネットワーク瞬断・APIの謎エラー)は「Claudeに状況をそのまま報告させ、ユーザの判断を仰ぐ」と書くだけで十分。全部を網羅しようとしない——運用で見つかった失敗を随時SKILL.mdに追記していく方が健全です。

原則の相対的な重要度と適用順

6つの原則は全部大事だが、Skillの失敗事例を集計すると効くポイントの重さが違う。現場で「まずここを直せ」という優先順位をレーダーで可視化します。

6原則の重要度レーダー(典型的な新規Skillの失敗寄与度)
経験的に「description不足」と「単一責任違反」がSkill失敗の2大要因。ここから先に直すとROIが高い。

さらに、Skill開発を時系列の作業として見ると、原則は適用タイミングが違います。

1
設計時
原則1: 単一責任を決める
2
命名時
原則2: descriptionで発火条件
3
実装時
原則3&4: テンプレ分離/self-contained
4
対話設計
原則5: 段階的ユーザ対話
5
仕上げ
原則6: 失敗時対処を書き切る
注意

「6番目の失敗時対処は後回しにしがち」だが、初回失敗時の体験こそがSkillの命運を決めます。書き終えた直後の自分なら想像できる失敗を最低3つは列挙しましょう。

ハンズオン: 既存Skillを6原則でリファクタ

ここまでの原則を使って、Ch02で作った(または既存の)Skillを点検リストで自己監査します。点検→赤信号があれば該当原則の節を読み直し→書き換え、の順で回します。

6原則チェックリスト(自分のSkillに✓)
6つ全部に✓が付けば一級品。未チェックがあれば該当セクションを読み直して書き換える。
今すぐ試す(10分)

Ch02で作った hello Skill(または手元の既存Skill)を開き、上の6項目で自己評価してください。未チェック項目があれば、以下の手順で埋める。

  1. 該当する原則の節を再読
  2. 自分のSKILL.mdの該当箇所を Before として写経
  3. After に書き換えて保存
  4. Claude Code を再起動して発火確認

「原則2(description)」の改善だけで発火率が体感で変わることが多いので、そこから始めると達成感があります。

Exercise 1
入門

Bad descriptionを Good に書き換える

以下の Bad description を、6原則(特に原則2)に従って Good に書き換えてください。「いつ + 入力 + 出力」の3点セットを意識します。

  1. Bad: description: コミットメッセージの支援
  2. 書き換え案を自分で考えてから、模範解答を見る
  3. 自分のPCで実際に ~/.claude/skills/commit-helper/SKILL.md を作り、Before/Afterで発火の違いを確認
模範解答を見る

Good description 例:

---
name: commit-helper
description: ユーザが「コミットメッセージ書いて」「コミットしたい」と言ったとき、
  ステージ済み変更(git diff --staged)の内容から Conventional Commits 形式の
  メッセージを1案生成する。type (feat/fix/docs/refactor/test/chore) と scope、
  subject を 50 字以内で提案し、ユーザ承認後にcommitを実行する。
---

改善ポイント: (1) トリガーフレーズ「コミットメッセージ書いて」を明示、(2) 入力「git diff --staged」を指定、(3) 出力フォーマット「Conventional Commits 50字以内」を固定、(4) 対話フロー「承認後にcommit」で原則5も内包。これだけで発火精度も生成品質も劇的に上がります。

Exercise 2
応用

自分の既存Skillを6原則で総監査

手元に既存Skillがあれば1つ選び、無ければ Ch02 で作ったhello Skillをメール下書きSkillに拡張する想定で、以下を実行してください。

  1. Section 6のチェックリスト6項目を、現状のSKILL.mdに対して実際に✓/✗で評価
  2. ✗の項目について「現状」「問題」「改善案」を各1行でメモ
  3. 改善案を反映した新SKILL.mdを書き、templates/配下も整備する
  4. 実際に ~/.claude/skills/ に配置して発火テスト
  5. 改善前後でプロンプト1回分のやりとり長さ・成果物の品質を比較してメモ
模範解答を見る

例: hello Skillをメール下書きに拡張する時の監査・改善例

【監査結果】
  1. 単一責任: ✓ (メール下書きのみに責任限定)
  2. description: ✗ → 「メール作成支援」と曖昧
  3. テンプレ分離: ✗ → 文面例がSKILL.md本体に埋め込み
  4. self-contained: ✓ (外部API不使用)
  5. 段階的対話: ✗ → 一度に全項目要求
  6. 失敗時対処: ✗ → セクション自体がない

【改善後SKILL.md骨格】
---
name: email-draft
description: ユーザが「メール書いて」「xxさんに送るメール作って」と言ったとき、
  宛先/目的/トーン/伝えたいポイント3つを受け取り、日本語ビジネスメール下書きを
  生成する。不足項目は1問ずつ対話で埋める。
---
## いつ使うか
- 「メール書いて」「〇〇さんに送るメール」等の発話時

## 入力収集(1問ずつ)
1. 宛先 (社内/社外/役職)
2. 目的 (依頼/報告/謝罪/お礼)
3. トーン (丁寧/カジュアル/緊急)
4. ポイント 3点

## 生成手順
- 目的に応じて templates/{request,report,apology,thanks}.md を選択
- 宛先とトーンで敬語レベルを調整
- ポイントを本文に織り込み

## エラー時の対処
- 目的が不明確 → 典型4種を提示して選ばせる
- templates/ファイル未導入 → 初回セットアップを案内
- 生成結果が長すぎ → 3段落以内に収める旨を再要求

【ディレクトリ】
~/.claude/skills/email-draft/
├── SKILL.md
└── templates/
    ├── request.md
    ├── report.md
    ├── apology.md
    └── thanks.md

観察ポイント: 6原則のうち4つが✗→✓に変わる。これだけで「使い捨てのhello」から「実務で毎日呼ぶSkill」に格上がる感覚が掴める。Ch05で実際のメール下書きSkillを完成形として扱うので、その予習として完璧。
