Skills と Subagents/Hooks の組み合わせ

3つを束ねて強力なワークフローを作る — 並列処理・自動発火・定期実行の統合設計

所要時間: 30分 Phase: Practice 難易度: 中級
この章のゴール

ここまで単独で覚えてきた Skill を、Subagent / Hook / Schedule&Loop の3つと組み合わせて「本当に手が離せる自動化」に昇華させます。Skillは「何を・どう書くか」の塊、Subagentは「誰に並列で任せるか」、Hookは「いつ勝手に発火するか」、Schedule&Loopは「どの周期で走らせ続けるか」。4つの軸を1つのワークフローに束ねるコツを、毎朝Slackサマリを送るSkill のハンズオンで体得します。

バージョン注記

本章の記述は 2026年1月時点 の Claude Code (Agent SDK / CLI / Skills / Hooks) 挙動を基準にしています。settings.json のhookイベント名や /loop / /schedule の引数体系は進化するので、コマンドが通らない時は 公式ドキュメント を必ず確認してください。

統合アーキテクチャ — 4本の軸と1つの中心

まず Skill が中心にあり、その周りに Subagent・Hook・Schedule&Loop が接続される、というのが全体像です。Skillは「呼ばれたら動く振る舞いのテンプレ」、他の3つは「どこから呼ぶか」の入口の違いと捉えると頭が整理されます。

Skill を中心にした4本の軸(統合アーキテクチャ)
Skillは「呼ばれて動く」側。User / Hook / Schedule&Loop の3つが入口、Subagent は Skill が 自分から呼ぶ 並列の手足。これを押さえれば組み合わせは無限に設計できる。

覚え方は 「Skillが頭脳、Subagentが手足、Hook/Loopが時計」。本章のハンズオンでは全部を1本のワークフローに束ねます。

組み合わせ例A：Skill から Subagent を並列呼び出し

Skillは単なる手順書なので、「この部分は並列で調査してきて」と書けば、Claudeは自動的に Agent tool で子を並列起動します。Skillに明記しておくのがポイント。

  • 調査系Skill — 3つ以上の独立した観点で同時に裏取りさせると、直列10分 → 並列2分に。
  • 比較系Skill — 候補A/B/Cを並列で触って評価させ、親はマージ表を作るだけ。
  • レビュー系Skill — セキュリティ/性能/可読性を並列で査読、親が要約だけ受け取る。

Skillの中で並列化を 明示的に指示する書き方 は次のようになります。このパターンを SKILL.md 本文に1箇所入れるだけで、Claudeは自発的に並列化してくれます。

---
name: tech-research
description: 新技術Xを導入すべきか、技術/コスト/学習/代替/リスクの5側面で裏取りしたい時に使う。「技術調査して」「導入検討して」で発火。
---

# tech-research スキル

## 手順

1. ユーザから対象技術名を受け取る(不足なら質問)
2. **並列調査フェーズ** — 次の5タスクを **1メッセージ内で Agent tool により並列起動する**:
   - A: 技術観点(API/成熟度/エコシステム) subagent_type=general-purpose
   - B: コスト観点(ライセンス/運用費/人的コスト)
   - C: 学習コスト観点(ドキュメント/日本語資料/チュートリアル)
   - D: 代替技術観点(最低3候補)
   - E: リスク観点(障害事例/サポート終了可能性)
   各子は **100語以内のサマリ+根拠URL3本** を返すこと
3. 統合フェーズ — 5つの要約を1つの比較表にマージ
4. 結論を 150 字以内で提示し、判断材料を整理
Tip: 並列が効く判定

「5観点の調査」のように 依存がない タスクなら必ず並列化してください。逆に「設計→実装→テスト」のように前段の結果が次段の入力になるタスクは、並列化すると破綻します。この見極めは本シリーズの姉妹 Subagents Mastery Ch04 で扱います。

もっと詳しく: Subagent の成果物はどう親に返すか?

子は要約しか返せない(Ch01の落とし穴2)ので、長文や生ログが必要な時は「成果物はファイルに書き、パスだけ返す」運用にします。Skill側でこう書いておけば、子はその通り動きます:

各子は成果物を /tmp/research_{A-E}.md に書き、
親には「ファイルパス+100語以内サマリ」だけを返すこと。
親はパスを読んで必要な部分だけ引用する。

これで親のコンテキストを汚さず、詳細は必要なときだけ読めます。

組み合わせ例B：Skill を Hook から自動発火

Hook は ~/.claude/settings.json に書いた「イベントで勝手に発動するシェルコマンド」です。ClaudeではなくClaude Code harness(ランタイム)が実行するため、「毎回必ず」発動する点が強力。コミット時にレビューSkillを自動発火させる、Stop時に日報Skillを自動発火させる、などの使い方ができます。

Memoryだけで頼る

抜けが出る

  • CLAUDE.mdに「コミット後にレビューして」
  • Claudeが忘れることがある
  • セッションを跨ぐと曖昧
  • 毎回必ずは保証されない
Hookで必ず発火

確実な自動化

  • settings.jsonに書く
  • harnessが毎回実行
  • セッションに依存しない
  • Skillと組み合わせて強力

設定の最小例はこうなります。「git commit を Bash ツールで実行した直後に /pr-prep Skill を叩く」パターン:

// ~/.claude/settings.json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "bash -c 'if echo \"$TOOL_INPUT\" | grep -q \"git commit\"; then claude --skill pr-prep --non-interactive; fi'"
          }
        ]
      }
    ]
  }
}
注意: Hookの設定は update-config skillで

MK自身の環境で実際に settings.json をいじる時は、Claude Codeの update-config skillに任せるのが安全です。「コミット後にpr-prepを自動発火させて」と伝えれば、正しいマッチャーと構文で settings.json を書き換えてくれます。手書きで壊してエディタが起動しなくなる事故を避けられます。

Hookで発火できる代表イベント(2026/1時点)

イベント発火タイミングSkill組み合わせ例

PostToolUse (Write)ファイル書き込み直後*.ts 書き込み → /type-check Skill

組み合わせ例C：Skill を /schedule や /loop で定期実行

/schedule は cron 式で絶対時刻トリガー、/loop は相対インターバルでのポーリング。両方とも「Skillを呼ぶだけ」の最小プロンプトを登録すれば、Skill本体の手順は全部SKILL.mdに書けるので、スケジューラ側はシンプルに保てます。

/schedule と /loop の使い分け
絶対時刻 = /schedule、相対間隔 = /loop。「毎朝」は scheduleの十八番、「5分ごと」は loopの十八番。
Skill呼び出しの1日タイムライン(統合イメージ)
朝のsummary、昼のreview、夕方のdeploy-check、夜のsession-log。Skill本体は各1つだが入口は4種類混在。

5本シリーズの統合設計 — MKの推奨構成

本家MKチュートリアルは Subagents / MCP / Hooks / Schedule&Loop / Skill Craft の5本立てです。5つをバラバラに覚えても強くなりません。Skillを真ん中に置いて他4つを入口や手足として並べる と、全体が一枚の設計図になります。

設計のコツ

「新しい業務を自動化したい」と思ったら、まずSkill(中心)を書き、次に入口(Hook/Schedule/Loop)を1つ選び、最後に手足(Subagent/MCP)を足す という順番で考えてください。いきなりHookから書くとロジックが分散して保守不能になります。

ハンズオン：毎朝Slackサマリを送るSkill

ここまでの全要素を束ねた統合ハンズオン。「毎朝9時に、GitHubとカレンダーとメールから今日のToDoをまとめ、Slackに投稿するSkill」 を作ります。Skill本体に並列Subagent呼び出しを書き、/loop(または /schedule)で定期起動します。

1

2

3観点をSubagent
3

MCPで送信
4

/loop or /schedule
5
運用
1週間回す

1. Skillのファイル

まず ~/.claude/skills/morning-summary/SKILL.md を作成。中身は次の通り:

---
name: morning-summary
description: 毎朝のToDoサマリをGitHub/カレンダー/メールから集約し、Slackの #daily チャンネルに投稿したい時に使う。「朝のまとめ」「デイリーサマリ」「今日のToDo」で発火。
---

# morning-summary スキル

## 手順

1. **並列収集フェーズ** — 次の3タスクを 1メッセージ内で Agent tool により並列起動する:
   - A: GitHubの今日締切Issue・自分にアサインされたPR一覧(subagent_type=general-purpose)
   - B: Googleカレンダーの今日の予定一覧(MCP google-calendar 経由)
   - C: 未読の重要メールの件名+差出人Top 5(MCP gmail 経由)
   各子は **Markdown箇条書きで15行以内** にまとめて返すこと。

2. **統合フェーズ** — 3つの要約を以下のテンプレートに流し込む:
   ```
   【おはようございます、{{DATE}} のまとめ】

   ◆ 今日のGitHub
   {{A_SUMMARY}}

   ◆ 今日の予定
   {{B_SUMMARY}}

   ◆ 注目メール Top 5
   {{C_SUMMARY}}

   今日も1日よろしくお願いします。
   ```

3. **Slack投稿** — MCP slack を使い、#daily チャンネルに上記テキストを投稿。
   失敗した場合は、同じテキストを /tmp/morning_summary_{{DATE}}.md に保存して終了。

## 発火条件

- `/morning-summary` が明示的に呼ばれた時
- ユーザから「朝のまとめ」「デイリーサマリ」「今日のToDo」と言われた時
- `/loop` や `/schedule` から自動呼び出しされた時

2. 並列化の確認

Skill本文の「並列起動する」の一文が重要。これがないとClaudeが順次実行してしまい、3倍遅くなります。「1メッセージ内で」 という語を必ず入れてください。

3. Slack接続(MCP)

Slack投稿は MCP slack サーバを使います。既に設定済みの前提ですが、未設定なら姉妹シリーズ「MCP」のSlack導入章を先に済ませてください。SKILL.md内では slack_post_message を呼ぶと意識するだけで、Claudeが自動で MCP ツールを選択します。

4. /loop から定期実行

Claude Code のセッションで次のように打ちます:

/loop 24h /morning-summary

これで24時間ごとに /morning-summary が自動実行されます。セッションを閉じずに放置するか、サーバ版で回すのが実運用。「毎朝9時」の絶対時刻にこだわるなら /schedule を使います:

/schedule "0 9 * * *" /morning-summary

5. 運用と改善

1週間運用したら、次の観点でSkillを見直します。2週目でグッとクオリティが上がります。

発火頻度
毎朝ちゃんと飛んでいるか / /loopがセッション切れで止まっていないか
要約の質
A/B/Cの各要約が15行以内に収まっているか / 冗長なら語数制限を厳しく
Slack投稿成功率
失敗時のフォールバック(/tmpへの保存)が機能しているか
コスト
Subagent3並列×毎日 = 月90回。トークン消費を月次で振り返る
1週間運用した時の改善ポイント分布(例)
要約冗長化が最多の改善対象。Skill文面に「必ずN行以内」と厳しく書くと大半が解消する。

組み合わせ時の落とし穴4選

3つを重ねる時ほど、単独では見えなかった罠が出ます。MKが実際にやらかしたやつも含め、代表的な4つを押さえてください。

落とし穴症状対策
Hook暴走ループSkill内の git commit がまた Hook を発火 → 無限ループHookで発火するSkillでは該当ツール使用を避ける / Hookに再入防止フラグ
並列数×定期 = コスト爆発5並列 × 毎時 × 24h × 30日 = 3600回/月並列上限を3にする + 頻度は日次か時間単位で
/loopセッション切れ夜にPC閉じたら朝のsummaryが動いてなかった絶対時刻なら /schedule(サーバ側) / ローカル運用ならPC常時起動
Skillとmemoryの混線CLAUDE.mdに「毎朝まとめて」と書いて、Skillと競合自動化はSkill+Hook/Schedule一本に統一。memoryは「知識」のみ
MKの失敗談

一度、PostToolUse(Bash) Hookで git commit 後にレビューSkillを起動した際、レビューSkillが内部で git log を Bash で叩いたせいで再度Hookが発火して、15回ループしてからCtrl+Cで止めました。ループ防止の環境変数(CLAUDE_HOOK_DEPTH 等、2026/1時点の目安)をHook側で見てスキップする設計が必須です。

Exercise 1
入門

既存Skillに並列Subagent指示を足す

Ch06で作った meeting-notes Skillを開き、「Decisions/Actions/Questions の3観点を並列で抽出する」ようにSKILL.mdを1箇所だけ書き換えてください。

  1. ~/.claude/skills/meeting-notes/SKILL.md を開く
  2. 本文に「次の3タスクを1メッセージ内でAgent toolにより並列起動する」の一文を追記
  3. 3観点をそれぞれ Subagent の prompt として明示
  4. 動作確認: 実際に長めの会議ログを渡して、Claudeが並列起動するかを見る
模範解答を見る

SKILL.md本文の「手順」直下にこう追記します:

## 手順

1. 生の会議ログをユーザから受け取る(なければ要求)

2. **並列抽出フェーズ** — 次の3タスクを 1メッセージ内で Agent tool により並列起動する:
   - A: Decisions(意思決定)を箇条書きで抽出 subagent_type=general-purpose
   - B: Actions(ToDo/担当者/期限)を箇条書きで抽出
   - C: Questions(保留/要確認)を箇条書きで抽出
   各子は 10行以内で Markdown 箇条書きを返すこと

3. 統合: Decisions / Actions / Questions の3セクションを持つ議事録として整形
4. ユーザに提示

観察ポイント: 並列化前は Claude が1つずつ抽出して10-15秒かかっていたのが、並列後は5秒前後で返ります。手応えとして3倍速くなるのが体感できます。

Exercise 2
応用

自分の業務から「Skill + Hook + Schedule」の統合ワークフローを1つ設計

本章の3つの組み合わせ例を参考に、自分の業務から Skill(中心)+ Hook or Schedule(入口)+ Subagent or MCP(手足) を全部使う統合ワークフローを1つ設計して、SKILL.md + settings.json の断片を書いてください。抽象で終わらせず、具体Skill名+具体イベントまで踏み込むこと。

  1. 対象業務を1つ選ぶ(デイリー振り返り / PRレビュー / 週次棚卸し 等)
  2. Skill名と frontmatter(name/description)を決める
  3. 入口を1つ決める(PostToolUse Hook / /schedule / /loop から選ぶ)
  4. 手足としてSubagent並列 or MCP外部接続を最低1つ使う
  5. 落とし穴(暴走ループ / コスト / セッション切れ)への対策を1行書く
模範解答を見る

例: MK自身の「週次チュートリアル棚卸し」ワークフロー

【Skill】 weekly-tutorial-review
---
name: weekly-tutorial-review
description: 自分が公開中の全チュートリアルを棚卸ししたい時に使う。各サイトの進捗・壊れリンク・視覚化不足箇所を並列調査して週次レポート化。「週次棚卸し」「チュートリアル健康診断」で発火。
---

## 手順
1. ~/Desktop/claude/26tutorial/*/ 配下のディレクトリを一覧化
2. 並列調査フェーズ — 各サイトごとにSubagentを並列起動(最大5並列):
   - 壊れリンクの検出 / 視覚化要素カウント / 最終更新日
   各子は 50語以内+結果ファイルパスで返却
3. 統合: 比較表にまとめ /tmp/weekly_tutorial_{{DATE}}.md に保存
4. MCP slack で #tutorial-ops チャンネルに投稿

【入口】 /schedule(毎週月曜9時)
  /schedule "0 9 * * 1" /weekly-tutorial-review

【手足】 Subagent 5並列 + MCP slack

【落とし穴対策】
  - 並列数は最大5(サイトが増えたらバッチング)
  - セッション切れ対策で /schedule を選択(/loopは不採用)
  - 再入防止: この Skill 自体は Bash で git commit しない

ポイント: 入口は1つに絞る(Hookと /schedule を両方つけると二重発火)。手足は「並列で効くか」で判定する。コスト対策は数字で書く(並列数や頻度)。この粒度で書けると最終章Ch08の「3つSkill化」演習がスムーズになります。
