実践：メール下書きSkill

テンプレと文脈から日本語ビジネスメールを量産する、自分専用アシスタントを作る

所要時間: 40分 Phase: Practice 難易度: 中級
この章のゴール

「xxさんに、MTG日程調整のメール書いて」と伝えるだけで、あなたの普段の書き方・トーン・定型句を反映したビジネスメール下書きが出てくる——そんな自分専用Skillを完成させます。Ch01〜04で学んだ frontmatter / 単一責任 / templates分離 がすべて具体コードで統合される章です。完成後は他章のSkill(議事録・組み合わせ)のひな型としてもそのまま再利用できます。

バージョン注記

本章の記述は 2026年1月時点 の Claude Code (Agent SDK/CLI) における Skill 仕様に基づきます。~/.claude/skills/{name}/SKILL.md ディレクトリ構造および frontmatter の name / description フィールドは 公式ドキュメント で最新仕様を必ず確認してください。挙動が変わった場合はまず description の書き方を見直すのが復旧の近道です。

シナリオ設計 — 何を、誰に、どう渡すか

Skillを書き始める前に、入力と出力を紙に書くくらいのつもりで明確化します。ここを曖昧にしたまま SKILL.md に入ると、descriptionが肥大化して 発火しない Skill が出来上がります。

メール下書きSkill の 4入力 → 1出力モデル
入力は最低1つ(目的)あれば発火、残りは対話で埋める。これが「段階的ユーザ対話」の原則(Ch03)の具体化。

4つの入力 の中でも、宛先 と 目的 は必須、トーン はデフォルト「標準」で省略可、ポイント は本文生成の核です。

曖昧なシナリオ(失敗)

「いい感じのメールを書く」

  • 入力が未定義 — 何を渡せばいい?
  • 出力が未定義 — 件名は付く? 署名は?
  • 発火条件が広すぎる — 雑談にも発火してうざい
明確なシナリオ(成功)

「4入力→ビジネスメール1通」

  • 宛先/目的/トーン/ポイント を明示
  • 件名+本文+署名の完成形を返す
  • 「メール書いて」「xxさん宛に送る」で発火

SKILL.md 全文 — コピペで動く完全版

これが email-draft Skill の本体 です。~/.claude/skills/email-draft/SKILL.md として保存してください。frontmatter の description は「発火判定の鍵」なので、本Skillの唯一無二性(=日本語ビジネスメールの骨格生成)が一読で分かるように書いています。

---
name: email-draft
description: 日本語ビジネスメールの下書きを作成する。ユーザが「メールを書いて」「xxさん宛に送るメール」「メール下書き」「日程調整メール」「依頼メール」「謝罪メール」「報告メール」等と言った時に発火する。宛先(取引先/社内/上司)、目的(依頼/日程調整/謝罪/報告)、トーン(丁寧/標準/カジュアル)、ポイント(本文の要点)の4つを受け取り、件名・本文・署名を含む完成したメール1通を返す。情報が不足する場合は1項目ずつ対話で埋める。単純な英語メール、チャット(Slack)投稿、社外広告文には使わない。
---

# email-draft Skill

日本語のビジネスメール下書きを、ユーザの意図を最短の対話で汲み取って生成する Skill。

## いつ発火するか

以下のいずれかに該当するユーザ発話で発火する:

- 「メール書いて」「メール下書き」
- 「xxさん宛のメール」「xx社に送るメール」
- 「日程調整のメール」「依頼のメール」「謝罪のメール」「報告のメール」
- 「文面考えて」+ 文脈がメールの時

下記は発火しない(別Skill または通常応答に任せる):
- Slack/Teams 投稿文面の相談 → 別Skill
- 英語メール → 本Skillは日本語前提のため
- すでに下書きがあって「推敲だけ」 → 通常応答で対応

## 4つの入力と収集ルール

| # | 項目 | 必須 | デフォルト | 例 |
|---|---|---|---|---|
| 1 | 宛先 | 必須 | — | 「A社 鈴木様」「上司の田中さん」 |
| 2 | 目的 | 必須 | — | 「キックオフMTGの日程調整」 |
| 3 | トーン | 任意 | 標準 | 丁寧 / 標準 / カジュアル |
| 4 | ポイント | 任意 | 目的から推論 | 「日時3候補提示」「資料事前送付」等 |

ユーザ発話から上記を抽出する。**不足している必須項目があれば、1項目ずつ順に質問する**(一気に全部聞かない)。任意項目はデフォルトで進め、完成後に「トーンやポイントを変えますか?」と聞く。

## 生成手順

1. **目的から骨格テンプレ選択** — `templates/` から以下を選ぶ:
   - 依頼系 → `templates/request.md`
   - 日程調整系 → `templates/schedule.md`
   - 謝罪系 → `templates/apology.md`
   - 報告系 → `templates/report.md`
   - 該当なし → `request.md` をベースに改変

2. **トーン規則の適用** — `templates/_tone_rules.md` を読み、宛先属性と指定トーンに応じて敬語レベル・定型句を決める。

3. **本文生成** — テンプレの `{{VAR}}` プレースホルダを、収集した入力から埋める。ポイントは箇条書きのまま残さず、**自然な文章に変換**する。

4. **出力形式** — 以下を必ずこの順で出す:

   ```
   【件名】xxx

   xxxx 様

   いつもお世話になっております。○○の△△です。

   (本文)

   (締めの文)

   何卒よろしくお願いいたします。

   ──
   署名(所属 / 氏名 / 連絡先)
   ```

5. **品質チェック** — 生成後、次を自己チェックしてから返す:
   - 件名は20字以内で、要件が一目で分かるか
   - 宛名の敬称(様/さん)がトーンと整合しているか
   - 二重敬語になっていないか(例: 「お伺いさせていただきます」 → 「伺います」)
   - ポイントが全て本文に反映されているか
   - 署名プレースホルダ `{{SIGNATURE}}` が残っていないか(ユーザの署名は初回に聞き、以後記憶を促す)

## 失敗時の対処

- **入力抽出失敗** — 「宛先は誰ですか?」のように1項目ずつ質問する。選択肢を3つ例示するとユーザが答えやすい
- **ポイントが曖昧** — 「伝えたい要点を3つ箇条書きで教えてください」と聞く
- **目的がテンプレに該当しない** — `request.md` をベースに使い、「このケースは新パターンとして templates/ に追加しますか?」と提案する

## テンプレートの更新方針

`templates/` 配下のファイルはユーザの実際のメール文体に寄せてカスタマイズする前提。初回実行時は同梱のひな型を使うが、2回目以降でユーザが「もう少しカジュアルに」等と指示したら、その差分を該当テンプレに反映するよう提案する(Ch04「テンプレと配布済みSkillの関係」参照)。
もっと詳しく: なぜ description をここまで長く書くのか

Claude Code は Skill の発火判断を、frontmatter の description 一箇所だけを見て行います(2026年1月時点)。短すぎる description は「メール書いて」のような曖昧発話に引っかからず、Skillがあるのに発火しないという残念な事故になります。逆に長すぎてもノイズですが、(1) 何をするか (2) いつ発火するか (3) 何をしないかの3点は最低限含めるのがこの章の推奨形です。上記SKILL.mdの description はこの3点を1文に詰めた実例です。

templates/ 全5ファイル — コピペで動く完全版

5つの テンプレート を配布します。下線付きの _tone_rules.md はトーン辞書として全骨格から参照されます。

templates/_tone_rules.md

# トーン規則

Skillはこのファイルを先読みし、宛先属性と指定トーンから敬語レベルを決める。

## トーン別パターン

| トーン | 宛名 | 冒頭 | 本文の語尾 | 締め |
|---|---|---|---|---|
| 丁寧 | 〇〇様 | いつも大変お世話になっております。 | 〜いただけますと幸いでございます。 | 何卒よろしくお願い申し上げます。 |
| 標準 | 〇〇様 / 〇〇さん | いつもお世話になっております。 | 〜いただけますと幸いです。 | 何卒よろしくお願いいたします。 |
| カジュアル | 〇〇さん | お疲れさまです。 | 〜だと助かります。 | よろしくお願いします! |

## 宛先属性によるデフォルト

- 取引先・社外 → 標準 or 丁寧(指定なければ標準)
- 社内の上司・目上 → 標準(指定なければ標準)
- 社内の同僚・チーム → カジュアル or 標準

## NG表現(Skillが自動回避)

- 二重敬語: 「お伺いさせていただきます」「拝見させていただきます」
- 冗長: 「〜の方」「〜になります」(物が主語の時以外)
- カジュアル混入: 「了解しました」→ 「承知いたしました」
- 過剰クッション: 「恐れ入りますが、お忙しいところ大変恐縮ですが」→ 1つに絞る

# 依頼メール骨格

【件名】{{SUBJECT_REQUEST}}

{{RECIPIENT}} 様

{{OPENING_TONE}}
{{SENDER_INTRO}}の{{SENDER_NAME}}です。

さて、このたび{{CONTEXT_SUMMARY}}につきまして、
{{RECIPIENT_SHORT}}にお願いがありご連絡いたしました。

{{REQUEST_BODY}}

{{DEADLINE_OR_CONDITION}}

ご多忙のところ恐縮ですが、{{CLOSING_TONE}}

{{SIGNATURE}}

# VAR ガイド(Skillが埋める)

- SUBJECT_REQUEST: 「〜のご依頼」「〜についてご相談」等、20字以内
- RECIPIENT: 「A社 鈴木」「田中」等、様/さんはトーン規則で決定
- OPENING_TONE / CLOSING_TONE: _tone_rules.md から引用
- REQUEST_BODY: ポイントを自然文に。必要なら「1. 〜 2. 〜」の箇条書きで列挙
- DEADLINE_OR_CONDITION: 「X月Y日までにご回答いただければ幸いです。」等、無ければ空行

{{RECIPIENT}} 様

{{OPENING_TONE}}

表題の件、{{MEETING_PURPOSE}}につきまして、
下記の候補日時から{{RECIPIENT_SHORT}}のご都合の良い日時をお知らせいただけますでしょうか。

▼候補日時(所要時間:{{DURATION}})
・{{CANDIDATE_1}}
・{{CANDIDATE_2}}
・{{CANDIDATE_3}}

場所:{{LOCATION_OR_URL}}
議題:{{AGENDA_SUMMARY}}

いずれもご都合が合わない場合は、{{RECIPIENT_SHORT}}のご都合の良い日時を

# VAR ガイド

- CANDIDATE_1〜3: ユーザ指定の3候補。無ければ「来週月〜金の午前中から3候補」を提案的に生成
- DURATION: 「30分」「1時間」等
- LOCATION_OR_URL: オンラインなら「Google Meet(URL別送)」等
- AGENDA_SUMMARY: 箇条書き2〜4点

【件名】{{ISSUE_SUBJECT}}のお詫び

{{RECIPIENT}} 様

{{OPENING_TONE}}

このたびは{{ISSUE_SUMMARY}}により、
{{RECIPIENT_SHORT}}に多大なご迷惑をおかけしましたこと、
心よりお詫び申し上げます。

▼発生事象
{{ISSUE_DETAIL}}

▼原因
{{ROOT_CAUSE}}

▼対応状況と再発防止策
{{RECOVERY_ACTION}}
{{PREVENTION_PLAN}}

{{RECIPIENT_SHORT}}におかれましては、ご不快な思いをおかけしましたことを

# VAR ガイド

- 謝罪メールは**トーンを常に「丁寧」に強制**(カジュアル指定でも)
- 言い訳を書かない。事実 → 原因 → 対応の順
- 「申し訳ありません」は冒頭と末尾の2回まで

templates/report.md

# 報告メール骨格

【件名】{{REPORT_TOPIC}}のご報告

{{RECIPIENT}} 様

{{OPENING_TONE}}

表題の件、以下のとおりご報告いたします。

■ 概要
{{SUMMARY_3LINE}}

■ 結果
{{RESULT_DETAIL}}

■ 次のアクション
{{NEXT_ACTION}}

ご不明な点やご質問がございましたら、お気軽にお申し付けください。

{{CLOSING_TONE}}

──
{{SIGNATURE}}

# VAR ガイド

- SUMMARY_3LINE: 3行で要点。上司向けは「忙しい人が3行で理解できる」を死守
- RESULT_DETAIL: 数字があれば必ず数字。定性なら「良好/課題あり」を明示
- NEXT_ACTION: 「いつ誰が何をやるか」を1文ずつ
プレースホルダ規約

すべて {{VAR}} の二重中括弧形式(Ch04で学んだ規約)。Skill本体はVARガイドを見ながら必要なだけ埋めるので、ユーザが全部を埋めてなくても自然文に成形してくれる。空欄のまま出さないことが重要です。

発火条件の書き方 — descriptionの精度

Skillがちゃんと発火するかは、description がどこまで具体的かにかかっています。下の分岐フローが判断ロジックの本質です。

「合致」の判定は description に書かれた動詞・名詞・トリガー語との類似度が鍵。

良いdescription / 悪いdescription

観点情報不足の例引き出せる例(本Skill採用)
動詞の明示「メール関連」「日本語ビジネスメールの下書きを作成する」
発火トリガー語(明示なし)「メール書いて / xxさん宛のメール / 日程調整メール…」とリテラル列挙
入出力「いい感じに書く」「宛先/目的/トーン/ポイントの4つを受け取り、件名・本文・署名を返す」
除外ルール(なし)「英語メール・Slack投稿・広告文には使わない」
対話方針(なし)「不足情報は1項目ずつ対話で埋める」
description の要素別スコア(自己採点)
本Skillのdescriptionは5軸すべてで8点以上を目安に書いている。
もっと詳しく: トリガー語を列挙する際のコツ

description にトリガー語を書くときは、ユーザが実際に打ちそうな自然文をそのまま並べるのが効きます。「xx機能」のような名詞形ではなく「xx書いて」「xxメール」のように動詞・接尾辞込みで列挙すること。逆に、トリガー語を増やしすぎると他Skillと競合して誤発火するので、最大5〜7個までに絞るのが本Skillのバランスです。

品質チェックの5項目

生成後、自分の目で5つの観点を確認します。Skill自身もこれを内部でチェックしていますが、最終判断は人間が握ります。

  • 件名は一目で分かるか — 20字以内。「ご連絡」「お疲れさまです」のような意味のない件名を弾く
  • 宛名と敬語が整合 — 取引先に「さん」はNG、社内同僚に過剰敬語もNG
  • 二重敬語ゼロ — 「お伺いさせていただきます」→「伺います」。自動検出リストがtone_rulesにある
  • ポイント全反映 — 入力した箇条書きが本文に反映されているか(抜けは一番やらかす事故)
  • 署名が入っているか — {{SIGNATURE}} のまま残っていないか、部署名まで入っているか

開発初期は二重敬語と署名抜けが多発。tone_rules.md と品質チェック項を足してから劇的に下がった。

Exercise 2
応用

description を改悪→改善して発火挙動を観察

Skillの発火判断は description にかかっていることを体感する演習です。

  1. 現行 SKILL.md の description を「メール書くSkill」だけの一言版に差し替えて保存
  2. Claude Codeで「鈴木さんに明日の資料送付のメールを書いて」と依頼 → 発火するか観察
  3. 発火しなかった/曖昧だった場合は、その発話で発火する最低限の description を自分で書き足す
  4. 元の長文 description に戻して同じ発話を投げ、発火の確実性の差を比較
  5. 「どの発火トリガー語が一番効くか」を3つ特定し、tone_rules.md にメモする
模範解答を見る

観察される典型パターン:

【一言版】description: "メール書くSkill"
→ 「鈴木さんに明日の資料送付のメールを書いて」
  でたまに発火、たまに通常応答になる(不安定)

【改善版】description: "日本語ビジネスメール下書きを作成。
  ユーザが『メール書いて』『xxさんに送るメール』と言った時に発火。
  宛先・目的・トーン・ポイントを受け取る。"
→ ほぼ100%発火する

【本章配布版】description: (長文・除外ルール込み)
→ 100%発火 + Slack投稿相談には誤発火しない

3つの効くトリガー語(経験則): (1) 動詞「書いて/下書き」 (2) 対象名「メール」 (3) 宛先表現「xxさん宛/xx様に」。この3語のうち2つ以上が発話に含まれれば発火確率は9割超。逆にこれらが一つも無い発話(例:「返信どうしよう」)では発火しないので、トリガー語として 返信 を足すか迷う——ここは他Skillとの誤発火リスクと天秤で決めます。

学び: description は「トリガー語の辞書 + 除外ルール」の2本柱で書く。本章のSKILL.md がそのリファレンス。
