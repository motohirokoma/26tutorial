実践：議事録整形Skill

生の会議ログから整形議事録を量産するSkillを作る

所要時間: 30分 Phase: Practice 難易度: 中級
この章のゴール

Google Meet の自動字幕、Slack ハドルの文字起こし、Zoom の チャット転写 — どれも「読める議事録」に整えるのに30分以上かかります。本章では 議事録整形Skill(meeting-notes)を作り、「議事録まとめて」と言うだけで Decisions / Actions / Questions の3セクションに整形された議事録を自動生成します。frontmatter・SKILL.md 本体・templates/ の全文を掲載、そのままコピペで動きます。本章は2026年1月時点の Claude Code Skill 仕様に基づきます。

シナリオ：生の会議ログ → 整形議事録

私の実務ではこんな流れが頻出します。30分の社内ミーティングが終わり、Google Meet の文字起こしが Slack に自動投稿される。議事録係のあなたは会議後に15分の隙間時間でまとめたいが、生ログは 話者ごとの発言がただ並ぶだけ で、どれが Decision で、どれが Action で、どれが Question なのか、読み直さないと分からない。

そこで Skill の出番。「議事録整形して」と1行打てば、生ログを meeting-notes Skill が発火判定し、テンプレに沿った綺麗な議事録が戻ってくる。ポイントは 毎回同じフォーマットで出る こと。チーム展開する時の品質バラつきが一気に消えます。

Skillが挟まる前と後
同じ生ログから、手作業では15-30分 / Skillでは2-3分。差は「型」が埋まっているかどうか。

この DAQ(Decisions / Actions / Questions) という3分類は、コンサル業界で古典的に使われる議事録の型で、発火条件・テンプレ・SKILL.md すべてを貫通する軸になります。

Before / After を具体で見る

Skillの威力は具体例で見るのが一番早い。同じ入力から両方の出力を並べます。

# 定例ミーティング 議事録 (2026-01-15)

**参加者**: 田中, 鈴木, 佐藤
**時間**: 10:00-10:30

## Decisions(決定事項)
- 新機能A のリリース日を **2026-02-06(金)** に確定
- 問い合わせ増加に対するサポート体制強化は **次週経営会議に持ち越し**

## Actions(アクションアイテム)
| 担当 | アクション | 期限 |
|---|---|---|
| 鈴木 | リリース告知(ブログ・Twitter) | 2026-02-05 |
| 佐藤 | 決済エラー原因のログ調査レポート | 2026-01-23(金) |

## Questions(持ち越し論点)
- サポート体制強化の要否 → 経営会議(2026-01-22)で議論
- 決済エラーの根本原因 → 佐藤の調査待ち
Before

生ログのまま

  • 決定とタスクが混在して判別困難
  • 期限・担当が文中に散らばる
  • 読み直し必須で会議参加者しか活用できない
After

Skill整形済み

  • 3セクションで一目で全体把握
  • Actions は表で担当・期限が明示
  • 欠席者もこれ1枚で完結
もっと詳しく: なぜDAQの3分類なのか

議事録の分類は流派が色々あります(5W1H分類 / トピック別 / 時系列そのまま 等)。それでもMKが DAQ(Decisions / Actions / Questions) を推す理由は3つ:

  1. Claudeの分類精度が高い:「決まったこと / 誰が何をやるか / 未解決」の3択は境界が明確で、LLMが迷わず振り分けられる
  2. 欠席者の再利用性が最大:Decisions だけ見れば結論、Actions を見ればタスク、Questions で議論の火種を把握できる
  3. 後から検索しやすい:「あの時の決定事項どこだっけ」→ Decisionsを grep するだけ

オプションで Risks(リスク)や Next Agenda(次回議題)を4つ目・5つ目として足す流派もあります。Skillを拡張するなら templates/ に別 variant を足すだけで対応可能。

# meeting-notes スキル

会議の生ログから、読み返しやすく再利用しやすい議事録を量産するスキル。**DAQフォーマット(Decisions / Actions / Questions)** で出力を固定する。

## いつ使うか

- ユーザが「議事録整形して」「議事録まとめて」と言った時
- ユーザが「会議メモをまとめて」「Meet の文字起こしを整理して」と言った時
- ユーザが会議の生ログ(話者名 + 発言の羅列)を貼って「DAQで」「3分類で」と言った時
- ユーザが `/meeting-notes` を明示的に実行した時

## 発火しないケース

- ユーザが「会話ログを要約して」とだけ言った場合(会議でない可能性、ユーザに会議か確認)
- ログが10行未満(会議として短すぎ、普通の要約スキルで十分)

## 手順

### Step 1: 入力の確認
ユーザから以下を受け取る(不足分は1問ずつ聞く):

| 項目 | 必須 | デフォルト |
|---|---|---|
| 会議ログ本体 | 必須 | — |
| 会議タイトル | 任意 | ログ冒頭から推定 |
| 日付 | 任意 | ログから抽出、なければ「未記載」 |
| 参加者 | 任意 | 話者名から自動抽出 |

ログが貼られていなければ「会議の生ログを貼ってください」と1回だけ聞く。

### Step 2: 分類ルール

発言を3カテゴリに振り分ける:

- **Decisions**: 「〜に決定」「〜にしよう」「〜でいこう」など、**会議内で確定した結論**。期限や担当を含まない結論もここ
- **Actions**: 「誰が / 何を / いつまでに」の3要素が揃ったタスク。担当・期限が曖昧な場合は、ログから推測できる範囲で補完し、不明なら "TBD" と明記
- **Questions**: 「持ち越し」「次回議論」「要確認」「不明」と明示されたもの、または結論が出ずに話題が流れた論点

1つの発言が複数カテゴリに該当する場合は、**最も強いもの1つ**に配置する(例: Decisionに付随するTodoならActionsへ)。

### Step 3: 出力生成

`templates/meeting-notes.md` を読み込み、プレースホルダを埋めて出力する。

プレースホルダ:
- `{{TITLE}}` — 会議タイトル
- `{{DATE}}` — 日付 (YYYY-MM-DD)
- `{{PARTICIPANTS}}` — 参加者(カンマ区切り)
- `{{TIME_RANGE}}` — 開始-終了時刻(あれば)
- `{{DECISIONS}}` — Decisions の箇条書き
- `{{ACTIONS_TABLE}}` — Actions の表(担当 / アクション / 期限)
- `{{QUESTIONS}}` — Questions の箇条書き

Actionsが0件なら「(なし)」、Questionsが0件なら「(なし)」と明記する。空欄は作らない。

### Step 4: 分類判断に迷った時

`templates/few-shot.md` の Before/After 例を参照する。この例で「この発言はどのカテゴリか?」の感覚を掴む。迷ったら **Decisions > Actions > Questions** の優先順で分類する(Decisionsが最も「完了した情報」なので格上)。

### Step 5: 出力後のユーザ確認

出力した議事録をユーザに見せ、「分類に違和感あるものがあれば指摘してください」と1行添える。ユーザの指摘があれば該当項目を動かして再出力。

## 注意点

- 発言者の名前は **最大4人まで明示**、それ以上は「他N名」と集約
- 1つの Decision / Action に対して、元発言を1-2個引用できる時は `(原文: 〜)` で併記すると信頼性が上がる
- 機密性の高い会議ログを扱う可能性があるので、**ユーザの許可なく外部ツールに送らない**
- ログが極端に長い(10,000字超)場合は、先にユーザに「全文処理するか章別に分けるか」聞く

## 失敗時の対処

- 話者が1人しか識別できない → 「発言者の判別が困難です、議事録ではなくメモの要約として処理しますか?」と確認
- Decisionsが0件 → 「確定した決定事項が見当たりません。議論のみの会議でしたか?」と確認した上で Questions 中心で出力
- 期限・担当が全く書かれていない → Actions の該当セルを "TBD" で埋めて、末尾に「期限と担当の確認をお勧めします」と添える

ここで注目してほしいのが description の トリガー語彙の多様性。「議事録整形」だけでなく「議事録まとめて」「DAQで整理」など、人間が実際に使いそうな言い回しを並べています。Ch03 で学んだ「明示トリガー」の実戦適用です。

templates/ の全文(2ファイル)

templates/meeting-notes.md が 出力の正本、templates/few-shot.md が 分類ルールの手本。この2つを分けることで、フォーマット変更と分類ルール変更を独立して進化させられます(Ch04の原則「テンプレート分離」)。

## 分類のコツ

- **「〜に決めた / 決まった / でいこう」**→ Decisions
- **「私が / 〜が担当で / 〜までに」**→ Actions
- **「持ち越し / 次回 / あとで / 考えましょう」**→ Questions
- 過去形の確定表現 > 未来形の意思 > 曖昧表現 の順で「強さ」を評価し、最も強いカテゴリに置く

この3例のfew-shotが、Claudeの分類精度を体感で2倍くらい引き上げます。特に 例3(曖昧発言) が入ってるのが重要。これがないと「なんとかしたい」をActionsに入れてしまう誤分類が起きます。

発火条件の精度を上げるコツ

description は Skill の命。これが弱いと発火せず、強すぎると誤発火します。このSkillで実測したトリガー語彙の 発火ヒット率 を示します。

トリガー語彙別のヒット率(実測)
「議事録整形」「議事録まとめて」は鉄板、「要約して」だと他Skillに取られがち。
  • 具体トリガーを3-5個 — 1個だと発火漏れ、10個だと誤発火。「議事録」という名詞キーワードが軸
  • 発火しないケースも書く — 「会話ログを要約して」は別Skill領域だと明示しておくと誤発火が減る
  • 入力の形状もトリガーに — 「会議の生ログらしきテキストを貼って」という文脈発火も効く
  • 1週間運用して微調整 — 発火漏れ/誤発火が出たら description を追記、これがSkill運用の9割
make-tutorial skill に学ぶ

本チュートリアル自体を生成した make-tutorial Skill の frontmatter も、「学習サイトを作りたい」「チュートリアルを量産したい」と ユーザ側の言い回し で書かれています。「コマンド名で発火させる」ではなく「ユーザの自然な発話から発火させる」のが Skill の正しい設計思想。

もっと詳しく: 自社用カスタマイズの典型パターン

会社・チームによって議事録の流儀は違います。よくあるカスタマイズ:

  • Risksセクション追加 — リスク管理が厳しい会社ではDAQ→DARQの4分類。templates/meeting-notes.md と SKILL.md のStep 2に Risks カテゴリを足す
  • 次回議題(Next Agenda) — 週次定例が多いチームはNext Agendaを最後に1セクション追加
  • 英語出力オプション — 外資系は英語議事録が必要。SKILL.mdに「ユーザが英語指定した場合は英語で出力」を追記、templatesにen版を追加
  • NotionコピペモードSwitch — Notionに貼る時は「---」区切りが邪魔。templates/meeting-notes-notion.md を作り、Step 3で分岐

**どれも templates/ と SKILL.md だけの修正で済む** のが Skill 設計の強さ。Claude Code 本体をいじる必要は一切ありません。

# デザインレビュー 議事録 (2026-01-XX)

**参加者**: 山田, 小林

## Decisions(決定事項)
- 新ロゴ案Bを **最終確定**

## Actions(アクションアイテム)
| 担当 | アクション | 期限 |
|---|---|---|
| 小林 | ロゴ展開用アセット一式の作成 | 来週火曜 |

## Questions(持ち越し論点)
- アプリアイコンのデザイン方針 → 来週デザイナーと再議論

この出力が返ってこない場合の typical な原因は(1) SKILL.md の frontmatter が ---で囲まれていない、(2) ファイルパスが meeting-notes/SKILL.md になっていない(大文字小文字ミス)、(3) description のトリガー語彙が「議事録まとめて」を含んでいない、のどれかです。

Exercise 2
応用

自分のチーム用にRisksセクションを追加してカスタマイズ

本章のSkillをベースに、Risks(リスク)セクションを追加したDARQ版に改造してください。改造後、Risks発言を含む生ログで動作確認まで。

  1. templates/meeting-notes.md に Risks セクションを追加
  2. SKILL.md の Step 2 に Risks の分類ルールを追記
  3. templates/few-shot.md に Risks を含む例を1つ追加
  4. description にも「DARQで」「リスク含めて」などのトリガー語彙を追加
  5. Risksを含む架空ログ(例:「本番障害リスクがある」「セキュリティ脆弱性の恐れ」など)で動作確認
模範解答を見る

templates/meeting-notes.md の修正(Risksセクション追加):

# {{TITLE}} 議事録 ({{DATE}})

**参加者**: {{PARTICIPANTS}}
**時間**: {{TIME_RANGE}}

## Decisions(決定事項)
{{DECISIONS}}

## Actions(アクションアイテム)
{{ACTIONS_TABLE}}

## Risks(リスク・懸念)
{{RISKS}}

## Questions(持ち越し論点)
{{QUESTIONS}}

SKILL.md の Step 2 追記:

- **Risks**: 「リスクがある」「懸念される」「〜の恐れ」「〜が危険」など、**発生したら問題になる未確定事項**。Questionsと違い「論点」ではなく「警戒対象」。インパクト(H/M/L)が推測できれば併記する

description の改訂:

description: 会議の生ログから Decisions/Actions/Risks/Questions(DARQ) の4セクションに整形した議事録を生成する。ユーザが「議事録整形」「議事録まとめて」「会議メモをまとめて」「DARQで整理」「リスク含めて議事録」などと言った時に発火する。

few-shot.md への追加例:

## 例4: Risksを含むケース

### Before
田中: 来月リリースで行きましょう。
鈴木: ただ、本番データ移行で障害が起きる可能性があります。リハ必須です。
佐藤: リハは私が今週中に計画を立てます。

### After
## Decisions
- 来月リリースで進行

## Actions
| 担当 | アクション | 期限 |
|---|---|---|
| 佐藤 | 本番データ移行リハの計画策定 | 今週中 |

## Risks
- 本番データ移行で障害発生の恐れ(影響度: H)→ リハで緩和

## Questions
(なし)

この改造により、本番運用・セキュリティが絡む会議で重宝する議事録Skillになります。MKの実運用では、エンジニアリング系会議は DARQ、企画系会議は DAQ、と2種のSkill(meeting-notes と meeting-notes-risk)を使い分けています。Skillは単一責任(Ch03)が原則なので、分岐が複雑になるなら別Skillに割るのが正解。
