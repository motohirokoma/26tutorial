Phase 2: Core > Chapter 04

従来型ML / 予測AI / データ基盤 / MLOps

「LLMで全部解ける」は誤解です。価格予測・需要予測・異常検知・離脱予測など、LLMより従来型MLの方が圧倒的に向いている領域があります。本章では LLM と並ぶもう一つの主役を扱い、それを支えるデータ基盤と運用(MLOps)も体系的に押さえます。

所要時間: 90分 Phase: Core 難易度: ★★★☆☆
本章の到達点

顧客の課題を聞いた瞬間に「これは LLM じゃなく予測ML」「これはデータ基盤の整備が先」と判断できる状態になります。本章の記述は2026年4月時点の情報に基づきます。

1. なぜ LLM だけでは足りないのか

2026年でも、エンタープライズの AI 案件の過半数は従来型MLが主役です。LLMがハマる領域とMLがハマる領域は、性質が違います。

LLM と 従来型ML の使い分け
LLMは「非構造データの汎用処理」、従来型MLは「構造化データの特化予測」。両者は競合ではなく補完関係。

判定の具体例

顧客の課題適切な手法理由
顧客の解約予測(チャーン)従来型ML構造化された顧客行動ログを学習、判断根拠の説明が必要
商品レビューの感情分析LLM非構造の文章、汎用言語理解が活きる
3ヶ月先の在庫需要予測従来型ML(時系列)過去データから数値を予測、季節性・トレンドを学習
顧客サポートのFAQ自動応答LLM + RAG多様な質問に汎用的に応答、知識更新も容易
製造ラインの不良品検出従来型ML(画像認識)専門特化型のCNN、低レイテンシが必要

教師あり学習 — 業務での主役

  • 回帰 — 連続値の予測。例: 来月の売上、機器の残存寿命、不動産価格
  • 2値分類 — Yes/Noを当てる。例: 解約する/しない、与信OK/NG、不良品/良品
  • 多クラス分類 — 複数カテゴリ。例: 商品カテゴリ判別、製品の不良種別

表形式データの予測課題は、まずLightGBMから試すのが定石。それで足りなければNN等を検討。「いきなり Deep Learning」と提案すると過剰設計の疑念を持たれます。

3. 評価指標 — 「精度99%」に騙されないために

顧客の前で「精度99%」と聞いたら、必ず「何の精度ですか?」と聞き返してください。ML の評価指標は1種類ではなく、用途ごとに使うべき指標が違います。

混同行列(Confusion Matrix) — 分類の評価基本
不良品検出では、FN(見逃し)が一番怖い。出荷後の顧客クレーム・リコール直結。「精度99%」と言われたらFNがどう数えられているかを確認。
指標定義使う場面
Accuracy(精度)全体に対する正解率 = (TP+TN)/全件クラスがバランスしている時のみ。不良品0.1%のような偏ったデータでは無意味
Precision(適合率)陽性予測のうち実際に陽性 = TP/(TP+FP)FP を避けたい時(スパムフィルタ等、誤判定で重要メールが消えると困る)
Recall(再現率)実際の陽性のうち捕捉した = TP/(TP+FN)FN を避けたい時(不良品検出、がん検診、不正検知)
F1スコアPrecision と Recall の調和平均両方バランスよく見たい時
AUC(ROC曲線下面積)分類の総合性能閾値に依存しない総合評価
もっと詳しく: 業界別の重要指標
  • 金融(不正検知): Recall重視(不正を見逃したくない)+ Precisionも一定確保(誤検知で正常顧客を停止すると問題)
  • 医療(疾患検出): Recall最優先(見逃しはダメ)、Precisionは二次
  • マーケ(レコメンド): Top-K Precision(上位K件のうち実際にクリック/購入された割合)
  • 製造(不良品検出): Recall最優先 + 全数検査と比較したコスト削減効果
  • 営業(リード予測): AUC + Lift(上位5%のリードがランダム抽出より何倍成約するか)

「AI 入れる前にデータ基盤を整える必要がある」案件は半数以上です。本章前半の全ては、整ったデータがある前提。整っていなければ、まずそこから手を入れる必要があります。

「データを集める」→「整える」→「特徴量化」→「モデル化」→「届ける」→「監視」の全行程。1つでも欠けると本番運用は崩れる。

2026年のデータ基盤論争 — DWH vs Lakehouse

DWH

データウェアハウス

長所: 構造化済み、高速SQL、BI連携◎
短所: 非構造データに弱い、コスト高、ベンダーロックイン
主要: Snowflake / BigQuery / Redshift

Lakehouse

レイク + DWH 統合

長所: 非構造もOK、安価、AI処理直結、オープン形式
短所: 設計が複雑、運用ノウハウが必要
主要: Databricks / Snowflake(Iceberg対応)/ Microsoft Fabric

データ品質 — 「ゴミからは金が出ない」

  • 完全性 — 必要なフィールドが欠けていないか(欠損値率)
  • 正確性 — 入力値が正しいか(明らかなノイズ・誤入力がないか)
  • 一貫性 — システム間で表記やスケールが揃っているか
  • 適時性 — どれくらい鮮度のあるデータか(リアルタイム/日次/月次)
  • 代表性 — 学習データが本番のデータ分布を代表しているか
PoC前の鉄板チェック

顧客から「データはあります」と言われても、実物を必ず見てから判断してください。よくある罠:
① 項目はあるが半分が空欄(過去のデータ移行で欠損)
② システム間で顧客ID体系が違う(突合が困難)
③ 「最新」と言われても3ヶ月前まで(運用上の遅延)
④ 個人情報マスク漏れ(PoCに使えない)

5. MLOps — モデルを育て続ける仕組み

本番運用で最も難しいのは、「モデルを作る」ことよりも「動き続けさせる」こと。MLOps はソフトウェアエンジニアリングの DevOps を ML に拡張した運用思想です。

MLOps ライフサイクル
本番デプロイは終着点ではなく、サイクルの一部。データ分布が変われば再学習、新しい要件があれば再評価。「作って終わり」だと運用が崩れる。

本番で必ず起きる問題 — Data Drift と Concept Drift

Data Drift

入力データの分布変化

例: コロナ後の旅行需要予測モデルが、平常時のデータで学習されているため精度が落ちる
対策: 入力統計量を継続監視、変化したら再学習

Concept Drift

正解の概念が変わる

例: 「優良顧客」の定義が、利用頻度から利用金額に変わった
対策: 定期的に評価指標を確認、業務側と概念を擦り合わせ

MLOps で押さえるべき5つの仕組み

  • バージョン管理 — コード(Git)+ データ(DVC等)+ モデル(Model Registry)
  • CI/CD パイプライン — 学習→評価→デプロイの自動化(GitHub Actions、Vertex Pipelines等)
  • 監視 — 入力分布・モデル出力分布・推論レイテンシをリアルタイム把握
  • A/B テスト — 新モデルを一部トラフィックに流して安全に評価
  • 再学習トリガー — 性能劣化を検知したら自動で再学習開始

主要MLOpsプラットフォームの選び方

プラットフォーム得意苦手

「データサイエンスの 80% は前処理」の現実

有名な格言ですが、本当です。ML案件の工数配分目安:
課題定義: 5%、データ収集・前処理: 60%、特徴量設計: 15%、モデリング: 10%、評価・本番化: 10%。
顧客に「PoC 3ヶ月」と提案した時、本気でモデリングしているのは2-3週間だけ、というのが現実。これを最初から織り込んでスケジュールを組んでください。

顧客に説明する時の罠 — 過剰な精度期待

  • 「精度99%出ます」 → 本番ではほぼ無理。学習データでの過学習を疑う
  • 「データを増やせば精度UP」 → 量より質。ノイズの多いデータは増やしても無意味
  • 「最新モデルなら大丈夫」 → 表データ系では、XGBoostがDeep Learningに勝ち続けている
  • 「AIで全自動化できる」 → 重要な判断には必ず人間レビューを残す。Human-in-the-Loop が現実解

7. 章のまとめ

  • LLM と従来型MLは補完関係。構造化データの予測は従来型MLが今も主役
  • 表データはまずLightGBM/XGBoostから。DLは画像・音声・時系列で
  • 評価指標は用途で使い分ける。Recall重視か Precision重視かを業務文脈で判断
  • AI案件の8割の障害はデータ品質。提案前に必ず実物確認
  • MLOps は「動き続ける仕組み」。Drift対策、再学習、監視を最初から設計
  • 2026年のデータ基盤の主流はLakehouse。Databricks / Snowflake が双璧

Exercise

Exercise 1
入門

用途に対する正しい評価指標を選ぶ

以下の各シナリオで最も重視すべき評価指標を選び、理由を述べてください。

  1. 銀行のクレジットカード不正取引検知
  2. ECサイトの商品レコメンド
  3. 製造ラインの不良品検出(出荷前)
  4. スパムメールフィルタ
模範解答を見る
  1. 不正検知: Recall優先。不正の見逃し(FN)が損害直結。ただし FP(正常を不正と誤判定)で正規顧客を停止すると問題なので、Precision も一定確保。F1 や AUC で総合判断
  2. 商品レコメンド: Top-K Precision(上位K件のうち実際にクリック/購入された割合)。下位の精度はユーザー体験に影響しない
  3. 不良品検出: Recall最優先。不良品を見逃して出荷したら顧客クレーム・リコール直結。FP(良品を不良と誤検出)は再検査コストで済むので許容
  4. スパムフィルタ: Precision優先。FP(正常メールをスパム判定)で重要連絡を見逃すと致命的。FN(スパムを通す)はユーザーが手動で削除可能

共通ポイント: 「FP と FN のどちらの方が業務インパクトが大きいか」で決める。これを顧客と最初に合意するのが鉄則です。

Exercise 2
応用

顧客に「データ基盤整備」を提案する

顧客企業(中堅小売、売上300億円、店舗50店、ECあり)から「AIで売上予測したい」と相談されました。

  1. そのままML案件を始める前に、データ基盤の状態を確認すべき5つの質問を作る
  2. もし「データ基盤の整備が先」と判断した場合、その理由と3-6ヶ月の整備プランを提案する
  3. 整備後に取り組むべき売上予測モデルの構成(データソース・アルゴリズム・評価指標)を提案する
模範解答を見る

5つの確認質問:

  1. 店舗POS・EC・在庫システムは別々ですか?それぞれの会社システムか、SaaS か?データを統合するハブはありますか?
  2. 顧客IDは店舗とECで連携していますか?(オムニチャネルとして一意の顧客が識別できるか)
  3. 過去何年分のデータを保持していますか?日次粒度?時間粒度?
  4. 商品マスタは正規化されていますか?(同じ商品が違うコードで登録されていないか)
  5. 外部要因データ(天気・イベント・競合動向)を取り込む仕組みはありますか?

整備プラン(6ヶ月):

  • Month 1-2: 現状アセスメント、データソース棚卸し、データレイク(S3)設計
  • Month 3-4: ETL パイプライン構築(POS+EC+在庫の統合)、DWH(BigQueryまたはSnowflake)構築
  • Month 5: マスタ整備(商品・顧客の名寄せ)、データ品質チェックの自動化
  • Month 6: BIダッシュボード(現状把握)、データ分析基盤としての試験運用

(整備に半年かかる案件も普通。これを「3ヶ月でAI導入」と約束すると詰みます)

整備後の売上予測モデル:

  • データソース: 過去3年のPOS+EC日次売上、商品マスタ、店舗マスタ、天気API、イベント情報、価格・販促履歴
  • アルゴリズム: 第1段階で LightGBM(店舗×商品×日次の予測)、第2段階で時系列専用(Prophet/DeepAR)を組み合わせ
  • 評価: WAPE(加重絶対誤差率)を主指標に、店舗別/商品カテゴリ別に分解。業務側の発注精度向上(欠品率と過剰在庫率の改善)を最終KPI
  • 運用: 週次で再学習、Data Drift 監視、月次で店長レビュー(Human-in-the-Loop)

ポイント: 「AI入れる」と「データ基盤整える」のどちらが先かを判断できるのが、本物の提案力です。データが整っていない状態でPoCしても、本番で再現せず信頼を失うだけ。
