戦略コンサル思考 — イシュー・仮説・構造化

技術を学ぶ前に「考える OS」を入れます。AI・DX 案件で最も多い失敗は、技術選定でも実装でもなく、「何を考えるべきか」を間違えること。本章ではトップ戦略コンサルが共通して使う5つの思考道具を、AI/DX 文脈で身につけます。

所要時間: 90分 Phase: Basics 難易度: ★★☆☆☆
この章のゴール

顧客の前で「論点はこれです」と30秒で示せること。技術解説や事例紹介の前に、まず考えるべきこと自体を構造化できる状態になります。

1. なぜ「考える前に何を考えるか」が肝心か

AI・DX案件は「何でもできる、けど何をやるべきか分からない」のが本質的な難しさです。技術が万能になればなるほど、論点設定の重要度が上がります。

考えるプロセスの全体像
graph LR
  A["状況"] --> B["問い
(イシュー)"]
  B --> C["仮説"]
  C --> D["構造化分解"]
  D --> E["検証"]
  E --> F["結論"]
  F --> G["伝える
(ピラミッド)"]
  E -.戻る.-> C
プロのコンサルは「問いを立てる」に最も時間を使う。検証で仮説が崩れても問いに戻る。
やるべきことのイシューを定めるところに、最も時間と頭を使うべきである。一度設定したイシューを、解くだけならその後は機械的にできる。
— 安宅和人『イシューからはじめよ』

「悪いコンサル思考」の典型 — これを避けるために学ぶ

よくある失敗

網羅型・収集型

  • 業界のトレンドをひたすら集める
  • 競合の事例を全部並べる
  • 結論の前に「分析100ページ」
  • 顧客の言うことを全部受ける
プロの仕事

イシュー型・仮説型

  • 「最も意思決定に効く問い」を1つ決める
  • その問いに対する答えを先に書く
  • 分析は仮説の検証のためだけに行う
  • 顧客の言うことを問いに翻訳する

2. イシュー — 良い問いの3条件

「イシュー」とは「今この瞬間に答えを出すべき問い」です。良いイシューは3つの条件を全て満たします。

良いイシューの3条件(全て必要)
3つの円が重なる中央だけが「価値ある問い」。1つでも欠けると、解いても価値がないか、解けない。

具体例で見る「良いイシュー」と「悪いイシュー」

状況悪いイシュー(漠然)良いイシュー(具体・本質)
顧客「AI で売上を伸ばしたい」AIで売上を伸ばすには?主要顧客の離脱率15%を、AIによる行動予測で10%以下に下げられるか?
顧客「DXを進めたい」DXの進め方は?営業現場で提案書作成に費やしている週20時間を、生成AI で5時間以下にできるか?
顧客「AIガバナンスを整えたい」AIガバナンスとは何か?EU AI Act 完全施行までに、社内100モデルの中で「高リスク」分類に該当する10モデルに対する管理プロセスを構築できるか?
良いイシューの作り方(実践テクニック)

顧客の漠然とした要望を、必ず以下の3要素で書き直してください:
「どの主体」が「どの状態」から「どの状態」になれるか?(期間限定)
例: 「AIで効率化したい」→「営業1人あたりの提案書作成時間を週20hから5hへ、3ヶ月で下げられるか?」

もっと詳しく: イシューの「サイズ感」

イシューには階層があります。

  • 大イシュー: 経営レベルの重大問い(例: 「AI時代に当社の競争優位は何か?」)
  • 中イシュー: 部門レベルの問い(例: 「営業生産性をどう倍にするか?」)
  • 小イシュー: 実行レベルの問い(例: 「提案書テンプレをどう設計するか?」)

提案の場面では、顧客のステークホルダーレベルに合わせることが鉄則。CxO相手に小イシューを話せば「なぜそんな細部の話を?」と思われ、現場相手に大イシューを話せば「壮大すぎて自分には関係ない」と思われます。

3. 仮説思考 — 答えを先に書く

イシューが決まったら、すぐに仮の答えを書きます。これが仮説思考。「分析してから結論」ではなく「結論を仮置きして、その検証のために分析する」逆転の発想です。

1
イシュー設定
問いを決める
2
仮説立案
答えを仮置き
3
検証設計
何を見れば検証できるか
4
分析実行
必要な分析だけ
5
結論
仮説を採用 or 修正

仮説思考が「できる人」の3つの違い

  • 速い — 分析途中で「これは違う方向だ」と気づける。情報が出揃う前に意思決定できる
  • 的が絞れる — 仮説検証に必要な分析だけ行う。100ページの分析資料を作らずに済む
  • 反復が効く — 仮説が外れても次の仮説にすぐ移れる。試行回数で精度を上げる
仮説検証ループ
graph LR
  Q["イシュー"] --> H["仮説"]
  H --> P["「もし◯◯なら△△が見えるはず」"]
  P --> A["分析・データ確認"]
  A --> R{"仮説と一致?"}
  R -- "Yes" --> Conf["仮説採用"]
  R -- "No" --> Rev["仮説修正"]
  Rev --> H
仮説は外れて当たり前。外れたら速く修正するのがプロの仕事。
AI/DX 案件での仮説思考

AI/DX 案件では、技術選定の局面で仮説思考が特に効きます。「LLM か従来MLか」を全部調べてから決めるのではなく、「この課題はLLMで解ける」と仮置きして、検証で外れたら従来MLへ移る。最初から正解を求めない設計が、PoC を速くします。

4. 構造化 — MECEとロジックツリー

仮説を検証するには、要素を漏れなく重複なく(MECE)分解します。これがコンサル思考の中核ツール。

MECE — Mutually Exclusive, Collectively Exhaustive

MECE: 重なりも漏れもなく分ける
「性別」と「年齢」を独立した軸として組めば自動的にMECE。1軸内で重なりがあったらNG。

ロジックツリー — イシューを階層的に分解

ロジックツリー例: 「営業生産性をどう倍にするか?」
graph TD
  Q["営業生産性を倍にする"] --> A["営業時間を増やす"]
  Q --> B["成約率を上げる"]
  Q --> C["案件単価を上げる"]
  A --> A1["事務作業の自動化"]
  A --> A2["移動時間の削減"]
  B --> B1["顧客選定の精度UP"]
  B --> B2["提案書の質UP"]
  C --> C1["上位顧客の獲得"]
  C --> C2["クロスセル"]
大きな問いを「足し算/掛け算で構成される要素」に分解する。各分岐がMECEになっているか毎回確認。
よく使うMECEの切り口

慣れないうちは、以下の定型の切り口を当てはめてみてください。

  • 足し算: 売上 = 客数 × 単価 × 頻度
  • 時系列: Before / During / After、短期/中期/長期
  • 主体: 自社 / 顧客 / 競合(3C)
  • 機能: 入力 / 処理 / 出力
  • 段階: 認知 / 興味 / 検討 / 購入(AIDMA)
もっと詳しく: AI/DX 案件で使うMECEの応用

AI/DX 案件では「業務の3層 × 技術の3層」という独特のMECE切り口があります。

業務の3層: 入力(データ収集) / 処理(判断・予測) / 出力(アクション・レポート)

技術の3層: データ基盤 / モデル / インターフェース

この6セルマトリクスで顧客の現状を分析すると、「データ基盤は弱いが、出力UIは整っている」など偏りが可視化されます。Ch07 (業界未来予測) で詳しく扱います。

5. ピラミッド原則 — 結論ファーストで伝える

考え抜いても、伝わらなければ価値ゼロ。コンサルが共通で使う伝え方が「ピラミッド原則」です。結論を頂上に置き、根拠を下に並べる。

ピラミッド原則の構造
下から上は「だから何が言える?(So What?)」、上から下は「なぜそう言える?(Why So?)」。両方向で論理が通っているか検証する。

悪いプレゼンと良いプレゼンの構造比較

悪い: 時系列・経緯型

「順を追って説明します」

1) 業界トレンドは…
2) 当社の現状は…
3) 検討した選択肢は…
4) 比較分析は…
5) 結論は…(50ページ目)

問題: 結論まで読まないと意味が分からない、忙しい意思決定者は途中で離脱

良い: 結論先行・ピラミッド型

「結論を先に言います」

1) 結論: AはBすべき(1ページ目)
2) 根拠: 3点(2-4ページ)
3) 詳細データ: Appendix

強み: 1ページで意思決定可能、詳細を聞きたい人だけ深掘れる、議論が活性化

重要な情報は、聞き手が30秒で席を立っても伝わるように構成しなければならない。 — バーバラ・ミント『考える技術・書く技術』

結論ファーストのテンプレート

PREP法
Point(結論) → Reason(理由) → Example(事例) → Point(再結論)
SCQA法
Situation(状況) → Complication(複雑化) → Question(問い) → Answer(答え)
エレベーターピッチ
30秒以内: 「誰が」「何の問題で」「どうなる」「だから何が必要」

6. AI/DX案件での実戦応用

5つの道具(イシュー / 仮説 / MECE / ロジックツリー / ピラミッド)を、AI/DX案件の実際の場面に当てはめます。

場面1: 顧客との初回ヒアリング

ヒアリングの構造
sequenceDiagram
  participant U as 顧客
  participant Y as あなた
  U->>Y: AIで何かやりたいんですよね
  Y->>U: 「何か」とは具体的にどの業務でしょうか?
  U->>Y: 例えば営業の効率化…
  Y->>Y: [仮説: 提案書作成かCRM管理か?]
  Y->>U: どちらが今、最も時間を取っていますか?
  U->>Y: 提案書作成ですね
  Y->>U: 1人あたり週何時間、何をされていますか?
  U->>Y: 20時間ほどテンプレ流用と数値計算
  Y->>Y: [イシュー確定: 提案書作成20h→Xh]
プロは「具体的な数字」と「現状の業務内訳」まで掘り下げる。これがイシュー設定の基礎データ。

場面2: 提案書の構造

章内容ピラミッドのどこか
エグゼクティブサマリー結論3行+主要根拠3点頂上
現状認識・イシューイシューを共有・合意頂上の前段
解決の方向性仮説と実装アプローチ2段目
期待効果定量的なROI試算2段目
実装計画フェーズ・スケジュール3段目
体制・コスト人員と予算3段目
リスク・前提制約条件と対応案下段
Appendix技術詳細・参考資料支える基盤

場面3: 社内議論で「決められない会議」を救う

  • 「結局何を決める会議?」 と問い直す → イシューが定義されていない可能性
  • 「論点を分解しましょう」 でロジックツリーを書く → 議論の骨格が見える
  • 「仮の結論を置いてみます」 で仮説を提示 → 反論することで議論が進む

7. 章のまとめ

  • イシュー: 良い問いの3条件(本質的・仮説あり・解ける)を満たす問いだけが価値を生む
  • 仮説思考: 答えを先に置き、検証で修正する。速さと精度を両立する
  • MECE: 漏れなく重複なく分ける。慣れないうちは定型の切り口を当てる
  • ロジックツリー: 大きな問いを階層的に分解、各分岐がMECEに
  • ピラミッド: 結論を頂上、根拠を下に。So What? と Why So? の双方向検証

Exercise

Exercise 1
入門

悪いイシューを良いイシューに書き直す

以下の3つの「悪いイシュー」を、本章の3条件(本質的・仮説あり・解ける)を満たす形に書き直してください。

  1. 「AIで何かしたい」
  2. 「DX人材を育成したい」
  3. 「コスト削減したい」
模範解答を見る
  1. 「AIで何かしたい」 → 「営業部門の50人が、提案書作成に費やす週20時間を、生成AIによる下書き生成で5時間以下に、3ヶ月以内にできるか?」
  2. 「DX人材を育成したい」 → 「事業部のミドル層30人に対し、6ヶ月のプログラムで、自部門のDX企画書を独力で書ける状態(具体的KPI: 修了時に各自が10億円規模の企画を1本提出)を実現できるか?」
  3. 「コスト削減したい」 → 「年間IT運用コスト100億円のうち、クラウド移行で削減可能なオンプレ運用部分(40億円)を、2027年度末までに20億円(50%削減)に下げられるか?」

共通ポイント: 主体・現状値・目標値・期間を全て埋めてください。これが揃っていないイシューは「考えるべき問い」になっていません。

Exercise 2
応用

「営業生産性を倍にする」をピラミッドで提案にまとめる

以下のシナリオでピラミッド原則に従った提案構造を作ってください。

  1. 顧客は中堅BtoBメーカー(売上500億円)、営業100名
  2. イシュー:「営業1人あたりの提案件数を、現在の月10件から月20件にできるか?」
  3. あなたが頂上に置く結論(1行)を書く
  4. 結論を支える3つの根拠を書く
  5. 各根拠の下に置く具体施策を1つずつ書く(計3つ)
  6. 提案をSCQA法でエレベーターピッチ(30秒)にまとめる
模範解答を見る

頂上(結論): 「生成AI による提案書作成の自動化と、CRM データを起点とした顧客選定の高度化により、営業1人あたり月20件の提案を実現できます。」

根拠と施策:

  • 根拠1: 提案書作成の所要時間を80%削減できる。
    施策: 過去の提案書5,000件を学習データに、顧客プロファイルを入力すれば10分でドラフト生成する社内LLMを構築
  • 根拠2: 顧客選定の精度向上で空振り訪問が半減する。
    施策: CRM 過去5年のデータから「成約しやすい顧客プロファイル」を機械学習で予測、営業に推薦リストを毎週配信
  • 根拠3: 提案フォローの自動化で再提案・追客が継続できる。
    施策: メール自動生成と AI による次回コンタクト推奨タイミングの算出

SCQA エレベーターピッチ:

「貴社の営業生産性は業界平均を15%下回っており(S: 状況)、提案書作成と顧客選定に時間が偏っています(C: 複雑化)。これを2倍にできるか?(Q: 問い)はい、3つの施策で月10→20件にできます(A: 答え)。生成AIによる提案書自動化、CRMデータの活用、フォローの自動化です。投資額3億円、回収1.5年、3ヶ月後にPoC完了します。」

ポイント: 答え(A)を最初の1文で言い切れているか? 根拠が3つにMECEに揃っているか? 数字が入っているか? を必ず自己チェックしてください。
