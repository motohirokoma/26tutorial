1. クラウドの3層と「責任の境界」

AI・DX の議論は、ほぼ全てが クラウドの上で起きています。まず3つの層を理解します。

クラウドの3層モデル(IaaS / PaaS / SaaS)
右に行くほど「自分でやる範囲」が減る。色付きが「クラウド事業者の責任」、薄色が「自分の責任」。

主要な単語の最小定義

用語1行定義例
IaaS仮想マシンとネットワークだけ借りて、その上は全部自分AWS EC2 / Azure VM / GCP Compute Engine
PaaSOSや実行環境まで管理してもらい、コードだけ持ち込むHeroku / AWS Elastic Beanstalk / Vercel
SaaS完成済みのソフトを使う、設定するだけSalesforce / Microsoft 365 / Notion
APIソフト同士が会話する窓口。HTTPで叩く呼び出し口OpenAI API / Stripe API / Google Maps API
コンテナアプリと依存物を箱詰めし、どこでも同じ動作にする仕組みDocker, Kubernetes
マイクロサービス大きなアプリを小さな独立サービスの集まりに分割NetflixやUberの内部構成
もっと詳しく: ハイパースケーラー3社の住み分け

AWS(Amazon Web Services): 最古参・最大シェア。サービス数が圧倒的に多く、エンタープライズの主流。

Microsoft Azure: 既存のWindows/Office環境との統合が強み。日本の大企業はAzure採用率が高い。

Google Cloud (GCP): AI/データ分析(BigQuery, Vertex AI)で技術的に先行。スタートアップに人気。

2026年現在、マルチクラウド(複数を併用)も増えています。「AzureがOpenAIと提携している、AWSがAnthropicに巨額投資、GoogleはGeminiを自前展開」という構図で、AIの観点から選定が分かれることもあります。

2. データ基礎 — 構造化と非構造化、そして「箱」

AIの燃料はデータです。データは大きく2種類に分かれ、それぞれを保管する「箱」があります。

種類特徴例主な保管先
構造化データ表の形(行と列)で整っている顧客マスタ、売上、在庫、ログイン履歴RDB / DWH
半構造化JSON / XML など、形はあるが表ではないAPI応答、ログ、設定ファイルNoSQL / データレイク
非構造化そのままでは集計できない文書、画像、音声、動画、メール本文データレイク / オブジェクトストレージ

「業務を回すRDB」→「貯めるデータレイク」→「整えるDWH」→「使うBI/ML」。データの流れの典型形。

覚えておきたい4つの箱

  • RDB(リレーショナルDB) — 業務を支える表形式DB。MySQL、PostgreSQL、Oracle。ACID 特性で正確性が命
  • DWH(データウェアハウス) — 分析専用に最適化された大量データ向けDB。Snowflake、BigQuery、Redshift
  • データレイク — とりあえず生データのまま貯める巨大ストレージ。AWS S3、Azure Data Lake
  • NoSQL — 表に縛られない柔軟なDB。MongoDB、DynamoDB、Redis
2026年のトレンド: Lakehouse

「レイク(柔軟・安い)」と「DWH(速い・整っている)」を統合する Lakehouse(Databricks、Snowflake等が推進)が主流になりつつあります。データ基盤の設計提案ではこの単語が頻出するので覚えておいてください。

3. 統計の入り口 — AIの背骨にある3つの概念

AIエンジンの中身は、突き詰めると確率と統計です。深く触れる必要はありませんが、3つの概念だけは「言葉として通じる」状態にしておきます。

正規分布(ベルカーブ) — 自然界の多くがこの形
平均(中央)を頂点に、両端へ向かって減衰する釣鐘型。多くの現象(身長、テスト点、誤差)がこれに従う。
期待値
「平均すると何が起きるか」の予測値
分散
「平均からどれくらいばらつくか」の指標
確率分布
どの値がどれくらい出やすいかの形

3つの概念の使われ方

  • 回帰: 「広告費を増やすと売上はいくらになるか?」など、連続値を予測する手法
  • 分類: 「このメールはスパムか?」「この画像は猫か?」など、カテゴリを当てる手法
  • クラスタリング: 「顧客を似た者同士のグループに分ける」など、正解なしで構造を発見する手法
もっと詳しく: 「過学習」という落とし穴

過学習(Overfitting)とは、AIが学習データに合わせすぎて、新しいデータには弱くなってしまう現象です。たとえば「過去5年の売上データ」だけを完璧に説明できるモデルを作ると、来年の予測には使えないということがよく起きます。

提案の現場で「精度99%」みたいな数字が出てきたら、それが学習データでの精度か本番に近いテストデータでの精度かを必ず確認してください。前者は危険信号です。

4. セキュリティ基礎 — 4つの英単語

エンタープライズの提案では、必ず「セキュリティはどうなっているのか」という質問が来ます。次の4語を区別できれば最低限の会話はできます。

Authentication

認証 — 誰?

「あなたが本当にあなたか」を確認する仕組み。
パスワード、生体認証、ICカードなど。

Authorization

認可 — 何ができる?

「あなたに何をする権利があるか」を制御する仕組み。
RBAC(役割ベース)、ACL(アクセス制御リスト)。

Encryption

暗号化

盗まれても読めない形に変換すること。
通信時(TLS)と保管時(at-rest)の両方が必要。

PII

PII(個人情報)

Personally Identifiable Information。氏名・住所・メールなど個人を特定できる情報。日本では個人情報保護法の対象。

よく出る場面

顧客から「LLMにデータを投げたら情報が漏れませんか?」と聞かれたら、
① 暗号化(通信・保管)
② API利用(学習に使われない契約か)
③ 認証認可(社内のどの権限の人が叩けるか)
の3点で答えるのが定石です。Ch12 で詳しく扱います。
