Phase 2: Core > Chapter 06

PoC設計から本番デリバリーまで

「PoC は成功したのに本番化しない」がエンタープライズAIの最大の失敗パターン。本章では PoC を本番に繋げる設計、精度・コスト・レイテンシのトレードオフ、運用立ち上げ、インシデント対応までを実戦目線で扱います。

所要時間: 120分 Phase: Core 難易度: ★★★★☆
本章の射程

顧客のAIプロジェクトに伴走するエンタープライズAI実装スタートアップの社員にとって、PoCから本番までの道筋を設計できることが最も重要なスキルです。本章で「PoCはやれば作れる、問題は本番に持ち込むまで」の構造を理解します。本章は2026年4月時点の情報に基づきます。

1. 「PoC死」のメカニズム

業界の調査では、AI PoCの70-85%が本番化しないとされます。これは技術の問題ではなく、ほとんどが設計の問題です。

PoC死の典型パターン(5つ)
graph TB
  PoC["PoC死"] --> P1["① 目的が曖昧"]
  PoC --> P2["② データが本番と違う"]
  PoC --> P3["③ ROIが見えない"]
  PoC --> P4["④ 運用設計がない"]
  PoC --> P5["⑤ ステークホルダー連携不足"]
  P1 --> P1d["「精度99%」だけで成功判定"]
  P2 --> P2d["きれいなテストデータで検証、本番のノイズ未考慮"]
  P3 --> P3d["「すごい」だけで投資判断できない"]
  P4 --> P4d["誰がモデルを運用するか決まっていない"]
  P5 --> P5d["IT部門と現業部門の合意取れず"]
技術的に成功したPoCも、5つのいずれか1つに引っかかると本番に進まない。

典型パターンの解剖

PoC死パターン

「精度99%、でも本番化しない」

  • PoCのデータが本番と質が違う
  • 運用コストが見積もられていない
  • 「精度」が業務インパクトと無関係
  • 誰がいつ運用するか不明
  • 現場が反対している
本番化する PoC

「精度85%、3ヶ月で本番」

  • 事業KPI(売上/コスト)で評価
  • 本番に近いデータで検証
  • ROI計算が完了済み
  • 運用体制が決まっている
  • 現場が「自分たちのもの」と感じている

2. PoC を本番に繋げる設計

「Production-aware PoC」 — 最初から本番を想定する

本番化するPoCの共通点は、PoC開始時から本番運用を想定して設計されていること。「とりあえず動くものを作る」発想を捨てます。

PoC設計の5層チェックリスト
5層全てが「Yes」になるまでPoCは未完。技術的に動くだけでは不十分。

PoCのスコープ管理

PoC期間中に「あれもやって」「これもやって」と要件が膨張するのが典型。「PoCで検証する仮説は1〜2個」と最初に明文化し、それ以外はPoC後の本番化フェーズに回します。

3. 精度・コスト・レイテンシのトレードオフ

本番設計の最大の判断軸が3つの軸のバランスです。「全部最高」は実現不可能、必ず優先順位をつける。

3軸トレードオフの三角形
3軸全部を上げることはできない。何を優先するかを顧客と最初に合意する。

用途別の優先順位例

用途精度コストレイテンシ選択例
カスタマーサポートチャット中★★★ 重要★★ 普通蒸留小モデル + RAG
金融与信判定★★★ 重要中★★ 普通(数秒OK)GPT-4 等の高性能モデル
自動運転判断★★★ 重要中★★★ 重要(50ms以下)専用エッジモデル
夜間バッチ分析★★★ 重要★★★ 重要不問(数時間OK)大型モデル + バッチ推論
社内文書検索★★ 普通★★ 普通★ 急がず(数秒OK)RAG + 中型モデル

コストを下げる5つの常套手段

  • 蒸留 — 大モデルから小モデルへ。精度95%維持、コスト1/10
  • キャッシング — 同じ質問は再計算しない、Anthropic Prompt Caching・Redis 活用
  • ルーティング — 簡単な質問は小モデル、難しい質問だけ大モデル
  • バッチ処理 — リアルタイム不要なら一括処理(API のバッチエンドポイントは50%引き)
  • 量子化 — int8/int4 でモデルサイズ縮小、推論速度UP(主にエッジ)

PoCは「動く」がゴール、本番は「動き続ける」がゴール。6つの門は全部通過必須。

各門で問われること

門問われること典型的な不合格例
① セキュリティ認証認可、暗号化、PII管理、ペネトレーションテスト「とりあえず社内IPで動かしている」
② 性能・SLA同時1000ユーザーで99%成功、レイテンシp95 < 2秒「1人なら動く」状態
③ 監視・ログDatadog等で異常検知、全推論ログ保管、トレーシング「ログはローカル保存のみ」
④ 障害対応夜間障害の連絡フロー、復旧手順書、フォールバック「データサイエンティストの個人スマホに連絡」
⑤ 運用体制運用担当の決定、夜間休日対応、再学習サイクル「決まっていない、業務側に丸投げ」
⑥ 規制遵守GDPR/個人情報法、業界規制(金融、医療)、AI関連法「法務に最後に持っていく」

SLA設計 — 「動き続ける」を数値化する

SLA項目典型値(エンタープライズ)注意点
稼働率99.5% (月3.6時間ダウン)99.9% は急に難しくなる、要望を慎重に
レイテンシ p50 / p951秒 / 3秒以下p99 は外れ値の影響大、p95 で評価
スループット同時 1,000 リクエスト/分ピーク時の3倍を想定
精度劣化検知週次評価、5%劣化で再学習業務影響度に応じて調整
復旧時間 (RTO)障害発生から30分以内夜間休日対応が必要

5. 運用設計 — 動き続けさせる仕組み

本番運用ダッシュボード(典型的な監視軸)
推論回数・レイテンシ・エラー率・コストを時系列で可視化。異常を早期発見する基盤。

運用フェーズで誰が何をやるか — RACI 表

タスク業務部門IT/SI当社(AI実装)
1次トラブル受付RAC
2次原因調査IRC
モデル不具合対応ICR
定期再学習ICR
性能劣化監視IRR
業務影響評価RIC

R = Responsible(実行責任)、A = Accountable(最終責任)、C = Consulted(相談)、I = Informed(報告先)

インシデント対応フロー

本番障害が起きた時の対応シーケンス
sequenceDiagram
  participant M as 監視
  participant O as 運用担当
  participant A as AIチーム
  participant B as 業務部門
  M->>O: 異常検知アラート
  O->>O: 一次切り分け
(5分以内)
  O->>A: モデル不具合の可能性
  A->>A: 切り戻し or 再学習判断
  A->>O: フォールバック適用
  O->>B: 業務影響を共有

監視→切り分け→切り戻し→業務影響共有 を5分単位で動く。事前に「夜間連絡先・切り戻し権限・コミュニケーションテンプレ」を準備。

6. プロジェクトマネジメント特有の罠

AIプロジェクト特有の予測しづらさ

  • 不確実性が高い — 「やってみないと分からない」が多い。固定スケジュールで管理すると破綻
  • データ依存 — 期待精度が出るまでデータの質に振り回される
  • 業務側の正解定義 — 「何が正解か」を業務側が言語化できない
  • 規制・法務の遅れ — 後工程で発覚して再設計が必要に

AIプロジェクト推奨の進め方

伝統的SI型

ウォーターフォール

要件定義→設計→開発→テスト→リリース
問題: 後で要件変更すると大幅遅延、AI特有の不確実性に対応できない

AI実装スタートアップ

段階的・伴走型

PoC(8週)→ 限定本番(3ヶ月)→ 全社展開(6ヶ月)
強み: 各段階で意思決定、軌道修正容易、業務側を巻き込みながら進む

公式ドキュメント必読

米国国立標準技術研究所 NIST が出しているAI Risk Management Frameworkは、エンタープライズAIプロジェクトのリスク管理の事実上の世界標準。提案で「リスク管理の根拠」が問われる場面では引用すると効果的です。

7. 章のまとめ

  • PoC死70-85%。原因の多くは技術ではなく設計の問題
  • 本番化PoCは5層チェック(事業KPI、本番データ、運用試算、スケール、ステークホルダー)を最初から設計
  • 3軸トレードオフ: 精度 vs コスト vs レイテンシ。全部最高は無理、用途で優先順位
  • 本番化の6つの門(セキュリティ・性能・監視・障害・運用・規制)を全て通過
  • 運用はRACI で誰が何をするかを明確化、インシデント対応の事前準備
  • AIプロジェクトは段階的・伴走型、ウォーターフォールでは破綻する

Exercise

Exercise 1
入門

PoC計画書の骨子を作る

顧客は中堅商社、課題は「営業からの問い合わせ対応で、過去案件参照に1案件30分かかっている」。8週間のPoC計画書の骨子を5層チェックリスト全てに沿って書いてください。

模範解答を見る

① 事業KPI

  • 主KPI: 1案件あたりの調査時間 30分 → 5分(83%削減)
  • 副KPI: 調査による営業活動阻害時間 月1000時間 → 200時間
  • 成功判定: 営業100名のうち70名が「使い続けたい」と回答

② 本番データ評価

  • 過去5年の案件データ(契約書、提案書、議事録、メール)5万件で構築
  • 本番に近い「実際の営業質問100問」で評価
  • 機密情報マスキングの仕組みも併せて検証

③ 運用試算

  • 月次運用コスト: API利用料50万円、ベクトルDB10万円、運用工数1名×0.3
  • 初期構築: 2,500万円(8週PoC含む)、本番化追加で1,500万円
  • 1案件30分→5分の削減効果: 月1,000万円相当(時給換算)
  • ROI: 4-5ヶ月でペイバック

④ スケーラビリティ

  • 営業100名・1人1日10回利用 → 月3万回問い合わせを想定
  • ピーク時の3倍まで耐える設計(月10万回まで)
  • Azure AI Search の Standard プラン、GPT-4 Turbo

⑤ ステークホルダー

  • 意思決定者: 営業本部長(主)、情報システム部長(副)
  • 運用責任者: 営業企画課が業務側、情シスがインフラ側
  • PoC期間中の週次定例: 営業3名+情シス1名+当社2名
  • 5週時点で「中間Go/NoGo判断」、業務側が継続意志を示すか確認

共通ポイント: 5層全部に具体的な数字と人名(または役職)が入っているか? 「やってみる」レベルではなく、Go/NoGo判断ができる粒度になっているかをチェック。

Exercise 2
応用

3軸トレードオフ + 本番化の壁の事前対応

顧客: 大手損保会社のコールセンター(オペレーター300名)。会話のリアルタイム要約とFAQ自動応答を提案する。以下を設計してください。

  1. 3軸(精度・コスト・レイテンシ)の優先順位と理由
  2. 選定するモデル構成(LLM、RAG構成、エッジ/クラウド)
  3. 本番化の6つの門のうち、特に厳しい門3つとその対応設計
  4. 3年で投資回収するROI試算(オペレーター時給3,000円、対応1件あたりの平均時間想定)
模範解答を見る

① 3軸の優先順位: レイテンシ > 精度 > コスト

  • レイテンシ最優先: リアルタイム要約は1秒以内に出ないと使われない
  • 精度: 損保業務の正確性は重要だが、Human-in-the-Loop(オペレーターが最終判断)で担保可
  • コスト: 1件あたり数銭程度なら許容(時給比較で十分回収できる)

② モデル構成

  • 音声認識: Whisper API + 業界特化追加学習(保険用語)
  • 会話要約(リアルタイム): Claude Haiku など低レイテンシ蒸留モデル
  • FAQ応答: Claude Sonnet + RAG(社内マニュアル数千ページ、Azure AI Search)
  • クラウド主体、エッジは音声入力デバイスのみ
  • 個人情報は通話開始時にマスキング、API送信は匿名化済み

③ 厳しい3つの門

  • セキュリティ: 顧客通話データの取り扱いが最も厳しい。Azure 国内リージョン、暗号化(at-rest+in-transit)、アクセスログ全保管、個人情報マスキング、監査要件対応
  • 規制遵守: 金融庁ガイドライン、保険業法、個人情報保護法。AI 関連法とEU AI Act の影響評価。コンプライアンス部門と早期合意
  • 運用体制: 24時間365日のコール業務に対応する監視・障害対応。当社+運用ベンダーで夜間休日体制構築。フォールバック(AI停止時は従来運用へ即切替)

④ ROI試算

  • オペレーター300名×時給3,000円
  • 1件あたり平均8分→AI支援で6分(25%削減)
  • 1日1人20件処理 → 削減時間 = 300人×20件×2分×250営業日 = 50万時間/年
  • 金額換算: 50万時間×3,000円 = 15億円/年
  • 年間運用コスト: API+ベクトルDB+運用 = 1.5億円
  • 初期構築: 5億円(PoC+本番化+運用立ち上げ)
  • 3年累積: 削減15×3 - 運用1.5×3 - 初期5 = 35.5億円ペイバック(初年度から黒字)

共通ポイント: 提案書の「投資効果」では、必ず人件費削減か売上増加で具体的な金額を出します。「業務効率化」だけでは経営層の意思決定に効きません。
