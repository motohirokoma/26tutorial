生成AI/LLM 本質編 — Transformer→RAG→Fine-tuning→Agent

2026年の主役技術 LLM(大規模言語モデル)を、ビジネスパーソンが顧客と対峙できる深さまで掘り下げます。「Transformer って何?」「RAG はなぜ必要?」「Fine-tuning との違いは?」に全て答えられる状態を目指します。

所要時間: 120分 Phase: Core 難易度: ★★★☆☆
本章の到達点 (L4)

章を終えると、顧客の課題に対して「これはRAG向き」「これはFine-tuning向き」「これはAgent向き」と技術選定の判断軸を持って提案できるようになります。本章の記述は2026年4月時点の情報に基づきます(LLM領域は3ヶ月で常識が変わるため、各章末リソースで最新を確認してください)。

1. LLMとは何か — 次の単語を予測する機械

LLM(Large Language Model)を一言で説明すると「次に来る単語を確率的に予測する機械」です。これだけです。難しく聞こえる仕組みも、根っこはこのシンプルな考え。

LLMの基本動作 — 単語の確率予測
入力に対して次の単語の確率分布を出力 → 確率の高いものを選んで入力に追加 → また予測…を繰り返すだけ。これが「自己回帰生成」と呼ばれる、LLMの基本動作。

これがすべて、というのはやや誇張ですが、本質はそうです。すごいのは「次の単語予測」だけを巨大データで超高精度に学習させると、要約も翻訳も会話もコード生成もできてしまうこと。「単純な目標の積み上げから、複雑な能力が創発する」ことが GPT 系の発見でした。

「学習」と「推論」を区別する

フェーズやることコスト頻度
事前学習(Pretraining)インターネット規模のテキストで「単語予測」を学ぶ数億〜数十億ドル1モデルにつき1回(GPT-5など)
事後学習(Post-training)人間の好みに沿うように調整(RLHF など)数百万ドル製品化のたび
推論(Inference)学習済みモデルで実際に応答を生成する1トークン数円〜数銭都度(ユーザー操作ごと)
提案で混同してはいけないポイント

「LLM のコスト」と顧客に言われたら、必ず「学習コスト」と「推論コスト」を分けて聞きます。普段の業務利用で気にするのはほぼ推論コストのみ(API 呼び出しの料金)。学習コストは、自社で基盤モデルを作る場合か、Fine-tuning する場合の話です。

2. Transformer — Attentionが全てを変えた

2017年に Google 研究者が発表した論文「Attention Is All You Need」で提唱された Transformer アーキテクチャが、現在の LLM の土台です。何が革命だったのか。

Transformerの構造(超簡略版)
graph TB
  Input["入力テキスト
「The cat sat on」"] --> Tok["トークン化
[The, cat, sat, on]"]
  Tok --> Emb["埋め込み(Embedding)
各トークンを数値ベクトルに"]
  Emb --> Att["Self-Attention
各単語が他の単語を見る"]
  Att --> FF["フィードフォワード層"]
  FF --> Att2["(N回繰り返し)"]
  Att2 --> Out["次トークン予測
「the」"]
入力を数値化 → Self-Attention で文脈を取り込む → 多層に重ねて次トークンを予測。GPT-4 では96層、Claude 3.5 ではさらに多層。

Self-Attention — 「他の単語をどれくらい見るか」

Attentionは「文中のある単語が、他のどの単語にどれくらい注目するか」を計算する仕組みです。これにより、文全体の文脈を踏まえた表現が作れます。

「The cat sat on the mat」の Attention を可視化
単語 sat が他の各単語にどれくらい「注意」を向けるかをスコア化(線の太さ)。文全体の文脈を全単語で同時に把握する。
もっと詳しく: なぜ Transformer が革命だったのか

Transformer 以前の主流は RNN(リカレントニューラルネット)で、単語を1つずつ順番に処理していました。これは:

  • 遅い: 並列化できない、長い文章の処理が極端に遅い
  • 長期依存が苦手: 文の前半を後半まで覚えていられない

Transformer は Self-Attention により全単語を同時並列で処理でき、長文の文脈把握も得意。GPU で大量並列学習できることが「巨大化のスケール則」を発見させ、結果として LLM の爆発的な進化に繋がりました。

3. プロンプト工学 — 使いこなしの基礎

LLMの能力を引き出す唯一のインターフェース、それがプロンプトです。「適切なプロンプト」と「適当なプロンプト」では、同じモデルでも出力品質が劇的に変わります。

プロンプトの基本構造(4要素)

  • Role(役割) — 「あなたはBtoBの営業マネジャーです」など、立場を指定
  • Task(タスク) — 「以下の会議録から決定事項を抽出してください」など、具体的指示
  • Context(文脈) — 関連情報・前提知識・参考資料
  • Format(形式) — 「箇条書きで5項目以内」「JSON形式で」など、出力フォーマット

4. RAG — 知識を外付けする

LLM の最大の弱点は2つ: 「学習時点までの情報しか知らない」と「知らないことを"それっぽく"作話する(ハルシネーション)」。これを解決するのが RAG(Retrieval-Augmented Generation)です。

RAG の仕組み(エンタープライズで最も使われる構成)
graph LR
  Q["ユーザー質問"] --> Embed["質問を
ベクトル化"]
  Embed --> Search["ベクトルDBで
類似文書を検索"]
  KB[(社内文書
事前ベクトル化済み)] --> Search
  Search --> Top["関連文書 Top-K"]
  Top --> Prompt["プロンプト構築:
質問 + 関連文書"]
  Prompt --> LLM["LLM"]
  LLM --> Ans["回答 + 引用元"]
質問が来たら → 社内文書から関連箇所を検索 → 質問と一緒にLLMへ → 文書に基づく回答が返る。エンタープライズの「自社データを使ったAI」のほぼ全ての実体はこれ。

RAG の主要構成要素

  • ドキュメントストア — 元の社内文書(PDF、Confluence、Slack ログ等)
  • チャンキング — 文書を意味のある単位に分割(数百〜数千文字)
  • Embedding モデル — チャンクと質問をベクトル化(text-embedding-3-large 等)
  • ベクトルDB — Pinecone、Weaviate、Qdrant、AWS OpenSearch、Azure AI Search
  • LLM — 検索結果を踏まえて回答(GPT-4、Claude、Gemini 等)

RAG の最小実装イメージ(疑似 Python)

# 1. 文書を事前にベクトル化(オフライン処理)
from openai import OpenAI
import pinecone

client = OpenAI()
index = pinecone.Index('company-docs')

for chunk in document_chunks:
    vec = client.embeddings.create(
        model='text-embedding-3-large', input=chunk.text
    ).data[0].embedding
    index.upsert([(chunk.id, vec, {'text': chunk.text})])

# 2. ユーザー質問が来たら検索 → LLM(オンライン処理)
def answer(question):
    q_vec = client.embeddings.create(
        model='text-embedding-3-large', input=question
    ).data[0].embedding

    # 類似する文書 Top 5 を取得
    results = index.query(vector=q_vec, top_k=5, include_metadata=True)
    context = '\n\n'.join([m.metadata['text'] for m in results.matches])

    # LLM に「文書 + 質問」を渡して回答生成
    response = client.chat.completions.create(
        model='gpt-4-turbo',
        messages=[
            {'role': 'system', 'content': '以下の文書のみを根拠に回答してください'},
            {'role': 'user', 'content': f'文書:\n{context}\n\n質問: {question}'}
        ]
    )
    return response.choices[0].message.content

※ 実際の本番では、リランカー導入・チャンキング戦略・引用元提示など多数の工夫が入りますが、骨格はこの形です。

RAG の使いどころと限界

向いている

頻繁に更新される情報

  • 社内マニュアル・FAQ
  • 製品仕様書・契約書
  • 過去の議事録・案件ログ
  • 法律・規制の最新版
向いていない

暗黙知・スタイル変更

  • 「弊社らしい」文章スタイル
  • 業界特有の言い回し・専門用語
  • 暗黙のルールや判断基準
  • (これらは Fine-tuning 領域)
もっと詳しく: RAG の精度を上げる5つの工夫
  1. チャンキング戦略: 単純な文字数分割ではなく、見出しや段落で意味的に切る。重複オーバーラップを入れる
  2. ハイブリッド検索: ベクトル検索 + キーワード検索(BM25)を組み合わせ、どちらの強みも活かす
  3. リランカー: 検索結果を二段階目の小型モデルで再ランキングして精度UP(Cohere Rerank、bge-reranker)
  4. クエリ拡張: ユーザー質問を LLM で言い換え3-5バージョン作り、それぞれで検索して結果を統合
  5. 引用元提示: 回答に「この部分は文書Xの第3章より」と引用元を必ず付ける(信頼性 + 検証可能性)

5. Fine-tuning と Distillation

RAG が「外付けハードディスク」だとすれば、Fine-tuningは「モデルの脳に直接書き込む」アプローチです。

Fine-tuning vs RAG — 使い分けの判断軸

どちらを選ぶか?の判断フロー
graph TD
  Start["顧客の課題"] --> Q1{"知識を増やしたい?
or スタイルを変えたい?"}
  Q1 -- "知識を増やす" --> Q2{"情報が頻繁に
更新される?"}
  Q1 -- "スタイル/振る舞い変更" --> FT["Fine-tuning"]
  Q2 -- "Yes" --> RAG["RAG"]
  Q2 -- "No (固定知識)" --> Q3{"件数は?"}
  Q3 -- "数千〜" --> RAG
  Q3 -- "数百以下" --> Both["Few-shot で十分"]
迷ったら RAG から始めて、それで足りなければ Fine-tuning を加える、が定石。逆ではない。
観点RAGFine-tuning
強み知識更新が容易、引用可能、低コストスタイル統一、振る舞いの細かい調整、推論が速い
弱み検索精度に依存、コンテキスト枯渇更新が大変、データ準備コスト大、忘却リスク
初期コスト低(ベクトルDB構築のみ)中〜高(訓練データ作成 + 学習)
運用コスト中(都度の検索 + LLM)低(LLMのみ)
典型ユースケース社内Q&A、ドキュメント検索、最新情報応答業界用語の口調、ブランドトーン、特定タスクの最適化

Distillation(蒸留) — 大モデルから小モデルへ

Distillationは「大きく賢いモデル(教師)に出力させたデータを使って、小さなモデル(生徒)を訓練する」手法。賢さは保ちつつコストを大幅に下げるテクニックで、2025年以降のエンタープライズ展開で爆発的に普及しました。

Distillation の構造

  Big["Teacher
大モデル
(GPT-4 等)"] -->|"出力ペアを生成"| Data["訓練データ
10万-100万件"]
  Data --> Train["Fine-tune"]

小モデル
(7B-13B)"] --> Train

速い・安い・賢い"]
小モデルが大モデルの「考え方」を真似する。コストが10分の1、レイテンシが3分の1、性能は90%、という設計が実現可能。
2026年のトレンド: 蒸留+RAG ハイブリッド

多くのエンタープライズが「RAG で知識を外付け + 業界特化モデルを蒸留で作る + その上で軽い Fine-tuning」というハイブリッド構成を採用しています。一つの手法だけでは最適にならないことが分かってきた、というのが現場感です。

6. AI Agent — 次のパラダイム

2024年後半から急速に立ち上がってきたのが AI Agent。LLM が単発の応答を返すのではなく、ツールを使い、複数ステップを自律的に実行する仕組みです。Anthropic の Claude、OpenAI の GPT-4 が agentic 機能を強化、Cursor / Devin / Manus 等の専用エージェントも登場。

Agent の動作ループ(ReAct パターン)
graph LR
  Goal["目標を受け取る"] --> Plan["計画を立てる
(LLM)"]
  Plan --> Act["ツールを実行
(検索/コード実行/API呼び出し)"]
  Act --> Obs["結果を観察"]
  Obs --> Eval{"目標達成?"}
  Eval -- "No" --> Plan
  Eval -- "Yes" --> Done["完了"]
「考える(Reasoning)」と「行動する(Acting)」を交互に繰り返す。複雑なタスクを多段で解ける反面、暴走や無限ループのリスクも。

Agent が使う「ツール」とは何か

  • Web 検索 — 最新情報を取りに行く
  • コード実行 — Python のサンドボックスでデータ処理・計算
  • データベース問い合わせ — SQL を発行して結果を取得
  • メール送信・カレンダー操作 — Gmail/Outlook と連携
  • クラウド API 呼び出し — Slack、Salesforce、社内システム
  • 他の AI モデル呼び出し — 画像生成・音声合成・専門モデル

Single Agent vs Multi-Agent

Single Agent

1人で全部

1つのLLMが複数ツールを使い分けて目標達成。
シンプルで扱いやすいが、複雑なタスクで詰まる。

Multi-Agent

専門家チーム

「企画担当」「分析担当」「実装担当」など複数Agentが協調。
高品質だが設計と運用が複雑、コスト高。

Agent の現実 — 2026年時点の率直な評価

公式ドキュメント必読

Agent 系は安全性と運用が極めて難しい領域です。Anthropic の Building Effective Agents は実装する全エンジニアが読むべき指針。「Agent は最終手段」「ワークフロー(プログラムで決まったフロー)で済むなら Agent にしない」という Anthropic の立場は、実務の判断軸として重要です。

  • 動く領域: コード生成・データ分析・調査タスク・カスタマーサポート1次対応
  • 苦戦する領域: 長期記憶を要する業務、感情労働、重要な意思決定の代行
  • 禁忌領域: 法的責任を負う行為(契約締結等)、人命に関わる判断、不可逆な金銭授受

顧客への提案で使う判断軸

  • データ主権 — 海外API送信OKか / 国内に置く必要があるか / オンプレ必須か
  • コスト感度 — 1リクエスト数銭か数円か。BtoB の数千件処理ならGPT-4、社内チャット10万人なら蒸留小モデル
  • レイテンシ — リアルタイム会話なら1秒以内、バッチ処理なら数分でもOK
  • マルチモーダル — 画像・音声・動画を扱うか
  • カスタム性 — Fine-tuning する予定か、ベースモデルを切り替えたいか

8. 章のまとめ

  • LLM は「次の単語予測」を巨大データで学習した汎用機械。Transformer 上の Self-Attention が革命の核
  • プロンプト4要素(Role/Task/Context/Format)を意識すれば品質は劇的に変わる
  • RAG: 知識の外付け、エンタープライズの主流。最初に試すべき手法
  • Fine-tuning: スタイル・振る舞いの調整。RAGで足りない時の二段階目
  • Distillation: 大モデル→小モデルでコスト10分の1へ。2026年の主流
  • Agent: ツールを使う自律実行。強力だが慎重に。「ワークフロー優先、Agent は最終手段」
  • モデル選定はデータ主権・コスト・レイテンシ・モダリティ・カスタム性の5軸で判断

以下のシナリオで、Role/Task/Context/Format の4要素を全て含むプロンプトを書いてください。

  1. シナリオ: BtoB SaaS 営業として、新規見込客向けの初回コンタクトメールを生成したい
  2. 顧客情報: 中堅製造業、IT予算削減プレッシャー、過去当社の競合製品を3年使用
  3. 各要素を明示的に書く(「Role:」「Task:」のように節を分ける)
模範解答を見る 
【Role】
あなたはBtoB SaaS の営業マネジャーで、製造業向けに10年の経験があります。

【Task】
中堅製造業の見込客に向けた、初回コンタクトメールを作成してください。

【Context】
- 顧客: 中堅製造業(売上500億円規模)
- 現状: 競合製品Xを3年使用中
- ペイン: IT予算削減のプレッシャーが強い、コスト効率の改善を経営層から指示されている
- 当社の強み: 同業他社で競合Xから乗り換えで30%コスト削減実績

【Format】
- 件名: 件名(15文字以内、開封したくなる文言)
- 本文: 200文字以内
- 構成: 「相手の状況の理解」→「具体的な数字を含む価値提案」→「次のアクション(15分電話)」
- トーン: 押し付けがましくない、相手の判断を尊重する書きぶり

ポイント: ContextとFormat が具体的であるほど、出力品質が安定します。「いい感じで書いて」ではなく「この観点でこの形式で」を必ず明示。

Exercise 2
応用

顧客に技術選定を提案する

以下の3つのシナリオそれぞれに対して、最適な技術アプローチ(Few-shot / RAG / Fine-tuning / Agent / それらの組み合わせ)と、選定理由を提案してください。

  1. シナリオA: 大手損保会社が、社内の保険商品マニュアル(数千ページ、毎月改訂)を対象に、コールセンター向けの社内Q&Aシステムを作りたい
  2. シナリオB: 中堅広告代理店が、社内のクリエイティブのブランドトーン(過去5年の優秀コピー数千本)を学習させ、新規コピー候補を生成するAIを作りたい
  3. シナリオC: 製造業の生産管理部門が、社内システム複数(ERP・MES・在庫DB)を横断して、自然言語で「明日の出荷予定は?」と質問できる仕組みを作りたい
模範解答を見る

シナリオA(損保Q&A): RAG が最適

  • 理由: 知識(マニュアル)が頻繁に更新される → Fine-tuningでは追従不能。引用元提示が必要(規制業種) → RAGで「この回答は商品約款p.42より」と明示できる
  • 構成: ベクトルDB (Azure AI Search等、規制対応のため国内リージョン) + GPT-4 or Claude(コンプラ厳格な業界では引用に強いClaude推奨) + ハイブリッド検索 + リランカー

シナリオB(広告コピー生成): Fine-tuning + Few-shot のハイブリッド

  • 理由: 求められるのはスタイル(ブランドトーン)であり、知識ではない → RAGよりFine-tuning向き。データ量(数千本)も Fine-tuningに十分
  • 構成: 中型モデル(Llama 4 13B など)を社内環境で Fine-tuning + プロンプトに3-5本の Few-shot 事例も併用 + 必要なら蒸留で軽量化
  • 注意: 著作権の扱い、人間ライターとの分業設計、生成物の選定プロセスは別途設計

シナリオC(横断質問): Agent が最適

  • 理由: 複数システムを跨ぐ問い合わせは、決まった検索ではなく「質問→どのDBを叩くか判断→SQL生成→結果統合→回答」の動的フローが必要
  • 構成: GPT-4 or Claude をAgent本体として、ツール = ERP API / MES API / 在庫DB SQL実行。Read系のみに権限制限、書き込みは禁止。Audit ログで全行動を記録
  • 注意: 「Agent は最終手段」原則を守るなら、まず一番頻度が高い質問パターン(出荷予定確認等)はワークフロー化、複雑な横断質問のみAgentに回す設計が安全

共通ポイント: 「LLMで何かやる」ではなく、知識の更新頻度・スタイルの重要度・処理の動的性で道具を選ぶ。提案では必ず「なぜ Fine-tuning ではなく RAG か」「なぜ Agent ではなくワークフローか」を1行で説明できる状態を目指してください。
